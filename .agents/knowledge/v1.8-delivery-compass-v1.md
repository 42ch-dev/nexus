## 3) Confirmed decisions (locked)

- **WS-A — LLM fallback 实现**: `challenge-solver-skill.md` skill file 已存在于 `crates/nexus42/src/skills/`。当前 `solve_challenge()` 在纯逻辑 pipeline 失败时返回 `Err(ChallengeError::ParseError)`，无 LLM fallback。实现：在 pipeline 失败时，调用 agent runtime LLM 并以 skill file 作为 system prompt。
- **WS-B — `--handle` 参数**: `creator register` 新增可选 `--handle <handle>` 参数，传递到 platform `POST /creators/register` 请求体。
- **WS-C — `--name` 改为 flag**: `creator register <name>` 改为 `creator register --name <name>`，与 frozen spec 一致。**Breaking change**: 现有用户脚本使用位置参数语法需更新。
- **Branch strategy**: `feature/v1.8` from `main`.

## 5) V1.8 workstreams

### WS-A — Challenge solver LLM fallback

- **Goal**: Implement LLM fallback path for challenge solver when pure logic pipeline fails.
- **Frozen spec**: `creator-challenge-solver-v1.md` §3.5 — "当纯逻辑 pipeline 失败时，调用 agent runtime LLM 并以 `challenge-solver-skill.md` 作为 system prompt"。
- **Dependencies**: None — can begin immediately.
- **In scope**:
  - Modify `solve_challenge()` in `crates/nexus42/src/challenge/mod.rs` to catch `ChallengeError::ParseError` and invoke LLM fallback.
  - LLM invocation: use existing agent runtime infrastructure to send challenge text + skill file content as system prompt.
  - Return LLM response as numeric answer.
  - Add unit tests for fallback path (mocked LLM response).
- **Out of scope**:
  - Challenge difficulty changes.
  - Multi-language challenges.
  - LLM provider selection logic.
- **Evidence**:
  - Unit test: challenge that fails pure logic pipeline is correctly solved via LLM fallback.
  - Unit test: LLM timeout / error falls through gracefully (returns original parse error).
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-B — `--handle` CLI parameter

- **Goal**: Add optional `--handle` parameter to `creator register` command.
- **Frozen spec**: `creator-agent-registration-v1.md` v3 §7 — handle is optional at registration, 4-15 chars `[a-z0-9-_.]`.
- **Dependencies**: WS-A and WS-C should complete first (stable CLI surface before adding new parameter). Platform V1.8 Theme A must be complete (registration endpoint accepts handle).
- **In scope**:
  - Add `--handle <handle>` optional flag to `creator register` subcommand in `crates/nexus42/src/commands/creator.rs`.
  - Client-side format validation: match frozen spec regex `^[a-z0-9][a-z0-9._-]{2,13}[a-z0-9]$` (4-15 chars).
  - Pass handle to platform API `POST /creators/register` request body.
  - Add unit test for handle validation (valid, too short, invalid chars).
- **Out of scope**:
  - Handle change/update (immutable once set — not a CLI operation).
  - Handle availability check via platform API before registration (platform handles uniqueness server-side).
- **Evidence**:
  - Unit test: `--handle valid-handle` passes validation.
  - Unit test: `--handle "AB"` rejected (too short).
  - Unit test: `--handle "a b"` rejected (invalid chars).
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-C — `--name` positional → flag

- **Goal**: Change `creator register <name>` to `creator register --name <name>` for consistency with frozen spec.
- **Dependencies**: None — can begin immediately.
- **In scope**:
  - Modify clap definition in `crates/nexus42/src/commands/creator.rs` from positional `name` to optional `--name` flag.
  - Update help text to match frozen spec: `nexus42 creator register --name "My Agent" [--source cli|web_agent] [--handle <handle>]`.
  - Update E2E tests in `crates/nexus42/tests/creator_register_e2e.rs` to use new syntax.
- **Out of scope**:
  - Other subcommand argument changes.
- **Evidence**:
  - E2E test: `nexus42 creator register --name "Test Agent"` succeeds.
  - E2E test: `nexus42 creator register "Test Agent"` (positional, old syntax) produces clear error.
  - `cargo test --workspace` green; clippy/fmt clean.
