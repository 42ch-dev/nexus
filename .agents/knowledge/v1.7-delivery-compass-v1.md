## 4) V1.7 workstreams (execution decomposition)

### WS-A — DTO residual closure

- **Goal**: Resolve 3 V1.6 WS-B QC residuals (DTO-C1 medium, DTO-C2 low, DTO-W1 nit) to zero open items in that category.
- **Dependencies**: none — can begin immediately.
- **Deliverables**:
  - **DTO-C1**: Propagate `mcp_servers` field from `NexusNewSessionRequest` to the SDK's `NewSessionRequest` in `sdk_new_session_request_from_nexus`. Add unit test verifying the field is not silently dropped. Scope: `crates/nexus-acp-host/src/client.rs`.
  - **DTO-C2**: Replace `.parse::<u16>().unwrap_or(1)` with explicit error handling — either return `Result<ProtocolVersion, ParseError>` or log at `warn!` level and use a documented fallback. Scope: `crates/nexus-acp-host/src/client.rs`.
  - **DTO-W1**: Add `#[derive(Eq)]` to `NexusContentBlock` (requires all fields to implement `Eq`). Scope: `crates/nexus-contracts/src/local/acp/types.rs`.
- **Evidence**:
  - Unit test: `NexusNewSessionRequest` with `mcp_servers: Some(vec![...])` produces SDK request with matching field.
  - Unit test: invalid protocol version string produces observable error or warning (not silent default).
  - Compile check: `NexusContentBlock` implements `Eq`.
  - `cargo test --workspace` green; `cargo clippy --all -- -D warnings` clean.

### WS-B — Permission residual closure

- **Goal**: Resolve 3 V1.6 WS-C QC residuals (PERM-W1, PERM-W2, PERM-W3) to zero open items in that category.
- **Dependencies**: none — can begin immediately (parallel with WS-A).
- **Deliverables**:
  - **PERM-W1**: Replace TOML serialization with `toml_edit` for round-trip preservation of comments and formatting in `policy.rs` save path. Scope: `crates/nexus-acp-host/src/policy.rs`.
  - **PERM-W2**: When `--agent <id>` filter is used in `nexus42 permission list`, include global (non-agent-specific) rules in the output alongside agent-specific rules. Scope: `crates/nexus42/src/commands/permission.rs`.
  - **PERM-W3**: Validate user-supplied TOML keys (agent name, capability name, effect) against known schema before persisting. Reject unknown keys with a clear error message. Scope: `crates/nexus42/src/commands/permission.rs`.
- **Evidence**:
  - Round-trip test: write permissions with comments, read back, verify comments preserved.
  - Filter test: `nexus42 permission list --agent test-agent` shows both global rules and agent-specific rules.
  - Validation test: `nexus42 permission grant test-agent unknown_capability` produces error; invalid TOML key rejected.
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-C — Schedule residual closure

- **Goal**: Resolve 2 V1.5 QC residuals (R4 low, R7 nit) from the orchestration scheduler module.
- **Dependencies**: none — can begin immediately (parallel with WS-A/WS-B).
- **Deliverables**:
  - **R4**: Either (a) implement DST wall-clock jump detection in `SystemClock` (detect when `Instant::now()` jumps > threshold relative to monotonic clock), or (b) downgrade documentation to accurately state that `SystemClock` does not provide DST-safety guarantees. Decision: implement option (a) if straightforward (compare wall-clock delta against monotonic delta), else option (b). Scope: `crates/nexus-orchestration/src/scheduler/mod.rs`.
  - **R7**: Clean up `schedule_guards` entries when schedules enter terminal states (completed, failed, cancelled). Add a `remove()` call in the schedule terminal-state transition path. Scope: `crates/nexus-orchestration/src/schedule/derivation.rs`.
- **Evidence**:
  - Unit test: `SystemClock` detects synthetic clock jump (if implemented) or documentation accurately reflects limitation.
  - Unit test: after schedule reaches terminal state, `schedule_guards` no longer contains entry for that schedule.
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-D — ACP SDK migration to v0.11.0+ (DF-04)

- **Goal**: Upgrade `agent-client-protocol` from `=0.10.4` to `0.11.x`, adopting stabilized new features while preserving the V1.6 DTO decoupling boundary.
- **Authoritative design**: [acp-client-tech-spec-v2.md](../archived/knowledge/acp-client-tech-spec-v2.md) §1 (adapter-trait policy), §11 (`nexus-acp-host` crate boundary).
- **Dependencies**: WS-A should complete first (DTO-C1/C2 fixes may affect conversion code that WS-D will also touch).
- **In scope**:
  - Bump `agent-client-protocol` dependency from `=0.10.4` to `^0.11.0` (or pin to specific patch, e.g., `=0.11.7` for stability).
  - Update `AcpSdkAdapter` internals to compile against v0.11.0 API (struct naming alignment per PR #637, `strum` 0.28 bump).
  - Adopt stabilized v0.11.1+ features: `session/list` and `session_info_update` (useful for multi-agent worker session discovery in WS-E).
  - Verify `NexusAcpClient` trait public API remains unchanged (V1.6 DTO decoupling should isolate consumers).
  - Handle `strum` 0.27→0.28 semver bump impact on Nexus code (if any direct strum usage).
- **Out of scope**:
  - Full sacp migration (Nexus does not depend on sacp; sacp 11.0.0 is a third-party crate).
  - Adopting `unstable` features (session/stop, message ID, auth methods) — these remain gated behind feature flags and are not stable API.
  - Client Handler rewrite.
  - Transport/registry/session_manager changes (no SDK type exposure beyond adapter).
- **Evidence**:
  - `cargo build --workspace` succeeds with new SDK version.
  - `cargo doc --package nexus-acp-host` shows `NexusAcpClient` trait unchanged (zero new SDK types in public signatures).
  - Consumers (e.g., `nexus-orchestration`) compile without changes.
  - Optional: integration test exercising `session/list` if adopted.
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-E — Multi-agent worker (DF-01, OQ-7)

- **Goal**: Extend the one-worker-per-creator MVP to support (1) a `WorkerRegistry` that manages one worker per active creator concurrently, and (2) each worker hosting >1 ACP agent concurrently with session routing. Agent configuration separates three concepts: agent (which ACP binary), role (which persona/system prompt), and model (which LLM).
- **Authoritative design**: [orchestration-engine-v1.md](../archived/knowledge/orchestration-engine-v1.md) §6 (worker model, IPC), §9.2 (concurrency contract), §11 OQ-7 (multi-agent open question).
- **Dependencies**: WS-D (ACP SDK v0.11.0+ provides `session/list` useful for session discovery in multi-agent context). WS-A/B/C should complete first (stable baseline).
- **Design phase deliverables** (before implementation):
  - **Approach selection**: Evaluate (a) multiplex one worker with session routing vs (b) spawn sibling workers. Document trade-offs and select approach. **Selected: Approach A** (multiplex one worker).
  - **WorkerRegistry design**: Daemon maintains `HashMap<CreatorId, WorkerHandle>` for multi-creator worker management. Includes lazy spawn, lookup, crash recovery, idle eviction.
  - **AgentConfig redesign**: Three-concept separation — `acp_agent_id` (ACP Registry), `role` (preset-defined persona), `model` (LLM model name). Agent/model selection split across three layers: preset `recommended_models`, user-level config (`~/.nexus42/agents.toml`), CLI `--agent-ref`.
  - **Role/persona feature**: Presets define roles with `system_prompt_file`; injected as first message in ACP session. Scoped within WS-E (not a separate workstream).
  - **IPC extension design**: Extend the JSON-RPC 2.0 message catalogue (§6.4) with `session_id`-routed requests for the selected approach.
  - **Concurrency contract update**: Define how multiple ACP sessions per creator interact with the engine's at-most-one-ACP-Schedule constraint (§9.2).
  - **Session lifecycle**: Define how agent switching works within a worker (new session, session handoff, session close).
  - **Error handling**: Define how worker failure affects multiple concurrent agent sessions.
- **Implementation deliverables** (after design approval):
  - WorkerRegistry: `HashMap<CreatorId, WorkerHandle>` with lazy spawn, lookup, crash recovery, idle eviction.
  - Worker process changes to support >1 concurrent ACP agent subprocess.
  - IPC protocol extension for `session_id`-routed requests/responses.
  - Engine integration: Worker Manager API to manage multi-agent sessions.
  - Preset YAML `agents` and `roles` sections for agent/role/model configuration.
  - `nexus42 acp-worker` CLI flags for specifying agents, models, and roles.
  - Role system prompt injection on ACP session creation.
- **Out of scope**:
  - Dynamic agent discovery (registry-based) — static configuration only.
  - Role marketplace / sharing — roles are preset-local.
  - Dynamic role switching mid-session — roles set at session creation, immutable.
- **Evidence**:
  - Design doc: approach selection with trade-off analysis (§4.1 deliverable).
  - Integration test: worker hosts 2 ACP agents concurrently; both respond to prompts independently.
  - Concurrency test: simultaneous prompts to different agents on the same worker produce correct responses.
  - Error test: one agent subprocess crashes; other agent continues unaffected.
  - Registry test: daemon manages workers for 2+ creators concurrently.
  - Role test: agent slot with role injects system prompt correctly.
  - `cargo test --workspace` green; clippy/fmt clean.

## 7) Cross-cutting policy

### Branch strategy

- Integration branch: `feature/v1.7` from `main`.
- Per-WS topic branches when ≥2 concurrent writers (expected: WS-A/B/C parallel, then WS-D, then WS-E).
- Merge to `main` only after M4 gate passes.

### CI requirements (mandatory before merge)

- `pnpm run validate-schemas` — clean.
- `pnpm run codegen` + `verify-codegen` — no diff on generated output (if schemas touched).
- `cargo +nightly fmt --all -- --check` — clean.
- `cargo clippy --all -- -D warnings` — clean.
- `SQLX_OFFLINE=true cargo sqlx prepare --workspace --all -- --all-targets --check` — passes.
- `cargo test --workspace` — green.
- CI linkage gate (no ACP SDK in daemon) — passes (inherited; verify after WS-D).

### sqlx compile-time macro policy

- Inherited from V1.4/V1.5/V1.6: all new queries use compile-time macros; runtime queries require `// SAFETY:` comment.
- After any query change: `cargo sqlx prepare --workspace --all -- --all-targets` and commit `.sqlx/` metadata.

### Platform sync cadence

- No platform schema changes expected in V1.7.
- If WS-D DTO changes affect published contract types, bump `@42ch/nexus-contracts` and notify platform team.
