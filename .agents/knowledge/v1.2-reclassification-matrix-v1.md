## 1) Purpose

This document maps every legacy roadmap item, ADR anchor, spec entry, and tech-debt item to the **V1.2 / V1.3 / V1.4 / Backlog** program tracks defined in [v1.2-delivery-compass-v1.md](v1.2-delivery-compass-v1.md) §3 and `pre-freeze-spec-log.md` §0. It serves as the single source of truth for version-tagging future plan rows.

**Conflict resolution**: If this matrix conflicts with `v1.2-delivery-compass-v1.md` §2 (locked decisions), the compass wins. If it conflicts with `pre-freeze-spec-log.md` §0.2 (ADR→version溯源), the spec log wins. Update this matrix to align.

## 2) Version-tagging Rules

When creating or reviewing a plan row, apply these rules **in order**:

| # | Rule | Rationale |
|---|------|-----------|
| R1 | **Hard platform dependency → V1.3 minimum.** If the item requires platform HTTP e2e to close its acceptance gate (user registration, Creator binding, billing, Explore ranking), it cannot be V1.2. | Compass §3 V1.2 Out; §7 dependency rule |
| R2 | **`local_only` path item → V1.2 candidate** if it can be verified without platform. If it also needs `local_first`/`cloud_enhanced` for full value, split: core path → V1.2; platform enhancement → V1.3. | Compass §2 locked decision; ADR-017 |
| R3 | **Switchable/runnable/degradable modes** → V1.2 if the *switch mechanism* and *degradation path* can be demonstrated with synthetic/mocked platform responses. Performance/accuracy thresholds → V1.3. | Compass §2 locked decision |
| R4 | **Billing / Stripe / subscription closure** → V1.4. No billing item ships before V1.4 program track. | Compass §3; pre-freeze §0 |
| R5 | **Explore community-facing (ranking, cold-start, publish compliance, E3 轻互动)** → V1.4. | Compass §3; ADR-011/012/013 |
| R6 | **Context Assembly optimization (platform-side enhanced recall, DSL, embedding)** → V1.3+. Basic read-only snapshot is V1.2 scope (already delivered in V1.0/V1.1). | ADR-009; compass §3 V1.3 |
| R7 | **Tech-debt backfill (no user-visible behavior change)** → assign to the earliest version where the underlying code is actively worked on, unless explicitly deferred. Batch E items follow the V1.2 WS5 parallel lane and do not block the release gate. | Compass §2 (Batch E non-blocking) |
| R8 | **Items without a clear product-owner or ADR anchor** → Backlog until elevated by a new ADR or roadmap row. Do not guess version assignment. | pre-freeze §1; v1.md §14 |
| R9 | **`nexus` OSS-only items** → eligible for V1.2 if they meet the `local_only` verification gate. **`nexus-platform`-only items** → V1.3 minimum unless they are pure documentation/spec deliverables usable by the OSS repo. | Compass §7 cross-repo |
| R10 | **Already-delivered items** (status.json `Done`) → mark `Delivered` in current version; do not re-assign to a future version unless the capability has an explicit post-delivery extension ADR. | status.json ground truth |

## 7) Maintenance

- A plan row moves to `InProgress` or `Done` — update the **Status** column

- Compass §2 locked decisions change (rare; requires PM sign-off)
