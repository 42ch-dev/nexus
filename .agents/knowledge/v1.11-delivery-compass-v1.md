## 2) V1.11 定位

**V1.11 = 「Daemon Cleanup + Security Hardening」**

V1.10 完成了 Device Flow Login + DeviceID Header + Daemon Auth/Context Mock Removal。Platform V1.10 (PR #44) 也已 Done，所有 active plans 完成。

Platform V1.11 是纯 platform 端功能收口（Plans 77-82），对 nexus OSS 的唯一交叉需求是 CLI refresh_token 支持（Theme F），但这依赖 platform 先行交付。

nexus OSS V1.11 聚焦三项工作：

1. **Daemon Stub Cleanup** — V1.10 compass §6 WS-C 明确推迟到 V1.11 的 14 项 daemon 非运行时代码清理（死模块、stub 函数、placeholder、dead_code 注解、测试债务）。
2. **DeviceID TOCTOU Fix** — R-V110-004（warning，target V1.11），`get_or_create_device_id()` 文件权限 TOCTOU 竞争。
3. **CLI refresh_token Support** — Platform V1.11 Theme F 将在 `/auth/device/token` 响应中签发 `refresh_token`。CLI 需同步扩展 `AuthStore` 与 token refresh 逻辑，与 platform 并行开发。

**V1.11 一句话**：清理 daemon 中 V1.10 未触达的遗留 stub/placeholder/dead-code，修复 DeviceID 文件创建的 TOCTOU 安全问题，实现 CLI 侧 refresh_token 存储与自动刷新，为后续版本建立干净基线。

**不包含**：Backlog 残余批量治理（13 条 R-WA/R-WC/R-M1 系列）、DF-11 Handlebars 模板引擎。

## 3) Confirmed decisions (locked)

- **Daemon cleanup scope**: 仅清理 V1.10 compass WS-C 记录的 14 项明确条目。不扩展到未记录的代码气味。
- **TOCTOU fix approach**: 使用原子文件操作（write-then-rename）替代 check-then-create 模式。保持现有 `device_id_path()` API 不变。
- **No new dependencies**: V1.11 不引入新 crate 依赖。
- **CLI refresh_token**: 纳入 V1.11 核心范围（WS-C）。与 platform V1.11 Theme F 并行开发。CLI 侧扩展 `AuthStore` 存储 `refresh_token`，实现 token 自动刷新逻辑。
- **Branch strategy**: `feature/v1.11` from `main`.

### WS-B — DeviceID TOCTOU Fix (R-V110-004)

- **Goal**: 修复 `get_or_create_device_id()` 中的 TOCTOU 竞争条件。当前实现先 `std::fs::metadata(path)` 检查文件是否存在，然后 `std::fs::write(path, uuid)` 创建文件，存在检查与创建之间的窗口可被利用。
- **Design inputs**: R-V110-004 residual; `crates/nexus-sync/src/device_id.rs`。
- **Dependencies**: None（independent of WS-A）。
- **In scope**:
  - **Atomic file creation**: 使用 `std::fs::OpenOptions::new().write(true).create_new(true)` 替代 check-then-create。`create_new(true)` 保证原子性（文件不存在则创建，已存在则返回 `AlreadyExists`）。
  - **File permission**: 保持现有 0600 permissions（Unix）。
  - **Error handling**: `AlreadyExists` 时 fallback 到读取现有文件（与当前行为一致）。
  - **Cross-platform**: Windows 上 `create_new` 同样可用。
  - **Tests**: 新增 unit test 验证并发调用不会覆盖已有 device-id。
- **Out of scope**:
  - DeviceID rotation / revocation
  - Multi-device sync
  - DeviceID format validation changes
- **Evidence**:
  - Unit test: concurrent `get_or_create_device_id()` calls produce consistent result
  - Unit test: existing device-id file is preserved, not overwritten
  - `cargo test -p nexus-sync` green
  - `cargo test --workspace` green

### WS-C — CLI refresh_token Support

- **Goal**: 扩展 CLI 认证流程以支持 refresh_token。当 `access_token` 过期时自动使用 `refresh_token` 获取新 token，避免用户频繁重新授权。与 platform V1.11 Theme F（`/auth/device/token` 响应增加 `refresh_token`）并行开发。
- **Design inputs**: Platform V1.11 Theme F; `crates/nexus42/src/auth/` 模块（`auth_store.rs`, `user_auth.rs`）。
- **Dependencies**: Platform Theme F 交付 `/auth/device/token` `refresh_token` 字段。CLI 侧先实现本地存储与 refresh 逻辑，通过 serde default 忽略尚不存在的 `refresh_token` 字段，无需等待 platform 完成。
- **In scope**:
  - **AuthStore 扩展**: 新增 `refresh_token: Option<String>` 和 `refresh_expires_at: Option<DateTime<Utc>>` 字段到持久化存储（`~/.nexus42/auth.json`）。
  - **Polling loop 更新**: `user_auth.rs` 的 device code polling 成功后，保存 `refresh_token` 及其过期时间。
  - **Token refresh logic**: 在 CLI 发起任何需认证的请求前，检查 `access_token` 是否即将过期（buffer，如 < 60s）；若过期则使用 `refresh_token` 调用 `/auth/device/token`（grant_type=refresh_token）获取新 token pair。
  - **`nexus42 auth status` 更新**: 显示 refresh token 状态（存在/不存在/已过期）。
  - **Graceful degradation**: 无 `refresh_token` 时回退到当前行为（提示用户重新授权）。
- **Out of scope**:
  - Token rotation / revocation 策略
  - Multi-device session 管理
  - Offline-first auth 缓存
- **Evidence**:
  - Unit test: `AuthStore` 正确持久化/读取 `refresh_token`
  - Unit test: token refresh logic 在 token 即将过期时触发，在有效时跳过
  - Unit test: 无 `refresh_token` 时 graceful degradation
  - Integration test: `nexus42 auth status` 显示正确的 refresh token 状态
  - `cargo test -p nexus42` green
  - `cargo test --workspace` green

## 8) Cross-cutting policy

### Branch strategy

- Integration branch: `feature/v1.11` from `main`.
- WS-A and WS-B can proceed in parallel (daemon-only vs nexus-sync-only, no code overlap).

### CI requirements (mandatory before merge)

- `cargo +nightly fmt --all -- --check` — clean.
- `cargo clippy --all -- -D warnings` — clean.
- `cargo test --workspace` — green.

### Platform sync

- V1.11 platform work (Plans 77-82) is **platform-only** except Theme F which adds `refresh_token` to `/auth/device/token`.
- WS-C (CLI refresh_token) develops in parallel with platform Theme F. CLI uses serde default to ignore `refresh_token` field until platform delivers it.
- WS-A and WS-B have no platform dependency.
