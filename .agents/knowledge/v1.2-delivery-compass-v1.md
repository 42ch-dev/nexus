Status: Active (implementation-period SSOT)

## 1) Purpose and SSOT boundary

This document is the implementation-period SSOT for V1.2 planning decisions and execution guardrails, based on confirmed PM brainstorming outcomes.

- Authoritative for: V1.2 scope classification, workstream decomposition, milestone gates, risks, cross-repo dependencies, and handoff rules.
- Not authoritative for: field-level wire contracts and behavior details defined in v1-spec mother documents and ADRs.
- Conflict resolution:
  - If conflict with plan execution status, `status.json` in `{PLAN_DIR}` wins for machine state.
  - If conflict with architecture or contract semantics, v1-spec / ADR source files win.

## 2) Confirmed decisions (locked)

- Release gate policy: V1.2 uses "local closed-loop + minimum automated regression gate" (not demo-only).
- `local_only` identity strategy: dual path in V1.2
  - anonymous/local temporary identity
  - local persistent identity (without platform registration dependency)
- `local_first` / `cloud_enhanced` target in V1.2:
  - switchable, runnable, and degradable
  - no mandatory performance/accuracy threshold in V1.2
- `v1-tech-debt-cleanup` Batch E:
  - parallel track
  - non-blocking for V1.2 core release gate
- Planning strategy:
  - Option A accepted: reclassify legacy planning content into V1.2/V1.3/V1.4 first, then execute detailed V1.2 plan.

## 3) Version reclassification compass (V1.2~V1.4)

### V1.2 (CLI MVP, local-first product focus)

In:

- CLI/daemon local closed-loop delivery (`nexus42`, `nexus42d`).
- creator runtime modes acceptance on CLI side (`local_only`, `local_first`, `cloud_enhanced`) with V1.2 constraints.
- Local source-of-truth path:
  - local SOUL
  - local long-term memory
  - local KB load and output generation
- No hard dependency on platform-CLI e2e for release.

Out:

- platform+CLI e2e as release hard gate.
- user registration/creator binding full story.
- billing and explore community feature closure.

### V1.3 (platform + CLI enhancement)

In:

- platform optimization for local workflows (context assembly path, memory sync strategy, token/accuracy improvements).
- user registration + creator binding completion.
- platform/CLI e2e integration as acceptance gate.

### V1.4 (billing + community explore)

In:

- subscription and billing closure.
- explore community-facing mechanisms (ranking/cold-start/compliance matrix and related interactions).

### Backlog

In:

- items explicitly marked as non-blocking for V1.2~V1.4 program tracks.
- advanced or deferred architecture/product topics requiring dedicated future plan rows.

## 4) V1.2 workstreams (execution decomposition)

## WS1: Version reclassification baseline

- Goal: transform legacy roadmap buckets into V1.2/V1.3/V1.4/Backlog with explicit ownership and blocking relationships.
- Deliverables:
  - reclassification matrix
  - blocking/non-blocking map
  - version-tagging rules for future plan rows
- Evidence:
  - reviewed matrix committed in repository docs/design space
  - references to source ADR/spec anchors

## WS2: `local_only` MVP core path

- Goal: no-platform CLI end-to-end path with dual local identity strategy.
- Deliverables:
  - runtime mode behavior and constraints implementation plan
  - minimal runnable flow and fallback rules
- Evidence:
  - end-to-end CLI scenario runs without platform hard dependency
  - negative check confirms prohibited platform path is not required in `local_only`

## WS3: Local truth pipeline

- Goal: SOUL + long-term memory + KB + output generation as local source-of-truth chain.
- Deliverables:
  - lifecycle path implementation tasks
  - state transition and data ownership notes
- Evidence:
  - scenario regression: create -> accumulate -> retrieve -> output
  - deterministic behavior checks for core path

## WS4: Runtime compatibility (`local_first` / `cloud_enhanced`)

- Goal: switchable/runnable/degradable compatibility in V1.2 (without V1.3-level optimization gates).
- Deliverables:
  - mode switch rules
  - degradation behavior spec for platform unavailable states
- Evidence:
  - mode switch test cases
  - fallback path validation records

## WS5: Batch E parallel cleanup (non-blocking)

- Goal: align Batch E debt items to new version taxonomy and execute in parallel lane.
- Deliverables:
  - debt reclassification table by target version
  - residual tracking updates
- Evidence:
  - progress captured in `status.json` and corresponding plan artifacts
  - no release-gate coupling with WS2/WS3 critical path

## 5) Milestone gates (M1/M2/M3)

## M1: Reclassification + minimal local path runnable

- Entry:
  - scope lock confirmed
  - version reclassification baseline drafted
- Exit:

  - `local_only` dual identity minimal flow runnable

## M2: Local truth pipeline complete

- Entry:

## M3: Release pre-gate

## 6) Minimum automated regression set (V1.2 hard gate)

- R1: `local_only` anonymous/temporary identity end-to-end
- R2: `local_only` local persistent identity end-to-end
- R3: local truth core chain regression (SOUL/memory/KB/output)
- R4: mode switch and degradation regression (`local_first`/`cloud_enhanced` fallback behavior)
- R5: no-platform dependency guard for `local_only` path

## 7) Cross-repo dependency compass

## Primary execution repo (current): `nexus` OSS

- Core implementation and acceptance for V1.2 happen here.
- CLI/daemon/contracts/codegen/doc alignment tasks belong here unless explicitly split.

## Upstream/adjacent repo: `nexus-platform`

- Treated as optional support dependency for V1.2 (`local_first`/`cloud_enhanced` runtime support paths).
- Not a hard release blocker for V1.2.
- Moves to hard integration gate in V1.3.

## Dependency rule

- Any requirement that implies mandatory platform e2e must be tagged as V1.3+ and excluded from V1.2 release gate.

## 8) Risk compass

## RISK-01 Scope drift

- Symptom: V1.3/V1.4 requirements leak into V1.2 acceptance.
- Control: enforce In/Out matrix and milestone gate checks before each execution batch.

## RISK-02 Runtime semantic drift (`local_only`)

- Symptom: `local_only` path implicitly depends on platform capabilities.
- Control: keep explicit no-platform guard checks (R5) and fail-fast on prohibited dependencies.

## RISK-03 Validation drift

- Symptom: "works in demo" replaces reproducible automated evidence.
- Control: M3 gate requires full minimum regression set, not ad hoc manual confirmation.

## RISK-04 Planning inconsistency across documents

- Symptom: roadmap/pre-freeze/plan rows diverge in version mapping.
- Control: treat this compass as implementation-period SSOT and update mapping before new plan rows are created.

## RISK-05 Parallel-lane confusion (Batch E)

- Symptom: non-blocking debt lane accidentally blocks core V1.2 delivery.
- Control: maintain explicit lane separation and gate policy in plan metadata and status updates.
