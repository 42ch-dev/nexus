## 0. Position

V1.31 made embedded Agentic Design Pattern presets real enough to demonstrate orchestration composition. V1.32 turns that shipped baseline into a **quality gate**: a preset that validates through CLI/API must be close to what the orchestration loader will actually accept and what users can run from embedded assets.

The locked axis is:

> **Orchestration preset quality gate + `SEC-V131-01`.**

This is not a new product-line expansion. Runtime preset assets under `crates/nexus-orchestration/embedded-presets/` are compiled into the binary, but they are **not** the normative preset-format source of truth. Normative preset semantics remain in [orchestration-engine.md](../knowledge/specs/orchestration-engine.md) §7-§8.

When this compass conflicts with an active locked plan, the active plan wins for delivery batching.

## 2. Architecture

### 2.1 Validation Ownership

```text
POST /v1/local/presets:validate
optional future CLI wrapper (P1-owned, not shipped yet)
user/system embedded scan paths
        |
        v
Shared preset validation facade
        |
        v
nexus-orchestration loader contract
  - parse + size/depth limits
  - manifest schema checks
  - graph build checks
  - asset path sandbox checks
  - capability compatibility checks
```

The target shape is **one semantic validator** with different presentation layers, not three independent validators.

### 2.2 Completeness Checks

The quality gate should reject presets that are syntactically valid YAML but operationally incomplete:

- `initial_state` reaches at least one terminal state.
- Every terminal state marker agrees with any terminal header/metadata convention in the preset.
- Bundle directory id and manifest id are consistent for user/system bundle validation.
- Inner graphs referenced by states exist; inner graphs defined but unreachable are warned or rejected per P1 decision.
- Prompt/template/system prompt file references exist inside the bundle sandbox and cannot traverse outside it.
- Capability names and argument keys align with the registered capability contract enough to catch obvious drift before runtime.

### 2.3 Embedded Preset Usability

Embedded presets are runtime assets, so V1.32 treats them like fixtures for the validation gate:

- Smoke-load every embedded preset through the same loader path users rely on.
- Keep existing V1.31 demonstrators (`reflection-loop`, `memory-augmented`) valid under the stricter gate.
- Fix likely `kb-extract` inner graph wiring if stricter validation exposes the issue.
- Preserve `ConditionalNotYetSupported`; do not add routing semantics to make a preset pass.

### 2.4 Security Follow-up

`SEC-V131-01` is a narrow defense-in-depth fix. `judge.llm` and `context.summarize` should follow the same ID boundary as the fixed `acp.prompt` path: only context-injected `_creator_id` / `_session_id` are trusted for capability execution identity.
