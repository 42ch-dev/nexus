## 2. Architecture

### 2.1 KB Extract lifecycle hardening

Embedded preset [`kb-extract`](../../crates/nexus-orchestration/embedded-presets/kb-extract/preset.yaml) (V1.29) passes `preset.input.work_entry_id` / `world_id` / `creator_id` into `kb.extract_work`; it does **not** poll `kb_extract_jobs` yet. V1.30 **R16 (full e2e)** requires queue **claim** semantics: when `job_id` is omitted, orchestration or the preset entry path must **claim the next queued job** (via `claim_job` in `nexus-local-db`) before `mark_running` → `acp.prompt` → parse → KeyBlock insert → `mark_done` / `mark_failed`.

```text
kb_extract_jobs table
  ├── claim_job() → SELECT ... FOR UPDATE + UPDATE in transaction (R15)
  ├── job_id: UUIDv4 with xj_ prefix (R18)
  ├── list_jobs(limit) → bounded pagination (R20)
  └── kb.extract_work capability
        ├── claim or load job_id → mark_running → extract → mark_done/mark_failed (R16)
        └── KeyBlock insertion on success
```

### 2.2 Session quality

```text
agent session start
  └── SessionCapture::new()  ← create here, not at stop (R14)
        ├── record(MessageEvent) on each message
        ├── record(ToolEvent) on each tool call
        └── close() on agent_stop → persist metrics

persistent child process
  └── Drop handler (R17)
        ├── Unix: retain child handles, send SIGTERM + timeout SIGKILL
        └── Non-Unix: documented limitation
```

### 2.3 Creator module refactor

```text
creator/
  ├── mod.rs (slim: re-exports + top-level commands)
  ├── kb.rs  (R19: extract kb queue/status/list handlers)
  └── ...existing submodules
```
