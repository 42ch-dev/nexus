## 2) V1.13 positioning

V1.13 is an **OSS-forward release**: prioritize nexus-side deliverables first, then bind platform coordination only where required by integration gates.

### Core principles

1. **OSS-first value**: all primary themes must produce direct nexus artifacts.
2. **No scope inflation**: avoid introducing unrelated V2.0+ capabilities.
3. **Governance before implementation when ambiguity is structural**: `DF-15` is handled as a policy/lifecycle decision, not a feature implementation.
4. **Integration-gated execution**: `DF-14` proceeds with staged gates so progress is possible even when cross-repo dependencies are pending.

### Theme B — DF-14 CLI + Platform e2e integration

**Goal**: Establish stable cross-repo end-to-end integration contract for CLI↔Platform critical path.
**Target item**: `DF-14`
**Effort**: M
**Type**: Integration delivery

| Gate | Description | Exit condition |
|---|---|---|
| Gate B1 (nexus-local) | Define and validate nexus-side e2e harness assumptions | Reproducible local integration baseline from nexus side |
| Gate B2 (cross-repo) | Validate against platform-side counterpart flow | End-to-end happy path passes with agreed fixture/profile |
| Gate B3 (hardening) | Capture failure-mode assertions | Non-happy-path behavior documented and testable |

**Acceptance evidence**:
- E2E path is runnable and documented
- At least one cross-repo integration scenario verified
- Failure behavior is explicit (timeout/auth/config mismatch class)

### Theme C — DF-15 governance closure (Cancelled — no OpenAPI implementation scope)

**Goal**: Resolve historical `DF-15` ambiguity by formal lifecycle decision.
**Target item**: `DF-15`
**Effort**: XS
**Type**: Governance / tracker hygiene

**Decision basis**:
- `nexus` is not an OpenAPI-first product boundary for runtime value delivery.
- Existing historical references came from early cross-repo planning context and no longer represent the current architecture intent.

**Required outputs**:
