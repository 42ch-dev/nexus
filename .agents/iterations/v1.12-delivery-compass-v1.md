## 5) V1.12 workstreams

### WS-A — Preset Module Completion (Theme A)

- **Goal**: 完成 preset 模块的 4 项遗留 residual 修复 + 更新过时的 AGENTS.md 文档。用户 preset 功能核心已实现（`resolve_preset()` 支持 user preset），但 CLI 和编排在健壮性与 UX 上有多处遗漏。
- **Design inputs**: R-M1-W03/W04/W05/W06 residual descriptions; `status.json` residual_findings。
- **Dependencies**: None（preset-only changes）。
- **Verified facts**:
  - `resolve_preset()` in `nexus-orchestration/src/preset/mod.rs` (lines 109-113) **does** support user presets (`scan_user_presets`, `find_user_preset`).
  - `nexus-orchestration/AGENTS.md` lines 20-22 incorrectly states "User-Installed Presets (Not Yet Implemented)" and "load_preset() is a stub" — both false.
  - No `preset validate` subcommand exists yet (R-M1-W05 scope includes adding it).
- **In scope**:

#### Item 1: R-M1-W03 — Preset list embedded preset daemon dependency

- **Problem**: `preset list` requires daemon running to display embedded presets, but user/system presets work offline.
- **Fix**: Decouple embedded preset listing from daemon. Embedded presets should be resolved from compile-time/bundled data, not via daemon API.
- **Scope**: `crates/nexus42/src/commands/preset.rs`

#### Item 2: R-M1-W04 — YAML depth/size limits

- **Problem**: User-supplied preset YAML files have no depth or size limits, risking stack overflow or memory exhaustion on malicious inputs.
- **Fix**: Add configurable depth limit (default ~10) and file size limit (default ~1MB) in `loader.rs`. Reject files exceeding limits with clear error messages.
- **Scope**: `crates/nexus-orchestration/src/preset/loader.rs`

#### Item 3: R-M1-W05 — init/scan prefix inconsistency + add `preset validate`

- **Problem**: `preset init` rejects `_` prefix in preset names, but `preset list` (scan) silently skips `_`-prefixed entries — inconsistent UX. No `preset validate` subcommand exists.
- **Fix**:
  1. Align behavior: `init` should either accept `_` prefix or `scan` should warn on `_`-prefixed entries. Recommended: warn on scan, accept on init (since user chose the name).
  2. Add `preset validate` subcommand that checks a preset YAML for structural validity, required fields, and potential issues.
- **Scope**: `crates/nexus42/src/commands/preset.rs`, `crates/nexus-orchestration/src/preset/`

#### Item 4: R-M1-W06 — O(1) preset lookup

- **Problem**: `find_user_preset()` scans all user presets linearly. With large numbers of user presets, this becomes slow.
- **Fix**: Build a `HashMap<String, Preset>` index at scan time (O(n) once), then use O(1) lookup by preset ID. Cache the index and invalidate on file system changes (mtime check).
- **Scope**: `crates/nexus-orchestration/src/preset/mod.rs`

#### Item 5: DOC-FIX — Stale AGENTS.md

- **Problem**: `crates/nexus-orchestration/AGENTS.md` incorrectly states user-installed presets are "Not Yet Implemented" and that `load_preset()` is a stub.
- **Fix**: Update AGENTS.md to reflect actual implementation status. Document the preset resolution flow including user preset support.
- **Scope**: `crates/nexus-orchestration/AGENTS.md`

- **Out of scope**:
  - New preset features beyond residual scope
  - Preset sharing/distribution mechanism
- **Evidence**:
  - `cargo test -p nexus42` green
  - `cargo test -p nexus-orchestration` green
  - `preset list` works without daemon running
  - `preset validate` correctly reports invalid YAML
  - Large preset collections (100+) resolve in reasonable time
  - AGENTS.md accurately describes preset module

## 8) Cross-cutting policy

### Branch strategy

- Integration branch: `feature/v1.12` from `main`.
- All three workstreams can proceed in parallel (WS-A in preset/orchestration, WS-B across multiple crates, WS-C in contracts/daemon). Overlap is minimal but `fullstack-dev` should stage commits to avoid merge conflicts.

### CI requirements (mandatory before merge)

- `cargo +nightly fmt --all -- --check` — clean.
- `cargo clippy --all -- -D warnings` — clean.
- `cargo test --workspace` — green.

### Platform sync

- Platform V1.12 is **platform-only** — no nexus OSS dependency from any platform theme.
- No coordination needed with platform team for this iteration.

### Residual closure process

- Each closed residual: update `status.json` (remove from `residual_findings`), add to `archived/residuals/<plan-id>.json`.
- Update `deferred-features-cross-version-tracker.md` to reflect V1.12 closures.
- Target: `status.json` `total_open` = 0 at M1.
