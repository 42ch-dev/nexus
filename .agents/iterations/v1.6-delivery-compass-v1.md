- **V1.5 deferred item evaluation**: QC-noted but untracked items (~15) evaluated individually. Real issues deferred to appropriate version; non-issues marked ignored. See §3 Backlog.
- **Branch strategy**: `feature/v1.6` from `main`.

### V1.6 (Residual Governance + ACP SDK Preparation + Creator Tooling)

**In (this compass)**:

- **V1.5 residual R1** — cancel signal `pause_schedule()` error silently ignored. Scope: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs` (line 545 `let _ =`). Fix: log the error at `warn!` level and continue with the cancel operation. The pause failure must NOT block the cancel signal — the schedule enters terminal state regardless. Both the cancel-path pause (line 545) and the standalone pause signal path (line 482) need the same fix.
- **V1.5 residual R2** — `resume_schedule()` TOCTOU race on concurrent callers. Scope: `crates/nexus-orchestration/src/schedule/supervisor.rs` (lines 596-718). Fix: check `rows_affected()` after the `resume_schedule()` UPDATE. If 0, return the schedule's current status without updating cache. This is sufficient given SQLite's single-writer model — no additional serialization needed (holding `Mutex` across async DB ops is an anti-pattern).
- **V1.5 residual R3** — `Scheduler::tick()` duplicates DB query with `tick_clocked()` — dead code. Scope: `crates/nexus-orchestration/src/scheduler/mod.rs` (lines 93-133). Fix: remove the redundant query in `tick()`; simplify to delegate to `tick_clocked()`. No test dependencies on the dead code path.
- **V1.5 residual R6** — Recovered sessions lack FlowRunner instances after daemon restart (`NoGraphLoaded`). Scope: `crates/nexus-orchestration/src/engine.rs` (lines 262-285, `recover_sessions()` + `run_step_internal()`). Fix: after recovering sessions, iterate non-terminal summaries and reconstruct FlowRunner from: (1) `preset_id` from `SessionSummary`, (2) `CapabilityRegistry` from engine, (3) graph construction via embedded preset loader, (4) graph-flow Session data from `SqliteSessionStorage` (preserves execution position). Also verify V1.5 QC2 S2 (apply_llm_summarize `[u8;32]` vs hex string) as part of this investigation — document outcome.
- **ACP SDK DTO decoupling** — Nexus-owned request/response types for `NexusAcpClient` trait. SDK types confined to `AcpSdkAdapter`. Client handlers remain unchanged.
- **ACP permission policy CLI** — `nexus42 permission list/grant/deny/ask/revoke/reset` commands backed by existing `policy.rs` rule engine. `revoke` removes a specific rule for an agent+capability tuple (distinct from `reset` which wipes all rules for an agent).
- **System-managed multi-preset scheduler** — configurable `_system.*` preset directory replacing hardcoded `_system.maintenance`.

**Out (deferred to V1.7+)**:

- **ACP SDK migration to sacp v1.0** — full migration blocked pending upstream stable release (RFD approved, response message changes pending). V1.6 DTO decoupling is preparation only.
- **Multi-agent worker** (single worker hosting >1 ACP agent) — requires ACP session multiplexing design. See [orchestration-engine-v1.md](../archived/knowledge/orchestration-engine-v1.md) §11 OQ-7.
- **User-authored capabilities** (shell / WASM plugin ABI) — requires plugin ABI + sandboxing spec. V1.7+.
- **Preset third-party registry / signing / publish** — requires trust model + distribution protocol. V1.7+.
- **Full ACP permission policy engine UI** — web-based UI for policy management. V1.6 delivers CLI surface only; web UI is V1.7+.
- **Preset hot-reload with in-flight session migration** — V1.4 snapshot semantics already handle new sessions picking up changed presets. In-flight migration requires state migration protocol design. V1.7+.
- **Capability schema registry sharing with platform** — cross-repo catalog sync. Evaluate in V1.7.
- **Wire/local drift auto-detect tooling** — automated detection of schema classification drift. V1.7+ (per WS5 OQ-S3).
- **Template_file path validation** — filesystem preset path traversal protection. V1.7+ (when third-party preset authoring opens).
- **WS4 Starting lifecycle edge cases** — HealthDegraded during Starting, Starting.exit in-flight cancel, ActionContext not used. V1.7+ (daemon lifecycle hardening).
- **CoreContext Handlebars template engine binding** — data produced by V1.4 WS7, template rendering not yet integrated. V1.7+.

## 4) V1.6 workstreams (execution decomposition)

### WS-A — Residual governance (Track A)

- **Goal**: Resolve all 4 V1.5 residuals targeting V1.6 (R1, R2, R3, R6) to zero open medium-severity items. Verify V1.5 QC2 S2 (apply_llm_summarize type mismatch) as part of R6 investigation.
- **Dependencies**: none — can begin immediately.
- **Deliverables**:
  - **R1**: Log the `pause_schedule()` error at `warn!` level and continue with the cancel operation. The pause failure must NOT block the cancel signal. Fix both the cancel-path pause (line 545) and the standalone pause signal path (line 482). Scope: `nexus42d/.../schedules.rs`.
  - **R2**: Check `rows_affected()` after the `resume_schedule()` UPDATE. If 0, return the schedule's current status without updating cache. SQLite's single-writer model makes this sufficient — no additional serialization needed. Scope: `nexus-orchestration/.../supervisor.rs`.
  - **R3**: Remove dead `Scheduler::tick()` code path that duplicates `tick_clocked()` DB query. Simplify to delegate to `tick_clocked()` directly. No test dependencies on removed code. Scope: `nexus-orchestration/.../scheduler/mod.rs`.
  - **R6**: Fix `NoGraphLoaded` error on recovered sessions — reconstruct FlowRunner from: (1) `preset_id` from `SessionSummary`, (2) `CapabilityRegistry` from engine, (3) embedded preset loader for graph construction, (4) graph-flow Session data from `SqliteSessionStorage`. Verify V1.5 QC2 S2 (`apply_llm_summarize` `[u8;32]` vs hex string) as part of context investigation; document outcome. Scope: `nexus-orchestration/.../engine.rs`.
- **Evidence**:
  - Unit test: `pause_schedule` error path emits `warn!` log when pause fails during cancel; cancel still succeeds.
  - Unit test: concurrent `resume_schedule` calls produce correct serial outcome (no double-resume).
  - Dead code removal: `rg` confirms no remaining reference to removed query path.
  - Integration test: daemon restart → recovered session resumes successfully without `NoGraphLoaded`.
  - Verification: `apply_llm_summarize` hex string handling is correct (document outcome).
  - `cargo test --workspace` green; `cargo clippy --all -- -D warnings` clean.

### WS-B — ACP SDK DTO decoupling (Track B, exploratory)

- **Goal**: Create a Nexus-owned DTO layer for the `NexusAcpClient` trait so that SDK types are entirely internal to `AcpSdkAdapter`. This prepares for a future SDK migration (to sacp v1.0 or later) without requiring changes to consumers of `NexusAcpClient`.
- **Authoritative design**: [acp-client-tech-spec-v2.md](../archived/knowledge/acp-client-tech-spec-v2.md) §1 (adapter-trait policy), §11 (`nexus-acp-host` crate boundary). DTO types follow `schemas-boundary-v1.md` placement convention → `crates/nexus-contracts/src/local/acp/`.
- **Dependencies**: none — can begin immediately (parallel with WS-A).
- **In scope**:
  - Define Nexus-owned types for: `NexusSessionId` (wrapper around `String`), `NexusProtocolVersion` (wrapper around `String`), `NexusInitializeRequest`, `NexusInitializeResponse` (with fields: `protocol_version`, `agent_capabilities`, `agent_info: Option<NexusAgentInfo>`, `auth_methods: Vec<NexusAuthMethod>`), `NexusPromptRequest`, `NexusPromptCompleted`, `NexusStopReason`, `NexusNewSessionRequest`, `NexusSessionCreated` (with field: `modes: Option<NexusSessionModeState>`), `NexusCancelResult`, `NexusAgentCapabilities`, `NexusAgentInfo` (wraps SDK `Implementation`), `NexusAuthMethod`, `NexusSessionModeState`, `NexusContentBlock`. **Total: 16 types**. Use newtype wrappers for simple cases (e.g., `NexusProtocolVersion(String)`); deep structural mirrors only where consumers read nested fields. **Hand-written** (per `schemas-boundary-v1.md` §3.1: local types have no JSON Schema/codegen). Do NOT create JSON Schema files for these types.
  - **`subscribe()` design decision**: Move `subscribe()` off the `NexusAcpClient` trait into a direct method on `AcpSdkAdapter`, or wrap the return type in a `NexusStreamEvent` enum + `NexusEventStream` wrapper. Document the chosen approach. Rationale: `subscribe()` returns `async-broadcast::StreamReceiver` which is tightly coupled to the SDK's broadcast implementation; moving it off the trait keeps the DTO scope contained.
- **Out of scope**:
  - Client Handler rewrite (`SimpleClientHandler`, `PolicyAwareClientHandler`) — remains on SDK v0.10.4 trait.
  - Full SDK migration to sacp v1.0.
  - Changes to transport, registry, session_manager, or policy modules (no SDK type exposure).
- **Evidence**:
  - `cargo doc --package nexus-acp-host` shows public API with no SDK types in trait signatures. SDK types are allowed only in `AcpSdkAdapter` impl blocks and `from_sdk_response()` conversion methods.
  - Manual verification: the `pub trait NexusAcpClient` definition block (and its return types) contains zero references to `agent_client_protocol` or `acp::`.
  - Consumers (e.g., `nexus-orchestration`) compile without `agent-client-protocol` in their dependency tree.
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-C — ACP permission policy CLI surface (Track C)

- **Goal**: Expose the existing `policy.rs` rule engine through a `nexus42 permission` CLI command group, so creators can manage agent permissions without hand-editing `.nexus42/permissions.toml`.
- **Dependencies**: none — can begin immediately (parallel with WS-A/WS-B).
- **Deliverables**:
  - `nexus42 permission list [--agent <id>]` — list current grant/deny/ask rules, optionally filtered by agent.
  - `nexus42 permission grant <agent> <capability>` — add a grant rule.
  - `nexus42 permission deny <agent> <capability>` — add a deny rule.
  - `nexus42 permission ask <agent> <capability>` — add an ask (prompt) rule.
  - `nexus42 permission revoke <agent> <capability>` — remove a specific rule for an agent+capability tuple (distinct from `reset` which wipes all rules for an agent).
  - `nexus42 permission reset [--agent <id>]` — remove all rules for an agent (or all agents).
  - Implementation reads/writes `.nexus42/permissions.toml` via existing `policy.rs` infrastructure.
- **Evidence**:
  - `nexus42 permission grant test-agent terminal.create` succeeds; `permission list` shows the rule.
  - `nexus42 permission deny test-agent terminal.create` overwrites previous grant.
  - `nexus42 permission revoke test-agent terminal.create` removes the specific rule; `permission list` confirms it's gone.
  - `nexus42 permission reset --agent test-agent` removes all rules for `test-agent`.
  - `.nexus42/permissions.toml` reflects changes after each command.
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-D — System-managed multi-preset scheduler (Track D)

- **Goal**: Replace the hardcoded `_system.maintenance` preset registration with a configurable system preset directory (`~/.nexus42/presets/_system/`) scanned at engine startup, enabling multiple system presets without engine code changes.
- **Dependencies**: none — can begin immediately (parallel with WS-A/WS-B/WS-C).
- **Deliverables**:
  - System preset directory convention: `~/.nexus42/presets/_system/<name>/` (same bundle format as user presets, using `kind: capability` enter actions for direct capability invocation — not just ACP prompts).
  - Engine startup scans `_system.*` directories and registers all discovered system presets (with `_system.` prefix in their identifier).
  - Existing `_system.maintenance` preset extracted from hardcoded Rust (`system_preset.rs`) into a directory bundle at `~/.nexus42/presets/_system/maintenance/`. **First-start fallback**: on first startup with no `_system/` directory, auto-create `_system/maintenance/` from the embedded Rust graph definition to ensure backward compatibility.
  - Graceful handling: missing system preset directory = no system presets (not an error); corrupted system preset = log warning and skip.
  - `nexus42 system preset list` — show registered system presets (required, not optional).
- **Evidence**:
  - Engine startup with `_system/maintenance/` directory present: system preset registered and functional.
  - Engine startup with additional `_system/health-check/` directory: both presets registered.
  - Engine startup with no `_system/` directory: no system presets, no error.
  - `nexus42 schedule start _system.maintenance` works identically to pre-V1.6 behavior.
  - `cargo test --workspace` green; clippy/fmt clean.

## 7) Cross-cutting policy

### Branch strategy

- Integration branch: `feature/v1.6` from `main`.
- Per-WS topic branches when ≥2 concurrent writers.
- Merge to `main` only after M3 gate passes.

### CI requirements (mandatory before merge)

- `pnpm run validate-schemas` — clean.
- `pnpm run codegen` + `verify-codegen` — no diff on generated output (if schemas touched).
- `cargo +nightly fmt --all -- --check` — clean.
- `cargo clippy --all -- -D warnings` — clean.
- `SQLX_OFFLINE=true cargo sqlx prepare --workspace --all -- --all-targets --check` — passes.
- `cargo test --workspace` — green.
- CI linkage gate (no ACP SDK in daemon) — passes (inherited; still applies after WS-B).

### sqlx compile-time macro policy

- Inherited from V1.4/V1.5: all new queries use compile-time macros; runtime queries require `// SAFETY:` comment.
- After any query change: `cargo sqlx prepare --workspace --all -- --all-targets` and commit `.sqlx/` metadata.

### Platform sync cadence

- No platform schema changes expected in V1.6 (unless platform V1.6 planning introduces new nexus-side requirements).
- If WS-B DTO changes affect published contract types, bump `@42ch/nexus-contracts` and notify platform team.
