## 1. Objective and Constraints

### 1.1 Objective

Redesign the daemon local HTTP API so it becomes a clean, reusable local control plane for:

1. the `nexus42` CLI as a thin HTTP client,
2. a future local web UI client,
3. scripting or automation clients that can authenticate with a local API key.

The redesign changes three architectural assumptions from the current implementation:

- Local API authentication is **not** Platform OAuth. It is a daemon-local API key.
- Local HTTP routes should expose stable daemon subsystems, not protocol internals such as ACP.
- CLI-direct local operations should move behind daemon endpoints so CLI and web UI share one local API surface.

### 1.2 Constraints

- `nexus-daemon-runtime` remains a **library crate only** consumed by `nexus42`.
- The daemon remains a **local supervisor / client-side runtime**, not an ACP agent/server.
- JSON Schema generated contract types remain the preferred DTO source where shared contracts exist.
- Breaking changes are allowed before first release, but this spec still names migration steps so downstream clients can update predictably.
- Platform OAuth tokens remain valid for **remote platform communication**, but must not authenticate local daemon HTTP calls.
- Runtime liveness endpoints must stay reachable without auth so `daemon start`, `status`, supervisors, and probes can distinguish “not running” from “auth failed”.

### 1.3 Non-goals

- No local web UI implementation in this spec.
- No new Platform API design for world/explore routes in this spec.
- No new ACP protocol plumbing; ACP remains behind `nexus-agent-host` / `nexus-acp-host`.
- No final CLI UX freeze. This spec defines daemon API shape and migration direction.

## 2. New Auth Model

### 2.1 Decision

Use a single daemon-local API key:

```text
NEXUS42_DAEMON_API_KEY=<opaque high-entropy local secret>
```

Authenticated clients must send:

```http
X-API-Key: <opaque high-entropy local secret>
```

The daemon validates this header for every non-runtime endpoint.

### 2.2 Startup behavior

**Dual-mode startup**: keyless-localhost or keyed-all.

- On daemon startup, read `NEXUS42_DAEMON_API_KEY` from environment exactly once.
- Trim surrounding whitespace for validation only; store the exact configured secret bytes for comparison after validation.
- **If the variable is set (non-empty)**: all HTTP requests must present `X-API-Key` matching the configured value. No IP-based bypass.
- **If the variable is empty or unset**: daemon starts in **keyless-localhost mode**:
  - Requests from loopback addresses (`127.0.0.1`, `::1`, `localhost`) are accepted without `X-API-Key`.
  - Requests from **non-loopback addresses** are rejected with `403 FORBIDDEN` (not `401 AUTH_REQUIRED` — distinguish "local without key" from "not allowed").
  - Log a warning at startup: `"NEXUS42_DAEMON_API_KEY not set; daemon running in keyless-localhost mode. Non-local connections will be rejected."`
- Do not log the key or echo it in errors.
- Background daemon spawn inherits the parent environment.

Rationale: keyless-localhost mode enables quick local development without key setup (CLI just works), while ensuring remote access is always secured. The explicit warning makes the security posture transparent.

### 2.3 Middleware design

Replace the current Bearer-token middleware with API-key middleware.

Current middleware behavior:

- reads `Authorization: Bearer <token>`,
- validates the value against `auth_tokens` in SQLite,
- injects `AuthenticatedUser` and `BearerToken` extensions.

Target middleware behavior:

- reads `X-API-Key`,
- validates it against the startup-loaded daemon API key using constant-time comparison,
- injects a lightweight `AuthenticatedLocalClient` extension, for example:

```rust
pub struct AuthenticatedLocalClient {
    pub auth_scheme: LocalAuthScheme, // ApiKey
}
```

Do not inject platform user identity from local API auth. If a handler needs a platform user/creator token, it should obtain that through the platform-auth/token subsystem explicitly.

### 2.4 Relationship to `TokenManager`

`crates/nexus-daemon-runtime/src/auth/token_manager.rs` should stop being part of local API auth.

Keep or refactor it as a **platform token manager** only:

- stores OAuth access/refresh tokens for remote Platform calls,
- supports `auth login/refresh/logout` flows coordinated by CLI,
- supplies platform clients used by daemon handlers that call remote services.

It must not decide whether an HTTP request to `127.0.0.1:<port>/v1/local/*` is authorized.

### 2.5 Unguarded endpoints

Only runtime liveness/status endpoints stay unguarded (even in keyed-all mode, these are reachable without auth):

| Method | Path | Reason |
|---|---|---|
| `GET` | `/v1/local/runtime/health` | minimal liveness check for daemon start/status/supervisors |
| `GET` | `/v1/local/runtime/status` | runtime diagnostic status used by `daemon status` |
| `GET` | `/v1/local/daemon/status` | daemon lifecycle snapshot / HSM state |

All other current and future local API endpoints require authentication. The specific auth requirement depends on the startup mode:

- **Keyed-all mode** (key set): `X-API-Key` required.
- **Keyless-localhost mode** (key empty/unset): `X-API-Key` is accepted but not required for loopback connections; non-loopback connections are rejected with `403`.

Note: in keyless-localhost mode, the 3 runtime endpoints above remain accessible from any source (they are never behind auth middleware). All other endpoints are still behind the auth middleware, which in keyless-localhost mode simply passes loopback requests through.

### 2.6 Auth error contract

- Missing header: `401 AUTH_REQUIRED`
- Empty header: `401 AUTH_REQUIRED`
- Malformed key format, if key format validation is added: `400 INVALID_API_KEY`
- Wrong key: `401 AUTH_REQUIRED`

Do not distinguish “wrong key” from “missing key” in the public response message.

## 4. Agent-Host API Redesign

### 4.1 Design principle

Expose **agent-host** as the public local API abstraction. Do not expose ACP as a first-class local HTTP namespace.

Current split:

- `/v1/local/acp/tool/execute` handles daemon-mediated file tools.
- `/v1/local/acp/sessions` reads/writes the `acp_sessions` SQLite table.
- `/v1/local/agent-host/*` exists but is incomplete and Wave-1 placeholder-like.

Target split:

- `nexus-agent-host` owns provider/session/operation lifecycle.
- `nexus-acp-host` remains the ACP transport/provider implementation detail.
- `nexus-daemon-runtime` exposes local HTTP routes only through `HostFacade`-level concepts.

### 4.2 Provider abstraction constraints from `nexus-agent-host`

The host crate already defines the right abstraction boundary:

- `ProviderAdapter`: provider-specific adapter (`AcpProvider`, native CLI providers).
- `HostFacade`: daemon-facing facade (`start`, `create_session`, `exec`, `cancel`, `health`, `shutdown`).
- `HostOperation`: execution-scoped variants only: `Prompt`, `SetModel`, `SetMode`.
- Cancel and health are control-plane actions, not operations.
- Sessions are managed-only: host-owned, observable, cancellable, and lifecycle-controlled.

The daemon API should mirror that boundary.

### 4.3 Target agent-host endpoints

| Method | Path | Purpose | Notes |
|---|---|---|---|
| `GET` | `/v1/local/agent-host/health` | Host-level health | guarded; running, active sessions, active operations |
| `GET` | `/v1/local/agent-host/providers` | Provider catalog | include `provider_id`, `display_name`, `protocol_kind`, capabilities, health, source/trust when available |
| `GET` | `/v1/local/agent-host/providers/{provider_id}` | Provider detail | optional but useful for web UI diagnostics |
| `POST` | `/v1/local/agent-host/sessions` | Create managed session | request maps to `CreateSessionRequest` |
| `GET` | `/v1/local/agent-host/sessions` | List managed sessions | real session registry, with `limit` and `cursor` |
| `GET` | `/v1/local/agent-host/sessions/{session_id}` | Session detail | state, provider, created_at, active_op_id, negotiated capabilities |
| `DELETE` | `/v1/local/agent-host/sessions/{session_id}` | Shutdown one session | must not call `HostFacade::shutdown()` for whole host |
| `POST` | `/v1/local/agent-host/sessions/{session_id}/operations` | Execute a normalized host operation | request body tagged as `prompt`, `set_model`, or `set_mode` |
| `GET` | `/v1/local/agent-host/operations/{operation_id}` | Operation status | for polling clients if streaming is not used |
| `POST` | `/v1/local/agent-host/operations/{operation_id}:cancel` | Cancel operation | maps to `HostFacade::cancel(op_id)` |
| `GET` | `/v1/local/agent-host/sessions/{session_id}/events` | Operation/session event stream | SSE or newline-delimited JSON; exact streaming transport can be a later implementation decision |

### 4.4 ACP route absorption

Remove public ACP routes:

- `POST /v1/local/acp/tool/execute`
- `GET /v1/local/acp/sessions`
- `DELETE /v1/local/acp/sessions/{id}`

Replacement model:

1. ACP provider sessions are represented as agent-host sessions with `protocol_kind: "acp"`.
2. Native CLI provider sessions are represented the same way with `protocol_kind: "native_cli"`.
3. Tool calls become `HostEvent::ToolCall` / `ToolCallUpdate` events from providers.
4. Workspace file access must be exposed to providers through an internal host tool mediation service, not a public `/acp/*` route.

Recommended internal service boundary:

```rust
pub trait HostToolExecutor {
    async fn execute_tool(
        &self,
        session_id: HostSessionId,
        operation_id: HostOperationId,
        provider_id: ProviderId,
        tool_name: &str,
        parameters: serde_json::Value,
    ) -> Result<serde_json::Value, NexusApiError>;
}
```

This can initially reuse the validated logic from `api/handlers/acp.rs`:

- permission checks,
- workspace path validation,
- file read/write execution,
- audit logging.

But it should live under an agent-host/tool subsystem or shared service, not under an ACP HTTP handler.

If a subprocess absolutely needs an HTTP callback during transition, use an internal route under agent-host and mark it as implementation detail:

```text
POST /v1/local/agent-host/internal/tool-executions
```

That route should still be API-key guarded and should require session/op/provider metadata. It must not be named `/acp/*`.

### 4.5 Fix current agent-host handler gaps

Current handler gaps to fix during implementation:

- `GET /agent-host/providers` returns an empty placeholder. It should return actual discovery/catalog output.
- `GET /agent-host/sessions` returns count-based placeholder rows with empty IDs. It should expose real sessions.
- `DELETE /agent-host/sessions/{id}` validates UUID and then shuts down the entire host. It must shut down only the requested session or return `404 NOT_FOUND`.
- There is no operation execution endpoint. Add `POST /sessions/{id}/operations`.
- There is no cancel endpoint. Add `POST /operations/{id}:cancel`.

## 5. New Endpoints for CLI-to-Daemon Migration

### 5.1 Migration principle

CLI should become a thin client for local state and daemon-managed operations. CLI may still own:

- user-facing argument parsing and terminal rendering,
- auth login/refresh/logout interaction,
- agent-host coordination UX,
- preset execution details and session-level business logic that are inherently interactive,
- reading local files that are explicitly CLI input payloads before sending to daemon, when that improves UX.

Daemon should own:

- workspace creation/selection/listing,
- local creator identity/cache/active selection operations,
- preset catalog/scaffold/validation operations,
- local KB and memory data operations,
- orchestration sessions/schedules,
- platform-facing operations that are reusable by CLI and web UI once auth state exists.

### 5.2 Workspace endpoints

Move current CLI-direct workspace logic from `crates/nexus42/src/commands/init.rs` and `creator workspace` subcommands behind daemon endpoints.

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/v1/local/workspaces` | list operational workspaces, filter by `creator_id` |
| `POST` | `/v1/local/workspaces` | create/materialize ADR-014 workspace (`creator_id`, `workspace_slug`, `creative_root`, `display_name`) |
| `GET` | `/v1/local/workspaces/active` | return active creator/workspace selection and resolved paths |
| `PUT` | `/v1/local/workspaces/active` | set active workspace for active or supplied creator |
| `GET` | `/v1/local/workspaces/{creator_id}/{workspace_slug}` | inspect one workspace registration |
| `DELETE` | `/v1/local/workspaces/{creator_id}/{workspace_slug}` | optional: unregister local workspace only; should not delete creative files by default |

Implementation notes:

- Reuse `nexus-home-layout` and existing path validation helpers.
- Do not hardcode `~/.nexus42/` in new runtime code.
- Preserve skeleton-free workspace policy: do not create `Stories/` or `References/` during init.
- Keep `.nexus42/workspace.json`, `.nexus42/.gitignore`, operational `meta.json`, and state DB initialization as daemon-owned operations.

### 5.3 Creator endpoints

The current CLI handles local identity cache, active creator selection, creator registration, credential storage, and status rendering. Move reusable state-changing behavior behind daemon endpoints.

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/v1/local/creators` | list known local creators with active marker and identity cache metadata |
| `POST` | `/v1/local/creators/registrations` | initiate platform creator registration; returns challenge metadata |
| `POST` | `/v1/local/creators/registrations/{verification_code}:verify` | submit challenge answer; stores creator API key and local identity on success |
| `GET` | `/v1/local/creators/{creator_id}` | local creator status/detail |
| `PUT` | `/v1/local/creators/active` | set active creator by creator ID or handle |
| `GET` | `/v1/local/creators/active` | get active creator |
| `POST` | `/v1/local/creators/{creator_id}:logout` | clear local creator credentials and active selection if applicable |
| `POST` | `/v1/local/creators/{creator_id}/credentials:rotate` | rotate creator credentials through platform and update local store |

Registration split rationale:

- CLI or web UI can display/solve the challenge interactively.
- The daemon owns platform communication and local credential/cache writes.
- The same flow is reusable by web UI.

### 5.4 Preset endpoints

Move CLI-direct preset filesystem operations from `crates/nexus42/src/commands/preset.rs` behind daemon endpoints.

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/v1/local/presets` | list embedded/system/user presets, grouped by source; replaces or aliases orchestration preset list |
| `GET` | `/v1/local/presets/{preset_id}` | inspect preset manifest and source metadata |
| `POST` | `/v1/local/presets` | scaffold a user preset bundle (`name`, optional template kind) |
| `POST` | `/v1/local/presets:validate` | validate a preset YAML or bundle path |
| `POST` | `/v1/local/presets/{preset_id}:reload` | reload user/system preset |

Implementation notes:

- Prefer a single preset service used by both these endpoints and orchestration preset loading.
- Maintain existing validation limits for file size and YAML depth.
- Prevent path traversal and reserved names such as `_system` where applicable.

### 5.5 Local KB and memory endpoints

Current `creator kb` local operations are CLI-direct filesystem operations. For local web UI parity, expose them through daemon endpoints.

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/v1/local/kb/entries?creator_id=&workspace_slug=&q=&limit=&cursor=` | list/search local KB entries |
| `POST` | `/v1/local/kb/entries` | add entry from content or uploaded/local-file reference |
| `GET` | `/v1/local/kb/entries/{entry_id}` | show one KB entry |
| `DELETE` | `/v1/local/kb/entries/{entry_id}` | remove one KB entry |

This is not a blocker for the auth/ACP cleanup, but it is part of the CLI-thinning direction.

## 6. Error Response Format

### 6.1 Target error shape

Keep the current broad shape for compatibility, but extend it for future clients:

```json
{
  "success": false,
  "error": {
    "code": "INVALID_INPUT",
    "message": "Invalid input: workspace_slug must be a single path segment",
    "details": {
      "field": "workspace_slug",
      "reason": "must be a single path segment"
    },
    "request_id": "req_01h..."
  }
}
```

Fields:

- `success`: always `false` for errors.
- `error.code`: stable public `UPPER_SNAKE_CASE` code.
- `error.message`: human-readable message safe for CLI/UI display.
- `error.details`: optional structured data for UI field highlighting or automated handling.
- `error.request_id`: optional correlation ID when request tracing middleware is available.

### 6.2 Recommended public error codes

| Code | HTTP | Usage |
|---|---:|---|
| `AUTH_REQUIRED` | 401 | missing/wrong `X-API-Key` |
| `INVALID_API_KEY` | 400 | malformed key if format rules are added |
| `FORBIDDEN` | 403 | authenticated but blocked by policy/path rules |
| `UNINITIALIZED` | 409 | workspace/runtime subsystem not initialized |
| `INVALID_INPUT` | 400 | request validation failure |
| `NOT_FOUND` | 404 | resource not found |
| `CONFLICT` | 409 | duplicate workspace/preset/session conflict |
| `CAPABILITY_UNSUPPORTED` | 422 | agent-host provider lacks requested capability |
| `PROVIDER_UNAVAILABLE` | 503 | selected provider unavailable |
| `OPERATION_TIMEOUT` | 504 | provider/operation timed out |
| `INTERNAL` | 500 | unexpected daemon error |

### 6.3 Success response conventions

Do not require all success responses to be wrapped in `{ "success": true, ... }` in V1.20. Existing handlers currently return direct DTOs. However, new list endpoints should use stable collection envelopes:

```json
{
  "items": [],
  "pagination": {
    "limit": 50,
    "next_cursor": null
  }
}
```

For existing list responses with `total`, keep `total` until a broader contract migration is planned.

## 8. Breaking Changes and Migration Notes

### 8.1 Local API clients

All clients except health/status probes must add:

```http
X-API-Key: <NEXUS42_DAEMON_API_KEY>
```

Remove local API use of:

```http
Authorization: Bearer <platform_access_token>
```

### 8.2 CLI client changes

`DaemonClient` must learn the daemon API key:

- read `NEXUS42_DAEMON_API_KEY` from env or a local config/secret handoff chosen by implementation,
- attach `X-API-Key` to all methods except `health_check()` and unauthenticated runtime status if desired,
- update error messages so 401 suggests setting the daemon API key, not running `auth login`.

`health_check()` should remain unauthenticated and continue to return `Ok(false)` for connection failures.

### 8.3 Daemon router changes

Router construction should shift from “some small route groups are protected” to “everything protected except runtime routes”. Recommended shape:

1. Build `runtime_routes` without middleware.
2. Build all other routes into `protected_routes`.
3. Apply `require_api_key` to `protected_routes` once.
4. Merge `runtime_routes` and `protected_routes`.

This avoids route-by-route omissions as the API grows.

### 8.4 Platform OAuth migration

- `TokenManager::validate_token()` should no longer be called from local API middleware.
- OAuth access/refresh token storage remains available for platform-bound handlers.
- `auth_tokens` table can remain until platform auth storage is redesigned; do not delete it as part of the local API auth refactor unless a separate migration owns that change.

### 8.5 Removed routes

Consumers must stop calling:

```text
POST   /v1/local/world/fork
POST   /v1/local/world/snapshot
POST   /v1/local/explore/browse
POST   /v1/local/explore/search
POST   /v1/local/acp/tool/execute
GET    /v1/local/acp/sessions
DELETE /v1/local/acp/sessions/{id}
```

World/explore should be reintroduced later as Platform API routes or daemon platform-client operations only if there is a local caching/sync reason.

ACP callers must migrate to agent-host sessions/operations.

## 9. Risk Assessment

### 9.1 Security risks

| Risk | Impact | Mitigation |
|---|---|---|
| Daemon starts without API key | Local API becomes open | fail closed at startup; test-only injection only |
| API key leaked via logs | Local API compromise | never log header or configured key; redact `X-API-Key` in HTTP tracing |
| Route missed by auth middleware | Unauthorized local access | protect one `protected_routes` aggregate instead of individual routes |
| Tool execution moved incorrectly | Workspace path escape or missing audit | extract existing validation/audit into shared service before deleting handler |
| Web UI stores key insecurely | Local compromise via browser storage | future UI should use same-origin loopback model and avoid persistent browser storage where possible |

### 9.2 Compatibility risks

| Risk | Impact | Mitigation |
|---|---|---|
| Existing CLI calls fail with 401 | CLI unusable until migrated | update `DaemonClient` first or in same batch as router auth |
| World/explore callers break | Removed functionality | document as platform concern; no local fallback |
| ACP worker/tool calls break | Preset workspace writes fail | migrate internal tool execution to agent-host before removing old ACP route |
| Preset list route move breaks schedule UX | CLI preset commands fail | temporary alias from old orchestration preset routes if needed |

### 9.3 Reliability risks

| Risk | Impact | Mitigation |
|---|---|---|
| Background daemon env mismatch | child lacks API key | parent preflight plus inherited env; print actionable error |
| Per-session shutdown unavailable | deleting one session shuts down whole host | extend `HostFacade` or manager API for per-session shutdown before exposing endpoint semantics |
| Placeholder provider/session listing persists | web UI cannot show real state | treat real catalog/session listing as acceptance criterion |
| Streaming operation API underspecified | clients cannot consume long operations | start with polling status + stored event log, add SSE/NDJSON as follow-up if needed |

### 9.4 Rollback considerations

Because this is pre-release, rollback can be source-level rather than compatibility-layer-heavy:

1. Keep old `auth_middleware::require_auth` implementation in git history only; do not maintain dual Bearer/API-key mode unless implementation discovers an external dependency.
2. If agent-host migration is not ready, temporarily keep `/v1/local/acp/tool/execute` behind `X-API-Key` while marking it deprecated. Do not keep it unguarded.
3. World/explore removal is low rollback cost because these are platform proxy concerns; re-add only with an explicit platform-client design.
4. If CLI-to-daemon migration proves too large for one release, prioritize auth + route removals + agent-host cleanup first, then migrate workspace/creator/preset endpoints in follow-up batches.

## 11. Resolved Decisions

### 11.1 API key provisioning UX ✅

**Decision**: env-only with localhost bypass.

- `NEXUS42_DAEMON_API_KEY` is read from environment at startup.
- **If the variable is empty or unset**: the daemon starts in **keyless mode for local connections only** (requests from `127.0.0.1` / `::1` / `localhost` are accepted without `X-API-Key`).
- **If the variable is set**: all connections must present `X-API-Key` matching the configured value, regardless of source IP.
- **Non-local connections** (any IP other than loopback) **always** require a key; a missing or empty env variable causes non-local requests to be rejected with `403 FORBIDDEN` (not `401 AUTH_REQUIRED` — distinguish "you are local but need a key" from "you shouldn't be here").
- This enables quick local development without key setup, while ensuring remote access is secured.

### 11.2 CLI/web UI key sharing ✅

**Decision**: same-origin session bootstrap is the target model.

- If the daemon serves a local web UI in the future, the initial page load bootstraps the API key into a same-origin session (e.g., via a one-time bootstrap token injected into the HTML at serve time, exchanged for a session cookie).
- The web UI never asks the user to manually configure an API key.
- CLI reads `NEXUS42_DAEMON_API_KEY` from env and attaches `X-API-Key` header (same pattern as `DaemonClient` today).
- This is a design target, not a V1.20 implementation requirement. The API design (consistent `X-API-Key` auth) must not block this future pattern.

### 11.3 Creator registration ownership ✅

**Decision**: CLI/web UI solves challenges; daemon owns platform calls and credential writes.

- Challenge display and user interaction remain client-side (CLI terminal or web UI).
- Daemon exposes `POST /v1/local/creators/registrations` (initiate) and `POST /v1/local/creators/registrations/{code}:verify` (submit answer).
- Daemon performs all platform communication and persists local credentials/cache.
- Same flow is reusable by any HTTP client.

### 11.4 Operation streaming transport for agent-host ✅

**Decision**: provider-native streaming, unified relay via SSE.

The daemon exposes a single `HostEvent`-based SSE endpoint (`GET /v1/local/agent-host/sessions/{id}/events`) that relays provider-side events to HTTP clients. The transport between provider and daemon is provider-specific:

- **ACP provider**: ACP SDK's native streaming (JSON-RPC 2.0 over newline-delimited stdio). The `AcpProvider` adapter manages this internally via `ActiveSession::read_update()`. The daemon relay consumes `HostEventStream` and publishes as SSE.
- **Non-ACP provider (e.g., native CLI)**: provider-adapter-specific transport (subprocess stdio, named pipes, etc.). Each native adapter normalizes its output into `HostEvent` variants per the per-provider stream adapter pattern. For V1.20, the Claude CLI adapter normalizes stdout lines to `MessageDelta` and stderr to `Status` warnings.

Both paths converge on the same `HostEventStream` type, so the SSE relay is provider-agnostic.

**Architecture SSOT**: All provider streaming design, Multica/OpenDesign research, ACP event mapping, and per-provider adapter patterns are documented in [agent-host-architecture.md](agent-host-architecture.md) §4 (Provider Implementations), §8 (Streaming and Event Transport), §11 (Multica), §12 (OpenDesign).

### 11.5 Batch scope ✅

**Decision**: all V1.20 batches are in scope (Batch 1–5), no staging to V1.21.
