## 0) Scope lock (merged in compass)

### In scope (frozen)

1. **Command IA big-bang 重排**（目标命令面）：
   - `nexus42 daemon`
   - `nexus42 acp`
   - `nexus42 creator`
   - `nexus42 sync`
   - `nexus42 platform`
   - `nexus42 system`
2. **ACP-first 执行约束**：推理、工具调用、文件输出统一走 ACP capability invocation。
3. **Creator 资产归一**：`soul` / `memory` / `kb` 收敛到 `nexus42 creator ...` 子命令。
4. **`creator kb` multi-scope 基线**：本地默认 `work`，并预留 `world`/扩展 scope 的对接语义。
5. **Creator 三层标识兼容**：`creator_id`（权威）/`handle`（别名）/`display_name`（展示）契约对齐。

### Out of scope (frozen)

- platform OpenAPI 字段级改名或破坏性 wire 合同调整
- 计费/套餐策略调整
- Creator challenge 注册机制改造

## 1) Purpose and SSOT boundary

- **Authoritative for**: `nexus` V1.16 的命令面重排与执行边界收敛。
- **Not authoritative for**: platform 字段级 API 合同（仍由 platform `v1-spec` 管理）。
- **Conflict resolution**: `status.json` 是机器状态 SSOT；跨仓合同以 platform `v1-spec` 为准；本 compass 约束 `nexus` V1.16 范围。

---

## 2) V1.16 positioning

V1.16 的产品定位是：

1. **CLI 是控制面，不是执行器**。  
2. **ACP 是执行能力面**（保持顶层独立）。  
3. **Creator 是本地知识资产面**（soul/memory/kb 一体化）。

---

## 3) Workstreams

### Theme B1 — Command surface refactor

- 引入 `daemon orchestrate` 族并统一运行控制入口
- 保持 `acp` 顶层独立，不并入 `daemon`
- 收敛 `soul/memory/kb` 到 `creator` 子树

### Theme B2 — Execution path consolidation

- 清理非 ACP 执行路径
- 统一执行追踪（`run_id` / `capability_call_id`）

### Theme B3 — Creator knowledge model

- `creator kb` multi-scope 最小模型
- 本地 `work` 默认行为 + `world` 映射规则

### Theme B4 — Compatibility and verification

- Creator 三层标识契约回归
- Creator-context 鉴权路径回归（Creator key vs User + `X-Creator-Id`）

## 5) Done definition (minimum)

1. `creator use <creator_id_or_handle>` 落盘规范化为 `creator_id`。
2. `creator status/list` 稳定输出三层标识（`creator_id`/`handle`/`display_name`）。
3. `daemon orchestrate` 与 `acp` 无职责重叠。
4. 执行路径 ACP 覆盖率 100%（无绕过路径）。
5. `creator kb` 默认 `work` 可用，`world` 映射可用。
6. User-path / Creator-path 鉴权行为符合 `auth-session-model-v1`。
