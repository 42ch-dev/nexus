## 3) Confirmed decisions (locked)

- **WS-A вАФ Third-party preset loading**: Extend the existing `presets/` directory structure. Currently `system_preset_dir.rs` handles `_system/` prefixed presets from `~/.nexus42/presets/_system/<name>/`. Add support for user-installed presets at `~/.nexus42/presets/<name>/` (no `_` prefix = third-party). `load_preset()` from directory is currently a stub вАФ implement it. Add CLI command `nexus42 preset init <name>` to scaffold a preset template.
- **WS-B вАФ Template_file path validation**: Add traversal protection when presets reference `template_file` paths. Must work with both embedded presets and third-party presets (WS-A). Use the same defensive pattern as `creator_soul_md_path` in `nexus-home-layout`.
- **WS-C вАФ WS4 Starting lifecycle edge cases**: Address three identified gaps: (1) `HealthDegraded` during `Starting` вАФ currently `Starting` only handles `SubsystemFailed`; (2) `Starting.exit` in-flight cancel вАФ what happens when a shutdown request arrives while subsystems are still being collected; (3) `ActionContext` not used in Starting entry вАФ ensure diagnostic context is available.
- **WS-D вАФ Wire/local drift auto-detect**: Build a validation tool (probably a test or CLI check) that detects when wire schemas in `schemas/` diverge from local Rust type definitions in `crates/nexus-contracts/`. Start with the `context_summarize` schema validation pattern and generalize.
- **Branch strategy**: `feature/v1.9` from `main`.

## 5) V1.9 workstreams

### WS-A вАФ Third-party preset loading (DF-17)

- **Goal**: Enable users to install and load third-party presets from `~/.nexus42/presets/<name>/`, and scaffold new presets via CLI.
- **Design inputs**: Deferred tracker DF-17; `system_preset_dir.rs` (existing `_system/` scanning); `loader.rs` (stub `load_preset()` from directory).
- **Dependencies**: None вАФ independent of other WS.
- **In scope**:
  - Implement `load_preset()` from directory in `crates/nexus-orchestration/src/preset/loader.rs` (replace stub). Scan `preset.yaml` in `~/.nexus42/presets/<name>/`, parse + validate via existing `load_preset_from_str()`.
  - Create a new module `user_preset_dir.rs` (analogous to `system_preset_dir.rs`) that scans `~/.nexus42/presets/` for non-`_` prefixed directories.
  - Register user preset directory in the composable search order: embedded вЖТ `_system/` вЖТ user `presets/` (user overrides system overrides embedded).
  - Add CLI subcommand: `nexus42 preset init <name>` вАФ scaffolds a preset directory with `preset.yaml` template + empty `prompts/` directory.
  - Add `nexus42 preset list` to show all available presets with source indicator (embedded/system/user).
  - Update `nexus-home-layout` if needed to expose user preset base path.
- **Out of scope**:
  - Preset registry / signing / distribution (DF-03 вАФ long-term backlog).
  - Preset hot-reload (cancelled DF-06 вАФ snapshot semantics correct).
  - Preset dependency resolution or versioning.
  - Validation beyond structural correctness (delegated to WS-B for path safety).
- **Evidence**:
  - Unit test: preset from `~/.nexus42/presets/test-strategy/preset.yaml` loads and validates correctly.
  - Unit test: preset with invalid YAML produces `PresetLoadError`.
  - Unit test: `nexus42 preset init my-strategy` creates valid template directory.
  - Integration test: user preset overrides embedded preset of same name.
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-B вАФ Template_file path validation (DF-09) + name validation (R-WC-001)

- **Goal**: Prevent path traversal attacks in `template_file` references within presets, and add input length validation.
- **Design inputs**: Deferred tracker DF-09; `nexus-home-layout/lib.rs` (existing traversal checks in `creator_soul_md_path`).
- **Dependencies**: WS-A should be stable first (third-party presets are the primary attack surface).
- **In scope**:
  - Add path traversal validation in preset loading: when a `template_file` path is resolved, verify it stays within the preset's bundle root directory (embedded) or the preset's install directory (user/system).
  - Pattern: reject paths containing `..`, starting with `/` (absolute), or containing null bytes.
  - Apply the same validation in `load_preset_from_str()` (embedded) and the new `load_preset()` from directory (WS-A).
  - **(R-WC-001 merge)** Add max-length validation for `name` in `creator register` вАФ `name` max 64 chars, with descriptive error.
- **Out of scope**:
  - Full audit of all input validation in the codebase.
  - Filesystem permission hardening beyond preset loading.
  - DF-09's original "must work with `~/.nexus/strategies/`" scope вАФ resolved to `~/.nexus42/presets/` in WS-A.
- **Evidence**:
  - Unit test: `template_file: "../../etc/passwd"` produces validation error.
  - Unit test: `template_file: "/absolute/path"` produces validation error.
  - Unit test: legitimate relative `template_file: "prompts/system.md"` passes validation.
  - Unit test: `--name` with 65+ chars produces clear error.
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-C вАФ WS4 Starting lifecycle edge cases (DF-10)

- **Goal**: Handle three identified edge cases in the daemon lifecycle FSM's Starting state.
- **Design inputs**: Deferred tracker DF-10; `crates/nexus42d/src/lifecycle/` (existing statig HSM).
- **Dependencies**: None.
- **In scope**:
  1. **HealthDegraded during Starting**: Currently `Starting` handles `SubsystemUp` (stay/вЖТRunning) and `SubsystemFailed` (вЖТFailed/retry). Does NOT handle `HealthDegraded`. Add `HealthDegraded` вЖТ transition to `Degraded` if any subsystem reports degraded before Starting completes. This means the HSM must accept `HealthDegraded` in `Starting` state.
  2. **Starting.exit in-flight cancel**: When `ShutdownRequested` arrives during Starting (subsystems still booting), ensure clean teardown: pending subsystem futures are cancelled, exit_actions run properly, transition goes to `Stopping` (not `Failed`).
  3. **ActionContext in Starting entry**: The `enter_starting` action currently doesn't use `ActionContext`. Add lightweight diagnostic context (start reason, timestamp, initiating event) so that logs during startup carry contextual information.
- **Out of scope**:
  - Full FSM audit beyond Starting state.
  - Runtime performance optimization of state transitions.
  - Adding new states to the FSM.
- **Evidence**:
  - Unit test: `HealthDegraded` event during `Starting` moves to `Degraded` state.
  - Unit test: `ShutdownRequested` during `Starting` transitions to `Stopping` with clean cancel.
  - Unit test: `ActionContext` populated in `enter_starting` (verify via mock/exposed context).
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-D вАФ Wire/local drift auto-detect tooling (DF-08)

- **Goal**: Build a tool (test or CLI check) that detects when wire schemas (`schemas/`) and local Rust DTOs (`crates/nexus-contracts/src/local/`) drift apart.
- **Design inputs**: Deferred tracker DF-08; existing `context_summarize_*_schema_valid()` tests as pattern reference.
- **Dependencies**: None.
- **In scope**:
  - Design and implement a schema drift check: for each schema in `schemas/`, validate that generated Rust types in `nexus-contracts/src/local/` are consistent. This is NOT about runtime checking вАФ it's a compile-time / test-time gate.
  - Two approaches to evaluate (design decision during implementation):
    - (a) **Structural comparison**: Parse wire schemas and local type definitions, compare field-by-field for drift.
    - (b) **Round-trip serialization test**: Serialize a sample value through wire schema вЖТ JSON вЖТ deserialize to local type вЖТ verify field preservation.
  - Add as a CI gate: `cargo test --test drift_detection` or similar.
  - Document the drift detection pattern so new schemas are included.
- **Out of scope**:
  - Runtime drift detection (production monitoring).
  - Auto-repair of drift.
  - Platform-side schema drift detection.
- **Evidence**:
  - CI test: known-matched schemas pass.
  - CI test: deliberately-drifted schema fails with clear error message.
  - `cargo test --workspace` green; clippy/fmt clean.

## 8) Cross-cutting policy

### Branch strategy

- Integration branch: `feature/v1.9` from `main`.
- WS-A and WS-C can proceed in parallel (independent modules: orchestration vs daemon lifecycle).
- WS-B depends on WS-A (needs stable preset loading surface).
- WS-D is independent вАФ can be done in parallel with WS-A/WS-C.

### CI requirements (mandatory before merge)

- `cargo +nightly fmt --all -- --check` вАФ clean.
- `cargo clippy --all -- -D warnings` вАФ clean.
- `cargo test --workspace` вАФ green.

### Platform sync

- V1.9 has **no expected platform dependency**. All workstreams are OSS-only.
- If WS-A's `nexus42 preset init` generates presets that reference platform APIs, document but don't block on platform availability.
