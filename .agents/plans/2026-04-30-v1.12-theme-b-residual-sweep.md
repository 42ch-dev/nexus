# V1.12 Theme B — V1.11 + Warning Residual Sweep

## Summary

Close the eight V1.11 and warning-class residuals assigned to V1.12 Theme B. This is a surgical quality sweep across device ID retry behavior, auth refresh handling, CLI/API error hygiene, sqlx-only field naming, challenge MSRV/telemetry notes, creator positional-argument migration UX, and deterministic test output.

## Background

- **V1.12 delivery compass**: `.agents/iterations/v1.12-delivery-compass-v1.md`
- **Theme B scope**: 4 V1.11 residuals plus 4 warning-class residuals.
- **Residual target**: R-V111-S01, R-V111-S02, R-V111-S03, R-V111-S04, R-WA-001, R-WA-002, R-WC-002, R-WC-003.
- **Execution intent**: Keep changes small and local; do not expand into new telemetry infrastructure or breaking CLI redesign.

## Design Decisions

1. **Retry loop boundedness**: Device ID retry behavior must have a small explicit max count and return a clear error after exhaustion.
2. **Auth refresh efficiency**: `refresh_access_token()` should load AuthStore once and reuse the loaded value instead of reloading multiple times.
3. **API error hygiene**: Non-`invalid_grant` 400 responses should not disclose upstream error bodies to CLI output.
4. **sqlx-only fields**: Prefer removing misleading `_` prefixes where possible; if suppression is required, it must include a justification comment.
5. **MSRV/telemetry warnings are closure items**: Either resolve with code or document why the current state is acceptable and how future telemetry infrastructure will handle metrics.
6. **Deterministic tests**: Replace nondeterministic `HashMap` ordering in the relevant test with deterministic ordering such as `BTreeMap`.

## Out of Scope

- New telemetry backend or metrics pipeline.
- Breaking CLI interface changes beyond documented migration/deprecation behavior.
- Broad auth subsystem refactors beyond `refresh_access_token()` and associated error handling.
- Unrelated clippy cleanup outside the residual scopes.
