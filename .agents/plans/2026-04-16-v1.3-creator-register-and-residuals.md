## 1. Version Positioning

V1.3 is the next nexus release. It delivers two workstreams on a single feature branch:

1. **Creator Register CLI (M20–M23)**: `nexus42 creator register` — full challenge-verify flow with a pure-logic challenge-solver and optional LLM fallback. Implements the platform v1.3-spec §4.3–4.5 and §9 endpoints from the nexus CLI side.
2. **V1.2 Residual Governance (35 items)**: Drive all open V1.2 residual findings to zero — fix code correctness issues, document accepted design decisions, and close/archive every entry in `status.json`.

Non-goals: Entitlements CLI, CLI+Platform e2e, OpenAPI export, Stripe integration (see compass §2 locked decisions).

---

## 2. Scope

### In

- `nexus42 creator register` command — full challenge-verify orchestration
- `challenge-solver` library (noise removal, number-word mapping, arithmetic evaluation)
- `challenge-solver-skill.md` — LLM fallback prompt for non-standard challenges
- Platform API client extension (`PlatformClient` module in `nexus-sync`) — `/creators/register` + `/creators/verify`
- `creator_api_key` local storage (daemon-first with CLI file fallback)
- Integration tests (mock API + unit tests for solver)
- All 35 V1.2 residual items governance (4 medium / 17 low / 14 nit per compass)

### Out

- Entitlements API consumption (`/me/entitlements`, `/official-creator/quota`)
- CLI+Platform end-to-end integration testing
- OpenAPI export work
- Stripe / billing integration
- Social / marketing features (V2.0+)
- Platform-side V1.3 work (auth loop, credential chain, entitlements read API, BFF audit)

## 6. Acceptance Criteria

### Creator Register CLI

- [ ] `nexus42 creator register --name "My Agent"` completes full flow (register → solve → verify → store key)
- [ ] Challenge-solver correctly parses standard obfuscation challenges (noise removal + number words + arithmetic)
- [ ] LLM fallback handles non-standard challenges when pure logic pipeline fails
- [ ] LLM fallback graceful degradation: when LLM is unavailable and pure pipeline fails, CLI returns a clear error message (not panic or hang)
- [ ] CLI checks `expires_at` before solving; expired challenges report actionable message
- [ ] After wrong verification answer, CLI auto-retries at most once (D4); then displays error and stops
- [ ] `creator_api_key` stored locally after successful verification
- [ ] Integration tests pass with mock platform API
- [ ] PlatformClient module cleanly separates creator registration from sync concerns

### Residual Governance

- [ ] All 35 V1.2 residuals closed (0 open in `status.json`)
- [ ] All closed residuals archived in `.agents/plans/archived/residuals/`
- [ ] `tech_debt_summary.total_open` reflects 0 (or updated count excluding V1.2 items)
- [ ] Fix-class items have corresponding code changes + tests
- [ ] Accept-class items have code comments or knowledge doc notes documenting rationale

### CI Gate

## 9. Implementation Notes

### Challenge-solver module placement

Per challenge-solver-design-v1, the module lives at `crates/nexus42/src/challenge/` for V1.3 (CLI-only). If shared with daemon in the future, migrate to `crates/nexus-domain/src/challenge/`.

### PlatformClient design decision

Per challenge-solver-design-v1 §5, **Option B** (new `PlatformClient` module in `nexus-sync`) is recommended. Creator registration is semantically distinct from sync operations. The new module shares `reqwest` client and base URL config from existing `SyncClient`.

### Credential storage decision

Per challenge-solver-design-v1 §4.1: daemon-first storage (extend `auth_tokens` or new `creator_credentials` table) with CLI file fallback (`~/.nexus42/auth.json`). Check daemon health before choosing path.

### Residual close process

1. **Fix**: code change + test → close in `status.json` → archive to `archived/residuals/`
2. **Accept**: code comment or knowledge doc note → close → archive
3. **Superseded**: close with `closure_note` referencing the surviving residual → archive
4. All items must reach lifecycle `resolved`/`waived`/`superseded` with `closed_at` + `closure_note`

### PM review integration

Product review approved scope with 2 preconditions (resolved):
1. **D1-D4 closed** — all open design decisions in challenge-solver-design-v1.md formalized to "Decided" status
2. **Wave 3 deduplication** — W-003 (slug collision) superseded by pipeline R10; S-006 (experience dup) superseded by pipeline R4. Both close as `superseded` before Wave 3 implementation starts
3. **Acceptance criteria updated** — added 3 new criteria: LLM graceful degradation, challenge expiry handling, wrong-answer retry limit (aligned with D4)
4. **Recommendation not adopted**: parallel Wave 2 with Wave 1 — single-track serial execution is simpler for this plan scope
