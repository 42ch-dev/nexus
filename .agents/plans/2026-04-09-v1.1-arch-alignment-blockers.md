# Plan: V1.1 Architecture Alignment Blockers

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

### Batch 2: JSON Schema Enum Typing (High)

**Priority:** P1 — No compile-time enforcement of enum values in wire contracts; 28+ fields typed as String.

**Estimated Effort:** M (~1-3 agent sessions)

#### Task 2.1: Add proper enum types to JSON Schema and regenerate contracts (TD-4)

**ID**: TD-4  
**Severity**: High  
**Source**: architecture-alignment-review-v1.md §2.2 (Enum Alignment table), §2.6

**Description**: The generated contract types use `String` for 28+ enum fields, while the domain layer has proper Rust enums with correct variants. Only 6 enums (`BlockType`, `MemoryType`, `ManuscriptPhase`, `TimePolicy`, `Visibility`, `BundleType`) are properly typed in contracts. The remaining fields (all `status`, `role`, `type` fields) need JSON Schema enum definitions so `typify` generates proper enum types in both Rust and TypeScript contracts.

**Scope**: This is the largest single task. It requires updating all schema files that lack enum definitions, then running `pnpm run codegen` to regenerate both Rust and TypeScript contracts.

**Files**:
- Modify: `schemas/*.json` — add `"enum"` arrays to fields that currently have only `"type": "string"`
  - Priority schemas (currently String-typed but domain has correct enums):
    - `creator.json`: `status` (`active`, `archived`, `locked`), `registration_source` (`cli`, `web_agent`, `platform`)
    - `pairing.json`: `pairing_source` (`auto_cli`, `manual_web`, `platform_auto`), `status` (`active`, `revoked`)
    - `world.json`: `status` (`active`, `paused`, `archived`) — ensure aligned with TD-2 fix
    - `world_membership.json`: `role` (`owner`, `maintainer`, `collaborator`, `official_creator`), `membership_status` (`active`, `invited`, `suspended`, `removed`) — ensure aligned with TD-3 fix
    - `key_block.json`: `status` (`provisional`, `confirmed`, `deprecated`, `merged`, `deleted`)
    - `timeline_event.json`: `event_type` (`story_advance`, `state_update`, `fork_marker`, `official_progression`, `publish_marker`), `status` (`canon`, `provisional`, `rejected`)
    - `fork_branch.json`: `status` (`active`, `archived`), `verification_status` (`unverified`, `requested`, `verified`, `rejected`)
    - `memory_item.json`: `memory_kind` (8 variants per ADR-001), `status` (`active`, `superseded`, `archived`)
    - `story_manifest.json`: `manifest_type` (`chapter`, `arc`, `story`, `excerpt`), `status` (`summary_ready`, `staged_for_publish`, `published`, `archived`), `manuscript_storage` (`none`, `local_workspace`, `platform_sandbox`)
    - `reference_source.json`: `source_type` (`file`, `pdf`, `url`, `note`), `scan_status` (`pending`, `scanned`, `failed`, `ignored`)
    - `sync_command.json`: `command_type`, `origin`, `status`
    - `delta.json`: `delta_type`, `operation`
    - `outbox_entry.json`: `delivery_state`
    - `workspace_binding.json`: `binding_status`
    - `agent_profile.json`: `profile_kind`, `selection_mode`, `transport`, `status`
- Regenerate: `packages/nexus-contracts/src/generated/` and `crates/nexus-contracts/src/generated/`
- Modify (if needed): `crates/nexus-domain/src/*.rs` — update domain types to use generated enums where they now match

**Acceptance Criteria**:
- [ ] All JSON Schema files in `schemas/` updated with `"enum"` arrays for previously-String-typed enum fields
- [ ] `pnpm run codegen` produces new generated types without errors
- [ ] `pnpm run validate-schemas` passes (all schemas valid)
- [ ] `git diff --exit-code packages/nexus-contracts/src/generated/ crates/nexus-contracts/src/generated/` shows zero drift (generated output committed)
- [ ] Generated Rust contracts have proper enum types (not `String`) for all 28+ previously-String fields
- [ ] Generated TypeScript contracts have union/enum types for the same fields
- [ ] `cargo test --all` passes (domain crate compiles and tests pass with new contract types)
- [ ] `cargo clippy --all -- -D warnings` passes
- [ ] Domain types still match contract enums (no `From` impl conflicts between domain enums and newly-generated contract enums)

**Effort (agent-oriented)**: M — ~1-3 agent sessions. Schema updates are systematic but require running codegen and verifying both Rust and TS outputs. Risk: domain types and generated contract types may need adaptation (e.g., `From` impl overlap).

**Dependency**: Should be done AFTER Batch 1 (TD-2/TD-3) to ensure enum values in WorldStatus and MembershipStatus schemas are already corrected before codegen.

### Batch 3: Sync Wiring — Core Functionality (Critical)

**Priority:** P0 — Users cannot actually push/pull structured deltas to/from the platform.

**Estimated Effort:** L (~3-5 agent sessions)

#### Task 3.1: Wire nexus-sync into CLI sync flow (TD-1)

**ID**: TD-1  
**Severity**: Critical  
**Source**: architecture-alignment-review-v1.md §2.5, §2.6

**Description**: The `nexus-sync` crate has comprehensive outbox, precheck, partial apply, and conflict resolution logic, but CLI `sync push/pull` commands call the daemon's HTTP API, which returns skeleton/stub data (501 or hardcoded). The `nexus-sync::SyncClient`, `Outbox`, and `Precheck` modules are not invoked in the actual sync flow. This task wires the existing `nexus-sync` functionality into the CLI→daemon→platform pipeline.

**Files**:
- Modify: `crates/nexus42/src/commands/sync.rs` — replace daemon HTTP stubs with actual sync orchestration
- Modify: `crates/nexus42d/src/api/handlers/sync.rs` — replace 501 stubs with handlers that invoke `nexus-sync::Outbox`, `Precheck`, `SyncClient`
- Modify: `crates/nexus42d/src/daemon.rs` or workspace state — inject `nexus-sync` components into daemon state
- Modify: `crates/nexus-sync/src/lib.rs` — ensure public API is clean for daemon consumption
- Potentially: `crates/nexus42d/Cargo.toml` — add `nexus-sync` dependency
- Potentially: integration test files for sync flow

**Acceptance Criteria**:
- [x] `nexus42 sync push` invokes `nexus-sync::BundleBuilder` → `Outbox` → `Precheck` → `SyncClient::push` flow
- [x] `nexus42 sync status` returns real outbox data (staged/ready/sent counts) instead of hardcoded stub
- [x] `nexus42 sync resolve` wires through conflict resolution (retry with strategy)
- [x] Daemon sync handlers return real data (not 501)
- [x] Outbox SQLite schema matches `nexus-sync::outbox` (not dual implementation — see TD-8)
- [x] `cargo test --all` passes (including any new integration tests)
- [x] `cargo clippy --all -- -D warnings` passes
- [ ] Manual end-to-end test: `nexus42 sync push` reaches platform API (or proper error if platform unavailable)

**Effort (agent-oriented)**: L — 3-5 agent sessions. Requires understanding the full sync pipeline (Outbox → Precheck → SyncClient → Platform API), integrating it into daemon state management, and ensuring daemon handlers properly orchestrate the flow. The nexus-sync crate is already well-implemented; main work is wiring, not building.

**Dependency**: Independent of Batches 1-2. Can start in parallel.

**Status**: ✅ Completed (c5f27d8) — CLI sync commands now call real Outbox → Precheck → SyncClient flow

### Batch 4: Domain Models — Canonical Hash & User Aggregate (High)

**Priority:** P1 — Required for idempotency (TD-5) and user identity in sync flows (TD-6).

**Estimated Effort:** S+M (~2-3 agent sessions total)

#### Task 4.1: Implement canonical_hash computation (TD-5)

**ID**: TD-5  
**Severity**: High  
**Source**: architecture-alignment-review-v1.md §2.5, §2.6

**Description**: `BundleBuilder::build()` currently generates an empty string for `canonical_hash`. The sync-contract spec requires idempotency via a content-based hash. Without it, duplicate detection won't work and retry safety is compromised. Implement SHA-256 hash computation over the delta payload.

**Files**:
- Modify: `crates/nexus-sync/src/delta_bundle.rs` — replace `canonical_hash: String::new()` with actual SHA-256 computation
- Potentially add: `crates/nexus-sync/src/canonical_hash.rs` (new module for hash computation)
- Modify: `crates/nexus-sync/Cargo.toml` — add `sha2` or `sha256` crate dependency
- Add: test file(s) for canonical hash verification

**Acceptance Criteria**:
- [ ] `canonical_hash` is computed as SHA-256 of the serialized delta payload (not empty string)
- [ ] Hash computation is deterministic: same delta payload → same hash
- [ ] Hash format matches spec: `sha256:<hex>` or as documented in sync-contract
- [ ] `idempotency_key` and `canonical_hash` are both set in built bundles
- [ ] Unit tests for: identical payloads produce identical hashes, different payloads produce different hashes
- [ ] Integration: `BundleBuilder::build()` no longer produces empty `canonical_hash`
- [ ] `cargo test --all` passes
- [ ] `cargo clippy --all -- -D warnings` passes

**Effort (agent-oriented)**: S — 1 agent session. Add sha2 dependency, compute SHA-256 over delta payload, replace placeholder.

#### Task 4.2: Add User aggregate to domain and contracts (TD-6)

The spec §5.1 defines User with `account_status` (`active`, `suspended`, `deleted`) and `subscription_tier` (`free`, `pro`, `studio`, `enterprise`). This task adds the User aggregate to both schemas and domain.

**Files**:
- Add: `schemas/user.json` — JSON Schema for User aggregate (fields per data-model-v1.md §5.1)
- Regenerate: `packages/nexus-contracts/src/generated/` and `crates/nexus-contracts/src/generated/` (via `pnpm run codegen`)
- Add: `crates/nexus-domain/src/user.rs` — User aggregate with domain logic and invariants
- Modify: `crates/nexus-domain/src/lib.rs` — register `user` module
- Add: test file(s) for User aggregate

**Acceptance Criteria**:
- [ ] `schemas/user.json` created with fields per spec §5.1: `id`, `username`, `email`, `display_name`, `account_status` (enum: `active`, `suspended`, `deleted`), `subscription_tier` (enum: `free`, `pro`, `studio`, `enterprise`), `created_at`, `updated_at`
- [ ] JSON Schema enum fields (`account_status`, `subscription_tier`) use proper `"enum"` arrays (not `"type": "string"`) — aligned with TD-4 convention
- [ ] `pnpm run codegen` generates valid Rust and TypeScript types for User
- [ ] `pnpm run validate-schemas` passes
- [ ] Generated contracts committed (zero drift after codegen)
- [ ] `crates/nexus-domain/src/user.rs` implements User aggregate with:
  - Domain invariants (e.g., `account_status` transitions, `subscription_tier` validation)
  - `UserStatus` enum matching spec values
  - `SubscriptionTier` enum matching spec values
- [ ] `crates/nexus-domain/src/lib.rs` exports `user` module
- [ ] `cargo test --all` passes
- [ ] `cargo clippy --all -- -D warnings` passes

**Effort (agent-oriented)**: M — 1-2 agent sessions. Schema creation is straightforward; domain aggregate implementation requires designing invariants and transitions. Codegen step needed.

**Dependency**: Should be coordinated with TD-4 (enum typing) — if TD-4 is done first, User schema already benefits from proper enum types. If done in parallel, ensure User schema uses enum arrays from the start.
