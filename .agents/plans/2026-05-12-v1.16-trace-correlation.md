# Plan: V1.16 Run and Capability-Call Trace Correlation

**Plan ID**: `2026-05-12-v1.16-trace-correlation`
**Theme**: B2 — Execution path consolidation
**Milestone**: M3
**Effort**: M
**Depends on**: `2026-05-12-v1.16-acp-execution-consolidation`
**Created**: 2026-05-12
**Status**: Todo

## Purpose

Seed a lightweight execution trace/correlation model for the ACP-first execution path introduced by Plan 6.

V1.16 already makes `nexus42 acp run` the primary user-facing execution entry, but there is no stable way to answer:

1. Which CLI/daemon run did this subprocess, orchestration session, or worker request belong to?
2. Which capability invocation produced this worker request or prompt output?
3. Where should support/debug tooling look for minimal run metadata?

This plan adds **run-level** and **capability-call-level** identifiers plus minimal metadata persistence. It intentionally avoids a full tracing system; the goal is a pragmatic pre-1.0 correlation seed that later plans can expand.

## Current State Summary

- `crates/nexus42/src/commands/acp.rs`
  - `acp run` is present and is the primary ACP-first execution command.
  - `cmd_run(agent_ref, message, cwd)` resolves an ACP registry entry, uses `AgentSpawner`, spawns the agent subprocess, and then handles either single-shot wait or an interactive local prompt loop.
  - The interactive loop still notes that full ACP prompt integration is pending; messages are not yet sent through `AcpSdkAdapter`.
  - There are no `run_id` flags, no trace file writes, and no subprocess environment propagation.
- `crates/nexus42/src/commands/daemon.rs`
  - Orchestration control is owned by `daemon schedule ...`; the older `daemon orchestrate ...` surface is not a current command surface.
  - No run/correlation argument exists on the schedule orchestration boundary.
- `crates/nexus-acp-host/src/`
  - `transport.rs` owns `AgentSpawner` / `AcpSession` subprocess lifecycle.
  - `client.rs` owns the SDK adapter and logs `agent_id` / `session_id`, but not `run_id` / `capability_call_id`.
  - `session_manager.rs` persists ACP sessions at `$HOME/.nexus42/acp/sessions.json`, keyed by ACP `SessionId`, agent, workspace, and timestamps only.
- `crates/nexus-contracts/`
  - Local ACP DTOs exist in `local/acp/types.rs` and daemon runtime DTOs in `local/acp_runtime/`.
  - No trace/correlation DTOs exist.
- Existing near-matches are not sufficient:
  - `acp_worker` has a per-worker numeric `request_id` counter for `worker/acp_prompt_chunk`; it is local to a worker process and cannot correlate full runs.
  - `graph-flow`/orchestration sessions have `session_id`, but session IDs are not equivalent to user-visible run IDs.
  - `context-assembly-v1` has `request_id`, but that belongs to platform/local API request correlation, not ACP execution tracing.

## Definitions

### `run_id`

`run_id` is the top-level correlation identifier for one user- or daemon-initiated execution attempt.

- For `nexus42 acp run`, one CLI invocation has exactly one `run_id`.
- For a `nexus42 daemon schedule ...` orchestration start attempt, one orchestration start attempt has exactly one `run_id` even if it later fans out into sessions, workers, and capability calls.
- It is opaque and stable for logs/metadata. Recommended generated form: `run_<32 lowercase hex uuid>`.
- Users may provide it explicitly via `--run-id <RUN_ID>` for reproducible external correlation; if omitted, the CLI generates one.

### `capability_call_id`

`capability_call_id` is the child correlation identifier for a single capability invocation under a `run_id`.

- A `CapabilityTask` invocation gets one `capability_call_id`.
- ACP prompt dispatch through `AcpPromptTask` also gets a `capability_call_id` so worker `request_id` chunks can be traced back to a graph node/preset state.
- It is opaque and unique within a process/run. Recommended generated form: `cap_<32 lowercase hex uuid>`.
- It must always be recorded with its parent `run_id` when a run is known.

## Non-Targets

- Do **not** add OpenTelemetry, spans, exporters, or distributed tracing.
- Do **not** change platform production schemas or generated `schemas/platform/*` DTOs.
- Do **not** introduce handwritten DTOs outside `nexus-contracts` for trace metadata.
- Do **not** make the trace store authoritative state for orchestration recovery; it is debug/support metadata only.
- Do **not** complete the pending full ACP prompt integration in `acp run`.

## Design

### Storage Location

Use a file-based pre-1.0 trace seed:

```text
$HOME/.nexus42/acp/runs/<run_id>/trace.jsonl
```

Each line is one `nexus-contracts` local trace event serialized as JSON. This keeps storage simple, inspectable, and independent of SQLite migrations while the execution model is still shifting.

Recommended event order for a simple `acp run`:

```json
{"schema_version":1,"event":"run_started","run_id":"run_...","entrypoint":"acp.run",...}
{"schema_version":1,"event":"run_finished","run_id":"run_...","status":"completed",...}
```

Recommended event order for future orchestration/capability calls:

```json
{"schema_version":1,"event":"run_started","run_id":"run_...","entrypoint":"daemon.schedule",...}
{"schema_version":1,"event":"capability_call_started","run_id":"run_...","capability_call_id":"cap_...","capability_name":"acp.prompt",...}
{"schema_version":1,"event":"capability_call_finished","run_id":"run_...","capability_call_id":"cap_...","status":"completed",...}
```

### Propagation Rules

- CLI entrypoints generate or accept `run_id` at the boundary.
- `acp run` passes `NEXUS_RUN_ID=<run_id>` into spawned ACP agent subprocesses.
- Orchestration stores `_run_id` in `graph_flow::Context` when starting a run/session.
- `CapabilityTask` and `AcpPromptTask` read `_run_id` from context and generate a fresh `capability_call_id` per invocation.
- `AcpPromptTask` includes `run_id` and `capability_call_id` in `worker/acp_prompt` params.
- `acp_worker` echoes those fields in `worker/acp_prompt_chunk` notifications and final responses when present.

## Tasks

### T1: Add local trace DTOs in `nexus-contracts`

**Files:**

- Create: `crates/nexus-contracts/src/local/acp_runtime/trace.rs`
- Modify: `crates/nexus-contracts/src/local/acp_runtime/mod.rs`

Add Nexus-owned local DTOs for trace metadata. Keep these outside `src/generated/` and outside platform schemas.

Required types:

- `NexusRunId(String)`
- `NexusCapabilityCallId(String)`
- `NexusRunTraceEvent`
- `NexusCapabilityCallTraceEvent`
- `NexusTraceEvent` enum tagged by `event` with variants:
  - `run_started`
  - `run_finished`
  - `capability_call_started`
  - `capability_call_finished`

Minimum fields:

- Common: `schema_version: u32`, `run_id`, timestamps as RFC3339 strings.
- Run events: `entrypoint`, optional `agent_id`, optional `cwd`, optional `pid`, `status` for finish.
- Capability events: `capability_call_id`, `capability_name`, optional `session_id`, optional `task_id`, optional `worker_request_id`, `status` for finish, optional `error`.

Tests:

- Round-trip JSON serialization for each variant.
- Assert event tag uses snake_case (`run_started`, `capability_call_started`).
- Assert newtypes serialize transparently.

### T4: Wire `--run-id` into `nexus42 acp run`

**Files:**

- Modify: `crates/nexus42/src/commands/acp.rs`
- Modify if needed: `crates/nexus-acp-host/src/transport.rs`

Update `AcpCommand::Run`:

```text
nexus42 acp run <agent_ref> [--message <TEXT>] [--cwd <PATH>] [--run-id <RUN_ID>]
```

Behavior:

- If `--run-id` is supplied, validate it and use it.
- If omitted, generate `run_<uuid32>`.
- Print the run ID near startup, e.g. `Run ID: run_...`.
- Write `run_started` before spawning the subprocess.
- Write `run_finished` after normal exit, timeout, Ctrl+C cleanup, or subprocess error. Use status values: `completed`, `failed`, `cancelled`, `timeout`.
- Pass `NEXUS_RUN_ID` to the spawned ACP agent process.

`AgentSpawner` currently accepts only `program` and `args`. Add the smallest extension that preserves existing callers:

- preferred: add `spawn_with_env(program, args, env: &[(&str, &str)])` and keep `spawn()` delegating to it with an empty env;
- avoid changing existing behavior for probe/tests.

- `command_surface_contract.rs`: `acp run --help` shows `--run-id`.
- Unit test for run-id validation/generation helper.
- `AgentSpawner::spawn_with_env` test with a mock command where practical on Unix; otherwise unit-test command construction indirectly if cross-platform support is awkward.

## Risks and Mitigations

- **Risk: ID semantics drift between CLI, daemon, and orchestration.**
  Mitigation: define all trace DTOs in `nexus-contracts` local modules and use the same newtypes everywhere.
- **Risk: trace writes fail and break user runs.**
  Mitigation: for V1.16, trace writes are best-effort for CLI/worker paths; warnings are acceptable, but execution should continue.
- **Risk: path traversal via user-provided `--run-id`.**
  Mitigation: validate IDs before constructing paths using `nexus-home-layout` safety helpers.
- **Risk: over-engineering.**
  Mitigation: no span hierarchy, no exporter, no DB schema, no platform wire schema. JSONL files plus IDs are enough for M3.

## Handoff Notes

- Run GitNexus impact analysis before modifying the named functions/symbols during implementation, especially `cmd_run`, `AgentSpawner::spawn`, `CapabilityTask::run`, `AcpPromptTask::run`, and `handle_acp_prompt`.
- Keep code changes surgical. Plan 7 should seed correlation only; deeper orchestration execution semantics belong to later plans.
- If a future plan replaces JSONL with SQLite, keep the `run_id` / `capability_call_id` DTOs and path helpers backward-compatible for at least the pre-1.0 development window.
