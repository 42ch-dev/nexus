### 🟡 Warning

#### W-1: Runtime `sqlx::query()` used for static production INSERTs in `narrative_write.rs`

**Issue**: The new `narrative_write.rs` write repository uses runtime `sqlx::query()` for static INSERT statements (`create_world`, `append_event`). The module doc comment justifies this as "to avoid requiring `cargo sqlx prepare` for each new query."

**Rule violated**: `crates/nexus-local-db/AGENTS.md` states:
> "Compile-time checked queries only — use `sqlx::query!()` / `sqlx::query_as!()` for all static SQL. Runtime `sqlx::query()` only for DDL, PRAGMAs, or truly dynamic SQL with a `// SAFETY:` comment."

While `// SAFETY:` comments are present, these are static INSERTs against known schema — they are eligible for compile-time checking. Using runtime queries for production write paths creates a precedent that undermines the crate's type-safety guarantees and makes schema drift harder to detect at build time.

**Impact**: Maintainability risk. If the DDL evolves (e.g., column added/removed/renamed), these runtime queries will fail at runtime rather than at compile time.

**Fix**: Migrate the three static queries (creator EXISTS, world INSERT, event INSERT) to `sqlx::query!()` and commit the `.sqlx/` query cache entries. The SELECT MAX query and the SELECT root_fork_branch_id in `world.rs` should also be compile-time checked.

---

#### W-2: Hardcoded `workspace_id = 'wrk_local'` without FK validation

**Issue**: `create_world` hardcodes `workspace_id = 'wrk_local'` in the INSERT statement. The DDL defines `workspace_id TEXT NOT NULL` but does **not** declare a FOREIGN KEY constraint to a `workspaces` table. There is no validation that `'wrk_local'` is a valid workspace ID.

**Impact**: Data consistency risk. If downstream read paths ever join against a `workspaces` table, worlds created by this function will have dangling workspace references. The seed helper (`seed_shared.rs`) uses `'wrk_test'`, indicating no canonical local workspace ID is established.

**Fix**: Either (a) add a `workspaces` table FK and validate the workspace exists, or (b) document in `narrative_write.rs` why `'wrk_local'` is the canonical local workspace identifier and ensure all local paths use it consistently.

---

#### W-3: Sequence conflict error handling code path is not tested

**Issue**: `append_event` contains logic to detect SQLite UNIQUE constraint violations (error code `2067`) and map them to `NarrativeWriteError::SequenceConflict`. None of the 8 unit tests in `narrative_write.rs` exercise this branch.

**Impact**: Low-confidence code path. A race condition or concurrent append (or a future bug in sequence allocation) could trigger this path. Without test coverage, regressions would go undetected.

**Fix**: Add a test that simulates a sequence conflict, e.g., by inserting an event directly with `sqlx::query()` to occupy a specific sequence number, then calling `append_event` and asserting `SequenceConflict` is returned.

### 🟢 Suggestion

#### S-1: Inconsistent ID prefix validation

`append_event` validates `world_id` starts with `wld_`, but `create_world` does not validate `owner_creator_id` starts with `ctr_`. Both functions perform FK existence checks, but prefix validation is only applied to one side. Consider adding `validate_id_prefix` for `creator_id` as well, or document why creator IDs are not validated.

#### S-2: CLI error wrapping loses structured error information

Both `world.rs` and `creator/mod.rs` map `NarrativeWriteError` / `KbStoreError` into `CliError::Other(format!("...: {e}"))`. This discards the structured error variants (`FkNotFound`, `InvalidId`, `SequenceConflict`) that could be used for richer CLI error messages or programmatic handling. Consider adding explicit variant mappings to `CliError`.

#### S-3: `block_type` default is implicit in code, not visible in CLI help

In `kb_add` (world scope), `block_type` defaults to `"InfoPoint"` when the CLI argument is omitted. The `clap` declaration marks `block_type` as `Option<String>` with no default, so `--help` will not show the default value. Users may not know what block type will be assigned.

**Fix**: Either add `#[arg(long, default_value = "InfoPoint")]` to the clap field, or document the default in the argument help text.

#### S-4: No integration tests for world KB CLI write paths in `nexus42`

The `kb_add` and `kb_remove` world scope implementations in `creator/mod.rs` are not covered by `nexus42` integration tests. The `nexus-local-db` tests cover the store layer, but the CLI dispatch, argument parsing, and `require_world_id` validation are untested at the crate level.

#### S-5: `generate_root_branch_id` uses low-entropy UUID truncation

`generate_root_branch_id` takes only 8 hex characters from a UUID (64 bits of entropy ≈ 1.8×10¹⁹). While collision probability is low for typical local usage, it is not zero. For long-running local databases or automated testing, this could eventually produce a duplicate. Consider using the full UUID or adding a uniqueness retry loop.
