### 🟡 Warning

**F-001: `is_user_authenticated()` does not check token expiry**

- **Location**: `crates/nexus42/src/auth/mod.rs:182-186`
- **Source Type**: manual-reasoning
- **Issue**: `is_user_authenticated()` checks `!t.access_token.is_empty()` but does **not** compare `expires_at` against current time. An expired JWT would still be considered valid, causing downstream platform requests to fail with 401 rather than proactively prompting re-authentication.
- **Impact**: `creator_auth::get_user_token()` relies on this method (line 64). Users with expired tokens will get opaque platform 401 errors instead of a clear "session expired, please re-login" message.
- **Fix**: Parse `expires_at` (ISO 8601 string) and compare with `chrono::Utc::now()`:
  ```rust
  pub fn is_user_authenticated(&self) -> bool {
      self.user_token.as_ref().is_some_and(|t| {
          !t.access_token.is_empty()
              && chrono::DateTime::parse_from_rfc3339(&t.expires_at)
                  .map(|dt| dt > chrono::DateTime::<chrono::Utc>::from(chrono::Utc::now()))
                  .unwrap_or(false)
      })
  }
  ```
- **Confidence**: High

**F-002: DeviceID resolution failure silently passes empty string to CLI**

- **Location**: `crates/nexus42/src/main.rs:233-240`
- **Source Type**: manual-reasoning
- **Issue**: When `get_or_create_device_id()` fails, only `tracing::warn!` is emitted. `config.device_id` remains `""` (default for `String`). Subsequent `auth login` will fail inside `DeviceFlowClient::new()` with `SyncError::SyncNotConfigured("device_id is required")`, producing a confusing error message that doesn't mention the root cause (device ID file creation failure).
- **Impact**: Users with filesystem permission issues on `~/.nexus42/` will see cryptic "device_id is required" errors instead of actionable guidance about fixing their home directory.
- **Fix**: Either (a) fail fast with a clear error during config resolution, or (b) change `CliConfig.device_id` to `Option<String>` and validate at the point of use with a user-friendly error:
  ```rust
  // In device_flow_client.rs or user_auth.rs:
  if config.device_id.is_empty() {
      return Err(CliError::Other(
          "Failed to initialize device ID. Ensure ~/.nexus42/ directory exists and is writable.".into(),
      ));
  }
  ```
- **Confidence**: High

### 🟢 Suggestion

**F-003: `login()` performs redundant `AuthStore::load()` after storing token**

- **Location**: `crates/nexus42/src/auth/user_auth.rs:72-82`
- **Source Type**: git-diff
- **Issue**: After `store.store_user_token(user_token)?` (which internally calls `self.save()` and writes to disk), the code immediately reloads: `let store = AuthStore::load()?;` just to display the stored values. This is an unnecessary second file I/O operation.
- **Fix**: Use the already-constructed `user_token` variable for display:
  ```rust
  println!("\u{2713} Authenticated successfully.");
  println!("  User: {}", user_token.user_id);
  println!("  Token type: {}", user_token.token_type);
  println!("  Expires: {}", user_token.expires_at);
  ```
- **Confidence**: High

**F-004: `DeviceFlowClient` uses inconsistent error return types between methods**

- **Location**: `crates/nexus-sync/src/device_flow_client.rs:125` vs `172`
- **Source Type**: git-diff
- **Issue**: `request_device_code()` returns `SyncResult<DeviceCodeResponse>` (i.e., `Result<..., SyncError>`) while `poll_device_token()` returns `Result<DeviceTokenResponse, DeviceFlowError>`. Callers in `user_auth.rs` must handle two different error types, requiring a manual mapping at the call site.
- **Impact**: Minor API inconsistency. Future consumers of `DeviceFlowClient` will need to know which error type each method returns.
- **Fix**: Consider either (a) making `poll_device_token` also return `SyncResult` and mapping `DeviceFlowError` → `SyncError` internally, or (b) making both return the domain-specific `DeviceFlowError`. Option (a) keeps the existing `SyncError` convention consistent with `PlatformClient`.
- **Confidence**: Medium

**F-005: Dead code retained in `context/client.rs` and `context/types.rs`**

- **Location**: `crates/nexus42/src/context/client.rs` (entire file), `crates/nexus42/src/context/types.rs:12-40`
- **Source Type**: git-diff
- **Issue**: `client.rs` is marked `#![allow(dead_code)]` and retains the full `ContextClient` wrapping `DaemonClient`. `types.rs` gates `is_error`, `error_code`, `error_message` with `#[cfg(test)]`. Per the compass (§9, compass §6), these are "retained for reference" but add compilation surface and maintenance burden.
- **Impact**: Zero runtime risk, but every future change to `DaemonClient` or context types must also update this dead code to keep it compiling. This is a slow accumulation of technical debt.
- **Fix**: Defer to V1.11 as planned (compass §6.3, WS-C), but add a tracking issue or TODO with an explicit deadline. Consider whether the `#[cfg(test)]`-only helpers in `types.rs` could also be removed now since the command itself no longer uses them.
- **Confidence**: High

**F-006: Dual reference to `device_code_sessions` in initial schema and migration**

**F-007: `PlatformClient::new()` breaking API change requires `device_id`**

- **Location**: `crates/nexus-sync/src/platform_client.rs:137`
- **Source Type**: git-diff
- **Issue**: `PlatformClient::new()` signature changed from `new(url, token)` to `new(url, token, device_id)`. All callers within this repo were updated (verified: `creator.rs`, `platform_client.rs` tests). However, the `nexus-sync` crate is published and consumed by `nexus-platform` (private repo) via npm/Rust. If the Rust crate is re-published, the platform repo will break compilation until updated.
- **Impact**: Coordinated update needed with platform repo if `nexus-sync` crate is published. The compass confirms `@42ch/nexus-contracts` major bump requires coordinated platform upgrade (§Versioning).
- **Mitigation**: Already documented in compass risk RISK-V110-04 (AuthStore JSON compat). Add a note about `PlatformClient` API break to the release notes.
- **Confidence**: Medium

## Long-Term Evolution Risk Assessment

| Design Choice | V2.0 Impact | Assessment |
|--------------|-------------|------------|
| Separate `DeviceFlowClient` from `PlatformClient` | ✅ Positive | Clean separation of unauthenticated (device flow) vs authenticated (creator API) flows. Makes adding OAuth2 PKCE or client credentials in V2.0 straightforward. |
