## Findings

### 🟡 Warning

#### W-QC2-001: `poll_device_token` silent fallback masks parse failures
- **File**: `crates/nexus-sync/src/device_flow_client.rs:241`
- **Severity**: Warning
- **Source**: manual-reasoning
- **Detail**: When `serde_json::from_str::<serde_json::Value>(&text)` fails on a 200 response, the code silently falls back to `serde_json::json!({ "data": &text })`, producing a misleading envelope that may parse to a `DeviceTokenResponse` with unexpected content. No error is logged or propagated.
- **Fix**: If `envelope` parse fails on a non-error status code, return `Err(DeviceFlowError::Other(format!("Failed to parse success response: {e}")))` instead of the silent fallback.
- **Confidence**: Medium (parsing would likely fail with a serde error anyway when `data` contains malformed text, but the fallback produces confusing diagnostics)

#### W-QC2-002: JWT user_id extraction silently degrades to "unknown" without logging
- **File**: `crates/nexus42/src/auth/user_auth.rs:57-58`
- **Severity**: Warning
- **Source**: manual-reasoning
- **Detail**: If `extract_user_id_from_jwt` fails (malformed base64 or non-JSON payload), the login proceeds with `user_id = "unknown"`. No warning is logged. While the token is still validated by the platform, a silent "unknown" user_id could confuse debugging and audit logs.
- **Fix**: Add `tracing::warn!("Failed to extract user_id from JWT, using 'unknown'")` before the `unwrap_or_else`.
- **Confidence**: High

#### W-QC2-003: `login()` error message inaccuracy on token store failure
- **File**: `crates/nexus42/src/auth/user_auth.rs:67-68`
- **Severity**: Warning
- **Source**: manual-reasoning
- **Detail**: When `store.store_user_token()` fails (e.g., I/O error on auth.json), the error propagates as `CliError::Other("Device authorization expired")` only if it's the `ExpiredToken` match arm. For other errors, the `?` would propagate the actual error... but if the match arm for `DeviceFlowError::ExpiredToken` at line 97 returns `Err(CliError::Other("Device authorization expired"))` and this is reached after a store failure, the error message is misleading.
- **Fix**: Ensure `store.store_user_token()` errors propagate with their actual message rather than being caught by the `ExpiredToken` arm. The current code structure does NOT actually have this bug (the `?` on line 68 would propagate before reaching the match), but the error handling around lines 52-77 could be clearer. Consider extracting the store operation result earlier.
- **Confidence**: Medium (code path unlikely but error messaging confusing if reached)

#### W-QC2-004: TOCTOU race in `get_or_create_device_id` file permission
- **File**: `crates/nexus-sync/src/device_id.rs:42-49`
- **Severity**: Warning
- **Source**: manual-reasoning
- **Detail**: Between `std::fs::write(&path, &new_id)` (line 42) and `std::fs::set_permissions(&path, perms)` (line 49), the file exists with whatever permissions the process umask allows (typically 0644). This is a narrow TOCTOU window. Additionally, `create_dir_all` on line 38 creates parent directories with default permissions before the file-specific 0600 is applied.
- **Fix**: Consider writing to a temp file with restrictive permissions first, then atomically renaming (or use `std::fs::OpenOptions` with explicit permissions). For the directory, `set_permissions` on the parent after creation.
- **Confidence**: Medium (low exploitability — device_id is not high-value secret; window is narrow)

### 🟢 Suggestion

#### S-QC2-001: AuthStore lacks file locking for concurrent access
- **File**: `crates/nexus42/src/auth/mod.rs:77-91`
- **Source**: manual-reasoning
- **Detail**: `AuthStore::save()` has no file locking. Concurrent calls to `login()` (or other operations that save) from multiple processes could produce garbled JSON. The `#[cfg(unix)]` permission setting is also not atomic with the write.
- **Note**: Acceptable for MVP since CLI is typically single-process and the auth flow is user-initiated. Document as tech debt for V2.0.
- **Confidence**: Low

#### S-QC2-002: Consider `VALIDATION_ERROR` handling in device flow
- **File**: `crates/nexus-sync/src/device_flow_client.rs:206-229`
- **Source**: manual-reasoning
- **Detail**: The platform contract (§5 delivery compass) lists `VALIDATION_ERROR` (400) as a possible platform error. The current code treats all non-4-error-code 400s as `DeviceFlowError::Other`. While this is technically correct (VALIDATION_ERROR would fall through to `Other`), it could be worth adding explicit handling for better error messages in the future.
- **Confidence**: Low
