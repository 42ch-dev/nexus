### 🟡 Warning

**W-1: Missing prompt injection queue cleanup implementation**
- **Source**: `crates/nexus-local-db/src/prompt_injection.rs`, `crates/nexus-local-db/migrations/20260530_creator_prompt_injections.sql`
- **Issue**: The schema includes `expires_at` and a `creator_prompt_injections_cleanup` index, but no cleanup/expiration function is implemented. The architect note (C0/J0) explicitly documents cleanup semantics (expire queued rows, requeue stale claimed rows, delete old consumed/expired rows), yet the codebase has zero cleanup code.
- **Impact**: Queue table grows unbounded over time. Expired and consumed rows accumulate indefinitely, degrading query performance and increasing disk usage.
- **Fix**: Add `cleanup_expired_prompt_injections(pool, now, retention_ms)` and call it opportunistically before enqueue/drain or during daemon maintenance.

**W-2: Unbounded IN clause parameter count in `mark_prompt_injections_consumed`**
- **Source**: `crates/nexus-local-db/src/prompt_injection.rs:196-211`
- **Issue**: `mark_prompt_injections_consumed` builds a dynamic SQL `IN (...)` clause with one placeholder per injection ID. SQLite has a hard limit of 32766 parameters per query. If a caller passes a slice larger than ~32764 IDs, the query will fail at runtime.
- **Impact**: Reliability failure at scale when batch-consuming large queues. No validation or chunking is performed.
- **Fix**: Chunk the input slice into batches of ≤1000 IDs (or ≤SQLITE_MAX_VARIABLE_NUMBER) and execute multiple UPDATE statements within a transaction.

**W-3: Unbounded prompt size in `build_summary_prompt`**
- **Source**: `crates/nexus-orchestration/src/capability/builtins/context_summarize.rs:157-168`
- **Issue**: `build_summary_prompt` concatenates `content`, `trace`, and `template` without any length validation before passing the result to `workers.call_acp_prompt()`. If `content` is very large (e.g., a full manuscript), the prompt could exceed worker/LLM token limits or cause memory pressure.
- **Impact**: Worker IPC failure, LLM context window overflow, or excessive memory allocation in the hot path.
- **Fix**: Add a configurable max prompt length (e.g., 32_768 chars) and truncate `content` with an ellipsis marker before building the prompt. Document the limit in the capability schema.

**W-4: JSON LIKE query performance in `list_fragments_filtered`**
- **Source**: `crates/nexus-local-db/src/memory_fragment.rs:177-239`
- **Issue**: Keyword filtering uses `LIKE '%"keyword"%'` against a JSON array stored as text in the `keywords` column. SQLite cannot use an index for this pattern (leading wildcard). The `memory_fragments` table has no index on `keywords`.
- **Impact**: Full table scan for every keyword-filtered memory read. Degrades linearly with fragment count.
- **Fix**: Add a `memory_fragment_keywords` junction table (fragment_id, keyword) with an index on keyword, or at minimum add a functional index. Alternatively, document this as a known limitation for pre-1.0.

**W-5: Pre-existing SQLX offline query cache incomplete** 
- **Source**: `crates/nexus-local-db/src/kb_store.rs`, `.sqlx/`
- **Issue**: `cargo test -p nexus-local-db` fails with "set DATABASE_URL to use query macros online, or run cargo sqlx prepare" for existing queries in `kb_store.rs`. This is a pre-existing issue not introduced by this PR, but it blocks running the full test suite for the changed crate.
- **Impact**: CI/test reliability. New code in `prompt_injection.rs` and `memory_fragment.rs` uses runtime queries (not sqlx macros) so it is unaffected, but the crate-wide test suite is broken.
- **Fix**: Run `cargo sqlx prepare` for `nexus-local-db` or configure CI with `DATABASE_URL`.

**W-6: `generate_ulid()` counter reset on process restart**
- **Source**: `crates/nexus-orchestration/src/capability/builtins/creator.rs:288-296`
- **Issue**: `generate_ulid()` uses a process-local `AtomicU64` counter. After process restart, the counter resets to 0. If two processes generate IDs within the same millisecond (or if the system clock goes backward), collisions are possible.
- **Impact**: Low probability, but a collision would cause a primary key violation on `memory_fragments` or `creator_prompt_injections`.
- **Fix**: Use a proper ULID library (e.g., `ulid` crate) or prepend a process-scoped entropy seed to the counter.

### 🟢 Suggestion

**S-1: Add hard limit to `claim_prompt_injections`**
- **Source**: `crates/nexus-local-db/src/prompt_injection.rs:129`
- **Issue**: The `limit` parameter accepts any `i64` value. A bug or malicious input could request an extremely large number of rows.
- **Fix**: Clamp `limit` to a reasonable maximum (e.g., 1000) at the API boundary.

**S-2: Add query timeout configuration**
- **Source**: All new DB operations in `prompt_injection.rs` and `memory_fragment.rs`
- **Issue**: No `query_timeout` is configured on `SqlitePool`. Long-running queries (e.g., unbounded scans) could block async executors.
- **Fix**: Configure `sqlx::sqlite::SqliteConnectOptions::busy_timeout()` or use `tokio::time::timeout` at the call sites for queue operations.

**S-3: Enforce TTL on memory fragment growth**
- **Source**: `crates/nexus-local-db/src/memory_fragment.rs`
- **Issue**: `write_memory` inserts fragments with `ttl: None` unconditionally. The `ttl` column exists but is never enforced.
- **Fix**: Add a periodic cleanup job that deletes fragments where `ttl` has expired, or default `ttl` to a reasonable value (e.g., 90 days) in `write_memory`.

**S-4: Optimize `judge_rule` operator matching**
- **Source**: `crates/nexus-orchestration/src/capability/builtins/judge_rule.rs:108-117`
- **Issue**: `evaluate_context_comparison` iterates through all 6 operators and calls `expr.find(op)` for each. For long expressions with early-match operators, this does redundant work.
- **Fix**: Use a single regex scan or parse the expression once with a small state machine. This is a micro-optimization; correctness is not affected.
