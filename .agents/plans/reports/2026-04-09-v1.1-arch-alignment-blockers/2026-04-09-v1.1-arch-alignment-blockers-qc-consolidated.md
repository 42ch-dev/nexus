# QC Consolidated Decision — V1.1 Architecture Alignment Blockers

## Executive summary

Three independent QC reports reviewed the same **`plan_id`**, **`Review cwd`**, and **`Diff basis`** (verbatim merge-base → HEAD range above).

| Reviewer | Verdict (individual) | Primary lens |
|----------|----------------------|--------------|
| qc-specialist (#1) | **Request Changes** | Architecture truth vs plan (TD-1 vs TD-5 on real push path), codegen/domain consistency |
| qc-specialist-2 (#2) | **Request Changes** | Process SSOT (`status.json` envelope), security/correctness boundaries, semver |
| qc-specialist-3 (#3) | **Approve** | Wire stability, hash preimage determinism, forward-compat / consumer foot-guns |

**Consolidated verdict: Request Changes — branch is not merge-ready** under this repository’s **plan / pre-merge SSOT conventions** (see `AGENTS.md` + `plan-convention.md`).

**Reason (short):** Reviewers #1 and #2 agree on two **program-blocking** themes: (1) **`.agents/plans/status.json` no longer uses the standard harness envelope** (missing `version`, `plans[]`, program `metadata`), and (2) **TD-1 “sync wiring” is misaligned** with shipped behavior: live `push` appends a `SyncCommand` only; **`BundleBuilder`, real `Precheck`, and `SyncClient::push` are not on the CLI→daemon path**, so **`canonical_hash` (TD-5) is not exercised end-to-end for real pushes**. Reviewer #3 does not contradict these facts; its **Approve** reflects “no critical security defect in wire/hash review scope,” not an override of repo process gates or TD-1 acceptance honesty.

**Automated gates (as reported by QC):** `cargo clippy -D warnings`, `cargo test --all`, `pnpm run validate-schemas`, `pnpm run typecheck` — **pass** on reviewed `HEAD`. This **does not** negate the SSOT / acceptance alignment blockers above.
