## Findings

### R-01 — critical — Plan SSOT (`status.json`) structure destroyed

**Issue:** At `merge-base`, `status.json` is a harness-compatible document with `version`, `plans` array, and root-level metadata. At `HEAD`, it is **only** one plan row object (no `plans`, no program `metadata`). All other plans and aggregate fields (e.g. `residual_findings`, `tech_debt_summary`) are absent from the hot file.

**Impact:** Breaks documented `jq` workflows in `AGENTS.md`, triple-QC/residual tracking conventions, and any automation expecting the array SSOT.

**Suggestion:** Restore the standard `status.json` envelope; add/update **one** `plans[]` row for this plan; keep residuals and rollups in root `metadata` per `plan-convention.md`.

---

### R-02 — high — TD-1 / sync push: documented pipeline vs implementation

**Evidence:**

- `crates/nexus42d/src/api/handlers/sync.rs` module comment claims wiring of `Outbox`, **`Precheck`**, **`SyncClient`**.
- `push` constructs `SyncCommand`, calls `outbox.append(&command)`, and optionally builds a `PrecheckSummary` from **empty-field checks only**. There is **no** `BundleBuilder`, **no** `nexus_sync::precheck::Precheck`, **no** `SyncClient::push` (confirmed via grep: only comments/types reference Precheck/SyncClient).
- `Outbox::append` persists `command_payload` with a synthetic `bundle_id` / `idempotency_key`; it does **not** store a built `Bundle` or `canonical_hash`.

**Impact:** Status/outbox counts may be “real”, but **push does not exercise bundle construction, content hash idempotency, or platform client push**—contrary to plan acceptance text and TD-5 integration on the real user flow.

**Suggestion:** Either (a) implement the stated pipeline in the daemon (build bundle → precheck → stage/mark ready → sync client), or (b) narrow plan/docs/acceptance criteria and responses so they do not claim full Precheck/SyncClient wiring.

---

### R-03 — medium — Unbound sync identity parameters

**Issue:** `SyncPushRequest` accepts `workspace_id`, `world_id`, and `creator_id` from JSON with no check against `WorkspaceState` / active binding.

**Impact:** For a **local-only** listener the risk is limited, but any future exposure or mistaken binding widens cross-workspace/command injection. Align with daemon workspace context or validate IDs against persisted bindings.

---

### R-04 — medium — `User` domain aggregate: weak typing vs contracts

**Issue:** `crates/nexus-domain/src/user.rs` stores `account_status` and `subscription_tier` as **`String`**, while contracts use generated enums (`AccountStatus`, `SubscriptionTier`) via `common.schema.json`.

**Impact:** Invalid strings can exist in domain instances unless every constructor path is audited; duplicates TD-4’s goal of compile-time enforcement at the domain boundary.

**Suggestion:** Use the domain enums (already defined in the same file) inside `User`, or use contract types consistently with conversion at the edges.

---

### R-05 — medium — npm semver / consumer breakage (TD-4)

**Issue:** Many TypeScript fields move from `string` to string-literal **union** types. `packages/nexus-contracts/package.json` remains **`0.2.0`**.

**Impact:** Under SemVer, narrowing types can break consumers that relied on `string`; even in `0.x`, release notes and version bump policy should match AGENTS.md (`schema_version` / package lockstep).

**Suggestion:** Confirm publish policy: bump minor (or document intentional tightening) before `npm publish`.

---

### R-06 — low — `sync pull` still explicitly incomplete

CLI `sync pull` prints that full pull is not implemented. Acceptable if out of scope for this plan; **plan checklist** should state residual scope to avoid “done” ambiguity.

---

### R-07 — nit — CLI default IDs for push

`world_id` defaults to `"unknown"` and workspace to `"local"` when omitted—easy to mis-stake commands without surfacing config gaps loudly.
