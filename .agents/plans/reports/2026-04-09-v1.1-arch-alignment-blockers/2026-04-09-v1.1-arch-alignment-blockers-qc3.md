## Findings

### Critical

- None. No blocking security defects identified in the reviewed wire/hash/contract changes (e.g. no new hardcoded secrets, no obvious injection surface in the schema-only path).

### Warning

- **F-001 — Canonical hash bytes depend on `serde_json` layout of `Delta` (especially `payload`)**  
  `canonical_hash_for_deltas` hashes `serde_json::to_vec` of `&[Delta]`. `payload` is `serde_json::Value`; object key order and any re-encoding differences across implementations (Rust vs TS `JSON.stringify`, or two Rust paths building the same logical object in different orders) can yield **different hashes for logically equal deltas**. That undermines cross-party verification of `canonical_hash` if the platform re-computes from its own serialization.  
  **Mitigation / fix direction:** Document the normative byte rule (“hash = SHA-256 of compact JSON for `deltas` produced by Nexus reference serialization”) or normalize payloads (e.g. canonical JSON / sorted keys) before hashing; add contract tests that assert parity with a reference fixture shared with `nexus-platform`.  
  **Source:** `crates/nexus-sync/src/canonical_hash.rs`, `crates/nexus-contracts/src/generated/delta.rs` (`payload: serde_json::Value`).

- **F-002 — Hash scope vs envelope**  
  Implementation hashes **only** the serialized `deltas` array, not the full bundle envelope (`idempotency_key`, `base_versions`, metadata). The schema describes `canonical_hash` as a generic “Content hash (`sha256:hexlower`)” without pinning the preimage. If the platform assumes a full-envelope or alternate preimage, dedupe/idempotency will disagree with the CLI.  
  **Mitigation:** Align `schemas/domain/bundle.schema.json` description (and sync-contract docs) explicitly with “hash of `deltas` as JSON array” or change implementation to match spec.  
  **Source:** `crates/nexus-sync/src/canonical_hash.rs`, `schemas/domain/bundle.schema.json` (`canonical_hash`).

- **F-003 — Precheck does not validate `canonical_hash` format or non-emptiness**  
  `precheck_bundle` never checks `canonical_hash` matches `sha256:` + 64 hex or is non-empty; unit tests still use `String::new()`. Bundles constructed outside `BundleBuilder` (tests, future call sites, or deserialized wire data) could carry an empty or malformed hash while passing precheck.  
  **Mitigation:** Add a precheck rule for production bundles; keep tests using valid placeholder hashes.  
  **Source:** `crates/nexus-sync/src/precheck.rs` (no `canonical_hash` references except test fixture).

- **F-004 — TD-4 enum typing: forward compatibility and semver pressure for `@42ch/nexus-contracts`**  
  JSON Schema `enum` → Rust closed enums (`serde`) and TypeScript string unions (`CommonTypes.ts`). **New wire values** require schema + package release; older clients fail deserialization / fail exhaustive TS switches. This is expected but is a deliberate **compatibility contract**: platform must not emit new enum literals until consumers bump, or must tolerate version skew via separate negotiation.  
  **Source:** `packages/nexus-contracts/src/generated/CommonTypes.ts`, `crates/nexus-contracts/src/generated/common_types.rs`, `schemas/common/common.schema.json`.

- **F-005 — `additionalProperties: false` on core objects (e.g. User, Bundle)**  
  Strict schemas help catch typos but **reject additive forward-compatible fields** from the server if consumers validate with the same schema. Platform must gate rollouts with schema version bumps or relax validation for unknown keys at the application layer.  
  **Source:** `schemas/domain/user.schema.json`, `schemas/domain/bundle.schema.json`.

- **F-006 — `BundleBuilder::build_validated` can panic**  
  After `validate()`, `build()` is unwrapped with `expect(...)`. If `build()` ever fails while validation passes, the daemon/CLI thread panics instead of returning `Result`. Low probability but poor degradation.  
  **Source:** `crates/nexus-sync/src/delta_bundle.rs` (`build_validated`).

### Suggestion

- **F-007 — User wire shape and privacy**  
  `User` requires `email`, `username`, `display_name` on the wire (`schemas/domain/user.schema.json`). Ensure platform/logging/telemetry policies **redact** these fields in traces; contracts do not enforce this—operational policy only.  
  **Source:** `schemas/domain/user.schema.json`, `packages/nexus-contracts/src/generated/User.ts`.

- **F-008 — Operational observability**  
  If the platform relies on `canonical_hash` metrics/alerts, define dashboards for mismatch rate between client-reported and server-recomputed hash **once** F-001/F-002 are nailed down.
