### 🟡 Warning

- **[W-QC3-001] TimeoutConfig defined but not enforced in provider execute paths**
  - **Source**: `crates/nexus-agent-host/src/config.rs` — `TimeoutConfig` is loaded and its Duration accessors (`launch_duration()`, `initialize_duration()`, `session_duration()`, `prompt_duration()`, `shutdown_duration()`) exist and are tested (lines 188–216, tests at lines 340–362). However, no `HostManager`, `AcpProvider`, or `ClaudeCliProvider` implementation actually applies these timeouts to async operations.
  - **Impact**: Long-running ACP operations (initialize, prompt streaming) or native CLI executions can run indefinitely without triggering a cancellation. The configured `launch_ms=15000`, `initialize_ms=15000`, `session_ms=30000`, `prompt_ms=180000`, `shutdown_ms=5000` are effectively dead code.
  - **Fix**: Wrap provider `execute()` streams and session creation futures with `tokio::time::timeout()` using the configured Duration values. For ACP: wrap `client.initialize()` with `initialize_duration()` and `client.stream_prompt()` with `prompt_duration()`. For native CLI: wrap the `child.wait()` background task and the stdout reader loop with `prompt_duration()`. Add `HostError::OperationTimeout` variants to timeout error paths.
  - **Confidence**: High
  - **Source Trace**: `manager.rs` line 247 (`adapter.execute()`), `providers/acp.rs` line 160 (`self.client.initialize()`), `providers/acp.rs` line 257 (`self.client.stream_prompt()`), `providers/native_cli/claude.rs` lines 270–323 (subprocess spawn without timeout).

- **[W-QC3-002] Session drain on shutdown is bounded but relies on no-op provider shutdown**
  - **Source**: `crates/nexus-agent-host/src/core/manager.rs` lines 320–347 — `shutdown()` transitions sessions to `Stopping → Stopped` and removes them from the registry, but does NOT wait for any provider adapter cleanup or child process drain. The `HostManager::shutdown()` is synchronous and does not coordinate with the ACP or native CLI providers' `shutdown()` methods (which are only called when the daemon-runtime subsystem manager calls `host.shutdown()`).
  - **Impact**: On daemon shutdown, if provider child processes are still running, they may become orphaned. The `shutdown_ms=5000` config value is never used to bound the drain phase.
  - **Fix**: Either (a) call `provider.shutdown()` for each active session during `HostManager::shutdown()` with a bounded await using `shutdown_duration()`, or (b) ensure the daemon-runtime `AgentHostSubsystem::shutdown()` (which does call `self.host.shutdown()`) properly awaits provider cleanup with a timeout guard. Currently `AgentHostSubsystem::shutdown()` at line 98 passes only `grace_ms` from the lifecycle layer, not the agent-host configured `shutdown_ms`.
  - **Confidence**: High
  - **Source Trace**: `manager.rs:320` (`shutdown()` — no provider teardown), `agent_host.rs:98` (subsystem shutdown ignores `agent-host` shutdown timeout).

- **[W-QC3-003] Native CLI subprocess zombie risk — background wait task not captured**
  - **Source**: `crates/nexus-agent-host/src/providers/native_cli/claude.rs` lines 319–322 — `tokio::spawn(async move { let _ = child.wait().await; })` spawns a background task to reap the child, but the `JoinHandle` is dropped immediately. If the stream consumer drops the stream early (cancel), the child may still be running and won't be reaped until the task completes.
  - **Impact**: Short-lived zombies or resource leaks if the stream is dropped before the process exits naturally. The `cancel()` implementation (lines 327–342) is a no-op for native CLI.
  - **Fix**: Store the `JoinHandle` in the `NativeSession` struct (currently has `child` and `stdin` fields but no handle for the background wait task). In `shutdown()`, await the handle with a timeout, or in `cancel()`, explicitly `kill()` the child process.
  - **Confidence**: Medium
  - **Source Trace**: `claude.rs:319–322`, `claude.rs:27–36` (`NativeSession` struct lacks wait handle).

### 🟢 Suggestion

- **[S-QC3-001] ACP `stream_prompt` backpressure not observable**
  - ACP provider `execute()` chains a `ReceiverStream` from the SDK directly onto the host event stream without any buffering configuration. If the consumer of the stream is slow, backpressure is applied to the ACP SDK's internal queue, but this is not visible to the host — no metrics or event signals indicate buffer depth.
  - **Suggestion**: Consider emitting a `Status` event or incrementing a counter when the buffer exceeds a threshold. Low priority for Wave 1.

- **[S-QC3-002] PATH scan is synchronous but non-blocking via `std::env::split_paths`**
  - `path_scan.rs` line 44 uses `std::env::var_os("PATH")` and `std::env::split_paths()` — these are synchronous syscalls but complete instantly on any正常工作 system. The implementation is correct (Wave 1 only scans `claude`, not a large command list). No async blocking concern. Documented for completeness.

- **[S-QC3-003] Telemetry system uses `chrono::DateTime<Utc>` — lightweight and appropriate**
  - `telemetry/events.rs` constructs `TelemetryContext` with `Utc::now()` per event. This is cheap enough for the configured event volumes. JSONL output via `EnrichedEvent::to_json()` is lazy (only computed when written). No performance concern.

- **[S-QC3-004] Provider discovery catalog sort is deterministic but uses `String` comparison**
  - `catalog.rs` lines 95, 108 sort by `provider_id.0.cmp()`. For ASCII provider IDs this is fine. For future Unicode provider IDs this could be non-deterministic across platforms due to `std::path::Path::display()` collation. Low risk.
