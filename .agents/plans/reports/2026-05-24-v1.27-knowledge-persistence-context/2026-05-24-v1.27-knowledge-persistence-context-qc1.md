### 🟡 Warning

**W-1: Schema drift between migration and spec document**

The actual migration file `20260526_knowledge_entries.sql` diverges from `.agents/knowledge/specs/local-db-schema.md` §4.1.3 in two ways:

1. **Missing CHECK constraint in spec**: The migration has `CHECK (entry_id LIKE 'kno_%')` but the spec's CREATE TABLE omits this constraint.
2. **Different indexes**: The migration creates `idx_knowledge_entries_user_created` (user_id, created_at DESC), while the spec documents `idx_knowledge_entries_tags` and `idx_knowledge_entries_user_tags` — neither of which exist in the migration.

This creates a documentation/spec drift that could confuse future implementers or DBAs. The spec should be updated to match the deployed schema.

→ Fix: Update `local-db-schema.md` §4.1.3 to match the migration:
- Add `CHECK (entry_id LIKE 'kno_%')` to the CREATE TABLE
- Replace index definitions with `idx_knowledge_entries_user_id` and `idx_knowledge_entries_user_created`

### 🟢 Suggestion

**S-1: Tag search uses JSON substring matching via LIKE**

The `list()` method in `SqliteKnowledgeStore` implements tag filtering via `tags_json LIKE '%"tag"%'` with quote-escaping. This is functional for the current use case but has two limitations:

1. **Substring false positives**: A tag `"rust"` could match a tag `"rust-tutorial"` because both contain the substring `"rust"`.
2. **Quote-escaping fragility**: The manual `"{}".replace('"', "\\\"")` escaping is correct for standard cases but doesn't handle all JSON edge cases (e.g., tags containing backslash-escaped quotes).

→ Recommendation: Consider migrating to SQLite's built-in JSON functions (`json_array_contains` or `json_each`) if the SQLite version supports it, or document this limitation in the spec. For V1.27, the current approach is acceptable given the controlled tag input (CLI comma-separated values).

**S-2: Error type mapping could be more precise**

All SQL errors in `SqliteKnowledgeStore` are mapped to `KnowledgeError::ValidationError`, including connection failures and constraint violations. This conflates validation errors (bad user input) with storage errors (DB issues).

→ Recommendation: Introduce a `KnowledgeError::StorageError` variant or map SQLx errors more precisely to distinguish user-fixable issues from infrastructure issues.
