### 🟡 Warning
- **W1 (Correctness / Search hygiene — low impact)**: In `knowledge_store.rs:161-181` (list()), tag filter uses heuristic `tags_json LIKE '%"escaped-tag"%'` (JSON string scan with manual escaping and bound params). This is safe from SQL injection (all ? are bound parameters; user_id scoping is always the leading `WHERE user_id = ? AND ...`). However, the LIKE pattern can interpret SQL LIKE metacharacters (`%`, `_`) present in user-supplied tag names as wildcards, leading to potential over-matching in multi-tag AND filters. Not a cross-user leak (user_id isolation is unconditional), but search results may not be exact for adversarial or unusual tag values. Tests cover normal cases and cross-user isolation but not metachar edge cases.
  - Source: dynamic SQL construction + `format!` for pattern before `.bind()`.
  - Fix recommendation: For exact tag membership, prefer `json_each(tags_json, '$.')` or a normalized tags table / FTS5 in a follow-up migration once volume grows. Document current heuristic limitations in knowledge/specs/local-db-schema.md.
  - Severity justification: Warning (not Critical) because (a) no injection vector, (b) user_id scope prevents data exfil, (c) demo/prod data unlikely to trigger, (d) CLI tags are user-controlled but result in only their own entries.

- **W2 (Code hygiene / Consistency — demo path only)**: In `creator/mod.rs:1418` (run_demo_seed cleanup on --force), the knowledge_entries DELETE uses a literal string in the SQL: `WHERE user_id = 'user_default'` instead of a bound parameter (contrast with the immediately following parameterized DELETE for narrative_worlds and the check query above it in the same function). This is a static, hardcoded, demo-only path with no user-controlled input reaching the query string, so no injection risk. However, it is inconsistent with the "all queries parameterized" and "SAFETY comments" discipline used everywhere else in the new knowledge code and Plan 1 write paths.
  - Source: git diff of mod.rs demo seed block.
  - Fix: Change to `WHERE user_id = ?` + `.bind("user_default")` for uniformity (even in demo code). Low priority as it touches only test/demo data.
  - Severity: Warning (hygiene/consistency for a non-production path; no security or data-integrity impact).

### 🟢 Suggestion

This would make the W1 edge case explicit and regression-proof.

Per plan non-goals, `user_default` policy is intentional until platform usr_* mapping. No path traversal, command injection, or size DoS vectors observed (content is plain text stored in SQLite TEXT). Suggestion (optional): add length limits or tag-name charset whitelist in a future CLI hygiene pass if user-generated knowledge grows.
