### 🟡 Warning

- **W-001 (Double DB pool open in `run_assemble_moment`)** — `context.rs:540-541` opens two independent `SqlitePool` instances to the same `state.db`:
  - `open_narrative_gateway(config)` → `Schema::init(&db_path)` → `open_pool(max_connections=8)` + `run_migrations` + `seed_versions`
  - `open_kb_store(config)` → `Schema::init(&db_path)` → `open_pool(max_connections=8)` + `run_migrations` + `seed_versions`
  This creates up to 16 connections (2 × max_connections=8) for a single CLI command. Both `SqliteNarrativeGateway` and `SqliteKbStore` wrap their pool in `Arc` and are compatible with pool sharing. The daemon runtime correctly shares one pool across stores via `WorkspaceState`; the CLI path should follow the same pattern.
  **Fix**: Open pool once and pass (clone) to both constructors:
  ```rust
  let pool = crate::db::Schema::init(&db_path).await?;
  let narrative = SqliteNarrativeGateway::new(pool.clone());
  let kb = SqliteKbStore::new(pool);
  ```
  This is not a correctness bug (migrations are idempotent, two pools won't corrupt data), but it wastes SQLite connections, re-executes PRAGMAs, and re-runs idempotent migrations for no benefit.

### 🟢 Suggestion

- **S-001 (Consolidate DB init)**: `nexus_local_db::init_pool()` and `nexus42::db::Schema::init()` perform identical operations (open_pool + run_migrations + seed_versions). Consider using `init_pool` directly or noting in a comment why the CLI has its own thin wrapper. Minor code duplication — no functional impact.

- **S-002 (List without pagination)**: `reference_source::list()` and `narrative_gateway::list_worlds()` return full result sets without offset/limit. Acceptable for V1.26 given per-creator data volumes are expected to be small (<1000 rows), but consider adding pagination params in a future iteration if collections grow.
