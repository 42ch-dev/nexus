## Detailed Performance & Reliability Analysis

### Crate Split Architecture
The core change is splitting the monolithic `nexus-domain` and `nexus-sync` crates into domain-specific crates with cloud/local isolation:

- **File moves (97-100% similarity)**: `nexus-sync/*.rs` → `nexus-cloud-sync/*.rs` — pure renames, zero code change, zero performance impact.
- **Domain extractions (60-97% similarity)**: `nexus-domain/src/{pairing,user}.rs` → `nexus-cloud-domain/`, `nexus-domain/src/{memory,experience}*.rs` → `nexus-creator-memory/` — minimal code changes, primarily module boundary adjustments.
- **New crates**: `nexus-creator`, `nexus-kb`, `nexus-knowledge`, `nexus-narrative`, `nexus-moment-context-assembly` — thin wrappers with clean dependency graphs.

### Trait Objects & Dynamic Dispatch
**No new `dyn Trait`, `Box<dyn>`, or `Arc<dyn>` introduced.** The crate split did not add any vtable indirection. Zero performance regression from dynamic dispatch.

### Clones & Copies
Clones in the new code are confined to cold paths:
- `creator_id.clone()` in `nexus-cloud-domain/` — used in admin operations (revoke, suspend, mark_deleted), not hot paths.
- `self.system_prefix.clone()` in `nexus-creator-memory/` — pre-existing pattern in memory assembly, not new.

### Mutex / RwLock Usage
Only 2 new lock calls, both in `nexus-agent-host` state transitions:
```rust
*self.state.lock().await = AgentHostState::Running;
*self.state.lock().await = AgentHostState::Shutdown;
```
Simple assign-after-lock pattern — no lock-holding across awaits, no deadlock risk. Tokio `Mutex` with `.await` is the correct choice for async state transitions.
