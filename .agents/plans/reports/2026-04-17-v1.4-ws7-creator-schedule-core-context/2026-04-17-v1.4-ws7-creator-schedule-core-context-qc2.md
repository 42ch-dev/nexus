### M1: CoreContextManager.apply() — Read-Modify-Write Race on Version Bump

**Severity**: Medium  
**Files**: `crates/nexus-orchestration/src/schedule/derivation.rs:54-132`  
**Finding ID**: M1-CCM-RACE

**Description**: The `apply()` method performs a non-atomic read-modify-write sequence:

1. `SELECT current_core_context_version FROM creator_schedules` (line 64)
2. `INSERT INTO core_context_versions (... version = N+1 ...)` (line 93)
3. `UPDATE creator_schedules SET current_core_context_version = N+1` (line 113)

Under concurrent calls (e.g., two rapid HTTP PATCH requests to `/schedules/{id}/core-context`), both calls may read version `N`, both compute `N+1`, and both attempt to insert. The `PRIMARY KEY (schedule_id, version)` constraint on `core_context_versions` will cause one insert to fail with a unique constraint violation, producing a confusing `Database error` rather than a retry or conflict response.

**Impact**: Concurrent edits to the same schedule will spuriously fail with a database error. No data corruption (constraint protects), but poor UX and lost edits.

**Recommended Fix**: Wrap the read+insert+update in a `sqlx::Transaction`. Alternatively, use an atomic `UPDATE ... SET version = version + 1 RETURNING version` pattern, or implement optimistic concurrency with a retry loop.

**Cross-Reviewer Note**: QC1 should verify this against their concurrency checklist.

### M2: signal_schedule — Missing `Failed` Status in Transition Checks

**Severity**: Medium  
**Files**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs:402-560`  
**Finding ID**: M2-FSM-FAILED-MISSING

**Description**: The `signal_schedule` handler matches on current status strings for `start`, `pause`, `resume`, `cancel`, and `advance` signals. However, the `Failed` terminal status is not explicitly handled:

- `start`: allows only from `pending` → `running`; `failed` falls through to the catch-all conflict error
- `pause`: allows from `running|pending`; `failed` falls through correctly
- `resume`: allows from `paused`; `failed` falls through correctly
- `cancel`: allows from `pending|running|paused`; `failed` is correctly rejected
- `advance`: allows from `running`; `failed` falls through correctly

While the fall-through behavior happens to be correct (returns CONFLICT), the `Failed` status is not explicitly mentioned in any match arm, making the code harder to verify against the spec.

**Impact**: Correct behavior by accident; maintainability risk. A future refactor could inadvertently change the implicit rejection of signals on `Failed` schedules.

**Recommended Fix**: Add an explicit early check:

```rust
if matches!(current_status, ScheduleStatus::Failed) {
    return Err((StatusCode::CONFLICT, "schedule is in failed status; signals not allowed".into()));
}
```

Or enumerate all statuses in each signal arm.

### M3: signal_schedule — `start` Signal Does Not Allow `paused` → `running` Transition

**Severity**: Medium  
**Files**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs:431-442`  
**Finding ID**: M3-FSM-PAUSED-START

**Description**: The `start` signal only accepts `pending` status:

```rust
"start" => match current_status_str.as_str() {
    "pending" => "running",
    _ => { return Err(CONFLICT); }
},
```

A schedule in `paused` status must first be `resume`d (→ `pending`), then `start`ed (→ `running`), or wait for the supervisor `tick()` to auto-admit. This two-step flow is not documented in the handler comments and may confuse users.

**Impact**: User experience friction. A user who pauses a running schedule and later wants to restart it must issue two commands (`resume` then `start`) instead of one.

**Recommended Fix**: Either (a) allow `start` from `paused` directly to `running`, or (b) document the two-step requirement clearly in CLI help text and API docs. Per spec §5, `resume` → `pending` is by design (allows admission gate re-check), so option (b) is acceptable if documented.

### M4: edit_core_context — Status Check and Edit Are Non-Atomic

**Severity**: Medium  
**Files**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs:249-287`  
**Finding ID**: M4-EDIT-TOCTOU

**Description**: The `edit_core_context` handler first queries the schedule status (line 258), checks it is not terminal, then proceeds to apply the edit (line 274). Between the status check and the edit application, another request could transition the schedule to a terminal status (e.g., `cancel`). The edit would then succeed on a terminal schedule, violating the intended invariant.

**Impact**: TOCTOU race allowing edits to terminal schedules under concurrent load.

**Recommended Fix**: Move the terminal-status check into the `apply_user_edit` flow as a database-level constraint, or use a transaction that re-checks status before inserting the core_context version.

### M5: status_of — Masks Errors and Unknown Statuses as `Pending`

**Severity**: Medium  
**Files**: `crates/nexus-orchestration/src/schedule/supervisor.rs:280-300`  
**Finding ID**: M5-STATUS-MASKING

**Description**: `status_of()` returns `ScheduleStatus::Pending` for both database errors AND unknown status strings:

```rust
match row {
    Ok(Some((status_str,))) => match status_str.as_str() {
        // ... known statuses ...
        _ => ScheduleStatus::Pending,  // unknown status → Pending
    },
    _ => ScheduleStatus::Pending,      // DB error → Pending
}
```

**Impact**: A database corruption or bug that writes an invalid status string will be silently reported as `Pending`. A database error (e.g., connection lost) will also be reported as `Pending`, potentially causing the caller to make incorrect state transitions.

**Recommended Fix**: 
- For unknown status strings: log a warning and return `Failed` or propagate an error.
- For database errors: propagate the error (change return type to `Result<ScheduleStatus, SupervisorError>`).

### L1: Schedule ID Generation — Collision Risk Under Load

**Severity**: Low  
**Files**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs:53`  
**Finding ID**: L1-ID-COLLISION

**Description**: Schedule IDs are generated with millisecond precision:

```rust
let schedule_id = format!("SCH{}", chrono::Utc::now().format("%Y%m%d%H%M%S%3f"));
```

Two rapid requests within the same millisecond will produce the same ID, causing the second `INSERT` to fail with a primary key violation.

**Impact**: Low probability in human-driven usage, but automated tools or rapid CLI scripts could trigger collisions.

**Recommended Fix**: Append a random suffix (e.g., `format!("SCH{}{:04}", timestamp, rand::random::<u16>())`) or use a ULID library as noted in the type doc comment.

### L2: get_core_context — Fragile Error Detection by String Matching

**Severity**: Low  
**Files**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs:302-314`  
**Finding ID**: L2-STRING-MATCH

**Description**: Error handling for "not found" uses substring matching on the error display string:

```rust
if e.to_string().contains("not found") {
    return Err((StatusCode::NOT_FOUND, ...));
} else {
    return Err((StatusCode::INTERNAL_SERVER_ERROR, ...));
}
```

**Impact**: Fragile — if the error message wording changes, the handler returns 500 instead of 404.

**Recommended Fix**: Match on the `CoreContextError` enum variant directly:

```rust
match e {
    CoreContextError::NotFound(_) => (StatusCode::NOT_FOUND, ...),
    _ => (StatusCode::INTERNAL_SERVER_ERROR, ...),
}
```

### L3: `scheduled_at` Type Mismatch Between SQL Schema and Rust

**Severity**: Low  
**Files**: 
- `crates/nexus-local-db/migrations/20260419_creator_schedules.sql:15`
- `crates/nexus-contracts/src/local/schedule/mod.rs:89`  
**Finding ID**: L3-TYPE-MISMATCH

**Description**: The migration declares `scheduled_at INTEGER`, but the Rust `Schedule` struct uses `Option<String>`. While SQLite is dynamically typed and will accept strings in INTEGER columns, this inconsistency could cause confusion and sorting issues.

**Impact**: Potential sorting/query bugs if `scheduled_at` is used in range queries.

**Recommended Fix**: Align types — either change SQL to `TEXT` (if storing ISO strings) or change Rust to `Option<i64>` (if storing Unix timestamps). Given `created_at`/`updated_at` are `INTEGER` (timestamps), `scheduled_at` should probably also be `Option<i64>`.

### L4: supervisor.tick() — Loads All Schedules Without Creator Filtering

**Severity**: Low  
**Files**: `crates/nexus-orchestration/src/schedule/supervisor.rs:69-145`  
**Finding ID**: L4-TICK-SCALE

**Description**: `tick()` loads ALL schedules from the database:

```rust
let all_rows = sqlx::query_as::<_, ScheduleRow>(
    "SELECT ... FROM creator_schedules",
)
.fetch_all(pool)
.await?;
```

The spec describes a "per-creator queue", but the implementation mixes all creators' schedules in a single scan. As the schedule table grows, this will become a scalability bottleneck.

**Impact**: Performance degradation with many creators/schedules.

**Recommended Fix**: Add a `creator_id` parameter to `tick()` and filter the query: `WHERE creator_id = ?1`. Callers (e.g., boot recovery, HTTP handlers) should iterate over distinct creators.
