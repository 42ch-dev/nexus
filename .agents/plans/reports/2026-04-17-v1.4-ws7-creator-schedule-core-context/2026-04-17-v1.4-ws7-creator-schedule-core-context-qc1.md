## Critical Findings

### C1: ScheduleSupervisor `tick()` never loads `depends_on` — admission gate is a no-op for dependencies

**File**: `crates/nexus-orchestration/src/schedule/supervisor.rs` (line 74–80)
**Spec ref**: §5.1 — "Eligible Schedules: `status == Pending` AND all `depends_on` satisfied."

The `tick()` method loads schedules with a raw SELECT that does **not** join or subquery `schedule_dependencies`:

```rust
let all_rows = sqlx::query_as::<_, ScheduleRow>(
    "SELECT schedule_id, creator_id, preset_id, preset_version,
            status, concurrency_kind, concurrency_whitelist,
            current_core_context_version, current_session_id,
            scheduled_at, label, created_at, updated_at, terminated_at
     FROM creator_schedules",
)
```

The `ScheduleRow` struct has no `depends_on` field, and `to_schedule()` hardcodes:

```rust
depends_on: Vec::new(), // loaded separately via schedule_dependencies
```

**Impact**: `candidate.depends_on` is always empty in `admit()`, so the dependency gate is never enforced. Schedules with `--after` will start immediately regardless of predecessor state. The entire `schedule_dependencies` table is unused at runtime.

**Fix**: Load dependencies in `tick()` — either via `JOIN`, a batch subquery, or a second query. The `inspect_schedule` handler (line 219) already does the right thing with a separate SELECT; the supervisor needs the same.

### C2: CoreContext v0 version numbering off-by-one

**File**: `crates/nexus-orchestration/src/schedule/derivation.rs` (line 54–64)
**Spec ref**: §3.2 — "`CoreContextVersion(pub u32)` — strictly increasing per Schedule; **v0 is initial**"; §6.2 — "`Seed` — written at `schedule add` time. ... **Version = 0**."

The `apply()` method always computes `new_version = current_version.0 + 1`. Since the SQLite schema initializes `current_core_context_version` to `0` (migration line 13), the first `Seed` call produces version **1**, not version **0**:

```rust
let current_version = self.current_version(schedule_id).await?; // returns 0
let new_version = CoreContextVersion(current_version.0 + 1);    // → 1, not 0
```

The test at line 576 confirms: `assert_eq!(record0.version, CoreContextVersion(1));` — the test asserts the wrong value.

**Impact**: The seed version is numbered 1 instead of 0, breaking the spec's versioning contract. Any downstream code that relies on "v0 = seed" will be off-by-one.

**Fix**: Handle the seed case specially — if `current_version.0 == 0` and step is `Seed`, write version 0. Or restructure so that `current_core_context_version` in the schedule row represents "next version to write" rather than "last written version".

### C3: `DerivationStep::PresetHook` serialization/deserialization mismatch

**File**: `crates/nexus-orchestration/src/schedule/derivation.rs` (lines 191–214, 375–388)

`apply_preset_hook` stores a **custom JSON blob** in `derivation_detail`:

```rust
let detail_json = serde_json::json!({
    "state_id": state_id,
    "hook_name": hook_name,
    "op": op,  // includes the EditOp enum
});
```

But `reconstruct_derivation` for `"preset_hook"` tries to deserialize this as a **`DerivationStep`** via serde, which expects the tagged enum shape `{"kind": "preset_hook", "state_id": ..., "hook_name": ...}` — **not** the custom JSON with `op`.

The code then falls back to manual extraction (`json["state_id"].as_str()`), which works by accident because the custom JSON has those keys, but:

1. The `"op"` field is silently lost — if you ever need to know *which* EditOp was used for a PresetHook, it's not recoverable via the type system.
2. The `"kind"` field is absent from the custom JSON, so `serde_json::from_slice::<DerivationStep>` would fail (it's not called, but the inconsistency is a maintenance risk).
3. For `"seed"`, `"user_edit"`, `"preset_seed_expansion"`, and `"llm_summarize"`, the code **does** call `serde_json::from_slice` directly — so there are **two different serialization conventions** in the same module.

**Impact**: Fragile coupling between write and read paths. Adding or changing fields on any `DerivationStep` variant risks breaking the `reconstruct_derivation` match arm for `PresetHook` silently (it uses `.unwrap_or("")`).

**Fix**: Use a single serialization strategy. Either:
- (a) Serialize the full `DerivationStep` via serde for all variants (including `PresetHook`), or
- (b) Use custom JSON for all variants consistently.

## High Findings

### H1: `ScheduleRow.to_schedule()` hardcodes `depends_on: Vec::new()`

**File**: `crates/nexus-orchestration/src/schedule/supervisor.rs` (line 399)

Related to C1 — even if `tick()` loaded dependencies separately, `to_schedule()` silently drops them. The `Inner` cache in `Inner.running_ids` is also never populated during `tick()` (the cache is updated only in `on_schedule_terminal` and the local `running_ids_so_far` variable). This means the supervisor's cache is stale after a `tick()` call that starts schedules.

### H2: `signal_schedule` resume always goes to `pending`, not `running` for mid-exec

**File**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs` (line 454–466)
**Spec ref**: §4 state machine — `Paused → Running [label="resume (user; if mid-exec)"]`; `Paused → Pending [label="resume (user; if pre-exec)"]`

The handler cannot distinguish pre-exec vs mid-execution Paused states. The spec says mid-execution resume should go to `Running`, but the code always goes to `Pending`. This means a paused mid-execution Schedule would need an additional manual `start` signal to resume, which is a UX regression.

---

### H3: `preset_version` hardcoded to `1` in `add_schedule` handler

**File**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs` (line 79)

```rust
preset_version: 1,
```

The `AddScheduleRequest` type doesn't include a `preset_version` field, and the handler doesn't look up the actual preset version from the loaded preset. If presets are ever versioned beyond 1, all schedules will be recorded with the wrong version snapshot, breaking the "snapshot of the preset at add-time" guarantee (spec §3.1).

## Medium Findings

### M1: HTTP `EditCoreContextRequest.op` is `String` — no compile-time validation

**File**: `crates/nexus-contracts/src/local/schedule/http.rs` (line 98)

The `op` field is a raw `String`, parsed at runtime in `parse_edit_op`. If a new `EditOp` variant is added to the domain type, the HTTP handler won't get a compile-time reminder to handle it. Using the domain `EditOp` type directly with proper serde would be safer.

### M2: `resume_running_as_paused` name contradicts its behavior

**File**: `crates/nexus-orchestration/src/schedule/supervisor.rs` (line 313)

The method is named `resume_running_as_paused` but it **pauses** Running schedules. "Resume" is misleading — it should be named `pause_running_on_boot` or `recover_running_as_paused`.

### M3: `status_of` silently returns `Pending` for non-existent schedules

**File**: `crates/nexus-orchestration/src/schedule/supervisor.rs` (line 298)

```rust
_ => ScheduleStatus::Pending,
```

If a schedule ID doesn't exist, `status_of` returns `Pending` instead of an error. Callers (like `edit_core_context` and `delete_schedule` handlers) rely on this for validation, but it conflates "doesn't exist" with "exists but is pending".

### M4: No per-creator scoping in supervisor admission

**File**: `crates/nexus-orchestration/src/schedule/admission.rs` + `supervisor.rs`

The spec §5 says "The Schedule supervisor maintains a **per-creator** queue." But `tick()` loads **all** schedules across all creators and runs admission globally. `RunningSet` and `CompletedSet` don't track creator IDs. A Serial schedule for creator A would block a Serial schedule for creator B.

### M5: `json_merge` is shallow-recursive but doesn't handle arrays

**File**: `crates/nexus-orchestration/src/schedule/derivation.rs` (line 490–505)

The `json_merge` function recursively merges objects but falls through to `patch.clone()` for non-object pairs. If a struct payload contains arrays, `StructMerge` will replace the entire array rather than appending or merging elements. This may be acceptable for V1.4 but should be documented.

## Architecture Assessment

### Strengths

1. **Clean crate boundaries**: Types in `nexus-contracts::local::schedule/`, orchestration logic in `nexus-orchestration::schedule/`, HTTP handlers in `nexus42d`. No cross-layer leaks.
2. **Pure-function admission gate**: `admit()` is side-effect-free, making it trivially testable. Good separation of concerns.
3. **Forward compatibility**: `DerivationStep::LlmSummarize` is reserved with `#[non_exhaustive]` and a public constructor `llm_summarize()`. Zero-migration path for V1.5.
4. **Boot recovery**: `resume_running_as_paused` correctly handles daemon restart by pausing stale Running schedules.
5. **Preset validation**: The loader correctly rejects `Replace`/`StructRemove` in `context_update` hooks at parse time.

### Integration Risks

1. **Supervisor ↔ Engine**: The supervisor doesn't yet create sessions or signal the engine to start executing admitted schedules. The `start` signal only flips the DB row to `running` but doesn't wire to `OrchestrationEngine`. This is expected for a phased implementation but should be tracked.
2. **CoreContext ↔ Template engine**: The `{{core_context.*}}` Handlebars binding is mentioned in the spec (§6.4) but not implemented in this change set. The derivation module produces the data, but the template rendering integration is deferred.
3. **Preset loader ↔ Derivation**: `CoreContextManager.apply_preset_hook` takes an `EditOp` but the preset loader's `ContextUpdateOp` is a different type. There's no conversion function between them in this change set.

## Residual Recommendations (non-blocking if above fixed)
