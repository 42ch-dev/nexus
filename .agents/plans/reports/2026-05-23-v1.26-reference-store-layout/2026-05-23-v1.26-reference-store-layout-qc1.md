### 🟡 Warning

#### W-1: Orphan body file risk on DB insert failure (`register` lacks cross-system transaction)
- **Location**: `crates/nexus-local-db/src/reference_source.rs:register`
- **Issue**: The `register` function writes `body.md` to disk *before* inserting the registry row into SQLite. If the DB insert fails after the file is written, an orphan `body.md` remains on disk with no registry entry pointing to it. There is no cleanup or rollback.
- **Impact**: Low for pre-1.0 (local persistence may be wiped per compass §9), but this will become a data-consistency concern after 1.0 when migration guarantees are required.
- **Fix**: Add a `tokio::fs::remove_file(&body_abs)` in an error path after the DB insert fails, or document this as accepted risk with a `// TODO(v1.0): add cleanup on DB failure` comment.
- **Confidence**: High

#### W-2: `list_references` returns all rows without workspace filtering
- **Location**: `crates/nexus-local-db/src/reference_source.rs:list`
- **Issue**: The `list` query does not accept a `workspace_id` filter, returning all reference sources across all workspaces. The `ReferenceSourceRow` struct carries a `workspace_id` field, and the CLI constructs a workspace-specific ID (`wrk_{slug}`), but listing is global. This is inconsistent with the spec's statement that "registration, scan state, and listing are workspace-local operations."
- **Impact**: Low for V1.26 (single-workspace local daemon is the common case), but will cause incorrect cross-workspace leakage when multi-workspace support arrives.
- **Fix**: Add an optional `workspace_id` parameter to `list` and filter in the SQL query. Update daemon handler and CLI to pass the active workspace.
- **Confidence**: Medium

### 🟢 Suggestion

#### S-1: Extract duplicated row-mapping boilerplate in `reference_source.rs`
- **Location**: `crates/nexus-local-db/src/reference_source.rs` (lines ~155-175, ~250-270, ~300-320)
- **Issue**: The anonymous closure mapping `sqlx` query results to `ReferenceSourceRow` is repeated three times (register, list, get_by_id). This is ~20 lines of identical boilerplate each.
- **Impact**: Minor maintainability tax — adding a new column requires editing three sites.
- **Fix**: Extract a private `fn row_into_reference_source_row(r: QueryResult) -> ReferenceSourceRow` or implement `From<QueryResult>` for `ReferenceSourceRow` if `sqlx` query result types unify.
- **Confidence**: High

#### S-2: `ReferenceSourceRow` used directly as API DTO in daemon handler
- **Location**: `crates/nexus-daemon-runtime/src/api/handlers/references.rs`
- **Issue**: The handler implements `From<nexus_local_db::ReferenceSourceRow> for ReferenceInfo`. This is clean for V1.26, but it couples the HTTP API response shape to the local-db repository row shape. If the DB adds internal columns (e.g. `scan_attempts`, `last_scan_error`), they will implicitly become available in the API unless explicitly excluded.
- **Impact**: Low now; becomes a boundary-drift risk as the schema evolves.
- **Fix**: Keep the `From` impl but add a compile-time assertion or note that `ReferenceInfo` is the explicit API contract; any new DB field must be opt-in in the `From` impl.
- **Confidence**: Medium

#### S-3: Content hash is always NULL for new rows
- **Location**: `crates/nexus-local-db/src/reference_source.rs:register` (INSERT sets `content_hash` to `NULL`)
- **Issue**: The body is written to disk but no hash is computed. The spec says `content_hash` is "Hash of the canonical `body.md` content when available." This is fine for V1.26 but leaves a partially implemented contract.
- **Impact**: None for V1.26; future refresh pipelines will need to backfill or compute hashes.
- **Fix**: Add a `// TODO: compute SHA-256 of body and store in content_hash` comment, or compute a simple hash now.
- **Confidence**: Low

### Rationale

The V1.26 reference store layout implementation correctly separates the two-layer model (SQLite registry + MD body on disk) across crate boundaries:

- **`nexus-home-layout`** owns path helpers (`creator_references_root`, `reference_unit_dir`, `reference_body_path`) with defense-in-depth path-traversal assertions for both `creator_id` and `reference_id`. Tests cover layout correctness and rejection of traversal attempts.
- **`nexus-local-db`** owns the repository layer (`register`, `list`, `get_by_id`), the `SourceMutability` domain enum, and the new `LocalDbError::Io` variant. The migration adds `content_path` and `source_mutability` with a sensible `DEFAULT 'static'`. The pre-1.0 wipe policy (nulling legacy `content`) is correctly applied.
- **`nexus-daemon-runtime`** exposes `GET /v1/local/references` and `GET /v1/local/references/{id}` through clean handler functions that delegate to the repository. The `ReferenceInfo` DTO is explicitly separated from the DB row via a `From` impl.
- **`nexus42`** provides `creator reference register/list/show` CLI commands with appropriate input validation (`source_type` enum check, `--file`/`--body` mutual exclusion, stdin support via `-`).

Build health is green: `cargo clippy --all -- -D warnings` passes with zero warnings, and `cargo test --workspace` passes all suites (591+ tests across affected crates).

The two Warnings identified (orphan file risk, missing workspace filter) are acknowledged architectural gaps that do not block V1.26 acceptance but should be tracked for V1.0 hardened persistence.
