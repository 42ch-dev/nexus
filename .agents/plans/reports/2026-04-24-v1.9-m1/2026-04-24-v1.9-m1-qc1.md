## Findings

### 🔴 Critical

**F-001: `user_preset_base_dir` duplicated across two crates with divergent implementations**

- **Location**: `crates/nexus-home-layout/src/lib.rs` vs `crates/nexus-orchestration/src/user_preset_dir.rs`
- **Issue**: `user_preset_base_dir` exists in both crates. The `nexus-home-layout` version uses `nexus_root_from_home(home).join("presets")` (canonical path resolution through the shared layout crate), while `nexus-orchestration/src/user_preset_dir.rs` uses `nexus_home.join("presets")` directly. This creates two sources of truth for the same path convention.
- **Impact**: If `nexus_root_from_home` ever performs additional path normalization (e.g., resolving `~`, validating the `.nexus42` suffix), the orchestration copy will silently diverge. The ADR-014 canonical paths policy (documented in `crates/nexus-home-layout/AGENTS.md`) mandates no hardcoded paths outside the layout crate.
- **Fix**: `user_preset_dir.rs` should import `user_preset_base_dir` from `nexus_home_layout` instead of defining its own. The same applies to `user_preset_bundle_dir` which is also duplicated. Note: `crates/nexus42/src/commands/preset.rs` already correctly imports from `nexus_home_layout`, but `user_preset_dir.rs` does not.

**F-002: `scan_user_presets_local` duplicates scanning logic from `nexus-orchestration::user_preset_dir`**

- **Location**: `crates/nexus42/src/commands/preset.rs` (lines 2210-2255)
- **Issue**: The CLI defines its own `LocalUserScan` struct and `scan_user_presets_local` function that re-implements directory scanning logic (filter `_`/`.` prefixes, check `preset.yaml` existence) already present in `nexus-orchestration::user_preset_dir::scan_user_presets`. The justification given ("fast listing, no dependency on the daemon") doesn't hold — the orchestration crate is already a dependency of `nexus42` (used for `resolve_preset`), and the CLI could call the orchestration scanner directly for listing, or share a common scan helper.
- **Impact**: Two implementations of the same filtering rules. If the filtering policy changes (e.g., adding a new reserved prefix), both locations must be updated. This violates the "single truth source" principle established in the repo's AGENTS.md.
- **Fix**: Either (a) expose a lightweight `list_user_preset_ids(nexus_home)` function from `nexus-orchestration::user_preset_dir` that does directory-only scanning without YAML parsing, or (b) reuse `scan_user_presets` and extract just the IDs.

**F-003: `resolve_preset` error matching relies on string search inside `PresetLoadError::Validation`**

- **Location**: `crates/nexus-orchestration/src/preset/mod.rs` lines 1344-1357
- **Issue**: The `resolve_preset` function distinguishes "not found" from other embedded preset errors by checking `problems.iter().any(|p| p.error.contains("not found"))`. This is a string-matching heuristic on the error message, not a typed enum variant. `load_embedded_preset` produces this "not found" string internally in its validation logic. Any future refactoring of the error message (e.g., i18n, rewording) will silently break the fallback logic.
- **Impact**: If `load_embedded_preset` changes its error message format, `resolve_preset` will return the wrong error to callers — masking "not found" as whatever the original embedded error was.
- **Fix**: Either (a) add a `NotFound` variant to `PresetLoadError`, or (b) have `load_embedded_preset` return `Result<Option<LoadedPreset>, PresetLoadError>` so callers can distinguish "not found" from "parse/validation error" without string matching.

### 🟡 Warning

**F-004: `start_handles` mutex contention silently drops handles**

- **Location**: `crates/nexus42d/src/lifecycle/actions.rs` lines 149-160
- **Issue**: `enter_starting` uses `try_lock()` to store `JoinHandle`s, and if the mutex is contested, it logs a warning but does NOT store the handle. This means `exit_starting` will not abort that subsystem's start task. In a race condition during rapid subsystem spawning, some tasks could continue running after the FSM has exited `Starting`.
- **Impact**: In-flight subsystem starts that were not tracked will not be cancelled on shutdown during Starting. The comment says the state transition makes their events "irrelevant" — but if they later dispatch `SubsystemUp` or `SubsystemFailed` events, the FSM may process them in the wrong state.
- **Fix**: Use `blocking_lock()` instead of `try_lock()` for handle storage (this is startup, not a hot path), or use a `Mutex` from `std::sync` instead of `tokio::sync::Mutex` since `enter_starting` is not async.

**F-005: `exit_starting` uses `blocking_lock()` in a non-async context — appropriate but worth noting potential contention**

- **Location**: `crates/nexus42d/src/lifecycle/actions.rs` line 166
- **Issue**: `exit_starting` calls `ctx.start_handles.blocking_lock()` which blocks the current thread. Since `exit_starting` is a synchronous function called from the FSM state machine, this is technically correct, but if any `enter_starting` task is still holding the lock, this will block the FSM transition. This is a minor liveness concern.
- **Mitigation**: Low probability in practice because `enter_starting` finishes quickly (it only spawns tasks and stores handles).
- **Severity**: Warning for documentation, not a blocker.

**F-006: Schema drift detection test uses `unwrap()` in workspace_root() — violates crate-level clippy rule**

- **Location**: `crates/nexus-contracts/tests/schema_drift_detection.rs` line 523
- **Issue**: The test file uses `dir.parent().unwrap().parent().unwrap()` to traverse to workspace root. While the test crate has `#[allow(clippy::unwrap_used)]` implicitly (tests are often exempted), the root `Cargo.toml` workspace denies `unwrap_used` in some crates. This is acceptable for integration tests but should be noted.
- **Impact**: If workspace structure changes (e.g., `nexus-contracts` moves deeper), this will panic at runtime during the test rather than fail gracefully.
- **Fix**: Add a comment or use `expect()` with a clear message.

**F-007: Schema drift test has hardcoded `ALL_SCHEMA_PATHS` list that must be manually maintained**

- **Location**: `crates/nexus-contracts/tests/schema_drift_detection.rs` lines 460-515
- **Issue**: The `ALL_SCHEMA_PATHS` constant (52 entries) is a separate hardcoded list from the `build_schema_map()` registry. If a schema file is added to the registry but forgotten from `ALL_SCHEMA_PATHS`, its `$ref` resolution will fail silently (returning "dummy" fallback values). Conversely, if a schema is removed from the filesystem but not from `ALL_SCHEMA_PATHS`, the test will fail.
- **Impact**: Maintenance burden. The two lists should ideally be derived from a single source or the cache should be built directly from `build_schema_map()`.
- **Fix**: Build the cache from `build_schema_map().iter().map(|e| e.schema_path)` instead of maintaining a separate list, or add a compile-time assertion that every entry's schema path is in the cache.

**F-008: `list_presets` CLI command depends on daemon being running**

- **Location**: `crates/nexus42/src/commands/preset.rs` lines 2133-2138
- **Issue**: `list_presets` calls `DaemonClient::get("/v1/local/orchestration/presets")` to get embedded presets. This means `nexus42 preset list` will fail if the daemon is not running. For a CLI command that also lists user presets (filesystem-based), this creates an asymmetric dependency.
- **Impact**: Users cannot run `preset list` offline or before daemon boot to check their presets.
- **Fix**: Consider reading embedded presets from the binary directly (via the orchestration crate's `list_embedded_preset_ids` function) rather than requiring a daemon round-trip. This would also make the command faster.

### 🟢 Suggestion

**F-009: `user_preset_dir.rs` re-exports `load_preset_from_str` from internal `loader` module**

- **Location**: `crates/nexus-orchestration/src/user_preset_dir.rs` line 1441
- **Issue**: The user preset scanner directly calls `crate::preset::loader::{load_preset_from_str, LoadedPreset, PresetLoadError}` rather than going through a public API. While this is internal crate code, it creates tighter coupling between modules. If `loader` internals change, `user_preset_dir` must change too.
- **Impact**: Minor — both are within the same crate.
- **Suggestion**: Consider exposing a `PresetLoader` trait or function in `preset/mod.rs` that abstracts file loading, so scanners (user, system, embedded) all go through the same interface.

**F-010: Schema drift detection does not validate `schema_version` field**

- **Location**: `crates/nexus-contracts/tests/schema_drift_detection.rs`
- **Issue**: The drift detection compares field names and types but does not validate that `schema_version` in each schema matches `LATEST_SCHEMA_VERSION`. This is a separate concern handled elsewhere, but the drift test is a natural place to catch version bumps.
- **Suggestion**: Consider adding a `schema_version` check as part of the drift detection pipeline.

**F-011: `PRESET_INIT_TEMPLATE` in CLI is a string with `{name}` placeholder — fragile templating**

- **Location**: `crates/nexus42/src/commands/preset.rs` lines 2017-2040
- **Issue**: The template uses simple `String::replace("{name}", name)`. If the YAML content ever legitimately contains `{name}` (e.g., in a prompt template), it would be unintentionally substituted.
- **Suggestion**: Use a more robust templating mechanism (e.g., `{{name}}` with a proper template engine, or construct the YAML programmatically via `serde_yaml`).

**F-012: `HealthDegraded` handling in Starting only checks `up_subsystems` set — doesn't record if transitioning to Degraded loses starting context**

- **Location**: `crates/nexus42d/src/lifecycle/state.rs` lines 108-123
- **Issue**: When `HealthDegraded` fires during Starting for an already-up subsystem, the FSM transitions directly to `Degraded`. This means the remaining subsystems that haven't reported `SubsystemUp` yet will still complete their starts, but the FSM is no longer in Starting to aggregate their completion. If a subsystem fails during this window, the FSM may be in an inconsistent state.
- **Suggestion**: Consider whether the Degraded state should track which subsystems are still booting and handle their eventual outcomes appropriately.

### Rationale

Three critical findings prevent Approve:

1. **F-001** (duplicated path functions) violates the repo's established ADR-014 policy of using `nexus-home-layout` as the single source of truth for `~/.nexus42/` path conventions. The duplication creates a concrete risk of future divergence.

2. **F-002** (duplicated scan logic) is the same class of issue — two implementations of the same filtering rules across crate boundaries, violating the "single truth source" principle documented in AGENTS.md.

3. **F-003** (string-matching error handling) introduces a fragility in the composable preset resolution API. The `resolve_preset` function's fallback logic depends on matching error message text, which is not a stable contract.

These are not subjective style issues — they are architectural violations of patterns already established in this codebase (single source of truth for paths, typed error handling). The changes should be addressed before merge to maintain the codebase's long-term maintainability.

The WS-C (lifecycle edges) and WS-D (schema drift detection) workstreams are well-structured with good test coverage. The critical findings are concentrated in the WS-A (preset loading) workstream, specifically around code reuse and module boundary discipline.
