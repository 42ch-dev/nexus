### 🟡 Warning

#### W-1: `skill_sync.rs` — O(n) full-file reads on every sync, no checksum caching
- **Source**: `crates/nexus-orchestration/src/skill_sync.rs:97-153`
- **Issue**: `sync_embedded_skills` reads the **entire byte content** of every on-disk skill file for comparison on every call. For `n` skills, this performs `n` full file reads + byte-wise equality checks.
- **Impact**: Currently low (only 1 skill, ~8KB total), but as the embedded skills catalog grows, this becomes a boot-time tax on every `nexus42 init` and session start. A 100KB skill file read on every init is unnecessary I/O.
- **Trigger**: Any call to `sync_embedded_skills`, which is wired into `init.rs`.
- **Fix**: Store a lightweight checksum (e.g., SHA-256 or blake3) in a sidecar file or metadata and compare hashes instead of full contents. Alternatively, use file modification time (`mtime`) as a fast-path skip before byte comparison.
- **Cross-Reviewer Ready Notes**: This is a degradation path — it won't break correctness but will slow down init times as skills grow. Expected runtime impact: linear growth in I/O per init.

#### W-2: `skill_link.rs` — TOCTOU race condition in symlink recreation
- **Source**: `crates/nexus-orchestration/src/skill_link.rs:56-73`
- **Issue**: The sequence `link_path.exists() || link_path.is_symlink()` → `read_link()` → `remove_file()`/`remove_dir_all()` is **not atomic**. Between the existence check and removal, another process (or a concurrent `nexus42` invocation) could modify the path, leading to:
  - Stale symlink left behind if `remove_file` races with another creator.
  - Unexpected error if the path is replaced with a directory between `read_link` and `remove_file`.
- **Impact**: Low probability in single-user CLI context, but real. `ensure_role_skills` is called per preset role, and parallel session starts could hit this.
- **Fix**: Use atomic symlink replacement where possible (Unix: `symlink` + `rename` pattern), or wrap the check-remove-create in a retry loop with exponential backoff.
- **Cross-Reviewer Ready Notes**: Reliability risk for concurrent workspace usage. Rollback is not directly applicable — symlink state is just inconsistent until next init.

#### W-3: `sync_module.rs` — Unbounded memory for large story collections
- **Source**: `crates/nexus-orchestration/src/sync_module.rs:126-169`
- **Issue**: `build_story_bundle` reads **every chapter file entirely into memory** as a `String`, computes SHA-256 on the full content, and stores all of it in a `Vec<ChapterContent>` within `StoryBundle`. For a novel with 100 chapters × 50KB each = 5MB per bundle, held entirely in RAM with no streaming or pagination.
- **Impact**: Acceptable for small projects, but does not scale. No upper bound on file size or chapter count. A maliciously large `.md` file could cause memory pressure.
- **Fix**: 
  1. Add a size limit (e.g., reject or truncate files > 1 MiB).
  2. Stream-read for SHA-256 computation instead of `read_to_string` + `digest(bytes)`.
  3. Consider lazy content loading — store paths and hashes, read content only when needed for sync transport.
- **Cross-Reviewer Ready Notes**: This is the primary hot path for story sync. Memory growth is unbounded. Observable via OOM or high RSS on large workspaces.

### 🟢 Suggestion

#### S-1: `embedded_skills.rs` — `get_embedded_skill` is O(n) with manifest re-parse
- **Source**: `crates/nexus-orchestration/src/embedded_skills.rs:129-131`
- **Issue**: `get_embedded_skill(id)` calls `list_embedded_skills()`, which re-parses the embedded JSON manifest and re-reads all skill files from the `include_dir::Dir` on every lookup, then does a linear search.
- **Impact**: Currently negligible (1 skill), but as the catalog grows, repeated lookups during session setup will waste CPU.
- **Fix**: Cache the parsed `Vec<EmbeddedSkillEntry>` in a `OnceLock` or `lazy_static`, or build a `HashMap<String, EmbeddedSkillEntry>` for O(1) lookup.

#### S-2: `init.rs` — `sync_embedded_skills` runs unconditionally on every init
- **Source**: `crates/nexus42/src/commands/init.rs:234-253`
- **Issue**: Embedded skill sync is executed on **every** `nexus42 init`, even if the user has not changed versions. This is a conservative but potentially wasteful boot step.
- **Impact**: Adds I/O and compute to a frequent CLI operation.
- **Fix**: Add a fast-path: skip sync if `~/.nexus42/skills/` exists and a `.synced_version` marker matches the CLI binary version. Only run full sync when the binary version changes.
