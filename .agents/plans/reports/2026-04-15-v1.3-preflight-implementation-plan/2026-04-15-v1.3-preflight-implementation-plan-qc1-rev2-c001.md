### Correctness Analysis

| Aspect | Before (buggy) | After (fixed) | Verdict |
|--------|---------------|---------------|---------|
| HashMap key type | `&'static str` (leaked) | `String` (owned) | ✅ Correct |
| Memory ownership | `Box::leak` creates `'static` reference, permanently leaks heap memory | `String` owned by HashMap, freed on drop | ✅ Correct |
| Token lowercasing | `tok.to_lowercase()` → `into_boxed_str()` → `Box::leak()` | `tok.to_lowercase()` directly | ✅ Correct — `to_lowercase()` already returns `String` |
| Semantic behavior | Deduplicates by lowercased token value | Identical deduplication logic | ✅ No behavioral change |
| API surface | `quality_signal(input: &str) -> QualitySignal` unchanged | Same signature | ✅ No breaking change |

**Root cause of original bug:** The previous code used `Box::leak` to convert a `String` into a `&'static str` so it could be stored as a key in `HashMap<&str, usize>`. `Box::leak` intentionally never frees the allocation, causing a permanent memory leak proportional to the number of unique tokens processed.

**Why the fix is correct:** `HashMap<String, usize>` owns its keys. The `String` returned by `to_lowercase()` is inserted directly into the map and will be dropped when the `seen` HashMap goes out of scope (end of `quality_signal()`). No leak, no `'static` lifetime requirement, identical deduplication semantics.
