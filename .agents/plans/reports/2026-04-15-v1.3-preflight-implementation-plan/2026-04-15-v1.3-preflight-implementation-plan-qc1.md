## Findings

### Critical (must fix before merge)

#### C-001: Unbounded memory leak in `quality_signal()` via `Box::leak`

**File:** `crates/nexus-domain/src/review_quality.rs`, line 43
**Severity:** Critical
**Finding:** The `quality_signal()` function uses `Box::leak(lower.into_boxed_str())` to intern lowercased token strings as `&'static str` keys in the `HashMap`. This intentionally leaks heap memory — every unique token across every call permanently allocates memory that is never freed.

```rust
// Line 43 — each unique token leaks its allocation
let entry = seen.entry(Box::leak(lower.into_boxed_str())).or_insert(0);
```

In a daemon processing reviews continuously, this creates unbounded memory growth. Even in CLI-only usage, processing large digests with many unique tokens accumulates leaked memory per invocation. This is especially problematic because the plan goal explicitly includes "memory review/fragments CLI-to-daemon flow" (T2) — the daemon will call this function repeatedly.

**Impact:** Memory exhaustion in long-running daemon processes. Violates the plan's "strengthen local runtime quality baseline" goal by introducing a reliability regression.

**Fix recommendation:** Use a `HashMap<String, usize>` (owned keys) instead. The cost is one extra allocation per unique token, but tokens are short-lived and the HashMap is dropped after the function returns:

```rust
let mut seen: HashMap<String, usize> = HashMap::new();
for tok in &tokens {
    let lower = tok.to_lowercase();
    *seen.entry(lower).or_insert(0) += 1;
    ...
}
```

Alternatively, use `HashMap<&str, usize>` with the original `&str` slices (no lowercase dedup) or use a `BTreeMap` / hashbrown with string interning if performance is critical.

### Warnings (should fix before merge)

#### W-001: Quality signal thresholds lack calibration evidence

**File:** `crates/nexus-domain/src/review_quality.rs`, lines 76-78
**Severity:** Warning
**Finding:** The thresholds in `is_high_signal()` (unique_ratio ≥ 0.35, alpha_ratio ≥ 0.65, repeated_token_ratio ≤ 0.45) appear arbitrary. The docstring says "Thresholds tuned to reject repetitive noise" but there's no evidence of tuning — no benchmark data, no corpus analysis, no boundary tests around the threshold values.

**Impact:** Classification behavior may produce surprising false positives/negatives on real creative content. The plan says "promote at this length" for creative tasks — without calibrated thresholds, content just below 80 chars with high signal is demoted to fragment, potentially losing valuable short-form creative content.

**Recommendation:** Add at least one boundary-condition test per threshold (e.g., content at exactly 0.35 unique_ratio) and document the rationale or calibration method. Consider making thresholds configurable via a `QualityThresholds` struct with `Default` impl for future tuning without code changes.

#### W-002: `ReviewResponse` and `FragmentRow` DTOs lack validation

**File:** `crates/nexus42/src/api/models.rs`, lines 40-57
**Severity:** Warning
**Finding:** The new DTOs (`ReviewResponse`, `FragmentRow`) have no field validation or `#[serde(default)]` guards. If the daemon returns a response missing `promoted`/`fragmented`/`dropped` fields, deserialization fails entirely rather than providing a sensible default.

**Impact:** A daemon version mismatch or transient API bug could cause the CLI `memory review` command to crash instead of showing a graceful error.

**Recommendation:** Consider `#[serde(default)]` on count fields, or at minimum document the expected wire contract and add an integration test against the actual daemon endpoint shape.

#### W-003: License change MIT → GPL-3.0-only is out of scope

**Files:** `Cargo.toml`, `LICENSE`, `README.md`, `package.json`, `packages/nexus-contracts/package.json`, `tooling/codegen/package.json`, `tooling/validation/package.json`

This change affects downstream consumers, published packages (`@42ch/nexus-contracts` on npm, `nexus-contracts` on crates.io), and any projects that depend on the MIT-licensed version.

#### W-004: `fragments` command table formatting may truncate long IDs

**File:** `crates/nexus42/src/commands/memory.rs`, lines 236-243
**Severity:** Warning
**Finding:** The fragments output uses a hardcoded 30-character width for `fragment_id`:

```rust
println!("{:<30} SUMMARY", "FRAGMENT_ID");
// ...
println!("{:<30} {}", f.fragment_id, f.summary);
```

If fragment IDs exceed 30 characters (likely as IDs grow with UUIDs or hashes), the output will misalign with no indication of truncation.

**Recommendation:** Use a dynamic width or switch to a table library (e.g., `comfy-table`) for future-proofing. At minimum, add an ellipsis for truncated IDs.

### Suggestions (may address later)

#### S-001: `QualitySignal` should derive `Default`

**File:** `crates/nexus-domain/src/review_quality.rs`, line 10
**Finding:** `QualitySignal` derives `Debug, Clone` but not `Default`. A `Default` impl returning all-zero signals would be ergonomic for initialization and testing.

#### S-002: Daemon client methods could share error-handling logic

**File:** `crates/nexus42/src/api/daemon_client.rs`, lines 228-288
**Finding:** Both `review_pending_memories()` and `list_memory_fragments()` duplicate the same `is_connect()` → `daemon_not_reachable` error mapping pattern. This is already partially abstracted in the existing `get()` and `post()` methods but the new methods don't reuse them.

**Recommendation:** Consider a private helper method or reuse the existing generic `get<T>()` / `post<T, B>()` methods where applicable. The `list_memory_fragments()` could use `self.get(path).await` directly.

#### S-003: Integration tests lack cleanup guarantees

**File:** `crates/nexus42d/tests/integration.rs`, lines 656-658, 719-721
**Finding:** The ACP tool tests use `std::fs::remove_file(&test_file).ok()` for cleanup, which silently ignores errors. If a test panics before reaching cleanup, temp files accumulate in the workspace directory.

**Recommendation:** Use `tempfile::NamedTempFile` or wrap in a `Drop` guard for guaranteed cleanup, even on panic.

#### S-004: `classify_pending_review()` new reason strings should be stable

**File:** `crates/nexus-domain/src/review.rs`, lines 171-186
**Finding:** The reason strings changed from `"{} task with substantial digest ({} chars)"` to `"{} task with high-signal digest ({} chars)"` and `"{} task with medium digest"` to `"{} task with medium or low-signal digest"`. Any downstream parsing of reason strings will break.

**Impact:** Low — reason strings are primarily for human audit, but if any code pattern-matches on them, this is a silent break.

**Recommendation:** Document that `reason` is for human consumption only, not programmatic parsing.
