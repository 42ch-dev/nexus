## Warnings

### W-01: `read_versions` discards the actual malformed value in error path

**File**: `crates/nexus-local-db/src/lib.rs:169,185`
**Severity**: Warning
**Category**: Correctness / Debuggability

Both `parse::<u32>()` error branches hardcode `value: "".to_string()` instead of preserving the actual database value that failed to parse:

```rust
// Line 169 — value is lost
.map_err(|e| LocalDbError::InvalidVersionValue {
    key: "db_schema_version".to_string(),
    value: "".to_string(),  // ← should be the raw string
    reason: e.to_string(),
})?;
```

The `InvalidVersionValue` variant has a `value` field designed to carry the offending string for debugging, but it is always empty. Fix: capture the raw string before parsing and pass it to the error variant.

**Fix**:
```rust
let raw = ...fetch_optional...?.ok_or(...)?.0;
let db_schema_version = raw.parse::<u32>().map_err(|e| LocalDbError::InvalidVersionValue {
    key: "db_schema_version".to_string(),
    value: raw.clone(),  // preserve the actual value
    reason: e.to_string(),
})?;
```

### W-02: `OutboxPool` sets WAL but not `foreign_keys` pragma

**File**: `crates/nexus-sync/src/pool.rs:48-50`
**Severity**: Warning
**Category**: Schema correctness / Consistency

`OutboxPool::new()` only sets `PRAGMA journal_mode = WAL` but omits `PRAGMA foreign_keys = ON`. In contrast, `nexus_local_db::open_pool()` sets both pragmas. While the outbox tables currently have no foreign key constraints, this inconsistency means:
1. Future outbox schema additions with FK constraints would silently be unchecked
2. Consumers of the outbox pool cannot rely on FK enforcement being uniform across pools

**Fix**: Add `sqlx::query("PRAGMA foreign_keys = ON").execute(&pool).await?;` after the WAL pragma in `OutboxPool::new()`.

### W-03: Daemon double-pool creation on initialization

**File**: `crates/nexus42d/src/workspace/mod.rs:129-130`
**Severity**: Warning
**Category**: Performance / Architecture

```rust
crate::db::schema::Schema::init(&db_path).await?;  // opens pool A, runs migrations, drops pool A
let db = DbPool::new(&db_path, PoolConfig::from_env()).await?;  // opens pool B
```

`Schema::init()` calls `nexus_local_db::init_pool()` which opens a pool, runs migrations, seeds versions, and returns the pool — which is immediately discarded. Then `DbPool::new()` opens a second independent pool to the same file. This is **correct** (migrations are idempotent, pragmas are per-connection) but wastes I/O on every daemon startup.

**Recommended fix**: Consider having `Schema::init()` return the initialized pool, or add a `nexus_local_db::ensure_migrated()` that only runs migrations without opening a separate pool. **Not blocking** — this is a pre-release tool and the overhead is small.

### W-04: `SoulMeta.schema_version` type changed from `u32` to `i64`

**File**: `crates/nexus-local-db/src/soul_meta.rs:15`
**Severity**: Warning
**Category**: API consistency

The old `schema.rs` used `u32` for `schema_version` in `SoulMeta`. The new sqlx version uses `i64` because SQLite `INTEGER` maps to `i64` in sqlx's `FromRow`. This is **safe for data** (no overflow risk) but creates internal API inconsistency: `SchemaVersions` in `lib.rs` still uses `u32` for both version fields, while `SoulMeta` uses `i64`.

**Recommended**: Either cast to `u32` in `FromRow` (via a custom type or `From<LocalIdentityRowRaw>` pattern like identity.rs uses for `platform_linked`), or align `SchemaVersions` to also use `i64`. **Not blocking** — no data loss, and version values are small integers.

## Suggestions

### S-01: `get_all_keywords` uses O(n²) dedup via `Vec::contains`

**File**: `crates/nexus-local-db/src/memory_fragment.rs:112-119`

```rust
let mut all_keywords: Vec<String> = Vec::new();
for row in keywords_json {
    if let Ok(keywords) = serde_json::from_str::<Vec<String>>(&row.0) {
        for kw in keywords {
            if !all_keywords.contains(&kw) {  // O(n) per keyword
                all_keywords.push(kw);
            }
        }
    }
}
```

Replace with `HashSet` for O(1) lookups, then convert to `Vec` at the end. Current behavior is fine for small fragment counts but will degrade at scale.

### S-02: Inconsistent `#[allow(clippy::unwrap_used)]` across test modules

`memory_fragment.rs` has `#[allow(clippy::unwrap_used)]` on its test module, but `identity.rs`, `soul_meta.rs`, and `pending_review.rs` do not. Consider adding the attribute uniformly or using `.expect("descriptive message")` instead.

### S-03: `DbPool` duplicates PRAGMA setup from `nexus_local_db::open_pool`

**File**: `crates/nexus42d/src/db/pool.rs:149-154`

The daemon's `DbPool::new()` independently runs WAL + foreign_keys pragmas, duplicating what `nexus_local_db::open_pool()` already does. Consider a shared helper or having `DbPool` accept an already-initialized pool. Low priority since the duplication is currently consistent.

### S-04: No `close()` method on `DbPool`

sqlx's `SqlitePool` has a `close()` method for graceful shutdown. The `DbPool` wrapper doesn't expose it. For a local daemon this is minor (the OS cleans up on process exit), but for correctness and testing it's good practice.
