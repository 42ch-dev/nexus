## Warnings

### W1: Outbox schema DDL inline â€” inconsistent with nexus-local-db migration pattern

**File**: `crates/nexus-sync/src/outbox.rs`, `init_pool_with_schema()` (lines 95-136)

The outbox creates its tables (`outbox_entries`, `partial_apply_states`) and indexes via hardcoded DDL strings executed one-by-one in a loop:

```rust
for ddl in &[
    "CREATE TABLE IF NOT EXISTS outbox_entries (...)",
    "CREATE INDEX IF NOT EXISTS idx_outbox_delivery_state ...",
    ...
] {
    sqlx::query(ddl).execute(pool.inner()).await?;
}
```

Meanwhile, `nexus-local-db` uses the established `sqlx::migrate!("./migrations")` infrastructure with timestamped SQL files in `crates/nexus-local-db/migrations/`.

**Risk**: Future outbox schema changes will require hand-writing ALTER TABLE code in Rust rather than adding migration files. The two approaches diverge in how schema versioning is tracked â€” nexus-local-db uses `db_schema_version` in `workspace_meta`, but the outbox has no equivalent versioning. This increases the cost of any future migration (e.g., adding columns, changing index definitions).

**Recommendation**: Move outbox DDL into `crates/nexus-local-db/migrations/` as additional migration files, or create a separate `crates/nexus-sync/migrations/` directory with its own `sqlx::migrate!()` call. This unifies schema management under one pattern.

### W2: CI `verify-sqlx-offline` step references missing `.sqlx/` metadata

**File**: `.github/workflows/ci.yml`, lines 98-114

The CI defines a `verify-sqlx-offline` job that runs:
```yaml
run: cargo sqlx prepare --workspace --all -- --all-targets --check
```

However, **no `.sqlx/` directory exists** in the repository (verified via `git log -- .sqlx` returning zero results, and file system glob returning zero matches). The `sqlx prepare --check` command validates that `.sqlx/` query metadata is up-to-date with the SQL strings in the codebase. Without this directory, the behavior depends on the exact sqlx-cli version:

- If `SQLX_OFFLINE=true` is set (it is), `sqlx prepare --check` may pass vacuously if there are zero compile-time-verified macros (`query!`, `query_as!`) to check.
- If any workspace crate ever adds a compile-time-verified macro without regenerating `.sqlx/`, this step will fail in CI without a clear error message.

**Cross-check**: A grep for `sqlx::query!`, `sqlx::query_as!`, `sqlx::query_scalar!` across all `crates/` returns **zero results** â€” all queries use the runtime-checked variants (`sqlx::query()`, `sqlx::query_as()`). This means the current codebase doesn't need `.sqlx/` metadata for compilation.

**Risk**: The step gives false confidence (passes vacuously) today, but masks a latent configuration gap. If a developer later adds `query!` macros, CI will break confusingly.

**Recommendation**: Either (a) remove the `verify-sqlx-offline` step since the project deliberately uses runtime-checked queries only, or (b) add documentation in `docs/` or a code comment explaining that this project opts out of compile-time query verification and the step exists only as a guard against accidental macro introduction. If (a), the three `SQLX_OFFLINE: "true"` env entries in other jobs remain correct (they prevent sqlx from trying to connect to a DB during clippy/test).

### W3: `SoulMeta::schema_version` type change from `u32` to `i64` â€” implicit contract change

**File**: `crates/nexus-local-db/src/soul_meta.rs`, line 19

```rust
// Before (rusqlite):
pub schema_version: u32,

// After (sqlx):
pub schema_version: i64,
```

This is a necessary adaptation for sqlx's SQLite type mapping (rusqlite maps INTEGER to i64 by default, but the old code used `row.get::<_, u32>()` which performed a narrowing cast). The change is functionally correct but is an **implicit breaking change** for any consumer pattern-matching on `SoulMeta`.

**Recommendation**: Add a doc comment on `SoulMeta` noting the type change, or add a `#[allow(clippy::cast_possible_truncation)]` / explicit `as u32` cast at read sites if the domain guarantees `schema_version` fits in u32.

## Suggestions

### S1: Redundant WAL PRAGMA execution

`OutboxPool::new()` already executes `PRAGMA journal_mode = WAL` (pool.rs line 48). Then `init_pool_with_schema()` executes it again (outbox.rs line 99). The second execution is harmless but adds an unnecessary round-trip. Remove the duplicate in `init_pool_with_schema()` or add a comment explaining the belt-and-suspenders intent.

### S2: `stage()` method lacks idempotency guard

`Outbox::stage(&self, bundle: &Bundle)` inserts unconditionally. If called twice with the same bundle, it creates duplicate outbox entries. The `stage_if_absent()` method provides the idempotent variant. Consider whether `stage()` should have an `#[doc(hidden)]` or `/// WARNING: not idempotent` annotation, or whether callers should always use `stage_if_absent()`.

### S3: `count_by_state` uses `i64` â†’ `usize` cast without overflow guard

```rust
let count: (i64,) = sqlx::query_as("SELECT COUNT(*) ...").fetch_one(...).await?;
Ok(count.0 as usize)
```

On 32-bit platforms, `i64::MAX as usize` would truncate. This is extremely unlikely for an outbox count, but `count.0.try_into().unwrap()` or `usize::try_from(count.0).expect("count overflow")` would be more defensive.

### S4: Cargo.lock dependency bloat from sqlx

The sqlx 0.8.0 dependency pulls in `sqlx-mysql`, `sqlx-postgres`, and their transitive dependencies (hkdf, hmac, rsa, pkcs1, pkcs8, stringprep, md-5, sha1, etc.) even though only SQLite is used. The workspace `Cargo.toml` correctly sets `default-features = false`, but `sqlx-core` still depends on some crypto crates via the `sqlx-macros-core` â†’ `dotenvy` â†’ temp file path. Consider auditing `cargo tree -p sqlx` to confirm no unnecessary crates are compiled.

### S5: `pool.rs` test helper creates file but doesn't clean up PRAGMAS

The test `create_test_db()` in `pool.rs` creates an empty file but doesn't run `PRAGMA foreign_keys = ON`. The production `nexus_local_db::open_pool()` does set this PRAGMA (lib.rs line 105-107), but `OutboxPool::new()` does not. If foreign key constraints are important for outbox data integrity, consider adding `PRAGMA foreign_keys = ON` to the pool initialization.

## Positive Observations

1. **Transaction safety improved**: The old `deadpool-sqlite` code used `conn.interact(move |conn| { txn.execute(...); txn.commit() })` which required moving data across async boundaries and double error unwrapping (`??`). The new `let mut tx = pool.inner().begin().await?; ... tx.commit().await?;` pattern is cleaner, with proper async error propagation via `?`.

2. **`stage_if_absent` transaction correctness**: The check-then-insert pattern correctly wraps both the `SELECT EXISTS` and the `INSERT` in a single transaction with explicit `tx.rollback().await?` on the early-return path. This is correct for SQLite's default SERIALIZED isolation mode.

3. **`mark_failed` retry logic preserved**: The read-modify-write pattern for `retry_count` (SELECT â†’ check MAX_RETRIES â†’ UPDATE) is correctly wrapped in a transaction. The error path (`OutboxMaxRetriesExceeded`) commits the final state before returning the error â€” no state leakage.

8. **Dependency cleanup verified**: `Cargo.lock` diff confirms complete removal of `rusqlite`, `deadpool-sqlite`, `deadpool`, `deadpool-sync`, `fallible-iterator`, `fallible-streaming-iterator`. New `sqlx` 0.8.0 and its sqlite-specific sub-crates (`sqlx-sqlite`, `sqlx-core`) added correctly. All four workspace crates (`nexus-sync`, `nexus-local-db`, `nexus42`, `nexus42d`) consistently depend on `sqlx.workspace = true`.
