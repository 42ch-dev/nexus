## Critical / Blocking Findings

### R1 — CRITICAL: Path Traversal via Unsanitized `creator_id`

**File**: `crates/nexus-home-layout/src/lib.rs:42-47`

```rust
pub fn creator_soul_md_path(home: &Path, creator_id: &str) -> PathBuf {
    nexus_root_from_home(home)
        .join("creators")
        .join(creator_id)   // ← No sanitization
        .join("SOUL.md")
}
```

`creator_id` is directly interpolated into the filesystem path with no validation. A malicious or malformed `creator_id` containing `../` sequences (e.g., `"ctr_evil/../../../etc/passwd"`) would escape the intended `~/.nexus42/creators/` sandbox and write files outside the intended directory.

**Impact**: Write arbitrary files on the filesystem, clobbering system or user files.

**Evidence**: `grep "creator_id"` across the diff shows no `normalize()` call, no `strip_prefix()` check, and no allowlist validation before `join(creator_id)`.

**Required fix**: Validate that `creator_id` contains only safe characters (e.g., `^[a-zA-Z0-9_-]+$`) and/or canonicalize the resolved path and verify it still starts under `nexus_root_from_home(home)/creators/`.

## High-Severity Findings

### R2 — HIGH: `soul_meta` Module is Dead Code

**File**: `crates/nexus-local-db/src/soul_meta.rs`

The `soul_meta` module defines `upsert`, `get`, and `delete` functions for per-creator SOUL metadata persistence, and `SOUL_META_TABLE` is added to the schema. However:

- `soul_io::create()`, `soul_io::load()`, `soul_io::save()`, `soul_io::delete()` never call any `soul_meta::*` functions.
- The CLI commands (`init`, `show`, `edit_personality`, `validate`) use `soul_io` directly and never touch the `soul_meta` table.
- `grep -r "soul_meta::" crates/` returns zero results outside of `soul_meta.rs` itself.

The table and all its functions are created but completely unwired. This is pure dead code that adds maintenance burden and compile-time cost.

**Evidence**: `grep "soul_meta::" crates/` — no callers.

**Required fix**: Either wire `soul_meta` into the soul I/O lifecycle (call `soul_meta::upsert` after create/save, `soul_meta::get` before load for caching, `soul_meta::delete` on soul deletion) or remove the entire module and the v3 migration.

## Medium-Severity Findings

### R3 — MEDIUM: `extract_frontmatter` Edge Case — Missing Leading Newline

**File**: `crates/nexus-domain/src/soul.rs:176-186`

```rust
fn extract_frontmatter(content: &str) -> String {
    let trimmed = content.trim_start();
    if !trimmed.starts_with("---") {
        return String::new();
    }
    if let Some(end) = trimmed[3..].find("\n---") {
        trimmed[3..3 + end].trim().to_string()
    } else {
        String::new()
    }
}
```

**Issue 1**: The closing delimiter search looks for `\n---` but standard YAML frontmatter can be `---\nkey: val\n---` (no leading newline before closing `---`). The current code would miss this and treat the entire content as body.

**Issue 2**: If the closing `---` is on the same line as content with no preceding newline (e.g., `---\nkey: val---`), it also won't match. The correct pattern for the closing delimiter is `\n---` at the start of a line, but the search should be `\n---` not `"---"` preceded by any whitespace + newline.

**Issue 3**: The regex `---\n` (or `^---$` multi-line) would be more correct.

**Impact**: SOUL.md files with frontmatter that don't have a blank line before the closing `---` would fail to parse correctly, returning an empty frontmatter.

**Workaround observed**: The `render()` method always produces `\n---\n` (newline before and after delimiters), so round-tripping works. But manually created SOUL.md files could easily trigger this.
