## Critical Findings

### C1 ŌĆö `soul_meta.rs` entirely `#[allow(dead_code)]`: DB table created but never written or read

**File**: `crates/nexus-local-db/src/soul_meta.rs` (lines 10, 22, 47, 72)

`SoulMeta` struct, `upsert()`, `get()`, `delete()` are ALL annotated `#[allow(dead_code)]`. The v3 migration creates `soul_meta` table, the DDL is defined in `schema.rs`, and the module is `mod`-declared in `lib.rs` ŌĆö but **no code path in CLI, domain, or any other module calls these functions**. The soul CLI commands use `soul_io` for file operations and never touch the SQLite metadata table.

This is dead infrastructure:
- `crates/nexus-local-db/src/lib.rs` declares `mod soul_meta` but does **not** re-export it (`pub use` is absent).
- `crates/nexus42/src/commands/soul.rs` imports only `nexus_domain::soul_io`.
- The table DDL, migration, and CRUD exist but are unreachable from any entry point.

**Required**: Either wire `soul_meta` CRUD into CLI/domain operations (init ŌåÆ upsert, show ŌåÆ get, delete ŌåÆ delete) OR remove the table/migration/CRUD from this plan and defer as a residual.

**Severity**: Critical ŌĆö ships unreferenced database schema and migration code that increases maintenance burden with zero functional coverage.

### C2 ŌĆö `render()` silently drops YAML frontmatter on `serde_yaml` serialization failure

**File**: `crates/nexus-domain/src/soul.rs` (line 81)

```rust
let yaml = serde_yaml::to_string(&self.frontmatter).unwrap_or_default();
```

`SoulFrontmatter` has `extra: HashMap<String, serde_yaml::Value>` which can contain arbitrary YAML values. If serialization fails (possible with cyclic or unsupported `serde_yaml::Value` variants), `unwrap_or_default()` produces an empty string ŌĆö the entire frontmatter block is silently omitted from rendered output.

**Required**: Replace with explicit error handling:

```rust
let yaml = serde_yaml::to_string(&self.frontmatter)
    .map_err(|e| DomainError::SoulFrontmatterError(format!("render: {e}")))?;
```

Or at minimum add a `tracing::warn!` before falling back to empty.

**Severity**: Critical ŌĆö silent data loss on roundtrip for custom frontmatter keys.

### C3 ŌĆö `extract_frontmatter` drops malformed frontmatter silently (no error)

**File**: `crates/nexus-domain/src/soul.rs` (lines 175-186)

When content starts with `---` but has no closing `---` (or closing `---` on the same line), `extract_frontmatter` returns `String::new()`. The subsequent `serde_yaml::from_str` sees an empty string and produces a default `SoulFrontmatter` ŌĆö **the malformed frontmatter is silently discarded** rather than raising `SoulFrontmatterError`.

Example: `"---key: val\n## Personality\n"` ŌåÆ frontmatter silently dropped ŌåÆ `SoulFrontmatter` defaults to empty.

**Required**: If content starts with `---` but no valid closing delimiter is found, return `DomainError::SoulFrontmatterError("missing closing ---")`.

```rust
fn extract_frontmatter(content: &str) -> Result<String, DomainError> {
    // return Ok or Err instead of String
}
```

**Severity**: Medium ŌĆö malformed user documents silently lose metadata.

## Warnings

### W1 ŌĆö SOUL error variants not mapped to structured CLI errors

**File**: `crates/nexus42/src/errors.rs` (lines 216-225)

`From<DomainError> for CliError` only special-cases `PlatformOperationProhibited`. `SoulNotFound`, `SoulMissingSection`, and `SoulFrontmatterError` fall through to:

```rust
other => CliError::Other(format!("Domain error: {}", other))
```

Users see generic `"Domain error: SOUL.md not found for creator '...' at ..."` instead of actionable SOUL-specific messages with suggestions (consistent with the pattern used for `DaemonNotRunning`, `WorkspaceNotInitialized`, etc.).

**Suggested**: Add dedicated `CliError::SoulNotFound`, `CliError::SoulInvalid` variants with user-friendly messages, matching the established error-pattern convention.

### W2 ŌĆö `creator_soul_md_path` has no path traversal defense

**File**: `crates/nexus-home-layout/src/lib.rs` (lines 42-47)

```rust
pub fn creator_soul_md_path(home: &Path, creator_id: &str) -> PathBuf {
    nexus_root_from_home(home)
        .join("creators")
        .join(creator_id)   // ŌåÉ raw join, no validation
        .join("SOUL.md")
}
```

If `creator_id` contains `../`, the resulting path escapes `~/.nexus42/creators/`. Mitigation exists upstream via `is_valid_creator_id` (re-exported from `local_identity`), but this path helper itself performs no validation.

**Suggested**: Add `canonicalize` check or validate that the resulting path is under `creators/` before returning. Or document the precondition that `creator_id` must pass `is_valid_creator_id`.

### W3 ŌĆö `soul_io::create()` has TOCTOU race between exists-check and write

**File**: `crates/nexus-domain/src/soul_io.rs` (lines 37-56)

The function checks `path.exists()`, creates the directory, renders, then calls `std::fs::write()` ŌĆö which silently overwrites any file created between the check and the write.

**Suggested**: Use `std::fs::OpenOptions::new().create_new(true).write(true).open(&path)` to fail atomically if the file already exists.

### W4 ŌĆö `save()` has no content validation before overwrite

**File**: `crates/nexus-domain/src/soul_io.rs` (lines 59-71)

`save()` renders and overwrites without calling `doc.validate()` first. If a caller modifies `doc.personality = None` then calls `save()`, the file is corrupted (missing required section).

**Suggested**: Call `doc.validate()` before `std::fs::write()` in `save()`.

### W5 ŌĆö `nexus-home-layout/src/lib.rs` has blanket `#![allow(dead_code)]`

**File**: `crates/nexus-home-layout/src/lib.rs` (line 5)

The entire crate is `#![allow(dead_code)]`. While acceptable for a shared utility crate consumed by other crates, this suppresses visibility into which helpers are actually used. Consider removing and using per-function `#[allow(dead_code)]` only where needed.

### W6 ŌĆö No `edit-experience` CLI command

**File**: `crates/nexus42/src/commands/soul.rs`

ADR-016 defines both `## Personality` and `## Experience` as required sections. The CLI has `edit-personality` but no `edit-experience`. This may be intentional (Experience read-only per ADR-016), but the assignment mentions "Experience read-only" as a compliance check ŌĆö this should be documented/validated.

## Suggestions

### S1 ŌĆö Consider using `pulldown-cmark` instead of hand-rolled heading parser

`Cargo.toml` adds `pulldown-cmark = "0.12"` and `pulldown-cmark-frontmatter = "0.2"` as workspace deps, and `nexus-domain/Cargo.toml` includes them. However, `soul.rs` uses **hand-rolled** `extract_frontmatter()` and `extract_h2_sections()` functions instead of the pulldown-cmark library. If the intent is to use pulldown-cmark for robust markdown parsing, wire it in. If not, consider removing the dependency to reduce the dependency tree.

### S2 ŌĆö Add `#[allow(dead_code)]` per-variant for new SOUL errors once wired

In `errors.rs`, `SoulNotFound`, `SoulMissingSection`, `SoulFrontmatterError` are all currently used, so no `#[allow(dead_code)]` needed. But if C1 (dead soul_meta) is resolved by removing the table, revisit.

### S3 ŌĆö Test `render()` with empty personality/experience (`None` vs `Some("")`)

`soul.rs:86-96` uses `as_deref().unwrap_or("")` ŌĆö this works for `Some("")` and `None`. But `new()` sets `Some(String::new())` while `parse()` without a section sets `None`. The roundtrip test (`render_roundtrip`) only tests `Some` content. Add a test for `None` rendering.

### S4 ŌĆö `soul_meta` module should be re-exported from `nexus-local-db`

If the CRUD functions are to be used, add to `lib.rs`:

```rust
pub use soul_meta::{SoulMeta, delete as delete_soul_meta, get as get_soul_meta, upsert as upsert_soul_meta};
```

### S5 ŌĆö Integration tests should cover `soul init ŌåÆ show ŌåÆ edit-personality ŌåÆ validate` e2e

The current integration tests only verify `soul --help` and `soul show/validate` error without active creator. Add a full lifecycle test:

```rust
#[test]
fn soul_init_show_edit_validate() {
    // init workspace, set active creator, soul init, soul show, edit-personality, validate
}
```

4. **Proper error propagation**: `soul_io.rs` wraps `std::fs` errors in `DomainError::ValidationError` with context strings ŌĆö no raw IO errors bubble up.

7. **Schema DDL follows existing conventions**: `SOUL_META_TABLE` in `schema.rs` matches the established `CREATE TABLE IF NOT EXISTS` pattern used by all other tables.
