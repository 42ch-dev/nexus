## Detailed Findings

### F1 — Path Traversal via `creator_id` (HIGH)

**Files**: `crates/nexus-home-layout/src/lib.rs`, `crates/nexus-domain/src/soul_io.rs`

**Issue**: `creator_soul_md_path` directly embeds `creator_id` into a `PathBuf` without validation:

```rust
pub fn creator_soul_md_path(home: &Path, creator_id: &str) -> PathBuf {
    nexus_root_from_home(home)
        .join("creators")
        .join(creator_id)   // ← unsanitized user input
        .join("SOUL.md")
}
```

A malicious or malformed `creator_id` such as `../../../etc` resolves to `/etc/SOUL.md`, allowing arbitrary file write/delete outside the nexus home directory. The `creator_id` flows from `CliConfig.active_creator_id` (user-controlled config) directly to filesystem operations.

**Impact**: Arbitrary file write/overwrite/deletion on the host filesystem.

**Recommendation**:
1. Validate `creator_id` against a safe pattern (e.g., `[a-zA-Z0-9_-]+`) before path construction.
2. Alternatively, sanitize path segments by rejecting `/`, `\`, and `..`.
3. Apply validation at the CLI entry point and/or in `soul_path`/`creator_soul_md_path`.

### F2 — `soul_meta` Table Is Never Populated (MEDIUM)

**Files**: `crates/nexus-local-db/src/soul_meta.rs`, `crates/nexus-domain/src/soul_io.rs`, `crates/nexus42/src/commands/soul.rs`

**Issue**: The `soul_meta` SQLite table and its CRUD module exist, but **no call site writes to or reads from it**. The module is decorated with `#[allow(dead_code)]` on structs and functions, confirming it is unintegrated. This directly violates acceptance criterion **AC8**:

> "soul_meta table stores metadata after SOUL creation/update"

**Impact**: The metadata persistence feature is incomplete; future consumers (T5/T6) will find an empty table.

**Recommendation**:
1. Wire `soul_meta::upsert` into `soul_io::create` and `soul_io::save` (or into the CLI commands that call them).
2. Populate `created_at`/`updated_at` with RFC 3339 timestamps and compute/store `personality_hash`/`experience_hash` (or remove the hash columns if out of T4 scope).
3. If wiring is intentionally deferred, document the gap explicitly and track it as a residual.

### F3 — Hand-Rolled Markdown Parser Has Edge Cases (MEDIUM)

**File**: `crates/nexus-domain/src/soul.rs`

**Issue**: The implementation adds `pulldown-cmark` and `pulldown-cmark-frontmatter` as dependencies (per decision D1) but uses a custom parser instead. The custom parser has real edge cases:

1. **Code fences are not respected**: A line starting with `## ` inside a fenced code block is incorrectly parsed as a new H2 section boundary.
   ```markdown
   ## Personality
   Some code:
   ```markdown
   ## Goals
   Write more.
   ```
   ## Experience
   None.
   ```
   Here `## Goals` would be extracted as a section, splitting the document incorrectly.

2. **Closing ATX hashes**: `## Personality ##` (valid Markdown H2) produces section name `Personality ##`, which fails to match `SECTION_PERSONALITY`.

3. **Tab separator**: `##\tPersonality` (tab instead of space) is not recognized as H2 because `strip_prefix("## ")` expects a literal space.

**Impact**: Users with valid Markdown formatting will experience section mis-detection and validation failures.

**Recommendation**:
1. Use `pulldown-cmark` + `pulldown-cmark-frontmatter` as originally planned (decision D1) to handle code fences, heading variants, and frontmatter robustly.
2. If keeping the custom parser, add unit tests for fenced code blocks and closing hashes, and document the unsupported constructs.

### F4 — TOCTOU Patterns in File I/O (LOW)

**File**: `crates/nexus-domain/src/soul_io.rs`

**Issue**: `create` and `save` check `path.exists()` before operating, creating a TOCTOU window:

```rust
if path.exists() { return Err(...); }   // check
std::fs::write(&path, content)          // use
```

**Impact**: Low for a single-user CLI, but the custom "already exists" / "not found" errors can be bypassed by concurrent filesystem modifications, leading to confusing error messages.

**Recommendation**: Prefer operating directly and matching on the resulting `std::io::ErrorKind` (e.g., `AlreadyExists`, `NotFound`) for atomic, race-free semantics.

---

### F5 — `soul show` Re-renders Instead of Displaying Raw File (LOW)

**File**: `crates/nexus42/src/commands/soul.rs`

**Issue**: The `show` command calls `doc.render()` rather than printing the original file bytes. Any round-trip differences in blank lines, comment preservation, or formatting will be silently normalized.

**Impact**: Users may see different content than what is actually on disk, making debugging difficult if parsing bugs exist.

**Recommendation**: Read and print the raw file content in `show` (or add a `--raw` flag). Validation and editing can continue to use the structured model.

---

### F6 — Integration Test Coverage Is Thin (LOW)

**File**: `crates/nexus42/tests/integration.rs`

**Issue**: Only three SOUL-related integration tests exist:
- `soul_help`
- `soul_requires_active_creator`
- `soul_validate_requires_active_creator`

Missing:
- `soul init` success and duplicate-init failure
- `soul show` with an existing SOUL.md
- `soul edit-personality` success and stdin path
- `soul validate` with a valid SOUL.md

**Impact**: Regressions in the primary user workflows could go undetected in CI.

**Recommendation**: Add integration tests for `init`, `show`, and `edit-personality` happy paths, ideally exercising the actual filesystem in a `TempDir`.
