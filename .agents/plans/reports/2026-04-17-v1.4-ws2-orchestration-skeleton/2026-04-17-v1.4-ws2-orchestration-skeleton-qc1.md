## 3. Findings

### Finding 1 — `RpcTransport::Send + 'static` bound blocks `StdioTransport` usage after `WorkerHandle::call_json_rpc`

**Severity**: high
**Location**: `crates/nexus-orchestration/src/worker/transport.rs:25`; `manager.rs:142-163`
**Spec ref**: §6.3 (IPC transport), §6.4 (IPC message catalogue)

The `RpcTransport` trait requires `Send + 'static`:

```rust
// transport.rs:25
pub trait RpcTransport: Send + 'static {
```

Meanwhile, `WorkerHandle::call_json_rpc` **consumes** the stored stdin/stdout pipe halves (`Option<ChildStdin>` / `Option<ChildStdout>`) via `.take()`, creates a `StdioTransport`, sends one request, and drops the transport. After one call, the pipes are permanently gone — subsequent calls fail with `WorkerError::Internal("stdin already consumed")`.

**This is a deliberate WS2 design** (the code comment at manager.rs:141-142 says: *"For multi-request sessions, use IpcClient (not yet wired in WS2)"*). However, the spec §6.4 defines **six** distinct request methods and multiple notifications that require a persistent channel, not one-shot. The `RpcTransport` trait's `'static` bound combined with the one-shot consumption pattern means that:

1. The trait is designed for persistent IPC but the only WS2 consumer (`call_json_rpc`) destroys the transport after one use.
2. If WS3 adds `IpcClient` on top of this, the `'static` bound will require re-architecting who owns the transport — `WorkerHandle` currently owns stdin/stdout, not the transport.

**Recommendation**: Make the ownership boundary explicit now. Either:
- (a) Add a `WorkerHandle::take_transport()` method that hands off the pipes to an `IpcClient` in WS3, or
- (b) Store the `StdioTransport` inside `WorkerHandle` rather than raw pipe halves, so the same transport survives across calls.

Marking as **high** because the current design makes WS3 integration non-trivial — the trait shape and the ownership model need to be compatible.

### Finding 2 — `GraphFlowEngine::new_session` creates orphan sessions with empty `current_task_id`

**Severity**: high
**Location**: `crates/nexus-orchestration/src/engine.rs:284-301`

The `new_session` impl (which satisfies `OrchestrationEngine::new_session`) creates a session with:

```rust
// engine.rs:288
let session = graph_flow::Session::new_from_task(session_id.clone(), "");
```

The empty string `""` as the start task ID is problematic:
- `graph_flow::Session` with an empty `current_task_id` is semantically invalid — `FlowRunner::run()` will fail or produce undefined behavior when trying to find the task.
- The session is persisted to storage and tracked in the in-memory `sessions` vec with `status: SessionStatus::Running` and `current_task_id: None`, but **no `FlowRunner` is registered** in the `runners` map.
- Subsequent `run_step()` calls for this session will return `EngineError::NoGraphLoaded`.

The spec §4.2 says `new_session` should *"Create a new session identified by `key`, seeded with `ctx`"* — the expectation is a session ready to execute, not an orphan stub.

**Recommendation**: Either:
- (a) `new_session` should accept a graph parameter (like `start_session` does) and create a proper `FlowRunner`, or
- (b) Document that `new_session` is a "registration-only" path and `start_session` is the only way to create a runnable session. If (b), the trait should make this distinction explicit in method names.

### Finding 3 — In-memory session tracking races with persistent storage on restart

**Severity**: high
**Location**: `crates/nexus-orchestration/src/engine.rs:184-186` (sessions field); `main.rs:140-156` (boot sequence)

The `GraphFlowEngine` maintains an in-memory `Vec<SessionSummary>` (`sessions` field at engine.rs:185). This is the only data structure used by `list_active()`, `get_status()`, `signal()`, and `update_session_status()`.

**Problem**: On daemon restart, this vec is empty. The SQLite `orchestration_sessions` table contains persisted sessions, but they are never loaded into the in-memory tracker at boot. The `_system.maintenance` session created in `main.rs:143-156` is the only session that gets tracked after restart.

This means:
- After a crash/restart, any user-created sessions in `running`/`paused`/`waiting_for_input` state are **invisible** to the API (`list_active` returns empty).
- The spec §6.1 "Crash recovery" explicitly says: *"daemon restart reads `orchestration_sessions` table, finds sessions in `running` / `waiting_for_input` state..."* — this is not implemented.
- `get_status()` and `signal()` will return `SessionNotFound` for all previously-active sessions.

**Recommendation**: Add a `resume_active_sessions()` method to `GraphFlowEngine` that:
1. Queries the SQLite storage for sessions in non-terminal states.
2. Reconstructs `SessionSummary` entries from the stored data.
3. Repopulates the in-memory `sessions` vec.
4. Optionally re-registers `FlowRunner` instances if the graph is available.

Call this in `main.rs` after constructing the engine and before starting `_system.maintenance`.

### Finding 4 — `Context` wrapper is a zero-utility newtype

**Severity**: medium
**Location**: `crates/nexus-orchestration/src/engine.rs:101-123`

The `Context` struct wraps `graph_flow::Context` but provides no additional behavior — `new()` simply delegates to the inner constructor, and there are no methods to access or modify the inner context. The `#[allow(dead_code)]` on the inner field confirms this.

The spec §4.2 says: *"In future tasks this will carry engine-specific metadata alongside the graph-flow context"* — but right now it's dead weight. Any caller that wants to pass data into `new_session` must use `Context::new()` which carries no data.

**Recommendation**: Remove the `Context` wrapper entirely for now. When engine-specific metadata is needed (WS3+), introduce it then. The current wrapper adds an indirection layer with zero benefit and a confusing API (callers pass `Context` but it carries nothing).

### Finding 5 — `system_preset::build()` creates its own `CapabilityRegistry` instead of accepting one

**Severity**: medium
**Location**: `crates/nexus-orchestration/src/system_preset.rs:119-120`

```rust
pub fn build() -> Arc<Graph> {
    let registry = Arc::new(CapabilityRegistry::with_builtins());
```

The system preset graph builder instantiates its own `CapabilityRegistry::with_builtins()`. Meanwhile, `main.rs:161` also creates a separate `CapabilityRegistry::with_builtins()`. These are two distinct registry instances:
- The preset graph uses registry A (created inside `build()`)
- The daemon wires registry B to `WorkspaceState`

If any capability's behavior depends on shared state (e.g., a sync registry cache), the two instances will operate independently.

**Recommendation**: Change `build()` to accept `Arc<CapabilityRegistry>` as a parameter, matching how `PresetCapabilityTask::new()` already accepts a registry. The caller (`main.rs`) should pass the same registry it wires to `WorkspaceState`.

### Finding 6 — `GraphFlowEngine::run_step` clones `FlowRunner` per call

**Severity**: medium
**Location**: `crates/nexus-orchestration/src/engine.rs:248-254`

```rust
let runner = {
    let runners = self.runners.read().await;
    runners
        .get(&session_id.0)
        .cloned()  // <-- clones FlowRunner
        .ok_or(EngineError::NoGraphLoaded)?
};
```

Cloning the `FlowRunner` on every `run_step` call means the cloned runner's internal state (including its reference to `SessionStorage`) diverges from the original. If `FlowRunner` holds mutable session state, the clone operates on a snapshot, not the live runner.

This is only safe if `FlowRunner` is `Clone` by design with copy-on-write or purely shared internals. Given that `graph-flow` is a third-party crate, this assumption should be verified.

**Recommendation**: Confirm that `graph_flow::FlowRunner::clone()` produces a usable runner that shares session state with the original. If it doesn't, store runners as `Arc<FlowRunner>` or use a different per-session tracking mechanism.

### Finding 7 — `WorkerHandle::shutdown()` has a 30-second grace but supervisor task has a hardcoded 30-second timeout

**Severity**: medium
**Location**: `crates/nexus-orchestration/src/worker/manager.rs:170-204` (shutdown); `manager.rs:285` (supervisor timeout)

The `WorkerHandle` has a configurable `shutdown_grace: Duration` (default 30s), but the supervisor task's `tokio::time::timeout` at line 285 is hardcoded to `Duration::from_secs(30)` and doesn't reference `shutdown_grace`. If the caller sets `shutdown_grace` to a different value, the supervisor timeout won't respect it.

**Recommendation**: Pass `shutdown_grace` to the supervisor task (or reference the handle's field) so the timeout matches the configured grace period.

### Finding 8 — Missing input validation on `signal_session` HTTP handler

**Severity**: medium
**Location**: `crates/nexus42d/src/api/handlers/orchestration/sessions.rs:113-123`

```rust
let signal = match body.signal.as_str() {
    "pause" => EngineSignal::Pause,
    ...
    other => {
        return Err((StatusCode::BAD_REQUEST, ...));
    }
};
```

The handler uses `body.signal.as_str()` which assumes `body.signal` is a `serde_json::Value::String`. If the client sends a non-string value (e.g., `"signal": 42` or `"signal": null`), `.as_str()` returns `None` and falls into the `other` branch, returning a confusing error message.

**Recommendation**: Use `serde` deserialization with a strongly-typed `SignalKind` enum for the request body, or add an explicit check for non-string values with a clearer error message.

### Finding 9 — `SessionSummary::status` serialization uses `Debug` derive format

**Severity**: medium
**Location**: `crates/nexus42d/src/api/handlers/orchestration/sessions.rs:55-56`

```rust
status: format!("{:?}", s.status).to_lowercase(),
```

This converts `SessionStatus::Running` → `"running"`, `SessionStatus::WaitingForInput` → `"waitingforinput"` (note: **no underscore**). The spec §4.3 schema defines status as `"running | paused | waiting_for_input | completed | failed"` — the expected value is `waiting_for_input` with underscores.

**Recommendation**: Implement `Display` for `SessionStatus` or use explicit match arms to serialize correctly:
```rust
status: match s.status {
    SessionStatus::WaitingForInput => "waiting_for_input",
    other => format!("{other:?}").to_lowercase(),
}.to_string(),
```

### Finding 12 — `get_session` handler does O(N) scan instead of direct lookup

**Severity**: low
**Location**: `crates/nexus42d/src/api/handlers/orchestration/sessions.rs:67-98`

The `get_session` handler calls `list_active()` and then does a linear scan:

```rust
let sessions = engine.list_active(...).await?;
let session = sessions.into_iter().find(|s| s.session_id == sid)...
```

This is O(N) for a single session lookup. The engine should provide a `get_session(session_id)` method.

**Recommendation**: Add `get_session(&self, session_id: &SessionId)` to `OrchestrationEngine` trait (or use the existing `get_status` and extend it to return a full summary). This is a low-priority concern for WS2 but should be addressed before production use.

### Finding 13 — `SqliteSessionStorage::save` hardcodes status as `"running"`

**Severity**: low
**Location**: `crates/nexus-orchestration/src/storage/sqlite.rs:72`

```sql
VALUES (... 'running', ...)
ON CONFLICT(session_id) DO UPDATE SET
    current_task_id = excluded.current_task_id,
    context_json    = excluded.context_json,
    updated_at      = excluded.updated_at
```

The `ON CONFLICT` clause does **not** update the `status` column. Once a session is saved as `"running"`, it stays `"running"` in SQLite even after the engine marks it `Completed` or `Failed` in memory. On restart (Finding 3), resumed sessions would all appear as `"running"` regardless of their actual terminal state.

**Recommendation**: Include `status = excluded.status` in the `ON CONFLICT DO UPDATE SET` clause, or better yet, have the engine persist status transitions through the storage layer.

### Finding 14 — `Capability::input_schema`/`output_schema` return `&'static str` not `&JsonSchema`

**Severity**: low
**Location**: `crates/nexus-orchestration/src/capability/mod.rs:48-51` vs spec §5.1

The spec §5.1 defines:
```rust
fn input_schema(&self) -> &JsonSchema;
fn output_schema(&self) -> &JsonSchema;
```

But the actual implementation returns `&'static str` (raw JSON Schema strings). This is actually a **reasonable deviation** — `serde_json::Value` or raw strings are easier to serialize in HTTP responses, and the local types (`nexus-contracts/src/local/orchestration/mod.rs`) ship properly typed `Input`/`Output` structs with proper `Serialize`/`Deserialize`.

**Not a finding to fix** — the `&'static str` approach is practical. The spec should be updated to reflect this decision.

### Verdict: **Request Changes**

The three **high** severity findings must be addressed before approval:

1. **F3 (Restart recovery gap)**: The most impactful — after daemon restart, all active sessions become invisible. The spec §6.1 explicitly requires crash recovery, and the Phase 2 acceptance criteria include *"SQLite storage roundtrip test: start session → kill daemon → restart → resume"*. This is a gap between spec and implementation.

2. **F2 (Orphan sessions from `new_session`)**: The `new_session` trait method creates un-runnable session stubs. Either this should be documented as intentional (and the method renamed) or it should accept a graph and create a proper `FlowRunner`.

3. **F1 (One-shot IPC pipe consumption)**: The `WorkerHandle` API destroys IPC pipes after one call. While documented as WS2 limitation, the `RpcTransport` trait's `'static` bound and the ownership model create a structural barrier for WS3's full IPC catalogue.

### This reviewer's unique findings (primary accent: architecture, maintainability, long-term evolution risk):

- **F1**: RpcTransport trait + WorkerHandle ownership model incompatibility
- **F3**: In-memory session tracking not repopulated on restart (spec compliance gap)
- **F5**: `system_preset::build()` creates isolated CapabilityRegistry
- **F6**: FlowRunner clone-per-step correctness question
- **F11**: Unused dependencies in Cargo.toml

### Findings that may overlap with other reviewers (correctness, safety, performance):

- **F8**: Missing input validation on signal_session (security/correctness)
