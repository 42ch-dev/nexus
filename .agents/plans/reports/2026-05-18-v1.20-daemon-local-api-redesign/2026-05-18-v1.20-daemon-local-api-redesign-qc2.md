### 🟡 Warning

#### W-001 — `HostToolExecutor::validate_file_path`: string-prefix bypass for non-existing paths

**File**: `crates/nexus-daemon-runtime/src/api/handlers/host_tool_executor.rs:134-153`

```rust
let abs_requested_str = abs_requested.display().to_string();
if !abs_requested_str.starts_with(&workspace_path_str) {
    return Err(NexusApiError::Forbidden { ... });
}
```

**Issue**: When the requested file path doesn't exist, validation uses a **string prefix check** instead of path-component-aware comparison. If `workspace_path_str` is `"/home/user/workspace"` (no trailing slash), paths like `/home/user/workspace_evil` or `/home/user/workspace_attacker` would pass the prefix check because they literally start with the workspace string.

**Impact**: Medium — an attacker could theoretically write to paths that share a string prefix with the workspace but are actually outside it. However, practical exploitation requires:
1. The path to not exist at validation time (new file creation scenario)
2. The workspace path string to lack a trailing separator
3. The attacker to control the file path parameter

**Mitigating factors**:
- `fs/write_text_file` creates parent directories with `create_dir_all`, but those paths are also validated
- Most legitimate workspace paths end with separator when canonicalized
- The file operations (read/write) use the validated path, so actual access is bounded

**Fix**: For non-existing paths, perform path-component-aware validation:
- Join the relative component to the workspace root if not absolute
- Normalize the path (resolve `.` and `..` without filesystem access)
- Check that each path component stays within the workspace boundary
- Or use `validate_workspace_path_under` from `nexus-agent-host/src/config.rs` which properly canonicalizes and checks prefix on canonical paths

**Source Type**: manual-reasoning + static-analysis
**Confidence**: High

#### W-002 — TOCTOU in `pending_registration.json` read after exists check

**File**: `crates/nexus-daemon-runtime/src/api/handlers/creator_registrations.rs:617-635`

```rust
let pending_path = nexus_home.join("pending_registration.json");
if !pending_path.exists() {
    return Err(...);
}
let pending_content = std::fs::read_to_string(&pending_path).map_err(...);
```

**Issue**: Classic TOCTOU pattern — check exists, then read. An attacker with write access to `.nexus42/` could:
1. Delete the file after the exists check passes
2. Create a malicious file with different verification_code
3. The read would get different content than expected

**Impact**: Low in practice — the attacker would need filesystem access to the user's home directory, which already implies full compromise. The verification code is validated against the platform anyway.

**Fix**: Use direct read pattern (like `load_config_from_path` in agent-host config.rs):
- Attempt `read_to_string` directly
- `io::ErrorKind::NotFound` → return appropriate error
- This eliminates the race window

**Source Type**: manual-reasoning
**Confidence**: Medium

### 🟢 Suggestion

#### S-001 — Add null-byte explicit rejection in path validation helpers

**Files**: `crates/nexus-agent-host/src/config.rs`, `crates/nexus-home-layout/src/lib.rs`

While `validate_creator_id_safe` and `validate_entry_id_safe` reject control characters (including null bytes) via `char::is_control`, the workspace path validators (`validate_workspace_path`, `validate_workspace_path_under`, `validate_config_path`) rely on canonicalize() to reject invalid paths. On some platforms, paths with embedded null bytes may pass initial checks before canonicalize fails.

**Suggestion**: Add explicit null-byte rejection to all path validation helpers:
```rust
if path.as_os_str().to_string_lossy().contains('\0') {
    return Err(HostError::policy_denied("path contains null byte"));
}
```
