### 🟡 Warning

#### W-001 — SSE broadcast channel: unfiltered per-session subscription (performance)

**File**: `crates/nexus-agent-host/src/core/manager.rs:102-107`

```rust
pub fn subscribe(
    &self,
    _session_id: &HostSessionId,
) -> tokio::sync::broadcast::Receiver<HostEvent> {
    self.event_tx.subscribe()
}
```

The `subscribe()` method ignores the `_session_id` parameter and returns an unfiltered broadcast receiver. All events from **all** sessions are broadcast to **every** SSE subscriber. Filtering is deferred to the receiver side in `session_events()` (`agent_host.rs:460-484`), which loops through events and skips non-matching sessions.

**Impact**:
- `EVENT_BROADCAST_CAPACITY = 256` (line 33) is a **shared buffer** across all sessions. Under moderate load with N concurrent sessions, the effective per-session capacity is `256 / N`.
- `RecvError::Lagged` causes **silent event loss** (warning logged, stream continues). Clients miss real data.
- Each `HostEvent` is a potentially large enum variant — 256 slots is small for high-throughput multi-session workloads.
- The `host_tool_executor.rs` and `acp.rs` handlers also send events through this channel, adding to pressure.

**Fix**: Either (a) use per-session broadcast channels created at session registration time, or (b) keep a single channel but increase `EVENT_BROADCAST_CAPACITY` to at least `1024` and add a pre-filter on the send side so non-matching sessions aren't broadcast.

#### W-002 — SSE zombie connections after session ends (reliability)

**File**: `crates/nexus-agent-host/src/api/handlers/agent_host.rs:455-487`

```rust
let stream = futures_util::stream::unfold(rx, move |mut rx| {
    let sid = sid.clone();
    async move {
        loop {
            match rx.recv().await {
                ...
                Err(tokio::sync::broadcast::error::RecvError::Closed) => {
                    return None; // only ends when channel closes
                }
            }
        }
    }
});
```

The SSE `unfold` loop runs **indefinitely** — it only terminates when the broadcast channel closes (which happens when `HostManager` is dropped — effectively at daemon shutdown). When a `SessionStopped` event for the subscribed session is received, it's yielded to the client, but the stream keeps polling.

**Impact**: Each closed session leaves an open SSE connection consuming memory and a subscriber slot on the broadcast channel. Over time, this degrades broadcast channel performance (each send must clone to all subscribers). In the `event_matches_session` filter, `SessionStopped` events for the matching session pass through — but the stream continues.

**Fix**: After yielding a `SessionStopped` event for the subscribed session, return `None` to terminate the stream. Add:

```rust
if matches!(&event, HostEvent::SessionStopped(e) if &e.session_id == &sid) {
    return Some((Ok(Event::default().data(json)), rx)); // yield, then terminate on next poll
}
```

Or use `futures_util::stream::try_unfold` to signal stream end.

#### W-003 — `execute_operation` blocks the HTTP handler thread draining the full event stream (reliability)

**File**: `crates/nexus-daemon-runtime/src/api/handlers/agent_host.rs:399-408`

```rust
let mut stream = host
    .exec(sid.clone(), host_op)
    .await
    .map_err(|e| map_host_error(&e))?;

while let Some(_result) = stream.next().await {
    // Events are consumed; SSE endpoint is the preferred way to relay them.
}
```

The handler synchronously drains the full operation stream before returning a response. For multi-turn agent conversations that may take minutes, this ties up:
- An async task on the tokio runtime
- The HTTP connection (subject to client/proxy timeouts)
- The session state machine (session stays Busy until the stream completes)

The comment correctly notes that "SSE endpoint is the preferred way to relay them" — but the handler itself blocks until completion anyway.

**Impact**: Long-running operations cause HTTP timeouts (default `DEFAULT_REQUEST_TIMEOUT` is 30s on the client side). The client receives no intermediate feedback. If the HTTP connection drops, the stream still runs server-side but with no consumer — the state machine transition (Busy→Ready) depends on the stream's `OpFinished`/`OpFailed` event, which occurs regardless but may leave the operation orphaned from the client's perspective.

**Fix**: Return the `operation_id` immediately after `exec()` and add a `GET /v1/local/agent-host/operations/{id}` polling endpoint. The `POST /operations` handler becomes fire-and-forget (or returns immediately after kicking off the stream in a background task that drives the state machine forward).

#### W-004 — Workspace scan is O(n) filesystem traversal on every paginated request (performance)

**File**: `crates/nexus-daemon-runtime/src/api/handlers/workspaces.rs:139-208`

`scan_workspaces()` performs a full filesystem traversal of `~/.nexus42/creators/*/workspaces/*/meta.json` on every request — even paginated requests with a cursor. The list is sorted in-memory and re-sorted on every call. No caching, no database index.

**Impact**: Linear degradation with number of creator+workspace directories. With 10 creators × 10 workspaces = 100 entries, performance is fine. With 100 creators × 50 workspaces = 5000 entries, each paginated page request does 5000 filesystem stat/read calls. This is especially visible when the client loops through all pages — each page repeats the full scan.

**Fix**: Add a short-lived in-memory workspace registry (e.g., `RwLock<Vec<WorkspaceSummary>>`) refreshed on daemon startup and invalidated on workspace create operations. For V1.20 bootstrapping, even a 60-second TTL cache would eliminate redundant scans during pagination loops.

#### W-005 — KB get/delete entries scan ALL creators and workspaces (performance)

**Files**:
- `crates/nexus-daemon-runtime/src/api/handlers/kb.rs:373-425` (`get_entry`)
- `crates/nexus-daemon-runtime/src/api/handlers/kb.rs:449-499` (`delete_entry`)

Both handlers iterate over `~/.nexus42/creators/*/workspaces/*/kb/entries/{id}.md` looking for a specific entry ID. Despite having a per-workspace `index.json` that maps entry IDs to metadata, neither handler uses the index for lookup — they do an O(n) filesystem scan across all creator/workspace directories.

**Impact**: With N creator/workspace combinations, each `GET /kb/entries/{id}` does N directory reads. Acceptable for N < 10, degrades noticeably beyond that.

**Fix**: Use the per-workspace `index.json` files for ID→workspace lookup. Build a daemon-wide index on startup by aggregating all `index.json` files, mapping entry_id → (creator_id, workspace_slug). This converts O(n) to O(1) lookups.

#### W-006 — KB add_entry: crash-consistency gap in write sequence (reliability)

**File**: `crates/nexus-daemon-runtime/src/api/handlers/kb.rs:315-344`

The `add_entry` handler writes in three steps:
1. **Step 1** (line 315-331): Write updated index to temp file (`index.json.tmp`)
2. **Step 2** (line 334-338): Write entry content directly to `{entry_id}.md` (**not** via temp+rename)
3. **Step 3** (line 340-344): Atomically rename temp index → `index.json`

If the process crashes between steps 2 and 3:
- `{entry_id}.md` exists on disk (content is written)
- `index.json` was NOT updated (the temp rename didn't happen)
- Result: an orphaned `.md` file that the index doesn't reference — invisible to list queries but consuming disk space

`write_kb_index` (line 133-154) correctly uses temp+rename, but the entry content write does not.

**Fix**: Write entry content to a temp file (`{entry_id}.md.tmp`) first, then rename it to `{entry_id}.md` after step 3 completes. Alternatively, reverse the order: write the content file first (if it exists, the entry is "pending"), then atomically commit the index — and have a startup recovery pass that removes orphaned content files.

### 🟢 Suggestion

#### S-001 — Broadcast channel: consider critical event backpressure

**File**: `crates/nexus-agent-host/src/core/manager.rs:33`

`EVENT_BROADCAST_CAPACITY = 256` with silent drop on lag (`_ = tx.send(...)`). For session lifecycle events (Created, Stopped) and operation state transitions (Started, Finished, Failed), consider a separate low-volume channel with blocking send or explicit error handling so critical state transitions are never silently dropped.

#### S-002 — DaemonClient: reqwest Client builder failure hides timeout configuration

**File**: `crates/nexus42/src/api/daemon_client.rs:86-93`

```rust
let http = reqwest::Client::builder()
    .connect_timeout(connect_timeout)
    .timeout(request_timeout)
    .build()
    .unwrap_or_else(|e| {
        tracing::error!(error = %e, "Failed to build reqwest Client, using default");
        reqwest::Client::new()
    });
```

If `Client::builder()` fails (which would be an unexpected environment issue), the fallback creates a client with **no timeouts**. This could lead to infinite hangs. The fallback to `Client::new()` is a silent degradation — better to panic with a clear error message since timeout configuration is a hard requirement.

#### S-003 — list_sessions pagination: cursor ordering is non-deterministic

**File**: `crates/nexus-daemon-runtime/src/api/handlers/agent_host.rs:290-298`

```rust
.skip_while(|s| {
    params.cursor.as_ref()
        .is_some_and(|cursor| s.id.to_string() <= *cursor)
})
```

Sessions from `list_sessions()` are returned in HashMap iteration order (from `SessionRegistry`), not in a deterministic sort. UUID-based cursors with `<=` comparison on unsorted collections mean pages may contain duplicate or missed items between calls. If the underlying collection order changes between paginated requests (e.g., a session is removed), the cursor can skip items unpredictably.

**Fix**: Sort sessions by creation timestamp or a monotonically increasing sequence number before applying cursor pagination.

#### S-004 — Request ID generation: consider collision-resistant algorithm

**File**: `crates/nexus-daemon-runtime/src/api/middleware.rs:152-166`

```rust
let mixed = count.wrapping_add(millis as u64);
format!("req_{:013}", base62_encode(mixed))
```

The `AtomicU64` counter + timestamp mix via `wrapping_add` is lightweight but deterministic. Under extreme contention (millions of requests/second), the wrapping nature combined with identical timestamps could produce collisions. In practice unlikely for the daemon's expected load, but consider using a UUIDv7 or ULID for stronger uniqueness guarantees at minimal cost.
