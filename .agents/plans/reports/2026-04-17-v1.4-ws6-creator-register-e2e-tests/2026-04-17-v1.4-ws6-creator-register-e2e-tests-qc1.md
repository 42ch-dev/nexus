## Findings

### W-1: Missing "stored-token absence" negative test (Medium)

**Location**: Plan architecture description (┬з"Architecture")
**Root cause**: The plan explicitly lists three failure paths: "bad challenge, verification 4xx, stored-token absence." The first two are covered by T3 tests; the third is absent.

**What is missing**: A test that runs `creator register` with an auth.json that has **no** `access_token` in any creator entry (or an empty `creators` map). The CLI's `obtain_auth_token` should return `CliError::AuthenticationRequired`, and the test should assert a non-zero exit with a user-visible error.

**Impact**: This is a real user-facing failure mode (first-time CLI user without prior auth). The existing `creator.rs` unit tests cover `obtain_auth_token` on an empty store, but the E2E integration does not validate the full CLI error path end-to-end.

**Recommendation**: Add a test:

```rust
#[tokio::test]
async fn creator_register_fails_without_auth_token() {
    let mock = MockServer::start().await;
    let home = tempfile::tempdir().unwrap();
    let nexus_dir = home.path().join(".nexus42");
    std::fs::create_dir_all(&nexus_dir).unwrap();
    // Write config.json only тАФ no auth.json
    let config = serde_json::json!({
        "platform_url": &mock.uri(),
        "daemon_url": "http://127.0.0.1:8420",
        "runtime_mode": "local_first"
    });
    std::fs::write(nexus_dir.join("config.json"), serde_json::to_string_pretty(&config).unwrap()).unwrap();

    let output = cli_cmd(home.path())
        .args(["creator", "register", "no-auth-creator"])
        .output()
        .expect("run nexus42");

    assert!(!output.status.success(), "should fail without auth token");
    let combined = format!(
        "{}{}",
        String::from_utf8_lossy(&output.stdout),
        String::from_utf8_lossy(&output.stderr)
    );
    assert!(
        combined.contains("auth") || combined.contains("Auth"),
        "error should mention authentication; got: {combined}"
    );
}
```

Alternatively, document this as an explicit residual (non-blocking) if PM decides coverage is sufficient via the existing unit tests in `creator.rs`.

### N-1: Idempotency test does not assert second-run exit status (Nit)

**Location**: `tests/creator_register_e2e.rs:233-280` (`creator_register_twice_does_not_double_write_credentials`)

The test captures `output2` from the second `creator register` call but deliberately does **not** assert `output2.status.success()` (comment: "Regardless of success/failure on second run, the credential for crt_e2e_test_12345 must still contain a valid api_key").

This is a defensible design choice тАФ the comment correctly identifies that the CLI currently lacks an idempotency guard. However, the test should still record the second-run exit code for regression tracking:

```rust
let second_run_succeeded = output2.status.success();
// Lock current behavior: second run currently succeeds (mock returns same response).
// If CLI adds an idempotency guard, this assertion should change to `!second_run_succeeded`.
assert!(
    second_run_succeeded,
    "second run currently succeeds; if this changes, update the assertion"
);
```

Without this, a future change that accidentally corrupts auth.json on the second run could go undetected if the CLI returns non-zero but the credential file still exists with the correct value.

### N-3: Verify rejection uses HTTP 200, not 4xx as plan suggested (Nit)

**Location**: `tests/creator_register_e2e.rs:153-157` vs plan Task 3, Step 2

The plan's template showed `ResponseTemplate::new(400)` for verify rejection. The implementation uses `ResponseTemplate::new(200)` with `{"status": "wrong_answer"}`.

**Assessment**: This is **correct**. The actual `PlatformClient` in `platform_client.rs` only treats HTTP status >= 400 as an error (`SyncError::PlatformError`). The platform contract uses HTTP 200 with a status enum body to communicate verification results. The test correctly mimics real platform behavior. This is a positive deviation from the plan.
