## Findings

### M1 — Missing HTTP 5xx / network-error coverage for verify endpoint *(Medium)*

**Location**: `crates/nexus42/tests/creator_register_e2e.rs`  
**Cross-reviewer**: Recommend @qc-specialist-1 verify if this is in-scope for E2E vs. unit-test coverage.

The `submit_with_retry` function in `creator.rs` handles `VerifyStatus::WrongAnswer` with auto-retry and also has a branch for `Err(CliError::Network(_))` (lines 386–392). The E2E tests cover the `wrong_answer` retry-exhaustion path via a 200 response with `wrong_answer` status, but **no test exercises an actual HTTP 500 or network timeout** from the verify endpoint.

**Impact**: If the platform changes its error representation (e.g. moves from HTTP 200+business-status to HTTP 4xx/5xx for verification failures), the E2E suite would not catch a regression in the CLI's error-handling path.

**Recommendation**: Add a negative test that mocks `POST /api/v1/creators/verify` with HTTP 502/503 to verify the retry-and-fail behavior. Alternatively, document in the test file that network-error coverage is delegated to the `submit_with_retry` unit tests in `creator.rs`.

### M2 — No challenge expiry boundary test *(Medium)*

**Location**: `crates/nexus42/tests/fixtures/creator_register/register_response.json`  
**Cross-reviewer**: Recommend @qc-specialist-3 confirm if boundary testing is part of this plan's acceptance criteria.

The happy-path fixture sets `expires_at` to `"2099-12-31T23:59:59.000Z"`, which is far in the future and will never trigger the expiry guard in `register_creator` (lines 266–275). The `register_creator` function has a **buffered expiry check** (`EXPIRY_BUFFER_SECS = 10`) and a **post-solve re-check** (lines 296–303), neither of which is exercised by the E2E suite.

**Impact**: If the challenge expiry logic drifts (e.g. timezone handling bug, buffer math error), the E2E tests will not catch it.

**Recommendation**: Add a fixture with `expires_at` in the near past (or use a mock that returns a backdated timestamp) and assert that the CLI exits with `ChallengeExpired`. This is a straightforward addition (~15 lines).

### L1 — Idempotency test does not assert second-run exit code *(Low)*

**Location**: `crates/nexus42/tests/creator_register_e2e.rs`, `creator_register_twice_does_not_double_write_credentials`  
**Cross-reviewer**: @qc-specialist-1 — please cross-check if your review found the same gap.

The idempotency test runs the CLI twice with the same creator name. It asserts that:
- The first run succeeds
- The credential file is not corrupted after the second run
- The seed token is preserved

However, it **does not assert `output2.status.success()` or its inverse**. The test comment explicitly states: "the CLI currently has no idempotency guard, so the second run will re-register." This codifies observed behavior rather than testing a guard.

**Impact**: If the CLI later adds an idempotency guard (e.g. returns error on duplicate register), this test will not detect the behavior change — it only checks data integrity.

**Recommendation**: Add an assertion for `output2.status.success()` (or `!success()`, depending on intended behavior) so the test becomes a true regression guard for the idempotency policy, not just a data-integrity check.

### L3 — Windows test portability concern *(Low)*

**Location**: `crates/nexus42/tests/creator_register_e2e.rs`, `cli_cmd` / `seed_home`  
**Cross-reviewer**: @qc-specialist-3 — please flag if this is out-of-scope for current CI targets.

The tests set `HOME` via `Command::env("HOME", home)` to redirect the CLI's data directory. On Windows, `dirs::home_dir()` (used by `nexus_home()` in `config.rs`) reads `USERPROFILE`, not `HOME`. The test will fail on Windows because the CLI will write to the real user home instead of the temp directory.

**Impact**: Tests cannot run on Windows CI runners without modification.

**Recommendation**: Document the Unix-only assumption in a module-level comment, or switch to setting both `HOME` and `USERPROFILE` env vars when `cfg(windows)`.

### N2 — Mocks do not verify request body shape *(Nit)*

**Location**: `crates/nexus42/tests/creator_register_e2e.rs`

All wiremock stubs match only on `method("POST")` and `path("/api/v1/creators/...")`. They do not assert the structure of the request body (e.g. that `register` sends `{display_name, registration_source}` or that `verify` sends `{verification_code, answer}`).

**Recommendation**: For stricter regression catching, add `body_json` or `body_string_contains` matchers to the mocks. This is optional for E2E tests — unit tests in `platform_client.rs` already cover serialization correctness.
