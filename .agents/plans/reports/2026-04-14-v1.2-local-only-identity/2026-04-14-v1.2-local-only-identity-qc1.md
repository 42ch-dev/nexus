## Critical (blocking)

### C1: `link_to_platform` DB layer misuses `QueryReturnedNoRows` for "already linked" — callers cannot distinguish error conditions

**File**: `crates/nexus-local-db/src/identity.rs`, lines 106-120

```rust
if affected == 0 {
    let existing = get_local_identity(conn, creator_id)?;
    match existing {
        Some(row) if row.platform_linked => {
            return Err(LocalDbError::Rusqlite(rusqlite::Error::QueryReturnedNoRows));
        }
        None => {
            return Err(LocalDbError::Rusqlite(rusqlite::Error::QueryReturnedNoRows));
        }
        _ => {
            return Err(LocalDbError::Rusqlite(rusqlite::Error::QueryReturnedNoRows));
        }
    }
}
```

**Problem**: All three branches return the *identical* error `LocalDbError::Rusqlite(rusqlite::Error::QueryReturnedNoRows)`, making it impossible for callers to distinguish:
- "identity not found" (should be `NotFound`)
- "identity already linked" (should be `AlreadyLinked` / `AlreadyInState`)

The domain model already has `DomainError::AlreadyInState` for this case, but the DB layer buries the semantic distinction inside a raw rusqlite error. The CLI `link_identity()` calls `link_to_platform()` directly (bypassing the domain model), so it cannot provide a differentiated error message to the user.

**Impact**: Users who try to link an already-linked identity get the same error as users who mistype a creator ID. Downstream consumers (WS2b runtime) will not be able to handle these cases differently.

**Fix**: Add a `AlreadyLinked` (or `IdentityAlreadyLinked { creator_id: String }`) variant to `LocalDbError`, and return it when `platform_linked = true`. Return a `NotFound` variant when the identity doesn't exist. Remove the blanket `QueryReturnedNoRows` wrapping.

## Warning (should-fix)

### W1: `From<nexus_contracts::LocalIdentity>` silently defaults unknown `identity_type` to `Anonymous`

**File**: `crates/nexus-domain/src/local_identity.rs`, lines 127-129

```rust
identity_type: LocalIdentityType::from_str(&c.identity_type)
    .unwrap_or(LocalIdentityType::Anonymous),
```

**Problem**: When deserializing contract data with an unknown or corrupted `identity_type` string, the conversion silently defaults to `Anonymous`. This masks data corruption and makes it impossible to detect version incompatibilities. If a future schema version adds `"managed"` as an identity type, older builds would silently treat it as anonymous.

**Fix**: Return `Result` from the conversion, or use `DomainError::InvalidIdFormat` / a new `InvalidIdentityType` variant to surface the error.

### W2: `link_to_platform` in DB layer bypasses domain `platform_creator_id` format validation

**File**: `crates/nexus42/src/commands/identity.rs`, lines 205-210

```rust
fn link_identity(creator_id: String, platform_id: String) -> Result<()> {
    let conn = open_global_db()?;
    link_to_platform(&conn, &creator_id, &platform_id)?;  // ← no format validation
    ...
}
```

**Problem**: The domain model's `LocalIdentity::link_to_platform()` validates `platform_creator_id` format via `is_valid_creator_id()`, but the CLI command calls the DB function directly, which accepts any string. A user could run `nexus42 identity link ctr_local123 --platform_id "not-a-creator-id"` and it would be stored without validation.

**Fix**: Either (a) have the CLI call the domain model's `link_to_platform` before persisting, or (b) add format validation to the DB layer's `link_to_platform`. Option (a) is preferred as it preserves the domain-invariant pattern used elsewhere.

### W3: `#[allow(dead_code)]` cluster — `resolve_active_identity` and related types not wired into any command

**File**: `crates/nexus42/src/commands/identity.rs`, lines 221-310

The following are all `#[allow(dead_code)]`:
- `resolve_active_identity()` (line 221)
- `ResolvedIdentity` struct (line 283)
- `ResolvedIdentity::is_ephemeral()` (line 296)
- `ResolvedIdentity::ephemeral_warning()` (line 301)

**Problem**: These functions are defined but never called by any CLI command or other module. They appear to be prepared for WS2b runtime integration but are not yet connected. While acceptable during staged implementation, the `#[allow(dead_code)]` suppressions reduce confidence that the resolution logic will work when wired in.

**Fix**: Either integrate at least one consumer in this plan, or add a `#[cfg(feature = "local_only_runtime")]` gate (if such a feature exists). If this is intentional staging, document in the plan that WS2b will wire these functions.

### W4: Minimal CLI test coverage — only 1 trivial test

**File**: `crates/nexus42/src/commands/identity.rs`, lines 312-321

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_kind_label() {
        assert_eq!(kind_label(&IdentityKindArg::Anonymous), "anonymous");
        assert_eq!(kind_label(&IdentityKindArg::Persistent), "persistent");
    }
}
```

**Problem**: The CLI identity module has only a single trivial test for a private helper function. All command handlers (`list_identities`, `create_identity`, `use_identity`, `link_identity`, `resolve_active_identity`) have zero test coverage. Compare to domain (16 tests) and DB layer (9 tests). Untested paths include:
- `use_identity` with nonexistent ID (error path)
- `create_identity` persisting to DB (integration path)
- `link_identity` error propagation
- `resolve_active_identity` anonymous vs persistent vs platform branching

**Fix**: Add at minimum tests for `use_identity` (not-found error), `resolve_active_identity` (all three branches), and `create_identity` (anonymous doesn't persist, persistent does). Use `Connection::open_in_memory()` + `init()` as the DB layer tests do.

### W5: No validation of `display_name` for empty/whitespace-only strings

**Files**: `crates/nexus-domain/src/local_identity.rs` line 89, `crates/nexus42/src/commands/identity.rs` line 108

**Problem**: `create_persistent(Some("  "))` and `nexus42 identity create --persistent --name "  "` both create identities with whitespace-only display names. The schema allows any string with `maxLength: 100` but no `minLength` constraint. The domain and CLI layers do not validate.

**Fix**: Add `minLength: 1` to the JSON Schema `display_name` property, and validate in `create_persistent()` with `DomainError::ValidationError`.

## Suggestions (nice-to-have)

### S1: Consider using `thiserror` for `LocalDbError`

**File**: `crates/nexus-local-db/src/error.rs`

The domain module uses `thiserror` for ergonomic error handling (`#[derive(Debug, Error, PartialEq)]`). `LocalDbError` uses manual `impl Display` and `impl Error`. Aligning with `thiserror` would reduce boilerplate and improve consistency across the crate boundary.

### S2: Duplicate `is_valid_creator_id` logic — consider centralizing

**File**: `crates/nexus-domain/src/local_identity.rs`, lines 193-195

```rust
fn is_valid_creator_id(s: &str) -> bool {
    s.starts_with("ctr_") && s.len() > 4 && s[4..].chars().all(|c| c.is_ascii_alphanumeric())
}
```

Other modules (e.g., `nexus-domain/src/creator.rs`) likely have similar CreatorId validation. If not already centralized, consider adding a shared `validate_creator_id()` function to avoid drift as the pattern evolves.

### S3: Migration `migrate_v2` creates a table that `init_shared_tables` also creates

**File**: `crates/nexus-local-db/src/migration.rs` line 38, `crates/nexus-local-db/src/schema.rs` line 168

Since `init_shared_tables` already includes `LOCAL_IDENTITIES_TABLE` via `CREATE TABLE IF NOT EXISTS`, the v2 migration is effectively a no-op when `init()` has already run. This is safe but may confuse future maintainers reviewing the migration history. Consider adding a comment explaining why the migration is idempotent-safe, or use a `SELECT` probe in the migration to detect pre-existing tables.

### S4: `resolve_active_identity` returns `"platform"` type for unknown `ctr_` IDs — not a valid enum value

**File**: `crates/nexus42/src/commands/identity.rs`, lines 246-255

```rust
Ok(Some(ResolvedIdentity {
    ...
    identity_type: "platform".to_string(),  // ← not in LocalIdentityType enum
    ...
}))
```

If `ResolvedIdentity` is ever serialized (e.g., for JSON output or daemon communication), `identity_type: "platform"` would not match the JSON Schema enum `["anonymous", "persistent"]`. This could cause downstream parsing failures.

### S5: `cargo fmt` shows pre-existing import-ordering drift in generated files

**Tool output**: `cargo fmt --check` reports import ordering differences in many `crates/nexus-contracts/src/generated/*.rs` files. This is a pre-existing issue (stable vs nightly `rustfmt` behavior with the `.rustfmt.toml` `ignore` directive), not caused by this change. Worth noting for the team but not blocking.

8. **Good CLI UX**: `identity list` output shows active mark, linked status, and helpful hints when no identities exist. The `create` command auto-sets the new identity as active and provides clear ephemeral vs persistent guidance.
