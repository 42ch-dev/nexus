## Rev2 Verification of Prior Findings

### W-1 (W3): CapabilityRegistry O(n) linear scan — FIXED

**Status: Resolved.**

The `CapabilityRegistry` in `crates/nexus-orchestration/src/capability/mod.rs` now builds a `HashMap<&'static str, usize>` index eagerly in **all constructors**:

- `with_builtins()` → `build_index()`
- `with_builtins_and_pool(...)` → `build_index()`
- `with_runtime_deps(...)` → `build_index()`
- `empty()` → `build_index()`

The `get()` method now performs O(1) amortized lookup via the pre-built index:

```rust
pub fn get(&self, name: &str) -> Option<&dyn Capability> {
    let idx = self.index.as_ref()?;
    let pos = idx.get(name)?;
    Some(self.capabilities[*pos].as_ref())
}
```

The `HashMap` keys are `&'static str` (from `cap.name()`), and the index is built once at construction with no mutation paths afterward. This correctly eliminates the O(n) linear scan identified in round 1. The existing test `registry_lookup_each_builtin` continues to pass, confirming all 17 built-in capabilities are reachable.

### 🟢 Suggestion

**S-4: `collect_asset_file_references` traversed redundantly per validation request**

The daemon `validate_preset` endpoint calls three validation surfaces that each independently traverse the manifest to collect asset file references:

1. `validate_path_safety` → `check_path_safety` → `collect_asset_file_references`
2. `validate_preset_semantic` → does NOT call `collect_asset_file_references`
3. `validate_assets_in_bundle` → `check_asset_file_existence` + `check_symlink_escapes` → `collect_asset_file_references` (twice)

This means a preset with *n* asset references is traversed up to **3×** per API request. For an authoring workflow that validates on every keystroke, this could become noticeable. The manifest is small (YAML < 1 MiB), so this is minor, but coalescing the collection into a single pass would be cleaner.

**Source:** `crates/nexus-orchestration/src/preset/validation.rs:416-499`
**Confidence:** Medium

**S-5: `CapabilityRegistry` rebuilt on every `validate_preset` API call**

The daemon endpoint constructs `CapabilityRegistry::with_builtins()` fresh on each request:

```rust
let caps = nexus_orchestration::CapabilityRegistry::with_builtins();
```

This builds all 17 capability instances and the HashMap index on every call. Since the built-in registry is immutable, a `once_cell::sync::Lazy` singleton would eliminate this per-request allocation overhead without adding global mutable state.

**Source:** `crates/nexus-daemon-runtime/src/api/handlers/preset_management.rs:295`
**Confidence:** Medium

**S-6: Loader and endpoint have divergent path-safety coverage for capability args**

The new `collect_asset_file_references` in `validation.rs` inspects `enter[].args.prompt_file` and `enter[].args.system_prompt_file` for path safety. The loader's `collect_template_file_entries` (in `loader.rs`) does **not** inspect capability args — it only checks `template_file` fields in known manifest locations. This means:

- A preset with `enter[].args.prompt_file: "../../etc/passwd"` will **pass** loader validation.
- The same preset will **fail** API endpoint validation.

This is a parity gap. The loader should ideally also check capability argument paths, or the endpoint should document that it performs stricter checks.

**Source:** `crates/nexus-orchestration/src/preset/loader.rs:363-405` vs `crates/nexus-orchestration/src/preset/validation.rs:416-499`
**Confidence:** Medium
