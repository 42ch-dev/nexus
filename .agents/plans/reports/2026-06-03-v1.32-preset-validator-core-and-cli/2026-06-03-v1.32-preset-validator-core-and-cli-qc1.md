## Findings

### 🔴 Critical

- **C-001 — Runtime loader does not use the new semantic validation facade, so P1 does not deliver the advertised single validation surface.**
  - Evidence: `validation.rs` documents `validate_preset_semantic()` as the shared surface for runtime loader and daemon API (`crates/nexus-orchestration/src/preset/validation.rs:1-17`, `124-157`), and `mod.rs` repeats that claim (`crates/nexus-orchestration/src/preset/mod.rs:11-24`). However, the actual loader path remains unchanged: `load_preset_from_str_with_limits()` calls only private `validate_manifest(&manifest, caps)` and never calls `validate_preset_semantic()` (`crates/nexus-orchestration/src/preset/loader.rs:197-209`). `load_preset()` only adds the older template sandbox check before delegating to `load_preset_from_str()` (`loader.rs:270-280`).
  - Impact: New A2/A4 checks such as initial-to-terminal reachability, terminal marker/header mismatch, orphan inner-graph warning semantics, and capability argument drift are API-side/unit-test diagnostics, not runtime-loader rejection semantics. This violates plan acceptance #1 (one semantic surface used by runtime loading and user-facing validation paths) and compass §2.1's target shape.
  - Required fix: Route `load_preset_from_str_with_limits()` / `load_preset()` through the new facade (or make the facade wrap the existing loader validator plus additive semantic checks) and convert error diagnostics into `PresetLoadError::Validation` for error-severity items while preserving warnings in an explicit, tested way.

- **C-002 — The daemon validate endpoint no longer runs loader-equivalent structural validation, creating API/runtime drift for existing defects.**
  - Evidence: `validate_preset()` replaced the old `validate_manifest_fields()` path with `validate_preset_semantic()` plus optional asset checks (`crates/nexus-daemon-runtime/src/api/handlers/preset_management.rs:290-333`). The new semantic validator intentionally skips missing `preset.initial` because it assumes loader validation already caught it (`validation.rs:221-226`), and it never reports a missing `preset.terminal` state unless a matching state exists but lacks `terminal: true` (`validation.rs:267-295`). The loader still has those structural checks (`loader.rs:455-469`).
  - Impact: `POST /v1/local/presets:validate` can report `valid: true` for a manifest that the runtime loader rejects, e.g. a manifest whose `preset.terminal` names a nonexistent state while another reachable state has no `next`/is terminal. This is the exact drift P1 is meant to remove and is a regression from the deleted endpoint-local checks.
  - Required fix: Ensure the daemon endpoint invokes a validator that includes loader structural validation before/additively with semantic diagnostics. Add daemon tests for missing `preset.initial`, missing `preset.terminal`, bad `next`, terminal-with-next, and unsupported conditional `next` so API parity cannot regress.

### 🟡 Warning

- **W-001 — Required P1 regression coverage is incomplete for the presentation path.**
  - Evidence: `cargo test -p nexus-daemon-runtime preset` passes, but the daemon handler test suite only includes a valid preset test for `validate_preset()` (`preset_management.rs:515-547`). It does not cover endpoint rejection for reachability, terminal/header mismatch, id mismatch, missing/orphan inner graphs, missing prompt/template files, unsafe paths, or capability drift.
  - Impact: The current tests do not prove A5 validation-surface parity or plan acceptance #2/#4 for the shipped Local API presentation path. The Critical drift above would have been caught by endpoint-level invalid fixture tests.
  - Required fix: Add invalid fixture tests at the daemon/API layer for each P1 presentation-path acceptance item, not only orchestration unit tests.

### 🟢 Suggestion

- **S-001 — Consider factoring diagnostic formatting into orchestration rather than reconstructing strings in the daemon handler.**
  - Rationale: The core validator now exposes structured `severity` and `category`, but `ValidatePresetResponse` still flattens diagnostics into `Vec<String>` (`preset_management.rs:305-323`). A small shared formatter or response DTO adapter would reduce future CLI/API drift if a CLI wrapper is added later.
