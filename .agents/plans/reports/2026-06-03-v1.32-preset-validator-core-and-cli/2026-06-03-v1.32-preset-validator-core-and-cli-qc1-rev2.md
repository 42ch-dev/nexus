- **New architecture concerns introduced by the fix:** The new validator API is split across `validate_preset_semantic()`, `validate_path_safety()`, and `validate_assets_in_bundle()`, but runtime loading only uses the first of these. The daemon endpoint uses all three. This split makes it easy for future callers to believe they are using the shared facade while still missing bundle-level validation.

## Findings

### 🔴 Critical

- **C-001 — Runtime bundle loading still does not use the full shared A3 validation surface, so API/runtime parity remains broken for missing files and prompt-file path safety.**
  - Evidence: The plan requires one semantic validation surface for runtime loading and user-facing validation paths, and tests/equivalent failures for missing prompt/template files and unsafe paths (`.agents/plans/2026-06-03-v1.32-preset-validator-core-and-cli.md:94-99`; compass §2.1/§2.2 at lines 105-135). Rev2 added `validate_assets_in_bundle()` and `validate_path_safety()` in `validation.rs`, and the daemon endpoint invokes both (`preset_management.rs:313-325`). The runtime bundle loader does not: `load_preset()` only runs the older `validate_template_files_in_sandbox()` and then delegates to `load_preset_from_str()` (`loader.rs:302-312`), while `load_preset_from_str_with_limits()` only invokes `validate_preset_semantic()` (`loader.rs:211-241`). The older sandbox helper explicitly treats missing referenced files as acceptable (`loader.rs:446-450`) and its collection path only covers `template_file` fields, not capability `args.prompt_file` / `args.system_prompt_file` (`loader.rs:359-405`, `731-739`).
  - Impact: The shipped Local API can reject a bundle that runtime loading still accepts, e.g. a bundle with a missing `exit_when.template_file`, or a capability `prompt_file`/`system_prompt_file` that is unsafe or missing. That is precisely the validator drift P1 is intended to remove. It also leaves `mod.rs`/`validation.rs` documentation claiming a single shared validation surface while callers must manually remember three separate functions.
  - Required fix: Make the runtime bundle path (`load_preset()`) invoke the same bundle-level validator used by the daemon endpoint, or introduce a single orchestration facade that composes structural + semantic + path-safety + bundle-asset checks for both runtime and API callers. Add orchestration-side loader tests proving `load_preset()` rejects missing file-backed assets and unsafe `prompt_file`/`system_prompt_file` references with the same fatality as the daemon endpoint.
