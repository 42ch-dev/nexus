## Findings

### 🔴 Critical

- **C-001: Path safety for asset references (`..` traversal, absolute paths, links outside sandbox) is not reliably or uniformly enforced on the daemon validate surface, creating security drift vs loader and incomplete A3 delivery.**
  - Evidence (from diff + file reads):
    - Daemon `validate_preset` (handler:293-303) calls `validate_preset_semantic` (pure in-memory A2/A4) then *conditionally* `validate_assets_in_bundle` *only if* `infer_bundle_root(file_path)` (i.e. client `path` ends exactly with `preset.yaml`).
    - `validate_assets_in_bundle` (validation:169-177) only does existence + `check_symlink_escapes`; it never calls (or duplicates) the loader's `assert_template_file_safe` string checks for `..`, `/` prefix (absolute), `\`, null, control chars.
    - `collect_asset_file_references` (validation:403-486) is a reimplementation (comment: "re-collect here for independence") that unconditionally includes `roles[*].system_prompt_file` (even empty strings) and all `prompt_file`/`system_prompt_file`/`template_file` without pre-filtering or structural safety.
    - `check_asset_file_existence` (489-512) does raw `bundle_root.join(rel_path).exists()` — no safety.
    - `check_symlink_escapes` (517-543): `canonicalize` + `starts_with` only; if `bundle_root.canonicalize()` fails → early return (no PathSafety added); if resolved target !exists on host FS → no PathSafety (existence check already emitted MissingAsset with the bad rel_path visible); absolute paths are "caught" only because `join(absolute)` ignores base and canonicalize fails the starts_with (message still says "symlink escape").
    - No tests exercise escape cases (search for PathSafety / "resolves outside" / symlink only finds the message string + 2 asset tests that are positive/negative for existence only, lines 1029-1099).
    - Loader still does its own `assert_template_file_safe` (loader:300-321, called in load path 682-684) + `validate_template_files_in_sandbox` (382-417, nearly identical canonical logic) and is *not* updated to use facade (mod.rs:16 explicitly says old `validate_manifest` continues for BC; grep for calls to new fns in loader returns none).
    - In daemon, when bundle mode, a ref like `prompt_file: "../../etc/passwd"` (if /etc/passwd exists) → PathSafety Error; if target does not exist on this host → MissingAsset (still blocks, but wrong category and message leaks the traversal attempt + full bundle_root path). Standalone yaml (non-`preset.yaml` name) → asset skipped → traversal ref accepted.
  - Impact: Violates plan A3 ("Reject traversal, absolute paths, and links outside the preset bundle sandbox"), compass §1.1 in-scope item 3, spec §7.6 ("Any `template_file` path escapes the bundle root or does not exist"). The "single shared validation surface" (validation.rs:1-17, mod.rs:11-24, handler doc 244-246) does not actually make daemon and loader reject the *same* path-safety defects. A crafted preset YAML can pass the user-facing validate endpoint (in common standalone or non-triggering naming) yet be rejected at runtime load (or vice-versa on FS state). This is a security boundary gap (A3 is explicitly a sandbox enforcement for untrusted preset content).
  - Required fix: 
    1. Add structural path safety (port of `assert_template_file_safe` logic or expose from loader) to the facade as an unconditional check (independent of bundle_root) — call it from `validate_preset_semantic` or a new `validate_preset_structure` entry.
    2. Always invoke path safety from daemon `validate_preset` (even for non-bundle).
    3. Make `validate_assets_in_bundle` (or a combined) the place that also enforces id/dir match + safety.
    4. Add dedicated tests (unit + daemon handler) for: `../` traversal (exist + nonexist target), absolute `/`, backslash, null, control, and actual symlink escape (create tmp symlink pointing outside). Assert `PathSafety` category + `!valid`.
    5. Wire facade into loader load paths (addresses QC1 C-001) so the checks are not duplicated.
    6. Filter empty refs in collect; make escape check safety-first (before or instead of raw exists); canonicalize request path in handler; improve error messages (do not hardcode "symlink escape" for non-symlink cases; consider omitting full FS paths or making them optional).

- **C-002: `check_bundle_id_match` only validates manifest id *format*; the actual "Bundle directory id vs manifest id mismatch" (plan A2, spec §7.2, compass) is never implemented or called.**
  - Evidence: Function at validation:303-338 checks `id.is_empty()`, invalid chars, !starts_with lowercase; category=IdMismatch; docstring even says "When loading from a bundle directory, the caller should compare the directory name with `manifest.preset.id`." But `validate_assets_in_bundle` receives `bundle_root` and never compares `bundle_root.file_name()` to `id`. Daemon `infer_bundle_root` + call (handler:298-302) does not pass or check dir name. No other call site does the match. (Loader also lacks it per similar grep.)
  - Impact: Presets with `preset.id` != containing directory name are accepted by the new validate surface (and likely loader). This is an explicit A2 acceptance item and spec contract violation. Correctness + potential for confusion/attack (mismatched identity between declared preset and on-disk bundle).
  - Required fix: Implement the actual comparison inside `validate_assets_in_bundle` (when a root is provided): if `manifest.preset.id != dir_name` → IdMismatch Error. Add to facade entry, tests (positive + negative), and call from loader too. Update `check_bundle_id_match` name/docs or split format vs match.

### 🟡 Warning

- **W-001: Daemon validate endpoint performs no sanitization/canonicalization on the client-provided `path` before using it for FS reads or as bundle root; combined with detailed path-containing errors, increases risk of FS layout disclosure or traversal in request affecting checks.**
  - Evidence: handler:254 (`Path::new(&req.path)`), 256 (exists), 264 (metadata), 276 (read_to_string), 340 (`infer_bundle_root` just checks file_name == "preset.yaml" and takes `.parent()` — no canonicalize, no starts_with under home or allowed dirs). No State (unlike scaffold/reload). Errors echo: NotFound includes the raw path (257), asset Missing includes `bundle_root.display()` (validation:505), parse errors include full yaml details. Request path with `..` or absolute is accepted if the target file exists on host.
  - Impact: While the endpoint is local-only (client is trusted in current threat model), this is a regression in "sandbox" thinking for a security-sensitive operation (validating untrusted preset content that itself contains FS paths). Errors can leak internal paths (e.g. if validate errors are logged centrally or surfaced via other channels). The path safety checks inside assets then run against a client-controlled root, which the escape check tries to defend but only after the read has already happened.
  - Required: Canonicalize the input `file_path` early (reject if fails or outside expected user preset tree if policy). Document the "arbitrary path" contract explicitly. Minimize full paths in error strings returned to clients (or gate behind debug/verbose). Add handler-level tests with crafted paths containing traversal.

- **W-002: Capability arg drift and schema parsing is defensive (no panic on malformed `input_schema`), but best-effort nature + "schema check skipped" warnings are not strongly tested for security-relevant malformed cases; latent correctness risk if schema strings ever carry privileged semantics.**
  - Evidence: check_capability_arg_compatibility (validation:557-622) does `if let Ok(schema_value) = serde_json::from_str...` else Warning SchemaCheckSkipped; `check_args_against_schema` (629-693) uses only safe `.and_then(as_object/as_array/as_str)` — no unwrap/panic paths. Registry `input_schema()` is trusted (built-in). But no tests for deliberately malformed JSON schema strings, missing "properties"/"required", non-object, etc. in the validation test suite (the capability tests are for unknown cap / missing arg, not schema parse failures).
  - Impact: Per prompt focus: "Capability arg parsing: tolerate malformed `input_schema` strings without panic" — it does tolerate (good), but the "best effort" (doc 627) means unknown args or drift can be accepted as Warning only. If a capability ever uses schema to gate privileged args, this could be a vector. Low current risk (builtins only), but correctness of the A4 check.
  - Required: Add tests with malicious/malformed `input_schema` (e.g. in a test registry) asserting the Warning is emitted and no panic/crash. Consider making SchemaCheckSkipped an explicit note in response.

- **W-003: Error messages and diagnostics include full internal bundle paths and raw rel_paths; while acceptable for local client-initiated validate, this plus the removed old checks creates a broader "information in errors" surface than before.**
  - Evidence: See W-001 + asset errors (validation:504-505 "in bundle at {}", 535), handler NotFound (258), parse Internal errors with yaml details (357,372). Old removed `validate_manifest_fields` errors were abstract ("preset.id must not be empty"); new ones from facade + asset are richer (good) but path-heavy.
  - Impact: Per prompt: "Error message content: does it leak internal paths or sensitive info to clients?" — yes, full FS paths when errors occur. For local use (nexus42 CLI calling daemon) the client already knows the path it sent, so low severity. But if errors are persisted, logged, or the validate surface is called from less-trusted contexts in future, this is a disclosure. Also, the traversal attempts are echoed back in the error (e.g. the bad rel_path appears), which may aid an attacker probing.
  - Required: Review all diagnostic messages for path leakage; prefer relative paths or omit root when possible; add redaction option if paths are sensitive. At minimum, document in spec.

### 🟢 Suggestion

- **S-001: The `check_symlink_escapes` function (and its loader counterpart) should be renamed `check_asset_resolves_within_bundle` or `check_path_safety`; messages should distinguish "traversal/absolute" from "symlink escape" (use the canonical result or a prior string check to decide message/category).**
- **S-002: Run the structural path safety *before* asset existence in the check sequence, so bad-pattern refs always get PathSafety (not MissingAsset) even if the target happens to exist on the host FS. This makes behavior deterministic and category accurate.**
- **S-003: Consider exposing a single `validate_preset(manifest, caps, optional_bundle_root) -> ValidationResult` entry point in the facade that always runs structural (incl path safety string checks) + semantic + (if root) asset/bundle checks. Update both loader and daemon to call only this. Removes dupe collect/check logic and makes "shared surface" real.**
- **S-004: Add a test in daemon handler (and/or integration) that exercises a bundle-named path with an escaping asset ref and asserts the expected error category + valid=false. Current daemon tests only have the happy valid case.**
- **S-005: The addition of `warnings` to ValidatePresetResponse (and the split of errors/warnings in handler:306-323) is a good usability improvement for P1. If a CLI wrapper is added later, ensure it surfaces warnings distinctly (not just as errors).**
- **S-006 (cross-cutting with QC3):** The registry linear scan (inside capability checks) is exercised more now by the new A4 path; the best-effort schema parse + arg checks add a few more string/JSON ops per capability. Acceptable for now (small N), but note for P3 or future.
