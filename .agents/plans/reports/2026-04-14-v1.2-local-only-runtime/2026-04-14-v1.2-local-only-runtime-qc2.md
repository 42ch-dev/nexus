## Summary

**Verdict: Request Changes.**

The T3 implementation is architecturally sound, clippy-clean, and follows the established module patterns. The domain model (`DomainRuntimeMode`), guard module (`runtime_guard.rs`), and daemon integration are well-structured. However, two medium-severity issues must be fixed before merge:

1. **`sync status` is incorrectly blocked** by a top-level guard that applies to ALL sync subcommands, even though `sync status` is a read-only local operation.
2. **`PlatformOperationProhibited` loses error classification** when converted to `CliError`, preventing downstream pattern-matching and making the error message fragile.

These issues independently affect correctness and maintainability. I concur with QC1's findings on both points.

## Warnings

### W1: `sync status` blocked by overly broad `require_platform` guard

**File**: `crates/nexus42/src/commands/sync.rs`, line 222  
**Severity**: Medium

The guard is placed at the top of `run()`, before the `match cmd`:

```rust
pub async fn run(cmd: SyncCommand, config: &CliConfig) -> Result<()> {
    runtime_guard::require_platform(&config.runtime_mode(), "sync")?;
    // ...
}
```

This blocks **all** sync subcommands in `local_only` mode, including `SyncCommand::Status`, which only queries local daemon state (outbox counts, staged/ready/sent/acked/failed/conflicted). Per ADR-017, `local_only` means "no platform HTTP dependency for Creator's end-to-end loop" — checking local sync state does not require platform connectivity and should remain available.

The integration test (`crates/nexus42/tests/integration.rs`, `sync_status_without_daemon`) was changed to expect this incorrect behavior, codifying the regression.

**Fix**: Move `require_platform` into the individual match arms for `Push`, `Pull`, and `Resolve`. Leave `Status` unguarded. Restore the integration test to assert success with the "not running" / daemon-status message when no daemon is present.

**Cross-check**: `publish` and `explore` do not have read-only subcommands in the current implementation, so their top-level guards are acceptable. `context assemble` has only one subcommand and is correctly guarded.

### W2: `DomainError::PlatformOperationProhibited` → `CliError::Other` loses classification

**File**: `crates/nexus42/src/errors.rs`, lines 199–203  
**Severity**: Medium

```rust
impl From<nexus_domain::errors::DomainError> for CliError {
    fn from(err: nexus_domain::errors::DomainError) -> Self {
        CliError::Other(format!("Domain error: {}", err))
    }
}
```

All domain errors, including the new `PlatformOperationProhibited`, are folded into `CliError::Other` with a "Domain error: " prefix. This means:
- Callers cannot pattern-match on `PlatformOperationProhibited`.
- The integration test relies on the string "not available in local_only mode" appearing inside the `Other` message — any change to `DomainError`'s `Display` impl would silently break the test.
- The user-facing output contains the extra "Domain error: " noise.

**Fix**: Add a dedicated `CliError::PlatformOperationProhibited { mode: String, operation: String }` variant and match it explicitly in the `From` impl, then give it a clean `Display` message.

## Suggestions

### S1: JSON Schema draft version inconsistent with plan spec

**File**: `schemas/domain/runtime-mode.schema.json`, line 2

The committed schema uses:
```json
"$schema": "http://json-schema.org/draft-07/schema#"
```

The plan's Task 1 specifies:
```json
"$schema": "https://json-schema.org/draft/2020-12/schema"
```

While this doesn't affect runtime behavior, consistency with the repo's schema draft and the plan spec is preferable. Recommend updating to `draft/2020-12`.

### S2: `classify_operation` defaults unknown operations to `LocalOnly` (fail-open)

**File**: `crates/nexus-domain/src/runtime_guard.rs`, line 43

```rust
_ => OperationCategory::LocalOnly,
```

If a future developer adds a platform-dependent command and forgets to update `classify_operation`, it will be silently allowed in `local_only` mode. Consider adding a `tracing::warn!` for unknown operations, or document this explicit design choice in a comment.

### S3: `auth_register` classified but command variant does not exist

**File**: `crates/nexus-domain/src/runtime_guard.rs`, line 34

`"auth_register"` is mapped to `OperationCategory::PlatformRequired`, but `AuthCommand` has no `Register` variant. Since the CLI uses manual `require_platform` calls rather than `check_operation`, this classification is dead code. Remove it or add a comment until the command is implemented.

### S4: Daemon runtime mode is read once at startup and becomes stale

**File**: `crates/nexus42d/src/workspace/mod.rs`, lines 99–101

The daemon loads `runtime_mode` from `CliConfigSnapshot` during `initialize()` and never re-reads it. If the user runs `nexus42 runtime-mode set ...` while the daemon is running, the status API will report the old mode. Acceptable for T3 (the plan notes T6 handles mode-switch degradation), but this should be documented as a known limitation or tracked as a residual.

### S5: Integration test no longer covers "daemon not running" path for `sync status`

**File**: `crates/nexus42/tests/integration.rs`, lines 271–281

The test was changed from asserting `.success().stdout(predicate::str::contains("not running"))` to `.failure().stderr(predicate::str::contains("not available in local_only mode"))`. Once W1 is fixed, restore the original assertion (or add a new test under `cloud_enhanced` mode) to preserve daemon-connectivity regression coverage.

### S6: `#[serde(rename_all = "snake_case")]` on `DomainRuntimeMode` is misleading

**File**: `crates/nexus-domain/src/runtime_mode.rs`, line 13

`DomainRuntimeMode` is a newtype struct (`pub struct DomainRuntimeMode(pub RuntimeMode)`). The `rename_all` attribute has no effect on newtype serialization — serde serializes it as the inner value. The attribute is redundant and may mislead future maintainers into thinking it controls the outer shape. Remove it; the inner `RuntimeMode` already carries the correct `rename_all`.
