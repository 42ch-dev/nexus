## Summary

The T3 implementation introduces a `local_only` runtime mode with fail-fast guards for platform-dependent operations. The design is sound and aligns with ADR-015/ADR-017. However, there is **one blocking issue**: `sync status` is incorrectly blocked in `local_only` mode despite being a pure local-daemon query. Additionally, `PlatformOperationProhibited` lacks a dedicated `CliError` variant, resulting in a generic `Other` wrapping.

**Verdict**: Request Changes — blocking issue must be fixed before merge.

---

## Critical (blocking)

### R1: `sync status` incorrectly blocked in `local_only` mode

**File**: `crates/nexus42/src/commands/sync.rs:222`

```rust
pub async fn run(cmd: SyncCommand, config: &CliConfig) -> Result<()> {
    runtime_guard::require_platform(&config.runtime_mode(), "sync")?;
```

**Problem**: The guard is placed at the top of the `run` function, blocking **all** sync subcommands including `sync status`. However, `sync status` only queries the local daemon via `/v1/local/sync/status` (line 383) — it makes no platform HTTP call.

**Evidence**:
- `SyncCommand::Status` handler (line 373-419) calls `client.get("/v1/local/sync/status")` — a purely local daemon endpoint
- `sync push/pull` would legitimately require platform, but `sync status` does not

**ADR-017 D3/D5 alignment**: `local_only` should allow local queries. Blocking `sync status` contradicts the ADR-017 intent of "no platform HTTP dependency for Creator's end-to-end loop" — `sync status` is a local status check, not a platform operation.

**Fix**: Move the `require_platform` guard into specific subcommand match arms (`Push`, `Pull`) rather than the top-level function. `sync status`, `sync list`, and `sync resolve` (local-only operations) should be allowed in `local_only` mode.

**Expected behavior**:
```rust
pub async fn run(cmd: SyncCommand, config: &CliConfig) -> Result<()> {
    let client = DaemonClient::from_config(config);
    match cmd {
        SyncCommand::Push { .. } | SyncCommand::Pull { .. } => {
            runtime_guard::require_platform(&config.runtime_mode(), "sync")?;
            // platform-dependent logic
        }
        SyncCommand::Status | SyncCommand::Resolve { .. } => {
            // local-only: no guard needed
        }
    }
}
```

**Impact**: Users cannot check sync queue status in `local_only` mode — a significant UX regression for local-only workflows.

## Warnings

### W1: `PlatformOperationProhibited` not a distinct `CliError` variant

**File**: `crates/nexus42/src/errors.rs:199-202`

```rust
impl From<nexus_domain::errors::DomainError> for CliError {
    fn from(err: nexus_domain::errors::DomainError) -> Self {
        CliError::Other(format!("Domain error: {}", err))
    }
}
```

**Problem**: `DomainError::PlatformOperationProhibited` converts to `CliError::Other("Domain error: ...")` rather than a dedicated error variant. This loses semantic structure — downstream error handling (e.g., exit code categorization, error page matching) cannot distinguish a mode-blocked operation from other domain errors.

**Checklist requirement**: "Error classification: PlatformOperationProhibited should be a distinct CliError variant."

**Suggestion**: Add `CliError::PlatformOperationProhibited { mode: String, operation: String }` and handle it separately. This would also allow setting a specific exit code (e.g., 1 vs 2) for mode violations vs other errors.

**Severity**: Medium — functional but semantically imprecise.

## Suggestions

### S1: Document why `sync` is `PlatformRequired` in `classify_operation`

**File**: `crates/nexus-domain/src/runtime_guard.rs:26-44`

The function classifies `sync` as `PlatformRequired` but doesn't explain that `sync status` is actually local-only. Adding a comment clarifying which subcommands are truly platform-dependent vs local would prevent future maintainers from incorrectly applying the blanket classification.

```rust
// Platform-required operations (blocked in local_only)
// Note: "sync" here refers to push/pull — sync status is local-only
// and is handled separately in the CLI command layer.
"sync" | "publish" | ...
```

### S2: Consider finer-grained operation classification

The current `classify_operation` lumps all of `sync` into `PlatformRequired`. If `sync push` and `sync pull` are the only truly platform-dependent operations, future work could add `sync_status` as a separate local-only operation.

## Cross-Reviewer Ready Notes

**R1 is a functional regression**: `sync status` is a local-only query that should work in `local_only` mode. This is not a test bug — the test correctly captures the current (incorrect) behavior. Fixing R1 will require updating the test to expect success for `sync status`.

**Runtime**: No performance concerns — guards are O(1) enum matching.

**Rollback urgency**: High if this reaches users — `sync status` is a common diagnostic command that would be unavailable in `local_only` mode.

**Effort to fix**: Low — requires moving 1 guard call from top-of-function to specific match arms in `sync.rs`.
