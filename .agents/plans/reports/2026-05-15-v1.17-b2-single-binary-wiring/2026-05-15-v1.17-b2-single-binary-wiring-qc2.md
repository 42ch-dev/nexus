### 🟢 Suggestion

**S-1: PID file atomic write pattern**

Location: `crates/nexus42/src/commands/daemon.rs:284-299`

The `write_pid_file()` function writes directly to the PID file. While permissions are set to 0600 (owner-only), a more robust approach would use an atomic write pattern (write to temp file, then rename) to reduce the TOCTOU window.

Current implementation is safe for single-user scenarios, but could be improved:
```rust
// Current: direct write
std::fs::write(&path, pid.to_string())?;

// Suggested: atomic rename pattern
let temp_path = path.with_extension("pid.tmp");
std::fs::write(&temp_path, pid.to_string())?;
std::fs::rename(&temp_path, &path)?;
```

This is low-priority since the PID file lives in user-owned `~/.nexus42/` with 0600 permissions, making cross-user attacks impractical.

**S-2: Unix socket path validation**

Location: `crates/nexus-daemon-runtime/src/boot.rs:341-356`

The Unix socket transport accepts any `socket_path` from configuration. While stale file cleanup prevents socket hijacking in most cases, additional validation could ensure the path is in a safe directory (e.g., user home or tmpdir).

Current implementation handles the main risk (stale socket removal), but could add:
```rust
// Validate socket path is in safe location
let canonical = path.canonicalize()?;
if !canonical.starts_with(dirs::home_dir()?) && !canonical.starts_with("/tmp") {
    anyhow::bail!("Socket path must be in home or /tmp");
}
```

This is informational; no immediate risk identified.

**S-3: Hidden command documentation**

Location: `crates/nexus42/src/cli.rs:166-167`, `crates/nexus42/src/commands/daemon_run.rs:1-10`

The `__internal daemon-run` hidden command is correctly marked with `#[command(hide = true)]`. The documentation could explicitly state that this command should not be invoked directly by users — it's an internal entry point only called by the parent `nexus42 daemon start` process.

Current documentation says "Hidden: Internal daemon-run entry point (self-spawned by daemon start)" which is adequate, but a stronger warning could be added to the module-level docs.
