### 🟡 Warning

#### W-001: Lifecycle HSM never transitions to Running state

- **Location**: `crates/nexus-daemon-runtime/src/boot.rs:218-222`, `state.rs:411-418`
- **Issue**: The lifecycle HSM is created with `new_with_subsystems()` which uses `DaemonHsm::default()` (no `ActionContext`). Per state.rs:411-418, this is intentional — entry actions log but don't spawn subsystem tasks. The result is:
  1. `ProcessStarted` event is dispatched (boot.rs:319) but falls through to Top and is ignored
  2. No `SubsystemUp` events are ever dispatched
  3. The machine stays in `Starting` state forever
  4. `wait_until_terminal()` only blocks on `Failed` state, so the program continues normally
- **Impact**: Moderate — the lifecycle state visible via HTTP API will always report "starting", not "running". This could confuse monitoring systems that expect proper state transitions.
- **Root Cause**: Subsystems are wired directly in `boot.rs` (lines 116-198) BEFORE the lifecycle is created. The HSM is effectively decorative — used for shutdown coordination and state visibility, but not actually orchestrating startup.
- **Fix**: Either:
  1. Dispatch synthetic `SubsystemUp` events after each subsystem is wired in boot.rs
  2. Update `new_with_subsystems()` to use proper two-phase initialization with `ActionContext`
  3. Document this as intentional and update spec
- **Decision**: Accept — documented in state.rs comments; QC1 also noted as deferred to B4.

#### W-002: Background mode health check polling has soft failure

- **Location**: `crates/nexus42/src/commands/daemon.rs:238-257`
- **Issue**: The health check polling loop uses 10 retries × 500ms = 5s timeout. If all retries fail, the function prints a warning but returns `Ok(())` anyway. The user sees:
  ```
  ⚠ Daemon process was spawned but health check failed after 10 retries.
  The daemon may still be starting. Check with: nexus42 daemon status
  ```
- **Impact**: Moderate — user may proceed thinking daemon started, but it might have failed during initialization. The PID file is written before health check, so subsequent `stop`/`status` commands will work.
- **Mitigation**: The warning is clear and suggests checking status. The daemon could legitimately take >5s to fully initialize under heavy load or slow disk I/O.
- **Fix**: Consider returning an error instead of `Ok()` when health check fails, or extending timeout to 10s for slower environments.
- **Decision**: Accept — warning message is adequate; 5s is reasonable for normal cases. B4 can refine.

- **Rationale**: The current polling loop uses `tokio::time::sleep()` in a serial loop. An async approach using `tokio::select!` with a deadline could provide cleaner timeout handling and allow other work during the wait period.

- **Alternative**: Spawn health check attempts with `select! { _ = health_check() => ..., _ = deadline => ... }`
- **Impact**: Low — current implementation is adequate for the use case.
- **Recommendation**: Consider refactor in B4 if startup gate becomes a performance bottleneck.

#### S-003: PID file written before health check passes

- **Location**: `crates/nexus42/src/commands/daemon.rs:222-226`
- **Issue**: In background mode, the PID file is written immediately after spawn, before health check confirms the daemon is running. If the daemon crashes during initialization, a stale PID file remains.
- **Mitigation**: `stop_daemon()` correctly cleans up stale PID files when the process doesn't exist (lines 329-334).
- **Impact**: Low — handled by existing cleanup logic.
- **Recommendation**: Consider writing PID only after health check passes, but current approach is acceptable since cleanup exists.

#### S-004: Unix socket stale file cleanup before bind

- **Location**: `crates/nexus-daemon-runtime/src/boot.rs:341-356`
- **Issue**: If the socket file exists from a previous run, it's removed before binding. This prevents socket hijacking but creates a small window where another process could bind to the same path.
- **Mitigation**: The window is ~1ms between `remove_file` and `UnixListener::bind`. For a user-owned socket path in `~/.nexus42/`, this is negligible risk.
- **Impact**: Low — acceptable for local development tool.
- **Recommendation**: Consider atomic bind-if-not-exists if socket is moved to shared location.

## Performance Observations

### Startup Sequence Timing

The initialization sequence in `run_daemon()` has natural ordering that avoids blocking:

1. **WorkspaceState::initialize()** (~100-500ms): Async DB schema init, pool creation
2. **Engine wiring** (~50-100ms): In-memory graph construction
3. **Session recovery** (~50-200ms): Database query for non-terminal sessions
4. **System preset scan** (~50-100ms): Directory scan + session start
5. **Schedule supervisor** (~100-300ms): Second pool creation + recovery query
6. **HTTP listener bind** (~10ms): TCP/Unix socket bind
7. **First health check response**: Available after all above complete

**Estimated total**: 300-1200ms under normal conditions → well within 5s health check window

### Shutdown Sequence Timing

Graceful shutdown with 20s default timeout:

1. **Signal received** → dispatch ShutdownRequested
2. **HTTP server drain** → stops accepting, waits for connections
3. **Schedule pause** → database update (fast)
4. **Session cancellation** → engine signal dispatch (fast)
5. **Subsystem shutdown** → reverse order (Db, Sync, WorkerMgr, Engine, Http)
6. **Process exit** → after ShutdownDrained or timeout

**Assessment**: 20s is adequate for typical cleanup. Slow DB flush or pending network operations could timeout.
