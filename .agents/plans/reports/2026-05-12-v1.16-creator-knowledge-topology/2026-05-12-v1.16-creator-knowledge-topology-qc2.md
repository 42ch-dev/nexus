### 🟡 Warning
- **F-001 — `creator logout` is documented as clearing credentials/session but is a successful no-op.**
  - Evidence: The plan scope says `creator logout` is in scope as “new (clear creator credentials/session)” and `CreatorCommand::Logout` is documented as “Logout and clear creator credentials”; however `creator.rs` only prints `Coming soon: creator logout — clear creator credentials/session.` and returns `Ok(())` without loading or mutating `AuthStore`.
  - Impact: A user can run a command named/logout-described as clearing credentials, receive a successful exit, and still retain local user/creator credentials. This is a security/correctness risk because it creates a false logout state.
  - Fix: Either implement actual credential/session clearing (reusing or extending existing auth clearing primitives as appropriate for creator scope) and add a regression test, or explicitly keep the command hidden/unsupported and return a clear non-success/unsupported result until implemented.

- **F-002 — `creator workspace clone` migration drops the existing `--dry-run` safety path.**
  - Evidence: Hidden legacy `nexus42 clone --help` still exposes `--dry-run` (“Print the JSON request and exit without calling the daemon”), but migrated `nexus42 creator workspace clone --help` exposes only `--source` and `--yes`; `creator.rs` constructs `CloneArgs { dry_run: false, ... }` unconditionally.
  - Impact: Users following the deprecation guidance from hidden top-level `clone` to `creator workspace clone` lose the non-mutating inspection mode and cannot verify the daemon request body without attempting daemon interaction. This is a correctness/safety regression in the migrated command surface.
  - Fix: Add a `dry_run: bool` flag to `CreatorWorkspaceCommand::Clone`, pass it through to `CloneArgs`, and add command-surface/behavior coverage that the migrated path exposes and honors `--dry-run`.
