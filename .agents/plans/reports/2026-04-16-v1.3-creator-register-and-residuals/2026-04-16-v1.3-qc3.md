## Warnings

**W1: `spawn_submit_capture` generates duplicate `pending_id` (session_capture.rs:345-349)**

In `spawn_submit_capture`, a new `pending_id` is generated via UUID at line 346-348. However, this is called *after* `submit_to_daemon` already generates and uses a `pending_id` at line 193-196. When daemon is unavailable, the local file fallback at line 355 uses the *second* `pending_id`, but `submit_to_daemon` already committed the *first* `pending_id` to the request body.

This creates a mismatch: the daemon receives `pending_id=A` but the local fallback saves `pending_id=B`.

- **Severity**: Low (data consistency; does not cause data loss since both paths write valid data)
- **Scope**: `session_capture.rs:340-374`
- **Evidence**: Two separate `uuid::Uuid::new_v4()` calls in the same capture flow
- **Runtime impact**: Low — only affects fire-and-forget path when daemon is down; exact ID tracking across daemon/local may be inconsistent
- **Rollback urgency**: Low

**Recommendation**: Use the same `pending_id` in both `submit_to_daemon` and local fallback. Pass the ID as a parameter or generate once in `spawn_submit_capture` and pass to `submit_to_daemon`.

## Suggestions

**S1: `context_assembly` output format change undocumented (context_assembly.rs:83-500)**

The `assemble_to_string` and related methods changed section separators from `"\n\n"` (double newline) to `"\n"` (single newline), and each section now ends with `\n`. This changes the markdown output format significantly.

- **Severity**: Nit (documented in code comments but not in plan/changelog)
- **Scope**: `context_assembly.rs` — all `assemble_to_string` implementations
- **Evidence**: Diff shows `parts.join("\n\n")` → `parts.join("\n")` and `format!("...\n")` trailing newlines
- **Risk**: If consumers of the assembled context rely on `"\n\n"` section separation, this could break downstream parsing

**Recommendation**: Add a changelog entry or inline comment noting this is an intentional format change, or verify consumers handle single-newline section separators.

**S2: `detect_operation` substring matching is greedy (parser.rs:62-71)**

The `detect_operation` function uses `text.contains(keyword)` which matches any substring. While the `OP_MAPPINGS` ordering (most specific first) partially mitigates this, the keyword `"and"` is included in Add operations (`["adds", "and", "more", "plus"]`). If the platform's challenge text ever contained the word "and" outside the operation context, it could cause misdetection.

- **Severity**: Nit (challenge text is controlled by platform; this is a defense-in-depth concern)
- **Scope**: `parser.rs:62-71`
- **Evidence**: `keywords: &["adds", "and", "more", "plus"]` — "and" is a common English word
- **Risk**: Very low — challenge text format is controlled by platform anti-bot system

**S3: `numbers.rs` compound number parsing only handles tens+units (numbers.rs:67-79)**

The compound number logic only recognizes patterns like "thirty five" → 35. Numbers like "one hundred" (100 + 0), "two hundred" (200 + 0), or "one hundred twenty three" (100 + 20 + 3) are not handled.

- **Severity**: Nit (matches the documented challenge spec which uses numbers 0-100)
- **Scope**: `numbers.rs:67-79`
- **Evidence**: Only `(20..=90).contains(&val) && i+1 < words.len()` branch for tens, no hundreds handling

**Recommendation**: Document clearly that challenges only use numbers 0-100 if this is a hard constraint.
