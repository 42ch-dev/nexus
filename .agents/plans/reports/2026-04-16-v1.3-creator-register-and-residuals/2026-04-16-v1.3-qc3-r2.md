## Findings

### 🔴 R1 — Medium: Retry logic does NOT re-solve challenge

**Location**: `crates/nexus42/src/commands/creator.rs`, `submit_with_retry` (lines 370–408)

**Evidence**:
```rust
// Line 383 — same `answer` used for every attempt
let response = client.verify_creator(verification_code, answer).await?;
```

**Problem**: The docstring on `submit_with_retry` (lines 365–369) says:
> "After the first wrong answer, re-solves the challenge and re-submits."

But the implementation passes the **same `answer`** string on every retry attempt. If the first answer was wrong due to a transient parsing error in `solve_challenge`, re-submitting the identical wrong answer will still fail.

**Impact**: Auto-retry is ineffective when the failure is in the challenge-solving step (e.g., ambiguous text leading to a wrong parse). The user exhausts retries without the system attempting correction.

**Suggested Fix**: Store the verification challenge text in the `VerifyResponse` or capture it from Step 3, and re-call `solve_challenge(&verification.challenge_text)` before each retry attempt.

### 🟡 R2 — Medium: API key stored in plain-text JSON auth store

**Location**: `crates/nexus42/src/auth/mod.rs` (lines 50–66, 83–101)

**Evidence**:
```rust
// AuthStore saves to ~/.nexus42/auth.json
pub fn save(&self) -> Result<()> {
    let content = serde_json::to_string_pretty(self)?;
    std::fs::write(&path, &content)?;  // plain-text write
    // ...
}
```

`CreatorAuthState.creator_api_key` is serialized as plain JSON and persisted to `~/.nexus42/auth.json`. Owner-only permissions (0o600) are set on Unix, but the key is still readable by any process running as the same user.

**Impact**: Malware or a malicious process running as the same user can exfiltrate the creator API key.

**Context**: AGENTS.md notes this is pre-1.0 and local persistence is not a migration contract. However, storing credentials in plain text without encryption is a security concern even in pre-release.

**Suggested Fix**: Either:
1. Use a proper secrets manager (e.g., OS keychain via `keyring` crate on Unix, or `windows-ss` on Windows), or
2. At minimum, document that `~/.nexus42/auth.json` contains sensitive credentials and should not be committed or shared.

### 🟡 R3 — Medium: `obtain_auth_token` conflates creator tokens with user tokens

**Location**: `crates/nexus42/src/commands/creator.rs`, `obtain_auth_token` (lines 410–427)

**Evidence**:
```rust
fn obtain_auth_token(auth_store: &auth::AuthStore) -> Result<String> {
    // Look for any authenticated creator's token as a proxy for the user token.
    // In a more complete implementation, we'd have a dedicated user token field.
    // For V1.3, we check if any creator has an access token stored.
    if let Some(creators) = &auth_store.creators {
        for state in creators.values() {
            if !state.access_token.is_empty() {
                return Ok(state.access_token.clone());
            }
        }
    }
    Err(CliError::AuthenticationRequired)
}
```

**Problem**: The code acknowledges in its own comment that it is a workaround ("proxy for the user token"). It iterates through creator tokens and returns the first non-empty one — which may belong to a different user or an expired session.

**Impact**: A multi-creator scenario could pick the wrong user's token. If the selected creator's token is expired but the current user's token exists elsewhere, the wrong token could be used for registration.

**Suggested Fix**: The `AuthStore` should have a dedicated `user_access_token` field separate from per-creator tokens, or `obtain_auth_token` should validate the token against the platform before use.

### 🟢 R4 — Low: Challenge expiry checked before solve, not re-checked before submit

**Location**: `crates/nexus42/src/commands/creator.rs`, `register_creator` (lines 279–307)

**Evidence**:
```rust
// Step 3: Check challenge expiry (with buffer)
if now > buffered_expiry { return Err(CliError::ChallengeExpired {...}); }

// Step 4: Solve challenge (may take time)
let answer = solve_challenge(&verification.challenge_text)?;

// Step 5: Submit answer — no re-check of expiry
let verify_response = submit_with_retry(&client, &verification.verification_code, &answer, ...).await?;
```

**Problem**: If the challenge expires between Step 3 and Step 5 (e.g., long computation in `solve_challenge`, or network delay), the submit will fail with a platform-side "expired" error that is not as user-friendly as the pre-check.

**Impact**: Minor UX degradation — user gets a less helpful error message after waiting for solve to complete.

**Suggested Fix**: Re-check expiry immediately before the first verify call, or build expiry-buffer margin into the constant `MAX_VERIFY_ATTEMPTS` retry window.

### 🟢 R5 — Low: Network errors during `verify_creator` propagate as generic errors

**Location**: `crates/nexus42/src/commands/creator.rs`, `register_creator` line 310

**Evidence**:
```rust
let verify_response = submit_with_retry(&client, &verification.verification_code, &answer, MAX_VERIFY_ATTEMPTS).await?;
```

`client.verify_creator()` returns `Result<VerifyResponse>` where `Err` variants are `SyncError` → `CliError` (via the `From` impl in `errors.rs`). Network errors (`SyncError::HttpError`) become `CliError::Network` which has a reasonable display. However, the retry loop in `submit_with_retry` only handles `VerifyStatus::WrongAnswer`, `Expired`, `Locked` — it does **not** retry on network errors.

**Impact**: A transient network failure on the first attempt returns immediately without retry, instead of retrying like it does for `WrongAnswer`.

## Runtime Behavior Notes (for @qa-engineer)

1. **Auto-retry behavior**: When WrongAnswer is received, the system retries once with the **same answer**. If the failure was due to challenge parsing ambiguity, the retry will also fail. Manual test: submit a challenge with an intentionally ambiguous number word ("forty" vs "fourty") that could parse differently.

2. **Expiry buffer**: 10-second buffer (`EXPIRY_BUFFER_SECS = 10`). If the challenge has <10 seconds remaining when registration starts, it fails immediately with `ChallengeExpired`.

3. **Token selection**: The system picks the first non-empty creator access token. In a multi-creator auth store, verify the correct creator's token is used for registration.
