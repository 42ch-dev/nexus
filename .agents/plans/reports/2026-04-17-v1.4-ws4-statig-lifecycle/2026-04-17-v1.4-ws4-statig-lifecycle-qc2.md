# QC Report: WS4 Statig Lifecycle HSM

**Plan ID**: `2026-04-17-v1.4-ws4-statig-lifecycle`
**Review Range**: `git diff 7c3dda6...f4deeb6`
**Reviewer Focus**: Architecture & long-term evolution risk — HTTP endpoint design, JSON Schema correctness, codegen integration, wire-compatibility with v1, subsystem trait design, extensibility, and interaction with future WS2 orchestration engine.

## Summary

This review covers the WS4 implementation that replaces nexus42d's single-state "running-probe" with a full 6-state statig HSM. The implementation demonstrates solid architectural direction with the `statig`-based HSM, proper trait boundaries (`Lifecycle` trait isolating statig internals), and comprehensive test coverage. However, several **Critical** and **Warning** issues around schema/spec alignment, wire compatibility, and production readiness must be addressed before approval.

**Verdict: Request Changes** (2 Critical, 8 Warning, 6 Suggestion)

## Critical Issues

### C1: Schema Field Name Mismatch with Spec — `uptime_ms` vs `uptime_seconds`

**Location**: `schemas/acp-runtime/daemon-status-v2.schema.json` line 28-31, spec §7.1

The spec §7.1 defines `uptime_ms` (milliseconds), but the schema uses `uptime_seconds` (seconds). This is a breaking divergence from the authoritative specification.

```json
// Spec §7.1 says:
"uptime_ms": 123456

// Schema has:
"uptime_seconds": { "type": "integer", "minimum": 0 }
```

**Impact**: Platform consumers expecting `uptime_ms` per spec will not find the field. v2 wire shape does not match its own spec.

**Recommendation**: Align with spec — either change schema to `uptime_ms` or update spec. Given the schema is the truth source for codegen, the schema should drive. But since spec §7.1 explicitly shows `uptime_ms`, this needs reconciliation.

**Cross-reviewer note**: @qc-specialist-1 should verify this against their review of spec compliance.

---

### C2: Deferred Initialization Pattern Is Incomplete — `new_with_subsystems` Never Initializes Machine

**Location**: `crates/nexus42d/src/lifecycle/state.rs` lines 379-398, `main.rs` lines 132-135

`StatigLifecycle::new_with_subsystems()` creates a placeholder machine (`None`) but never initializes it. The `dispatch()` method handles uninitialized machines by logging a warning and returning early:

```rust
let machine_ref = match m.as_mut() {
    Some(machine) => machine,
    None => {
        tracing::warn!("dispatch called on uninitialized lifecycle");
        return;
    }
};
```

This means in production (`main.rs`), the first `ProcessStarted` dispatch will be silently dropped, and the lifecycle will never transition from `Starting`. The `main.rs` code:

```rust
let lifecycle = Arc::new(StatigLifecycle::new_with_subsystems(
    subsystems,
    args.shutdown_grace_ms,
));
// ...
lifecycle.dispatch(Event::ProcessStarted); // This will be dropped!
```

**Impact**: Production daemon will hang in `Starting` state forever, never reaching `Running`. The HSM is non-functional in production builds.

**Recommendation**: Either:
1. Initialize the machine immediately in `new_with_subsystems` (preferred), or
2. Add an explicit `initialize()` method that `main.rs` must call before first dispatch, or
3. Remove the deferred initialization pattern entirely — it adds complexity without clear benefit given the current usage.

**Cross-reviewer note**: This is a correctness issue that @qc-specialist-3 should verify in their functional testing.

## Warning Issues

### W1: `daemon_status` Handler Returns Stub Data Instead of Real Health

**Location**: `crates/nexus42d/src/api/handlers/runtime.rs` lines 62-76

The handler builds subsystem health as hardcoded JSON with all statuses "up" and zero values:

```rust
let subsystems = serde_json::json!({
    "http": {"status": "up", "last_check_ms": 0},
    "db": {"status": "up", "last_check_ms": 0},
    // ... all "up", all zeros
});
```

**Impact**: The v2 endpoint is functional but returns misleading data. In `Degraded` state, affected subsystems still show `"status": "up"`. This defeats the purpose of the health monitoring system.

**Recommendation**: Wire the handler to the actual `SubsystemBootstrap` health checks, or at minimum, correlate with `lifecycle_state()` to return appropriate degraded/down statuses. Add a TODO with a plan-id reference for the follow-up work.

---

### W2: `last_error` Field Always Returns `None` in Handler

**Location**: `crates/nexus42d/src/api/handlers/runtime.rs` line 96

The handler sets `last_error: None` unconditionally, even though the HSM tracks `last_error` in `DaemonHsm`:

```rust
last_error: None, // Could be set from lifecycle in Failed state
```

**Impact**: When `lifecycle_state = "failed"`, the endpoint omits the error message that explains why the daemon failed.

**Recommendation**: Read `last_error` from the lifecycle state machine when available, similar to how `exit_code` is handled (lines 79-83).

---

### W3: Parallel Arrays in `degraded` Object Lack Ordering Guarantees

**Location**: `schemas/acp-runtime/daemon-status-v2.schema.json` lines 43-64

The degraded info uses parallel arrays:

```json
"degraded": {
    "subsystems": ["db", "sync"],
    "reasons": ["slow", "lag"]
}
```

There is no schema constraint ensuring `subsystems.len() == reasons.len()`, and consumers must rely on index correlation which is fragile.

**Impact**: Schema allows `subsystems: ["db"], reasons: []` which is nonsensical. Future code generation may not preserve array ordering.

**Recommendation**: Restructure as an array of objects:

```json
"degraded": {
    "entries": [
        {"subsystem": "db", "reason": "slow"},
        {"subsystem": "sync", "reason": "lag"}
    ]
}
```

Or add `"required": ["subsystems", "reasons"]` and document the zipper invariant.

---

### W4: Rust `DaemonStatusV2` Uses Untyped `serde_json::Value` for Nested Objects

**Location**: `crates/nexus-contracts/src/generated/daemon_status_v2.rs` lines 24-27

```rust
pub degraded: Option<serde_json::Value>,
pub subsystems: Option<serde_json::Value>,
```

**Impact**: This defeats type safety for the two most complex fields. The generated Rust type does not leverage the strongly-typed `SubsystemHealthEntry` struct defined in the same file. Consumers must manually match JSON structure.

**Recommendation**: The codegen should generate proper nested structs. If the current codegen cannot handle this, either:
1. Enhance the codegen template, or
2. Hand-write a wrapper type in `crates/nexus-contracts/src/enum_conversions.rs` style.

**Cross-reviewer note**: This is a codegen issue that may need coordination with tooling owner.

---

### W5: `AcpRegistry` Subsystem Is in Schema but Not in Production Bootstrap

**Location**: `schemas/acp-runtime/daemon-status-v2.schema.json` lines 84-86, `crates/nexus42d/src/main.rs` lines 294-309

The schema includes `acp_registry` as a subsystem, but `create_subsystems()` only creates 5 subsystems (Http, Db, Sync, Engine, WorkerMgr). `AcpRegistry` is missing from the bootstrap.

**Impact**: Schema claims 6 subsystems, but production only bootstraps 5. The handler returns stub data for all 6, but `acp_registry` will never transition from "up" to anything else.

**Recommendation**: Either add `AcpRegistrySubsystem` to the bootstrap (even as a mock like Engine/WorkerMgr), or remove it from the schema's required subsystem list and mark it as optional/deprecated.

---

### W6: Tests Rely on `tokio::time::sleep` Instead of Proper Synchronization

**Location**: Multiple test files (`lifecycle_transitions.rs`, `lifecycle_graceful_shutdown.rs`, etc.)

Tests use fixed sleep durations:

```rust
tokio::time::sleep(Duration::from_millis(100)).await;
assert_eq!(m.current_state(), LifecycleState::Running);
```

**Impact**: Flaky tests on slow CI runners. Race conditions possible if dispatch hasn't completed within 100ms.

**Recommendation**: Use `tokio::sync::Notify`, `broadcast` channels, or poll with timeout instead of fixed sleeps. The `LifecycleTransition` broadcast channel already exists — tests could subscribe and await specific transitions.

---

### W7: Missing `started_at` Timestamp Population

**Location**: `crates/nexus42d/src/api/handlers/runtime.rs` line 91

`started_at` is always `None`:

```rust
started_at: None, // Could be set from lifecycle Running.entry timestamp
```

**Impact**: Consumers cannot determine when the daemon entered the Running state.

**Recommendation**: Track `started_at` in `DaemonHsm` (set in `Running.entry`) and expose it via the `Lifecycle` trait for the handler to read.

---

### W8: Commented-Out `engine.start_session()` in `Running.entry` Not Tracked

**Location**: `crates/nexus42d/src/lifecycle/actions.rs` lines 132-139

```rust
// Stub: _system.maintenance session would be started here.
// WS2 will implement actual engine session management.
tracing::info!(
    "Running.entry stub: would start _system.maintenance session (engine not yet available)"
);
```

**Impact**: This is a known gap (WS2 dependency) but there is no residual finding or TODO linking it to a plan. Risk of being forgotten.

**Recommendation**: Add a formal `// TODO(WS2, plan-id): ...` comment or create a residual finding in `status.json` to track this. The spec §5.2 explicitly requires this behavior.

## Suggestions

### S1: Add `stopped` to `LifecycleState` Enum for Completeness

**Location**: `crates/nexus42d/src/lifecycle/mod.rs` lines 30-36

The `LifecycleState` enum omits `Stopped` (the initial pseudo-state). While the spec says it's "never externally visible," having it in the enum would make the state machine model more complete and aid debugging.

**Recommendation**: Add `Stopped` variant with `#[serde(skip)]` or document explicitly why it's excluded.

---

### S2: Schema Should Mark `exit_code` and `last_error` as Conditional

**Location**: `schemas/acp-runtime/daemon-status-v2.schema.json` lines 90-97

These fields are only meaningful when `lifecycle_state == "failed"`, but the schema does not express this constraint.

**Recommendation**: Add `"if": { "properties": { "lifecycle_state": { "const": "failed" } } }` conditional validation, or at minimum document the invariant in the `description` fields more prominently.

---

### S3: `SubsystemHealthEntry` Mixed Concerns — Consider Per-Subtype Schemas

**Location**: `schemas/acp-runtime/daemon-status-v2.schema.json` lines 101-132

`SubsystemHealthEntry` combines generic fields (`status`, `last_check_ms`) with subsystem-specific fields (`active_sessions`, `active_workers`, `cache_age_ms`).

**Recommendation**: For v2.1, consider using `oneOf` or separate entry types per subsystem to enforce that `active_sessions` only appears on `engine`, etc. Current design is acceptable for v2 but limits schema validation power.

---

### S4: `DaemonHsm` `exit_code` Type Inconsistency

**Location**: `crates/nexus42d/src/lifecycle/state.rs` line 42

`exit_code` is `Option<i32>` in `DaemonHsm` but mapped to `i64` in the generated Rust contract and `number` in TypeScript.

**Recommendation**: Standardize on `i32` throughout (matches `std::process::exit` signature) or document the widening rationale.

---

### S5: Add Residual Finding for Deferred Admin Shutdown Endpoint

**Location**: Spec §11 LQ-3

The spec leaves open whether to add `POST /v1/local/daemon/shutdown` in V1.4 or defer to V1.5.

**Recommendation**: Make a decision and track in `status.json` residual findings. If deferred, close LQ-3 with a note.

---

### S6: Consider Adding `schema_version` to `LifecycleState` Enum or Trait

**Location**: `crates/nexus42d/src/lifecycle/mod.rs`

The `Lifecycle` trait could expose a `schema_version()` method to make version negotiation explicit.

**Recommendation**: Low priority, but would make future v3 transitions cleaner.

## Positive Findings

1. **Excellent trait boundary**: The `Lifecycle` trait (mod.rs lines 69-81) properly isolates `statig` internals. HTTP handlers and future orchestration engine will never touch statig types directly.

2. **Comprehensive test coverage**: 5 test files covering transitions, events, subsystems, graceful shutdown, and endpoint shape. Tests verify both happy path and edge cases (abort-on-start, timeout, multiple degradations).

3. **Proper wire compatibility strategy**: v2 response is a structural superset of v1 (spec §8). v1 clients ignoring unknown fields will continue to work.

4. **Good use of statig features**: Hierarchical superstates (`Alive` grouping `Running`/`Degraded`), entry/exit actions, and introspection callbacks are all correctly applied.

5. **Subsystem trait design is extensible**: `SubsystemBootstrap` trait (subsystems/mod.rs lines 43-64) cleanly separates concerns and will accommodate real Engine/WorkerMgr implementations in WS2 without HSM changes.

6. **Signal handling and panic hooks wired correctly**: `main.rs` lines 142-177 properly bridge OS signals and panics into HSM events.

7. **Codegen integration correct**: Schema changes propagate to both Rust (`crates/nexus-contracts/src/generated/`) and TypeScript (`packages/nexus-contracts/src/generated/`) with proper `LATEST_SCHEMA_VERSION` bump to 2.
