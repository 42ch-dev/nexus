## Suggestions (Low / Nit)

### Q3-1: `StatigLifecycle::new_with_subsystems` is a placeholder (Low)

**Location**: `crates/nexus42d/src/lifecycle/state.rs` lines ~490–510

**Finding**: The `new_with_subsystems` constructor creates a placeholder `Arc<Mutex<Option<...>>` with `None` inside and stores `_subsystems` / `_shutdown_grace_ms` as unused parameters. The comment explicitly reads "TODO: In production, main.rs should properly set up subsystems".

```rust
pub fn new_with_subsystems(
    _subsystems: Vec<Arc<dyn SubsystemBootstrap>>,
    _shutdown_grace_ms: u64,
) -> Self {
    // ...
    let placeholder_machine = Arc::new(Mutex::new(None));
    // TODO: In production, main.rs should properly set up subsystems
    Self { machine: placeholder_machine, ... }
}
```

**Impact**: Production `main.rs` calls this constructor (line ~132–135), but the machine stays uninitialized. Subsystem lifecycle is entirely stubbed — subsystems don't actually start/stop through the HSM. This is by design for WS4 (WS2 will wire `nexus-orchestration` crate), but it's worth flagging that the subsystem bootstrap loop in `enter_starting` won't run real subsystem tasks until WS2 integration.

**Recommendation**: Add a `#[deprecated]` attribute or a `todo!()` assertion to make it explicit at runtime that production wiring is pending. Alternatively, add a comment referencing the WS2 ticket that will complete the wiring.

**Severity**: Low (by-design gap for WS4; won't block merge)

### Q3-2: `enter_starting` / `enter_running` / `enter_stopping` spawn tasks but don't track them (Low)

**Location**: `crates/nexus42d/src/lifecycle/actions.rs`

**Finding**: Each `enter_*` function spawns `tokio::spawn(async move { ... })` tasks but does not retain the `JoinHandle`. If the lifecycle state transitions before these tasks complete, they continue running as orphaned tasks.

Example from `enter_starting`:
```rust
tokio::spawn(async move {
    for kind in subsystems.iter() {
        let result = s.start().await;
        // ...
        if result.is_ok() {
            lc.dispatch(Event::SubsystemUp(*kind));
        }
        // ...
    }
});
```

**Impact**: Minor — tasks complete quickly and `dispatch` is idempotent, but the pattern is fragile if entry actions become heavier in WS2.

**Recommendation**: Store `JoinHandle`s in `ActionContext` and cancel them in the corresponding `exit_*` action. A follow-up in WS2 scope would be appropriate.

**Severity**: Low (acceptable for WS4 mock stubs)

### Q3-3: Hardcoded 100 ms pause before `std::process::exit` (Low)

**Location**: `crates/nexus42d/src/lifecycle/actions.rs` `enter_failed()`

```rust
std::thread::sleep(std::time::Duration::from_millis(100));
std::process::exit(exit_code);
```

**Finding**: Per spec §5.5 this 100ms pause is intentional ("for log flush"), but `std::thread::sleep` in a `thread::spawn` context is not cancelable. If the process is killed during this window, the pause won't be cut short.

**Impact**: Negligible — this is the desired behavior per spec.

**Recommendation**: Document this explicitly in the spec as "process will sleep up to 100ms before exit, making it non-cancelable during that window."

**Severity**: Nit

### Q3-4: Test uses `tokio::time::sleep` for async dispatch settling (Nit)

**Location**: All 5 test files

**Finding**: Tests use `tokio::time::sleep(Duration::from_millis(50..100))` to wait for async `dispatch` to propagate. This is fragile — on a loaded machine these could flake.

Example from `lifecycle_transitions.rs`:
```rust
tokio::time::sleep(std::time::Duration::from_millis(100)).await;
assert_eq!(m.current_state(), LifecycleState::Running);
```

**Recommendation**: Consider using a channel-based completion notification instead of arbitrary sleeps, or add a `wait_for_state` helper that polls with a timeout. For now, this is acceptable given the test scope.

**Severity**: Nit

## Cross-Reviewer Ready Notes (for QC1/QC2)

- **Performance note**: The `Arc<Mutex<Option<statig::awaitable::StateMachine<DaemonHsm>>>>` pattern in `StatigLifecycle` has two mutex layers (std + tokio). The std `Mutex` for `current_state` mirror is held only briefly during dispatch. The tokio `Mutex` wraps the statig machine and is locked for the duration of `handle()`. This should be fine for the daemon's single-thread event throughput.
- **Reliability note**: `enter_starting` iterates subsystems sequentially (`for kind in shutdown_order`). Spec §5.1 says "each subsystem bootstrap is its own tokio task". The current impl does sequential await in a spawned task — correct for shutdown order, but startup is also sequential. May want to parallelize startup in WS2.
- **Observability gap**: No `tracing` spans or metrics emitted from lifecycle transitions. `on_transition` callback logs at `debug` level but no structured metrics (histogram of time in each state, count of transitions per event type). WS2 or follow-up could add this.
