## Critical Findings

### C1: Race condition in `enter_failed` — async entry action spawns thread for `process::exit`

**File**: `crates/nexus42d/src/lifecycle/state.rs`, lines 303–327

The `enter_failed` entry action is an `async fn` that spawns a separate `std::thread` to call `enter_failed(exit_code, last_error)`, then awaits `tokio::time::sleep(10ms)`. Meanwhile, `enter_failed()` in `actions.rs` sleeps 100ms and calls `std::process::exit`.

**Problem**: The 10ms sleep in the statig action is far shorter than the 100ms in `enter_failed()`. After the 10ms, the statig dispatch context resumes and may execute additional code (e.g., updating mirrors, broadcasting transitions) while the exit thread is still sleeping. This creates a window where state mirrors can be written to after `process::exit` is imminent, and in worst case, the tokio runtime may drop futures before the exit thread fires.

**Spec expectation** (§5.5): "Call `std::process::exit(exit_code)` after 100ms pause (for log flush)."

**Recommended fix**: Remove the `std::thread::spawn`. Since `enter_failed()` in `actions.rs` already contains the 100ms sleep + `process::exit`, call it synchronously from the async entry action. The 100ms sleep blocks the async task, but that's acceptable — this is the terminal state and no further work should happen.

```rust
// In state.rs enter_failed():
if self.context.is_some() {
    let exit_code = self.exit_code.unwrap_or(1);
    let last_error = self.last_error.clone();
    // Call synchronously — process::exit will terminate everything
    enter_failed(exit_code, last_error);
}
```

**Severity**: Critical — reliability of daemon shutdown path is a core commitment.

### C2: `dispatch()` spawns independent tokio tasks — event ordering not guaranteed

**File**: `crates/nexus42d/src/lifecycle/state.rs`, lines 462–513

Each `dispatch(event)` call spawns a new `tokio::spawn` task. When the caller dispatches multiple events sequentially (as in tests and `main.rs`), there's no guarantee of processing order. Two events dispatched in sequence may execute in reverse order depending on tokio scheduler timing.

**Evidence** — test pattern from `lifecycle_transitions.rs`:
```rust
for kind in SubsystemKind::mandatory() {
    m.dispatch(Event::SubsystemUp(*kind));  // 5 spawned tasks, order undefined
}
```

**Impact**: While the current HSM logic is largely idempotent (inserting into a `HashSet`), a future where events have side-effect ordering dependencies would break silently. The `ProcessStarted` followed immediately by `SubsystemUp` dispatch in tests is particularly vulnerable.

**Recommended fix**: Introduce a single background dispatcher task with an `mpsc::Receiver<Event>` channel. `dispatch()` sends into the channel; one task sequentially processes events. This ensures FIFO ordering.

**Severity**: Critical for architectural integrity; low immediate risk but high future cost.

## Warnings

### W1: `force_state_for_test` corrupts HSM invariants

**File**: `crates/nexus42d/src/lifecycle/state.rs`, lines 427–433

`force_state_for_test()` only updates the `current_state` mirror (`std::sync::Mutex<LifecycleState>`) but does **not** update the actual statig state machine. Subsequent `dispatch()` calls will process events from the **real** statig state, not the forced mirror state.

This creates a split-brain condition: `current_state()` returns the forced value while the HSM internals are in a different state. If any test relies on the forced state persisting across dispatches, it will produce incorrect results.

**Current usage**: Not called in any existing test, so no active breakage. But the public API is a footgun.

**Recommended fix**: Either remove `force_state_for_test()` or make it also update the statig machine (requires statig's internal state manipulation, which may not be publicly exposed). If keeping, rename to `force_state_mirror_for_test` and add documentation warning.

### W2: `new_with_subsystems` has deferred initialization — fragile contract

**File**: `crates/nexus42d/src/lifecycle/state.rs`, lines 379–398

The `StatigLifecycle::new_with_subsystems()` constructor accepts `subsystems` and `shutdown_grace_ms` but creates a placeholder machine (`Arc::new(Mutex::new(None))`). The actual statig machine is only created on first `dispatch()`. The `subsystems` parameter is stored nowhere — meaning `enter_starting` cannot access subsystem bootstraps in the current implementation.

This means `new_with_subsystems` currently **does not wire up subsystems** despite accepting them as parameters. The production `main.rs` calls it, then immediately dispatches `ProcessStarted`, which triggers deferred init. But the subsystems are lost.

**Recommended fix**: Complete the deferred initialization to actually create `ActionContext` with the subsystems and initialize the statig machine in `new_with_subsystems` or on first dispatch.

### W3: `daemon_status` endpoint uses hardcoded subsystem health — not wired to lifecycle

**File**: `crates/nexus42d/src/api/handlers/runtime.rs`, lines 63–70

```rust
let subsystems = serde_json::json!({
    "http": {"status": "up", "last_check_ms": 0},
    "db": {"status": "up", "last_check_ms": 0},
    // ... all hardcoded "up"
});
```

Spec §7.1 requires the endpoint to reflect real subsystem health from the `SubsystemBootstrap::health()` method and degraded tracking. Current implementation returns static stub data regardless of actual state.

Similarly, `degraded` field is always empty. `last_error` is always `None` even in Failed state.

**Note**: This is acceptable as WS4 scope deferral if explicitly documented as stub. The endpoint structure is correct and wire-compatible with v1.

### W4: Panic hook hardcodes `SubsystemKind::Engine`

**File**: `crates/nexus42d/src/main.rs`, lines 171–174

```rust
lifecycle_for_panic.dispatch(Event::FatalError {
    kind: SubsystemKind::Engine, // Generic subsystem
    err: info.to_string(),
});
```

Panics can originate from any subsystem. Hardcoding `Engine` misattributes the failure source. Spec §3 lists `{ kind, err: String }` for `FatalError` where `kind` should identify the actual source.

**Recommended fix**: Use `SubsystemKind::Engine` only as fallback; attempt to extract origin from panic info, or introduce a dedicated `SubsystemKind::Panic` variant.

### W5: `Stopping.entry` does not set HTTP endpoint state to "stopping"

**File**: `crates/nexus42d/src/lifecycle/actions.rs`, lines 190–259

Spec §5.4 requires: "Set HTTP endpoint `lifecycle_state` to `"stopping"`; new work requests get `503 Service Unavailable`." The `enter_stopping` action only handles subsystem shutdown coordination. It does not update the HTTP endpoint's lifecycle state or configure 503 responses.

**Note**: The `LifecycleState::Stopping` is available via `WorkspaceState::lifecycle_state()`, so the endpoint *could* return it. But the handler `daemon_status()` reads from the lifecycle mirror, which **is** updated by dispatch's mirror sync. So this may work implicitly — the concern is that `enter_stopping` doesn't explicitly perform the spec-required HTTP configuration.

### W6: Subsystem shutdown in `enter_stopping` clones entire vector

**File**: `crates/nexus42d/src/lifecycle/actions.rs`, lines 197

```rust
let subsystems = ctx.subsystems.clone();
```

The `Vec<Arc<dyn SubsystemBootstrap>>` is cloned for the shutdown coordinator. While `Arc` cloning is cheap, the vector lookup by `kind` inside the loop is O(n*m). For 5 subsystems this is trivial, but the pattern doesn't scale. A `HashMap<SubsystemKind, Arc<dyn SubsystemBootstrap>>` in `ActionContext` would be cleaner.

## Suggestions

### S1: Legacy stub functions in `actions.rs` should be removed

**File**: `crates/nexus42d/src/lifecycle/actions.rs`, lines 285–320

Functions `enter_starting_stub`, `exit_starting_stub`, `enter_running_stub`, `enter_degraded_stub`, `enter_stopping_stub`, `enter_failed_stub` are dead code. They're not referenced anywhere in the current codebase. Removing them reduces confusion about which functions are the "real" entry/exit actions.

### S2: Add missing transition test: `Starting → Failed` via non-retryable `SubsystemFailed`

The test suite covers `Starting → Running`, `Running → Degraded → Running`, `Stopping → Failed`, `FatalError → Failed`, but not the `Starting → Failed` path when a subsystem fails during bootstrap with `retryable: false`. The HSM code handles it (state.rs lines 96-109), but there's no dedicated test.

### S3: Consider adding `SubsystemKind::Unknown` or `SubsystemKind::Panic`

For the panic bridge (W4), a dedicated variant would be cleaner than misattributing to `Engine`. This also future-proofs for panics in subsystem-agnostic code (e.g., workspace state, config parsing).

### S4: HealthDegraded during Starting (LQ-2 from spec)

The spec's open question LQ-2 asks: "Should `HealthDegraded(kind=...)` that fires *during* `Starting` (before `Running`) delay `Running` transition, or be buffered and replayed?" The current implementation does not handle `HealthDegraded` in the `Starting` state (it defers to `Super` → `Top` → dropped). If this is an intentional deferral decision, it should be documented.

### S5: `exit_starting` does not cancel in-flight subsystem tasks

Spec §5.6: "Cancel any in-flight subsystem start tasks." Current implementation only logs. The TODO comment acknowledges this. For production reliability, tracking `JoinHandle`s and calling `abort()` on them would prevent orphaned subsystem initialization after `Starting → Stopping` transition.

### S6: `enter_degraded` and `enter_running` don't receive `ActionContext` data

The `_ctx: Arc<ActionContext>` parameter is unused in `enter_degraded` and `enter_running`. If these actions need to interact with subsystems (e.g., updating HTTP state, starting maintenance sessions), the context is available but not used. This is fine for WS4 stubs but should be noted as deferred.

## Verdict: Request Changes

**Rationale**: Two critical findings (C1: race condition in `enter_failed`, C2: event ordering nondeterminism) affect the core reliability of the lifecycle HSM. C1 directly impacts the shutdown path which is a core daemon commitment. C2 is an architectural concern that will silently break when event ordering becomes semantically significant.

Six warnings (W1–W6) also require attention, particularly W2 (`new_with_subsystems` not wiring subsystems despite accepting them) and W1 (`force_state_for_test` creating split-brain state).

The HSM transition matrix, statig hierarchical design, and test coverage are all strong. The issues are concentrated in the async/spawning patterns around the HSM wrapper rather than in the state machine logic itself.
