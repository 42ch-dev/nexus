### 🟡 Warning

**F-001: `SessionCapture` created at agent-stop time produces zero-duration metrics**
- **File:** `crates/nexus42/src/commands/acp_worker/mod.rs:689-706`
- **Severity:** Warning
- **Issue:** In `handle_agent_stop`, a new `SessionCapture` is instantiated via `SessionCapture::new(...)`, and `capture_session_end()` is called immediately after creation. The internal `start_time` is set to `Instant::now()` in `SessionCapture::new()`, so `duration_secs` will always be effectively 0. Additionally, `message_count` and `tool_calls` are always 0 since no `record_message()` or `record_tool_call()` calls are made before capture. The submitted `PendingReviewRecord` carries no meaningful session metrics.
- **Impact:** Pending review entries captured at agent stop provide no useful session statistics. The `task_kind` heuristic (from `agent_id`) is the only useful field in the digest. This undermines the Session Review Loop's value proposition.
- **Fix:** Create the `SessionCapture` at session-creation time (in `initialize` or `agent_start` handler), call `record_message()` / `record_tool_call()` on each relevant IPC event, and call `capture_session_end()` only at `agent_stop`.

**F-002: `list_by_creator` query has no LIMIT**
- **File:** `crates/nexus-local-db/src/kb_extract_job.rs:158-182`
- **Severity:** Warning
- **Issue:** The `list_by_creator` function fetches all rows matching a creator ID with `ORDER BY created_at DESC` and no `LIMIT` clause. While this is returned to the CLI via `kb extract-status` (which prints all rows in a terminal table), there is no explicit guard against unbounded growth.
- **Impact:** For a creator with thousands of extract jobs, this could load excessive data into memory and produce an unreadable terminal output. The `extract-status` command has no pagination.
- **Fix:** Add a configurable `LIMIT` (e.g., `LIMIT 100`) or implement pagination via `OFFSET` + `LIMIT`. At minimum, add a soft warning if result set exceeds a threshold.

**F-003: `generate_job_id` uses non-standard collision-resistant scheme**
- **File:** `crates/nexus-local-db/src/kb_extract_job.rs:38-44`
- **Severity:** Warning
- **Issue:** The job ID generator mixes `subsec_nanos()` and `as_millis()` with bitwise OR: `u64::from(now.subsec_nanos()) | (now.as_millis() as u64).wrapping_shl(20)`. This is not a standard unique ID scheme. Within the same wall-clock millisecond, `now.as_millis()` is identical, and `subsec_nanos()` within a single millisecond is limited to 1,000,000 values (nanos 0..999,999). Two calls within the same millisecond on a fast system could theoretically produce the same value. The SQLite PRIMARY KEY constraint on `job_id` catches collisions at INSERT time, but the error propagation from a collision is a raw `sqlx::Error` (not a domain-specific retryable error).
- **Impact:** Low probability collision, but the error is not handled gracefully. If it occurs, the caller receives an opaque SQL error.
- **Fix:** Use `uuid::Uuid::new_v4().to_string()` for ID generation, consistent with the rest of the codebase (used in `session_capture.rs`, `claude.rs`, and elsewhere with `uuid` crate already in dependency tree).

### 🟢 Suggestion

**F-004: `pending_show` fetches all pending reviews for a single-ID lookup**
- **File:** `crates/nexus42/src/commands/creator/memory.rs:307-315`
- **Severity:** Suggestion
- **Issue:** The `pending_show` function calls `list_pending_reviews(creator_id)` which fetches all pending reviews for the creator, then filters client-side with `.find(|r| r.pending_id == pending_id)`. There is no server-side `GET /v1/local/memory/pending-review/{id}` route to fetch a single record by ID.
- **Impact:** Negligible for the expected scale (pending reviews per creator are unlikely to exceed dozens), but violates the principle of fetching only what's needed.
- **Fix:** Add a `GET /v1/local/memory/pending-review/{id}` endpoint to the daemon API for direct single-record lookup by `pending_id`.

**F-005: Per-invocation stderr drain and child-wait background tasks accumulate without cleanup**
- **File:** `crates/nexus-agent-host/src/providers/native_cli/claude.rs:820-838`
- **Severity:** Suggestion
- **Issue:** In `execute_per_invocation`, a `tokio::spawn` task to drain stderr and another to `child.wait()` are spawned for every execute call. These tasks are fire-and-forget — they complete naturally when stderr closes and the child exits, but for short-lived executes on a busy system, dozens of these tasks could accumulate in the tokio runtime's task list before completing.
- **Impact:** Minimal — each task is a lightweight `BufReader::lines()` loop or a single `.await`. In pathological scenarios (100+ concurrent short-lived per-invocation sessions), tokio scheduler overhead could increase marginally.
- **Fix:** Consider using `tokio::task::JoinSet` to clean up completed tasks, or accept current behavior as acceptable for typical usage.

**F-006: Drop-time synchronous process kill could briefly block runtime thread**
- **File:** `crates/nexus-agent-host/src/providers/native_cli/claude.rs:713-746`
- **Severity:** Suggestion
- **Issue:** The `Drop` implementation for `ClaudeCliProvider` uses `std::process::Command::new("kill").arg("-9")` synchronously for orphaned persistent child cleanup. If `kill` hangs (unlikely but possible under extreme system load), it would block the thread executing the Drop.
- **Impact:** Very low — Drop cleanup is a best-effort safety net for program shutdown. The normal path (via `shutdown()`) kills children properly with tokio. The Drop path only activates when the provider is dropped without explicit `shutdown()`.
- **Fix:** The current approach is reasonable for Drop-time best-effort cleanup. Document the Unix-only behavior and that Windows persistent children may be orphaned on Drop (already noted via `tracing::warn`).

### Positive findings

1. **Timeout enforcement is comprehensive.** Session launch, per-session shutdown, per-invocation prompt phase, and Drop-time cleanup all have explicit timeout bounds. The `HangingMockProvider` test validates that shutdown does not hang with a stuck child.

2. **Resource lifecycle is well-managed.** The `Drop` implementation on `ClaudeCliProvider` serves as a safety net for orphaned persistent children. PID tracking in `persistent_pids` is synchronized with `shutdown()` and `cancel()` to avoid double-kill. The `shutdown()` method calls `ProviderAdapter::shutdown()` for every active session, including lost-session-path handling.

3. **Session state machine is defensive.** When `adapter.execute()` returns an error instead of a stream, `transition_busy_to_ready` is called before propagating the error — preventing stuck-in-Busy sessions (QC3 F-001 defense-in-depth from the plan).

4. **Fire-and-forget capture is non-blocking.** `spawn_submit_capture` uses `tokio::spawn` with a short client timeout (2s connect, 5s total) and falls back to local file persistence on daemon unavailability.

5. **SQLite queries use compile-time checking.** All `kb_extract_job` queries use `sqlx::query_as!()` / `sqlx::query!()` macros. Indexes are defined for the idempotency check (`idx_kb_extract_jobs_idempotent` with partial uniqueness on non-failed rows) and the `next_queued` query pattern (`idx_kb_extract_jobs_status` + `idx_kb_extract_jobs_creator`).

6. **No breaking CLI changes.** New commands (`pending-list`, `pending-show`, `pending-dismiss`, `kb queue-extract`, `kb extract-status`, `soul refresh-experience`) are purely additive. The `acp-worker` gains an optional `--daemon-url` flag with a sensible default.

7. **Token budget truncation preserves invariants.** Personality is never truncated; truncatable sections are trimmed from the end (memories → keywords → experience), matching spec §9.3.
