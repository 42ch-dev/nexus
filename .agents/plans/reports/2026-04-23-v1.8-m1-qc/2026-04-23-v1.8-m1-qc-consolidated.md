## Critical Findings (Must Fix)

### C-001: LLM Fallback Not Wired into `register_creator` — Feature Inaccessible in Production

- **Cross-review**: QC1 W-001, QC2 F-001, QC3 F-002
- **Scope**: WS-A
- **Source**: `crates/nexus42/src/commands/creator.rs:286`, `crates/nexus42/src/challenge/mod.rs:148`
- **Issue**: `register_creator()` calls `solve_challenge()` (pure logic path). The new `solve_challenge_with_fallback()` and `LlmSolver` trait are defined and tested but never invoked from the production registration flow. No concrete `LlmSolver` implementation exists outside tests.
- **Impact**: WS-A deliverable provides no end-user value; the fallback infrastructure is dead code in production paths.
- **PM Decision**: **Fix in this wave**. Dev must:
  1. Wire `solve_challenge_with_fallback()` into `register_creator()`.
  2. Provide a default `UnavailableLlmSolver` implementation that returns `None` immediately, so the fallback gracefully degrades when no LLM provider is configured.
  3. This makes the fallback infrastructure active without requiring a real LLM provider in M1.

### C-002: `LlmSolver` Trait Lacks Timeout Protection

- **Cross-review**: QC3 F-001
- **Scope**: WS-A
- **Source**: `crates/nexus42/src/challenge/mod.rs:72-81`
- **Issue**: The `LlmSolver::solve` async method has no timeout guarantee. A hanging LLM call would block the creator registration flow indefinitely.
- **Impact**: Denial of service for registration; poor UX with no progress indicator or escape.
- **PM Decision**: **Fix in this wave**. Dev must wrap the LLM call inside `solve_challenge_with_fallback()` with `tokio::time::timeout` (e.g., 30s default), falling through to the original parse error on timeout.
