### 🔴 Critical

#### F-001: WS-A LLM fallback is dead code in production — creator.rs never calls solve_challenge_with_fallback
- **Severity**: Critical (architectural gap — feature not wired in)
- **Source Type**: manual-reasoning
- **Source Reference**: `crates/nexus42/src/commands/creator.rs:286` vs `crates/nexus42/src/challenge/mod.rs:148`
- **Confidence**: High

The `solve_challenge_with_fallback` async function and the `LlmSolver` trait are implemented with test coverage (7 unit/integration tests), but the **production registration flow in `creator.rs` still calls the synchronous `solve_challenge` directly**:

```rust
// creator.rs:286 — production call site
let answer: String = match solve_challenge(&verification.challenge_text) {
```

There is **no concrete implementation of `LlmSolver`** anywhere in the production codebase (grep confirms only `MockLlmSolver` in tests). Even if an implementation existed, `creator.rs` does not import or invoke `solve_challenge_with_fallback`.

**Impact**: WS-A "LLM fallback for challenge solver" is scaffold-only. When the pure-logic pipeline fails on a challenge, users get a `ChallengeFailed` error instead of an LLM-assisted solve. The entire WS-A deliverable is non-functional in the end-to-end registration flow.

**Fix**: Either (a) integrate `solve_challenge_with_fallback` into `register_creator` with a concrete `LlmSolver` implementation, or (b) document the trait as a future integration point and gate WS-A acceptance on a follow-up story.

### 🟡 Warning

#### F-002: T5 positional-syntax rejection test is overly lenient on error-message assertion
- **Severity**: Warning
- **Source Type**: manual-reasoning
- **Source Reference**: `crates/nexus42/tests/creator_register_e2e.rs:577-583`
- **Confidence**: Medium

```rust
assert!(
    combined.contains("--name")
        || combined.contains("unexpected")
        || combined.contains("required")
        || combined.contains("Usage"),
    "error output should mention --name or usage hint; got: {combined}"
);
```

The test accepts any of four broad substrings. The `"unexpected"` and `"Usage"` branches are particularly broad — clap's error output always contains "Usage" in its standard format, making the assertion pass even if `--name` is never mentioned and no migration hint is shown. This weakens regression protection for the T5 migration-hint requirement.

**Fix**: Narrow the assertion to require `"--name"` specifically, or check for the clap error pattern `error: missing required option '--name'`.

#### F-003: No maximum-length validation on `name` argument before platform call
- **Severity**: Warning
- **Source Type**: manual-reasoning
- **Source Reference**: `crates/nexus42/src/commands/creator.rs:31-38`
- **Confidence**: Medium

The `name` field is declared as `#[arg(long)] name: String` with no `value_parser` length limit. A caller passing a multi-thousand-character name will embed it in the `POST /creators/register` request body. While the platform presumably has its own limits, defensive length validation in the CLI before network I/O is absent.

**Fix**: Add `value_parser = clap::value_parser!(String).range(1..=256))` or equivalent, consistent with platform display-name limits.
