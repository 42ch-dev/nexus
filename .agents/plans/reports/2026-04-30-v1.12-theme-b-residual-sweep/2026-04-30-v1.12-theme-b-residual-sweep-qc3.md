### 🟢 Suggestion

**S-001: `refresh_access_token` success path still incurs 2x file read + 1x file write per call**
- **Location**: `crates/nexus42/src/auth/user_auth.rs:259,354`
- **Issue**: The consolidation correctly reduced `AuthStore::load()` calls from 4 to 3 (2 in success path). However, every successful token refresh still reads the auth file twice and writes it once. For a CLI this is acceptable, but if this path becomes a daemon hot path (per `auth/mod.rs` comments about future daemon migration), the repeated file I/O will be a bottleneck.
- **Recommendation**: If `refresh_access_token` is ever called in a tight loop or daemon context, consider:
  1. Holding an in-memory `AuthStore` with file-based invalidation (e.g., check mtime before reload), or
  2. Using a file lock to avoid the defensive re-read before write.
  Current CLI usage is low-frequency enough that this is not blocking.
- **Impact**: Low — CLI auth refresh is user-initiated and infrequent.
- **Decision**: Accept as-is; document as performance note for future daemon migration.

**S-002: `device_id.rs` retry uses fixed sleep without backoff or jitter**
- **Location**: `crates/nexus-sync/src/device_id.rs:78`
- **Issue**: The retry loop sleeps a fixed 5ms per iteration. With `MAX_EMPTY_RETRIES = 3`, worst-case latency is 15ms — negligible. However, if `MAX_EMPTY_RETRIES` is ever increased (e.g., under heavy contention), fixed sleep can cause thundering herd or unnecessary latency.
- **Recommendation**: If retries are ever increased beyond 3, switch to exponential backoff with jitter (e.g., 1ms, 2ms, 4ms). Current bound makes this optional.
- **Impact**: Low — bounded to 3 retries; no thundering herd risk at current scale.
- **Decision**: Accept as-is; future-proofing note only.

**S-003: BTreeMap-to-HashMap `.collect()` in test helper is redundant**
- **Location**: `crates/nexus42/src/commands/creator.rs:651`
- **Issue**: The test helper builds a `BTreeMap`, then does `.into_iter().collect()` to produce a `HashMap` for `store.creators`. The intermediate BTreeMap provides determinism during construction, but the final collect into HashMap re-hashes all entries. In test code this is harmless, but it adds unnecessary work.
- **Recommendation**: Since `AuthStore` already uses `HashMap` in production, the helper could directly construct a `HashMap` with deterministic insertion order (e.g., using `HashMap::from([...])` with a fixed seed or FNV hasher in tests) to avoid the BTreeMap-to-HashMap conversion. This is a micro-optimization for test code only.
- **Impact**: Negligible — test helper, not a hot path.
- **Decision**: Accept as-is; test determinism is the priority.

### AuthStore Reload Reduction Correctness (R-V111-S02)

- **Original**: `refresh_access_token` loaded `AuthStore` at function entry (for `refresh_token`), then again before constructing the new token (for `user_id`), then again before saving — 4 total loads, 3 on success path.
- **New**: Loads once at entry, capturing both `refresh_token` and `user_id` from the same snapshot. Then loads once more before saving — 3 total loads, 2 on success path.
- **Correctness**: `user_id` does not change during a token refresh (it is an account-scoped identity, not a token-scoped field). Using the initial snapshot is safe. The pre-save load remains as a defensive measure against concurrent mutation.
- **Race window**: Between initial load and HTTP response, another process could modify `auth.json`. The pre-save load mitigates this by reading fresh state before writing. This is the same mitigation as before, just with one fewer intermediate load.
- **Verdict**: Correct and safe. A modest I/O reduction with no new concurrency hazards.

### Retry Loop Boundedness (R-V111-S01)

- **Counter**: `empty_retries` is `u8`, initialized to 0, incremented before check.
- **Bound**: `empty_retries > MAX_EMPTY_RETRIES` (where `MAX_EMPTY_RETRIES = 3`).
  - 1st empty: 1 > 3? No -> retry
  - 2nd empty: 2 > 3? No -> retry
  - 3rd empty: 3 > 3? No -> retry
  - 4th empty: 4 > 3? Yes -> return `TimedOut` error
- **Sleep**: Fixed 5ms per retry. Max sleep = 15ms.
- **Overflow**: `u8` could theoretically overflow at 255, but the `>` check triggers at 4, so overflow is unreachable.
- **Verdict**: Properly bounded. No infinite loop, no integer overflow, no spin-lock behavior.

### BTreeMap vs HashMap Tradeoff (R-WC-003)

- **Scope**: Test code ONLY. Production `AuthStore.creators` remains `HashMap<String, CreatorAuthState>`.
- **Rationale**: BTreeMap provides deterministic iteration order, eliminating flaky test failures from HashMap randomized hashing.
- **Performance**: BTreeMap has O(log n) lookup/insert vs HashMap O(1) average. In test code with tiny maps (1-2 entries), this is irrelevant.
- **Risk**: None. No production code path is affected.
- **Verdict**: Correct test-only fix. No hot path regression.

### Pre-existing Test Isolation Debt

Both failures are symptoms of a **shared filesystem anti-pattern** in the auth test suite: tests read/write the real `~/.nexus42/auth.json` instead of using temporary directories. This creates:

1. **Order-dependent failures**: Test outcome depends on which tests ran before it.
2. **Parallel execution hazards**: `cargo test` runs tests in parallel by default; file races are inevitable.
3. **CI noise**: Flaky failures waste developer time and erode trust in CI.

**Recommendation (not blocking this diff)**: Migrate auth tests to use `tempfile::TempDir` with injected paths, or mock the `auth_store_path()` function in test builds. This is a known reliability debt that should be tracked separately.
