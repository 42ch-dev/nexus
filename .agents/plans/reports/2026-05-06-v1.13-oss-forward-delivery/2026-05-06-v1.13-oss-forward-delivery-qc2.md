### 🟡 Warning
- **QC2-W-001 — DF-11 handlebars renderer is not on the runtime `AcpPromptTask` path.** The changed code adds `render_core_context_template()` and a unit-style integration test for that helper, but `AcpPromptTask::run()` still calls `self.render_template(&context).await`, whose implementation remains the pre-existing flat `{{key}}` string replacement. That path does not call the new handlebars renderer, does not support nested paths such as `{{world.title}}` unless a flat context key named `world.title` exists, and still treats malformed templates as literal text rather than surfacing render errors. This is a correctness gap for the claimed “bind core_context template rendering path” delivery because the actual prompt sent to the worker is not using the new renderer. -> Fix by wiring `AcpPromptTask` rendering through the handlebars helper (or the intended CoreContext JSON payload adapter), propagating render failures into `GraphError`, and adding an end-to-end `AcpPromptTask`/worker or context test that proves nested CoreContext values render through the real task execution path.
- **QC2-W-002 — CLI-visible staged platform errors can include unsanitized upstream bodies.** `classify_platform_error()` preserves `SyncError::PlatformError { body }`, and `StagedPlatformError::Display` prints that body verbatim as `platform integration failed: HTTP {status} — {body}`. Because platform error bodies are external input and may include echoed request fields, API keys, tokens, or internal diagnostics, the new “CLI-visible error bucket” can leak sensitive data despite the code comment stating it avoids leaking internal details. -> Fix by separating stable status/category from a sanitized user message: redact token/API-key-looking values, cap body length, prefer structured error codes/messages from the platform contract, and add regression coverage proving `creator_api_key`, bearer-token-like values, and similar secrets are not displayed.

### 🟢 Suggestion
- **QC2-S-001 — Decide prompt escaping semantics before using `handlebars::Handlebars::new()` for natural-language prompts.** The default handlebars renderer HTML-escapes interpolated values. That is usually safe for HTML, but prompt/CoreContext rendering is plain text; values containing `&`, `<`, `>`, quotes, or manuscript text can be silently transformed (for example `&` to `&amp;`) once the helper is wired into runtime. -> Add an explicit test for plain-text prompt values and configure escaping deliberately (`no_escape` if plain-text fidelity is desired, or documented escaping if not).
