## Detailed Findings

### C1 — ConnectionTo<Agent> Send/Safety Concern (Critical)

**File**: `crates/nexus-acp-host/src/client.rs` ~L532-550

**Problem**: In `with_connection()`, the `connect_with` callback stores the `ConnectionTo<Agent>` handle into the shared `Arc<RwLock<Option<SdkConnection>>>` and then awaits `std::future::pending::<()>()`. The entire callback runs inside the `LocalSet` (non-Send context). The `ConnectionTo<Agent>` handle is then accessed from trait methods via `connection.spawn(async move { ... })`.

The code clones `connection_for_callback` and stores `cx` inside the callback, but the concern is: `ConnectionTo<Agent>` from SDK v0.11.0 is documented as `Clone + Send`. If it truly is `Send`, this is safe. However, the pattern of storing it inside a `!Send` LocalSet closure and then using it from `spawn` tasks (which run on the dispatch loop) requires verification that:

1. `ConnectionTo<Agent>` is indeed `Send` (SDK v0.11.0 claim)
2. The `spawn()` method correctly dispatches to the internal dispatch loop

**Evidence**: The trait methods all use `connection_handle.spawn(async move { ... })` pattern (e.g., `initialize` at L662, `create_session` at L737, `list_sessions` at L987, `set_config_option` at L1062). Each one calls `.block_task()` which "requires the dispatch loop context."

**Risk**: If `ConnectionTo<Agent>` is not actually `Send` or if `spawn` doesn't properly dispatch to the LocalSet thread, this creates a data race or undefined behavior. The `SdkConnection` struct stores `ConnectionTo<Agent>` (L429) and `sessions: HashMap<String, ActiveSession<'static, Agent>>` (L437). The `ActiveSession<'static, Agent>` may also have `Send` requirements.

**Recommendation**: 
- Add a compile-time `assert_send::<ConnectionTo<Agent>>()` helper in tests to verify the SDK's `Send` claim
- Add documentation to `SdkConnection` explaining the thread-safety invariant
- Verify that `ActiveSession<'static, Agent>` is also `Send` (required for storage in `HashMap` accessed from `spawn` closures)

### W1 — Prompt Content Flattening Loses ResourceLink Semantics (Warning)

**File**: `crates/nexus-acp-host/src/client.rs` ~L800-811

**Problem**: The `prompt()` method extracts text from `NexusContentBlock`s by:
```rust
let prompt_text = request.prompt.iter()
    .map(|block| match block {
        NexusContentBlock::Text(t) => t.text.clone(),
        NexusContentBlock::ResourceLink(r) => format!("resource:{}", r.uri),
    })
    .join("\n");
```

This flattens structured content blocks into a single `&str` and then calls `active_session.send_prompt(&prompt_text)`. The `ResourceLink` variant loses all semantic information — only the URI is preserved as a `resource:` prefix string. The agent receives a plain text string, not structured content blocks.

**Root cause**: SDK v0.11.0's `ActiveSession::send_prompt()` apparently accepts `&str` rather than structured content blocks (unlike the old SDK which had `PromptRequest` with `Vec<ContentBlock>`).

**Risk**: Agents that understand `ResourceLink` semantics (file references, URLs, etc.) will only see the URI as text, not as a structured resource reference. This degrades the protocol fidelity.

**Recommendation**: 
- Verify if SDK v0.11.0 has an alternative `send_prompt` that accepts structured content
- If not, document this as a known limitation and plan for structured prompt support
- At minimum, preserve the original `sdk_prompt_request_from_nexus` conversion path for future use (currently `#[allow(dead_code)]`)

### W2 — Drop Implementation Fire-and-Forget (Warning)

**File**: `crates/nexus-acp-host/src/client.rs` ~L597-611

**Problem**: The `Drop` impl for `AcpSdkAdapter`:
```rust
if let Ok(handle) = tokio::runtime::Handle::try_current() {
    handle.spawn(async move {
        let _ = setup_task.await;
    });
}
```

This spawns a cleanup task but immediately returns from `drop()`. There is no guarantee the setup task (which holds the `LocalSet` and the SDK connection) completes before the process exits. If `Drop` is called during process shutdown, the spawned task may be killed mid-execution.

**Risk**: The agent subprocess may not receive proper shutdown signals. The `LocalSet` bridge thread may be terminated abruptly, potentially leaving the subprocess in a zombie state.

**Recommendation**: Consider adding a `shutdown()` async method that consumers should call explicitly before dropping, similar to how the `LocalSetBridge` itself has a graceful shutdown sequence with join timeouts.

### W3 — Silent Discard of Non-Select Config Kinds (Warning)

**File**: `crates/nexus-acp-host/src/client.rs` ~L336-343

**Problem**: In `sdk_config_option_to_nexus`:
```rust
let kind = match &opt.kind {
    SessionConfigKind::Select(sel) => NexusConfigKind::Select(...),
    _ => NexusConfigKind::Select(NexusConfigSelect {
        current_value: String::new(),
        options: NexusConfigSelectOptions::Ungrouped(vec![]),
    }),
}
```

The `_` arm silently produces an empty `Select` for any non-Select SDK kind (e.g., `Boolean`, `Number`, `String`, or future variants). This means consumers receive an apparently valid but empty config option without knowing data was lost.

**Risk**: If the SDK adds new `SessionConfigKind` variants, this code silently discards them without warning or error.

**Recommendation**: 
- Add `tracing::warn!` in the `_` arm with the kind name
- Consider adding a `NexusConfigKind::Unknown { raw: String }` variant
- Add `#[allow(clippy::match_wildcard_for_single_variants)]` with a comment explaining the forward-compatibility rationale

### S1 — Prompt Text Extraction Outside Bridge Closure (Suggestion)

**File**: `crates/nexus-acp-host/src/client.rs` ~L798

The `prompt_text` is built before entering the `bridge.execute()` closure. This is a good pattern (avoids capturing `request` which is needed for the `move` closure), but the text extraction logic is duplicated between `prompt()` and `sdk_prompt_request_from_nexus`. Consider extracting into a shared `fn extract_prompt_text(blocks: &[NexusContentBlock]) -> String`.

### S2 — Dead Code: sdk_prompt_request_from_nexus (Suggestion)

**File**: `crates/nexus-acp-host/src/client.rs` ~L207

The `sdk_prompt_request_from_nexus` function is marked `#[allow(dead_code)]` because `prompt()` uses `send_prompt(&prompt_text)` instead of the structured `PromptRequest`. If the structured path is planned for future use, add a TODO comment. Otherwise, consider removing it to reduce maintenance burden.

### S3 — Limited ContentBlock Variants (Suggestion)

**File**: `crates/nexus-contracts/src/local/acp/types.rs` ~L181

`NexusContentBlock` only has `Text` and `ResourceLink`. The ACP spec also defines `ImageContent`, `ToolUse`, `ToolResult`, and others. Adding a `#[serde(other)]` or `Unknown` variant would provide forward compatibility.

---

### S4 — Missing Negative-Path Tests (Suggestion)

New trait methods `list_sessions` and `set_config_option` have unit tests for the "no connection" case but lack tests for:
- `list_sessions` with a valid connection but SDK returns empty/error
- `set_config_option` with a valid connection but invalid config_id
- `prompt` with a valid connection but nonexistent session (the test at L1425 only verifies "no connection" path, not "no session in HashMap" path)

## Cross-Reviewer Ready Notes

**Integration risk**: This migration changes the core ACP communication path. Consumers of `NexusAcpClient` trait (currently minimal in this repo — `nexus42` CLI's agent commands reference the adapter but don't fully exercise the trait yet) should be unaffected since trait signatures use Nexus DTOs. The main risk is in the internal `AcpSdkAdapter` implementation.

**Migration cost**: The old `subscribe()` method was removed without replacement. Any future streaming requirements will need a new DTO-wrapped API (documented as future work in module docs).

**SDK dependency pin**: The SDK is pinned to `=0.11.0` (exact version) in `Cargo.toml`. This is appropriate for a major API migration but should be revisited when the SDK stabilizes.
