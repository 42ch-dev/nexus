# V1.2 Program Review — QC2 Report (Architecture / Integration)

**Review Scope**: `git diff c6a911d..HEAD` on `feature/v1.2`  
**Branch**: `feature/v1.2`  
**Primary Focus**: Security, correctness, and integration-layer contract alignment for architecture-level changes.

---

## Executive Summary

The V1.2 architecture additions are structurally sound: schema evolution is minimal, wire-contract generation is consistent, and the CLI→daemon→platform layering is well-separated. However, **two high-impact integration bugs** were found in the CLI-daemon boundary that will cause runtime failures, and **two medium-severity contract/validation gaps** need correction before merge. The recommendation is **Request Changes**.

---

## Findings

### 🔴 H1 — Daemon client `call_assemble` hits wrong URL path

- **File**: `crates/nexus42/src/api/daemon_client.rs:186`
- **Issue**: `call_assemble` constructs the URL as `{base_url}/api/v1/context/assemble`, but the daemon serves the endpoint at `/v1/local/context/assemble` (see `crates/nexus42d/src/api/mod.rs:105` and `crates/nexus42/src/context/client.rs:31`).
- **Impact**: Any CLI two-stage assembly call that reaches this method will receive HTTP 404 and fall back (or surface a confusing error). The method is actively used in `crates/nexus42/src/commands/context.rs:394`.
- **Fix**: Change the path to `/v1/local/context/assemble` and remove the stale `#[allow(dead_code)]` annotation (see L1).
- **Cross-reviewer note**: QC#1 may have covered domain-side assembly logic; this is the **integration boundary** defect.

### 🔴 H2 — `clone` command calls a daemon endpoint that does not exist

- **File**: `crates/nexus42/src/commands/clone.rs:124`
- **Issue**: The CLI posts to `/v1/local/world/clone`, but no such route is registered in `crates/nexus42d/src/api/mod.rs`. The command performs a `health_check()` first, so a missing daemon is handled, but if the daemon *is* running the user gets a 404 / JSON deserialization error.
- **Impact**: User-visible runtime failure for a shipped command. The prior tech-debt report (`v1-tech-debt-cleanup-qc-self-20260415-batch-e.md`) already noted this as deferred.
- **Fix**: Either (a) implement the daemon handler and route, or (b) remove the command from the public CLI surface until the endpoint is ready. If keeping it, add an integration test that exercises the daemon round-trip (not just `--dry-run`).

### 🟡 M1 — CLI `RuntimeStatus` model is missing the new `runtime_mode` field

- **File**: `crates/nexus42/src/api/models.rs:7`
- **Issue**: The daemon’s `StatusResponse` (in `crates/nexus42d/src/api/handlers/runtime.rs:31`) now includes `runtime_mode: String`, but the CLI’s deserializing `RuntimeStatus` struct does not declare it. Serde ignores unknown fields by default, so no crash occurs, but the CLI cannot observe or assert on the daemon’s runtime mode.
- **Impact**: Silent contract drift; downstream features (e.g., `doctor` checking mode consistency) cannot be implemented without fixing the model.
- **Fix**: Add `pub runtime_mode: String` to `RuntimeStatus` and update the unit-test JSON fixture in the same file.

### 🟡 M2 — Inconsistent `creator_id` validation in memory pending-review handlers

- **File**: `crates/nexus42d/src/api/handlers/memory.rs`
- **Issue**: `create_pending_review` validates `creator_id` with only `.starts_with("ctr_")` (line 166), whereas `list_pending_reviews` and `count_pending_reviews` use the stricter `nexus_domain::is_valid_creator_id` (lines 220, 291). This was previously flagged in `2026-04-14-v1.2-local-memory-pipeline-qc2.md` and is still not fully resolved.
- **Impact**: Malformed IDs such as `ctr_../../../etc` can reach the create path. SQL injection is mitigated by parameterized queries, but downstream file-system or logging consumers are at risk.
- **Fix**: Replace the `starts_with` check in `validate_pending_review_input` with `nexus_domain::is_valid_creator_id(&req.creator_id)`.

### 🟡 M3 — `RuntimeMode` enum absent from TypeScript contract package

- **Files**: `schemas/domain/runtime-mode.schema.json`, `packages/nexus-contracts/src/generated/index.ts`
- **Issue**: The Rust codebase manually maintains `RuntimeMode` in `crates/nexus-contracts/src/enum_conversions.rs` (with `Display`, `FromStr`, etc.), but the TypeScript codegen does not emit a corresponding `RuntimeMode` export. Pure string-enum schemas appear to fall outside the current codegen pipeline.
- **Impact**: TypeScript consumers of `@42ch/nexus-contracts` must use bare strings (`'local_only' | ...`) instead of a shared, typed constant, increasing the risk of drift when modes are added or renamed.
- **Fix**: Extend the codegen pipeline to emit a standalone string-union type (or const enum) for pure `enum` schemas, or hand-maintain a `RuntimeMode.ts` next to the Rust equivalent and export it from `index.ts`.

### 🟢 L1 — Stale `#[allow(dead_code)]` on `call_assemble`

- **File**: `crates/nexus42/src/api/daemon_client.rs:178`
- **Issue**: `call_assemble` is actively invoked from `commands/context.rs:394`, yet it carries an `allow(dead_code)` attribute.
- **Fix**: Remove the attribute.

### 🟢 L2 — Minor formatting drift in non-generated files

- **Files**: `crates/nexus-domain/src/lib.rs`, `crates/nexus42/src/commands/config.rs`, `crates/nexus42/src/commands/context.rs`
- **Issue**: `cargo fmt --check` (stable) reports diffs against these files. The project CI uses nightly rustfmt, which ignores generated directories, but the non-generated drift is still present.
- **Fix**: Run `cargo +nightly fmt --all` and commit.

### 🟢 L3 — `config.rs` missing trailing newline

- **File**: `crates/nexus42/src/commands/config.rs:168`
- **Fix**: Add trailing newline.
