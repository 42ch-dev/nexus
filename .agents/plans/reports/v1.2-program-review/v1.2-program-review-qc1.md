## Findings

### Critical (C-001)

#### C-001: Review classification is purely length-based — no content quality signal

**Files**: `crates/nexus-domain/src/review.rs:150-224`
**Severity**: Critical (architectural)

The `classify_pending_review()` function uses only `raw_digest.len()` and `task_kind` to decide between Drop/FragmentOnly/PromoteToLongTerm. There is zero content-quality analysis:

```rust
// Line 151-157: Only length matters
let digest_len = record.raw_digest.len();
// ...
const DROP_THRESHOLD: usize = 50;
const CREATIVE_PROMOTE_THRESHOLD: usize = 100;
const DEFAULT_PROMOTE_THRESHOLD: usize = 200;
```

**Risk**: A 250-character string of repeated whitespace or "aaaa..." will be promoted to long-term memory. Conversely, a dense, information-rich 90-character brainstorm summary will be downgraded to FragmentOnly.

**Recommendation**: Add at minimum:
1. Whitespace/stopword filtering before length check
2. Entropy or vocabulary-diversity check (even a simple unique-word-count ratio)
3. Configurable thresholds via `DegradationPolicy` or a new `ReviewPolicy`

**This finding is unique to this reviewer** — QC#2 and QC#3 may focus on different angles.

### Warnings (W-001 to W-004)

#### W-001: `create_pending_review` API lacks `creator_id` format validation

**File**: `crates/nexus42d/src/api/handlers/memory.rs:166-170`
**Severity**: Warning (security/input validation)

The `validate_pending_review_input()` function checks `creator_id.starts_with("ctr_")` but does not validate the full CreatorId pattern `^ctr_[a-zA-Z0-9]+$`. A value like `ctr_../evil` or `ctr_foo bar` would pass:

```rust
// Line 166-170 — insufficient validation
if !req.creator_id.starts_with("ctr_") {
    return Err(NexusApiError::InvalidInput { ... });
}
```

**Contrast**: The `list_pending_reviews` and `count_pending_reviews` handlers correctly use `nexus_domain::is_valid_creator_id()` (lines 220, 291).

**Recommendation**: Use `nexus_domain::is_valid_creator_id(&req.creator_id)` in `validate_pending_review_input()` for consistency and security.

#### W-002: `delete_pending_review` has no creator_id authorization check

**File**: `crates/nexus42d/src/api/handlers/memory.rs:337-377`
**Severity**: Warning (security)

The DELETE endpoint accepts any `pending_id` without verifying the requesting creator owns that record:

```rust
// Line 337-377 — no creator_id in path or query
pub async fn delete_pending_review(
    State(state): State<WorkspaceState>,
    Path(pending_id): Path<String>,
) -> Result<Json<DeletePendingReviewResponse>, NexusApiError> {
```

Any client with daemon access can delete any pending review regardless of ownership.

**Recommendation**: Add `creator_id` to the query params and verify ownership before deletion, or include it in the path.

#### W-003: `promote_to_long_term` slug collision risk

**File**: `crates/nexus-domain/src/review.rs:628`
**Severity**: Warning (data integrity)

The slug is derived from `memory_id` by replacing `mem_` with `memory-`:

```rust
let slug = memory.frontmatter.memory_id.replace("mem_", "memory-");
```

If two sessions produce memory IDs that result in the same slug after this transformation (unlikely but possible with non-standard ID generators), the second promotion silently overwrites the first without warning. The `save_memory` function uses `std::fs::write` which truncates.

**Recommendation**: Add a check before `save_memory` — if slug already exists, append a suffix or return an error. Alternatively, ensure `memory_id` generation is globally unique (UUID-based, which it is — but document this assumption).

#### W-004: `open_editor_temp` leaks temp file on editor crash

**File**: `crates/nexus42/src/commands/memory.rs:237-269`
**Severity**: Warning (resource leak)

If the editor process crashes or is killed (SIGKILL), the temp file is not cleaned up:

```rust
let status = std::process::Command::new(&editor)
    .arg(&temp_path)
    .status()?;

if !status.success() {
    let _ = std::fs::remove_file(&temp_path);  // Only cleaned on non-zero exit
    return Err(...);
}

let content = std::fs::read_to_string(&temp_path)?;
let _ = std::fs::remove_file(&temp_path);  // Only cleaned on success
```

If `read_to_string` fails after a successful editor exit (e.g., file permission changed), the temp file leaks.

**Recommendation**: Use a guard/drop pattern or `tempfile` crate for automatic cleanup.

### Suggestions (S-001 to S-008)

#### S-001: `extract_h2_sections` loses newline preservation

**File**: `crates/nexus-domain/src/soul.rs:204`
```rust
current_body.push_str(line);
current_body.push('\n');
```

Each line gets a trailing `\n` appended, but the original markdown may have used `\r\n` or different line endings. This is minor since markdown parsers are lenient, but worth noting for roundtrip fidelity.

#### S-002: `classify_pending_review` uses `unwrap_or(TaskKind::Unknown)` — silent fallback

**File**: `crates/nexus-domain/src/review.rs:152`
```rust
let task_kind: TaskKind = record.task_kind.parse().unwrap_or(TaskKind::Unknown);
```

Invalid task kinds silently fall back to `Unknown` with a higher promotion threshold (200 chars vs 100). This means a typo like `"brianstorm"` gets treated as Unknown and may be dropped when a valid `"brainstorm"` would have been promoted at 100 chars.

**Recommendation**: Log a warning when fallback occurs, or make the error explicit in the `ReviewDecision.reason`.

#### S-003: `DegradationGuard` has no `Send`/`Sync` bounds

**File**: `crates/nexus-domain/src/degradation.rs:229`

`DegradationGuard` holds `chrono::DateTime<Utc>` and `DomainRuntimeMode` fields but does not derive or explicitly implement `Send + Sync`. If this is intended for use across async task boundaries (e.g., in the daemon's health check loop), it should implement these traits.

**Current state**: The struct is `pub` and used in the daemon, but `Send`/`Sync` are auto-derived only if all fields are `Send`/`Sync`. `chrono::DateTime<Utc>` and `DomainRuntimeMode` are `Send + Sync`, so this works by accident — but should be explicit.

#### S-004: `context_assembly.rs` uses `chars/4` heuristic which is locale-blind

**File**: `crates/nexus-domain/src/context_assembly.rs:224-226`
```rust
pub fn estimate_tokens(text: &str) -> usize {
    text.len().div_ceil(4)
}
```

This uses `.len()` (bytes), not `.chars().count()`. For ASCII text, `len == chars`, but for CJK or emoji, the ratio is significantly different (e.g., a Chinese character is 3 bytes but may be 1-2 tokens). This may lead to over-estimation for multilingual content.

**Recommendation**: Use `text.chars().count().div_ceil(4)` for better accuracy, or document the byte-based assumption.

#### S-005: `memory.rs` `delete` command confirmation is bypassable

**File**: `crates/nexus42/src/commands/memory.rs:202-213`

The confirmation check uses `eq_ignore_ascii_case("y")` — only single-letter `y` is accepted. Users who type `yes` will have the deletion aborted, which is a UX issue. Most CLI tools accept `y` or `yes`.

#### S-006: `experience_aggregation.rs` has duplicated logic

**Files**: `crates/nexus-domain/src/experience_aggregation.rs:78-139` and `146-194`

`aggregate_experience()` and `aggregate_experience_preview()` share ~80% identical code (listing, filtering, sorting, synthesis). The only difference is step 5 (updating SOUL.md).

**Recommendation**: Extract a shared private function `collect_experience_entries()` and a common synthesis step, then have both public functions call it.

#### S-007: `runtime_mode.rs` command `set` doesn't warn about daemon restart

**File**: `crates/nexus42/src/commands/runtime_mode.rs:79-106`

When the user changes the runtime mode via `nexus42 runtime-mode set`, the daemon (if running) is not notified of the change. The daemon reads its own config independently. The CLI config and daemon config can drift.

**Recommendation**: At minimum, print a warning: "Note: if the daemon is running, restart it for the change to take effect."

#### S-008: `clone.rs` command depends on daemon but doesn't check runtime mode

**File**: `crates/nexus42/src/commands/clone.rs:92-143`

The clone command calls the daemon but does not check if the current runtime mode allows platform operations. In `local_only` mode, cloning from platform should be blocked.

**Recommendation**: Add a runtime mode check before the daemon call, or let the daemon enforce it (but the CLI should also guard).

### Nits (N-001 to N-003)

#### N-001: `identity.rs` `ResolvedIdentity` has many `#[allow(dead_code)]`

**File**: `crates/nexus42/src/commands/identity.rs:246-328`

Five `#[allow(dead_code)]` annotations suggest this type was designed for future use but not yet wired into any command. Consider marking the module or struct as `#[doc(hidden)]` or wiring it into at least one command.

#### N-002: `doctor.rs` `version_check` reports `Ok` when daemon is unreachable

**File**: `crates/nexus42/src/commands/doctor.rs:352-359`

When the daemon is unreachable, `version_check` returns `HealthStatus::Ok` with detail "CLI vX (daemon not reachable, cannot compare)". This is misleading — a version check that cannot compare versions should be `Warning` at minimum.

#### N-003: `memory_pending_review` table has no index on `creator_id`

**File**: `crates/nexus-local-db/src/schema.rs:186-195`

The `list_pending_reviews` and `count_pending_reviews` queries filter by `creator_id` but there's no index on that column. For small datasets this is fine, but as the review queue grows, a full table scan will occur.

**Recommendation**: Add `CREATE INDEX IF NOT EXISTS idx_pending_review_creator ON memory_pending_review(creator_id);`

## Cross-Reviewer Ready Notes

**Integration risks identified**:
1. **C-001** (review classification) is the highest-risk finding — if the classification algorithm is too naive, it will produce poor memory promotion decisions that compound over time. This is the reviewer's primary concern for long-term evolvability.
