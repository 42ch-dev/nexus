### 🟡 Warning

#### W-001: Cross-platform test failure — `kill -0` without platform guard
The `nt1_2_shutdown_kills_persistent_child` test in `crates/nexus-agent-host/src/providers/native_cli/claude.rs` uses `tokio::process::Command::new("kill").arg("-0")` to verify child process termination. `kill -0` is POSIX-specific and will fail on Windows CI.

- **Impact**: Test suite breakage on Windows; the production `shutdown()` code itself is cross-platform (uses `child.kill()` via tokio).
- **Fix**: Add `#[cfg(unix)]` guard to the test, or provide a Windows-equivalent verification (e.g., `tasklist` / `Get-Process` check, or `OpenProcess` + `WaitForSingleObject`).
- **Source**: `crates/nexus-agent-host/src/providers/native_cli/claude.rs:1227-1235`

#### W-002: Silent CLI argument failure for `--kb-type`
In `crates/nexus42/src/commands/platform/context.rs:594-599`, invalid `--kb-type` values are silently ignored:

```rust
if let Some(bt_str) = kb_type {
    if let Ok(bt) = serde_json::from_value::<nexus_contracts::BlockType>(
        serde_json::Value::String(bt_str.to_string()),
    ) {
        request = request.with_kb_block_type(bt);
    }
}
```

The `if let Ok(...)` pattern silently drops parse failures. A user passing `--kb-type invlaid` receives no feedback that their filter was ignored.

- **Impact**: Poor UX; user may incorrectly believe a filter is active.
- **Fix**: Surface a warning or error when `kb_type` is provided but cannot be parsed to a valid `BlockType`.
- **Source**: `crates/nexus42/src/commands/platform/context.rs:594-599`

#### W-003: Missing unit test for `kb_block_type` filter
`MomentRequest` carries `kb_block_type: Option<BlockType>` and the `assemble_moment` function passes it to `KbQuery::with_block_type()`. However, there is no test verifying that `kb_block_type` actually filters results by block type. Tests exist for `kb_limit` (line 712) and `kb_text_search` (line 756), but not for `kb_block_type`.

- **Impact**: Coverage gap; regression risk if `KbQuery` behavior changes.
- **Fix**: Add a test similar to `kb_query_respects_text_search` that seeds blocks of different `BlockType`s and asserts only the matching type is returned.
- **Source**: `crates/nexus-moment-context-assembly/src/moment.rs`

### 🟢 Suggestion

#### S-001: Magic number in personality section split
`split_stage0_personality()` hardcodes `14` as the byte offset for `"## Personality"`.len():

```rust
let end_of_personality = after_personality_header[14..] // skip "## Personality"
```

- **Rationale**: If the heading format changes (e.g., extra space, different case), the magic number silently breaks.
- **Fix**: Use `"## Personality".len()` or a named constant.
- **Source**: `crates/nexus-moment-context-assembly/src/moment.rs:262`

#### S-002: `run_assemble_moment` parameter count
`run_assemble_moment` has 11 positional parameters, requiring `#[allow(clippy::too_many_arguments)]`. The inline comment acknowledges this should be "refactored into builder" but no timeline is given.

- **Rationale**: High parameter count reduces readability and increases caller-side error risk.
- **Fix**: Refactor into a `MomentAssemblyRequest` builder or config struct. This is acceptable as pre-release technical debt but should be tracked.
- **Source**: `crates/nexus42/src/commands/platform/context.rs:540-557`

#### S-003: Consider truncation budget accounting precision
`truncate_section` returns `0` when truncation occurs, but `truncate_text` may drop additional characters to break at a newline boundary. This means the total output may undershoot the budget by up to one line length per truncated section. The behavior is safe (never exceeds budget) but could leave usable tokens unused.

- **Rationale**: Minor inefficiency in token budget utilization.
- **Fix**: Document the "break-at-newline" behavior in the public API docs, or consider allowing the last section to reclaim leftover budget.
- **Source**: `crates/nexus-moment-context-assembly/src/moment.rs:274-298`
