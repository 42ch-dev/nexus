### 🟡 Warning
- **F-001: Non-Send futures in `assemble_moment` limit runtime composability** → `assemble_moment` produces non-`Send` futures (annotated `#[allow(clippy::future_not_send)]`) due to the transitive dependency on `InMemoryKbStore`'s `std::sync::RwLock` guards inside `KbStore::async fn` methods. This means `assemble_moment` cannot be spawned via `tokio::spawn()` on a multi-threaded runtime — it must be `.await`'d directly. For the daemon runtime where moment assembly may need to run concurrently with other tasks, this limits deployment flexibility. The `KbStore` trait's `#[allow(async_fn_in_trait)]` design choice precludes `Send` bounds on returned futures. **Fix**: Consider migrating `KbStore` to `#[async_trait]` (as `KnowledgeStore` already does) to enable `Send` bounds, or document the spawn limitation explicitly in the daemon integration path.
- **F-002: Silent error swallowing in `assemble_moment` with no observability** → `assemble_moment` (moment.rs:203-229) uses `Err(_) => (None, None)` and `_ => None` to gracefully degrade on domain storage failures. While graceful degradation is correct and documented, there is no `tracing::warn!`, `tracing::error!`, or any diagnostic emission when a domain source fails. A persistent storage error (e.g., corrupted SQLite in a future backend) would silently produce partial context with zero signal to the operator. **Fix**: Add `tracing::warn!("moment assembly: {domain} query failed: {err}")` for each caught failure, gated behind `tracing` dependency.
- **F-003: Nested lock acquisition in `get_narrative_context` Phase 2 — fragile pattern** → `get_narrative_context` (gateway.rs:280-303) acquires `events` read lock and, while holding it, calls `self.read_forks()?` to acquire a second `forks` read lock. This is currently safe because both are read locks (`std::sync::RwLock` allows multiple concurrent readers), and no write lock inversion exists. However, the nested lock pattern is fragile: if future code adds a write acquisition path (e.g., mutating events while holding the forks write lock), this would deadlock. **Fix**: Restructure Phase 2 to release the events lock before acquiring forks: collect `branch_events` data first, drop the events guard, then acquire forks.

### 🟢 Suggestion
- **F-004: Lock type inconsistency across in-memory stores** → `InMemoryKbStore` uses `std::sync::RwLock` (blocking) while `InMemoryKnowledgeStore` uses `tokio::sync::RwLock` (async-aware). This is intentional and documented (different trait designs: `KbStore` avoids `#[async_trait]`, `KnowledgeStore` uses it), but creates a footgun for maintainers. If future code adds an `.await` inside a `std::sync::RwLock` guard in `InMemoryKbStore`, it would block a tokio worker thread. **Fix**: Add a doc comment to `InMemoryKbStore` warning about the blocking lock in async context, or add a CI lint to detect blocking locks held across `.await`.
- **F-005: `get_narrative_context` acquires events read lock up to 3 times per query** → Phase 2 branch resolution (gateway.rs:281), Phase 2 fork detection (gateway.rs:296), and Phase 3 event resolution (gateway.rs:327 or 331) each call `self.read_events()?` independently. For a heavy-read workload, consider caching the read guard across phases within the method scope. In practice, the in-memory read lock is uncontended and cheap; this is only relevant for future disk-backed stores.
- **F-006: `InMemoryKnowledgeStore::store` silently overwrites on duplicate keys** → The composite key `user_id::entry_id` uses a UUID-based `entry_id`, making collisions practically impossible. However, if the store is seeded with externally-generated IDs, a duplicate insert silently replaces the previous entry with no diagnostic. **Fix**: Consider returning an error or at minimum logging a `tracing::debug!` on overwrite.
- **F-007: `InMemoryNarrativeGateway::kb_store` field is `#[allow(dead_code)]`** → The field is a required constructor parameter and adds a `K: KbStore` generic bound to the struct, but is never accessed in the V1.23 read-only gateway methods. The comment documents this as "wired for future context enrichment." This is acceptable for V1.23 but adds type-level complexity without runtime benefit. **Fix**: Consider making it `Option<K>` or wrapping in a feature gate until the enrichment path is implemented.
- **F-008: Case-insensitive comparison allocates on hot path** → `KnowledgeEntry::content_contains` (knowledge.rs:62-64) calls `content.to_lowercase()` and `query.to_lowercase()`, allocating two `String` per comparison. In `InMemoryKnowledgeStore::list` (store.rs:139-157), this runs for every entry on a text-filtered query — O(n) allocations. For in-memory prototyping this is negligible, but for future production stores it may become a hot-path bottleneck. **Fix**: Consider `unicase` or manual case-insensitive comparison without allocation when porting to production.
