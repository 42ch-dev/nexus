## Strengths

### 1. Schema Design Quality

All three new schemas follow JSON Schema best practices:

| Schema | Required Fields | Enum Fields | Refs to Common | `additionalProperties` |
|--------|----------------|-------------|----------------|----------------------|
| `delta.schema.json` | 4 (`delta_type`, `operation`, `payload`, `local_timestamp`) | 2 (inline) | `SourceAnchor`, `Timestamp` | ✅ `false` |
| `workspace-binding.schema.json` | 7 | 1 (`binding_status`) | `SchemaVersion`, `WorkspaceId`, `WorldId`, `CreatorId`, `Timestamp` | ✅ `false` |
| `agent-profile.schema.json` | 6 | 4 (inline) | `Timestamp` | ✅ `false` |

**Security Note**: `additionalProperties: false` prevents schema extension attacks where malicious clients could inject unexpected fields.

### 2. Correct Enum Definitions

**Delta Schema** (aligned with v1-spec §5.12):
```json
"delta_type": {
  "enum": ["world", "key_block", "timeline_event", "fork_branch", "memory_item", "story_manifest"]
}
```

**WorkspaceBinding Schema** (§5.14):
```json
"binding_status": {
  "enum": ["active", "unlinked", "stale"]
}
```

**AgentProfile Schema** (§5.15):
```json
"profile_kind": ["primary", "specialist", "fallback"]
"selection_mode": ["explicit", "registry", "auto"]
"transport": ["local", "http", "stdio"]
"status": ["active", "inactive", "error"]
```

All enums use snake_case string values, ensuring consistent serialization across Rust and TypeScript.

### 3. Codegen Tooling Improvements

The codegen tooling (`rust-generator.ts`, `ts-generator.ts`) was updated to handle:

1. **Cross-module imports**: When a schema references another schema's top-level type (e.g., `Delta`), the generator now emits proper `use` statements:
   ```rust
   // Rust: use crate::generated::delta::Delta;
   // TypeScript: import type { Delta } from './Delta';
   ```

2. **Inline enum naming**: Uses PascalCase path prefixes (e.g., `AgentProfileProfileKind`) to avoid name collisions across modules.

3. **Module ordering**: `mod.rs` now sorts modules alphabetically for deterministic output.

### 4. Generated Code Quality

**TypeScript** (`Delta.ts`):
```typescript
export type DeltaDeltaType = 'world' | 'key_block' | 'timeline_event' | 'fork_branch' | 'memory_item' | 'story_manifest';
export type DeltaOperation = 'create' | 'update' | 'upsert' | 'delete' | 'append';

export interface Delta {
  delta_type: DeltaDeltaType;
  operation: DeltaOperation;
  target_entity_type?: string;
  target_entity_id?: string;
  payload: Record<string, unknown>;
  source_anchor?: SourceAnchor;
  local_timestamp: string;
}
```

**Rust** (`delta.rs`):
```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
#[serde(rename_all = "snake_case")]
pub struct Delta {
    pub delta_type: String,
    pub operation: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub target_entity_type: Option<String>,
    // ...
}
```

Both outputs are idiomatic and type-safe.

## Security & Correctness Review (Reviewer #2 Primary Accent)

### Input Validation ✅

| Schema | Validation Strategy |
|--------|-------------------|
| `delta.schema.json` | Enum validation on `delta_type` and `operation`; `payload` is `Record<string, unknown>` (intentionally flexible for delta payloads) |
| `workspace-binding.schema.json` | Enum validation on `binding_status`; all IDs use common type refs (string with format constraints at application layer) |
| `agent-profile.schema.json` | Enum validation on 4 fields; `protocol_version` is string (semver validated at application layer) |

**Assessment**: Schemas correctly use JSON Schema constraints. Runtime validation will catch invalid enum values.

### Auth Boundaries ✅

**AgentProfile Schema**:
- `profile_kind` enum includes `primary`, `specialist`, `fallback` — enables role-based agent selection
- `selection_mode` enum includes `explicit`, `registry`, `auto` — controls how agents are resolved
- `status` enum includes `active`, `inactive`, `error` — enables state-based access control

**WorkspaceBinding Schema**:
- `binding_status` enum includes `active`, `unlinked`, `stale` — enables workspace-world relationship validation
- `creator_id` field is required — ensures audit trail for binding creation

**Assessment**: Schemas provide the data model for auth boundaries; application-layer enforcement is out of scope for this review.

### Sensitive Data Handling ✅

- No PII fields in any of the three schemas
- `local_root` in `WorkspaceBinding` is a filesystem path (not transmitted over network in V1.0)
- `protocol_version` in `AgentProfile` is a version string (not user-controlled)

**Assessment**: No sensitive data exposure risks in schema design.

### Exception Paths ✅

- All optional fields use `?` (TypeScript) or `Option<T>` (Rust) — correctly handles missing data
- `payload` in `Delta` is `Record<string, unknown>` / `serde_json::Value` — allows flexible error payloads
- `source_anchor` in `Delta` is optional — provenance tracking is not required for all deltas

**Assessment**: Schemas handle partial data correctly.

### State Consistency ✅

**Delta Schema**:
- `local_timestamp` is required — ensures temporal ordering
- `delta_type` + `operation` + `target_entity_type` + `target_entity_id` form a complete addressing scheme

**WorkspaceBinding Schema**:
- `created_at` is required, `updated_at` is optional — tracks lifecycle
- `binding_status` is required — state machine invariant

**AgentProfile Schema**:
- `status` is required — lifecycle tracking
- `created_at` and `updated_at` are optional (not required for in-memory profiles)

**Assessment**: Temporal and state tracking fields are appropriately required/optional.

### 🟡 Warning

#### W-001: Permission Policy File Race Condition

**Location**: `crates/nexus42/src/acp/policy.rs:87-101` (`PermissionPolicy::save()`)

**Issue**: The `save()` method performs a non-atomic read-modify-write operation when called multiple times in quick succession. If two CLI invocations simultaneously modify permissions, the second write may overwrite the first.

**Code**:
```rust
pub fn save(&self, workspace_root: &Path) -> anyhow::Result<()> {
    let policy_path = Self::policy_path(workspace_root);
    // ...
    let content = toml::to_string_pretty(self)?;
    std::fs::write(&policy_path, content)?;  // Non-atomic write
    Ok(())
}
```

**Impact**: Low for single-user scenarios. In team environments where multiple developers might run `nexus42 policy grant` simultaneously, permission changes could be lost.

**Recommendation**: 
1. Use file locking (`fs2` crate) for exclusive access
2. Or implement atomic writes via temp file + rename pattern
3. Add a warning in documentation about concurrent modification

**Cross-Reviewer Note**: QC#1 may have different perspective on operational risk vs implementation complexity tradeoff.

### Warning (Should Fix)

#### W-1: `interact_to_rusqlite_err` misuses `InvalidParameterName` error variant

- **File**: `crates/nexus-sync/src/pool.rs:163-171`
- **Task**: DEBT-X1
- **Severity**: Warning
- **Evidence**:
  ```rust
  fn interact_to_rusqlite_err(e: InteractError) -> rusqlite::Error {
      match e {
          InteractError::Panic(p) => {
              rusqlite::Error::InvalidParameterName(format!("Connection interact panicked: {:?}", p))
          }
          InteractError::Aborted => {
              rusqlite::Error::InvalidParameterName("Connection interact aborted".into())
          }
      }
  }
  ```
- **Issue**: Both panic and abort conditions are mapped to `InvalidParameterName`, which is semantically wrong. This error variant is intended for named parameter binding failures, not for general runtime errors. Callers who match on specific `rusqlite::Error` variants will get incorrect categorization. For example, an `InvalidParameterName` match arm expecting a SQL bind parameter name would incorrectly catch a connection panic.
- **Recommendation**: Use a more appropriate variant. Options:
  - `rusqlite::Error::ToSqlConversionFailure(Box::new(...))` with a custom error type
  - `rusqlite::Error::ExecuteReturnedResults` (semantically closer for general operation failures)
  - Create a custom error wrapper that preserves the `InteractError` type information

#### W-2: Body size enforcement reads full response before size check

- **File**: `crates/nexus-sync/src/sync_client.rs:258-284` (push_bundle), `:344-369` (pull_sync_state)
- **Task**: SYNC-R5
- **Severity**: Warning
- **Evidence**: Both `push_bundle` and `pull_sync_state` follow this pattern:
  ```rust
  // Check Content-Length header (only if present)
  if let Some(content_length) = response.headers().get(reqwest::header::CONTENT_LENGTH) {
      // ... parse and check ...
  }

  // Read ENTIRE body into memory
  let text = response.text().await?;

  // THEN check actual size
  if text.len() > self.body_max_size {
      return Err(...);
  }
  ```
- **Issue**: If the `Content-Length` header is missing (possible with chunked transfer encoding), streaming responses, or a malicious/broken server, the full response body is read into memory *before* the size check. This defeats the purpose of the body size limit for memory protection — a 1GB response would consume 1GB of memory before the check triggers.
- **Recommendation**: Use `reqwest`'s streaming body reader with a byte limit, or set the limit at the HTTP client level (e.g., `reqwest::Client::builder().read_timeout()` combined with manual chunk reading). Alternatively, use `response.bytes_stream()` with `take(n)` to enforce the limit incrementally.
