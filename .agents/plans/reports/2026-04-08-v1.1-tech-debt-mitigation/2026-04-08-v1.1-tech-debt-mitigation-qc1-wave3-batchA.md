#### W-2: HTTP body size check uses Serialization error variant
- **Issue**: `sync_client.rs:263-267, 278-284` returns `SyncError::Serialization` for body size limit violations. This conflates serialization errors with resource limit errors, potentially confusing monitoring/alerting systems.
- **Impact**: Error categorization for monitoring becomes ambiguous (serialization errors may trigger different alerting than resource limits).
- **Recommendation**: Consider adding a dedicated `SyncError::BodySizeLimitExceeded { limit: usize, actual: usize }` variant for clearer error handling. This aligns with the error_code() standardization goal.
- **Severity**: Warning (current implementation is functional, but error taxonomy could be clearer)
- **Source**: Manual review of sync_client.rs:258-284

#### W-3: Pool status monitoring not exposed in Outbox API
- **Issue**: `OutboxPool::status()` method exists (pool.rs:59-61) but `Outbox` struct does not expose pool health/status monitoring capabilities.
- **Impact**: Callers cannot monitor pool utilization, connection leaks, or performance degradation without accessing the pool directly.
- **Recommendation**: Add `Outbox::pool_status()` method delegating to the internal pool's status. This enables operational monitoring without breaking encapsulation.
- **Severity**: Warning (nice-to-have for observability, not blocking)
- **Source**: Architecture review of pool.rs + outbox.rs

#### S-1: Connection pool timeout configuration
- **Issue**: `pool.rs` uses deadpool-sqlite with default timeout configuration. No explicit timeout/waits configuration for pool exhaustion scenarios.
- **Impact**: Under high load with pool exhaustion, operations may hang indefinitely or timeout with unclear errors.
- **Recommendation**: Consider adding `Config::wait_timeout()` and `Config::create_timeout()` to the pool builder for predictable behavior under load.
- **Severity**: Suggestion (default behavior is acceptable for current pool size)

- **Impact**: Future changes to either outbox_entries or daemon outbox schema may diverge unintentionally.

- **Severity**: Suggestion (explicitly noted as future work in closure_note)
- **Source**: Manual review of outbox.rs:12-18, status.json DEBT-X2 closure_note

#### S-3: Test isolation for in-memory databases
- **Issue**: `Outbox::new_in_memory()` (outbox.rs:148-160) leaks tempfile::TempDir via `std::mem::forget(tmp)` for test isolation. While this is acceptable for tests, it creates unnecessary disk I/O and potential cleanup issues in long-running test suites.
- **Impact**: Minimal (test-only code), but could accumulate temp directories in `/tmp` during long test runs.
- **Recommendation**: Consider using `rusqlite::Connection::open_in_memory()` directly for pure in-memory tests, or document the tempfile leak rationale in the test method.
- **Severity**: Suggestion (test code quality improvement)
- **Source**: Manual review of outbox.rs:148-160
