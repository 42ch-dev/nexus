### Warning (Should Fix)

#### W-1: `interact_to_rusqlite_err` misuses `InvalidParameterName` error variant

- **File**: `crates/nexus-sync/src/pool.rs:163-171`
- **Task**: DEBT-X1
- **Severity**: Warning
- **Evidence**:
  ```rust
  fn interact_to_rusqlite_err(e: InteractError) -> rusqlite::Error {
      match e {
          InteractError::Panic(p) => {
              rusqlite::Error::InvalidParameterName(format!("Connection interact panicked: {:?}", p))
          }
          InteractError::Aborted => {
              rusqlite::Error::InvalidParameterName("Connection interact aborted".into())
          }
      }
  }
  ```
- **Issue**: Both panic and abort conditions are mapped to `InvalidParameterName`, which is semantically wrong. This error variant is intended for named parameter binding failures, not for general runtime errors. Callers who match on specific `rusqlite::Error` variants will get incorrect categorization. For example, an `InvalidParameterName` match arm expecting a SQL bind parameter name would incorrectly catch a connection panic.
- **Recommendation**: Use a more appropriate variant. Options:
  - `rusqlite::Error::ToSqlConversionFailure(Box::new(...))` with a custom error type
  - `rusqlite::Error::ExecuteReturnedResults` (semantically closer for general operation failures)
  - Create a custom error wrapper that preserves the `InteractError` type information

#### W-2: Body size enforcement reads full response before size check

- **File**: `crates/nexus-sync/src/sync_client.rs:258-284` (push_bundle), `:344-369` (pull_sync_state)
- **Task**: SYNC-R5
- **Severity**: Warning
- **Evidence**: Both `push_bundle` and `pull_sync_state` follow this pattern:
  ```rust
  // Check Content-Length header (only if present)
  if let Some(content_length) = response.headers().get(reqwest::header::CONTENT_LENGTH) {
      // ... parse and check ...
  }

  // Read ENTIRE body into memory
  let text = response.text().await?;

  // THEN check actual size
  if text.len() > self.body_max_size {
      return Err(...);
  }
  ```
- **Issue**: If the `Content-Length` header is missing (possible with chunked transfer encoding), streaming responses, or a malicious/broken server, the full response body is read into memory *before* the size check. This defeats the purpose of the body size limit for memory protection — a 1GB response would consume 1GB of memory before the check triggers.
- **Recommendation**: Use `reqwest`'s streaming body reader with a byte limit, or set the limit at the HTTP client level (e.g., `reqwest::Client::builder().read_timeout()` combined with manual chunk reading). Alternatively, use `response.bytes_stream()` with `take(n)` to enforce the limit incrementally.

#### W-3: Test helper `new_in_memory()` leaks temp directories

- **File**: `crates/nexus-sync/src/outbox.rs:204-213`
- **Task**: DEBT-X1
- **Severity**: Warning
- **Evidence**:
  ```rust
  #[cfg(test)]
  pub async fn new_in_memory() -> SyncResult<Self> {
      let tmp = tempfile::TempDir::new().map_err(|e| SyncError::OutboxDatabase(e.to_string()))?;
      let db_path = tmp.path().join("test_outbox.db");
      std::mem::forget(tmp); // Keep temp dir alive by leaking it (test only)
      Self::new(&db_path).await
  }
  ```
- **Issue**: Each call to `new_in_memory()` leaks a `TempDir` via `std::mem::forget()`. With 28+ tests using this helper, every test run leaks ~28 temp directories. While acceptable for local dev, this could accumulate disk usage in CI environments running thousands of test iterations. The temp directories are never cleaned up.
- **Recommendation**: Consider using a shared test fixture or `TempDir` stored in a test-scoped static, or accept the leak with a comment noting CI cleanup strategy.

#### S-2: `Outbox::with_pool` documents caller responsibility but still initializes WAL

- **File**: `crates/nexus-sync/src/outbox.rs:191-200`
- **Task**: DEBT-X1
- **Severity**: Suggestion
- **Evidence**:
  ```rust
  /// Note: Caller is responsible for ensuring the schema is initialized.
  pub async fn with_pool(pool: OutboxPool) -> SyncResult<Self> {
      let conn = pool.get().await.map_err(|e| SyncError::OutboxDatabase(e.to_string()))?;
      conn.execute_batch("PRAGMA journal_mode=WAL;").await?;
      drop(conn);
      Ok(Self { pool })
  }
  ```
- **Issue**: The doc says "caller is responsible for ensuring the schema is initialized" but the method itself initializes WAL mode. This is slightly inconsistent — if the caller is responsible, why is WAL mode set here? If WAL mode is required, it should be documented as a precondition rather than silently applied.
- **Recommendation**: Either document that `with_pool` sets WAL mode (making it a documented side effect), or move WAL initialization to the caller's responsibility.
