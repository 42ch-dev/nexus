## Findings

### 🔴 Critical

#### QC3-ARCH-1: SimpleClientHandler has daemon_client field but doesn't use it
- **Issue**: `SimpleClientHandler` in `client.rs` (lines 166-168) stores a `daemon_client` but `execute_tool()` in `nexus42d/src/api/handlers/acp.rs` (lines 137-195) performs **direct file system operations** via `std::fs::read_to_string()` and `std::fs::write()`, not through the daemon client.
- **Impact**: The "daemon-mediated" architecture (ACP-R8) is not actually implemented. Agents calling `fs/read_text_file` or `fs/write_text_file` will have their file operations execute directly on the CLI machine, bypassing the daemon's workspace path validation and audit logging.
- **Fix**: Remove `daemon_client` from `SimpleClientHandler` OR wire `SimpleClientHandler` to call the daemon's `/v1/local/acp/tool/execute` endpoint for file operations.

#### QC3-ARCH-2: max_file_size parameter parsed but not passed to daemon
- **Issue**: In `crates/nexus42/src/commands/context.rs` (lines 72-74, 123-126), the `max_file_size` option is parsed from CLI arguments but is **never passed** to the daemon API. Comment at line 126-128 explicitly states: "It will be used when SummaryGenerator is integrated into the context assembly workflow."
- **Impact**: The CTX-R2 feature (file size limit) is not functional despite being committed. Users specifying `--max-file-size` will see no effect.
- **Fix**: Either (a) wire `max_file_size` to the daemon API request, or (b) implement SummaryGenerator in the CLI and remove the TODO comment. Do not accept commits with non-functional features.

### 🟡 Warning

#### QC3-MAINTAIN-1: Interactive permission prompt not implemented (always defaults to Deny)
- **Issue**: In `crates/nexus42/src/acp/client.rs` (lines 93-96), `PolicyAwareClientHandler::prompt_user()` has a TODO comment and defaults to `PermissionDecision::Deny` with a warning log. The "proper interactive prompting" is deferred.
- **Impact**: When `PermissionPolicy::evaluate()` returns `PermissionDecision::Ask`, the user is **not actually prompted**. Permission is automatically denied. This breaks the intended UX where users should grant/deny permissions interactively.
- **Fix**: Implement interactive prompting with `dialoguer` or similar crate before marking ACP-R7 as resolved. Until then, `PolicyAwareClientHandler` cannot be used in production.

#### QC3-MAINTAIN-2: Audit log agent_id and session_id always NULL
- **Issue**: In `nexus42d/src/api/handlers/acp.rs` (line 278), the INSERT statement passes `NULL` for `agent_id` and `session_id`:
  ```rust
  conn.execute(
      "INSERT INTO acp_tool_audit_log (tool_name, path, outcome, agent_id, session_id)
       VALUES (?1, ?2, ?3, NULL, NULL)",
      ...
  );
  ```
- **Impact**: Audit log entries cannot be correlated to specific agents or sessions. Future debugging and security auditing will be severely limited.
- **Fix**: Pass actual `agent_id` and `session_id` from the request context.

#### QC3-MAINTAIN-3: Path validation for non-existent paths uses string prefix comparison
- **Issue**: In `nexus42d/src/api/handlers/acp.rs` (lines 87-101), when a path doesn't exist (e.g., for write operations), the validation checks if the absolute path **string starts with** the workspace path string. This is a lexigraphic check, not a proper path canonicalization.
- **Impact**: A path like `/workspace/foo/../etc/passwd` would pass the string check (starts with `/workspace/`) but could resolve to `/etc/passwd` after canonicalization. Though the actual fs operations would fail for non-existent paths outside workspace, the validation is semantically incorrect.
- **Fix**: For non-existent paths, canonicalize the parent directory path and compare canonical forms.

### 🟢 Suggestion

#### QC3-SUGGEST-1: Consider adding tests for PermissionPolicy edge cases
- **Issue**: `policy.rs` has good unit tests for basic grant/deny logic, but missing tests for:
  - Concurrent modification of policy (RwLock semantics not tested)
  - TOML file corruption handling
  - Empty policy file edge case
- **Suggestion**: Add tests for these scenarios before V1.1 GA.

#### QC3-SUGGEST-2: SessionManager cleanup uses <= 24 hours (boundary condition)
- **Issue**: In `session_manager.rs` (line 127), the cleanup condition is `now.signed_duration_since(s.last_used_at) <= max_age`. Sessions at exactly 24 hours are removed.
- **Suggestion**: Consider using `< max_age` (strict less-than) so sessions at exactly the boundary are retained. Document the behavior clearly.

#### QC3-SUGGEST-3: WorldId validation function is not used by all consumers
- **Issue**: `validate_world_id()` in `context.rs` is used as a clap validator for the `world_id` argument, but other code paths that create WorldIds may not use this validation.
- **Suggestion**: Consider extracting `validate_world_id` to a shared location (e.g., `nexus-domain`) and using it consistently across all WorldId creation points.
