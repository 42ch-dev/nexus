### 🟡 Warning

#### W-001: Permission Policy File Race Condition

**Location**: `crates/nexus42/src/acp/policy.rs:87-101` (`PermissionPolicy::save()`)

**Issue**: The `save()` method performs a non-atomic read-modify-write operation when called multiple times in quick succession. If two CLI invocations simultaneously modify permissions, the second write may overwrite the first.

**Code**:
```rust
pub fn save(&self, workspace_root: &Path) -> anyhow::Result<()> {
    let policy_path = Self::policy_path(workspace_root);
    // ...
    let content = toml::to_string_pretty(self)?;
    std::fs::write(&policy_path, content)?;  // Non-atomic write
    Ok(())
}
```

**Impact**: Low for single-user scenarios. In team environments where multiple developers might run `nexus42 policy grant` simultaneously, permission changes could be lost.

**Recommendation**: 
1. Use file locking (`fs2` crate) for exclusive access
2. Or implement atomic writes via temp file + rename pattern
3. Add a warning in documentation about concurrent modification

**Cross-Reviewer Note**: QC#1 may have different perspective on operational risk vs implementation complexity tradeoff.

### 🟢 Suggestions

#### S-001: Session Cleanup Could Be Configurable

**Location**: `crates/nexus42/src/acp/session_manager.rs:127-152` (`cleanup_expired()`)

**Observation**: The 24-hour expiration is hardcoded. Some users may want shorter sessions (security-conscious) or longer sessions (long-running projects).

**Recommendation**: Expose session TTL via workspace config or environment variable.

```rust
// Current: hardcoded 24 hours
let max_age = chrono::Duration::hours(24);

// Suggested: configurable
let max_age = chrono::Duration::hours(
    std::env::var("NEXUS_SESSION_TTL_HOURS")
        .ok()
        .and_then(|s| s.parse().ok())
        .unwrap_or(24)
);
```

**Location**: `crates/nexus42/src/commands/context.rs:11-35` (`validate_world_id()`)

**Observation**: Manual character-by-character validation works but is verbose. A regex would be more maintainable if the format evolves.

**Code**:

// Current: manual checks
if !s.starts_with("wld_") { ... }

**Tradeoff**: Current approach has zero dependencies; regex adds maintenance burden but centralizes format logic.

## Thread Safety Analysis

### SessionManager ⚠️

**Location**: `crates/nexus42/src/acp/session_manager.rs`

**Observation**: `SessionManager` is `Clone` but does not use internal synchronization. If shared across threads:

1. `load_sessions()` → race condition if another thread calls `save_session()`
2. `save_session()` → non-atomic read-modify-write

**Current Usage Pattern**: In `commands/session.rs`, the manager is created per-command, so concurrent access is unlikely in practice.

**Recommendation**: If future usage involves shared state (e.g., long-running daemon context), wrap in `Arc<Mutex<SessionManager>>` or use a database backend.

---

### PermissionPolicy ✅

**Location**: `crates/nexus42/src/acp/policy.rs`

**Observation**: `PermissionPolicy` is loaded once per command invocation and mutated in-memory before saving. No shared mutable state across threads.

**Verdict**: Thread-safe for current usage patterns.
