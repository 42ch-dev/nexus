### 🟡 Warning

**W-1: Double construction of `build_schema_map()` wastes ~51 boxed closures per test run**
- **Issue**: In `schema_drift_detection()`, `build_schema_map()` is called directly to get `entries`. Then `build_schema_cache()` calls `collect_all_schema_paths()`, which calls `build_schema_map()` a second time to iterate over `schema_path` values. Each invocation constructs ~51 `Box<dyn Fn(Value) -> Result<...>>` checkers. This is pure overhead with no benefit.
- **Fix**: Pass `&entries` into `build_schema_cache()` (or `collect_all_schema_paths()`) so the already-built map is reused. Alternatively, split schema-path collection from checker construction.
- **Source**: `crates/nexus-contracts/tests/schema_drift_detection.rs` lines ~352, ~760

**W-2: Drift detection time threshold (`5_000 ms`) is too generous to catch regressions**
- **Issue**: The test currently runs in ~18 ms for 52 schemas. A 5 s threshold allows a ~250x slowdown before failure, making the assertion ineffective at catching schema-growth or algorithmic regressions on typical dev machines. The comment acknowledges the threshold is generous for "slow CI runners," but this pushes the problem to CI while dev regressions go unnoticed.
- **Fix**: Lower the default limit to `500 ms` (still ~25x headroom) and allow CI override via an environment variable (e.g., `NEXUS_DRIFT_LIMIT_MS`). This keeps the test useful locally without flaking on slow runners.
- **Source**: `crates/nexus-contracts/tests/schema_drift_detection.rs` line 754

**W-3: Compiler warning — unused variable `entries` in `collect_all_schema_paths()`**
- **Issue**: `cargo test -p nexus-contracts --test schema_drift_detection` emits:
  ```
  warning: unused variable: `entries`
     --> crates/nexus-contracts/tests/schema_drift_detection.rs:359:15
  ```

  Per harness rules, compiler warnings in the changed scope default to `>= Warning` and must be resolved before `Approve`.
- **Fix**: Rename `entries` to `_entries`, or remove the outer `if let Ok(entries) = ...` guard since `collect_schema_files_recursive` performs its own `read_dir`.
- **Source**: `crates/nexus-contracts/tests/schema_drift_detection.rs` line 359

**W-4: `exit_starting` doc comment promises a grace window that does not exist**
- **Issue**: The rustdoc for `exit_starting` states:
  > "Gracefully cancels in-flight subsystem start tasks by signalling a shared `CancellationToken`, then awaits their completion within a short grace window. Falls back to `abort()` only if tasks don't respond to cancellation in time."
  The actual implementation calls `ctx.start_cancel.cancel()`, then immediately checks `handle.is_finished()` and aborts if not finished. There is **no** grace window, no `await`, and no `timeout`. The comment is misleading for future maintainers and creates reliability risk if someone later assumes cooperative cancellation has time to clean up resources.
- **Fix**: Either (a) add a `tokio::time::timeout(Duration::from_millis(100), handle).await` before falling back to `abort()`, or (b) update the comment to accurately describe the immediate-abort behavior.
- **Source**: `crates/nexus42d/src/lifecycle/actions.rs` lines ~191-210

**W-5: Non-Unix worker supervisor timeout branch may orphan processes**
- **Issue**: In the `#[cfg(not(unix))]` branch of the worker supervisor, after `child.start_kill()` and a grace timeout, the code sends `WorkerEvent::Crashed` and returns without reaping the child (`child.wait()` is not called in the timeout branch). Dropping a `tokio::process::Child` does **not** guarantee the OS process is terminated, so a stubborn child could become an orphan.
- **Fix**: In the `Err(_)` (timeout) branch of the non-Unix path, add `let _ = child.wait().await;` to ensure the handle is reaped before the supervisor task drops it.
- **Source**: `crates/nexus-orchestration/src/worker/manager.rs` non-Unix `Err(_)` branch inside supervisor cancel handler

### 🟢 Suggestion

**S-1: Make `DRIFT_DETECTION_TIME_LIMIT_MS` overrideable via environment variable**
- A `const` limit is inflexible. Reading `std::env::var("NEXUS_DRIFT_LIMIT_MS").ok().and_then(|s| s.parse().ok()).unwrap_or(5_000)` would let CI set a high threshold while keeping dev runs strict.
- **Source**: `crates/nexus-contracts/tests/schema_drift_detection.rs` line 754

**S-2: Cache `workspace_root()` result**
- `workspace_root()` is called on every `load_json()` invocation and inside `collect_all_schema_paths()`. The path is immutable for the lifetime of the test. A `lazy_static!` or `OnceLock` would avoid redundant path manipulation, though the cost is negligible compared to JSON parsing.
- **Source**: `crates/nexus-contracts/tests/schema_drift_detection.rs` lines ~386-391
