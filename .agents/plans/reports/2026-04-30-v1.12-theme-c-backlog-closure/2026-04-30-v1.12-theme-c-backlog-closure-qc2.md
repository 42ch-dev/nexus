### Suggestions

#### S01 — `reset_agent_doc` does not clear action tables before removing agent (minor correctness edge case)

- **File**: `crates/nexus-acp-host/src/policy.rs:340-362`
- **Commit**: 6d49bf7
- **Description**: The `reset_agent_doc` function first clears all action sub-tables (`grant`, `deny`, `ask`), then removes the agent entry, then checks if `agents` table is empty. However, the clearing of action sub-tables is unnecessary since the entire agent entry is removed immediately after. The function works correctly but the action-clearing loop is dead code. This is cosmetic, not a bug.
- **Recommendation**: Simplify by removing the action-clearing loop (lines 344-349) since `t.remove(agent)` on line 353 already removes the entire agent subtree. Or document why the two-step approach is intentional (e.g., for future hooks).

#### S02 — `exit_starting` aborts tasks without awaiting them after cancellation signal

- **File**: `crates/nexus42d/src/lifecycle/actions.rs:215-228`
- **Commit**: 60c98b7
- **Description**: After `ctx.start_cancel.cancel()` is called, `exit_starting` drains handles and calls `handle.abort()` on non-finished tasks. However, `abort()` is called without awaiting the handle first. The cancellation token signal is cooperative — the task will check `cancel.is_cancelled()` at its next await point. Calling `abort()` immediately means the task may not get a chance to observe the cancellation and perform any cleanup at await points between the signal and the abort. The code comment acknowledges this ("abort as last resort"), but there is no grace window between `cancel()` and `abort()`.
- **Risk**: Low — the documented risk RISK-WSC-02 already notes that cancellation does not guarantee full resource cleanup. The abort fallback is a reasonable safety net. However, adding even a very short `tokio::time::sleep` (e.g., 10ms) between `cancel()` and `abort()` would give tasks a chance to observe the signal.
- **Recommendation**: Consider adding a short grace window in `exit_starting` between `cancel()` and `abort()`, or document that immediate abort is intentional to avoid blocking daemon startup transitions.

#### S03 — Schema drift glob does not guard against symlink traversal

- **File**: `crates/nexus-contracts/tests/schema_drift_detection.rs:367-381`
- **Commit**: 30984c3
- **Description**: The `collect_schema_files_recursive` function follows `read_dir` and `is_dir()` without checking for symlinks. If a symlink points outside the `schemas/` directory, it could traverse into unexpected locations. This is a test-only function, so the security impact is minimal, but it differs from the hardened `scan_with_traversal_attempt_fails` test in `nexus42`'s context summary module which explicitly rejects path traversal.
- **Recommendation**: Add a canonicalization or `starts_with` check after `strip_prefix` to ensure collected paths remain under `schemas/`. Low priority since this is test code operating on a trusted repository tree.

#### S04 — `reset_all_agents_doc` is a nuclear option with no confirmation

- **File**: `crates/nexus-acp-host/src/policy.rs:365-367`
- **Commit**: 6d49bf7
- **Description**: `reset_all_agents_doc` removes the entire `[agents]` table in one operation. The CLI caller (`run_reset` with no `--agent` flag) does check that agents exist before calling, but there is no dry-run or confirmation mechanism at the policy layer. This is a permission/security-relevant operation.
- **Note**: The CLI layer should handle user confirmation; the policy layer correctly provides a low-level primitive. This is an observation, not a defect.

## Security & Correctness Assessment by Change

### 1. `cfg(unix)` gating for `nix` dependency (2ec934f)
- **Correctness**: `#[cfg(unix)]` on imports and `#[cfg(unix)]` / `#[cfg(not(unix))]` on usage blocks are correctly paired. The `Cargo.toml` change to `target.'cfg(unix)'.dependencies` is the idiomatic way to gate platform-specific dependencies.
- **Security**: No security concerns. The non-Unix path uses `child.start_kill()` which is the standard tokio approach.
- **Event broadcasting**: The supervisor task correctly sends `WorkerEvent::Stopped` or `WorkerEvent::Crashed` in all branches.

### 2. Permission reset refactoring (6d49bf7)
- **Correctness**: The extracted `reset_agent_doc` and `reset_all_agents_doc` methods faithfully replicate the original inline logic. The pre-check (verifying agent rules exist before resetting) is preserved in `run_reset`.
- **TOML safety**: Uses `toml_edit::DocumentMut` mutation methods which preserve comments and formatting — no data loss risk.
- **Security**: No auth boundary changes. Reset operations are CLI-local and require filesystem access to the workspace.

### 3. Schema drift auto-derivation (30984c3)
- **Correctness**: `BTreeSet` ensures deterministic ordering. The dual-source approach (registered map + glob) correctly catches schemas both with and without Rust struct checkers.
- **CI gate**: **Fails** `clippy --all --all-targets -- -D warnings` due to unused `entries` variable (W01).
- **Performance**: The timing threshold (5000ms) is generous and includes observability logging — well-designed.

### 4. Graceful shutdown replacement (60c98b7)
- **Correctness**: Replacing `JoinHandle::abort()` with `CancellationToken` is a meaningful improvement. The `tokio::select!` pattern correctly races the subsystem start against cancellation.
- **State consistency**: The cancellation check after `Ok(())` and `Err(e)` prevents stale `SubsystemUp`/`SubsystemFailed` events from being dispatched during shutdown.
- **Residual risk**: `exit_starting` still uses `abort()` as a fallback, but now only after the cancellation signal. This is a best-effort graceful shutdown pattern.

### 5. Drift detection time threshold (0dcf997)
- **Correctness**: The `Instant` timing and `assert!` threshold are straightforward. The `eprintln!` observability line ensures timing data is visible even when the threshold is not exceeded.
- **No security concerns**.
