### 🟡 Warning

**W1 — Narrative handlers construct fresh gateway per request instead of using `WorkspaceState`**
- **Source**: `crates/nexus-daemon-runtime/src/api/handlers/narrative.rs:47,66`
- **Issue**: Both `list_worlds` and `get_world` accept `State<WorkspaceState>` but ignore it (`State(_state)`), constructing `InMemoryNarrativeGateway::new(InMemoryKbStore::new())` on every request. This means:
  - No narrative state is shared across requests (worlds seeded in one request are invisible to others)
  - Per-request allocation overhead
  - Inconsistent with other handler patterns that use `WorkspaceState` for store injection
- **Context**: Plan C1.1 explicitly noted "Store injection should be explicit in `WorkspaceState`; avoid constructing a new empty gateway per request unless PM accepts that stateless behavior in C0." C0 confirmed the in-memory approach, so this is an accepted tradeoff for the minimal surface, but it creates a future refactoring burden when persistent narrative storage is introduced.
- **Fix**: Add a shared `Arc<InMemoryNarrativeGateway>` (or `Arc<dyn NarrativeGateway>`) to `WorkspaceState` and inject it into handlers. This aligns with existing patterns (engine, worker_manager, capability_registry) and makes the transition to persistent stores a single-field change.

**W2 — No integration tests exercise the `creator kb --scope world` execution path**
- **Source**: `crates/nexus42/tests/command_surface_contract.rs:521-534`
- **Issue**: The only CLI-level test for `--scope world` is `v2_target_kb_scope_flag`, which validates help text only. There is no test that runs `nexus42 creator kb list --scope world --world-id <id>` (or `search`/`show`) and verifies it reaches `InMemoryKbStore` and produces deterministic output. The underlying `nexus-kb` crate has 48 unit tests, but the CLI wiring gap means regressions in argument parsing, error mapping, or scope routing could go undetected.
- **Fix**: Add a targeted integration test in `command_surface_contract.rs` or a new `tests/creator_kb_world_scope.rs` that exercises the `--scope world --world-id wld_test` path against the CLI binary and asserts on stdout (e.g., "No key blocks in world wld_test." for an empty store).

### 🟢 Suggestion

**S1 — Missing justification comment for `#![allow(clippy::missing_errors_doc)]`**
- **Source**: `crates/nexus-daemon-runtime/src/api/handlers/narrative.rs:14`
- **Issue**: The module-level `allow` suppresses clippy's `missing_errors_doc` lint without a justification comment. Per root `AGENTS.md`: "Do not suppress with `#[allow(...)]` without a brief justification comment."
- **Fix**: Add a `//` comment on the line above explaining why this is acceptable (e.g., `// Early iteration — docs will stabilize before persistence is added.`).

**S2 — `world_id` parameter lacks format validation in `require_world_id()`**
- **Source**: `crates/nexus42/src/commands/creator/mod.rs:790-800`
- **Issue**: `require_world_id()` only checks presence, not format. The module already has `validate_world_id()` (requires `wld_` prefix + alphanumeric suffix) used for clone commands. Inconsistent validation means a malformed `world_id` will propagate to `InMemoryKbStore` before failing, producing a less clear error.
- **Fix**: Reuse `validate_world_id()` inside `require_world_id()` (or as a follow-up check) to fail fast with the same clear message used elsewhere.
