## Warning Findings

### W1: `should_degrade` has a logic error in failure window check (Medium)

**File**: `crates/nexus-domain/src/degradation.rs`, lines 334–346

```rust
fn should_degrade(&self) -> bool {
    if self.failure_count >= self.policy.failure_threshold {
        if let Some(last) = self.last_failure_time {
            let elapsed = chrono::Utc::now().signed_duration_since(last).num_seconds();
            if elapsed <= self.policy.failure_window_secs as i64 {
                return self.degradation_state.can_degrade_more(self.policy.max_degradation_depth);
            }
        }
    }
    false
}
```

**Problem**: `last_failure_time` is set to the **most recent** failure. The window check `elapsed <= failure_window_secs` only verifies that the **latest** failure happened within the window — it does **not** verify that all `failure_threshold` failures occurred within the window. The guard effectively treats `failure_count` as a **lifetime counter** once any recent failure occurs.

**Example**: With `failure_threshold=3` and `failure_window_secs=60`, if failures occur at t=0, t=120, t=180 — the check at t=180 sees `last_failure_time=t=180` (elapsed=0 <= 60) and `failure_count=3 >= 3`, so it triggers degradation. But the first failure at t=0 is **180 seconds** outside the window.

**Impact**: Premature degradation when failures are sparse. Contradicts the policy intent that failures must cluster within a time window.

**Recommendation**: Store a sliding window of failure timestamps (e.g., `Vec<DateTime<Utc>>`) and count only those within `failure_window_secs`. Or, reset `failure_count` when the window expires between failures.

### W2: `try_platform_assemble` uses hardcoded placeholder values (Medium)

**File**: `crates/nexus42/src/commands/context.rs`, lines 385–399

The function builds a request with `world_id: "wld_default"` and `workspace_id: "wrk_default"`. These placeholders will result in 404/422 from a real platform, treated as "platform unavailable" — triggering fallback to Stage0.

**Impact**: In V1.3 when a real platform is connected, every call will fail with 404 and trigger fallback, making `local_first`/`cloud_enhanced` modes behave identically to `local_only`.

**Recommendation**: Accept `world_id` and `workspace_id` from config or caller. The `assemble_local_with_routing` flow should pass actual values.

### W3: Degradation state restoration may trigger unintended side effects (Medium)

**File**: `crates/nexus42/src/commands/context.rs`, lines 212–227

```rust
pub fn create_degradation_guard(config: &CliConfig) -> DegradationGuard {
    let mode = config.runtime_mode();
    let mut guard = DegradationGuard::with_defaults(mode);
    if let Some(snap) = config.degradation_snapshot() {
        if snap.state != DegradationState::Normal {
            for _ in 0..snap.failure_count {
                guard.record_platform_result(false, Some("restored from snapshot".to_string()));
            }
        }
    }
    guard
}
```

**Problem**: Looping `record_platform_result(false, ...)` may trigger **additional degradation** during restoration. If `failure_count=2` and `failure_threshold=2`, the loop will trigger degradation on restore — potentially pushing the state further than it was when snapshotted.

**Recommendation**: Add a `DegradationGuard::restore_from_snapshot(&mut self, snapshot: &DegradationSnapshot)` method that sets state fields atomically without triggering side effects.

### W4: `call_assemble` URL path mismatches handler route (Low)

**File**: `crates/nexus42/src/api/daemon_client.rs`, line 186
**Handler**: `crates/nexus42d/src/api/handlers/context.rs`, line 12

Client calls `POST /api/v1/context/assemble` but daemon handler is at `POST /v1/local/context/assemble`. This path mismatch causes 404. The method is `#[allow(dead_code)]` and not wired into the active CLI path (CLI uses `ContextClient`), so this is latent.

**Recommendation**: Align the URL path or remove the dead method.

## Suggestions

### S1: `HealthCheckResult` vs `HealthCheckSnapshot` — consider `From` impl

Two nearly identical types exist: `HealthCheckResult` (chrono) and `HealthCheckSnapshot` (String). Consider implementing `From<&HealthCheckResult> for HealthCheckSnapshot` to reduce manual mapping in `DegradationSnapshot::from_guard`.

### S2: `assemble_stage0_fallback` duplicates `Stage0Assembly::assemble`

`TwoStageAssembly::assemble_stage0_fallback()` repeats the same ordering as `Stage0Assembly::assemble()`. Consider constructing a temporary `Stage0Assembly` and delegating to its `assemble()` method to eliminate duplication.

### S3: `degradation_depth_to` returns 0 for upgrade directions

Line 87: `_ => 0, // Upgrades or invalid combinations`. Consider returning a distinct sentinel value or documenting this more clearly.

### S4: Emoji in CLI stderr output

Line 255 in `context.rs`: `eprintln!("⚠ Degradation: ...")`. May not render in all terminals. Consider `[WARN]` prefix or configurable output.
