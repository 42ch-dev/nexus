## Findings

### 🔴 Critical

#### C-1: CLI depends on daemon-runtime internals (`nexus42` → `nexus-daemon-runtime::db`)
**Location**: `crates/nexus42/src/commands/creator/mod.rs:814-819`

The CLI `open_world_kb_store()` function directly constructs `nexus_daemon_runtime::db::kb_store::SqliteKbStore`. This violates crate layering: `nexus42` (CLI executable) should not depend on `nexus-daemon-runtime` (library crate) internals. The daemon-runtime AGENTS.md explicitly states "Library only — no `[[bin]]` section; this crate is consumed by `nexus42` CLI", but consumption should be at the API/service level, not by reaching into internal `db` modules.

**Impact**: Tight coupling prevents independent evolution of CLI and daemon DB strategies. Any refactoring of `SqliteKbStore` location or signature breaks the CLI.

**Fix**: Move `SqliteKbStore` and `SqliteNarrativeGateway` to `nexus-local-db` (which both CLI and daemon already depend on), or create a shared `nexus-local-db` gateway module that both crates can use. The `nexus-local-db` crate is the canonical owner of SQLite concerns per its AGENTS.md.

---

#### C-2: Duplicate seed helper logic (`world()` function)
**Location**: 
- `crates/nexus-daemon-runtime/src/db/narrative_gateway.rs:32-64`
- `crates/nexus-daemon-runtime/src/db/kb_store.rs:32-64`

The `seed::world()` function is copy-pasted identically between `narrative_gateway` and `kb_store` test modules. Both insert a creator row and a narrative_worlds row with identical SQL.

**Impact**: DRY violation. Changes to the seed pattern (e.g., adding a new required column) require editing two places. Divergence risk is high as the codebase grows.

**Fix**: Extract a shared `nexus_local_db::test_helpers` module (behind `#[cfg(test)]`) or have one module re-export from the other. Since both gateways live in the same crate (`nexus-daemon-runtime`), at minimum deduplicate within `crate::db`.

---

#### C-3: `clippy::future_not_send` suppression without justification
**Location**: 
- `crates/nexus-daemon-runtime/src/db/narrative_gateway.rs:193`
- `crates/nexus-daemon-runtime/src/db/kb_store.rs:176`

Both `impl` blocks for trait implementations carry `#[allow(clippy::future_not_send)]` without a `// SAFETY:` or justification comment. The `nexus-daemon-runtime/AGENTS.md` explicitly states: "Do not suppress with `#[allow(...)]` without a brief justification comment."

**Impact**: Lint debt accumulates silently. The suppression may mask a real `Send` requirement issue for multi-threaded runtime usage.

**Fix**: Add a comment explaining why `future_not_send` is acceptable (e.g., "// All callers are within the tokio runtime on the main thread; `SqlitePool` is `Send` but the returned futures are not `Send` due to sqlx internal borrows, which is safe for our single-threaded SQLite usage").

### 🟡 Warning

#### W-1: Semantically incorrect error mapping (`db_err` → `ValidationError`)
**Location**: `crates/nexus-daemon-runtime/src/db/narrative_gateway.rs:189-191`

The `db_err` helper maps all `sqlx::Error` variants to `NarrativeError::ValidationError`. A database connection failure, lock timeout, or corruption is not a "validation" error. This misleads callers and complicates error handling.

**Impact**: Consumers cannot distinguish between "world not found" (legitimate 404) and "database is locked" (retryable 500). The handler in `narrative.rs` uses string-matching (`msg.contains("not found")`) as a workaround.

**Fix**: Introduce `NarrativeError::Storage` or `NarrativeError::Database` variant, or map sqlx errors to a generic internal error variant. At minimum, distinguish `RowNotFound` from other sqlx errors.

---

#### W-2: In-memory query pagination for `KbQuery`
**Location**: `crates/nexus-daemon-runtime/src/db/kb_store.rs:289-342`

The `query()` method fetches **all** active blocks for a world into memory, then applies filters, offset, and limit in Rust. The comment acknowledges this ("efficient for per-world datasets (typically small)"), but there's no upper bound or safeguard.

**Impact**: If a world accumulates thousands of key blocks, this will cause unbounded memory growth and latency spikes. The `list_by_world` query lacks a `LIMIT` clause.

**Fix**: Add a database-level `LIMIT` to `list_by_world` (e.g., `LIMIT 1000`) as a safety cap, or implement proper indexed pagination in SQL for the `query` method.

---

#### W-3: `block_type` serialization uses `Debug` formatting
**Location**: `crates/nexus-daemon-runtime/src/db/kb_store.rs:191`, `448`, `414`

`format!("{:?}", kb.block_type)` is used to serialize `BlockType` to the database. This couples storage format to Rust's `Debug` derive implementation. If `BlockType` gains a new variant or the derive changes, the stored values may drift.

**Impact**: Silent data corruption risk if `Debug` output changes (e.g., renaming a variant). No compile-time check that `Debug` output matches the parser in `parse_block_type()`.

**Fix**: Implement `Display` for `BlockType` or use an explicit `as_str()` method, and store/parse via a stable string representation.

---

#### W-4: `delete_key_block` uses soft-delete but `insert` may conflict on `key_block_id`
**Location**: `crates/nexus-daemon-runtime/src/db/kb_store.rs:477-495`

`delete_key_block` sets `status = 'deleted'`, but `insert_key_block` uses the primary key `key_block_id` (which is generated by `KeyBlock::new()` using UUID). A re-insert with the same `canonical_name` + `block_type` succeeds (the partial unique index excludes deleted), but a re-insert with the same `key_block_id` would fail on PK.

**Impact**: The `test_deleted_allows_reinsertion` test only verifies name+type reinsertion, not same-ID reinsertion. If a client attempts to restore a deleted block by re-inserting with the same ID, it will fail with an opaque sqlx error.

**Fix**: Document the deletion semantics (is "deleted" permanent or recoverable?). If recoverable, implement `restore_key_block`. If permanent, consider renaming `delete` to `soft_delete` and document that the old ID is burned.

### 🟢 Suggestion

#### S-1: Remove stale comment in `narrative_routes()` docstring
**Location**: `crates/nexus-daemon-runtime/src/api/mod.rs:192-196`

The docstring says "in-memory, read-only (V1.25 Theme C, C1.1)" and "No persistence across daemon restarts." This is now false — the routes use `SqliteNarrativeGateway` which is SQLite-backed and persistent.

**Fix**: Update the docstring to reflect V1.26 persistence.

---

#### S-2: `narrative.rs` handler string-matches error messages
**Location**: `crates/nexus-daemon-runtime/src/api/handlers/narrative.rs:69-77`

The handler matches `NarrativeError::ValidationError(msg) if msg.contains("not found")` to return 404. This is fragile — changing the error message text in `narrative_gateway.rs` breaks the API contract.

**Fix**: Add a dedicated `NarrativeError::NotFound` variant, or check for `NarrativeError::ValidationError` with a structured context rather than string matching.

---

#### S-3: `WorkspaceState::narrative_gateway()` returns concrete type, not trait object
**Location**: `crates/nexus-daemon-runtime/src/workspace/mod.rs:191-194`

The method returns `Arc<SqliteNarrativeGateway>` rather than `Arc<dyn NarrativeGateway>`. While this avoids a vtable indirection, it prevents swapping the implementation (e.g., for tests or future caching layers) without changing `WorkspaceState`.

**Fix**: Consider returning `Arc<dyn NarrativeGateway>` for flexibility, or document why the concrete type is required.

---

#### S-4: Migration comment claims schema v7 but file is dated 20260524
**Location**: `crates/nexus-local-db/migrations/20260524_narrative_worlds.sql:1`

The comment says "V1.26 narrative persistence" but the plan says this is schema v7. The `version.rs` says `DB_SCHEMA_VERSION = 8`. The relationship between migration dates, plan versions, and `DB_SCHEMA_VERSION` is not obvious.

**Fix**: Add a comment in `version.rs` linking each schema version number to its migration files, or adopt a convention where the migration file header states the schema version it targets.

**Verdict**: Request Changes

**Rationale**: Three Critical findings block approval:

1. **C-1 (CLI → daemon-runtime internals)**: This is the most significant architectural concern. The CLI depending on daemon-runtime's internal `db` modules breaks the library boundary. The fix (moving gateways to `nexus-local-db`) is structural and should be done before merge.

3. **C-3 (clippy suppression without justification)**: Violates the project's own AGENTS.md rules. Two-line fix.

The Warning findings (W-1 through W-4) are architectural/semantic issues that don't block merge but should be addressed or explicitly deferred with residual tracking. W-1 (error mapping) and W-3 (Debug serialization) are particularly concerning for long-term maintainability.
