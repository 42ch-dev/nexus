### 🟡 Warning

#### W01 — Missing world_id format validation for `--scope world` KB CLI commands (correctness / UX gap)

- **Severity**: Warning (correctness, input hygiene)
- **Files**:
  - `crates/nexus42/src/commands/creator/mod.rs:798-808` (`require_world_id` only checks `Some`)
  - `crates/nexus42/src/commands/creator/mod.rs:923-934,1008-1019,1103-1110` (kb_list / kb_search / kb_show for World scope)
  - `crates/nexus-daemon-runtime/src/db/kb_store.rs:261-287` (list_by_world etc. — no format gate)
  - `crates/nexus-daemon-runtime/src/db/narrative_gateway.rs:195-218` (get_world_state etc. — no prefix gate)
  - `crates/nexus-daemon-runtime/src/api/handlers/narrative.rs:61-78` (get_world Path<String> → gateway, no validate)
- **Description**: `validate_world_id` (enforces `^wld_[A-Za-z0-9]+$`) exists and is used for `clone` command (line 402), but **not invoked** by `require_world_id` or the three `kb_*` world-scope functions. `require_world_id` returns any non-None string. Malformed IDs (e.g. `wld_foo;`, `../../../etc`, empty suffix, or non-alnum) reach `list_by_world(&wid)` / `get_world_state` / HTTP `GET /.../{world_id}` and produce silent empty result or 404 instead of clear "invalid world ID format" error (400). Inconsistent with clone path and domain convention (table CHECK `LIKE 'wld_%'`, id prefixes in seeds/tests).
- **Risk**: Low security (parameterized queries prevent injection even with weird IDs; no fs path join from world_id in V1.26). Medium correctness / UX (silent failure, surprises users, makes future path-based features riskier if world_id ever influences layout).
- **Evidence**:
  ```
  // CLI
  if scope == &KbScope::World {
      let wid = require_world_id(world_id)?;  // no validate!
      let store = open_world_kb_store(...)?
      store.list_by_world(&wid)...
  }
  // Handler
  Path(world_id): Path<String>  // arbitrary
  gateway.get_world_state(&world_id)...
  ```
- **Fix**: Update `require_world_id` (or add `validate_world_id_for_kb`) to call `validate_world_id` (or share the logic). Add equivalent lightweight validation in `SqliteNarrativeGateway` entrypoints and/or narrative handler (return 400 with "WorldId must start with 'wld_' + alnum" before DB query). Mirror pattern used in clone.
- **Status**: Unresolved — recommend fix before or in follow-up (not blocking Approve for V1.26 since no data loss / injection risk).

#### W02 — No early validation of world_id / key_block_id prefixes in Sqlite adapters (defense-in-depth)

- **Severity**: Warning (correctness / robustness)
- **Files**: `narrative_gateway.rs:196,225,282`, `kb_store.rs:178,234,261` (all query methods take `&str` with zero prefix/length checks before `?` bind)
- **Description**: DB-level CHECK constraints (`world_id LIKE 'wld_%'`, `key_block_id LIKE 'kb_%'`, status enums) protect writes, but read/query paths and KB inserts accept any string. FK violation on bad world_id for KB insert surfaces as opaque "database error" or Duplicate/Storage instead of "world does not exist or invalid format".
- **Risk**: Low (current callers from CLI/daemon already pass well-formed IDs from seeds or validated paths; no untrusted direct writes in V1.26 non-goals). Future direct use of the stores (e.g. via ACP or new endpoints) could benefit from early rejection.
- **Recommendation**: Add `validate_world_id` / `validate_key_block_id` helpers (reuse or copy from CLI) called at the start of gateway/store methods; return domain-specific ValidationError / KbStoreError::InvalidId. Keeps DB as last line of defense.
