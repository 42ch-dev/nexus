## 3. Critical Findings (Block Merge)

### C-1: IPC `worker/acp_prompt` echoes prompt as response — no actual ACP dispatch

**File**: `crates/nexus42/src/commands/acp_worker.rs`, `handle_acp_prompt()` (lines ~200-230)
**Severity**: **Critical**
**Category**: IPC correctness / functional gap

The `handle_acp_prompt` implementation sends the **prompt text itself** as the "full_text" response:

```rust
// Stream a chunk notification (simulated — full impl streams real chunks).
let chunk = json!({
    "jsonrpc": "2.0",
    "method": "worker/acp_prompt_chunk",
    "params": {
        "request_id": request_id,
        "text": prompt      // ← echoes the input prompt
    }
});

// Send final result.
let result = json!({
    "done": true,
    "full_text": prompt     // ← echoes the input prompt, NOT agent response
});
```

**Impact**: Any orchestration flow that depends on `worker/acp_prompt` to produce **agent-generated content** will receive the original prompt text instead. The novel-writing pipeline (gathering → brainstorming → outlining → drafting) will output user prompts as "novel content." This is a functional regression for the primary use case.

**Expected**: Per `acp-client-tech-spec-v2.md` §2.3, the worker should dispatch the prompt via `nexus-acp-host` to the ACP agent subprocess and stream the agent's response back. At minimum, this should return a clear stub marker (like `[acp_prompt stub: ...]`) rather than echoing the input.

**Fix**: Either (a) integrate `nexus-acp-host` dispatch in the worker, or (b) return a clearly-marked stub response so downstream code can detect the incomplete path.

### C-2: Echo-worker test fixture has shell injection via `printf`

**File**: `crates/nexus-orchestration/tests/fixtures/echo-worker.sh`, line 18
**Severity**: **Critical** (test fixture, but reveals design concern for production worker IPC)
**Category**: IPC security / test correctness

```bash
printf '{"jsonrpc":"2.0","id":1,"result":{"done":true,"full_text":"%s"}}\n' "$prompt"
```

The `$prompt` variable (extracted from JSON-RPC params via Python) is interpolated into `printf`'s **format string position** (`%s`), but the **actual prompt value** is passed as the argument to `%s`. This is technically correct for printf. However, the `python3` extraction on line 15 uses:

```bash
prompt=$(echo "$req" | python3 -c "import json,sys; d=json.loads(sys.stdin.read()); print(d.get('params',{}).get('prompt',''), end='')" 2>/dev/null)
```

If `print()` outputs bytes that include a `%` character (e.g., the prompt contains `%s`), the subsequent `printf` on line 18 will interpret it as a format specifier. While `printf '%s' "$prompt"` is safe (the `%s` is in the format, `$prompt` is the arg), the actual code passes `$prompt` as the argument, so this is **safe** in this specific case.

**However**, the bigger concern is that **the JSON-RPC request ID is always hardcoded to `1`** in the echo worker:

```bash
printf '{"jsonrpc":"2.0","id":1,"result":...}'
```

This means the worker never echoes back the actual request ID from the client. If the daemon's IPC client correlates responses by request ID (standard JSON-RPC 2.0), **concurrent `worker/acp_prompt` calls would have unmatchable responses**.

**Impact**: The `acp_prompt_via_worker.rs` test passes because it's a single-threaded single-call test. In production, concurrent prompt dispatches would lose response correlation.

**Fix**: Echo worker should parse and echo the `id` field from the incoming request.

## 4. Warning Findings

### W-1: `AcpPromptTask.render_template` uses naive `{{key}}` substitution — vulnerable to ReDoS and silent data loss

**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, `AcpPromptTask::render_template()` (lines ~530-550)

```rust
while let Some(start) = rendered.find("{{") {
    if let Some(end) = rendered[start..].find("}}") {
        let key = rendered[start + 2..start + end].trim().to_string();
        let value: Option<String> = context.get(&key).await;
        let replacement = value.unwrap_or_default();
        rendered.replace_range(start..start + end + 2, &replacement);
    } else {
        break;
    }
}
```

**Issues**:
1. **ReDoS risk**: Input like `{{{{{{{{...` causes repeated `find("{{")` scans over shrinking strings — O(n²) worst case.
2. **Silent data loss**: Unknown keys are replaced with empty string. A prompt template `Hello {{user_name}}!` becomes `Hello !` with no error — the user gets no indication the variable wasn't resolved.
3. **Nested template attack**: `{{{{evil}}}}` → first pass resolves `{{evil}}` to empty → result is `{{}}` → second pass sees `{{}}` as empty key → no crash but confusing behavior.

**Recommendation**: Use the `handlebars` crate (already in `Cargo.toml` for the project) or at least return an error for unresolvable keys rather than silently dropping them.

### W-2: `requires_capabilities` in preset manifest is NOT validated during load

**File**: `crates/nexus-orchestration/src/preset/loader.rs`, `load_preset_from_str()`

The `PresetManifest` has a `requires_capabilities` field, and the comment in `preset.rs` says "loader rejects if any are missing." However, `load_preset_from_str()` does **not** validate `manifest.preset.requires_capabilities` against the `CapabilityRegistry`.

**Impact**: A preset declaring `requires_capabilities: ["nonexistent.cap"]` loads successfully, and the failure only occurs at runtime when the capability is invoked.

**Fix**: Add validation in `validate_manifest()`:
```rust
for cap_name in &manifest.preset.requires_capabilities {
    if caps.get(cap_name).is_none() {
        problems.push(ValidationProblem {
            path: "preset.requires_capabilities".into(),
            error: format!("unknown required capability: '{}'", cap_name),
        });
    }
}
```

### W-3: `StateCompositeTask` resume flag collides across sessions

**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, `StateCompositeTask::run()`

The resume flag uses `format!("_state_{}_resumed", self.id)` as a key in the shared `graph_flow::Context`. If two sessions run the same preset concurrently, they share the same context namespace (per session). However, if the context is somehow shared or serialized/deserialized across sessions, the flag could leak.

**Risk level**: Low in current architecture (context is per-session), but the naming convention doesn't include the session ID for future-proofing.

**Recommendation**: Include session ID in the resume key, e.g., `format!("_state_{}@{}_resumed", session_id, self.id)`.

### W-4: `EngineProxy` duplicates `run_step` logic from `GraphFlowEngine`

**File**: `crates/nexus-orchestration/src/engine.rs`, `EngineProxy` impl (lines ~197-260)

`EngineProxy` re-implements the same `run_step` logic as `GraphFlowEngine`. The two implementations must be kept in sync manually. Any bug fix in one must be mirrored in the other.

**Impact**: Maintenance burden; risk of divergence. Currently they are byte-for-byte similar but not shared.

**Recommendation**: Extract `run_step` into a shared helper function that both `GraphFlowEngine` and `EngineProxy` call.

### W-5: `ToolPolicy::from_str` silently defaults unknown values to `AutoGrantReadOnly`

**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, `ToolPolicy::from_str()`

```rust
_ => Ok(ToolPolicy::AutoGrantReadOnly), // safe default
```

While `AutoGrantReadOnly` is the "safest" default, silently accepting unknown policy strings means YAML typos (e.g., `auto_grant_readonl` missing the 'y') go undetected.

**Recommendation**: Return `Err(...)` for unknown values. Let validation catch typos.

### W-6: No preset ID format validation — path traversal via `load_embedded_preset`

**File**: `crates/nexus-orchestration/src/preset/mod.rs`, `load_embedded_preset()`

The `id` parameter is used directly in `format!("{id}/preset.yaml")` to look up files in the `include_dir!` tree. While `include_dir!` prevents traversal **outside** the embedded directory, an ID like `../../etc/passwd` would still produce a valid lookup path within the embedded tree (which simply returns "not found"). The `PresetHeader` doc comment says IDs must match `/^[a-z][a-z0-9._-]*$/`, but this regex is **not enforced** anywhere.

**Impact**: Low — `include_dir!` is a compile-time snapshot. But the lack of validation means invalid IDs propagate deeper into the stack before failing.

**Recommendation**: Validate `id` against the regex at the entry points (`load_embedded_preset`, `reload_preset` HTTP handler).

## 5. Suggestions (Nit / Future Improvements)

### S-1: `judge_llm.parse_judge_response` substring matching is fragile

**File**: `crates/nexus-orchestration/src/capability/builtins/judge_llm.rs`

`GO_WORDS` contains `"go"` and `NOGO_WORDS` contains `"no"`. The substring search `lower.contains("go")` would match `"going"`, `"logo"`, etc. Similarly `"no"` matches `"knowledge"`.

**Suggestion**: Split on word boundaries and match whole words, or use structured output format (JSON) from the judge agent.

### S-2: `InnerGraphTask` hard-coded 256 iteration limit

**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, `InnerGraphTask::run()`

```rust
for _ in 0..256 {
    let outcome = self.engine.run_step(&child_sid).await...
}
```

No error or warning is emitted if the loop exhausts 256 iterations without reaching completion. This is a silent failure mode.

**Suggestion**: Add a log warning and return an error outcome after max iterations.

### S-3: `handle_initialize` doesn't validate params

**File**: `crates/nexus42/src/commands/acp_worker.rs`, `handle_initialize()`

The `_params` argument is unused. The daemon may send initialization parameters (e.g., `agent_ref` override, `tool_policy` defaults) that the worker ignores.

**Suggestion**: Parse and validate params; at minimum log unexpected params.

### S-4: `InnerGraphNodeTask` is a complete stub

**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, `InnerGraphNodeTask::run()`

```rust
let output = format!("inner_node:{}:stub_output", self.id);
```

This returns a hard-coded string regardless of the node's `template_file` or `tool_policy`. The E2E tests check for non-empty output but not for meaningful content.

**Suggestion**: At minimum, read and return the template file content. T4 should replace this with `AcpPromptTask` dispatch.
