### 🟡 Warning

**W-1: Unbounded file read in `daemon_logs`**
- **Location**: `crates/nexus42/src/commands/daemon.rs:529`
- **Issue**: `std::fs::read_to_string(&log_path)?` reads the **entire** log file into memory before extracting the last `N` lines. The `lines` parameter only bounds the output, not the input. For a long-running daemon with verbose logging, the log file could grow to hundreds of MB or GB, causing excessive memory usage and potential OOM.
- **Fix**: Use a bounded ring-buffer approach (e.g., `rev` + `tail`-like line scanning from the end of the file, or a streaming reader with a fixed-size circular buffer). At minimum, cap the read to a reasonable max (e.g., read only the last 1 MB of the file and extract lines from that).
- **Risk**: Memory pressure / OOM on systems with large log files; CLI unresponsiveness.

**W-2: `restart_daemon` may race with port release after SIGKILL**
- **Location**: `crates/nexus42/src/commands/daemon.rs:500-505`
- **Issue**: `restart_daemon` calls `stop_daemon().await?` then immediately `start_daemon().await`. In `stop_daemon`, after sending `SIGKILL`, there is only a `tokio::time::sleep(Duration::from_millis(200)).await` before returning. On a loaded system, 200ms may not be sufficient for the kernel to fully tear down the process and release the listening port. `start_daemon` then attempts to bind the same port; if the old process still holds it, the new daemon will fail to start.
- **Fix**: Increase the post-SIGKILL wait to at least 500ms-1s, or better: poll the port (or PID) until it is confirmed free before returning from `stop_daemon`.
- **Risk**: Restart command fails intermittently; poor UX for daemon management.

**W-3: Potential zombie/defunct process in `probe_agent` timeout path**
- **Location**: `crates/nexus42/src/commands/acp.rs:420-427`
- **Issue**: When the 3-second timeout fires (`Err(_)` branch), the code calls `let _ = child.kill().await;` and discards the result. If `kill()` fails (e.g., permission denied, process already in uninterruptible sleep), the child is never reaped. Although `AgentSpawner::spawn` sets `.kill_on_drop(true)`, the `child` handle is still alive in this scope and is not dropped until after the `match`. More critically, there is no `child.wait().await` after `kill()` to reap the process. The `_stdin` and `_stdout` handles are also ignored (underscore-prefixed) but they *are* dropped at end of scope, which closes the pipes.
- **Fix**: After `child.kill().await`, always call `let _ = child.wait().await;` to ensure the process is reaped. Handle the `kill()` result explicitly.
- **Risk**: Zombie processes accumulating on systems where `acp probe --agent` is used frequently.

### 🟢 Suggestion

**S-1: `daemon_logs` requires daemon to be running**
- **Location**: `crates/nexus42/src/commands/daemon.rs:510-519`
- **Issue**: The function checks `client.health_check().await?` and returns early if the daemon is not running. This is counter-productive for debugging: users most often want to read logs *because* the daemon crashed or failed to start. The log file should be readable regardless of daemon state.
- **Fix**: Remove the health-check gate, or demote it to a warning printed above the logs. Always attempt to read the log file if it exists.

**S-2: Unnecessary allocation in `probe_agent` arg passing**
- **Location**: `crates/nexus42/src/commands/acp.rs:384-391`
- **Issue**: `args.iter().map(std::string::String::as_str).collect::<Vec<_>>()` allocates a new `Vec<&str>` just to pass to `spawn()`. Since `resolve_launch_command` already returns `Vec<String>`, and `spawn` takes `&[&str]`, this is an avoidable intermediate allocation.
- **Fix**: Change `resolve_launch_command` to return a small struct or change `spawn` signature, or use `args.iter().map(|s| s.as_str())` without collecting if the signature can be relaxed.

**S-3: Old `agent` module still compiled with dead-code allowed**
- **Location**: `crates/nexus42/src/commands/agent.rs`
- **Issue**: The entire old `agent` module is retained with `#![allow(dead_code)]` and is still a public module. All its code is compiled and linked into the release binary, increasing binary size. The comment says it is for "internal reference and test compatibility," but the integration tests have already been migrated to `acp` commands.
- **Fix**: Either remove the module entirely (if tests no longer need it), or gate it behind a `#[cfg(test)]` attribute so it is not compiled into the release binary.

**S-4: `cmd_probe` ignores the `registry` flag**
- **Location**: `crates/nexus42/src/commands/acp.rs:325-329`
- **Issue**: The `_registry_flag` parameter is prefixed with underscore and never used. The `Probe` command accepts `--registry`, but the behavior is always "probe registry if no `--agent` given." This is harmless for now but is a silent behavioral mismatch if future users expect `--registry` to have an effect.
- **Fix**: Either use the flag (e.g., require `--registry` explicitly to probe the registry), or remove it from the CLI definition to avoid confusion.

**S-5: `print_list_json` builds intermediate `serde_json::Value` objects**
- **Location**: `crates/nexus42/src/commands/acp.rs:537-550`
- **Issue**: The JSON output is constructed by mapping each agent to a `serde_json::Value` and collecting into a `Vec`, then wrapping in another `Value`. This is idiomatic but allocates more than necessary for a simple listing. For the expected registry size (tens of agents), this is negligible; for a very large registry it could matter.
- **Fix**: Define a small output struct and serialize directly with `serde_json::to_string_pretty` instead of building `json!` macros.
