### 🟢 Suggestion 1: device_id.rs - Thread sleep duration in concurrent retry loop

**File:** `crates/nexus-sync/src/device_id.rs:60`

**Issue:**
The `get_or_create_device_id` function uses a fixed 5ms sleep when encountering an empty file during concurrent creation. While this is acceptable for low-contention scenarios, it could become a bottleneck if:
- Many threads/processes start simultaneously
- The filesystem has high latency
- The device ID is accessed on every startup path

**Current code:**
```rust
if trimmed.is_empty() {
    // Empty file — another thread may be writing.
    // Wait briefly and retry rather than removing,
    // to avoid NotFound races for concurrent readers.
    std::thread::sleep(std::time::Duration::from_millis(5));
    continue;
}
```

**Impact:** Low. This is a rare edge case for a one-time initialization operation. The empty file scenario only happens during the race window between `create_new` succeeding and the write completing.

**Recommendation:**
Consider adding a maximum retry count (e.g., 10-20 attempts) to prevent an infinite loop in case of filesystem corruption that creates a permanently empty device ID file.

**Verdict:** Acceptable for now; recommended to add max retry count for robustness.

### 🟢 Suggestion 2: user_auth.rs - Chrono parsing overhead on every status call

**File:** `crates/nexus42/src/auth/user_auth.rs:196`

**Issue:**
The `status` command parses the `refresh_expires_at` timestamp on every call using `chrono::DateTime::parse_from_rfc3339`. While this is a lightweight operation for a CLI command called infrequently, the parsing happens in a loop over all tokens. Parsing RFC 3339 strings involves:
- Regex-like pattern matching
- String splitting and allocation
- TZ offset calculation

**Current code:**
```rust
match chrono::DateTime::parse_from_rfc3339(re) {
    Ok(expiry) => {
        if expiry > chrono::Utc::now() {
            println!("  Refresh expires: {} (valid)", re);
        } else {
            println!("  Refresh expires: {} (expired)", re);
        }
    }
    Err(_) => {
        println!("  Refresh expires: {} (unparseable)", re);
    }
}
```

**Impact:** Very low. This is only for human-readable output in the status command, not on any hot path.

**Recommendation:**
No change needed. This is acceptable for a CLI status command. The code is clear and correct.

### 🟢 Suggestion 3: user_auth.rs - ensure_valid_token performs filesystem I/O on every call

**File:** `crates/nexus42/src/auth/user_auth.rs:340-382`

**Issue:**
The `ensure_valid_token` function loads the full `AuthStore` from disk on every call:
1. Reads `~/.nexus42/auth.json` from disk
2. Parses the JSON
3. Checks token expiry
4. If refresh needed, does HTTP and then reloads AuthStore again

Since `creator_auth.rs:67` calls `ensure_valid_token` for every authenticated operation, this means:
- Every CLI command that needs authentication does at least one disk read
- If token refresh is triggered, there are additional disk writes and reads

**Hot path consideration:**
For long-running CLI operations or scripts that invoke the CLI frequently, this could add up. However, typical CLI usage patterns are single-invocation, so the amortized cost is low.

**Impact:** Low to Medium. Acceptable for CLI usage patterns, but something to monitor for future daemon-based auth.

**Recommendation:**
Consider in-memory caching for the token state (with file mtime-based invalidation) if:
1. The CLI ever becomes a long-running process
2. Auth token checks happen more than once per invocation
3. Scripts invoke the CLI in tight loops

For now, this is acceptable given typical CLI usage patterns.

### 🟢 Suggestion 4: refresh_access_token - Multiple AuthStore reloads during refresh

**File:** `crates/nexus42/src/auth/user_auth.rs:230-332`

**Issue:**
The `refresh_access_token` function loads `AuthStore` multiple times:
1. Line 231: Load to get refresh token
2. Line 312: Load again to get user_id (could reuse from first load)
3. Line 328: Load a third time to store new token (actually, `store_user_token` internally loads again)

**Current flow:**
```
Load 1 (line 231) → Read refresh_token
  HTTP request
Load 2 (line 312) → Read user_id
Load 3 (implied in store_user_token) → Write new token
```

**Impact:** Low. File operations are cached by the OS, and refresh only happens hourly or less frequently.

**Recommendation:**
Minor optimization opportunity: Pass the already-loaded `AuthStore` through instead of reloading. This would save 1-2 file reads per refresh.

```rust
// Instead of reloading:
let user_id = store
    .user_token
    .as_ref()
    .map(|t| t.user_id.clone())
    .unwrap_or_else(|| "unknown".to_string());
```

This is a clean-up item, not a performance issue per se.

### 🟢 Suggestion 5: runtime.rs - daemon_status performs synchronous DB health check on every call

**File:** `crates/nexus42d/src/api/handlers/runtime.rs:75-81`

**Issue:**
The `daemon_status` endpoint performs a live DB health check on every request:
```rust
let db_status = match sqlx::query_scalar!("SELECT 1 as \"count!\"")
    .fetch_one(state.pool())
    .await
{
    Ok(_) => HealthStatus::Up,
    Err(_) => HealthStatus::Down,
};
```

While this provides accurate health status, it:
1. Uses a SQLite connection from the pool on every call
2. Does a round-trip to the database
3. Could contribute to pool exhaustion if this endpoint is polled very frequently

**Impact:** Low to Medium. SQLite connection overhead is minimal, and daemon status polling is typically infrequent (seconds or minutes between calls).

**Recommendation:**
Consider adding a short cache (e.g., 1-5 seconds) for the DB health status if this endpoint becomes a hot path due to aggressive monitoring. For now, this is acceptable.
