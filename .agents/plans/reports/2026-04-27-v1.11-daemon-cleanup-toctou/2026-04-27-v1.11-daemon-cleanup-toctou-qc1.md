### 🟢 Suggestion

#### F-001: `_` prefix convention on struct fields is non-idiomatic

**Source**: `crates/nexus42d/src/lifecycle/subsystems/http.rs`, `crates/nexus42d/src/api/handlers/orchestration/schedules.rs`

The `_` prefix is used to suppress `dead_code` warnings on struct fields:
- `HttpState::Running { _port: u16 }` (http.rs)
- `InspectRow { _concurrency_whitelist }` with `#[sqlx(rename = "concurrency_whitelist")]` (schedules.rs)
- `HistoryRow { _content }` with `#[sqlx(rename = "content")]` (schedules.rs)

While `_`-prefixed names do suppress dead_code warnings in Rust, this convention is idiomatic for **unused local variables and function parameters**, not struct fields. For struct fields that are selected by sqlx but not used in code, the `#[sqlx(rename = "...")]` approach is correct, but the `_` prefix on the Rust field name reads as a variable convention applied to a struct field.

**Impact**: Low — purely stylistic; does not affect correctness.

**Suggested fix**: Either (a) accept the `_` prefix as a practical workaround, or (b) keep the original field names and use `#[allow(dead_code)]` on the specific field (more explicit about intent), or (c) use the field somewhere (e.g., in debug/logging output).

#### F-002: TOCTOU fix lacks explicit `flush()` before returning

**Source**: `crates/nexus-sync/src/device_id.rs`, `get_or_create_device_id()`

The atomic `create_new(true)` + `write_all()` pattern correctly eliminates the TOCTOU race. However, `write_all()` writes to the OS buffer but does not guarantee an immediate flush to disk. If the process crashes between `write_all()` and the OS flush, the file could contain a partial UUID string.

The existing empty-file retry loop (5ms sleep + retry) would handle a truly empty file, but a partial UUID (e.g., "a1b2c3") would be returned as the device ID, producing an invalid UUID.

**Impact**: Low — crash-during-write is rare; the original code had the same risk.

**Suggested fix**: Add `file.sync_all()?` after `write_all()` for durability, or add a UUID validation check when reading back the file:

```rust
let trimmed = existing.trim();
if uuid::Uuid::parse_str(trimmed).is_err() {
    // Corrupted/partial file — remove and retry
    std::fs::remove_file(&path)?;
    continue;
}
```

#### F-003: Integration test fixtures extracted inline, not to shared module

**Source**: `crates/nexus42d/tests/integration.rs`

Plan T13 stated: "Extract hardcoded mock test data ... to **shared** test fixtures or builders." The extraction produced three well-named helpers (`seed_base_fixtures`, `seed_pending_review`, `seed_acp_session`) plus two constants (`FIXTURE_CREATOR_ID`, `FIXTURE_REF_SOURCE_ID`), all defined inline at the top of `integration.rs`.

While the extraction improves readability within `integration.rs`, the fixtures are not in a shared module (e.g., `crates/nexus42d/tests/common/` or `crates/nexus42d/src/test_utils.rs`), so they cannot be reused from other integration test files or benchmark suites.

**Impact**: Low — currently only `integration.rs` exists; shared fixtures would only matter when more test files are added.

**Suggested fix**: When a second integration test file is added, move the fixture helpers to `crates/nexus42d/tests/common/fixtures.rs` or a similar shared location.

#### F-004: `refresh_access_token` preserves user_id from stale store

**Source**: `crates/nexus42/src/auth/user_auth.rs`, `refresh_access_token()`

On successful token refresh, the function reads `user_id` from the existing `AuthStore` rather than extracting it from the new JWT response:

```rust
let user_id = store
    .user_token
    .as_ref()
    .map(|t| t.user_id.clone())
    .unwrap_or_else(|| "unknown".to_string());
```

If the platform ever changes the user_id between refreshes (unlikely but possible in multi-tenant migration scenarios), the stale `user_id` would be preserved instead of being updated from the new token.

**Impact**: Low — platform behavior change required to trigger.

**Suggested fix**: Extract `user_id` from the new JWT in the refresh response (reuse the existing `extract_user_id_from_jwt` helper on `token_response.access_token`).
