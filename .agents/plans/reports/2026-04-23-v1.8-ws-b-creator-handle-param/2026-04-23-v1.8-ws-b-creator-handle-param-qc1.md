## Findings

### 🟡 Warning

**W-001: Unreachable error branches in `validate_handle()`**

- **File**: `crates/nexus42/src/commands/creator.rs`, lines 156–176
- **Issue**: The `validate_handle()` function uses an if-else chain to determine a human-readable error reason after the regex fails. However, the regex `^[a-z0-9][a-z0-9._-]{2,13}[a-z0-9]$` **already enforces** all structural constraints: length 4–15, starts/ends with `[a-z0-9]`, interior only `[a-z0-9._-]`. This means:
  - `handle.len() < 4` branch: **UNREACHABLE** — regex requires ≥4 chars.
  - `handle.len() > 15` branch: **UNREACHABLE** — regex requires ≤15 chars.
  - `!handle.starts_with(...)` branch: **UNREACHABLE** — regex requires `[a-z0-9]` at start.
  - Only the `else` (invalid characters) branch and the final fallback can actually be reached.
- **Impact**: Dead code that will never execute. The specific error messages ("too short", "too long", "must start with", "must end with") are never shown to users — they always get the generic "contains invalid characters" message (or the fallback). This defeats the purpose of having detailed error feedback.
- **Fix**: Either (a) remove the regex and use only the if-else chain for validation (preserving detailed error messages), or (b) simplify the error branch to a single message like `"does not match the required format: 4–15 chars, lowercase letters, digits, dots, hyphens, underscores"`. Option (a) is preferred for better UX.

- **Fix**: Consider `regex = "1.10"` or pinning via `Cargo.lock` (which is already committed).
