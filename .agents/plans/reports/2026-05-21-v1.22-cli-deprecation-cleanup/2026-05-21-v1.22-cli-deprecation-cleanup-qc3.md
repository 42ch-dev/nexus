### 🟡 Warning

- **W-001: `creator/mod.rs` is 2715 lines — large single-file module**
  The `creator/mod.rs` absorbed init workspace utilities (lines 46–312), clone utilities (lines 314–464), and retains the full creator command suite (register/status/use/list/pair/unpair/credentials/workspace/kb/logout) plus their KB sub-command implementations (lines 725–1165). At 2715 lines, this is substantially larger than the other directory module `mod.rs` files (acp: 1362, daemon: 955, sync: 433, platform: 57, system: 295). The soul and memory sub-commands were correctly extracted to `creator/soul.rs` (228 lines) and `creator/memory.rs` (337 lines), but further extraction remains advisable — for example, KB operations (`kb_list`, `kb_search`, `kb_show`, `kb_add`, `kb_remove` + their helpers) span ~440 lines and could live in a `creator/kb.rs` sub-module.
  → **Mitigation**: This is a structural note, not a regression — the PR improved modularity by extracting soul and memory. Further splitting can be done in a follow-up PR. Not blocking merge.

- **W-002: "migrated from" doc comments are slightly misleading**
  In `creator/mod.rs` lines 637–653, doc comments state `"migrated from nexus42 init"` and `"migrated from nexus42 clone"` on `CreatorWorkspaceCommand::Init` and `CreatorWorkspaceCommand::Clone`. In `daemon/mod.rs` line 119, `"migrated from nexus42 schedule"`. These doc comments are accurate historical notes but suggest a deprecated path that no longer exists. Users reading `--help` may be confused.
  → **Fix**: Consider rephrasing to past tense or removing the migration note from user-facing doc comments. E.g. "Initialize a new workspace" instead of "Initialize a new workspace (migrated from `nexus42 init`)". Not blocking; purely cosmetic.

### 🟢 Suggestion

- **S-001: `print_next_steps()` references new-style paths — good**
  In `creator/mod.rs:301-311`, the next-steps output uses the current command paths (`nexus42 system preset list`, `nexus42 daemon schedule add`, etc.). This is consistent and correct. No change needed.

- **S-002: `open_global_db()` duplication between `creator/soul.rs` and `system/identity.rs`**
  The `creator/soul.rs:187-203` contains `open_global_db()` which is noted in the comment at line 185 as mirroring `identity.rs::open_global_db()`. This is a benign duplication (different error semantics — soul returns `None` gracefully, identity errors). Could be consolidated into a shared utility in a follow-up.

- **S-003: `read_tail_lines()` in `daemon/mod.rs` is a nice performance optimization**
  The tail-reading function at `daemon/mod.rs:596-655` uses chunked backward seek to avoid loading entire log files into memory. This is a good pattern and should be preserved.
