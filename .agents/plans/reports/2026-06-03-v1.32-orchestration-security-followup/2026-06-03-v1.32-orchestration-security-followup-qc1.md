## Findings

### ЁЯФ┤ Critical

- **C-001 тАФ Generic preset capability calls do not inject `_creator_id` / `_session_id` into capability input, so the new contract is not wired at the orchestration boundary.**
  - Evidence: `StateCompositeTask::run` copies `EnterAction::Capability { args }` into `_capability_input` unchanged (`crates/nexus-orchestration/src/tasks/mod.rs:536-540`), then `CapabilityTask::run` passes that JSON directly to `cap.run(input)` (`crates/nexus-orchestration/src/tasks/mod.rs:66-76`). The changed capabilities now only read `input["_creator_id"]` / `input["_session_id"]` and otherwise fall back to `"default"` (`judge_llm.rs:92-103`, `context_summarize.rs:113-124`).
  - Impact: The code fixes direct spoofing inside `judge.llm` and `context.summarize`, but it does not implement the promised task-execution injection layer for generic preset capability actions. Legitimate internal executions that rely on graph context identity can silently route worker IPC to `default` instead of the active creator/session, and existing preset args cannot be used as a safe replacement because raw `creator_id`/`session_id` are now intentionally ignored.
  - Required fix: Before invoking a capability from `CapabilityTask`/`StateCompositeTask`, merge trusted context values into the capability input (at minimum `_creator_id` and `_session_id` when present), without allowing raw user/preset `creator_id` / `session_id` to override them. Add a regression test that exercises the actual task path, not only direct `cap.run(json!(...))`.
