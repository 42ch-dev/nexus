### 🟡 Warning

#### F-001: `system completion` reports success but does not generate shell completions

**Issue**: The plan and command help describe `system completion` as shell completion generation, but `SystemCommand::Completion` only calls `print_completion_help()` and returns `Ok(())`. The output explicitly says completion is not available and suggests `eval "$(nexus42 --help | tail -n +2)"`, which is not a valid completion script for bash/zsh/fish/elvish/powershell.

**Security/correctness impact**: This is a correctness failure for a newly public command. Users or install scripts may treat exit code 0 as successful completion generation, then evaluate non-completion help text. The command accepts arbitrary shell names without validation and never rejects unsupported shells, so invalid invocations also look successful.

**Evidence**:
- `crates/nexus42/src/commands/system.rs:44-48` advertises `Completion { shell: String }` as “Generate shell completion script”.
- `crates/nexus42/src/commands/system.rs:121-124` returns `Ok(())` after printing placeholder help.
- `crates/nexus42/src/commands/system.rs:149-160` prints “not yet available” and the invalid `eval` suggestion.
- Smoke check: `cargo run -q -p nexus42 -- system completion bash` exits successfully but does not emit a bash completion script.

**Fix**: Either implement real completion generation (for example with `clap_complete`, validating the shell enum and writing the generated script to stdout) before exposing this as a passing command, or mark the subcommand as a stub that exits non-zero / clearly out-of-scope and update the plan/tests so acceptance criteria do not claim completion generation is delivered.

#### F-002: `platform auth` exposes the development token path while plan docs describe a different auth surface

**Issue**: `PlatformCommand::Auth` delegates the full existing `AuthCommand`, which exposes `token <ACCESS_TOKEN>` as a visible `platform auth` subcommand. The plan/test comments describe the platform auth surface as login/logout/status/profiles, not token injection, and the token value is accepted as a positional CLI argument.

**Security/correctness impact**: Positional access tokens are commonly captured in shell history and process listings. This was present in the legacy command, but the refactor makes it part of the new visible `platform` command surface while simultaneously hiding the legacy top-level `auth`. If `token` is intended to remain development-only, the new public surface should either hide it, gate it, or use a safer input channel.

**Evidence**:
- `crates/nexus42/src/commands/platform.rs:20-24` delegates `platform auth` to `super::auth::AuthCommand` without filtering.
- `crates/nexus42/src/commands/auth.rs:14-23` defines `Token { access_token: String, ... }` as a visible subcommand argument.
- `cargo run -q -p nexus42 -- platform auth --help` lists `token` and does not list `profiles`.

**Fix**: Decide and encode the intended V2 auth contract. If `token` remains supported, document it explicitly and consider safer input (`--access-token-stdin`, env var, or hidden/dev-only command). If the intended surface is login/logout/status/profiles, add/route `profiles` and hide or remove `token` from the visible `platform auth` help.

### 🟢 Suggestion

#### F-003: Strengthen command-surface tests to assert actual help sections rather than substring presence

**Issue**: Several contract tests use broad `help_text.contains(subcmd)` checks. This can pass when a word appears in descriptions or options rather than as an actual command entry.

**Impact**: The current tests passed even though `system completion` is only a placeholder; stricter parsing would make command-surface regressions easier to catch.

**Fix**: For command-surface tests, extract the `Commands:` section and assert indented command entries (for example `  completion`) or use clap’s command metadata directly if exposed to tests.
