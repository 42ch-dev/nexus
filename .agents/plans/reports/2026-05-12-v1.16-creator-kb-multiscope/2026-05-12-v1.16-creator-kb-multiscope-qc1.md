### 🟢 Suggestion

**S-001: KB module extraction for maintainability**

The KB logic (~200 lines: `KbIndex`, `KbIndexEntry`, `run_kb`, `kb_list`, `kb_search`, `kb_show`, `kb_add`, `generate_entry_id`, `resolve_kb_paths`) is embedded directly in `creator.rs` which is already ~1000+ lines. As KB evolves (Plan mentions "edit, delete, update" deferred to post-V1.16), this code will grow.

**Recommendation**: When implementing delete/edit/update, consider extracting KB into a separate module (e.g., `crates/nexus42/src/commands/kb.rs` or `crates/nexus42/src/kb/`). This follows the pattern used for `soul.rs` and `memory.rs` subcommands and keeps `creator.rs` focused on creator identity/workflow operations.

**Impact**: Low — current baseline is functional and maintainable. Extraction is a future optimization.

---

**S-002: Entry ID collision consideration**

Entry ID generation uses `kb_` + lower 32 bits of nanosecond timestamp (`(now.as_nanos() & 0xFFFF_FFFF) as u32`). This provides ~4.3 billion unique suffixes but collision is possible in rapid concurrent execution (within same nanosecond tick window).

**Recommendation**: For current local single-user CLI baseline, collision risk is negligible. When implementing `world` scope or batch operations, consider UUID v4-based IDs (`kb_<uuid8hex>`) for guaranteed uniqueness across distributed systems.

**Impact**: Low — not a concern for current scope.

---

**S-003: Content search deferred**

Plan §Design mentions "search entries by title/content" but implementation only searches title (case-insensitive substring match against `index.json`). Content search (grepping inside `.md` files) was not implemented.

**Recommendation**: Acceptable for baseline — title search via index is fast. For future enhancement, add optional `--full` flag to search entry content (would require loading each file, more expensive). Document current limitation in help text if needed.

**Impact**: Low — title search covers primary use case.

---

**S-004: KB-specific error variants**

KB errors use generic `CliError::Other(String)`. Examples:
- `"KB entry {entry_id} not found in workspace {slug}"`
- `"Source file not found: {}"`

**Recommendation**: When KB module is extracted, consider adding dedicated error variants (e.g., `CliError::KbEntryNotFound { entry_id, workspace_slug }`, `CliError::KbSourceNotFound { path }`) for better error categorization and programmatic handling.

**Impact**: Low — current errors are user-friendly.

---

**S-005: Path traversal defense-in-depth**

KB path helpers (`creator_kb_dir`, `creator_kb_entries_dir`) don't use `assert_creator_id_safe()` like `creator_soul_md_path()` does. The `creator_id` comes from validated `CliConfig` and `workspace_slug` is validated by `init::validate_slug()`.

**Recommendation**: Acceptable for current implementation — creator_id validation happens upstream. If KB operations are exposed to untrusted input in future, add defense-in-depth assertions.

**Impact**: Low — current path construction is safe.
