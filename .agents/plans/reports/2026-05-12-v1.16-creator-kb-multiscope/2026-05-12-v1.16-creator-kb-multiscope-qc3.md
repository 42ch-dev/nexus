### 🟡 Warning

#### W-1: Non-atomic index read-modify-write without file locking
- **Source**: `crates/nexus42/src/commands/creator.rs`, `kb_add()` (lines 454–502)
- **Issue**: `kb_add` performs a non-atomic sequence: (1) `std::fs::copy` entry file, (2) `read_kb_index`, (3) push to `Vec`, (4) `write_kb_index`. Multiple concurrent `nexus42 creator kb add` CLI processes can race on the index file, causing lost entries or a corrupted JSON index.
- **Impact**: Data loss under concurrent usage; corrupted `index.json` if write is interrupted.
- **Fix**: Implement atomic write for `index.json` (write to temp file + `std::fs::rename`), and add advisory file locking (e.g., `fs2::FileExt::lock_exclusive`) around the read-modify-write window. Alternatively, document that concurrent `kb add` is unsupported.

#### W-2: Orphan files on partial failure
- **Source**: `crates/nexus42/src/commands/creator.rs`, `kb_add()` (lines 480–498)
- **Issue**: If `std::fs::copy` succeeds but `write_kb_index` fails (disk full, permissions, serialization error), the copied `.md` file remains in `entries/` without an index entry. There is no rollback or cleanup.
- **Impact**: Orphan KB entry files that are invisible to `list`/`search`/`show` but consume disk space.
- **Fix**: On `write_kb_index` failure, delete the copied `dest` file before returning the error. Or use a temp file + atomic rename for the entry copy as well.

#### W-3: Silent data loss on index corruption
- **Source**: `crates/nexus42/src/commands/creator.rs`, `read_kb_index()` (lines 331–339)
- **Issue**: `read_kb_index` silently returns an empty default if the index file is missing **or** unreadable **or** corrupt JSON. A partially-written or corrupted `index.json` is discarded without warning, and a subsequent `kb_add` will overwrite it with a new index containing only the new entry — all previous entries are lost.
- **Impact**: Complete KB index loss on corruption.
- **Fix**: Distinguish "missing file" (return empty) from "unreadable/corrupt" (return an explicit error to the user so they can investigate/recover). Consider keeping a backup copy of `index.json` before overwrite.

#### W-4: ID collision risk in rapid successive calls
- **Source**: `crates/nexus42/src/commands/creator.rs`, `generate_entry_id()` (lines 352–359)
- **Issue**: Uses lower 32 bits of nanosecond timestamp. On fast systems or if the system clock has low resolution, two rapid `kb add` calls could generate the same `entry_id`. There is no check for existing IDs before using the generated one.
- **Impact**: Second `kb_add` would overwrite the first entry file (same `dest` path) and append a duplicate ID to the index.
- **Fix**: Check `entries_dir` for an existing file with the generated ID; if it exists, regenerate. Or use a counter/random component in addition to timestamp.

### 🟢 Suggestion

#### S-1: `kb_show` loads entire file unbounded
- **Source**: `crates/nexus42/src/commands/creator.rs`, `kb_show()` (line 448)
- **Issue**: `std::fs::read_to_string(&entry_path)?` loads the entire entry file into memory without size limits. A very large `.md` file could cause excessive memory use.
- **Fix**: Add a reasonable size limit (e.g., 10 MiB) and return an error if exceeded, or stream the file to stdout.

#### S-2: `kb_list` lacks pagination / output limits
- **Source**: `crates/nexus42/src/commands/creator.rs`, `kb_list()` (lines 382–389)
- **Issue**: All entries are printed unconditionally. A large KB could produce thousands of lines of output.
- **Fix**: Add a default limit (e.g., 50 entries) with a message like "Showing N of M entries" when truncated.

#### S-3: `kb_search` only searches titles, not content
- **Source**: `crates/nexus42/src/commands/creator.rs`, `kb_search()` (lines 394–430)
- **Issue**: The help text says "Search knowledge base entries by title/content", but the implementation only filters on `title.to_lowercase().contains(&query_lower)`. Content is not searched.
- **Fix**: Either update help text to "by title" only, or implement content search (requires reading each `.md` file — trade-off against performance).

#### S-4: Missing unit tests for KB I/O operations
- **Source**: test suite
- **Issue**: Only CLI surface contract tests exist (`command_surface_contract.rs`). There are no unit tests covering `read_kb_index`, `write_kb_index`, `kb_add`, `kb_list`, `kb_search`, or `kb_show` behavior.
- **Fix**: Add unit tests in `crates/nexus42/src/commands/creator.rs` (or a dedicated `kb.rs` module) that exercise happy path, missing index, corrupt index, empty search, and duplicate ID scenarios using a temporary directory.

#### S-5: Consider extracting KB logic to a dedicated module
- **Source**: `crates/nexus42/src/commands/creator.rs` (lines 289–502)
- **Issue**: ~210 lines of KB-specific logic (structs, I/O helpers, subcommand handlers) are embedded in `creator.rs`. This will grow as `world` scope and more features are added.
- **Fix**: Extract to `commands/kb.rs` to keep `creator.rs` focused on creator lifecycle commands.
