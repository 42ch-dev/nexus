## Findings

### 🔴 Critical

- **F-001 — `creator kb show` allows path traversal through user-controlled `entry_id`.**
  - Evidence: `kb_show` builds a path with `entries_dir.join(format!("{entry_id}.md"))` without validating that `entry_id` is a single safe path segment (`crates/nexus42/src/commands/creator.rs:432-448`). In Rust, joining an absolute RHS can replace the base path, and relative components such as `..` remain meaningful to filesystem calls.
  - Impact: a user-provided entry id can read outside the KB entries directory when the resulting `*.md` path exists, e.g. absolute paths or enough `../` components. This violates the plan’s explicit path traversal safety concern for entry IDs.
  - Required fix: validate `entry_id` against a strict ID regex such as `^kb_[0-9a-f]{8}$` (or the frozen entry-id contract), reject `/`, `\\`, `..`, absolute paths, control characters, and ideally construct paths only from a validated filename. Add regression tests for `../`, `..\\`, absolute paths, and control characters.

- **F-002 — KB write paths trust unvalidated `active_creator_id`, enabling writes outside `~/.nexus42/creators/...`.**
  - Evidence: `resolve_kb_paths` takes `config.active_creator_id` directly (`crates/nexus42/src/commands/creator.rs:304-313`), then `kb_add` creates directories and copies files under `creator_kb_entries_dir` (`crates/nexus42/src/commands/creator.rs:471-482`). `creator_kb_dir` / `creator_kb_entries_dir` simply join `creator_id` and `workspace_slug` (`crates/nexus-home-layout/src/lib.rs:106-120`), unlike `creator_soul_md_path`, which has an explicit `assert_creator_id_safe` defense-in-depth guard (`crates/nexus-home-layout/src/lib.rs:128-165`). Separately, `creator use` persists arbitrary strings as `active_creator_id` (`crates/nexus42/src/commands/creator.rs:899-907`).
  - Impact: `nexus42 creator use ../../outside` followed by `creator kb add --file <path>` can create directories and write copied content outside the intended ADR-014 creator workspace subtree. This is a direct path traversal write primitive.
  - Required fix: validate creator IDs before using them in KB path helpers, preferably in both the CLI resolution path and the shared `nexus-home-layout` helper as defense-in-depth. Use the existing domain creator-id validator or a shared safe path-segment assertion, and add tests proving KB paths reject `/`, `\\`, `..`, and control characters. Validate workspace slugs too at the boundary, even if current workspace commands usually validate them.

### 🟡 Warning

- **F-003 — Corrupt `index.json` is silently treated as empty and then overwritten, causing data loss.**
  - Evidence: `read_kb_index` returns `KbIndex::default()` if the file cannot be read or parsed (`crates/nexus42/src/commands/creator.rs:330-339`). `kb_add` then appends one entry to that default index and `write_kb_index` overwrites `index.json` via `std::fs::write` (`crates/nexus42/src/commands/creator.rs:490-498`).
  - Impact: a malformed or partially written index makes existing entries disappear from list/search and the next add can replace index metadata with a single new entry. That leaves orphaned entry files and hides the corruption from the operator.
  - Required fix: distinguish missing from corrupt index. Missing can initialize an empty index; corrupt/unreadable should return an error or back up the corrupt file before rebuilding. Write through a temp file in the same directory and atomic rename to reduce partial-write corruption risk.

- **F-004 — Entry ID generation can collide and overwrite an existing entry file.**
  - Evidence: `generate_entry_id` uses only the lower 32 bits of the current nanosecond timestamp (`crates/nexus42/src/commands/creator.rs:351-358`), and `kb_add` copies directly to `entries/<entry_id>.md` without checking whether the destination already exists (`crates/nexus42/src/commands/creator.rs:479-482`).
  - Impact: fast repeated adds, clock anomalies, or collisions can overwrite an existing KB entry file while appending another index row with the same ID.
  - Required fix: generate stronger IDs (UUID/random or monotonic with retry) and open/copy with create-new semantics or explicitly retry on destination existence.

### 🟢 Suggestion

- **F-005 — Add behavior tests for KB file operations, not only help surface tests.**
  - Current `command_surface_contract` additions verify `--help` output only (`crates/nexus42/tests/command_surface_contract.rs:535-581`). Add tests around add/list/show/search with a temporary HOME/config, including missing index, corrupt index, duplicate/collision prevention, and the traversal cases above.
