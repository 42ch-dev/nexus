## Findings

### 🟡 Warning

#### W-001: Test flakiness due to parallel HOME environment manipulation

**Location**: `crates/nexus42/src/creator_identity.rs:157-215` (tests module)

**Issue**: Tests manipulate the `HOME` environment variable to isolate test directories. When tests run in parallel (default cargo test behavior), multiple tests may simultaneously modify `HOME`, causing race conditions. The test `cache_roundtrip_save_load` failed in batch execution but passed when run individually, confirming this is a parallel execution issue.

**Evidence**: 
- Batch test run showed `cache_roundtrip_save_load` assertion failure (left: 0, right: 2)
- Individual test run of the same test passed
- All tests in this module use the pattern: `std::env::set_var("HOME", tmp.path())`

**Impact**: CI reliability — intermittent test failures may block merges incorrectly.

**Fix**: Add `#[serial]` attribute from `serial_test` crate, or use a mutex-based synchronization mechanism for tests that manipulate global environment variables. Alternative: use `tempfile::TempDir` with explicit path injection rather than `HOME` manipulation.

### 🟢 Suggestion

#### S-001: Test isolation via explicit path injection

**Location**: `crates/nexus42/src/creator_identity.rs`

**Issue**: The current test approach uses environment variable manipulation which is inherently global and causes parallel test flakiness.

**Improvement**: Consider refactoring the cache functions to accept an optional explicit path parameter for testing purposes, similar to how `nexus_home_layout` functions accept `home: &Path` as a parameter. This would allow tests to inject temp directories directly without modifying global state.

- **Format**: Simple `HashMap<String, CreatorIdentityEntry>` keyed by `creator_id` — appropriate for V1.16 pre-1.0 disposable scope

### Identity Resolution Logic ✅

`resolve_creator_ref()` correctly implements the three-step resolution per plan:
1. Exact `creator_id` match → return that ID
2. Exact `handle` match → return matched `creator_id`
3. Path-safe unknown → pass through (backward compat)
4. Unsafe characters → error

Security: Properly uses `validate_creator_id_safe()` from `nexus_home_layout` for path safety check.

### Auth Header Selection ✅

`creator_auth_headers()` correctly implements auth path selection:
- Path 1: Creator API key → `Authorization: Bearer <key>` (no X-Creator-Id)
- Path 2: User token → `Authorization: Bearer <token>` + `X-Creator-Id: <creator_id>`
- Path 3: No credentials → clear error message

Pure function design (no network calls) matches plan intent.
