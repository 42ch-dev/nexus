### 3.2 Warning — **2**

#### W1: Spec §11.2 — `capabilities/` directory absent

**Spec §11.2 target layout** declares:
```
crates/nexus-acp-host/src/
└── capabilities/
    ├── session_persistence.rs
    └── session_modes.rs
```

**Actual**: The `capabilities/` directory does not exist. The V1.4 capability IDs (`session.persistence`, `session.modes`) from spec §5.2a are not declared in `skills.rs` — only the V1.0 frozen IDs (`file_system.read`, `file_system.write`, `terminal.create`, `terminal.output`, `terminal.release`) are present.

**Impact**: This is a **spec compliance gap**. The spec explicitly says these capabilities are "declared by the worker during `initialize`" in V1.4. The worker stub (`acp_worker.rs`) does not call `build_v1_0_capabilities` or any capability builder, so the gap is not yet runtime-visible. However, WS2 will need these constants when the worker main loop calls `initialize` with the full capability set.

**Recommendation**: Either (a) add the two capability constants to `skills.rs` (matching the `capabilities` module pattern already established) and update `build_v1_0_capabilities` to `build_capabilities` that includes V1.4 additions, or (b) file a residual noting this is deferred to WS2. The spec says "Declared" for V1.4, so (a) is preferred.

**Severity**: Warning — not blocking WS1 merge (worker is a stub), but blocks WS2 correctness.

#### W2: `SimpleClientHandler` docstring says "auto-grants all tool requests" but the old doc referenced ACP-R8 daemon-mediated routing

**Diff in `client.rs`**:
```diff
-/// Simple client handler for V1.0 — routes tools through daemon (ACP-R8).
+/// Simple client handler for V1.0 — auto-grants all tool requests.
```

The `daemon_client` field was removed from `SimpleClientHandler` and `AcpSdkAdapter::new()` / `with_connection()` signatures. The new docstring correctly describes current behavior, but the `PolicyAwareClientHandler` (which also exists in `client.rs`) still references `PermissionPolicy` and `PermissionDecision` — these are intact.

**Impact**: The removal of `daemon_client` is **correct** for Phase 1 (ACP-R8 is "plumbing only" per spec Appendix B). However, the signature change is a breaking API change for any code that was constructing `AcpSdkAdapter::new(agent_id, agent_path, daemon_client)`. The compat shim (`pub use nexus_acp_host::*`) propagates this change to all `crate::acp::*` consumers.

**Verification**: Grepped all `use crate::acp` consumers in `nexus42/src/`:
- `errors.rs`: `use crate::acp::AcpError` ✅ (type-only, unaffected)
- `commands/session.rs`: `use crate::acp::{SessionEntry, SessionManager}` ✅ (types, not constructors)
- `commands/agent.rs`: `use crate::acp::registry::...` + `use crate::acp::transport::AgentSpawner` ✅ (constructors use different types)
- `commands/policy.rs`: `use crate::acp::{DefaultPolicy, PermissionPolicy}` ✅ (trait impls, not affected)

No existing caller passes `DaemonClient` to `AcpSdkAdapter::new()`. The callers in `agent.rs` use `AcpSdkAdapter::with_connection()` which takes already-spawned stdio pipes — those call sites were updated to remove the `daemon_client` parameter. ✅

**Severity**: Warning — API surface changed; no broken callers found, but downstream consumers (if any exist in `nexus-platform`) would need updating.

### 3.3 Suggestions — **4**

#### S1: `nix` dependency in `nexus-acp-host/Cargo.toml` is unused

The crate declares `nix = { version = "0.28", features = ["signal", "process"] }` under `[target.'cfg(unix)'.dependencies]`, but no source file under `crates/nexus-acp-host/src/` imports from `nix`. It was likely copied from `nexus42/Cargo.toml` where `nix` is used in `daemon.rs`. **Recommendation**: Remove until needed (WS2 will use it for worker signal handling).

#### S2: `tempfile` as a production dependency

`tempfile = "3"` is listed under `[dependencies]` in `nexus-acp-host/Cargo.toml`. It is typically a test-only crate (used in `wiremock`-style integration tests). If it's only used in tests, move it to `[dev-dependencies]`.

#### S3: `toml` dependency justification unclear

`toml = "0.8"` appears in the new crate's dependencies. The original `nexus42` crate had it for config file parsing. If `nexus-acp-host` doesn't parse TOML files directly (it doesn't in the moved modules), this dependency is unnecessary and should be removed.

#### S4: CI gate regex is narrow — won't catch version-gated or workspace-style deps

The CI gate regex `^\s*(nexus-acp-host|agent-client-protocol)\s*=` matches lines like:
```toml
nexus-acp-host = "0.1.0"
agent-client-protocol = "=0.10.4"
```

But would **not** catch:
```toml
nexus-acp-host = { path = "../nexus-acp-host" }
```
Wait — actually it **would** because `\s*=` matches the `=` before `{`. Let me re-check: the regex is `^\s*(nexus-acp-host|agent-client-protocol)\s*=`. For `nexus-acp-host = { path = "..." }`, the `=` after the space would match. ✅ The regex is adequate for both forms.

However, it would not catch a **transitive** dependency — e.g., if someone added `nexus42` as a dependency of `nexus42d` (which would transitively bring in `nexus-acp-host`). This is an acceptable limitation: the workspace topology should be reviewed separately. The gate protects the explicit-linking rule stated in §11.3.

## 6. Architecture & Maintainability Assessment

**Strengths**:
- Clean module boundary: all ACP client logic now lives in one crate with `#![deny(clippy::unwrap_used)]`.
- Compat shim is minimal and well-documented with removal timeline (Phase 2 / WS2).
- `session_capture` correctly stays in `nexus42` due to its coupling with `DaemonClient` and `config`.
- The `LocalSetBridge` and `NexusAcpClient` trait abstraction remain intact — future SDK migration (`sacp`) is still straightforward.
- CI linkage gate provides automated regression protection for the `!Send` / LocalSet isolation invariant.

**Risks**:
- **SDK pin tightness**: `agent-client-protocol = "=0.10.4"` is exact-pinned in the new crate. This is correct per spec §1 and protects against unexpected SDK drift. When `sacp` v1.0 arrives, the adapter-trait policy will contain the migration cost.
- **`capabilities/` gap** (W1 above): The spec anticipates V1.4 capability declarations but the code only has V1.0 IDs. This is acceptable for WS1 but must close before WS2.

## 7. Security Assessment

- **No new attack surface**: The extraction is a structural refactor — no new network listeners, no credential handling changes, no permission model weakening.

- **`acp_worker` stub**: Phase-1 stub outputs JSON to stdout; no input parsing, no network I/O. Safe.
