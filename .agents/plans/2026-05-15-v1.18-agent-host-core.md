## Objective

Create `crates/nexus-agent-host`, a Rust library crate that provides a Managed-only hybrid host core for local agent execution. The host owns provider subprocesses, normalizes ACP and native CLI behavior behind narrow traits, and integrates with `nexus-daemon-runtime` without leaking provider-specific code into the runtime.

## Research Findings

### Current repository state

- `crates/nexus-daemon-runtime/` exists and is a library-only local supervisor. It currently wires lifecycle subsystems in `boot.rs`, builds Axum routes in `api/mod.rs`, and stores runtime state in `WorkspaceState`.
- `crates/nexus-acp-host/` already exists and wraps the official `agent-client-protocol = 0.11.1` SDK. It provides registry caching, subprocess spawning, ACP SDK adapter wiring, policy helpers, and session persistence under `~/.nexus42/acp/`.
- `crates/nexus42/` already exposes `nexus42 acp` commands for registry/probe/run/session/policy/permission. Current `acp run` still has incomplete prompt-loop behavior and should not be treated as the daemon host core.
- Workspace dependencies already include `tokio`, `async-trait`, `futures-util`, `tokio-util`, `serde`, `serde_json`, `thiserror`, `tracing`, `reqwest`, `toml`, `uuid`, and `chrono`; v1.18 should reuse them before adding new crates.

### ACP protocol findings

- ACP is JSON-RPC 2.0 over newline-delimited stdio in current mainstream deployments, but this repo must use the official Rust SDK via `nexus-acp-host`, not a hand-rolled parser.
- Real lifecycle is `initialize -> session/new | session/load -> session/prompt -> session/update* -> stop reason`.
- `initialize` negotiates `protocolVersion`, `clientCapabilities`, `clientInfo`, `agentCapabilities`, `agentInfo`, and `authMethods`.
- Runtime-relevant ACP capabilities include session loading, prompt content modalities, MCP transport capabilities, session config options for mode/model, cancellation, and permission requests.
- ACP providers can request permissions (`session/request_permission`) with allow/reject options. Host policy must choose deterministic outcomes in Managed-only mode.

### ACP Registry findings

- Registry-backed discovery is now materially useful: many mainstream agents are listed, including Claude Agent, Codex CLI, Gemini CLI, Cursor, Cline, OpenCode, Kimi CLI, Qwen Code, Kiro CLI, OpenHands, Docker cagent, and others.
- Existing `nexus-acp-host::registry::RegistryClient` already fetches and caches `https://cdn.agentclientprotocol.com/registry/v1/latest/registry.json`; `nexus-agent-host` should consume it instead of duplicating registry models.

### OpenDesign patterns to borrow

- Stage-level timeout management: separate launch, initialize, session creation, prompt, and shutdown timeouts.
- One process per managed provider session with explicit stdin/stdout ownership.
- Graceful model/mode configuration fallback: failure to set a requested model/mode should downgrade only when policy permits.
- Permission outcome policy should be centralized and deterministic; provider adapters should not decide safety.
- Stream normalization should produce host events for thinking, text deltas, tool calls, status changes, and terminal results.

### Multica patterns to borrow

- PATH scanning for known local CLI command names.
- A provider catalog that separates provider identity, launch strategy, trust metadata, and runtime health.
- Managed task/session state that can detect orphaned processes and emits deterministic terminal events.
- Per-provider config for model, custom args/env, MCP config, and concurrency limits.

### Native Integration Reference: OpenDesign (nexu-io/open-design)

Source: <https://github.com/nexu-io/open-design/tree/main/apps/daemon/src>

OpenDesign is a TypeScript daemon implementing multi-agent ACP communication with a rich connector/tool ecosystem. Key patterns for future native adapter development:

**ACP adapter (`acp.ts`)**:
- `detectAcpModels()`: spawns agent → initialize → session/new → reads `models` from response → returns model list. Used for model discovery before session creation.
- `attachAcpSession()`: full session lifecycle manager — initialize → session/new → optional set_model → session/prompt → stream updates → result. Returns a handle with `hasFatalError()`.
- `createJsonLineStream()`: newline-delimited JSON-RPC parser that feeds chunks and handles partial lines. Ignores non-JSON stdout lines (agent log output).
- `buildAcpSessionNewParams()`: constructs `session/new` params with resolved `cwd` and normalized MCP server configs (type/name/command/args/env).
- Stage-level timeouts: `DEFAULT_TIMEOUT_MS = 15_000` (initialization), `DEFAULT_STAGE_TIMEOUT_MS = 180_000` (prompt/execution). Each RPC stage gets its own timer.
- Model switching graceful fallback: if `session/set_model` fails with -32603 or -32602, falls back to default model and proceeds with prompt.
- Permission auto-approval strategy: `choosePermissionOutcome()` tries `approve_for_session` > `allow_always` > `allow_once`. This is configurable; nexus-agent-host should centralize this in policy.

**Multi-provider stream adapters**:
- `claude-stream.ts`, `copilot-stream.ts`, `qoder-stream.ts`: per-provider stream normalization. Each wraps provider-specific output format into unified agent events.
- All adapters follow the same contract: spawn → attach → stream events → handle error/timeout → cleanup.

**Connector system (`connectors/`)**:
- `catalog.ts`: connector catalog with tool safety classification.
  - `ConnectorToolSafety { sideEffect, approval, reason }` — three axes: `read`/`write`/`destructive` × `auto`/`confirm`/`disabled`.
  - `classifyConnectorToolSafety()`: regex-based auto-classification. Checks tool name/description for destructive hints (drop, truncate, purge, erase, wipe) → disabled; write hints (create, update, delete, send, manage) → confirm; read hints (get, list, search, fetch, view) → auto. Default is write/confirm for unknown tools.
  - `isRefreshEligibleConnectorToolSafety()`: only `read + auto` tools can be auto-refreshed (no confirmation needed for periodic operations).
- `service.ts`: connector service managing lifecycle and state.
- `routes.ts`: API routes for connector CRUD.

**Provider-agent architecture (`agents.ts`)**:
- Agent definitions with provider identity, capabilities, and configuration.
- Multi-agent concurrent execution support.

**Relevance to nexus-agent-host**:
- Wave N native adapters for Codex/Gemini/OpenCode should follow OpenDesign's per-provider stream adapter pattern (each provider gets its own normalization module).
- Tool risk auto-classification (R-006) should borrow OpenDesign's regex-based approach for future waves.
- Stage-level timeout values (15s init, 180s prompt) are validated production defaults worth adopting.

### Native Integration Reference: Multica (multica-ai/multica)

Source: <https://github.com/multica-ai/multica> (Go server + local daemon)

Multica is a hosted agent platform with a local daemon that discovers agent CLIs on PATH, registers as a Runtime, and executes tasks server-assigned tasks. Key patterns for future native adapter and orchestration development:

**Daemon architecture**:
- Local daemon scans `PATH` for known agent CLI commands on startup.
- Registers the machine as a "Runtime" (compute environment) with the cloud server.
- Dual transport: WebSocket (real-time priority) + HTTP polling (reliable fallback).
- Receives task assignments from server → claims tasks → executes via local agent CLIs → streams progress → reports completion/failure.

**Task lifecycle (daemon → server API)**:
- `POST /api/daemon/runtimes/{id}/tasks/claim` — atomically claim next queued task.
- `POST /api/daemon/tasks/{id}/start` — mark task as running.
- `POST /api/daemon/tasks/{id}/progress` — report execution progress (streaming updates).
- `POST /api/daemon/tasks/{id}/messages` — batch report agent execution messages.
- `POST /api/daemon/tasks/{id}/complete` — mark done with output, branch, session info.
- `POST /api/daemon/tasks/{id}/fail` — mark failed with error and failure reason.
- `POST /api/daemon/tasks/{id}/session` — persist session_id + work_dir mid-execution (crash recovery).
- `GET /api/daemon/tasks/{id}/status` — check if task was cancelled during execution.
- `POST /api/daemon/runtimes/{id}/recover-orphans` — mark previously-running tasks as failed (daemon restart recovery).

**Agent configuration (server-side)**:
- Agent as first-class workspace citizen: owner, visibility (public/private), config, custom env, model selection, MCP config, concurrency limits.
- Per-agent custom environment variables and command arguments.
- Per-agent MCP server configuration.
- Agent-to-Agent (A2A) traffic always allowed (even for private agents); trust boundary is member↔agent, not agent↔agent.

**Task queue and scheduling**:
- Task lease model with atomic claim (database-level uniqueness).
- Candidate index optimization for claim queries.
- Per-provider concurrency limits enforced at queue level.
- Autopilot mode: periodic scheduled task creation and agent assignment.

**Crash recovery and reliability**:
- Orphan task detection: on daemon restart, `recover-orphans` marks tasks from previous daemon session as failed.
- Session persistence: `session_id` and `work_dir` saved mid-execution for potential resume.
- `force_fresh_session` option to override session restore.

**Skills system**:
- Skills are portable Markdown instruction sets injected into agent working directories before execution.
- Multi-to-many relationship between skills and agents.
- Structured skill storage (not free-text) with file attachments and workspace-level ownership.

**Known supported CLI agents (from Multica quickstart)**:
- Claude Code (`claude`)
- Codex CLI
- GitHub Copilot CLI
- Gemini CLI
- Kimi CLI
- Kiro CLI

**Relevance to nexus-agent-host**:
- Task lease + claim pattern is a proven model for future multi-session concurrent execution (post-Wave 1 hardening).
- Orphan recovery pattern should inform session cleanup under daemon crash (spec Section 10.19 open risk #5).
- Per-agent config (model, env, args, MCP, concurrency) validates the plan's `[[providers]]` TOML structure.
- Skills injection pattern could inform future "pre-session context injection" for nexus-agent-host sessions.
- PATH scan discovery is directly applicable to `discovery/path_scan.rs`.

### Spec Section 10 gap analysis

- **Adjust**: Section 10 implies building an ACP adapter inside `nexus-agent-host`. In this repo, `nexus-acp-host` already owns ACP SDK, registry, and transport primitives. v1.18 should create a higher-level host crate that depends on `nexus-acp-host` for ACP implementation.
- **Adjust**: Section 10 lists worker lease management in Wave 1. Full task lease/retry semantics are too large for this plan; Wave 1 should implement session/op admission and in-memory operation tracking, leaving cross-daemon crash recovery for a hardening wave.
- **Add**: Capability negotiation must map actual ACP initialize fields rather than a generic capability enum only.
- **Add**: Session config operations (`set_model`, `set_mode` or SDK equivalent config options) should be part of session launch/execution planning, with graceful fallback.
- **Add**: Permission handling must account for ACP `session/request_permission` and native CLI tool-risk classification.
- **Narrow**: Host API routes should land in `/v1/local/agent-host/` in runtime, but CLI UX changes should be minimal in v1.18; existing `nexus42 acp` commands remain registry/probe surface.

## Architecture Design

### Selected approach

Build `nexus-agent-host` as the orchestration/facade crate above `nexus-acp-host`:

```text
nexus42 CLI
  └─ nexus-daemon-runtime
       ├─ lifecycle subsystem: AgentHostSubsystem
       ├─ Axum routes: /v1/local/agent-host/*
       └─ Arc<dyn HostFacade>
            └─ nexus-agent-host
                 ├─ core: HostManager, SessionRegistry, OpRegistry
                 ├─ capability: normalized host operations + negotiation
                 ├─ discovery: config + PATH + ACP registry catalog
                 ├─ providers/acp: official SDK via nexus-acp-host
                 └─ providers/native_cli/claude: Wave 1 native adapter
```

This keeps the daemon runtime a supervisor/client, avoids duplicate ACP plumbing, and preserves Section 10's Hybrid + Managed-only decision.

### ACP provider implementation

The ACP provider must use `nexus-acp-host` primitives:

1. Resolve provider launch strategy from explicit config or registry entry.
2. Spawn provider with `nexus_acp_host::transport::AgentSpawner`.
3. Build `nexus_acp_host::AcpSdkAdapter::with_connection(...)` from child stdio.
4. Send `initialize` with:
   - latest supported protocol version,
   - client info `nexus42` + crate version,
   - client capabilities for filesystem, terminal, and MCP as policy allows.
5. Convert initialize response into `CapabilityDescriptor`:
   - `load_session` from `agentCapabilities.loadSession`,
   - prompt modality support from prompt capabilities when exposed by SDK DTOs,
   - MCP transport support from MCP capabilities,
   - auth methods from initialize response.
6. Create or load a session:
   - Wave 1 default: always `session/new` for Managed-only reproducibility.
   - If request explicitly asks restore and `load_session` is supported, allow `session/load` only for host-owned persisted sessions.
7. Apply optional model/mode:
   - Use SDK config-option APIs where available.
   - If setting fails and policy allows fallback, emit `Status` warning and continue with default.
   - If policy disallows fallback, return `CapabilityUnsupported` or `ProviderProtocolError`.
8. Execute prompt:
   - Translate `HostContentBlock` into ACP content blocks.
   - Stream ACP `session/update` messages into `HostEvent` variants.
   - Emit exactly one terminal `OpFinished` or `OpFailed`.
9. Permission requests:
   - Central policy chooses from provider options in this preference order when allowed: session-scoped approval, `allow_always`, `allow_once`; reject otherwise.
   - Destructive or unknown tool classes default to deny/ask per config, never adapter-local auto-allow.
10. Shutdown:
   - Cancel active op if any, wait for configured graceful timeout, terminate child, then force kill if needed.

### Native CLI provider strategy (Wave 1)

Wave 1 implements **Claude Code CLI native provider** (`providers/native_cli/claude.rs`). Rationale:

- It is the most common native target in the existing CLI-agent ecosystem and is already represented conceptually by current `nexus42 acp` examples/tests.
- It provides a useful non-ACP baseline for the Hybrid architecture without adding multiple protocol-drift surfaces.
- Native support should be intentionally narrower than ACP in Wave 1: managed subprocess ownership, prompt execution, cancellation by process termination, stdout/stderr event normalization, health/probe, and configured model/args/env.

Native provider constraints:

- Default command lookup: `claude` on PATH, overridable in `~/.nexus42/agent-host/config.toml`.
- Default mode: non-interactive single-turn prompt through configured args. Do not claim full Claude-specific session restore unless verified by implementation tests.
- Streaming: normalize stdout lines/chunks to `MessageDelta`; stderr lines become `Status` warnings unless the process exits non-zero.
- Tool-call visibility: native CLI output is unstructured in Wave 1; report `structured_tool_calls = false` in `CapabilityDescriptor`.
- Cancellation: send child kill/termination sequence; map to `OperationCancelled` when host initiated.

### Discovery mechanism

Discovery is deterministic and ordered:

1. **Static config** from `~/.nexus42/agent-host/config.toml`:
   - explicit provider ids,
   - protocol kind,
   - command template,
   - args/env,
   - allow/deny state,
   - timeout/concurrency overrides.
2. **PATH scan** for known native commands:
   - Wave 1: `claude` only.
   - Later waves: `codex`, `gemini`, `opencode`, `cursor`, `kimi`, etc.
3. **ACP registry** via `nexus_acp_host::RegistryClient`:
   - include registry entries with runnable distributions for current platform,
   - annotate as `protocol_kind = acp`,
   - preserve registry metadata and trust source.

Catalog entries:

```rust
pub struct ProviderCatalogEntry {
    pub provider_id: ProviderId,
    pub display_name: String,
    pub protocol_kind: ProtocolKind, // Acp | NativeCli
    pub launch: LaunchStrategy,
    pub source: DiscoverySource,     // Config | PathScan | AcpRegistry
    pub trust: TrustLevel,           // Explicit | Registry | LocalPath
    pub capabilities: CapabilityDescriptor,
    pub health: ProviderHealth,
}
```

Deduplication rule: explicit config wins over PATH, PATH wins over registry aliases, and disabled config entries suppress matching auto-discovered entries.

**ACP-vs-native coexistence (R-004, R-008)**: dedupe applies per `provider_id`, not per agent identity. ACP registry and PATH scan produce different provider IDs (e.g., `claude` from registry vs `claude-native` from PATH), so both can coexist in the catalog. When the same agent has both ACP and native entries and no explicit config, ACP registry entry takes priority (richer protocol). Explicit config can suppress either or both.

### Session state machine

Host-level sessions use the spec state machine with explicit op substates:

```text
Created
  -> Starting
  -> Ready
  -> Busy(op_id)
  -> Ready
  -> Stopping
  -> Stopped

Busy(op_id) -> Cancelling(op_id) -> Ready | Stopped
Starting | Busy | Cancelling | Stopping -> ErrorRecoverable | ErrorTerminal
ErrorRecoverable -> Starting     # only when restart policy permits
ErrorTerminal -> Stopped         # new explicit session required
```

Rules:

- One active op per session in Wave 1.
- `exec(Prompt)` requires `Ready` and transitions to `Busy`.
- `cancel(op_id)` requires `Busy(op_id)` and transitions to `Cancelling`.
- Provider process exit during `Busy` is `ErrorRecoverable` only if no host state is corrupted and retry caps allow relaunch; otherwise `ErrorTerminal`.
- Every path out of `Busy` emits one terminal host event.
- `shutdown()` transitions all sessions through `Stopping` and drains with configured grace timeout.

### Error handling

Host-visible error taxonomy remains Section 10.15, with concrete mappings:

| Host error | ACP mapping | Native mapping |
| --- | --- | --- |
| `ProviderUnavailable` | registry entry unavailable, missing command, health probe failed | command absent on PATH |
| `LaunchFailed` | spawn/stdio/SDK connection setup failed | process spawn failed |
| `CapabilityUnsupported` | initialize capabilities or config options lack required feature | native descriptor lacks operation |
| `PolicyDenied` | permission outcome denied by host policy | command/provider/capability denied |
| `OperationTimeout` | stage timeout around initialize/session/prompt/shutdown | process timeout |
| `OperationCancelled` | ACP cancel acknowledged or host cancelled stream | host killed/cancelled process |
| `ProviderProtocolError` | invalid ACP response, SDK protocol error, unexpected stop | malformed native output when structured mode is expected |
| `InternalHostError` | registry/session/op invariant violation | same |

`HostError` should carry `provider_id`, optional `session_id`, optional `op_id`, `category`, `message`, and optional `source` string safe for local logs.

### Capability negotiation

Normalize capabilities into:

```rust
pub struct CapabilityDescriptor {
    pub text_prompt: bool,
    pub streaming: bool,
    pub cancellation: bool,
    pub session_restore: bool,
    pub structured_tool_calls: bool,
    pub mcp_http: bool,
    pub mcp_sse: bool,
    pub mcp_stdio: bool,
    pub images: bool,
    pub audio: bool,
    pub embedded_context: bool,
    pub set_model: bool,
    pub set_mode: bool,
    pub diagnostics: bool,

```

Negotiation happens at session creation:

1. Merge static provider descriptor with ACP initialize response or native descriptor.
2. Apply host policy deny/allow toggles.
3. Validate requested `CreateSessionRequest` features.
4. Store negotiated descriptor on `HostSession`; all later ops check it before provider dispatch.

### Daemon-runtime integration

Integration is via narrow facade only:

- `WorkspaceState` stores `Option<Arc<dyn HostFacade>>` plus setters/getters, similar to engine/worker manager patterns.
- `boot.rs` constructs `HostManager`, loads config, calls `start`, stores the facade, and adds `AgentHostSubsystem` to lifecycle.
- `lifecycle::SubsystemKind` gains `AgentHost` if needed for health reporting.

## PM Review Notes (2026-05-15)

Cross-referenced plan against ACP protocol docs, ACP registry, OpenDesign source (`apps/daemon/src/acp.ts`, `connectors/catalog.ts`), Multica docs (daemon communication, agent backend), and actual `crates/nexus-acp-host` source. 20 core findings accurate; 8 issues identified below.

### Issue R-001: `NexusAcpClient` trait lacks streaming method [MEDIUM]

`NexusAcpClient::prompt()` returns `AcpResult<NexusPromptCompleted>` (one-shot), not a stream. The old `subscribe()` broadcast mechanism was removed (client.rs line 29-33: "will be replaced with a proper DTO-wrapped streaming API in a future task"). The SDK's `ActiveSession::read_update()` can read streaming `session/update` messages, but this is not exposed through the public trait.

**Resolution for T12**: ACP provider must either (a) extend `nexus-acp-host` with a new streaming method wrapping `ActiveSession::read_update()` before building `AcpProvider::execute()`, or (b) use the SDK type directly within `nexus-acp-host` (already `!Send`-bridged via `LocalSetBridge`). This may add scope to T12; PM should allow a small T12a sub-task if needed. The plan's Risk row about "lacks a rich public streaming DTO API" already acknowledges this.

### Issue R-002: `HostOperation::Cancel` and `HostOperation::Health` are not operations [MEDIUM]

`Cancel` and `Health` are control-plane actions, not execution operations. ACP protocol sends `session/cancel` as a notification (no id). `HostFacade` already has dedicated `cancel(op_id)` and `health()` methods. Having them as `HostOperation` variants passed to `exec()` creates semantic confusion.

**Resolution**: Remove `HostOperation::Cancel` and `HostOperation::Health` from the enum. Cancel flows through `HostFacade::cancel()` / `ProviderAdapter::cancel()`. Health is a separate `HostFacade::health()` query. Update `HostOperation` to only contain execution-scoped variants: `Prompt`, `SetModel`, `SetMode`. Update T3 done criteria accordingly.

### Issue R-003: Permission policy integration with existing `nexus-acp-host::PermissionPolicy` unclear [MEDIUM]

`nexus-acp-host::policy::PermissionPolicy` already implements a full permission engine with grant/deny/ask decisions, per-agent scoped rules, and TOML persistence at `.nexus42/permissions.toml`. The plan mentions `policy/permission.rs` for "ACP permission outcome selection" but does not state how it relates to the existing policy module.

**Resolution**: `nexus-agent-host`'s permission handling for ACP providers must **delegate** to `nexus-acp-host::PermissionPolicy`. The host-level `policy/admission.rs` adds provider/capability/session admission gates on top (e.g., "is this provider allowed at all?"), but once an ACP `session/request_permission` arrives, the outcome selection calls through to `PermissionPolicy::evaluate_for_agent()`. Native CLI providers use the host-level risk classification only (no ACP permission delegation). Add this integration note to T10 and T12 descriptions.

### Issue R-004: Claude CLI native provider command name ambiguity [LOW]

The ACP registry lists "Claude Agent" (an ACP wrapper), while the native CLI is "Claude Code" (`claude` binary). If a user has both installed, discovery may produce two entries for the same underlying agent.

**Resolution**: Discovery dedupe rule should clarify: when the same agent identity (e.g., "claude") appears in both ACP registry and PATH scan, explicit config wins; if no config, ACP registry entry takes priority (protocol-rich) unless user has explicitly configured a native CLI entry. Native CLI entries should use distinct provider IDs like `claude-native` to avoid collision with ACP registry's `claude` agent ID.

### Issue R-005: Claude CLI `args = ["-p"]` is a placeholder [LOW]

Claude Code CLI's exact non-interactive flag needs verification at implementation time (may be `-p`, `--print`, or version-dependent).

**Resolution**: Mark config example args as placeholder. T13 implementation must verify actual CLI flags and update defaults accordingly.

### Issue R-006: OpenDesign tool risk auto-classification not fully leveraged [LOW]

OpenDesign's `connectors/catalog.ts` implements `classifyConnectorToolSafety()` with regex-based pattern matching: destructive patterns (drop, truncate, purge) → disabled; write patterns (create, update, delete) → confirm; read patterns (get, list, search) → auto. The plan's `capability/risk.rs` mentions risk classes but not auto-classification.

**Resolution for future waves**: `capability/risk.rs` should define a `ToolRiskClassifier` trait in Wave 1 that supports static declaration, with an auto-classification implementation based on name/pattern heuristics (borrowing OpenDesign's regex approach) in a later wave. This is an extension point, not a Wave 1 blocker.

### Issue R-007: Session persistence vs `nexus-acp-host::SessionManager` relationship unclear [LOW]

`nexus-acp-host::session_manager::SessionManager` already persists sessions under `~/.nexus42/`. The plan's config contract mentions `~/.nexus42/agent-host/sessions.json` without clarifying reuse.

**Resolution**: Wave 1 uses in-memory session tracking only (no host-level persistence). ACP sessions can optionally use existing `SessionManager` for restore; native sessions have no persistence in Wave 1. The `sessions.json` path in config contract is reserved for future use. Update config contract section to note this explicitly.

### Issue R-008: Missing discovery dedupe rule for same-agent ACP-vs-native coexistence [LOW]

Related to R-004 but broader: the dedupe rule "config > PATH > registry" handles priority but not the case where a user wants both ACP and native entries for the same agent simultaneously.

**Resolution**: Dedupe rule applies per `provider_id`, not per agent identity. ACP registry and PATH scan produce different `provider_id` values (e.g., `claude` from registry vs `claude-native` from PATH), so both can coexist. Explicit config can suppress either. This is already implied by the plan but should be documented in the discovery section.
