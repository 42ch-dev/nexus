# V1.4 WS3 — Preset loader + `novel-writing` end-to-end

> **For agentic workers:** Bite-sized tasks below; commit after each unless noted.

**Plan ID**: `2026-04-17-v1.4-ws3-preset-loader-novel-writing`
**Wave**: V1.4 WS3 — see [knowledge/v1.4-delivery-compass-v1.md](knowledge/v1.4-delivery-compass-v1.md) §4 WS3
**Wave-0 spec (authoritative)**: [knowledge/orchestration-engine-v1.md](knowledge/orchestration-engine-v1.md) §7 (preset bundle format), §8 (loader mapping rules), §4.4 (Task impls — `AcpPromptTask` here), §5.2 (remaining ACP capabilities)
**Working branch**: `create feature/v1.4-ws3-preset-loader from feature/v1.4-ws2-orch-skeleton` (or main once WS2 merges).

**Goal**: Light up dynamic preset execution end-to-end: YAML loader → `graph-flow::Graph`, ACP-facing capabilities (`acp.prompt`, `acp.session_load`, `judge.llm`, `creator.*` remaining), `AcpPromptTask` + `InnerGraphTask` full implementations, embedded `novel-writing` preset that an E2E test can drive through all four outer states against a mocked ACP agent.

**Architecture**: Loader parses `preset.yaml` + `prompts/*.md` into a validated `LoadedPreset`, then maps state/node declarations onto `graph-flow` primitives per §8.2. `AcpPromptTask` bridges to the Worker Manager via IPC (`worker/acp_prompt` + stream of chunks). Embedded preset ships via `include_dir!`.

**Tech Stack**: Rust, `serde_yaml`, `handlebars`, `include_dir` (for embedded presets), existing `graph-flow` + `nexus-acp-host` + Worker Manager from WS1/WS2.

- [x] **Step 2: Implement `AcpPromptTask`**

Per [knowledge/orchestration-engine-v1.md](knowledge/orchestration-engine-v1.md) §4.4 (AcpPromptTask row) + §6.4 (worker/acp_prompt request/response shapes):

- Constructor `AcpPromptTask::new(worker_manager, creator_id, state_id, template, tool_policy)` and the test helper `new_for_test(handle, state_id, template, tool_policy)`.
- `run(ctx)`:
  1. Resolve the creator's `WorkerHandle` (via Worker Manager; test helper passes one in directly).
  2. Render the template with `handlebars` against `ctx` (bindings: `core_context.*`, `state.*`, `preset.input.*`).
  3. Call `worker/acp_prompt { prompt, tool_policy, session_id }` via `WorkerHandle::call_json_rpc`.
  4. Stream `worker/acp_prompt_chunk` notifications into `ctx.chat_history` (use `graph_flow::Context::add_assistant_message`).
  5. On final `worker/acp_prompt` reply, store `result.full_text` at `ctx["state.<state_id>.output"]`.
  6. Return `TaskResult { response: Some(full_text), next_action: NextAction::Continue }`.

- [x] **Step 3: Implement the three capabilities**

Each thin-wraps around existing functionality:
- `acp.prompt` — input schema `{ prompt: string, tool_policy: enum, session_id?: string }`; output `{ full_text: string }`. `run()` delegates to the current-creator's worker (reads the creator id off `ctx`).
- `acp.session_load` — input `{ session_id: string }`; calls `worker/acp_session_load`.
- `judge.llm` — input `{ prompt: string, creator_id?: string }`; calls `acp.prompt` with `tool_policy: DenyAll`; parses final text as boolean ("go" / "yes" / "proceed" etc. → true; "wait" / "no" → false; ambiguous → error).

Register all three in `CapabilityRegistry::with_builtins()` extended for WS3.

- [x] **Step 4: Extend `acp_worker` subcommand to the real JSON-RPC loop**

Replace the Phase-1 stub (WS1 T7) in `crates/nexus42/src/commands/acp_worker.rs` with:
- Spawn the agent subprocess via `nexus-acp-host::transport`.
- Main loop reads JSON-RPC frames from stdin, dispatches:
  - `worker/initialize` → reply with `capabilities`, `worker_pid`.
  - `worker/acp_prompt` → call into `NexusAcpClient::prompt`, stream chunks back as `worker/acp_prompt_chunk` notifications, send final reply.
  - `worker/acp_cancel`, `worker/acp_session_load`, `worker/health`, `worker/shutdown` — per §6.4.
- Upcall dispatch: agent tool requests map to `worker/agent_tool_request` notifications; daemon replies over stdin.
- Uses `tokio::task::LocalSet` (since `agent-client-protocol` futures are `!Send`).
