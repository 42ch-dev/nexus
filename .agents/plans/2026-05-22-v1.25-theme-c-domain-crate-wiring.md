## 1. Problem Statement

V1.25 KCA-25-006 and KCA-25-007 found that the domain crates for narrative state, World KB, and user knowledge are Cargo-reachable but not product-wired through CLI or daemon Local API surfaces. Current `/v1/local/kb/*` routes are work-scope file index routes and must not be treated as `nexus-kb` World KB wiring.

This plan adds a minimal read-only narrative Local API surface and a minimal `creator kb --scope world` path using existing in-memory store implementations only. It also records the persistence decision needed for `reference_sources` before deeper `nexus-knowledge` product wiring.

---

## 2. Goals

1. Add a minimal daemon narrative read surface under `/v1/local/narrative/*` using `NarrativeGateway` and in-memory stores.
2. Wire `creator kb --scope world` to `nexus-kb` instead of short-circuiting through the current deferred path.
3. Decide and document the persistence posture for `reference_sources` versus `nexus-knowledge::ReferenceSource` before changing production reference storage.
4. Preserve the existing work-scope `/v1/local/kb/*` contract and avoid a full KB route redesign.
5. Keep Theme C explicitly in-memory for C1/C2; production persistence is deferred.

---

## 3. Non-goals / Out of Scope

- No SQLite-backed `nexus-narrative`, `nexus-kb`, or `nexus-knowledge` stores.
- No cloud sync for narrative, World KB, user knowledge, or reference sources.
- No full Local API KB route redesign and no change to existing work-scope `/v1/local/kb/*` semantics.
- No write operations for narrative state.
- No production persistence across daemon restarts for C1 or C2.
- No `nexus-platform` private-repo work.
- No implementation of future `platform knowledge` commands in this plan.

### C1.2 — `creator kb --scope world` via `nexus-kb` (DF-42)

| Field | Value |
| --- | --- |
| Owner | `@fullstack-dev` |
| Files | `crates/nexus42/src/commands/creator/kb.rs` or the current `creator kb` command module; any local helper module required for world ID resolution; tests matching existing CLI command conventions |
| Change | Replace the `world_scope_deferred()` short-circuit with actual `nexus-kb` wiring backed by `InMemoryKbStore`. The `--scope world` path must resolve a `world_id`, query/list world-scoped key blocks through `KbStore`, and return a deterministic CLI result rather than “deferred”. |
| World ID resolution | Use the existing CLI inputs if `world_id` is already provided. If the command only has work/creator context, add the smallest explicit `--world-id <id>` requirement for `--scope world` and return a clear error when omitted. Do not silently infer an unstable world ID. |
| Product posture | In-memory only; this proves CLI-to-crate reachability and scope correctness, not production persistence. |
| Acceptance | `nexus42 creator kb --scope world --world-id <id> ...` reaches `InMemoryKbStore` and returns a valid empty or seeded world-scoped KB result. The command no longer exits through `world_scope_deferred()` for the supported world-scope read/list path. |
| Verify | Targeted CLI tests show: missing `--world-id` fails with a clear message; provided `--world-id` uses the world-scope path; existing work-scope KB behavior remains unchanged. |

Implementation notes:

- Preserve V1.24 policy: default `creator kb` and `/v1/local/kb/*` are work-scope file index surfaces, not World KB.
- Keep World KB support narrow to the command shape accepted in C0/C1; do not add broad write/mutation semantics unless already present in `creator kb` and explicitly approved by PM.
- `InMemoryKbStore` may return an empty result for a new world. That is acceptable if the CLI output clearly represents an empty World KB rather than a deferred feature.

## 8. Future Platform Knowledge — Out of Scope

Future `platform knowledge` CLI work remains out of scope for this plan. If planned later, the recommended CLI posture is:

- Treat `platform knowledge` as **User-scoped** knowledge, not World KB.
- Keep it distinct from `creator kb --scope world` and from work-scope `/v1/local/kb/*`.
- Require a persistence decision for `nexus-knowledge` before claiming production user knowledge storage.

## 11. Risks and Controls

| Risk | Severity | Control |
| --- | --- | --- |
| In-memory only means no persistence across daemon restarts | High | C0 must confirm this posture; route and CLI wording must not imply production persistence. |
| `creator kb --scope world` world ID resolution may be ambiguous | High | Require explicit `--world-id` when no stable context exists; fail clearly rather than guessing. |
| Existing work-scope `/v1/local/kb/*` may be confused with World KB | Medium | Keep route names and docs distinct; QC checks no full KB route redesign occurred. |
| Dual `reference_sources` models remain after C3 | Medium | ADR must name the selected production owner and future migration trigger. |
| DTO/schema drift if narrative contracts are handwritten | Medium | Use JSON Schema / generated contract patterns as the source of truth; avoid parallel handwritten DTOs. |
| Scope creep into narrative writes, SQLite stores, or cloud sync | High | Out-of-scope list forbids these; QC verifies diff scope. |
