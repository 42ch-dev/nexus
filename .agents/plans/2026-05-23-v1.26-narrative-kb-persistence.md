## 1. Problem Statement

V1.25 Theme C added narrative HTTP routes and `creator kb --scope world` using **fresh `InMemory*` stores per request/invocation**, so product surfaces always appear empty and data does not survive restarts.

V1.26 delivers **workspace-scoped SQLite persistence** and **shared runtime handles** in the daemon (and CLI access via daemon or shared DB layer).

---

## 2. Goals

### Theme N — Narrative

1. DDL for narrative entities in workspace `state.db`.
2. `SqliteNarrativeGateway` (or equivalent) implementing `NarrativeGateway`.
3. Daemon `WorkspaceState` holds one gateway per active workspace pool.
4. Remove per-handler `InMemoryNarrativeGateway::new` in [`narrative.rs`](../../crates/nexus-daemon-runtime/src/api/handlers/narrative.rs).
5. Integration test: create world → restart daemon → `GET /v1/local/narrative/worlds` still lists world.

### Theme K — World KB

1. DDL for `key_blocks` / anchors (normalized names per `nexus-kb` model).
2. `SqliteKbStore` implementing `KbStore`.
3. Share store with narrative gateway where `NarrativeGateway` requires `KbStore`.
4. `creator kb list|search|show --scope world` uses persistent store (same DB).
5. Optional: minimal daemon `GET /v1/local/narrative/worlds/{id}/kb` — **only if** low cost; else CLI-only for V1.26.
6. Close residual **R4** (CLI test for world scope); rebind **R3** to explicit scope in work-KB handler or defer comment in DF-42.

---

## 3. Non-goals

- Narrative write/fork/manuscript APIs (read-only minimal surface OK).
- World KB write via CLI (`add`/`remove` world scope) — may remain deferred with clear message.
- Full DF-42 route redesign for `/v1/local/kb/*` work index.
- Cloud sync of narrative/KB payloads.
