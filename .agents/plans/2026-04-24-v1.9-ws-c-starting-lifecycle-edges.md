# V1.9 WS-C — Starting Lifecycle Edge Cases

## Summary

Handle three identified edge cases in the daemon lifecycle FSM's `Starting` state: (1) accept `HealthDegraded` event during `Starting` with transition to `Degraded`, (2) implement in-flight cancel of subsystem start tasks when `ShutdownRequested` arrives during `Starting`, (3) populate diagnostic `ActionContext` data in `enter_starting` for better observability.

## Background

- **Deferred feature**: DF-10 ("WS4 Starting remaining edge cases") tracked in `.agents/knowledge/deferred-features-cross-version-tracker-v1.md`.
- **Current FSM implementation**: `crates/nexus42d/src/lifecycle/state.rs` uses `statig` crate. The `Starting` state (lines 79-123) handles `SubsystemUp`, `SubsystemFailed`, `ShutdownRequested`, and `FatalError`. It does NOT handle `HealthDegraded`.
- **Current `exit_starting`**: In `actions.rs` lines 115-124, the exit action logs "in-flight starts will complete or be ignored" — a TODO. Subsystem start tasks spawned in `enter_starting()` (lines 85-113) are NOT tracked or cancelled.
- **Current `enter_starting`**: In `state.rs` lines 245-253, calls `enter_starting(ctx)` but does NOT populate diagnostic context (start reason, timestamp, initiating event) into the `ActionContext`. The `ActionContext` struct in `actions.rs` lines 21-28 currently has `lifecycle`, `subsystems`, and `shutdown_grace_ms` but no diagnostic fields.

## Design Decisions

1. **HealthDegraded during Starting**: Add `Event::HealthDegraded` match arm to the `starting()` method in `state.rs`. When received, record degraded subsystem + reason in `DaemonHsm` (same pattern as `running()` at lines 137-139), then `Transition(State::degraded())`. This is correct: if a subsystem health check fires during boot and reports degraded, we should enter Degraded rather than staying in Starting indefinitely.
2. **In-flight cancel during Starting exit**: Replace the TODO in `exit_starting()` (actions.rs lines 120-124) with actual task cancellation:
   - In `enter_starting()`, collect `JoinHandle`s for spawned subsystem tasks and store them in `ActionContext` (add `start_handles: Mutex<Vec<tokio::task::JoinHandle<()>>>` field).
   - In `exit_starting()`, iterate handles and call `.abort()` on each. This cancels any subsystem start tasks that haven't completed yet.
   - The `ShutdownRequested` transition already goes to `Stopping` (line 111-114 in state.rs); `exit_starting` fires before the transition, cancelling in-flight starts. Then `Stopping.entry` handles graceful shutdown of subsystems that DID start successfully.
3. **ActionContext diagnostics in enter_starting**: Add fields to `ActionContext`:
   - `start_reason: String` — e.g., `"daemon_boot"`, `"restart_after_failure"`
   - `started_at: chrono::DateTime<chrono::Utc>`
   - `initiating_event: Option<String>` — the event that triggered the start
   - Populate these in `enter_starting()` and emit them as structured tracing events with `target: "daemon_lifecycle"`.

## Risks

| ID | Risk | Mitigation |
|----|------|------------|
| RISK-WSC-01 | `HealthDegraded` during Starting may be triggered by subsystems that haven't fully booted | Only transition to Degraded if the degraded subsystem IS in `up_subsystems` (has reported SubsystemUp). Ignore HealthDegraded for subsystems still booting. |
| RISK-WSC-02 | `abort()` on JoinHandles may leave subsystem resources leaked (file handles, sockets) | This is a known limitation. `SubsystemBootstrap` trait doesn't have an `abort()` method. Add a note that full resource cleanup requires WS7+ work. The current `abort()` prevents the FSM from hanging, which is the M1 priority. |
| RISK-WSC-03 | Regression: existing lifecycle tests break | Run full test suite before committing; `cargo test -p nexus42d` must remain green. Manual review of `lifecycle_transitions.rs` tests. |
| RISK-WSC-04 | `ActionContext` diagnostic fields not populated in `new_for_test()` (test path) | Add `ActionContext::new_for_test()` variant that populates diagnostics as well, or accept that tests without context won't have them (current behavior: `context.is_none()` → log-only). |
