# Multi-Agent Worker — Design Specification & Implementation Plan (WS-E, DF-01, OQ-7)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extend the one-worker-per-creator MVP (V1.4) to support (1) a `WorkerRegistry` that manages one worker per active creator concurrently, and (2) each worker hosting >1 ACP agent session with session routing.

**Architecture:** Selected approach is **Approach A — multiplex one worker with session routing**: each creator gets one worker process that hosts multiple ACP agent subprocesses (each with its own `AcpSession`), routing IPC requests by `session_id`. The daemon's `WorkerRegistry` maintains a `HashMap<CreatorId, WorkerHandle>` to manage multiple concurrent creators. Within each worker, session routing multiplexes agent slots.

**Tech Stack:** Rust, `agent-client-protocol` v0.11.x, `tokio` (LocalSet per agent), JSON-RPC 2.0 over stdin/stdout, SQLite for session persistence.

**V1.7 Compass Alignment:** WS-E → Milestone M3. Depends on WS-D (ACP SDK v0.11.0+ for `session/list`).

**Revision History:**

## 1. Current State Analysis

### 1.1 Existing model — one worker per creator, one agent per worker

The current implementation (shipped in V1.4, iterated through V1.5/V1.6) follows this topology:

```
┌──────────────────────────────────────────────────────────────────┐
│                          nexus42d                                │
│                                                                  │
│  ┌───────────────────────────────────────────────────────────┐   │
│  │  WorkerManager                                            │   │
│  │  (broadcast channel only — no HashMap<CreatorId, Handle>) │   │
│  └───────────────────────────┬───────────────────────────────┘   │
│                              │ spawns & returns WorkerHandle      │
│                              │ (but does NOT index by CreatorId)  │
│              ┌───────────────┼───────────────┐                   │
│              ▼               ▼               ▼                   │
│        ┌──────────┐   ┌──────────┐   ┌──────────┐               │
│        │ Worker A │   │ Worker B │   │ Worker C │               │
│        │ creator 1│   │ creator 2│   │ creator 3│               │
│        └──────────┘   └──────────┘   └──────────┘               │
└──────────────────────────────────────────────────────────────────┘
```

**Note:** `nexus42d` already supports running multiple workers for different creators simultaneously (each creator gets its own worker process). However, the current `WorkerManager` (`crates/nexus-orchestration/src/worker/manager.rs`) does **not** maintain a `HashMap<CreatorId, WorkerHandle>` — it only holds a `broadcast::Sender<WorkerEvent>`. Workers are spawned via `spawn()` which returns a `WorkerHandle`, but there is no registry that maps `CreatorId → WorkerHandle`. This means the daemon has no way to look up an existing worker for a given creator, making crash recovery and multi-creator coordination unreliable.

Each individual worker process currently follows this internal topology:

```
┌──────────────────────────────────────┐
│  nexus42 acp-worker --creator <id>   │
│  --agent <agent_ref>                 │
│                                      │
│  ┌──────────────────────┐            │
│  │  WorkerState {       │            │
│  │    creator_id,       │            │
│  │    agent_ref,        │            │
│  │    initialized,      │            │
│  │    shutdown          │            │
│  │  }                   │            │
│  └──────────┬───────────┘            │
│             │ LocalSet               │
│             ▼                        │
│  ┌──────────────────────┐            │
│  │  ACP Agent subprocess│            │
│  │  (single session)    │            │
│  └──────────────────────┘            │
└──────────────────────────────────────┘
```

| File | Role |

### 1.3 Current constraints

- **One agent per worker**: `WorkerState` holds a single `agent_ref` and a single `initialized` flag. Switching agents requires worker restart (`orchestration-engine-v1.md` §6.2).
- **One-shot IPC**: `WorkerHandle::call_json_rpc` consumes stdin/stdout pipe halves on each call (design limitation from WS2). This means only one outstanding request at a time.
- **At-most-one-ACP-Schedule constraint**: Per §9.2 of `orchestration-engine-v1.md`, at most one ACP-calling Schedule may touch the worker at any instant. Capability-only Schedules may run in parallel.
- **Worker process identity**: `WorkerSpec` encodes `--creator <id>` and `--agent <ref>` as command-line arguments. Worker is spawned once per creator.
- **LocalSet constraint**: ACP SDK futures are `!Send`, so each agent connection requires a `tokio::task::LocalSet`. The current worker runs a single LocalSet.
- **No WorkerRegistry**: The daemon's `WorkerManager` has no `HashMap<CreatorId, WorkerHandle>`. Workers are spawned and returned, but the caller must track the handles. This prevents the daemon from looking up a creator's worker for crash recovery, prompt routing, or health checks.

### 1.4 Multi-creator WorkerRegistry (new design)

The daemon must simultaneously manage **multiple running workers for different creators**. This is not a new concept — `orchestration-engine-v1.md` §6.1 states "one long-lived worker subprocess per **active** creator" — but the current `WorkerManager` lacks the index to support it reliably.

**New internal structure:**

```rust
/// Extended WorkerManager with per-creator worker indexing.
pub struct WorkerRegistry {
    /// Broadcast channel for worker lifecycle events.
    event_tx: broadcast::Sender<WorkerEvent>,
    /// Index: CreatorId → WorkerHandle.
    /// One worker per creator (invariant preserved from V1.4).
    workers: HashMap<CreatorId, WorkerHandle>,
    /// Maximum concurrent workers (configurable, default: 16).
    max_workers: usize,
}
```

**WorkerRegistry lifecycle:**

1. **Lazy spawn**: When the engine first needs an ACP session for creator C, the registry checks `workers.get(&C)`. If absent, it spawns a new worker via `WorkerSpec`, stores the handle, and returns it.
2. **Lookup**: Any engine component can call `registry.get(&creator_id)` to obtain the `WorkerHandle` for IPC.
3. **Supervision**: The registry monitors `WorkerEvent::Crashed` and removes the entry, triggering crash recovery for all sessions on that creator's worker.
4. **Graceful shutdown**: On daemon shutdown, the registry iterates all active workers and sends `worker/shutdown` with configurable grace period.
5. **Eviction**: When a creator has no active Schedules, the registry may evict its worker after an idle timeout (configurable, default: 5 minutes). This prevents unbounded resource consumption.

**Why this is needed now:**

- The multi-agent feature (this plan) extends each worker to host multiple ACP agent sessions. The daemon must be able to route prompts to the correct worker for the correct creator, then within that worker to the correct session.
- Crash recovery must scan all sessions for a specific creator's worker, which requires knowing which worker belongs to which creator.
- The existing `WorkerMgrSubsystem` (`crates/nexus42d/src/lifecycle/subsystems/worker_mgr.rs`) is currently a mock stub. WS-E must replace it with real `WorkerRegistry` integration.

**Invariants preserved:**

- One worker process per creator (unchanged from V1.4).
- Multiple creators can have workers running simultaneously (new: explicit registry).
- Each worker can host multiple ACP agent sessions (the core WS-E extension).

## 2. Approach Evaluation

### 2.1 Approach A: Multiplex one worker with session routing

The single worker process for a creator is extended to host multiple ACP agent subprocesses concurrently. IPC requests are routed by `session_id` (which maps 1:1 to an agent subprocess + its `AcpSession`).

```
┌──────────────────────────────────────────────────────────────┐
│                        nexus42d                               │
│  ┌────────────────────────────────┐                           │
│  │  Worker Manager                │                           │
│  │  (tracks sessions per creator) │                           │
│  └──────────────┬─────────────────┘                           │
│                 │ stdin/stdout JSON-RPC 2.0                    │
└─────────────────┼────────────────────────────────────────────┘
                  ▼
┌──────────────────────────────────────────────────────────────┐
│  nexus42 acp-worker --creator <id>                            │
│                                                               │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  MultiplexedWorkerState {                                │ │
│  │    creator_id,                                           │ │
│  │    sessions: HashMap<SessionId, AgentSlot>,              │ │
│  │    active_prompt: Option<(SessionId, RequestId)>,        │ │
│  │  }                                                       │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                               │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐          │
│  │ AgentSlot 1  │ │ AgentSlot 2  │ │ AgentSlot N  │          │
│  │ (LocalSet)   │ │ (LocalSet)   │ │ (LocalSet)   │          │
│  │ ┌──────────┐ │ │ ┌──────────┐ │ │ ┌──────────┐ │          │

│         ▼                ▼                ▼                   │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐              │
│  │Agent proc│     │Agent proc│     │Agent proc│              │
│  │(Claude)  │     │(Codex)   │     │(Gemini)  │              │
│  └──────────┘     └──────────┘     └──────────┘              │
```

### 2.2 Approach B: Spawn sibling workers per agent

Keep the one-agent-per-worker invariant but spawn multiple worker processes for a creator when multiple agents are needed.

```
┌──────────────────────────────────────────────────────────────┐
│                        nexus42d                               │
│  ┌────────────────────────────────────────────────────────┐   │
│  │  Worker Manager                                        │   │
│  │  (tracks workers per creator)                          │   │
│  │  creator_workers: HashMap<CreatorId, Vec<WorkerHandle>>│   │
│  └───────┬───────────────────────────┬────────────────────┘   │
│          │                           │                        │
└──────────┼───────────────────────────┼────────────────────────┘
           ▼                           ▼
┌───────────────────────┐   ┌───────────────────────┐
│ Worker 1              │   │ Worker 2              │
│ --creator <id>        │   │ --creator <id>        │
│ --agent claude-sonnet │   │ --agent codex-agent   │
│                       │   │                       │
│  ┌────────────────┐   │   │  ┌────────────────┐   │
│  │ AcpSession     │   │   │  │ AcpSession     │   │
│  │ (single agent) │   │   │  │ (single agent) │   │
│  └───────┬────────┘   │   │  └───────┬────────┘   │
│          ▼            │   │          ▼            │
│   ┌──────────┐        │   │   ┌──────────┐        │
│   │ Claude   │        │   │   │ Codex    │        │
│   └──────────┘        │   │   └──────────┘        │
└───────────────────────┘   └───────────────────────┘
```

### 2.3 Trade-off Matrix

| Dimension | Approach A (Multiplex) | Approach B (Sibling workers) |
|-----------|----------------------|------------------------------|
| **Process count** | 1 worker process per creator (unchanged) | N worker processes per creator |
| **IPC complexity** | Medium — add `session_id` routing to existing channel; must multiplex request IDs | Low — each worker has its own IPC channel; no protocol change |
| **Resource isolation** | Shared process memory; agent subprocesses are isolated OS processes | Full process isolation between workers |
| **Error isolation** | Agent subprocess crash → only that slot fails; worker process crash → all agents lost. Worker-level code bugs affect all sessions. | Worker crash → only that agent's session lost. Better blast radius. |
| **Compatibility with single-agent mode** | Excellent — single-agent is the degenerate case (1 slot) | Excellent — single-agent is 1 worker (current behavior) |
| **Concurrency contract impact** | Must serialize ACP prompts within worker (1 active prompt at a time across all slots) or manage concurrent LocalSets | Each worker has independent prompt queue; natural per-agent serialization |
| **Memory footprint** | Lower — 1 process, shared runtime | Higher — N processes, each with own tokio runtime + LocalSet |
| **Daemon complexity** | Worker Manager unchanged (1 handle per creator) | Worker Manager must track multiple handles per creator; coordinate prompt routing |
| **LocalSet management** | Each AgentSlot has its own `LocalSet` on a dedicated OS thread | Each worker process has its own `LocalSet` (current behavior) |
| **Backward compatibility** | Requires worker binary change; IPC extension is additive | Minimal worker change; mostly daemon-side |
| **Debugging / observability** | Single process → simpler log aggregation | Multiple processes → more PID tracking |
| **Configuration complexity** | Worker receives agent roster at initialize time | Daemon manages worker-per-agent lifecycle |

### 2.4 Recommendation

**Selected: Approach A — Multiplex one worker with session routing.**

Rationale:

1. **Preserves the one-worker-per-creator invariant** that the engine, supervisor, and scheduling logic depend on. `WorkerRegistry` maps `CreatorId → WorkerHandle` (singular). This mapping underpins crash recovery, session scanning, and the statig lifecycle integration. Changing this to `CreatorId → Vec<WorkerHandle>` would ripple through `ScheduleSupervisor`, `EngineSharedState`, the daemon's `WorkerMgrSubsystem`, and the HTTP session endpoints.

2. **Lower daemon complexity** — the daemon doesn't need to learn how to manage multiple IPC channels per creator. The existing `WorkerHandle` abstraction continues to work with minimal extension.

3. **Resource efficiency** — one process per creator is already the stated design goal. Adding more processes per creator contradicts the architecture's process topology.

4. **Pre-1.0 pragmatism** — Approach A is simpler to implement correctly. The primary risk (shared-process crash taking all sessions) is acceptable in pre-release: we document it and mitigate with session-state persistence (SQLite) so sessions can be recovered on worker restart.

5. **The one-shot IPC limitation** in `WorkerHandle` needs fixing regardless of approach. With Approach A, we fix it once; with Approach B, each sibling worker still has the same limitation independently.

The key downside — all sessions lost on worker crash — is already the case in the current single-agent model. Multi-agent doesn't make this worse; it just increases the number of sessions at risk. The crash recovery path (scan `orchestration_sessions`, mark `paused` with `worker_crash` reason) handles both cases identically.

## 3. IPC Protocol Extension

### 3.1 Design principles

1. **Additive only** — new fields and methods; existing methods continue to work unchanged for backward compatibility.
2. **`session_id` routing** — all agent-specific requests include a `session_id` parameter that routes to the correct `AgentSlot`.
3. **Backward-compatible single-agent mode** — when only one agent is configured, `session_id` defaults to the primary session; old callers don't need to change.

### 3.2 Extended message catalogue (v2)

#### Requests (daemon → worker) — additions and changes

| Method | Params | Response | Change |
|--------|--------|----------|--------|
| `worker/initialize` | `{ creator_id, agents: Vec<AgentConfig>, workspace_root }` | `{ capabilities, worker_pid, sessions: Vec<SessionInfo> }` | **Breaking** — `agent_ref` replaced by `agents` array with `acp_agent_id`/`role`/`model` |
| `worker/agent_start` | `{ session_id, acp_agent_id, role?, model?, workspace_root?, acp_session_id? }` | `{ ok, session_id, agent_info? }` | **New** |
| `worker/agent_stop` | `{ session_id, grace_ms }` | `{ ok }` | **New** |
| `worker/acp_prompt` | `{ session_id, prompt, tool_policy }` | streaming chunks + final `{ done, full_text }` | `session_id` now **required** |
| `worker/acp_cancel` | `{ session_id }` | `{}` | Unchanged |
| `worker/acp_session_load` | `{ session_id, acp_session_id }` | `{ ok, error? }` | Unchanged |
| `worker/agent_list` | `{}` | `{ sessions: Vec<SessionSummary> }` | **New** |
| `worker/health` | `{}` | `{ uptime_ms, sessions: Vec<SessionHealth>, last_error? }` | `acp_session_state` → `sessions` array |
| `worker/shutdown` | `{ grace_ms }` | `{}` | Unchanged |

#### New types

```rust
/// Agent configuration passed at worker initialize.
/// See §7 for the three-concept design (agent / role / model).
///
/// `session_id` is a runtime-generated slot identifier — the daemon
/// creates it when assembling the initialize payload, NOT from preset
/// or user config.  See §7.2 for the preset→runtime mapping.
struct AgentConfig {
    /// Unique session ID for this agent slot (runtime-generated by daemon).
    session_id: String,
    /// Which ACP agent binary to launch (from ACP Registry).
    acp_agent_id: String,
    /// Which role/persona to use (references preset's roles section).
    role: Option<String>,
    /// Which LLM model to use (free-form string, passed through to agent).
    model: Option<String>,
    /// Optional ACP session ID to resume (session/load).
    acp_session_id: Option<String>,
    /// Tool policy override for this agent slot.
    tool_policy: Option<String>,
}

/// Summary of a session slot in the worker.
struct SessionSummary {
    session_id: String,
    acp_agent_id: String,
    role: Option<String>,
    model: Option<String>,
    state: "initializing" | "ready" | "prompting" | "error" | "stopped",
    acp_session_id: Option<String>,
    uptime_ms: u64,
    last_error: Option<String>,
}

/// Per-session health info.
struct SessionHealth {
    session_id: String,
    acp_agent_id: String,
    role: Option<String>,
    model: Option<String>,
    state: String,
    last_error: Option<String>,
}
```

#### Notifications (worker → daemon) — additions

| Method | Params | Change |
|--------|--------|--------|
| `worker/acp_prompt_chunk` | `{ session_id, request_id, text }` | `session_id` field added |
| `worker/agent_tool_request` | `{ session_id, tool_name, args, request_id }` | `session_id` field added |
| `worker/agent_tool_request_result` | `{ request_id, grant, output? }` | Unchanged (request_id links to session) |
| `worker/agent_session_event` | `{ session_id, event, detail? }` | **New** — `event` = `"started" \| "stopped" \| "crashed" \| "error"` |
| `worker/acp_permission_request` | `{ session_id, reason, request_id }` | `session_id` field added |
| `worker/log` | `{ level, message, fields }` | Unchanged |
| `worker/unrecoverable_error` | `{ kind, detail }` | Unchanged |

### 3.3 Backward compatibility

The `worker/initialize` message is the only breaking change (flat `agent_ref` → `agents` array with `acp_agent_id`/`role`/`model`). Since the daemon always starts workers, both sides upgrade together. For the transition:

- Old `worker/initialize { creator_id, agent_ref, ... }` → treated as `agents: [{ session_id: "default", acp_agent_id: "<agent_ref>", role: None, model: None }]`.
- The worker detects the old format by checking for the presence of `agents` vs `agent_ref` in the params JSON.

### 3.4 IPC client upgrade

The current `WorkerHandle::call_json_rpc` consumes stdin/stdout pipe halves on each call (one-shot). This must be replaced with a **persistent multiplexing IPC client** that:

1. Owns the stdin/stdout halves for the worker's lifetime.
2. Maintains a pending-request map (`request_id → oneshot::Sender`).
3. Reads responses in a background task and routes them to waiting callers.
4. Supports concurrent requests (multiple `call_json_rpc` calls in flight).

This is essentially the `IpcClient` mentioned in the existing `ipc.rs` doc comments:

```rust
/// Persistent IPC client with pending-request tracking.
pub struct IpcClient {
    transport: Box<dyn RpcTransport>,
    pending: Arc<Mutex<HashMap<u64, oneshot::Sender<Result<Value, IpcError>>>>>,
    next_id: AtomicU64,
    cancel: CancellationToken,
}
```

## 4. Concurrency Contract Revision

### 4.1 Current contract (§9.2 of orchestration-engine-v1.md)

> "At most one ACP-calling Schedule may touch the worker at any instant."

This means: even if a creator has multiple Schedules, only one can be sending ACP prompts at a time. Capability-only Schedules may run in parallel.

### 4.2 Revised contract

The at-most-one constraint **remains at the Schedule level** but is relaxed at the agent session level:

1. **Per-creator worker constraint** (unchanged): one worker process per creator.
2. **Per-agent prompt serialization** (new): within a worker, each `AgentSlot` serializes ACP prompts (at most one in-flight prompt per slot). This is natural since each slot has its own `LocalSet` and `AcpSession`.
3. **Cross-slot concurrency** (new): multiple agent slots may have prompts in flight **simultaneously**, since each operates on an independent `LocalSet` + subprocess. The IPC client routes responses by `request_id`.
4. **Schedule-level serialization** (unchanged): the engine's `ScheduleSupervisor` still admits at most one ACP-calling Schedule to `Running` at a time. The multi-agent capability enables **different Schedules to use different agents** on the same creator, not parallel ACP Schedules.

**Why this is safe:**

- Each `AgentSlot` has its own `AcpSession` and subprocess. The `!Send` constraint is contained within the slot's `LocalSet`.
- The IPC channel is shared, but NDJSON framing + request-ID matching prevents interleaving.
- The engine already serializes ACP-calling Schedules; multi-agent just allows the single active Schedule to work with multiple agents (e.g., a preset that delegates sub-tasks to different ACP agents).

### 4.3 Engine integration

The `AcpPromptTask` currently takes an `Option<Arc<Mutex<Option<WorkerHandle>>>>` for IPC. This changes to:

```rust
pub struct AcpPromptTask {
    ipc_client: Option<Arc<IpcClient>>,
    session_id: String,  // which agent slot to target
    state_id: String,
    template: String,
    tool_policy: ToolPolicy,
}
```

The engine passes the `session_id` based on the preset's inner-graph node configuration. For presets that don't specify a session, the default (primary) agent slot is used.

## 5. Session Lifecycle

### 5.1 Agent initialization

```
Daemon                        Worker                    Agent
  │                             │                         │
  │── worker/initialize ──────►│                         │
  │   { agents: [              │                         │
  │     { session_id: "s1",    │                         │
  │       acp_agent_id: "claude-acp",│                    │
  │       role: "writer" },    │                         │
  │     { session_id: "s2",    │                         │
  │       acp_agent_id: "codex-acp",│                     │
  │       role: "reviewer" }   │                         │
  │   ] }                      │                         │
  │                             │── spawn agent proc ────►│
  │                             │── ACP initialize ──────►│
  │                             │◄── initialize resp ─────│
  │                             │── spawn agent proc ────►│
  │                             │── ACP initialize ──────►│
  │                             │◄── initialize resp ─────│
  │◄── { sessions: [...] } ────│                         │
  │                             │                         │
```

**Worker behavior on `initialize`:**

1. Parse the `agents` array.
2. For each `AgentConfig`:
   a. Spawn the agent subprocess (via `nexus-acp-host` transport).
   b. Create a `LocalSet` on a dedicated OS thread.
   c. Perform ACP `initialize` handshake via `AcpSdkAdapter`.
   d. Optionally call `session/new` or `session/load` if `acp_session_id` is provided.
   e. Store the `AgentSlot` in `sessions: HashMap<SessionId, AgentSlot>`.
3. Reply with session summaries.

### 5.2 Session creation (dynamic)

After initial `worker/initialize`, the daemon can add agent slots dynamically:

```
Daemon                        Worker                    Agent
  │── worker/agent_start ────►│                         │
  │   { session_id: "s3",     │── spawn agent proc ────►│
  │     acp_agent_id: "gemini",│── ACP initialize ──────►│
  │     role: "researcher" }  │◄── initialize resp ─────│
  │◄── { ok: true,            │                         │
  │      session_id: "s3" } ──│                         │
```

This enables the daemon to add agents on-demand when a new Schedule requires a different agent than those already loaded.

### 5.3 Prompt routing

```
Daemon                        Worker                    Agent (s1)
  │── worker/acp_prompt ─────►│                         │
  │   { session_id: "s1",     │── route to slot "s1" ──►│
  │     prompt: "...",        │── ACP prompt ──────────►│
  │     tool_policy: "..." }  │◄── streaming chunks ────│
  │                           │                         │
  │◄── worker/acp_prompt_chunk│                         │
  │   { session_id: "s1",     │                         │
  │     request_id: 42,       │                         │
  │     text: "chunk1" }      │                         │
  │                           │                         │
  │◄── { done: true,          │                         │
  │      full_text: "..." } ──│                         │
```

The worker's IPC loop:
1. Reads the `session_id` from the request.
2. Looks up the `AgentSlot` in `sessions` HashMap.
3. Dispatches the prompt to that slot's `AcpSession` via its `LocalSet`.
4. Streams chunks back to daemon with the same `session_id`.

### 5.4 Session handoff

When the engine needs to switch which agent handles a state transition (e.g., a preset that uses different agents for different phases):

1. The engine sends `worker/acp_prompt` with the target `session_id`.
2. The previous agent's session remains alive (no need to tear down).
3. If a new agent is needed, the engine sends `worker/agent_start` first.

No explicit "handoff" message is needed — it's implicit in which `session_id` the engine targets.

### 5.5 Session close

```
Daemon                        Worker                    Agent (s1)
  │── worker/agent_stop ─────►│                         │
  │   { session_id: "s1",     │── cancel in-flight ops ─►│
  │     grace_ms: 5000 }      │── ACP cancel ──────────►│
  │                           │── wait for drain ───────►│
  │                           │── kill subprocess ──────►│
  │                           │── remove slot            │
  │◄── { ok: true } ──────────│                         │
```

The worker:
1. Cancels any in-flight prompt on the slot.
2. Sends ACP `cancel` to the agent.
3. Waits up to `grace_ms` for the subprocess to exit.
4. Kills the subprocess if it hasn't exited.
5. Removes the `AgentSlot` from the sessions map.
6. Notifies the daemon via `worker/agent_session_event { session_id, event: "stopped" }`.

### 5.6 Agent crash recovery

When an agent subprocess exits unexpectedly:

1. The worker's supervisor task (per-AgentSlot) detects the exit.
2. The worker sends `worker/agent_session_event { session_id, event: "crashed", detail: "exit code 1" }`.
3. The worker marks the slot as `state: "error"` but does **not** remove it.
4. Other agent slots continue operating normally.
5. The daemon's engine receives the notification and:
   - Marks the affected `orchestration_sessions` row as `paused` with reason `agent_crash`.
   - Other sessions on the same worker are unaffected.

## 6. Error Handling

### 6.1 Worker-level failure

When the entire worker process crashes:

1. Daemon's `WorkerManager` supervisor detects `WorkerEvent::Crashed`.
2. Engine scans `orchestration_sessions` for all sessions associated with this creator's worker.
3. Each session is marked `paused` with reason `worker_crash`.
4. This is identical to the current single-agent crash recovery — the recovery path doesn't change.

### 6.2 Per-agent failure

When a single agent subprocess fails:

1. Worker sends `worker/agent_session_event { event: "crashed" }`.
2. Other agent slots remain operational.
3. Engine marks only the affected orchestration session as `paused` (not the entire worker).
4. The slot remains in `state: "error"` — the daemon can attempt `worker/agent_start` with the same session_id to restart it, or use `worker/agent_stop` to clean it up.

### 6.3 Partial session state recovery

On worker restart (after crash):

1. Worker starts with empty sessions map.
2. Daemon sends `worker/initialize` with agents from the persisted configuration.
3. For each agent that had an active `acp_session_id`, daemon includes it in `AgentConfig::acp_session_id`.
4. Worker calls `session/load` to resume the ACP session.
5. If `session/load` fails (agent doesn't support it or session expired), the slot starts in `state: "error"` and the daemon is notified.

### 6.4 IPC errors

| Scenario | Worker behavior | Daemon behavior |
|----------|----------------|-----------------|
| Request to non-existent `session_id` | Return JSON-RPC error `-32001: session not found` | Engine logs error; marks session `failed` if persistent |
| IPC timeout on `acp_prompt` | Worker keeps running; agent may be slow | Engine retries or marks session `paused` |
| Worker receives unknown method | Return `-32601: Method not found` | Log and skip |
| Duplicate `session_id` in `agent_start` | Return `-32002: session already exists` | Engine uses `agent_stop` first or picks new ID |

## 7. Configuration Model

### 7.0 Design clarification — three distinct concepts

> **User feedback (2026-04-21):** The previous plan conflated `agent_ref` into a single concept that ambiguously meant "which ACP agent to launch" or "which model to use". The user identified three distinct concepts that must be separated.

The ACP configuration for a worker session involves **three independent concepts**:

| Concept | What it controls | Example | Source |
|---------|-----------------|---------|--------|
| **Agent** (`acp_agent_id`) | Which ACP agent binary to launch | `claude-acp`, `codex-acp`, `gemini` | ACP Registry (`registry.json`) |
| **Role** (`role`) | Which prompt template / persona to use in the session | `"writer"`, `"reviewer"`, `"researcher"` | Preset YAML (preset-level role definitions) |
| **Model** (`model`) | Which LLM model the agent uses | `claude-sonnet-4-20250514`, `o3`, `gemini-2.5-pro` | User configuration / CLI flags |

**Why they must be separate:**

1. **Agent ≠ Model**: The ACP Registry entry `claude-acp` (id: `claude-acp`) is the binary wrapper for Anthropic's Claude. The *model* it uses is configured separately (e.g., `claude-sonnet-4-20250514` vs `claude-opus-4-20250514`). Similarly, `codex-acp` is the agent binary for OpenAI's Codex; the model it uses is a separate parameter.
2. **Role is a preset-level concern**: The same agent + model can be used with different prompt templates to produce different behaviors (e.g., Claude as a "writer" vs Claude as a "reviewer"). Roles are defined in the preset, not in the agent configuration.
3. **ACP Registry provides agent discovery**: The registry at `https://cdn.agentclientprotocol.com/registry/v1/latest/registry.json` lists available agents with their `id`, `distribution` (binary/npx/uvx), `cmd`, and `args`. This is how the system knows how to launch a given agent.

**ACP Registry structure (relevant fields):**

```json
{
  "id": "claude-acp",
  "name": "Claude Agent",
  "version": "0.30.0",
  "distribution": {
    "npx": {
      "package": "@agentclientprotocol/claude-agent-acp@0.30.0"
    }
  }
}
```

The registry provides the **launch method** (how to start the agent subprocess). The `acp_agent_id` references an entry in this registry to resolve the binary path / npx package / uvx package at spawn time.

### 7.1 Configuration layer separation

> **User feedback (v3, 2026-04-21):** The `agents` block was removed from preset YAML. Agent/model selection is now split across three layers: preset `recommended_models` (defaults), user-level config file (persistent preferences), and CLI `--agent-ref` (ephemeral overrides).

There are **three configuration layers** with distinct responsibilities:

| Layer | File / Source | Responsibility | Mutability |
|-------|-------------|---------------|------------|
| **Preset** | `preset.yaml` | Role definitions + `recommended_models` (defaults) | Preset author controls |
| **User config** | `~/.nexus42/agents.toml` | Per-strategy agent/model preferences | User edits directly |
| **CLI** | `--agent-ref` flag | Ephemeral overrides for a single run | Per-invocation |

**Config format policy (two-tier):**

Nexus uses a layer-based format split for files under `~/.nexus42/`:

| Tier | Format | Files | Rationale |
|------|--------|-------|-----------|
| **Runtime persistence** | JSON | `auth.json`, `acp/sessions.json` | Machine-generated/read via `serde_json`; program reads and writes; no human editing needed, no comments required. |
| **Human-edited / user-facing config** | TOML | `config.toml`, `agents.toml`, `permissions.toml` (future) | Users write and maintain manually; TOML supports inline comments and has better ergonomics for hand-editing. All user-facing config defaults to TOML. |

- **Migration from `config.json` → `config.toml`** (pre-1.0): On startup, if `config.toml` is absent but `config.json` exists, the CLI/daemon reads the JSON, converts to TOML, writes `config.toml`, and renames the old file to `config.json.migrated`. This is a one-time migration — no backward compatibility concerns per repo's pre-1.0 policy.
- **After migration**, only `auth.json` and `acp/sessions.json` remain as JSON (machine-written, never hand-edited).
- **New config files** (e.g., future permission rules) should default to TOML.

**Key design decisions:**

1. **`session_id` is runtime-only** — never appears in preset or user config. The daemon generates `session_id` values when assembling `worker/initialize` payloads from the resolved configuration.
2. **`acp_agent_id` replaces `agent_id`** — the full name makes it clear this references the ACP Registry agent entry, not a generic "agent".
3. **Preset has no `agents` block** — presets only define `roles` with `recommended_models`. This keeps presets focused on behavioral definitions (what the agent should do) rather than infrastructure choices (which binary to run).
4. **User config bridges preset defaults to actual agent/model** — users override preset recommendations with their own preferences without modifying the preset file.

### 7.2 Preset YAML — roles with recommended_models

Presets define **roles only** (no `agents` block). Each role specifies its behavioral contract and an ordered list of recommended agent+model combinations.

```yaml
preset:
  id: novel-writing
  version: 4

  # Optional: define reusable role/persona templates.
  # Each role is a named prompt template that shapes the agent's behavior.
  roles:
    - id: "writer"
      description: "Creative writer focused on narrative prose"
      system_prompt_file: prompts/roles/writer-system.md
      recommended_models:
        - "claude-acp:claude-sonnet-4-20250514"
        - "codex-acp:o3"

    - id: "reviewer"
      description: "Critical reviewer focused on consistency and quality"
      system_prompt_file: prompts/roles/reviewer-system.md
      recommended_models:
        - "codex-acp:o3"
        - "claude-acp:claude-sonnet-4-20250514"

    - id: "researcher"
      description: "Research-oriented agent for gathering and synthesis"
      system_prompt_file: prompts/roles/researcher-system.md
      recommended_models:
        - "gemini:gemini-2.5-pro"
        - "claude-acp:claude-sonnet-4-20250514"

states:
  - id: gathering
    enter:
      - kind: acp_prompt
        agent: researcher       # references the "researcher" role
        template_file: prompts/gathering.md
        tool_policy: auto_grant_read_only

  - id: brainstorming
    enter:
      - kind: inner_graph
        name: brainstorm_graph

inner_graphs:
  brainstorm_graph:
    nodes:
      - id: diverge
        kind: acp_prompt
        agent: writer            # references the "writer" role
        template_file: prompts/brainstorm-diverge.md
      - id: review
        kind: acp_prompt
        agent: reviewer          # references the "reviewer" role
        template_file: prompts/brainstorm-review.md
        depends_on: [diverge]
```

**`recommended_models` format:** `{acp_agent_id}:{model_name}` — an ordered list where the first entry is the default choice. This format makes it clear which agent binary goes with which model.

**Per-node `agent` field:** References a role ID (not a session ID). At runtime, the daemon resolves each role reference to an `acp_agent_id` + `model` using the priority resolution order (§7.5).

### 7.3 Role definition schema

```yaml
roles:
  - id: <string>                        # Unique role ID within this preset
    description: <string>               # Human-readable description
    system_prompt_file: <path>          # Prompt template for system message
    recommended_models:                 # Ordered list of agent:model pairs
      - "<acp_agent_id>:<model_name>"   # First entry = default
      - "<acp_agent_id>:<model_name>"   # Second entry = fallback
```

**How roles interact with agent sessions:**

1. When a worker needs an agent for a role, it reads the role's `recommended_models[0]` to get the default `acp_agent_id:model`.
2. The user-level config or CLI `--agent-ref` can override this (see §7.5).
3. The role's `system_prompt_file` is resolved relative to the preset bundle root.
4. The system prompt is injected into the ACP session's initial message (via `session/new` or the first prompt's system message).

### 7.4 User-level agent configuration file

**Location:** `~/.nexus42/agents.toml`

This file follows the existing `~/.nexus42/` directory convention (same location as `config.json`, `auth.json`, `acp/sessions.json`). It maps strategy preset IDs to per-role agent/model preferences.

```toml
# ~/.nexus42/agents.toml
#
# User-level agent/model overrides per strategy preset.
# Priority: CLI --agent-ref > this file > preset recommended_models
#
# Format: [strategies."<preset-id>"]
#   <role-id>.agent = "<acp_agent_id>"
#   <role-id>.model = "<model_name>"

[strategies."novel-writing"]
reviewer.agent = "codex-acp"
reviewer.model = "o3"
writer.agent = "claude-acp"
writer.model = "claude-sonnet-4-20250514"

[strategies."code-review"]
reviewer.agent = "codex-acp"
reviewer.model = "o3"
```

**Loading behavior:**

1. File is optional — if absent, only preset `recommended_models` and CLI flags are used.
2. File is loaded once at daemon startup and cached. Changes require daemon restart (or future hot-reload, out of scope for V1.7).
3. Unknown preset IDs or role IDs are ignored silently (allows config to be forward-compatible).
4. Uses TOML for consistency with other configuration in the Rust ecosystem and readability.

**Rust types:**

```rust
/// User-level agent preferences loaded from ~/.nexus42/agents.toml.
#[derive(Debug, Clone, Default, Serialize, Deserialize)]
pub struct UserAgentsConfig {
    /// Map of preset ID → per-role agent/model overrides.
    #[serde(default)]
    pub strategies: HashMap<String, StrategyOverrides>,
}

/// Per-strategy agent/model overrides.
#[derive(Debug, Clone, Default, Serialize, Deserialize)]
pub struct StrategyOverrides {
    /// Map of role ID → agent/model selection.
    /// Serde flatten: TOML keys under [strategies."<id>"] become role IDs.
    #[serde(flatten)]
    pub roles: HashMap<String, RoleOverride>,
}

/// Override for a single role's agent and/or model.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RoleOverride {
    pub agent: Option<String>,
    pub model: Option<String>,
}
```

### 7.5 Priority resolution order

When resolving the effective `acp_agent_id` and `model` for a role, the system uses this priority (highest wins):

1. **CLI `--agent-ref` flag** — `role:acp_agent_id[:model]` format (ephemeral, per-invocation)
2. **User-level config** (`~/.nexus42/agents.toml`) — per-strategy role overrides
3. **Preset `recommended_models`** — first entry in the role's ordered list (preset author's default)

**Resolution algorithm (per role):**

```
fn resolve_agent_model(role_id, preset, user_config, cli_overrides) -> (acp_agent_id, model):
    // 1. CLI override
    if let Some(override) = cli_overrides.get(role_id) {
        return (override.acp_agent_id, override.model)
    }
    
    // 2. User config
    if let Some(strategy_config) = user_config.strategies.get(preset.id) {
        if let Some(role_config) = strategy_config.roles.get(role_id) {
            if role_config.agent.is_some() || role_config.model.is_some() {
                // Merge: user config fills in, recommended_models fills gaps
                let default = preset.roles[role_id].recommended_models[0]
                return (
                    role_config.agent.unwrap_or(default.acp_agent_id),
                    role_config.model.unwrap_or(default.model),
                )
            }
        }
    }
    
    // 3. Preset recommended_models (first entry)
    let default = preset.roles[role_id].recommended_models[0]
    return (default.acp_agent_id, default.model)
```

**`--agent-ref` flag format:** `role:acp_agent_id[:model]`

The flag takes a colon-separated value where the first segment is the **role ID** (matching a role defined in the preset), the second is the `acp_agent_id`, and the optional third segment is the model name.

### 7.6 CLI flags

The `acp-worker` subcommand gains multi-agent flags with three-concept separation:

```
nexus42 acp-worker --creator <id> \
  --agent <acp-agent-id> \                      # primary agent (from ACP Registry)
  [--model <model-name>] \                      # primary model override
  [--role <role-id>] \                          # primary role override
  [--agent-ref <role>:<acp_agent_id>[:<model>]]...
```

Example:
```
nexus42 acp-worker --creator novel-42 \
  --agent claude-acp \
  --model claude-sonnet-4-20250514 \
  --agent-ref reviewer:codex-acp:o3 \
  --agent-ref researcher:gemini:gemini-2.5-pro
```

The `--agent` flag remains the primary/default agent for backward compatibility. `--agent-ref` adds per-role overrides. The colon-separated format specifies the `role` ID, `acp_agent_id`, and optional `model`.

**Backward compatibility:**
- `--agent <acp-agent-id>` without `--agent-ref` → single-agent mode (session_id: "default")
- `--agent-ref` without matching role in preset → error at validation time
- Presets without `roles` section → all nodes use the primary agent from `--agent`

### 7.7 Schedule configuration

When creating a Schedule, the user can specify agent refs:

```
nexus42 schedule add novel-writing --creator <id> \
  --agent claude-acp \
  --model claude-sonnet-4-20250514 \
  --agent-ref reviewer:codex-acp:o3
```

If the user-level config (`~/.nexus42/agents.toml`) has overrides for this strategy, those apply automatically unless overridden by CLI flags.

### 7.8 Model configuration summary

Users configure LLM model names through three layers (highest priority first):

1. **CLI flags** `--model` and `--agent-ref ...:<model>` — ephemeral override
2. **User config** (`~/.nexus42/agents.toml`) — persistent per-strategy preferences
3. **Preset YAML** `roles[].recommended_models[].model` — preset author's recommended default

The model name is a free-form string that the ACP agent interprets. Nexus does not validate model names — it passes them through to the agent. This is intentional: different agents support different models, and the model namespace varies by provider.

### 7.9 Runtime session_id generation

The daemon generates `session_id` values when assembling the `worker/initialize` payload:

1. For each role referenced by a preset's graph nodes, the daemon generates a unique `session_id` (e.g., `"s1"`, `"s2"`, or a UUID).
2. The daemon resolves the `acp_agent_id` and `model` for each role using the priority resolution order (§7.5).
3. The daemon constructs `AgentConfig { session_id, acp_agent_id, role, model, ... }` for each slot.
4. The worker receives these configs and creates `AgentSlot` instances indexed by `session_id`.

This means `session_id` never appears in preset YAML or user config — it is purely a runtime coordination mechanism between daemon and worker.

### 7.10 Default behavior (backward compatibility)

If no multi-agent configuration is provided:

- The daemon generates `agents: [{ session_id: "default", acp_agent_id: "<primary>" }]` from the `--agent` flag.
- All ACP prompts target `session_id: "default"`.
- No role is applied (the agent uses its default behavior).
- This is functionally identical to the current single-agent behavior.
- Existing presets without `roles` sections continue to work unchanged.
- The old `agent_ref` CLI flag continues to work as an alias for `--agent` (mapped to `acp_agent_id` internally).

### 7.11 Interaction with DF-17 (`~/.nexus42/strategies/`)

The user-level config file (`~/.nexus42/agents.toml`) uses preset IDs as strategy keys. When DF-17 third-party preset loading is implemented:

- Third-party presets installed under `~/.nexus42/strategies/<name>/` will have their preset IDs available as keys in `agents.toml`.
- Users can add overrides for any installed strategy without modifying the preset itself.
- The `recommended_models` in third-party presets provide defaults; users customize via `agents.toml` or CLI.
- This design is forward-compatible: adding DF-17 does not require changes to the agent configuration resolution logic.

### 7.12 Redesigned AgentConfig (IPC protocol type)

The `AgentConfig` struct used in the `worker/initialize` IPC message. Note: this is the **runtime type** assembled by the daemon, not a config file type.

```rust
/// Configuration for one agent slot in a multi-agent worker.
/// Assembled by the daemon from preset roles + user config + CLI flags.
/// Passed to worker via worker/initialize IPC.
pub struct AgentConfig {
    /// Unique session ID for this agent slot (runtime-generated by daemon).
    pub session_id: String,

    /// Which ACP agent to launch — references an agent ID from the ACP Registry.
    /// Example: "claude-acp", "codex-acp", "gemini".
    pub acp_agent_id: String,

    /// Which role/persona to use when starting the ACP session.
    /// References a role defined in the preset's `roles` section.
    pub role: Option<String>,

    /// Which LLM model to use.
    /// Example: "claude-sonnet-4-20250514", "o3", "gemini-2.5-pro".
    pub model: Option<String>,

    /// Optional ACP session ID to resume (session/load).
    pub acp_session_id: Option<String>,

    /// Tool policy override for this agent slot.
    pub tool_policy: Option<String>,
}
```

### 7.13 Role/persona feature scope recommendation

**Recommendation: Roles are a WS-E concern, not a separate workstream.**

The user stated "需要先实现" (should be implemented first), but the role/persona feature is tightly coupled with the multi-agent worker architecture for the following reasons:

1. **Roles only make sense with multi-agent**: A single-agent worker has no need for role switching — the system prompt is part of the preset's regular prompt template flow. Roles become meaningful when multiple agent slots can each have their own persona.

2. **Roles are a configuration-layer concern**: The role definition is a YAML schema extension to the preset format. It does not require a new Rust crate, new IPC protocol, or new engine integration — it extends the same `AgentConfig` → `AgentSlot` → `AcpPromptTask` pipeline that WS-E already builds.

3. **Minimal additional implementation cost**: The core role feature is: read `system_prompt_file` from the preset, inject it as the first message in the ACP session. This is ~S effort on top of the multi-agent infrastructure.

4. **Implementation order**: The role's `system_prompt_file` should be injected when the worker calls `session/new` (or equivalent) for an agent slot. This happens inside T2 (AgentSlot abstraction), making it a natural extension point.

**What stays out of WS-E scope:**

- **Role marketplace / sharing** — roles are preset-local, not globally shared
- **Dynamic role switching mid-session** — roles are set at session creation and are immutable for the session's lifetime
- **Role-level tool policy overrides** — tool policies remain per-prompt, not per-role (the preset's per-node `tool_policy` already handles this)

If the PM determines that the role feature significantly increases WS-E's scope, it can be split into a follow-up task within WS-E (e.g., T8: Role system prompt injection) rather than a separate workstream.

## 10. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Multiple LocalSets on OS threads cause resource contention | Low | Medium | Each LocalSet runs on its own OS thread with bounded task queue. Limit max agents per worker (configurable, default: 4). |
| IPC response routing mixes up concurrent requests | Low | High | `IpcClient` uses unique numeric request IDs with pending-map routing. Integration test verifies 10+ concurrent requests. |
| Agent subprocess crash causes worker panic | Medium | High | Each `AgentSlot` catches subprocess exits in its supervisor. The worker's main loop is isolated from per-slot panics via `std::thread` + catch_unwind. |
| Breaking `worker/initialize` format breaks existing deployments | Low | Medium | Worker binary and daemon upgrade together. Old-format detection provides fallback path. |
| Memory usage scales linearly with agent count | Medium | Low | Each agent subprocess is an external process (its own memory space). Worker's memory overhead per slot is minimal (~1 LocalSet + 2 channels). Default max 4 agents. |
| Preset YAML `roles` with `recommended_models` complexity confuses users | Low | Low | Both sections are optional. Single-agent presets don't need them. Clear validation error messages. Roles section can be omitted for single-agent presets. |
| V1.7 WS-D (ACP SDK v0.11.x) introduces type changes affecting AgentSlot | Medium | Medium | AgentSlot uses `NexusAcpClient` trait (V1.6 DTO boundary). SDK changes are confined to `AcpSdkAdapter`. |
| `session/list` from ACP SDK v0.11.x doesn't work as expected | Low | Low | `session/list` is used for session discovery, not critical path. Worker can operate without it (session tracking is internal). |
| WorkerRegistry unbounded growth if creators are never evicted | Medium | Medium | Idle eviction task with configurable timeout (default: 5 minutes). Max workers cap (default: 16). Monitoring via health endpoint. |
| Role `system_prompt_file` path traversal security risk | Low | Medium | Path must be relative to preset bundle root; loader rejects absolute paths and `..` traversal. Validates at load time (aligns with DF-09 requirements for V1.8+). |
| ACP Registry acp_agent_id doesn't match user's expected naming | Low | Low | The registry uses stable IDs (e.g., `claude-acp`, `codex-acp`, `gemini`). Users can discover available agents via `nexus42 agent list`. The `acp_agent_id` field references these registry IDs, not model names. |
