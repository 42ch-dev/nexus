# Multi-Agent Worker — Design Specification & Implementation Plan (WS-E, DF-01, OQ-7)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extend the one-worker-per-creator MVP (V1.4) to support a single worker hosting >1 ACP agent concurrently, enabling multiple ACP sessions within one worker process.

**Architecture:** Selected approach is **Approach A — multiplex one worker with session routing**: the worker process hosts multiple ACP agent subprocesses (each with its own `AcpSession`), routing IPC requests by `session_id`. The daemon's Worker Manager tracks per-session state. This preserves the one-worker-per-creator invariant while extending the worker internals to multiplex agents.

**Tech Stack:** Rust, `agent-client-protocol` v0.11.x, `tokio` (LocalSet per agent), JSON-RPC 2.0 over stdin/stdout, SQLite for session persistence.

**V1.7 Compass Alignment:** WS-E → Milestone M3. Depends on WS-D (ACP SDK v0.11.0+ for `session/list`).

## 1. Current State Analysis

### 1.1 Existing one-worker-per-creator model

The current implementation (shipped in V1.4, iterated through V1.5/V1.6) follows this topology:

```
┌──────────────────────────────────────┐

│            │ stdin/stdout             │
│            │ JSON-RPC 2.0             │
└────────────┼─────────────────────────┘
             ▼
┌──────────────────────────────────────┐
│  nexus42 acp-worker --creator <id>   │
│  --agent <agent_ref>                 │
│                                      │
│  ┌──────────────────────┐            │
│  │  WorkerState {       │            │
│  │    creator_id,       │            │
│  │    agent_ref,        │            │
│  │    initialized,      │            │
│  │    shutdown          │            │
│  │  }                   │            │
│  └──────────┬───────────┘            │
```

### 1.3 Current constraints

- **One agent per worker**: `WorkerState` holds a single `agent_ref` and a single `initialized` flag. Switching agents requires worker restart (`orchestration-engine-v1.md` §6.2).
- **One-shot IPC**: `WorkerHandle::call_json_rpc` consumes stdin/stdout pipe halves on each call (design limitation from WS2). This means only one outstanding request at a time.
- **At-most-one-ACP-Schedule constraint**: Per §9.2 of `orchestration-engine-v1.md`, at most one ACP-calling Schedule may touch the worker at any instant. Capability-only Schedules may run in parallel.
- **Worker process identity**: `WorkerSpec` encodes `--creator <id>` and `--agent <ref>` as command-line arguments. Worker is spawned once per creator.
- **LocalSet constraint**: ACP SDK futures are `!Send`, so each agent connection requires a `tokio::task::LocalSet`. The current worker runs a single LocalSet.

## 2. Approach Evaluation

### 2.1 Approach A: Multiplex one worker with session routing

The single worker process for a creator is extended to host multiple ACP agent subprocesses concurrently. IPC requests are routed by `session_id` (which maps 1:1 to an agent subprocess + its `AcpSession`).

```
┌──────────────────────────────────────────────────────────────┐
│                        nexus42d                               │
│  ┌────────────────────────────────┐                           │
│  │  Worker Manager                │                           │
│  │  (tracks sessions per creator) │                           │
│  └──────────────┬─────────────────┘                           │
│                 │ stdin/stdout JSON-RPC 2.0                    │
└─────────────────┼────────────────────────────────────────────┘
                  ▼
┌──────────────────────────────────────────────────────────────┐
│  nexus42 acp-worker --creator <id>                            │
│                                                               │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  MultiplexedWorkerState {                                │ │
│  │    creator_id,                                           │ │
│  │    sessions: HashMap<SessionId, AgentSlot>,              │ │
│  │    active_prompt: Option<(SessionId, RequestId)>,        │ │
│  │  }                                                       │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                               │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐          │
│  │ AgentSlot 1  │ │ AgentSlot 2  │ │ AgentSlot N  │          │
│  │ (LocalSet)   │ │ (LocalSet)   │ │ (LocalSet)   │          │
│  │ ┌──────────┐ │ │ ┌──────────┐ │ │ ┌──────────┐ │          │
│  │ │AcpSession│ │ │ │AcpSession│ │ │ │AcpSession│ │          │
│  │ └────┬─────┘ │ │ └────┬─────┘ │ │ └────┬─────┘ │          │
│  └──────┼───────┘ └──────┼───────┘ └──────┼───────┘          │
│         ▼                ▼                ▼                   │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐              │
│  │Agent proc│     │Agent proc│     │Agent proc│              │
│  │(Claude)  │     │(Codex)   │     │(Gemini)  │              │
│  └──────────┘     └──────────┘     └──────────┘              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 Approach B: Spawn sibling workers per agent

Keep the one-agent-per-worker invariant but spawn multiple worker processes for a creator when multiple agents are needed.

```
┌──────────────────────────────────────────────────────────────┐
│                        nexus42d                               │
│  ┌────────────────────────────────────────────────────────┐   │
│  │  Worker Manager                                        │   │
│  │  (tracks workers per creator)                          │   │
│  │  creator_workers: HashMap<CreatorId, Vec<WorkerHandle>>│   │
│  └───────┬───────────────────────────┬────────────────────┘   │
│          │                           │                        │
└──────────┼───────────────────────────┼────────────────────────┘
           ▼                           ▼
┌───────────────────────┐   ┌───────────────────────┐
│ Worker 1              │   │ Worker 2              │
│ --creator <id>        │   │ --creator <id>        │
│ --agent claude-sonnet │   │ --agent codex-agent   │
│                       │   │                       │
│  ┌────────────────┐   │   │  ┌────────────────┐   │
│  │ AcpSession     │   │   │  │ AcpSession     │   │
│  │ (single agent) │   │   │  │ (single agent) │   │
│  └───────┬────────┘   │   │  └───────┬────────┘   │
│          ▼            │   │          ▼            │
│   ┌──────────┐        │   │   ┌──────────┐        │
│   │ Claude   │        │   │   │ Codex    │        │
│   └──────────┘        │   │   └──────────┘        │
└───────────────────────┘   └───────────────────────┘
```

### 2.3 Trade-off Matrix

| Dimension | Approach A (Multiplex) | Approach B (Sibling workers) |
|-----------|----------------------|------------------------------|
| **Process count** | 1 worker process per creator (unchanged) | N worker processes per creator |
| **IPC complexity** | Medium — add `session_id` routing to existing channel; must multiplex request IDs | Low — each worker has its own IPC channel; no protocol change |
| **Resource isolation** | Shared process memory; agent subprocesses are isolated OS processes | Full process isolation between workers |
| **Error isolation** | Agent subprocess crash → only that slot fails; worker process crash → all agents lost. Worker-level code bugs affect all sessions. | Worker crash → only that agent's session lost. Better blast radius. |
| **Compatibility with single-agent mode** | Excellent — single-agent is the degenerate case (1 slot) | Excellent — single-agent is 1 worker (current behavior) |
| **Concurrency contract impact** | Must serialize ACP prompts within worker (1 active prompt at a time across all slots) or manage concurrent LocalSets | Each worker has independent prompt queue; natural per-agent serialization |
| **Memory footprint** | Lower — 1 process, shared runtime | Higher — N processes, each with own tokio runtime + LocalSet |
| **Daemon complexity** | Worker Manager unchanged (1 handle per creator) | Worker Manager must track multiple handles per creator; coordinate prompt routing |
| **LocalSet management** | Each AgentSlot has its own `LocalSet` on a dedicated OS thread | Each worker process has its own `LocalSet` (current behavior) |
| **Backward compatibility** | Requires worker binary change; IPC extension is additive | Minimal worker change; mostly daemon-side |
| **Debugging / observability** | Single process → simpler log aggregation | Multiple processes → more PID tracking |
| **Configuration complexity** | Worker receives agent roster at initialize time | Daemon manages worker-per-agent lifecycle |

### 2.4 Recommendation

**Selected: Approach A — Multiplex one worker with session routing.**

Rationale:

1. **Preserves the one-worker-per-creator invariant** that the engine, supervisor, and scheduling logic depend on. `WorkerManager` maps `CreatorId → WorkerHandle` (singular). This mapping underpins crash recovery, session scanning, and the statig lifecycle integration. Changing this to `CreatorId → Vec<WorkerHandle>` would ripple through `ScheduleSupervisor`, `EngineSharedState`, the daemon's `WorkerMgrSubsystem`, and the HTTP session endpoints.

2. **Lower daemon complexity** — the daemon doesn't need to learn how to manage multiple IPC channels per creator. The existing `WorkerHandle` abstraction continues to work with minimal extension.

3. **Resource efficiency** — one process per creator is already the stated design goal. Adding more processes per creator contradicts the architecture's process topology.

4. **Pre-1.0 pragmatism** — Approach A is simpler to implement correctly. The primary risk (shared-process crash taking all sessions) is acceptable in pre-release: we document it and mitigate with session-state persistence (SQLite) so sessions can be recovered on worker restart.

5. **The one-shot IPC limitation** in `WorkerHandle` needs fixing regardless of approach. With Approach A, we fix it once; with Approach B, each sibling worker still has the same limitation independently.

The key downside — all sessions lost on worker crash — is already the case in the current single-agent model. Multi-agent doesn't make this worse; it just increases the number of sessions at risk. The crash recovery path (scan `orchestration_sessions`, mark `paused` with `worker_crash` reason) handles both cases identically.

## 3. IPC Protocol Extension

### 3.1 Design principles

1. **Additive only** — new fields and methods; existing methods continue to work unchanged for backward compatibility.
2. **`session_id` routing** — all agent-specific requests include a `session_id` parameter that routes to the correct `AgentSlot`.
3. **Backward-compatible single-agent mode** — when only one agent is configured, `session_id` defaults to the primary session; old callers don't need to change.

### 3.2 Extended message catalogue (v2)

#### Requests (daemon → worker) — additions and changes

| Method | Params | Response | Change |
|--------|--------|----------|--------|
| `worker/initialize` | `{ creator_id, agents: Vec<AgentConfig>, workspace_root }` | `{ capabilities, worker_pid, sessions: Vec<SessionInfo> }` | **Breaking** — `agent_ref` replaced by `agents` array |
| `worker/agent_start` | `{ session_id, agent_ref, workspace_root?, acp_session_id? }` | `{ ok, session_id, agent_info? }` | **New** |
| `worker/agent_stop` | `{ session_id, grace_ms }` | `{ ok }` | **New** |
| `worker/acp_prompt` | `{ session_id, prompt, tool_policy }` | streaming chunks + final `{ done, full_text }` | `session_id` now **required** |
| `worker/acp_cancel` | `{ session_id }` | `{}` | Unchanged |
| `worker/acp_session_load` | `{ session_id, acp_session_id }` | `{ ok, error? }` | Unchanged |
| `worker/agent_list` | `{}` | `{ sessions: Vec<SessionSummary> }` | **New** |
| `worker/health` | `{}` | `{ uptime_ms, sessions: Vec<SessionHealth>, last_error? }` | `acp_session_state` → `sessions` array |
| `worker/shutdown` | `{ grace_ms }` | `{}` | Unchanged |

#### New types

```rust
/// Agent configuration passed at worker initialize.
struct AgentConfig {
    /// Unique session ID for this agent slot (generated by daemon).
    session_id: String,
    /// Agent reference (e.g. "claude-sonnet-4-20250514").
    agent_ref: String,
    /// Optional ACP session ID to resume (session/load).
    acp_session_id: Option<String>,
    /// Tool policy override for this agent slot.
    tool_policy: Option<String>,
}

/// Summary of a session slot in the worker.
struct SessionSummary {
    session_id: String,
    agent_ref: String,
    state: "initializing" | "ready" | "prompting" | "error" | "stopped",
    acp_session_id: Option<String>,
    uptime_ms: u64,
    last_error: Option<String>,
}

/// Per-session health info.
struct SessionHealth {
    session_id: String,
    agent_ref: String,
    state: String,
    last_error: Option<String>,
}
```

#### Notifications (worker → daemon) — additions

| Method | Params | Change |
|--------|--------|--------|
| `worker/acp_prompt_chunk` | `{ session_id, request_id, text }` | `session_id` field added |
| `worker/agent_tool_request` | `{ session_id, tool_name, args, request_id }` | `session_id` field added |
| `worker/agent_tool_request_result` | `{ request_id, grant, output? }` | Unchanged (request_id links to session) |
| `worker/agent_session_event` | `{ session_id, event, detail? }` | **New** — `event` = `"started" \| "stopped" \| "crashed" \| "error"` |
| `worker/acp_permission_request` | `{ session_id, reason, request_id }` | `session_id` field added |
| `worker/log` | `{ level, message, fields }` | Unchanged |
| `worker/unrecoverable_error` | `{ kind, detail }` | Unchanged |

### 3.3 Backward compatibility

The `worker/initialize` message is the only breaking change (flat `agent_ref` → `agents` array). Since the daemon always starts workers, both sides upgrade together. For the transition:

- Old `worker/initialize { creator_id, agent_ref, ... }` → treated as `agents: [{ session_id: "default", agent_ref }]`.
- The worker detects the old format by checking for the presence of `agents` vs `agent_ref` in the params JSON.

### 3.4 IPC client upgrade

The current `WorkerHandle::call_json_rpc` consumes stdin/stdout pipe halves on each call (one-shot). This must be replaced with a **persistent multiplexing IPC client** that:

1. Owns the stdin/stdout halves for the worker's lifetime.
2. Maintains a pending-request map (`request_id → oneshot::Sender`).
3. Reads responses in a background task and routes them to waiting callers.
4. Supports concurrent requests (multiple `call_json_rpc` calls in flight).

This is essentially the `IpcClient` mentioned in the existing `ipc.rs` doc comments:

```rust
/// Persistent IPC client with pending-request tracking.
pub struct IpcClient {
    transport: Box<dyn RpcTransport>,
    pending: Arc<Mutex<HashMap<u64, oneshot::Sender<Result<Value, IpcError>>>>>,
    next_id: AtomicU64,
    cancel: CancellationToken,
}
```

## 4. Concurrency Contract Revision

### 4.1 Current contract (§9.2 of orchestration-engine-v1.md)

> "At most one ACP-calling Schedule may touch the worker at any instant."

This means: even if a creator has multiple Schedules, only one can be sending ACP prompts at a time. Capability-only Schedules may run in parallel.

### 4.2 Revised contract

The at-most-one constraint **remains at the Schedule level** but is relaxed at the agent session level:

1. **Per-creator worker constraint** (unchanged): one worker process per creator.
2. **Per-agent prompt serialization** (new): within a worker, each `AgentSlot` serializes ACP prompts (at most one in-flight prompt per slot). This is natural since each slot has its own `LocalSet` and `AcpSession`.
3. **Cross-slot concurrency** (new): multiple agent slots may have prompts in flight **simultaneously**, since each operates on an independent `LocalSet` + subprocess. The IPC client routes responses by `request_id`.
4. **Schedule-level serialization** (unchanged): the engine's `ScheduleSupervisor` still admits at most one ACP-calling Schedule to `Running` at a time. The multi-agent capability enables **different Schedules to use different agents** on the same creator, not parallel ACP Schedules.

**Why this is safe:**

- Each `AgentSlot` has its own `AcpSession` and subprocess. The `!Send` constraint is contained within the slot's `LocalSet`.
- The IPC channel is shared, but NDJSON framing + request-ID matching prevents interleaving.
- The engine already serializes ACP-calling Schedules; multi-agent just allows the single active Schedule to work with multiple agents (e.g., a preset that delegates sub-tasks to different ACP agents).

### 4.3 Engine integration

The `AcpPromptTask` currently takes an `Option<Arc<Mutex<Option<WorkerHandle>>>>` for IPC. This changes to:

```rust
pub struct AcpPromptTask {
    ipc_client: Option<Arc<IpcClient>>,
    session_id: String,  // which agent slot to target
    state_id: String,
    template: String,
    tool_policy: ToolPolicy,
}
```

The engine passes the `session_id` based on the preset's inner-graph node configuration. For presets that don't specify a session, the default (primary) agent slot is used.

## 5. Session Lifecycle

### 5.1 Agent initialization

```
Daemon                        Worker                    Agent
  │                             │                         │
  │── worker/initialize ──────►│                         │
  │   { agents: [              │                         │
  │     { session_id: "s1",    │                         │
  │       agent_ref: "claude" },│                         │
  │     { session_id: "s2",    │                         │
  │       agent_ref: "codex" } │                         │
  │   ] }                      │                         │
  │                             │── spawn agent proc ────►│
  │                             │── ACP initialize ──────►│
  │                             │◄── initialize resp ─────│
  │                             │── spawn agent proc ────►│
  │                             │── ACP initialize ──────►│
  │                             │◄── initialize resp ─────│
  │◄── { sessions: [...] } ────│                         │
  │                             │                         │
```

**Worker behavior on `initialize`:**

1. Parse the `agents` array.
2. For each `AgentConfig`:
   a. Spawn the agent subprocess (via `nexus-acp-host` transport).
   b. Create a `LocalSet` on a dedicated OS thread.
   c. Perform ACP `initialize` handshake via `AcpSdkAdapter`.
   d. Optionally call `session/new` or `session/load` if `acp_session_id` is provided.
   e. Store the `AgentSlot` in `sessions: HashMap<SessionId, AgentSlot>`.
3. Reply with session summaries.

### 5.2 Session creation (dynamic)

After initial `worker/initialize`, the daemon can add agent slots dynamically:

```
Daemon                        Worker                    Agent
  │── worker/agent_start ────►│                         │
  │   { session_id: "s3",     │── spawn agent proc ────►│
  │     agent_ref: "gemini" } │── ACP initialize ──────►│
  │                           │◄── initialize resp ─────│
  │◄── { ok: true,            │                         │
  │      session_id: "s3" } ──│                         │
```

This enables the daemon to add agents on-demand when a new Schedule requires a different agent than those already loaded.

### 5.3 Prompt routing

```
Daemon                        Worker                    Agent (s1)
  │── worker/acp_prompt ─────►│                         │
  │   { session_id: "s1",     │── route to slot "s1" ──►│
  │     prompt: "...",        │── ACP prompt ──────────►│
  │     tool_policy: "..." }  │◄── streaming chunks ────│
  │                           │                         │
  │◄── worker/acp_prompt_chunk│                         │
  │   { session_id: "s1",     │                         │
  │     request_id: 42,       │                         │
  │     text: "chunk1" }      │                         │
  │                           │                         │
  │◄── { done: true,          │                         │
  │      full_text: "..." } ──│                         │
```

The worker's IPC loop:
1. Reads the `session_id` from the request.
2. Looks up the `AgentSlot` in `sessions` HashMap.
3. Dispatches the prompt to that slot's `AcpSession` via its `LocalSet`.
4. Streams chunks back to daemon with the same `session_id`.

### 5.4 Session handoff

When the engine needs to switch which agent handles a state transition (e.g., a preset that uses different agents for different phases):

1. The engine sends `worker/acp_prompt` with the target `session_id`.
2. The previous agent's session remains alive (no need to tear down).
3. If a new agent is needed, the engine sends `worker/agent_start` first.

No explicit "handoff" message is needed — it's implicit in which `session_id` the engine targets.

### 5.5 Session close

```
Daemon                        Worker                    Agent (s1)
  │── worker/agent_stop ─────►│                         │
  │   { session_id: "s1",     │── cancel in-flight ops ─►│
  │     grace_ms: 5000 }      │── ACP cancel ──────────►│
  │                           │── wait for drain ───────►│
  │                           │── kill subprocess ──────►│
  │                           │── remove slot            │
  │◄── { ok: true } ──────────│                         │
```

The worker:
1. Cancels any in-flight prompt on the slot.
2. Sends ACP `cancel` to the agent.
3. Waits up to `grace_ms` for the subprocess to exit.
4. Kills the subprocess if it hasn't exited.
5. Removes the `AgentSlot` from the sessions map.
6. Notifies the daemon via `worker/agent_session_event { session_id, event: "stopped" }`.

### 5.6 Agent crash recovery

When an agent subprocess exits unexpectedly:

1. The worker's supervisor task (per-AgentSlot) detects the exit.
2. The worker sends `worker/agent_session_event { session_id, event: "crashed", detail: "exit code 1" }`.
3. The worker marks the slot as `state: "error"` but does **not** remove it.
4. Other agent slots continue operating normally.
5. The daemon's engine receives the notification and:
   - Marks the affected `orchestration_sessions` row as `paused` with reason `agent_crash`.
   - Other sessions on the same worker are unaffected.

## 6. Error Handling

### 6.1 Worker-level failure

When the entire worker process crashes:

1. Daemon's `WorkerManager` supervisor detects `WorkerEvent::Crashed`.
2. Engine scans `orchestration_sessions` for all sessions associated with this creator's worker.
3. Each session is marked `paused` with reason `worker_crash`.
4. This is identical to the current single-agent crash recovery — the recovery path doesn't change.

### 6.2 Per-agent failure

When a single agent subprocess fails:

1. Worker sends `worker/agent_session_event { event: "crashed" }`.
2. Other agent slots remain operational.
3. Engine marks only the affected orchestration session as `paused` (not the entire worker).
4. The slot remains in `state: "error"` — the daemon can attempt `worker/agent_start` with the same session_id to restart it, or use `worker/agent_stop` to clean it up.

### 6.3 Partial session state recovery

On worker restart (after crash):

1. Worker starts with empty sessions map.
2. Daemon sends `worker/initialize` with agents from the persisted configuration.
3. For each agent that had an active `acp_session_id`, daemon includes it in `AgentConfig::acp_session_id`.
4. Worker calls `session/load` to resume the ACP session.
5. If `session/load` fails (agent doesn't support it or session expired), the slot starts in `state: "error"` and the daemon is notified.

### 6.4 IPC errors

| Scenario | Worker behavior | Daemon behavior |
|----------|----------------|-----------------|
| Request to non-existent `session_id` | Return JSON-RPC error `-32001: session not found` | Engine logs error; marks session `failed` if persistent |
| IPC timeout on `acp_prompt` | Worker keeps running; agent may be slow | Engine retries or marks session `paused` |
| Worker receives unknown method | Return `-32601: Method not found` | Log and skip |
| Duplicate `session_id` in `agent_start` | Return `-32002: session already exists` | Engine uses `agent_stop` first or picks new ID |

## 7. Configuration Model

### 7.1 Preset YAML extension

Preset YAML gains an optional `agents` field at the top level and per-node `agent` overrides:

```yaml
preset:
  id: novel-writing
  version: 2
  # ... existing fields ...

  # Optional: declare which agents this preset uses.
  # If absent, all nodes use the default agent (session_id: "default").
  agents:
    - session_id: "writer"
      agent_ref: "claude-sonnet-4-20250514"
      default: true          # used when node doesn't specify
    - session_id: "reviewer"
      agent_ref: "o3"
    - session_id: "researcher"
      agent_ref: "gemini-2.5-pro"

states:
  - id: gathering
    enter:
      - kind: acp_prompt
        agent: researcher       # uses the "researcher" slot
        template_file: prompts/gathering.md
        tool_policy: auto_grant_read_only
    # ...

  - id: brainstorming
    enter:
      - kind: inner_graph
        name: brainstorm_graph
    # ...

inner_graphs:
  brainstorm_graph:
    nodes:
      - id: diverge
        kind: acp_prompt
        agent: writer            # uses the "writer" slot
        template_file: prompts/brainstorm-diverge.md
      - id: review
        kind: acp_prompt
        agent: reviewer          # uses the "reviewer" slot
        template_file: prompts/brainstorm-review.md
        depends_on: [diverge]
```

### 7.2 CLI flags

The `acp-worker` subcommand gains a multi-agent flag:

```
nexus42 acp-worker --creator <id> --agent <primary-agent> [--agent-extra <session_id>:<agent_ref>]...
```

Example:
```
nexus42 acp-worker --creator novel-42 \
  --agent claude-sonnet-4-20250514 \
  --agent-extra reviewer:o3 \
  --agent-extra researcher:gemini-2.5-pro
```

The `--agent` flag remains the primary/default agent for backward compatibility. `--agent-extra` adds additional named slots.

### 7.3 Schedule configuration

When creating a Schedule, the user can specify agents:

```
nexus42 schedule add novel-writing --creator <id> \
  --agent claude-sonnet-4-20250514 \
  --agent-extra reviewer:o3
```

If the preset declares its own `agents` section, those take precedence (the CLI flags are for presets that don't declare agents, or to override the preset's defaults).

### 7.4 Default behavior (backward compatibility)

If no multi-agent configuration is provided:

- `worker/initialize` receives `agents: [{ session_id: "default", agent_ref: "<primary>" }]`.
- All ACP prompts target `session_id: "default"`.
- This is functionally identical to the current single-agent behavior.
- Existing presets without `agents` sections continue to work unchanged.

## 10. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Multiple LocalSets on OS threads cause resource contention | Low | Medium | Each LocalSet runs on its own OS thread with bounded task queue. Limit max agents per worker (configurable, default: 4). |
| IPC response routing mixes up concurrent requests | Low | High | `IpcClient` uses unique numeric request IDs with pending-map routing. Integration test verifies 10+ concurrent requests. |
| Agent subprocess crash causes worker panic | Medium | High | Each `AgentSlot` catches subprocess exits in its supervisor. The worker's main loop is isolated from per-slot panics via `std::thread` + catch_unwind. |
| Breaking `worker/initialize` format breaks existing deployments | Low | Medium | Worker binary and daemon upgrade together. Old-format detection provides fallback path. |
| Memory usage scales linearly with agent count | Medium | Low | Each agent subprocess is an external process (its own memory space). Worker's memory overhead per slot is minimal (~1 LocalSet + 2 channels). Default max 4 agents. |
| Preset YAML `agents` section complexity confuses users | Low | Low | `agents` section is optional. Single-agent presets don't need it. Clear validation error messages. |
| V1.7 WS-D (ACP SDK v0.11.x) introduces type changes affecting AgentSlot | Medium | Medium | AgentSlot uses `NexusAcpClient` trait (V1.6 DTO boundary). SDK changes are confined to `AcpSdkAdapter`. |
| `session/list` from ACP SDK v0.11.x doesn't work as expected | Low | Low | `session/list` is used for session discovery, not critical path. Worker can operate without it (session tracking is internal). |
