#### DTO-C1 — Propagate `mcp_servers` from `NexusNewSessionRequest` to SDK's `NewSessionRequest`

**Severity**: medium
**Source**: V1.6 WS-B QC

**Current code state**:

- `crates/nexus-acp-host/src/client.rs` lines 150–152:
  ```rust
  fn sdk_new_session_request_from_nexus(req: NexusNewSessionRequest) -> acp::NewSessionRequest {
      acp::NewSessionRequest::new(req.cwd)
  }
  ```
  The `req.mcp_servers` field is **discarded** — never mapped to the SDK's `NewSessionRequest`.

- `crates/nexus-contracts/src/local/acp/types.rs` lines 206–254: `NexusMcpServer`, `NexusMcpServerHttp`, `NexusMcpServerSse`, `NexusMcpServerStdio` are already defined with correct structure.

- SDK `agent-client-protocol` v0.10.4 (via `agent-client-protocol-schema` v0.11.4): `NewSessionRequest` has `pub mcp_servers: Vec<McpServer>` field and a builder-style `.mcp_servers()` method. The SDK types are `McpServer::Http(Http)`, `McpServer::Sse(Sse)`, `McpServer::Stdio(Stdio)`.

**Fix strategy**:

1. Add a conversion function `nexus_mcp_server_to_sdk(server: NexusMcpServer) -> acp::McpServer` in `client.rs` alongside the existing conversion helpers. Map each variant:
   - `NexusMcpServer::Http(NexusMcpServerHttp { name, url })` → `acp::McpServer::Http(acp::McpServerHttp::new(name, url))`
   - `NexusMcpServer::Sse(NexusMcpServerSse { name, url })` → `acp::McpServer::Sse(acp::McpServerSse::new(name, url))`
   - `NexusMcpServer::Stdio(NexusMcpServerStdio { name, command })` → `acp::McpServer::Stdio(acp::McpServerStdio::new(name, command))`

2. Update `sdk_new_session_request_from_nexus` to propagate:
   ```rust
   fn sdk_new_session_request_from_nexus(req: NexusNewSessionRequest) -> acp::NewSessionRequest {
       let mcp_servers: Vec<acp::McpServer> = req.mcp_servers
           .into_iter()
           .map(nexus_mcp_server_to_sdk)
           .collect();
       acp::NewSessionRequest::new(req.cwd).mcp_servers(mcp_servers)
   }
   ```

**Edge cases**:
- Empty `mcp_servers` vec (default) → propagates as empty vec, no behavior change
- SDK `McpServerHttp` has `headers: Vec<HttpHeader>` field that Nexus DTO doesn't carry → defaults to empty via `McpServerHttp::new()` constructor (acceptable for V1.7; headers can be added to `NexusMcpServerHttp` later if needed)

**Files touched**:

**Tests**:
- Unit test: construct `NexusNewSessionRequest` with one of each MCP server type, convert via `sdk_new_session_request_from_nexus`, verify the resulting `acp::NewSessionRequest` has matching `mcp_servers` entries

#### DTO-C2 — Replace `.parse::<u16>().unwrap_or(1)` with explicit error handling

**Severity**: low
**Source**: V1.6 WS-B QC

**Current code state**:

- `crates/nexus-acp-host/src/client.rs` lines 136–141:
  ```rust
  let protocol_version: acp::ProtocolVersion = serde_json::from_value(serde_json::json!(req
      .protocol_version
      .0
      .parse::<u16>()
      .unwrap_or(1)))
  .unwrap_or(acp::ProtocolVersion::LATEST);
  ```
  Two problems:
  1. `parse::<u16>().unwrap_or(1)` silently swallows parse errors — if the string is "2" it works, but "abc" silently falls back to 1, which is misleading
  2. Double `unwrap_or` chain is hard to reason about

**Fix strategy**:

1. Replace with a clear, logged fallback:
   ```rust
   fn sdk_protocol_version_from_nexus(version: &NexusProtocolVersion) -> acp::ProtocolVersion {
       match version.0.parse::<u16>() {
           Ok(v) => serde_json::from_value(serde_json::json!(v))
               .unwrap_or(acp::ProtocolVersion::LATEST),
           Err(e) => {
               tracing::warn!(
                   version = %version.0,
                   error = %e,
                   "Failed to parse protocol version, defaulting to LATEST"
               );
               acp::ProtocolVersion::LATEST
           }
       }
   }
   ```

2. Update `sdk_initialize_request_from_nexus` to use the new helper:
   ```rust
   fn sdk_initialize_request_from_nexus(req: NexusInitializeRequest) -> acp::InitializeRequest {
       let protocol_version = sdk_protocol_version_from_nexus(&req.protocol_version);
       let mut builder = acp::InitializeRequest::new(protocol_version);
       // ... rest unchanged
   }
   ```

**Edge cases**:
- Valid version string "1" → parsed correctly, no warning
- Invalid version string "abc" → logged warning, fallback to `LATEST`
- Valid but unsupported version "99" → `serde_json::from_value` handles it via the existing `unwrap_or(ProtocolVersion::LATEST)` fallback

**Files touched**:
- `crates/nexus-acp-host/src/client.rs`

**Tests**:
- Unit test: valid version "1" → correct `ProtocolVersion`
- Unit test: invalid version "not-a-number" → returns `ProtocolVersion::LATEST`
- Unit test: empty string → returns `ProtocolVersion::LATEST`

#### PERM-W1 — Use `toml_edit` for round-trip TOML preservation in `policy.rs`

**Severity**: low
**Source**: V1.6 WS-C QC

**Current code state**:

- `crates/nexus-acp-host/src/policy.rs` lines 155–165 (`save` method):
  ```rust
  pub fn save(&self, workspace_root: &Path) -> anyhow::Result<()> {
      // ...
      let content = toml::to_string_pretty(self)?;
      std::fs::write(&policy_path, content)?;
      Ok(())
  }
  ```
  Uses `toml` crate serialization which **destroys** comments, formatting, and key ordering. The CLI (`permission.rs`) already uses `toml_edit` for round-trip preservation in its write paths (`run_grant`, `run_deny`, etc.), but `policy.rs`'s own `save()` method still uses plain `toml`.

- `crates/nexus-acp-host/Cargo.toml` line 25: `toml_edit = "0.22"` — **already in workspace** as a dependency of `nexus-acp-host`.
- `crates/nexus42/Cargo.toml` line 30: `toml_edit = "0.22"` — also already in the CLI crate.

**Fix strategy**:

1. Refactor `PermissionPolicy::save()` to use `toml_edit` for round-trip preservation:
   - Load existing file as `toml_edit::DocumentMut` (if it exists) to preserve comments/formatting
   - Serialize the `PermissionPolicy` fields into the document via `toml_edit` operations
   - Write back the modified document

2. Alternative (simpler, recommended): Add a `save_toml_edit(&self, workspace_root: &Path)` method that takes the `toml_edit` approach, and keep `save()` as-is for backward compatibility. The CLI commands that need preservation already use `toml_edit` directly (via `load_toml_edit` / `save_toml_edit` helpers in `permission.rs`). The real fix is to move those shared helpers into `policy.rs` so both the CLI and any other callers get round-trip preservation.

3. **Recommended approach**: Move `load_toml_edit`, `save_toml_edit`, and the mutation helpers (`ensure_agents_table`, `ensure_agent_action_table`, `set_agent_capability`, `remove_agent_capability`, `clean_empty_agent_tables`) from `permission.rs` into `policy.rs` as public methods on `PermissionPolicy`. This centralizes TOML I/O and the CLI calls the policy methods. Then deprecate/remove the old `save()` method that uses plain `toml` serialization.

**Edge cases**:
- File doesn't exist yet → create fresh document (current behavior)
- File has user comments → preserved
- File has unknown keys from future schema versions → preserved by `toml_edit`

**Files touched**:
- `crates/nexus-acp-host/src/policy.rs` — add `toml_edit`-based load/save methods
- `crates/nexus42/src/commands/permission.rs` — delegate to policy methods instead of inline `toml_edit` helpers

**Tests**:
- Existing `test_save_and_load_policy` still passes
- New test: save with comment → reload preserves comment
- New test: save with unknown future key → reload preserves unknown key

**Severity**: low

  - `print_list_json` (lines 186–218): JSON output **never includes global rules**. The output structure is `{ "default": "...", "agents": {...} }` with no `grant`/`deny` global keys.

**Fix strategy**:

   ```rust
   fn print_list_json(policy: &PermissionPolicy, agent_filter: Option<&str>) -> Result<()> {
       let (global_granted, global_denied) = policy.list_permissions();

       let mut result = serde_json::Map::new();
       result.insert("default".to_string(), serde_json::json!(format!("{:?}", policy.default).to_lowercase()));

       // Add global rules
       if !global_granted.is_empty() || !global_denied.is_empty() {
           let mut global = serde_json::Map::new();
           if !global_granted.is_empty() {
               global.insert("grant".to_string(), serde_json::json!(global_granted));

           if !global_denied.is_empty() {
               global.insert("deny".to_string(), serde_json::json!(global_denied));

           result.insert("global".to_string(), serde_json::Value::Object(global));

       // Agent rules (existing logic, with agent_filter)
       // ...
       result.insert("agents".to_string(), serde_json::Value::Object(agents_json));
       println!("{}", serde_json::to_string_pretty(&result)?);
       Ok(())
   }
   ```

**Edge cases**:
- No global rules configured → no `global` key in output (skip_serializing_if pattern)

#### PERM-W3 — Validate TOML keys against known schema

**Severity**: low
**Source**: V1.6 WS-C QC

**Current code state**:

- `crates/nexus-acp-host/src/policy.rs` `PermissionPolicy` struct (lines 115–132): Deserialization uses `#[serde(default)]` on all fields. Unknown keys in the TOML file are silently ignored by `toml::from_str` / `toml_edit::DocumentMut`.
- No validation exists to warn users about unrecognized keys in `permissions.toml`.

**Fix strategy**:

1. Add a validation function that checks top-level keys against known schema:
   ```rust
   impl PermissionPolicy {
       /// Validate a TOML document for unknown keys. Returns a list of warnings.
       pub fn validate_toml_keys(doc: &toml_edit::DocumentMut) -> Vec<String> {
           let known_keys = ["default", "grant", "deny", "agents"];
           let mut warnings = Vec::new();
           for key in doc.keys() {
               if !known_keys.contains(&key) {
                   warnings.push(format!("Unknown top-level key '{}' in permissions.toml", key));
               }
           }
           // Could extend to validate agent sub-keys: grant/deny/ask
           warnings
       }
   }
   ```

2. Call validation on load (optional, with `--strict` flag) or in `nexus42 permission list` as informational warnings.

**Recommended minimal approach for V1.7**: Add validation to the `list` command output — print warnings for unknown keys without rejecting the file. This catches typos without breaking existing setups.

**Edge cases**:
- Future schema additions (new keys) → would generate false warnings. Mitigate by documenting that new keys should be added to the known set.
- Nested unknown keys under `[agents.*]` → validate `grant`/`deny`/`ask` sub-keys similarly.

**Files touched**:
- `crates/nexus-acp-host/src/policy.rs` — add `validate_toml_keys`
- `crates/nexus42/src/commands/permission.rs` — call validation in `run_list` and print warnings

#### R4 — Implement DST jump detection or downgrade docs

**Severity**: low
**Source**: V1.5 QC

**Current code state**:

- `crates/nexus-orchestration/src/scheduler/mod.rs`:
  - `SystemClock` (lines 30–36) uses `chrono::Utc::now().timestamp()` — always UTC, **tz-unaware**.
  - `Scheduler::tick()` (lines 92–106) passes `now_unix` (UTC) to `supervisor.tick_clocked(now)`.
  - `ClockSource` trait (lines 24–27) returns Unix timestamp.
  - Module doc (lines 1–12) claims: "DST / clock-jump safety: `ClockSource` abstraction allows mock for testing." But there is **no DST detection logic** — the claim is aspirational.

- The `scheduled_at` column stores a Unix timestamp (integer, UTC). Comparison is `now >= scheduled_at` in UTC. This means a schedule set for "2pm EST" is stored as its UTC equivalent and the comparison is always correct in UTC.

**Analysis**:

The current implementation is actually **safe for DST** because:
1. All times are stored and compared in UTC (Unix timestamps)
2. The `scheduled_at` field is already a Unix timestamp
3. There is no wall-clock time comparison that could be affected by DST

The risk would only materialize if:
- Users specify schedules in local time (e.g., "every day at 8am") — not currently supported
- A cron parser that produces local-time-based timestamps is added

**Fix strategy (recommended: document downgrade)**:

1. Update the module doc to accurately reflect the current state:
   ```rust
   //! # DST Safety
   //!
   //! The scheduler operates entirely in UTC via Unix timestamps. `scheduled_at`
   //! values are stored as UTC Unix timestamps, and all comparisons use UTC.
   //! This design is inherently safe against DST transitions.
   //!
   //! **Limitation**: The scheduler does not support wall-clock recurrence rules
   //! (e.g., "every day at 8am local time"). If wall-clock recurrence is added
   //! in a future version, DST jump detection will be required at that time.
   ```

2. No code changes needed — the current UTC-only approach is correct for the pre-1.0 scope.

**Edge cases**: None — this is a documentation fix.

**Files touched**:
- `crates/nexus-orchestration/src/scheduler/mod.rs` — update module-level docs

**Tests**:
- No new tests needed (existing `MockClock` tests verify time comparison correctness)

#### R7 — Clean up schedule_guards on terminal state

**Severity**: nit
**Source**: V1.5 QC

**Current code state**:

- `crates/nexus-orchestration/src/schedule/derivation.rs`:
  - `CoreContextManager` struct (lines 46–52):
    ```rust
    pub struct CoreContextManager {
        pool: Arc<SqlitePool>,
        schedule_guards: Mutex<HashMap<String, Arc<tokio::sync::Mutex<()>>>>,
    }
    ```
  - `schedule_guard()` method (lines 67–74): gets-or-creates a per-schedule `Arc<Mutex<()>>`:
    ```rust
    async fn schedule_guard(&self, schedule_id: &ScheduleId) -> Arc<tokio::sync::Mutex<()>> {
        let key = schedule_id.0.clone();
        let mut guards = self.schedule_guards.lock().await;
        guards.entry(key).or_insert_with(|| Arc::new(tokio::sync::Mutex::new(()))).clone()
    }
    ```

- **Problem**: Guards are never removed when a schedule reaches terminal state (Completed/Cancelled/Failed). Over time, the `HashMap` grows unboundedly. For pre-1.0 with low traffic, this is acceptable but worth cleaning up.

**Fix strategy**:

1. Add a `cleanup_guard` method:
   ```rust
   impl CoreContextManager {
       /// Remove the per-schedule guard for a terminal schedule.
       /// Call this when a schedule reaches Completed/Cancelled/Failed status.
       pub async fn cleanup_guard(&self, schedule_id: &ScheduleId) {
           let key = schedule_id.0.clone();
           let mut guards = self.schedule_guards.lock().await;
           guards.remove(&key);
       }
   }
   ```

2. Call `cleanup_guard` from the schedule supervisor when a schedule transitions to terminal state. The supervisor already handles terminal transitions — add a call there.

**Alternative (simpler, for pre-1.0)**: Add periodic cleanup — if the guard map exceeds a threshold (e.g., 1000 entries), prune entries for schedules that are in terminal status. This avoids coupling `CoreContextManager` to schedule lifecycle knowledge.

**Recommended approach for V1.7**: Implement the explicit `cleanup_guard()` method and document where callers should invoke it. The actual integration into the supervisor is a one-line addition at the terminal transition point.

**Edge cases**:
- Cleanup called for non-existent schedule → `HashMap::remove` is a no-op, safe
- Schedule transitions to terminal while a write is in progress → the guard is `Arc`-shared, so the active write holds a clone and completes safely; the map entry is removed but the `Arc<Mutex<()>>` stays alive until all holders drop

**Files touched**:
- `crates/nexus-orchestration/src/schedule/derivation.rs` — add `cleanup_guard` method

**Tests**:
- Unit test: `cleanup_guard` removes the entry → subsequent `schedule_guard` creates a new one
- Unit test: `cleanup_guard` on non-existent schedule → no panic

## §5 Shared Concerns

### 5.1 `toml_edit` dependency

**Already in workspace**: Both `nexus-acp-host` and `nexus42` already depend on `toml_edit = "0.22"` in their respective `Cargo.toml` files. No new dependency needed for PERM-W1.

### 5.2 DTO-C1/C2 overlap with WS-D (SDK migration)

DTO-C1 and DTO-C2 are small, contained fixes to the existing `AcpSdkAdapter` conversion layer. They do **not** overlap with a future WS-D SDK migration (which would replace `AcpSdkAdapter` internals). These fixes are worth doing now because:

1. DTO-C1 fixes a real data loss bug (`mcp_servers` silently dropped)
2. DTO-C2 improves observability (logged fallback instead of silent)
3. Both are confined to conversion functions that will be replaced wholesale during SDK migration

**WS-A does not need to finish before a future WS-D**, but WS-A fixes should land independently.

### 5.3 R4 DST detection approach recommendation

**Recommendation: Document the current UTC-only approach as correct.**

Rationale:
- All `scheduled_at` values are stored as UTC Unix timestamps
- All time comparisons are `now_utc >= scheduled_at` — inherently DST-safe
- Wall-clock recurrence ("every day at 8am local") is not a pre-1.0 feature
- If wall-clock recurrence is added in a future version, that's when DST detection should be implemented

The fix is **documentation-only** — update the module doc to accurately describe the current safety properties and the known limitation.

### 5.4 NexusContentBlock `Eq` derive safety

All inner types (`NexusTextContent`, `NexusResourceLink`) already derive `Eq`. Adding `Eq` to `NexusContentBlock` is safe and makes the type consistent with other DTO types in the same module (all of which derive both `PartialEq` and `Eq`).
