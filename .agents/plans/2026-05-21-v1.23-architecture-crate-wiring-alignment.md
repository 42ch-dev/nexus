## 1. Problem Statement

V1.21 split the OSS workspace into focused crates (`nexus-kb`, `nexus-narrative`, `nexus-knowledge`, `nexus-cloud-domain`, `nexus-moment-context-assembly`, etc.), but several crates are not yet wired into the executable product graph. Some knowledge specs describe the intended target graph, while the current `Cargo.toml` graph and CLI/daemon behavior still reflect partial implementation.

This drift is serious because it creates three risks:

1. Implementers may trust knowledge SSOT that describes behavior not actually reachable from `nexus42`.
2. Core domain modules may continue evolving as compile-only islands without product validation.
3. Naming and scope semantics (`kb`, `knowledge`, `narrative`, `moment`) are currently ambiguous enough to cause future boundary mistakes.

The plan aligns product semantics, knowledge specs, crate responsibilities, and actual runtime wiring.

## 2. User Design Decisions (Locked Inputs)

These are treated as authoritative design inputs for V1.23 unless the user revises them.

### 2.1 Entity scope hierarchy

The architecture should model entity ownership using the following scope levels:

| Scope | Meaning |
| --- | --- |
| `Global` | Product-wide global scope |
| `User` | Human user scope |
| `Creator` | Agent / creator persona scope |
| `World` | World / narrative universe scope |
| `Timeline` | Timeline scope inside a world |
| `Event` | Event scope inside a timeline |
| `Moment` | Moment / session-start context point |

For non-fungible unique entities, the same class/template has at most one instance under the same scope ID. Different classes/templates do not share this uniqueness constraint.

### 2.2 Crate intent

| Crate | Intended role |
| --- | --- |
| `nexus-moment-context-assembly` | Session-start **moment** context aggregation. It runs before a session begins and should aggregate all relevant local system domains needed for that moment. |
| `nexus-narrative` | Core entry point for creative-work narrative state: current work background, world state, timeline/event position, and narrative context. |
| `nexus-knowledge` | **User-scoped** global knowledge base. Tag-driven index and storage module for user-level knowledge that can be pulled into moment context assembly. Not Creator-scoped. |
| `nexus-cloud-sync` | Cloud transport must use `nexus-cloud-domain` for User/Pairing domain invariants rather than bypassing it. |
| `nexus-kb` | Main module for Creator knowledge assets under narrative scope. KB scope belongs under a `World` / narrative entity, not directly under Creator. It should become the key module for KB graph insertion and query. |

## 4. Goals

1. Define a single, explicit domain model for scope ownership and entity uniqueness.
2. Update knowledge specs so intended crate roles and current implementation state are not contradictory.
3. Wire previously split crates into the runtime graph where they are product-critical.
4. Rename or clarify command/module boundaries where `kb` / `knowledge` / `narrative` terms overlap.
5. Make `nexus-moment-context-assembly` the session-start aggregation point over creator memory, narrative, world-scoped KB, and user-scoped knowledge.
6. Make `nexus-cloud-sync` consume `nexus-cloud-domain` for User/Pairing invariants.
7. Add focused tests and architectural checks so the graph does not drift again.

---

## 5. Non-goals

- Do not redesign platform HTTP APIs unless a wire-contract mismatch is found.
- Do not implement a full production KB graph engine beyond the minimal interfaces needed to connect current crates safely.
- Do not introduce a new database engine or external service.
- Do not preserve backward compatibility for pre-release CLI semantics if the corrected model requires breaking command changes.
- Do not move user-facing documentation into `.agents/knowledge/`; keep `docs/` for human-facing docs and `.agents/knowledge/specs/` for normative architecture.

## 6. Architecture Direction

### 6.1 Scope ownership

Introduce a normative scope model in knowledge specs and, where needed, in local contract/domain types:

```text
Global
└── User
    ├── Creator
    │   └── World
    │       ├── Timeline
    │       │   └── Event
    │       │       └── Moment
    │       └── KB graph / narrative knowledge assets
    └── User knowledge index
```

Key distinction:

- `nexus-knowledge`: User-scoped, tag-driven global knowledge index and storage.
- `nexus-kb`: World/narrative-scoped KB graph assets and query primitives.
- `nexus-narrative`: The narrative state gateway that ties World, Timeline, Event, Moment, and KB context together.
- `nexus-moment-context-assembly`: The pre-session aggregator that asks the domain modules for context slices and produces the final moment context.

### 6.2 Expected crate graph (target)

```text
nexus42
  ├── nexus-daemon-runtime
  │   ├── nexus-orchestration
  │   ├── nexus-agent-host
  │   ├── nexus-creator
  │   ├── nexus-creator-memory
  │   ├── nexus-narrative
  │   ├── nexus-kb
  │   ├── nexus-knowledge
  │   ├── nexus-moment-context-assembly (default features)
  │   └── nexus-local-db
  ├── nexus-cloud-sync
  │   └── nexus-cloud-domain
  └── nexus-moment-context-assembly (cloud-stage only for CLI/platform flows)

nexus-moment-context-assembly
  ├── nexus-creator-memory
  ├── nexus-narrative
  ├── nexus-kb
  ├── nexus-knowledge
  └── nexus-contracts
```

Daemon must still not depend on platform HTTP code paths. If `nexus-moment-context-assembly` keeps an optional `cloud-stage`, daemon must use default features only.
