## 1. Scope and Goal

### 1.1 What WS-D Delivers

1. **SDK version bump**: `agent-client-protocol` from `=0.10.4` to `=0.11.7` in `crates/nexus-acp-host/Cargo.toml`.
2. **Adapter compilation**: Update `AcpSdkAdapter` internals so the crate compiles and all tests pass against v0.11.7.
3. **Stabilized feature adoption**: Add `session/list` and `session_info_update` support through the `NexusAcpClient` trait (new trait methods with Nexus DTO wrappers).
4. **`NexusAcpClient` trait public API verification**: Zero SDK types in public signatures â€” enforced by `cargo doc --package nexus-acp-host`.
5. **Consumer impact verification**: `nexus42` and `nexus-orchestration` compile without source changes (other than adopting optional new methods).

### 1.2 What WS-D Does NOT Deliver

| Item | Reason |
|------|--------|
| Full `sacp` migration | Nexus does not depend on sacp; it's a third-party crate. Per compass: out of scope. |
| Unstable features (`session/stop`, message ID, auth methods) | These remain behind SDK feature flags; not stable API. Per compass: out of scope. |
| Client Handler rewrite | The `SimpleClientHandler` and `PolicyAwareClientHandler` implementations work with v0.11.x without structural changes. |
| Transport/registry/session_manager changes | No SDK type exposure beyond the adapter. These modules don't directly use SDK types (except `session_manager.rs` using `acp::SessionId`). |
| `strum` workspace upgrade | The workspace uses `strum = "0.26"` with `features = ["derive"]`. The ACP SDK's internal `strum 0.28` dependency is a transitive dep that does not affect Nexus code (see Â§3.3). |
| `nexus42d` changes | The daemon does **not** link `nexus-acp-host` (enforced per Â§11.3 of tech spec). No daemon changes needed. |

### 1.3 Dependencies

- **WS-A should complete first**: DTO-C1/C2 fixes may affect `client.rs` conversion code that WS-D will also touch. If WS-A is not done, WS-D should merge WS-A's DTO fixes as a prerequisite.
- **WS-D is a dependency for WS-E**: The multi-agent worker (WS-E) needs `session/list` for session discovery within a multi-agent worker.

### 3.2 Impact on `AcpSdkAdapter` internals

- `acp::SessionId` â€” used in `sdk_prompt_request_from_nexus()`, `cancel()`, and `session_manager.rs`.
- `acp::CancelNotification` â€” used in `cancel()`.
- `acp::StopReason` â€” used in `nexus_stop_reason_from_sdk()`.

- Various request/response types in `Client` trait impls.

**Mitigation**: The compilation check is the primary gate. After bumping the version, `cargo build -p nexus-acp-host` will reveal all broken type references. The fix is mechanical: update renamed types in conversion functions and trait impls.

**Builder pattern cleanup (PR #635)**: May affect method names on builders like `InitializeRequest::new()`, `ClientCapabilities::new()`, etc. The `skills.rs` `build_v1_0_capabilities()` function uses `ClientCapabilities::new().fs(...).terminal(true)` builder chain â€” may need adjustment.

### 3.3 Impact on `NexusAcpClient` trait

**Zero impact expected.** The trait uses only Nexus-owned DTO types. SDK changes do not affect trait signatures. The DTO boundary (established in V1.6) ensures this.

### 3.4 Impact on `session_manager.rs`

The `SessionEntry` struct uses `agent_client_protocol::SessionId` directly. If v0.11.x renames `SessionId`, this import and all usages need updating. Based on the changelog, `SessionId` was not renamed â€” it's a fundamental type.

### 3.5 Impact on `Client` trait implementations

`SimpleClientHandler` and `PolicyAwareClientHandler` implement `acp::Client`. The SDK may have added new optional methods to this trait. Since the SDK provides default trait implementations for optional capability-based methods (since v0.4.4), any new methods should have defaults. If not, compilation will fail and we'll need to add stub implementations.

**Known new methods in v0.11.x (all unstable/optional)**:
- No new stable methods on `Client` trait between v0.10.4 and v0.11.7.

### 3.6 Impact on `strum` usage

None. The workspace uses `strum 0.26`. The ACP SDK bumped its internal `strum` to `0.28`, but this is a transitive dependency within the SDK's own crate graph. The two versions coexist without conflict because `strum` is a derive-macro crate â€” the generated code is self-contained.

## 4. Stabilized Feature Adoption Plan

### 4.1 `session/list` (stabilized in v0.11.1, PR #705)

**Purpose**: List active sessions on the agent. Essential for WS-E multi-agent worker session discovery.

**Integration approach**:

1. **Add new Nexus DTOs** in `crates/nexus-contracts/src/local/acp/types.rs`:
   ```rust
   /// Request to list sessions on the agent.
   pub struct NexusSessionListRequest {
       /// Optional session ID prefix filter.
       pub filter: Option<String>,
   }

   /// Response from session/list.
   pub struct NexusSessionListResponse {
       /// List of session summaries.
       pub sessions: Vec<NexusSessionSummary>,
   }

   /// Summary of a single session from session/list.
   pub struct NexusSessionSummary {
       pub session_id: NexusSessionId,
       pub task: Option<String>,
       pub modes: Option<NexusSessionModeState>,
   }
   ```

2. **Add trait method** to `NexusAcpClient`:
   ```rust
   fn list_sessions(
       &self,
       request: NexusSessionListRequest,
   ) -> impl Future<Output = AcpResult<NexusSessionListResponse>> + Send;
   ```

3. **Implement in `AcpSdkAdapter`**: Bridge through `LocalSetBridge`, convert SDK response to Nexus DTOs.

4. **Consumer impact**: `nexus42` CLI commands (`agent list` / `agent show`) can optionally adopt this for richer session display. `nexus-orchestration` worker can use this for multi-agent session tracking in WS-E.

### 4.2 `session_info_update` (stabilized in v0.11.1, PR #705)

**Purpose**: Receive session info updates from the agent (e.g., task description changes, mode changes). Useful for WS-E's session tracking.

**Integration approach**:

1. **Add new Nexus DTOs** in `crates/nexus-contracts/src/local/acp/types.rs`:
   ```rust
   /// Session info update event from the agent.
   pub struct NexusSessionInfoUpdate {
       pub session_id: NexusSessionId,
       pub task: Option<String>,
       pub modes: Option<NexusSessionModeState>,
   }
   ```

2. **Integration point**: The `subscribe()` method on `AcpSdkAdapter` already returns a `StreamReceiver` that delivers session notifications. `session_info_update` is a variant of the `SessionUpdate` enum that the stream delivers. The existing `subscribe()` mechanism handles this â€” **no new trait method needed**.

3. **Consumer impact**: Workers that already call `subscribe()` will receive `session_info_update` events as part of the stream. The `SessionManager` can optionally persist updated task descriptions.

### 4.3 `session_config_options` (stabilized in v0.10.8, PR #411)

**Purpose**: Allow clients to present session configuration options (e.g., model selection, temperature) during session creation.

**Decision**: **Deferred to WS-E.** Session config options are most useful when creating multi-agent sessions with different models per agent slot. The WS-E multi-agent worker plan will adopt this feature when implementing `AgentConfig` â†’ `session/new` with model parameters.
