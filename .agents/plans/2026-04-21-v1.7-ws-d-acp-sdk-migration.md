# ACP SDK Migration to v0.11.0 — Implementation Plan (WS-D)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade the `agent-client-protocol` dependency from `=0.10.4` to `=0.11.0`, rewrite `AcpSdkAdapter` internals to use the new session-based streaming API, and adopt the stabilized `session/list` and `session_info_update` features — while keeping the `NexusAcpClient` trait public API unchanged.

**Architecture:** The V1.6 DTO decoupling boundary isolates all SDK types inside `AcpSdkAdapter`. The `NexusAcpClient` trait uses only Nexus-owned DTOs from `nexus_contracts::local::acp`. This plan leverages that boundary: all changes are confined to `crates/nexus-acp-host/` internals. Consumers (`nexus42`, `nexus-orchestration`) require zero changes to their source code.

**Tech Stack:** Rust, `agent-client-protocol = "=0.11.0"`, `strum 0.26` (workspace), `tokio` (LocalSet bridge), `nexus-contracts` DTOs.

**V1.7 Compass Alignment:** WS-D → Milestone M2. Dependency for WS-E (multi-agent worker needs `session/list` for session discovery).

**Revision History:**
- v1 (2026-04-21): Initial implementation plan
- v2 (2026-04-21): **Major revision after T1 completion** — SDK v0.11.0 is a full architectural rewrite (session-based streaming vs old request/response). ## 1. Scope and Goal

### 1.1 What WS-D Delivers

1. **SDK version bump**: `agent-client-protocol` from `=0.10.4` to `=0.11.0` in `crates/nexus-acp-host/Cargo.toml`. ✅ **T1 DONE** (commit `f28d0c5`).
2. **Adapter compilation**: Update `AcpSdkAdapter` internals so the crate compiles and all tests pass against v0.11.0. ✅ **T1 DONE** — compilation achieved by stubbing trait methods.
3. **Trait method reimplementation**: Rewrite all 4 `NexusAcpClient` trait methods to use the new session-based streaming API (`ConnectionTo<Agent>` → `SessionBuilder` → `ActiveSession`). ✅ **T1.5 DONE** (commit `a69f9ce`).
4. **Stabilized feature adoption**: Add `session/list` and `session_info_update` support through the `NexusAcpClient` trait (new trait methods with Nexus DTO wrappers).
5. **`NexusAcpClient` trait public API verification**: Zero SDK types in public signatures — enforced by `cargo doc --package nexus-acp-host`.
6. **Consumer impact verification**: `nexus42` and `nexus-orchestration` compile without source changes (other than adopting optional new methods).

### 1.2 What WS-D Does NOT Deliver

| Item | Reason |
|------|--------|
| Full `sacp` migration | Nexus does not depend on sacp; it's a third-party crate. Per compass: out of scope. |
| Unstable features (`session/stop`, message ID, auth methods) | These remain behind SDK feature flags; not stable API. Per compass: out of scope. |
| `Client` trait reimplementation | The old `acp::Client` trait no longer exists in v0.11.0. `SimpleClientHandler` and `PolicyAwareClientHandler` were removed in T1. Permission handling will be reimplemented using the new `Builder::on_receive_request_from` pattern in a future plan. |
| Transport/registry/session_manager changes | No SDK type exposure beyond the adapter. These modules don't directly use SDK types (except `session_manager.rs` using `acp::SessionId`). |
| `strum` workspace upgrade | The workspace uses `strum = "0.26"` with `features = ["derive"]`. The ACP SDK's internal `strum 0.28` dependency is a transitive dep that does not affect Nexus code (see §3.3). |
| `nexus42d` changes | The daemon does **not** link `nexus-acp-host` (enforced per §11.3 of tech spec). No daemon changes needed. |
| Streaming response proxying | WS-E will implement full streaming from `ActiveSession.read_update()`. WS-D only needs to collect the full prompt response into `NexusPromptCompleted` for API compatibility. |

### 1.3 Dependencies

- **WS-A should complete first**: DTO-C1/C2 fixes may affect `client.rs` conversion code that WS-D will also touch. If WS-A is not done, WS-D should merge WS-A's DTO fixes as a prerequisite.
- **WS-D is a dependency for WS-E**: The multi-agent worker (WS-E) needs `session/list` for session discovery within a multi-agent worker.

## 3. SDK v0.11.0 Architecture (Actual, post-T1)

> ⚠️ **v2 revision**: T1 revealed that SDK v0.11.0 is a **full architectural rewrite**, not a mechanical migration. The original §3 assumed mechanical renames; this section replaces it with the actual API surface discovered during T1.

### 3.1 Architectural change summary

| Aspect | v0.10.4 (old) | v0.11.0 (new) |
|--------|--------------|---------------|
| **Core pattern** | Request/response (`Client` trait) | Session-based streaming (`ConnectionTo` → `SessionBuilder` → `ActiveSession`) |
| **Connection** | `ClientSideConnection` | `ConnectionTo<Agent>` (obtained via `Client::builder().connect_with()`) |
| **Initialize** | `Client::initialize()` separate step | Handled internally during `SessionBuilder::start_session()` |
| **Create Session** | `Client::create_session(req)` → response | `connection.build_session(cwd).start_session().await` → `ActiveSession` |
| **Prompt** | `Client::prompt(session_id, messages)` → single response | `session.send_prompt(prompt)` (fire-and-forget) + `session.read_update().await` loop (streaming) |
| **Cancel** | `Client::cancel(session_id)` | `CancelNotification` JSON-RPC notification (no client wrapper method; use `send_notification_to`) |
| **List Sessions** | `Client::list_sessions()` | `connection.send_request(ListSessionsRequest)` → `SentRequest<ListSessionsResponse>` (raw JSON-RPC) |
| **Set Config** | N/A | `connection.send_request(SetSessionConfigOptionRequest)` → `SentRequest<SetSessionConfigOptionResponse>` |
| **Handler pattern** | `impl Client for MyHandler` | `Builder::on_receive_request_from()` callbacks |

### 3.2 Key types in SDK v0.11.0

**Connection:**
- `acp::Client` — zero-sized struct, entry point
- `Client::builder().connect_with(transport)` → `ConnectionTo<Agent>` (Clone + Send)
- `ConnectionTo<Agent>` — the base connection handle; has `send_request()` / `send_request_to()` / `send_notification()` / `send_notification_to()` methods for raw JSON-RPC

**Session:**
- `connection.build_session(cwd)` → `SessionBuilder<Counterpart, NullRun>`
- `SessionBuilder::start_session().await` → `ActiveSession<'static, Counterpart>`
- `SessionBuilder::build_session_from(request)` — create from existing `NewSessionRequest` (for MCP injection)
- `ActiveSession` — mutable, holds session state:
  - `.session_id()` → `&SessionId`
  - `.modes()` → `&Option<SessionModeState>`
  - `.meta()` → `&Option<Map<String, Value>>`
  - `.response()` → `NewSessionResponse`
  - `.connection()` → `ConnectionTo<Link>`
  - `.send_prompt(prompt)` → `Result<(), Error>` (fire-and-forget)
  - `.read_update().await` → `Result<SessionMessage, Error>` (streaming)
  - `.read_to_string().await` → `Result<String, Error>` (collect all text until StopReason)

**Messages:**
- `SessionMessage::SessionMessage(Dispatch)` — streaming content updates
- `SessionMessage::StopReason(StopReason)` — prompt completion

**Schema types (JSON-RPC):**
- `ListSessionsRequest` / `ListSessionsResponse` — `session/list`
- `SetSessionConfigOptionRequest` / `SetSessionConfigOptionResponse` — `session/set_config_option`
- `CancelNotification` — `session/cancel` notification (no response)
- `NewSessionRequest` / `NewSessionResponse` — `session/new`
- `PromptRequest` / `PromptResponse` — `session/prompt`
- `InitializeRequest` / `InitializeResponse` — `initialize`

### 3.3 Impact on `AcpSdkAdapter` internals (actual, post-T1)

**What was done in T1:**
- SDK bumped from `=0.10.4` to `=0.11.0`
- `with_connection()` rewritten to use `Client::builder().connect_with()` → `ConnectionTo<Agent>`
- `LocalSetBridge` kept for `!Send` futures
- Connection kept alive via `std::future::pending::<()>().await` in callback
- `Drop` impl joins setup task
- All 4 trait methods **stubbed** with "not fully implemented" error returns
- DTO conversion functions preserved (some marked `#[allow(dead_code)]`)
- `SimpleClientHandler` and `PolicyAwareClientHandler` **removed** (old `Client` trait no longer exists)

**T1.5 completed (commit `a69f9ce`):**
- `initialize()` → sends `InitializeRequest` via `connection.spawn()` + `block_task()`, caches response
- `create_session()` → maps to `connection.build_session_from(req).block_task().start_session()`, stores `ActiveSession` in `SdkConnection.sessions`
- `prompt()` → maps to `session.send_prompt()` + `read_update()` loop until `StopReason`
- `cancel()` → sends `CancelNotification` via `connection.send_notification_to()`
- `SdkConnection` refactored to hold `init_response` and `sessions: HashMap<String, ActiveSession<'static, Agent>>`
- All verification gates passed: build, test, fmt, clippy, doc

### 3.4 Impact on `NexusAcpClient` trait

**Zero impact expected.** The trait uses only Nexus-owned DTO types. SDK changes do not affect trait signatures. The DTO boundary (established in V1.6) ensures this.

### 3.5 Impact on `strum` usage

None. The workspace uses `strum 0.26`. The ACP SDK bumped its internal `strum` to `0.28`, but this is a transitive dependency within the SDK's own crate graph. The two versions coexist without conflict because `strum` is a derive-macro crate — the generated code is self-contained.

## 4. Stabilized Feature Adoption Plan (v2)

### 4.1 `session/list` (stabilized in v0.11.1, PR #705)

**Purpose**: List active sessions on the agent. Essential for WS-E multi-agent worker session discovery.

**SDK API**: `ConnectionTo<Agent>` has no dedicated `list_sessions()` method. Instead, use raw JSON-RPC:
```rust
let response = connection.send_request(ListSessionsRequest::new()).await?;
// response is ListSessionsResponse
```

**Integration approach**:

1. **Add new Nexus DTOs** in `crates/nexus-contracts/src/local/acp/types.rs` (types TBD based on actual `ListSessionsResponse` shape from SDK).
2. **Add trait method** to `NexusAcpClient`:
   ```rust
   fn list_sessions(
       &self,
   ) -> impl Future<Output = AcpResult<NexusSessionListResponse>> + Send;
   ```
3. **Implement in `AcpSdkAdapter`**: Bridge through `LocalSetBridge`, use `connection.send_request(ListSessionsRequest)`, convert response to Nexus DTOs.
4. **Consumer impact**: WS-E multi-agent worker can use this for session tracking.

### 4.2 `session_info_update` → `session/set_config_option`

**SDK API**: Raw JSON-RPC:
```rust
let response = connection.send_request(SetSessionConfigOptionRequest { ... }).await?;
// response is SetSessionConfigOptionResponse
```

**Note**: The original plan described `session_info_update` as a stream event, but the actual SDK v0.11.0 uses `session/set_config_option` as a request/response pattern for updating session metadata. The integration approach in T3 will be adjusted accordingly.

### 4.3 `session_config_options` (stabilized in v0.10.8, PR #411)

**Decision**: **Deferred to WS-E.** Session config options are most useful when creating multi-agent sessions with different models per agent slot.

### T1.5: Wire trait methods to new session-based API (L) — **DONE** ✅

**Scope:**
Replace stub implementations with real implementations using the new SDK v0.11.0 session-based API.

- [x] **Step 1: `initialize()` implementation**

Sent `InitializeRequest` via `connection.spawn()` + `block_task()` with a oneshot channel to relay the response back. Response is cached in `SdkConnection.init_response` for subsequent calls.

- [x] **Step 2: `create_session()` implementation**

Map to the new SDK session creation flow:
```rust
let sdk_request = sdk_new_session_request_from_nexus(request);
let builder = connection.build_session_from(sdk_request);
let active_session = builder.block_task().start_session().await?;
let session_id_str = active_session.session_id().to_string();
// Store active_session in SdkConnection.sessions HashMap
```
```

Key considerations:
- `ActiveSession` is `!Send` if it captures non-Send state — need `LocalSetBridge`
- Need to store `ActiveSession` for subsequent `prompt()` calls
- May need to store in `SdkConnection` behind a `Mutex` (since `ActiveSession` is `&mut self`)

- [x] **Step 3: `prompt()` implementation**

Implemented using `ActiveSession::send_prompt()` + `read_update()` loop. Streaming updates are logged but not accumulated in `NexusPromptCompleted` (the DTO only carries `stop_reason`; streaming content is a future API).

- [x] **Step 4: `cancel()` implementation**

Sends `CancelNotification` via `connection.send_notification_to(Agent, CancelNotification::new(session_id))`. Uses SDK constructor (`#[non_exhaustive]` struct).

- [x] **Step 5: Refactor `SdkConnection` state**

`SdkConnection` now holds:
- `connection: ConnectionTo<Agent>` — the raw SDK connection
- `init_response: Option<InitializeResponse>` — cached init response
- `sessions: HashMap<String, ActiveSession<'static, Agent>>` — active sessions keyed by ID

The bridge's single-threaded guarantee provides exclusive `&mut` access, no Mutex needed.

## 7. Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Session state management complexity in T1.5** | High | High | `ActiveSession` is `!Send` and requires `LocalSetBridge`. Need careful design of `SdkConnection` to store sessions. Consider using `tokio::sync::Mutex` for access. |
| **`ActiveSession` lifetime management** | Medium | High | `ActiveSession<'static>` obtained from `start_session()` should be storable. If not, use `self-referencing` crate or restructure to avoid storage. |
| **`cancel()` notification shape mismatch** | Medium | Low | `CancelNotification` may not accept a session_id in v0.11.0. Check SDK schema types and use raw JSON-RPC if needed. |
| **`session/list` response shape mismatch** | Medium | Low | DTOs written to match SDK shape; verified at compile time. |
| **`strum` version conflict** | Very Low | Low | Workspace `strum 0.26` and SDK's transitive `strum 0.28` coexist without conflict. |
| **`nexus42` fails to compile after T1.5** | Low | Medium | `nexus42` consumes SDK types only through `nexus-acp-host`. The trait API is unchanged. |
| **T1.5 blocked by `LocalSetBridge` limitations** | Medium | Medium | The bridge pattern works for `!Send` futures. If `ActiveSession` requires spawning, may need to restructure. |
