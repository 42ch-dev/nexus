## T1 Research Decision: Claude CLI Multi-turn

**Date**: 2026-05-15
**Researcher**: `@fullstack-dev`
**Status**: Decision locked — Option B selected

### Research Findings

#### Claude CLI Flags Discovered (official docs, 2026-05-15)

| Flag | Behavior | Multi-turn relevance |
|------|----------|---------------------|
| `--print` / `-p` | Single-turn: reads stdin (or arg), produces response, exits. | Current default — single-shot only. |
| `--continue` / `-c` | Loads the most recent session in the current directory. Works with `-p`: `claude -c -p "query"`. | **Enables multi-turn**: each invocation resumes most recent conversation. |
| `--resume` / `-r` | Resume a specific session by ID or name. Accepts UUID or session name. Works with `-p`. | **Enables multi-turn**: explicit session-ID continuity across invocations. |
| `--session-id` | Use a specific UUID as the session ID for the conversation. | **Enables session-ID injection**: host can predetermine the session ID. |
| `--output-format json` | Print mode outputs structured JSON including session metadata. | Enables session-ID extraction from first invocation output. |
| `--output-format stream-json` | Streaming JSON output with per-event structured data. | Enables real-time event parsing + session-ID capture. |
| `--fork-session` | When resuming, creates a new session ID instead of reusing the original. Combine with `--resume` or `--continue`. | Enables session branching without mutating the original. |
| `--name` / `-n` | Sets a display name for the session, resumable by name. | Alternative to UUID-based resume. |
| `--max-turns` | Limit agentic turns (print mode only). | Host can bound agent loop iterations. |
| `--no-session-persistence` | Disables session transcript persistence. Print mode only. | Useful for ephemeral sessions that should not leave disk traces. |
| `--input-format stream-json` | Structured streaming input protocol (not persistent stdin). | Not relevant — this is a structured input protocol, not persistent-process multi-turn. |

#### Key Finding: Option A (persistent stdin) is NOT feasible

Claude CLI `--print` mode is explicitly single-turn: it reads stdin once, processes the prompt, writes response to stdout, and exits. There is no documented or observable mechanism to keep the process alive and feed multiple prompts through persistent stdin. The `--input-format stream-json` flag is a structured input protocol for a single invocation, not a mechanism for multi-prompt persistence.

#### Key Finding: Option B (multi-process resume) is fully supported

Claude CLI natively supports session continuity across process invocations:

1. **Session creation**: `claude -p "first prompt" --output-format json` creates a session and returns a session ID in its JSON output.
2. **Session resume**: `claude -p "second prompt" --resume <session-id> --output-format json` continues the conversation with full context.
3. **Continue shortcut**: `claude -c -p "follow-up"` resumes the most recent session in the current directory without needing the session ID explicitly.
4. **Session-ID injection**: `claude --session-id <uuid> -p "first prompt"` lets the host predetermine the session ID, eliminating the need to parse it from output.

The official Agent SDK uses exactly this pattern: capture `session_id` from the first `query()` call's init message, then pass `resume=session_id` to subsequent `query()` calls.

### Decision: Option B — Multi-process session continuity via `--resume`

**Selected option**: B (multi-process `--resume` / `--session-id` continuity)

**Rationale**:

1. **Option A is infeasible**: Claude CLI `--print` mode exits after one prompt. No persistent-stdin multi-turn mechanism exists.
2. **Option B is first-class**: `--resume` and `--session-id` are documented, stable, and used by the official Agent SDK for the same purpose.
3. **Option C is unnecessary**: Multi-turn IS feasible via Option B; no need to restrict native provider to single-turn only or document as ACP-only.
4. **Session-ID injection** (`--session-id`) is the cleanest path: the host generates a UUID at launch time, passes it on every invocation, and avoids the fragility of parsing session IDs from output.

### Implementation Path

#### Changes to `claude.rs`

1. **`default_config()` args**: Change from `vec!["--print"]` to `vec!["--print", "--output-format", "stream-json"]` to get structured output for reliable parsing.
2. **`NativeSession` struct**: Add `session_id: Option<String>` field to store the Claude CLI session ID (distinct from `HostSessionId`). Initially `None`; set on first `execute()`.
3. **`launch()`**: Generate a UUID and store it as the pending `claude_session_id`. Do NOT spawn a process yet (matches current Wave 1 behavior).
4. **`execute()` first call**: Spawn with `--session-id <uuid> --print --output-format stream-json`. Write prompt to stdin. Parse response for confirmation. Store session ID in `NativeSession`.
5. **`execute()` subsequent calls**: Spawn with `--resume <session-id> --print --output-format stream-json`. Write prompt to stdin. Response includes full conversation context.
6. **`shutdown()`**: No need to kill a persistent process (there isn't one). Clear the session map entry.

#### Session ID Strategy

Two sub-options exist for session ID management:

- **Sub-option B1 (host-generated, recommended)**: Use `--session-id <uuid>` on first invocation. The host generates a UUID v4 at `launch()` time and uses it consistently. No parsing of session IDs from output needed.
- **Sub-option B2 (output-parsed)**: Let Claude CLI generate the session ID, parse it from `--output-format json` output, store it, and use `--resume <id>` on subsequent calls. More fragile, not recommended.

**Selected sub-option**: B1 (host-generated UUID via `--session-id`).

### Expected Test Strategy

1. **Unit tests**: Mock `tokio::process::Command` to verify correct flag assembly for first vs. subsequent `execute()` calls. Verify `--session-id` is present on first call and `--resume` on subsequent calls.
2. **Integration fixture**: Create a test fixture script that simulates Claude CLI behavior (reads stdin, writes JSON response with session echo, exits). Verify multi-turn flag progression.
3. **No live Claude CLI dependency**: All tests use mock/fixture scripts, not the real `claude` binary. CI does not require Claude CLI installation.
4. **Capability descriptor**: `session_restore` capability should be set to `true` after this change, since `--resume` provides true session continuity.

### Fallback

If `--session-id` + `--resume` proves unreliable in practice (e.g., session transcript corruption, context window limits across resumes):

1. Downgrade to single-turn only (`--print` without resume flags).
2. Set `CapabilityDescriptor.session_restore = false`.
3. Document that multi-turn requires ACP provider.
4. Add `--no-session-persistence` flag when operating in single-turn mode to avoid disk pollution.

### Risks

1. **Context window accumulation**: Each `--resume` re-sends the full transcript. Long conversations may hit context limits. Mitigation: document a recommended max-turns-per-session or implement host-side compaction triggers.
2. **Transcript disk usage**: Claude CLI stores transcripts in `~/.claude/projects/<project>/<session-id>.jsonl`. Long-running hosts may accumulate large transcript files. Mitigation: use `--no-session-persistence` for ephemeral sessions, or document cleanup expectations.
3. **Claude CLI version dependency**: `--session-id` and `--resume` flags must be available on the target Claude CLI version. Mitigation: `probe()` should verify CLI version or test flag availability.

## T2 Research Decision: ACP SetModel/SetMode

**Status**: Decision locked — Option C (Hybrid) selected

### Research Findings

#### ACP Protocol Architecture (SDK v0.11.1 / schema v0.12.0)

The ACP protocol provides **three distinct mechanisms** for model and mode changes:

| Mechanism | RPC Method | SDK Feature Flag | Status |
|-----------|-----------|-------------------|--------|
| `session/set_mode` | `SetSessionModeRequest` | **Stable** (always compiled) | Production-ready |
| `session/set_model` | `SetSessionModelRequest` | `unstable_session_model` | Behind feature gate |
| `session/set_config_option` | `SetSessionConfigOptionRequest` | **Stable** (always compiled) | Production-ready |

#### Key Finding 1: `session/set_mode` is a first-class stable RPC

`SetSessionModeRequest` (at `agent.rs:2071`) takes `{ session_id, mode_id }` and sends `session/set_mode`. This is **not** behind a feature flag — it is part of the stable ACP protocol. The `NexusAcpClient` trait currently does **not** expose this method, but the underlying SDK fully supports it.

The `NewSessionResponse` returns `modes: Option<SessionModeState>` which tells the client which modes are available. Our `NexusSessionCreated` already maps this field.

#### Key Finding 2: `session/set_model` is unstable and feature-gated

`SetSessionModelRequest` (at `agent.rs:3381`) is behind `#[cfg(feature = "unstable_session_model")]`. Our `nexus-acp-host/Cargo.toml` does **not** enable this feature. The `NewSessionResponse` includes `models: Option<SessionModelState>` behind the same feature gate.

**Conclusion**: We cannot use `session/set_model` directly without enabling the unstable feature, and even then, ACP agents are not guaranteed to support it.

#### Key Finding 3: `session/set_config_option` is the general-purpose stable mechanism

`SetSessionConfigOptionRequest` (at `agent.rs:2599-2655`) takes `{ session_id, config_id, value }` and sends `session/set_config_option`. This is stable and already implemented in our `NexusAcpClient::set_config_option()`.

**Critical detail**: The `config_id` is an agent-defined string identifier. It is **not** standardized across agents. The protocol provides a `SessionConfigOptionCategory` enum with variants `Mode`, `Model`, `ThoughtLevel`, and `Other(String)`, but these are **UX hints**, not lookup keys.

Config options are discovered dynamically via:
1. `NewSessionResponse.config_options` — returned at session creation
2. `SetSessionConfigOptionResponse.config_options` — returned after every set call
3. `SessionUpdate::ConfigOptionUpdate` — pushed during prompt processing

Each `SessionConfigOption` has:
- `id: SessionConfigId` — the actual key to use in `set_config_option`
- `category: Option<SessionConfigOptionCategory>` — UX hint (Mode/Model/ThoughtLevel/Other)
- `kind: SessionConfigKind` — either `Select` (with current_value and options list) or unstable `Boolean`

#### Key Finding 4: Our NexusSessionCreated does not capture config_options

The SDK's `NewSessionResponse` (at `agent.rs:1006-1037`) returns `config_options: Option<Vec<SessionConfigOption>>`, but our `NexusSessionCreated` (at `types.rs:363-368`) only maps `session_id` and `modes`. The `config_options` field is **dropped** during conversion.

#### Key Finding 5: The `NexusAcpClient` trait does not expose `session/set_mode`

The `NexusAcpClient` trait (at `client.rs:407-469`) defines: `initialize`, `create_session`, `send_prompt`, `cancel`, `list_sessions`, and `set_config_option`. It does **not** have a dedicated `set_mode()` method that maps to the `session/set_mode` RPC. The SDK does support this RPC through `send_request_to(Agent, SetSessionModeRequest)`.

### Option Evaluation

#### Option A: Implement via `set_config_option()` only

- **Mode**: Search `config_options` from `NewSessionResponse` for an option with `category == Mode`. Use its `id` as the `config_id` in `set_config_option()`. This works but is indirect.
- **Model**: Search `config_options` for `category == Model`. Same indirect path.
- **Problem**: Not all agents expose model/mode as config options. Some agents use the dedicated `session/set_mode` RPC instead. Using only `set_config_option` would miss agents that support `set_mode` but don't expose a config option with `category: Mode`.
- **Verdict**: Insufficient — doesn't cover the `session/set_mode` stable RPC path.

#### Option B: Remove claims, enable after negotiation

- Set `set_model = false` and `set_mode = false` in `acp_full()`.
- Only enable after `NewSessionResponse` confirms the provider supports model/mode.
- **Problem**: `acp_full()` is a static const function. It cannot be dynamic. The capability descriptor would need to become a runtime-computed value based on negotiation results.
- **Verdict**: Correct in principle, but the current architecture doesn't support dynamic capability descriptors per-session. Significant refactoring needed for the descriptor to become per-provider-instance rather than per-provider-type.

#### Option C: Hybrid — use `set_config_option()` for model, remove `set_model` claim; use `session/set_mode` for mode

- **Mode**: The `session/set_mode` RPC is stable and first-class. Add a `set_mode()` method to `NexusAcpClient` that sends `SetSessionModeRequest`. In `AcpProvider::execute()`, map `SetMode { mode }` to `client.set_mode(session_id, mode)`. Available modes are already returned in `NexusSessionCreated.modes`.
- **Model**: Use `set_config_option()` to search `config_options` for a `category: Model` entry. If found, set it. If not found, return `CapabilityUnsupported` with graceful fallback.
- **Descriptor**: Set `set_mode = true` in `acp_full()` (mode switching IS supported via the stable RPC). Set `set_model = false` in `acp_full()` (model switching depends on dynamic discovery). Add a new `acp_full_without_model()` or make model capability per-session after negotiation.
- **Verdict**: Best balance of protocol correctness, capability honesty, and implementation feasibility.

### Decision: Option C (Hybrid)

**Selected option**: C

**Rationale**:

1. **`session/set_mode` is stable**: The ACP protocol provides a dedicated, stable RPC for mode switching. Not using it would be ignoring a first-class protocol feature. Our codebase already maps `SessionModeState` in `NexusSessionCreated`, so mode availability is already discoverable.
2. **`session/set_model` is unstable**: The dedicated model-switching RPC is feature-gated and may change. Using `set_config_option()` with dynamic discovery is the safer path for model switching.
3. **Capability honesty**: `acp_full()` should claim `set_mode = true` (we CAN do it via the stable RPC) but `set_model = false` (we can't guarantee it — depends on agent config options). This eliminates the current lie.
4. **Graceful degradation**: If an agent doesn't expose a model config option, `SetModel` returns `CapabilityUnsupported` — same as today, but now the descriptor honestly reflects this possibility.

### Config Option ID Mapping (for model switching via `set_config_option`)

Model config option IDs are **not standardized** — they are agent-defined. Discovery flow:

1. After `create_session()`, the `NewSessionResponse` may include `config_options`.
2. Search for a `SessionConfigOption` where `category == Some(SessionConfigOptionCategory::Model)`.
3. If found, extract its `id` (the `config_id`) and the `current_value` from its `Select` kind.
4. To set a model: call `set_config_option(session_id, config_id, new_model_value_id)`.
5. If no model config option exists, `SetModel` returns `CapabilityUnsupported`.

**Required change**: `NexusSessionCreated` must capture `config_options: Option<Vec<NexusConfigOption>>` from the SDK's `NewSessionResponse`. Currently this field is dropped.

### Graceful Fallback Behavior

| Operation | Success Path | Fallback Path |
|-----------|-------------|---------------|
| `SetMode` | Send `session/set_mode` RPC → success | Agent returns error → map to `HostError::capability_unsupported` |
| `SetModel` | Find model config option → `set_config_option()` → success | No model config option found → `HostError::capability_unsupported` |
| `SetModel` | Found config option but `set_config_option()` fails | Map SDK error → `HostError::OpFailed` with context |

### Implementation Path

#### Changes to `nexus-acp-host`:

1. **`NexusAcpClient` trait**: Add `set_mode(session_id, mode_id)` method that sends `SetSessionModeRequest`.
2. **`AcpSdkAdapter`**: Implement `set_mode()` using `connection.send_request_to(Agent, SetSessionModeRequest)`.
3. **`NexusSessionCreated`**: Add `config_options: Option<Vec<NexusConfigOption>>` field and map it from SDK's `NewSessionResponse`.

#### Changes to `nexus-agent-host`:

1. **`CapabilityDescriptor::acp_full()`**: Change `set_model: false` (honest — depends on dynamic discovery). Keep `set_mode: true` (stable RPC exists).
2. **`AcpProvider::execute()` for `SetMode`**: Map to `client.set_mode(acp_session_id, mode)`. On success, emit a status event. On failure, map to `CapabilityUnsupported` or `OpFailed`.
3. **`AcpProvider::execute()` for `SetModel`**: Look up model config option from cached `config_options` in session state. If found, call `client.set_config_option()`. If not found, return `CapabilityUnsupported`.
4. **Session state**: Store `config_options` from `create_session()` response in `AcpProvider`'s session map for later `SetModel` lookups.

### Test Strategy

1. **Unit: set_mode conversion**: Test `SetSessionModeRequest` construction from `SetMode { mode }`. Verify session_id and mode_id mapping.
2. **Unit: config option lookup**: Test that searching a `Vec<NexusConfigOption>` for `category == Model` correctly finds or misses the model option.
3. **Unit: set_config_option for model**: Test that `SetModel { model }` maps to `NexusSetConfigOptionRequest` with the discovered config_id and model value.
4. **Unit: fallback paths**: Test that missing config_options returns `CapabilityUnsupported`. Test that missing model category returns `CapabilityUnsupported`.
5. **Integration fixture**: Create a mock ACP agent that responds to `session/set_mode` and `session/set_config_option`. Verify the full flow.
6. **Capability descriptor test**: Verify `acp_full()` has `set_mode = true, set_model = false`. Verify `native_cli_limited()` has both `false`.

### SDK Limitations Discovered

1. **`session/set_model` is feature-gated**: The `SetSessionModelRequest` type is behind `unstable_session_model`. We cannot use it without enabling the feature and accepting instability. Using `set_config_option()` as a workaround is the correct approach for model switching.
2. **`config_options` dropped in `NexusSessionCreated`**: Our DTO mapping drops the `config_options` field from `NewSessionResponse`. This must be fixed to enable model switching.
3. **`NexusAcpClient` missing `set_mode`**: The trait doesn't expose the stable `session/set_mode` RPC. A new trait method is needed.
4. **No standardized model config_id**: Model config option IDs are agent-specific. Dynamic discovery from `NewSessionResponse.config_options` is required; hardcoding config_ids would be incorrect.
