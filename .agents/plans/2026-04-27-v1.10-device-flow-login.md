# V1.10 — Device Flow Login + DeviceID Header + Daemon Cleanup

## Summary

Replace the mock daemon-proxy device flow with real platform-direct RFC 8628 Device Authorization Grant implementation (closing TD-10). Generate a persistent machine UUID v4 and inject `X-Device-ID` header into all platform HTTP requests. Fully remove all mock auth HTTP endpoints and context mock from the daemon — eliminating the CLI → daemon local HTTP dependency for auth, which was an incorrect architectural coupling.

## Background

- **Platform V1.10**: Done (PR #44, plans 70–76). Device flow endpoints deployed at `POST /api/v1/auth/device/code` and `POST /api/v1/auth/device/token`. Rate limit prefers `X-Device-ID` over IP.
- **TD-10 residual**: `verify_device_code()` in `crates/nexus42d/src/auth/device_flow.rs` is a deliberate stub returning `Ok(false)`. The entire daemon device flow path (4 mock handlers, device_flow.rs stub, device_code_sessions SQLite table) needs removal — not upgrade.
- **Current architecture problem**: CLI calls daemon over local HTTP for auth (`user_auth.rs` → `DaemonClient` → daemon mock handlers). This is an incorrect design — the CLI should call platform directly.
- **Design compass**: `.agents/knowledge/v1.10-delivery-compass-v1.md`

## Design Decisions

1. **CLI calls platform directly for device flow** — no daemon proxy. Rationale: platform compass §12 says "CLI 调用 platform POST /api/v1/auth/device/code"; daemon is a local supervisor, not an auth proxy.
2. **Token storage**: Platform JWT stored in `~/.nexus42/auth.json` (extend existing `AuthStore` with new `user_token` field). This avoids daemon SQLite dependency for auth state and makes `nexus42 auth status` work without daemon.
3. **DeviceID generation**: One-time generation on first use. Store a random UUID v4 to `~/.nexus42/device-id` (plain text, 0600 permissions). Add `device_id_path()` to `nexus-home-layout` crate.
4. **DeviceID injection**: Add `X-Device-ID` header to `PlatformClient::build_request()` and the new device flow HTTP client. SyncClient may also need it (TBD during implementation).
5. **Daemon mock endpoint removal**: **Fully delete** `auth.rs` handler (4 mock endpoints), `device_flow.rs` module, `context.rs` handler. Remove from router, cleanup imports. Drop `device_code_sessions` SQLite table via new migration. Keep `auth_tokens` table (used by `require_auth` middleware for daemon-internal route protection).
6. **CLI context fallback**: After removing daemon context mock, `nexus42 context assemble` returns a "not yet available" error with TODO markers. No platform integration for context assembly in V1.10.
7. **WS-A + WS-B can proceed in parallel** — WS-A touches `crates/nexus42/` + `crates/nexus42d/`; WS-B touches `crates/nexus-sync/` + `crates/nexus-home-layout/`. No code conflicts.

## Out of Scope

- Token refresh (platform JWT refresh mechanism — future version)
- Creator API key handling changes (existing flow unchanged)
- `--flow device` flag (CLI device flow is the default/only auth flow)
- DeviceID rotation or revocation
- Multi-device sync (platform side handles per-device rate limiting)
- Daemon non-HTTP stub cleanup (engine.rs, session.rs, legacy stubs, etc. — deferred to V1.11)
