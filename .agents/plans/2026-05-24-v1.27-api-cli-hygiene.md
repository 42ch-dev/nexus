## 1. Problem Statement

Several surfaces still misrepresent shipped behavior:

- `creator clone` calls non-existent `/v1/local/world/clone` with a warning then failure.
- `CreatorCommand::Kb` doc comment says world scope is ‚Äúcoming soon‚Äù while read (and after Plan 1, write) is implemented.
- `api/mod.rs` `narrative_routes` module comment still says in-memory / no persistence.
- Daemon work-KB handlers ignore scope confusion (residual **R3**): clients may assume `/v1/local/kb/*` serves World KB.

---

## 2. Goals

1. **Remove or hard-deprecate** `creator clone` ‚Äî no HTTP call to `/v1/local/world/clone`; message points to future platform/sync docs.
2. Fix **clap / module docs** for `creator kb` world scope (reads + writes).
3. Update **`narrative_routes`** comment in `api/mod.rs` to SQLite persistence (V1.26 reality).
4. **R3**: If request body or query includes `scope != work` (when field added) or document-only path: return **400** with explicit ‚Äúwork-scope file index only‚Äù message; alternatively add optional `scope` query param that rejects non-`work` values (minimal DF-42 slice).
5. Update [iterations/README.md](../iterations/README.md): V1.26 ‚Üí Shipped; V1.27 ‚Üí Active.
6. Optional nit: populate `content_hash` on **static** reference register (R7 partial) ‚Äî only if zero-risk in same PR.

---

## 3. Non-goals

- Full DF-42 route redesign or new `/v1/local/world-kb/*` routes.
- Implement local world clone/fork.
- Platform HTTP changes.
- Close R5/R6/R8/R9 unless trivial one-liners.
