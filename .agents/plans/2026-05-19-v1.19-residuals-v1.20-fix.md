## Findings and Fix Plan

### Group A — Manager path validation (`core/manager.rs`, `config.rs`)

| Finding | Severity | Current source observation | Fix approach |
|---|---:|---|---|
| `QC2-F002` — `CreateSessionRequest.cwd` not validated for path traversal | medium | `HostManager::create_session()` now calls `validate_workspace_path(&request.cwd)` before building `LaunchSpec`, but that helper only verifies absolute/no-`..`/exists. It does not prove `cwd` remains under the `HostStartConfig.workspace_root` trust boundary or other configured boundary. | Store the canonical `HostStartConfig.workspace_root` when `start()` succeeds. Replace session cwd validation with `validate_workspace_path_under(cwd, workspace_root)` that canonicalizes both paths and checks `cwd.starts_with(workspace_root)`. Reject relative paths, `..` traversal, nonexistent paths, and symlink escapes. Pass only the canonical cwd into `LaunchSpec`. |
| `QC2-F003` — TOCTOU in config path validation (`exists()` pre-check) | low | `HostManager::start()` avoids a direct pre-check but still branches on `config.config_path.exists()` after a failed canonicalization. `config::load_config()` also uses `exists()` before `read_to_string()`. | Introduce an optional-config loader/validator that does not use `exists()` as an authorization decision. Read the file directly: `Ok(NotFound)` means defaults; `Ok(content)` is parsed; any other read error fails. For path containment, validate the intended path lexically/canonically against the canonical allowed config dir without an `exists()` race, and canonicalize/read only after containment is established. |

### Group B — Manager shutdown handle correctness (`core/manager.rs`)

| Finding | Severity | Current source observation | Fix approach |
|---|---:|---|---|
| `QC2-F004` — `shutdown()` constructs `ManagedSessionHandle` with hardcoded `acp_full()` capabilities | low | `shutdown()` now attempts to read negotiated capabilities from `SessionRegistry`, but still falls back to `CapabilityDescriptor::acp_full()` if a session is missing. That fallback can overstate capabilities and reintroduce the original issue in edge cases. | Remove the `acp_full()` fallback from shutdown handle construction. If a session ID lacks a session registry entry, log a warning and skip/clean its stale provider mapping rather than creating a fabricated full-capability handle. Add a test provider that records the shutdown handle capabilities and assert custom negotiated capabilities are preserved. |

### Group C — ACP permission handler registration (`providers/acp.rs`)

| Finding | Severity | Current source observation | Fix approach |
|---|---:|---|---|
| `QC2-F005` — permission handler race in `AcpProvider::new()` lacks explicit synchronization | low | `AcpProvider::new()` builds a handler then calls `tokio::spawn(async move { client_for_handler.set_permission_handler(handler).await; })`. Constructor returns before the handler is definitely installed. | Make handler registration synchronous from the caller's perspective. Preferred: change construction to `pub async fn new(...) -> Self` and `await client_arc.set_permission_handler(handler)` before returning. If a sync constructor must remain for tests, split into `build_permission_handler(...)` plus `new_with_registered_client(...)` and require production construction through the async path. |
| `QC3-F003` — permission handler registration is racy (`spawn` without await) | low | Same root cause as `QC2-F005`. | Covered by the same constructor change. Add a focused unit test around a pure helper that maps tool risk + resolver outcome to `AcpPermissionOutcome`; add an integration/constructor test if `AcpSdkAdapter` test seams permit. |

### Group D — ACP execute timeout cancellation (`providers/acp.rs`)

| Finding | Severity | Current source observation | Fix approach |
|---|---:|---|---|
| `QC3-F002` — orphaned ACP session on execute timeout (no cancel sent) | low | `AcpProvider::execute()` emits `OpFailed` on `stream_prompt` setup timeout and streaming item timeout, but the timeout paths do not call `self.client.cancel(acp_session_id)`. The stream timeout branch is currently synchronous `.map(...)`, so it cannot await cancellation. | On `stream_prompt` setup timeout, call `self.client.cancel(acp_session_id.clone()).await` before returning the error stream; log but do not mask the timeout if cancel fails. For streaming timeout, wrap the receiver stream in an async stream/unfold or helper that can await a single best-effort cancel before yielding `OpFailed`. Ensure exactly one terminal event is emitted. |
