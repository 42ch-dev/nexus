## Goal

Close the V1.16 compatibility gap for Creator identity selection/display and Creator-context authentication behavior, with pragmatic CLI-focused regressions rather than new auth infrastructure.

## Success criteria

This plan satisfies Delivery Compass B4 Done items:

1. `creator use <creator_id_or_handle>` persists a normalized `creator_id`.
2. `creator status` and `creator list` stably display the three-layer identity model: `creator_id`, `handle`, `display_name`.
3. User-path and Creator-path authentication behavior follows `auth-session-model-v1`: Creator-context HTTP uses either `Authorization: Bearer <creator_api_key>` or `Authorization: Bearer <user_access_token>` plus `X-Creator-Id: <creator_id>`.

## Current state analysis

### `creator use`

Current implementation: `crates/nexus42/src/commands/creator.rs` has `use_creator(_config, creator_ref)` that:

- loads `CliConfig`,
- writes `cli_config.active_creator_id = Some(creator_ref.to_string())`,
- removes that exact `creator_ref` key from `active_workspace_slug_by_creator`,
- saves the config,
- prints `✓ Active Creator set to: {creator_ref}`.

Implications:

- It accepts any string and persists it verbatim.
- It does **not** resolve a handle to `creator_id`.
- It does **not** validate that the persisted value is path-safe or matches the Creator ID pattern.
- If the user passes a handle, subsequent `creator soul|memory|kb|workspace` paths bind to the handle string as though it were a `creator_id`, which violates the V1.16 rule that local knowledge subcommands bind only to canonical `creator_id`.

### `creator status`

Current implementation: `creator_status(config, creator_id)` resolves the requested ID or active config value, then prints:

```text
Creator: <id>
  Auth: ✓ Token cached | ✗ No cached token

⚠ V1.0 skeleton: full status requires daemon + platform API.
```

Implications:

- It displays only one identifier string.
- It does **not** show `handle` or `display_name`.
- It treats cached `access_token` as the only auth indicator; `creator_api_key` stored by registration is not reflected.

### `creator list`

Current implementation: `list_creators(_config)` prints only the active creator ID from `CliConfig` and a skeleton warning.

Implications:

- It does not enumerate the local creator auth cache.
- It does not show `handle` or `display_name`.
- It cannot support `creator use <handle>` because there is no local handle index.

### Config storage

`crates/nexus42/src/config.rs` stores active Creator as:

```rust
pub active_creator_id: Option<String>
```

This field is the right canonical storage slot. The compatibility fix should keep this field as `creator_id` only and add any alias/display metadata outside the field.

### Creator identity types and contracts

- `crates/nexus-domain/src/creator.rs` defines `Creator { creator_id, user_id, display_name, ... }` with no `handle` field.
- `schemas/domain/creator.schema.json` and generated `crates/nexus-contracts/src/generated/creator.rs` likewise expose `creator_id` and `display_name`, but no `handle`.
- `crates/nexus-sync/src/platform_client.rs` sends `handle` in `RegisterRequest`, but `RegisterResponse` does not carry it back.

Implication: the current OSS wire/domain contract cannot provide three-layer status/list output by itself. For V1.16, the least disruptive path is a CLI-local metadata cache for `handle` + `display_name` keyed by canonical `creator_id`, fed by registration inputs and local auth cache. A future schema/platform follow-up can add `handle` to shared contracts if the platform contract is formally updated.

### Auth paths

User auth path:

- `nexus42 auth login` uses device flow and stores `AuthStore.user_token`.
- `auth::user_auth::ensure_valid_token(config)` returns a valid/refreshed user access token.
- Creator-context requests made with this token must also send `X-Creator-Id` matching the body/path creator ID.

Creator auth path:

- Creator registration stores `creator_api_key` in `AuthStore.creators[creator_id].creator_api_key`.
- `auth::creator_auth::ensure_valid_token` currently checks/rotates `CreatorAuthState.access_token`, but credential rotation is a skeleton and does not use `creator_api_key` as the actual bearer credential.
- No shared helper currently selects Creator-path vs User-path headers for Creator-context HTTP clients.

Implication: this plan should not build new auth infrastructure. It should introduce a small, testable CLI-side header-selection helper and use it where Creator-context clients are assembled in this repo, while preserving existing device-flow and registration behavior.

## Selected approach

1. Keep `CliConfig.active_creator_id` as the canonical persisted active Creator ID.
2. Add a small CLI-owned Creator identity cache in `~/.nexus42` for display/alias metadata, keyed by `creator_id`:

   ```json
   {
     "creators": {
       "ctr_abc": {
         "creator_id": "ctr_abc",
         "handle": "my-handle",
         "display_name": "My Creator"
       }
     }
   }
   ```

   Recommended file: `creator-identities.json` next to `auth.json` and `config.toml`. Keep it CLI-local and pre-1.0 disposable.

3. On `creator register`, cache `creator_id`, the user-provided `handle` if present, and `display_name` from the platform response.
4. On `creator use <creator_id_or_handle>`, resolve the input against the local cache:
   - exact `creator_id` match → persist that ID;
   - exact `handle` match → persist the matched `creator_id`;
   - no match but input looks like a valid/path-safe Creator ID → persist as explicit ID for backward compatibility;
   - otherwise return a clear error telling the user to run `creator list` or use a `creator_id`.
5. Update `creator status/list` to render a stable table/fields containing all three identity layers. When `handle` or `display_name` is unknown, print `-` rather than omitting the column.
6. Add a minimal auth-header helper for Creator-context requests:
   - prefer `creator_api_key` for the active `creator_id` and emit only `Authorization: Bearer <creator_api_key>`;
   - otherwise use a valid User token and emit `Authorization: Bearer <user_access_token>` plus `X-Creator-Id: <creator_id>`;
   - ensure body/path creator ID mismatch checks remain with callers/tests where applicable.

## Suggested regression cases

### Identity normalization

1. Seed cache with `{ creator_id: "ctr_alpha", handle: "alpha", display_name: "Alpha Creator" }`.
2. Run `creator use alpha`.
3. Assert `config.toml` contains `active_creator_id = "ctr_alpha"`, not `alpha`.

### Three-layer output

1. Seed cache with one complete identity and one ID-only identity.
2. Assert `creator list` contains columns/labels for `creator_id`, `handle`, and `display_name`.
3. Assert missing handle/display name are represented as `-`.
4. Assert active row is visibly marked.

### Auth compatibility

1. With `AuthStore.creators[ctr_alpha].creator_api_key = "nexus_live_alpha"`, assert Creator-context headers contain `Authorization: Bearer nexus_live_alpha` and do **not** require `X-Creator-Id`.
2. Without `creator_api_key` but with valid `AuthStore.user_token`, assert headers contain `Authorization: Bearer <user_token>` and `X-Creator-Id: ctr_alpha`.
3. With neither credential, assert a clear authentication-required error.
4. If a caller supplies a body/path creator ID, assert mismatch with active/header creator ID is rejected before sending.

## Validation commands

Use the narrowest checks first, then widen:

```bash
cargo test -p nexus42 creator_identity
cargo test -p nexus42 creator_context_auth
cargo test -p nexus42 --test creator_register_e2e
cargo +nightly fmt --all
cargo clippy --all -- -D warnings
```

If existing flaky tests involving shared `$HOME/.nexus42/auth.json` appear, record them against existing residuals rather than mixing them into this plan’s acceptance.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| `handle` is not present in shared Creator schema | Use CLI-local cache only for V1.16; do not hand-edit generated contracts. |
| Local cache can become stale relative to platform | Treat cache as best-effort display/alias data; explicit `creator_id` remains accepted. |
| Secret leakage in status/list/auth tests | Never print `creator_api_key`; report only present/absent or masked prefixes if needed. |
| User token vs Creator API key ambiguity | Centralize header selection in one helper and cover both branches with tests. |

## Non-goals

- Full platform `GET /api/v1/creators` integration for `creator list`.
- New OAuth, pairing, credential-rotation, or daemon auth infrastructure.
- Schema/codegen changes for `Creator.handle` unless separately planned.
- Secret storage hardening beyond the current pre-1.0 local persistence model.

## Handoff notes

- Preserve pre-1.0 pragmatism: the goal is stable CLI behavior and regression coverage.
- Keep all local knowledge subcommands bound to canonical `creator_id`; never route by `display_name`.
- Prefer small helpers and tests over broad refactors of `creator.rs`.
