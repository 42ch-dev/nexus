## 1) Purpose and SSOT boundary

This document is the **implementation-period SSOT** for V1.4 planning decisions and execution guardrails for the `nexus` OSS repository. It translates the 2026-04-17 orchestration brainstorming outcomes into waves, milestones, acceptance criteria, and cross-cutting policy (including the long-deferred `schemas/` boundary refactor that V1.4 finally absorbs into a program track).

- Authoritative for: V1.4 scope classification, wave decomposition, milestone gates, minimum regression gate, cross-repo dependency boundaries, risk controls.

## 2) Confirmed decisions (locked)

Outputs of the 2026-04-17 orchestration brainstorming (see answer traces Q1–Q5):

- **Strategy shape**: hierarchical — outer long-task state machine + inner short DAG ("graph-of-graphs").
- **Daemon role**: `nexus42d` becomes **orchestration engine + capability registry**. Existing HTTP-era capabilities (sync, outbox, workspace ops, registry refresh) are reclassified as first-class registered capabilities; HTTP surface retreats to a trigger/query layer over the same engine.
- **ACP host locus**: Extract `crates/nexus-acp-host` (worker-only linkage); `nexus42d` never links `agent-client-protocol`; orchestration-driven ACP sessions run inside **per-creator long-lived `nexus42 acp-worker` subprocess**; interactive `nexus42 agent run` unchanged.
- **Daemon ↔ worker IPC**: parent↔child stdin/stdout JSON-RPC 2.0 (owned-pipe graceful shutdown).
- **Transition mechanism**: rule-first + LLM-judge fallback + user manual override (`nexus42 schedule advance`).
- **Preset physical form**: filesystem folder bundle (YAML manifest + Markdown prompt templates + optional local schemas).
- **Orchestration runtime library**: `graph-flow = "=0.2.3"` (rs-graph-llm) — both outer SM and inner DAG, with custom SQLite `SessionStorage`, behind an adapter trait; `rig` feature **not** enabled.
- **Daemon lifecycle library**: `statig = "0.3"` — 6-state HSM closing TD-9.
- **Libraries are orthogonal**: statig for compile-time daemon lifecycle; graph-flow for dynamic preset execution; **no overlap, no dep cycle**.
- **`schemas/` boundary refactor is V1.4 scope** (was "parallel small plan" C-track; now formally folded in as WS5 — see §4 below).
- **`schemas/` wire rule (tightened 2026-04-17)**: `schemas/` hosts **only** what `nexus-platform` actually observes over the wire. Local-only types (including CLI↔daemon local HTTP, daemon↔worker IPC, ACP registry manifest, daemon-status-v2, orchestration runtime types, Schedule / core_context types) live as hand-written Rust in `crates/nexus-contracts/src/local/`. See [schemas-boundary-v1.md](schemas-boundary-v1.md) for the full rule + audit procedure.
- **B-track is folded into V1.4 as WS7** (user decision 2026-04-17, reversing the initial "deferred" call). V1.4 ships Creator Schedule (multi-Schedule per creator, concurrency declared per-add, CRUD by ID) and immutable versioned `core_context` (seed + user edits + preset hooks). The LLM-driven iteration mechanism is **not** implemented in V1.4 but the data model reserves `DerivationKind::LlmSummarize` so V1.5 can add a `context.summarize` capability with zero schema migration. See [creator-schedule-and-core-context-v1.md](creator-schedule-and-core-context-v1.md) for the full spec (incl. forward-compat rationale in its §11).
- **Planning strategy**: waves ordered by dependency graph; WS5 (schemas refactor) parallelisable with WS2 (orchestration skeleton); WS4 (statig lifecycle) parallelisable with WS2; WS7 (Schedule + core_context) lands after WS2/WS3 (depends on engine + preset loader).
- **Branch strategy**: `feature/v1.4-orchestration` from `main` as plan integration branch; per-wave topic branches off the integration branch when ≥2 concurrent writers.
- **Dependency policy (2026-04-17)**: crate selection for any third-party Rust dependency introduced in V1.4 (JSON-RPC IPC, orchestration SessionStorage, JWT validation, file watcher guidance, cron deferral, etc.) follows [crate-selection-best-practices-v1.md](crate-selection-best-practices-v1.md). Binding for V1.4 are: (a) worker IPC = `jsonrpsee-core` + `tokio_util::codec::LinesCodec` NDJSON framing + `RpcTransport` trait (no hand-rolled framing); (b) **DB engine unified on `sqlx` across the workspace** — see next bullet; (c) no `oauth2` crate introduced in V1.4 — platform auth stays on `jsonwebtoken` only; (d) cron / wall-clock triggers remain **deferred to V1.5+** per orchestration-engine-v1 and the four hard constraints recorded in the best-practices doc §3.7.
- **DB engine unification (new, 2026-04-17, WS8)**: `nexus-local-db` / `state.db` migrates from `rusqlite` + `deadpool-sqlite` + bespoke sequential migrations to **`sqlx` (sqlite + runtime-tokio + macros + migrate)**. This is a **hard prerequisite for WS2 Task 3** (`SqliteSessionStorage` shares the same pool on the same `state.db`, not a separate DB file). WS8 is a dedicated workstream (plan row `2026-04-17-v1.4-ws8-local-db-sqlx-migration`); `rusqlite` / `deadpool-sqlite` removed from `[workspace.dependencies]` on WS8 T9. PM-approved 2026-04-17 after SSOT conflict review (dual-engine same-file pattern rejected as unstable; dual-file pattern rejected as permanent operational tax).

## 3) Version reclassification compass (V1.4 in-tree)

This compass covers V1.4 only. V1.2 → [v1.2-delivery-compass-v1.md](v1.2-delivery-compass-v1.md); V1.3 → [v1.3-delivery-compass-v1.md](v1.3-delivery-compass-v1.md); cross-version mapping matrix → [v1.2-reclassification-matrix-v1.md](v1.2-reclassification-matrix-v1.md) (includes V1.4 column).

### V1.4 (Orchestration + schemas boundary)

In (this compass):

- **Orchestration foundation** — daemon engine + capability registry + worker model + preset bundle + first user-facing preset end-to-end.
- **TD-9 closure** — `statig` 6-state daemon lifecycle HSM + v2 `GET /v1/local/daemon/status` superset response.
- **`nexus-acp-host` crate extraction** — ACP client logic moves out of `crates/nexus42` into its own crate (worker-only linkage enforced by CI).
- **DB engine unification on `sqlx`** — `nexus-local-db` / `state.db` fully migrates from `rusqlite` + `deadpool-sqlite` + bespoke sequential migrations to `sqlx` + `sqlx migrate`; 34 files / ~193 references ported; orchestration `SessionStorage` consumes the same pool on the same file (WS8 → WS2 T3 dependency).
- **`schemas/` boundary refactor** — audit every `schemas/**/*.schema.json` file; keep only what `nexus-platform` observes over the wire; move local-only types into `crates/nexus-contracts/src/local/` (hand-written Rust, no codegen); see [schemas-boundary-v1.md](schemas-boundary-v1.md).
- **Creator Schedule + `core_context` (WS7, new in V1.4)** — multi-Schedule per creator (concurrency declared per-add), CRUD by ID, immutable versioned `core_context` (seed + user edits + preset `context_update` hooks), `nexus42 schedule add/edit/list/inspect/context/context-history/start/pause/resume/cancel/advance/remove/timeline` CLI + `/v1/local/orchestration/schedules/*` HTTP. LLM iteration mechanism reserved in data model but not emitted in V1.4 (V1.5 lights up `context.summarize` capability zero-migration). See [creator-schedule-and-core-context-v1.md](creator-schedule-and-core-context-v1.md).
- **Rescoped ACP residuals** — ACP-R6 (session persistence), ACP-R7 plumbing (permission policy I/O path), ACP-R8 (daemon-mediated tool access, re-scoped), ACP-R11 (session modes) — status changes tracked in [acp-client-tech-spec-v2.md](acp-client-tech-spec-v2.md) Appendix B.
- **V1.3 backlog** — `V1.3 Creator Register — E2E Integration Tests (T6)` (currently the sole `Todo` in `status.json`) — executed in V1.4 as a non-orchestration lane (platform M10 dependency).

Out (deferred):

- **LLM-driven `core_context` summarisation / auto-iteration** — `context.summarize` capability that emits `DerivationKind::LlmSummarize`; V1.4 reserves the variant, V1.5+ implements the capability. See [creator-schedule-and-core-context-v1.md](creator-schedule-and-core-context-v1.md) §11.
- **Schedule cron / wall-clock triggers** — `scheduled_at` column reserved in V1.4; V1.5 adds the clock poller zero-migration.
- **User-authored capabilities** (shell / WASM plugin ABI) — V1.5+.
- **Preset third-party registry / signing / publish** — V1.5+.
- **Multi-agent worker** (single worker hosting >1 ACP agent) — V1.5+.
- **ACP SDK migration** to `sacp` v1.0 — governed by adapter-trait policy; no forced move in V1.4.
- **Full ACP permission policy engine UI** — only plumbing in V1.4; UI + decision engine in V1.5.
- **Preset hot-reload with in-flight session migration** — V1.4 uses snapshot semantics (running sessions keep their preset snapshot; new sessions pick up the new preset).

Backlog (discovered during V1.4 brainstorming, park for later):

- System-managed multi-preset scheduler for `_system.maintenance` family (if V1.4 system Schedule grows beyond the embedded single preset).
- Capability schema registry sharing with platform (cross-repo capability catalog sync) — evaluate after V1.4 ships.

## 4) V1.4 workstreams (execution decomposition)

Mapped 1:1 to [orchestration-engine-v1.md](orchestration-engine-v1.md) §10 phases; WS5 is the compass's addition.

### WS1 — `nexus-acp-host` crate extraction (orchestration Phase 1)

- **Goal**: Extract ACP client logic out of `crates/nexus42/src/acp/` into new crate `crates/nexus-acp-host`; `nexus42d` crate never links it (CI-enforced linkage gate); hidden `nexus42 acp-worker` subcommand stub lands.
- **Authoritative design**: [acp-client-tech-spec-v2.md](acp-client-tech-spec-v2.md) §11.
- **Deliverables**: new crate scaffold; migrated source files; updated `Cargo.toml`s; re-exports where needed; CI `verify-no-acp-in-daemon` gate; hidden `acp-worker` subcommand returning initialize OK.
- **Evidence**: `cargo build --workspace` clean; `cargo test --workspace` green (existing ACP tests move with crate); interactive `nexus42 agent list/show/probe/run` outputs unchanged vs pre-migration; linkage gate CI job passes; `acp-worker --creator <id>` runs and exits cleanly.

### WS2 — Orchestration skeleton (orchestration Phase 2)

- **Goal**: Introduce the runtime plumbing of the engine + capability registry + Worker Manager + daemon↔worker IPC; register non-ACP built-in capabilities; land `_system.maintenance` hardcoded graph (not yet via file loader); add orchestration HTTP endpoints.
- **Authoritative design**: [orchestration-engine-v1.md](orchestration-engine-v1.md) §§3–6, §10.3.
- **Dependency**: **WS8 is a hard prerequisite for WS2 Task 3** (`SqliteSessionStorage`) — the orchestration pool consumes `nexus-local-db`'s `Arc<SqlitePool>` on the unified `state.db`. WS2 Tasks 1–2 + 4–10 MAY run in parallel with WS8 if convenient, but Task 3 waits.
- **IPC crate selection**: `jsonrpsee-core` + custom `RpcTransport` trait + NDJSON via `tokio_util::codec::LinesCodec` — see [crate-selection-best-practices-v1.md](crate-selection-best-practices-v1.md) §3.1. (Decision closed; WS2 Task 5 is implementation, not selection.)
- **Deliverables**: new crate `crates/nexus-orchestration`; `OrchestrationEngine` trait + `GraphFlowEngine` impl; `SqliteSessionStorage` (sqlx, shares `nexus-local-db`'s pool; `orchestration_sessions` added as one more `.sql` migration under `crates/nexus-local-db/migrations/`); `Capability` trait + registered built-ins (`sync.*`, `outbox.*`, `workspace.*`, `registry.refresh`, `creator.*` non-ACP); Worker Manager lifecycle; `worker/initialize`, `worker/health`, `worker/shutdown` IPC verbs (jsonrpsee-core); `/v1/local/orchestration/*` HTTP endpoints.
- **Evidence**: engine can create/step/pause/resume/cancel a hardcoded 3-state test graph; roundtrip test (start → kill daemon → restart → resume) passes; Worker Manager integration test with dummy worker script; `GET /v1/local/orchestration/sessions` returns `_system.maintenance` session when engine is running; clippy/fmt clean.

### WS3 — Preset loader + first user preset end-to-end (orchestration Phase 3)

- **Goal**: Enable filesystem preset bundles; register ACP-related capabilities (`acp.*`, `judge.llm`, `judge.rule`); implement `AcpPromptTask` dispatching through Worker Manager; ship embedded `novel-writing` preset as E2E demonstrator; add user-facing `nexus42 schedule start/status/advance` CLI (minimal, not B-track full).
- **Authoritative design**: [orchestration-engine-v1.md](orchestration-engine-v1.md) §§7–9, §10.4.
- **Deliverables**: `load_preset()` with §7.6 validation; bundled preset `embedded-presets/novel-writing/`; `AcpPromptTask` + `JudgeTask` + `InnerGraphTask` + `ManualWaitTask` implementations; CLI commands `schedule start` / `schedule status` / `schedule advance`; at least one tool-policy enforcement path (`auto_grant_read_only`).
- **Evidence**: `nexus42 schedule start novel-writing --creator <id>` returns a session id; mocked-agent E2E test advances through all 4 outer states and the `brainstorm_graph` inner DAG; daemon restart mid-`brainstorming` resumes from last completed inner-graph node; `nexus42 schedule advance` progresses from `outlining` → `drafting`.

### WS4 — Daemon lifecycle HSM (orchestration Phase 4) — **parallel with WS2**

- **Goal**: Replace the v1 "running-probe" shim with the full 6-state `statig` HSM, closing TD-9.
- **Authoritative design**: [daemon-lifecycle-api-v2.md](daemon-lifecycle-api-v2.md).
- **Deliverables**: new `crates/nexus42d/src/lifecycle/` module; `Lifecycle` trait + `StatigLifecycle` impl; entry/exit actions; SIGINT/SIGTERM integration; panic-hook bridge; v2 `GET /v1/local/daemon/status` (with `schema_version: 2`, `subsystems{}`, `degraded{}`); new `schemas/acp-runtime/daemon-status-v2.schema.json` (wire) + regenerated contracts.
- **Evidence**: integration test — cold start → `Running` within threshold; SIGTERM → `Stopping` → exit 0 within grace; induced subsystem failure → `Degraded` with populated fields; `FatalError` → non-zero exit; backward-compat test: v1 clients still get `lifecycle_state: "running"` on healthy daemon.

### WS5 — `schemas/` boundary refactor (**parallel with WS2; no dependency on WS1/WS3/WS4**)

- **Goal**: Enforce the rule "`schemas/` hosts only what `nexus-platform` actually observes over the wire"; move local-only types into hand-written Rust under `crates/nexus-contracts/src/local/` so `pnpm run codegen` stops emitting TypeScript shapes platform never consumes.
- **Authoritative policy and audit procedure**: [schemas-boundary-v1.md](schemas-boundary-v1.md) — see §2 for the wire/local rule, §3 for the placement convention (`crates/nexus-contracts/src/local/`), §5 for the audit procedure + template, §6 for the initial classification sketch, §7 for step-by-step migration.
- **Alignment with v1-spec**: per `{v1-spec}/schema/codegen-strategy-v1.md` §3, this boundary decision is explicitly delegated to the contract repo (nexus OSS). No v1-spec update required for V1.4; an optional footnote pointing to `schemas-boundary-v1.md` may be added to the platform-side codegen-strategy spec post-V1.4 (see `schemas-boundary-v1.md` §4.2).
- **Deliverables** (all details in [schemas-boundary-v1.md](schemas-boundary-v1.md) §7):
  - Audit table populated to 100% coverage of `schemas/**/*.schema.json`.
  - Local schemas moved from `schemas/**` to `crates/nexus-contracts/src/local/<module>/*.rs` (hand-written Rust + `serde`; no JSON Schema, no codegen).
  - `packages/nexus-contracts` **minor** version bump with release note.
  - Cross-repo handoff note posted in `nexus-platform` tracker.
- **Evidence**: see [schemas-boundary-v1.md](schemas-boundary-v1.md) §8 acceptance criteria (codegen clean, clippy clean, validate-schemas clean, npm publish green, platform-side regression green on next contracts upgrade).

### WS6 — V1.3 backlog closure (non-orchestration lane)

- **Goal**: Close the single remaining V1.3 backlog item (`V1.3 Creator Register — E2E Integration Tests (T6)` per `.agents/plans/status.json`) opportunistically in V1.4.
- **Dependency**: platform M10 must be complete (creator register + verify API endpoints reachable in integration-test environment).
- **Deliverables**: integration tests in `crates/nexus42/tests/` with `wiremock` or platform test harness; covers token lookup → platform register → challenge solve → verify → credential store.
- **Evidence**: tests pass against mocked platform; residual row removed from `status.json.plans[]`.

### WS7 — Creator Schedule + `core_context` (depends on WS2 + WS3)

- **Goal**: Ship multi-Schedule per creator, CRUD by ID, and immutable versioned `core_context` so users can (1) inject preset runs by name, (2) see what the creator plans to do, (3) edit / append / remove prompts without destabilising in-flight execution.
- **Authoritative design**: [creator-schedule-and-core-context-v1.md](creator-schedule-and-core-context-v1.md) — see that doc for data model (§3), lifecycle (§4), concurrency rules (§5), `core_context` derivation (§6), preset YAML additions (§7), CLI surface (§8), HTTP surface (§9), SQLite schema (§10), and the V1.5-forward-compat rationale (§11).
- **Key scope decisions locked (2026-04-17)**:
  - Multi-Schedule per creator **is** supported; concurrency declared per-add (`--parallel-with <id>` / `--parallel-any` / default serial).
  - No priority / preemption in V1.4 (explicit `schedule pause/resume/cancel` only).
  - `core_context` is **immutable-versioned** with a `DerivationStep` trace; user can query history.
  - V1.4 emits derivation kinds `Seed`, `UserEdit`, `PresetHook`, `PresetSeedExpansion` (the last opt-in per preset).
  - V1.4 **reserves** `DerivationKind::LlmSummarize` in the data model but does not implement the capability; V1.5 adds `context.summarize` capability with zero schema migration.
  - Trigger model in V1.4 is on-demand only; `scheduled_at` column reserved for V1.5 cron.
- **Deliverables**:
  - Rust types under `crates/nexus-contracts/src/local/schedule/` (per WS5 convention, hand-written, not codegen'd).
  - `ScheduleSupervisor` module in `crates/nexus-orchestration/`.
  - SQLite migration adding `creator_schedules`, `schedule_dependencies`, `core_context_versions` tables.
  - `nexus42 schedule` CLI command group (13 subcommands per [creator-schedule-and-core-context-v1.md](creator-schedule-and-core-context-v1.md) §8).
  - `/v1/local/orchestration/schedules/*` HTTP endpoints (8 endpoints per §9).
  - Preset YAML additions (`initial_action`, per-state `context_update`) — existing V1.4 presets updated to declare defaults explicitly (no behavioural change).
  - Demonstrator: `novel-writing` preset gets a `context_update` hook on the `outlining` state so the `drafting` state sees edited context.
- **Evidence**:
  - R11 (regression gate, §6 below): `schedule add --preset novel-writing --seed ...` → `schedule inspect <id>` roundtrip; `core_context` v0 equals seed.
  - R12: mid-execution `schedule edit` creates v1; running state finishes on v0; next state reads v1.
  - R13: preset `context_update` hook writes a `PresetHook` version at state exit; template at next state renders with updated `{{core_context.*}}`.
  - R14: dependency chain `A → B` auto-advances when A completes.
  - `cargo test --workspace` / `clippy` / `fmt` clean.

### WS8 — `nexus-local-db` sqlx migration (unified DB engine) — **parallel with WS1/WS4; blocks WS2 Task 3**

- **Goal**: Migrate the entire workspace from `rusqlite` + `deadpool-sqlite` + bespoke sequential migrations to `sqlx` + `sqlx migrate`, so that `state.db` has a single engine, a single pool implementation, and a single migration runner — enabling WS2 Task 3 to share that pool directly.
- **Authoritative design**: [crate-selection-best-practices-v1.md](crate-selection-best-practices-v1.md) §2.3 + §3.3 (decision SSOT); [local-db-refactor-v1.md](local-db-refactor-v1.md) (to be revised to v2 at WS8 T9).
- **Scope**: ~34 files / ~193 `rusqlite` / `deadpool-sqlite` references across `nexus-local-db`, `nexus42`, `nexus42d`, `nexus-sync`. Four existing code-form migrations (v2/v3/v4 + implicit baseline) are ported to timestamped `.sql` files under `crates/nexus-local-db/migrations/`. `workspace_meta.db_schema_version` key preserved for backwards compat (mirrored from sqlx's `_sqlx_migrations`).
- **Deliverables**: new `crates/nexus-local-db/migrations/*.sql` (5 files incl. version-sync); sqlx pool bootstrap (`open_pool`, `run_migrations`); all CRUD modules async-ported; all consumer callsites `.await`-chained; `sqlx-cli` offline metadata committed under `.sqlx/`; CI `SQLX_OFFLINE=true` + `cargo sqlx prepare --check`; `rusqlite` + `deadpool-sqlite` removed from `[workspace.dependencies]`; `local-db-refactor-v2.md` + archived v1.
- **Evidence**: every V1.2/V1.3-landed scenario (SOUL lifecycle, memory pipeline, local-only identity, challenge solver, creator register) still passes its existing test; `rg 'rusqlite|deadpool[_-]sqlite' crates/` returns zero hits; `sqlx migrate run` idempotent on fresh DB; R15 regression row (§6) green.

## 8) Risk compass

### RISK-01 — Orchestration scope creep **past** V1.4's Schedule MVP

- **Symptom**: tasks start depending on LLM-driven `core_context` summarise, cron triggers, multi-agent workers, or user-authored capabilities — all explicitly deferred past V1.4.
- **Control**: enforce §4 In/Out matrix at wave entry; point at [creator-schedule-and-core-context-v1.md](creator-schedule-and-core-context-v1.md) §13 "Open Questions Deferred Past V1.4" as the authoritative deferral list.

### RISK-02 — `graph-flow` 0.2.x crate drift during V1.4 build

- **Symptom**: upstream releases a 0.3.x with API changes mid-implementation.
- **Control**: pin `=0.2.3` in `Cargo.toml`; `OrchestrationEngine` trait isolates callers; if upstream breaks, upgrade in a **separate plan** — V1.4 does not chase version churn.

### RISK-03 — LocalSet contagion into `nexus42d` runtime (regression)

- **Symptom**: some refactor accidentally links `agent-client-protocol` from `nexus42d` or a shared downstream crate, re-introducing `!Send` contagion.
- **Control**: CI `verify-no-acp-in-daemon` job (R9) runs on every PR; failing this gate blocks merge.

### RISK-04 — WS5 audit incompleteness

- **Symptom**: some `schemas/domain/*.schema.json` gets mis-classified as local and removed from `schemas/`, breaking platform.
- **Control**: WS5 publishes audit table for per-file review **before** any file move; `rg` on `nexus-platform` for each removal candidate; npm bump reviewed by at least one reviewer familiar with platform consumers; post-merge, watch `nexus-platform` CI for green status on the next contracts upgrade.

### RISK-11 — WS8 regression surface (DB engine swap touches 34 files)

- **Symptom**: a V1.2/V1.3-landed feature (SOUL lifecycle, memory pipeline, local-only identity, challenge solver, creator register) silently regresses because the rusqlite→sqlx port introduces subtle behaviour changes (transaction semantics, error mapping, row counts, `RETURNING` support).
- **Control**: WS8 Task-level acceptance mandates "every existing test for the touched module passes after the port" — no new tests needed to advance the plan, but **no existing test may be weakened or skipped**. QC triple-review at WS8 T10 specifically diffs `sqlite3 .schema` output pre- vs post-migration on a real workspace DB to confirm schema identity. Reviewer ALSO runs the CLI smoke suite (`nexus42 init / doctor / soul / identity / creator`) against a rebuilt binary.

### RISK-12 — `db_schema_version` key drift between sqlx metadata and `workspace_meta`

- **Symptom**: external readers (e.g. `nexus42 doctor`) consume `workspace_meta.db_schema_version`, but sqlx tracks migration state in its own `_sqlx_migrations` table. If a new migration is added and the `.rs` constant is updated but the value-sync migration is not, the two sources diverge silently.
- **Control**: WS8 T2 delivers `20260417_000005_workspace_meta_schema_version_sync.sql` as the single seed. Any future migration that advances the logical DB version MUST include an `INSERT OR REPLACE` update to the `db_schema_version` key in its own `.sql` file, and update `crates/nexus-local-db/src/version.rs::DB_SCHEMA_VERSION` constant in the same PR. Reviewer checklist (added to AGENTS.md at WS8 T9) enforces this.

### RISK-05 — Worker IPC protocol instability

- **Symptom**: first-release IPC verbs undergo churn as WS3 integration surfaces edge cases; future presets / external contributors depend on an unstable shape.
- **Control**: document IPC catalogue in [orchestration-engine-v1.md](orchestration-engine-v1.md) §6.4 as v1 explicitly; breaking changes bump a minor in the daemon↔worker protocol version handshake declared on `worker/initialize` reply; no external-author exposure in V1.4. IPC implementation crate-selection (jsonrpsee-core + `RpcTransport` trait) is locked in [crate-selection-best-practices-v1.md](crate-selection-best-practices-v1.md) §3.1 — the trait is the explicit insurance layer if jsonrpsee-core ever needs replacement.

### RISK-06 — V1.4 wall-clock drag if WS5 audit uncovers deep wire/local entanglement

- **Symptom**: some "obvious" local-only domain type turns out to be transmitted in an obscure bundle path, forcing redesign instead of simple move.
- **Control**: WS5 first task is **audit only** (no file moves); if the audit reveals >30% of `schemas/domain/*` needs to stay wire, WS5 is **rescoped mid-flight** to close only the clearly-local subset, with the remainder moved to a V1.5 successor plan.

### RISK-07 — TD-9 parity with cli-spec §10.1 drifts during implementation

- **Symptom**: implementer abbreviates the state graph or skips `Degraded` semantics for expediency.
- **Control**: [daemon-lifecycle-api-v2.md](daemon-lifecycle-api-v2.md) is wave-0 spec; QC triple-review must cite §2–§6 of that doc when validating WS4 PR.

### RISK-08 — Preset bundle format churn after first external author

- **Symptom**: once external authors try the YAML schema, corner cases (Handlebars edge escapes, variable scoping) force breaking bundle format changes.
- **Control**: V1.4 ships **only embedded presets** (`_system.maintenance`, `novel-writing`); no third-party distribution story; format is `preset.version: 1` — format `v2` can be introduced non-breakingly because loaders can dispatch by `version` key.

### RISK-09 — `core_context` versioning blob size growth

- **Symptom**: a long-running Schedule accumulates hundreds of `core_context_versions` rows with large `content` blobs, bloating the SQLite database.
- **Control**: [creator-schedule-and-core-context-v1.md](creator-schedule-and-core-context-v1.md) §10 table `core_context_versions` is keyed `(schedule_id, version)` with an index; V1.4 does not enforce retention; a follow-up capability `schedule.compact <id> --keep-last N` is listed as a V1.5 chore (open but not a V1.4 gate). Monitor via SQLite size metrics post-launch.

### RISK-10 — Forward-compat promise for V1.5 LLM summarise silently breaks

- **Symptom**: a V1.4 refactor (e.g. moving `DerivationStep` enum) makes the reserved `LlmSummarize` variant unreachable or changes its serialized form, forcing a V1.5 migration.
- **Control**: [creator-schedule-and-core-context-v1.md](creator-schedule-and-core-context-v1.md) §11 declares the forward-compat contract explicitly; unit test `core_context_versions_parse_unknown_derivation_kind_safely` must exist to confirm the enum's `#[non_exhaustive]` handling; CI guards against variant deletion.
