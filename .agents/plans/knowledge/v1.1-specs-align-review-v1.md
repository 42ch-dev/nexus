## 3. Detailed findings

### 3.1 Entitlements / quota wire — **resolved (2026-04-10)**

- **Was**: Gap vs `entitlements-wire-v1.md` / codegen-strategy §6.5 — no dedicated JSON Schemas in nexus.  
- **Now**: `schemas/platform/me-entitlements-response.schema.json`, `schemas/platform/official-creator-quota-response.schema.json`; codegen emits `MeEntitlementsResponse` / `OfficialCreatorQuotaResponse`; `@42ch/nexus-contracts` bumped (see root `AGENTS.md` table).  
- **Remaining**: Platform must **consume** generated types and keep HTTP handlers aligned (cross-repo).

### 3.2 Knowledge / overview SSOT — **largely resolved (2026-04-10)**

- [architecture-alignment-review-v1.md](architecture-alignment-review-v1.md) §2.2 and enum narrative updated for User / ForkBranch / TD-2 / TD-3 closure.  
- [v1.1-overview-v2.md](v1.1-overview-v2.md) §2–§3 refreshed from `status.json` (counts, Done vs Todo, residuals).  
- **Ongoing rule**: after material `status.json` edits, re-run §2–§3 refresh per `v1.1-overview-v2.md` §7.

### 3.3 SQLite path vs cli-spec — **still a documented deferral**

- **Implementation**: `state_db_path()` → `~/.nexus42/state.db` (single file under nexus home).  
- **Spec tension**: cli-spec / local-db narrative anticipates workspace-scoped layout for **multiple worlds/workspaces** without path collisions.  
- **Interpretation**: Acceptable for **single active workspace** era; promoting **multi-workspace** requires ADR + `nexus-local-db` / daemon path strategy — **not** automatically a “V1.1 plan row” but a **product/version** decision.

### 3.4 Domain coverage vs data-model — **intentional thin domain for sync plane**

- Sync aggregates exist on the wire and in `nexus-sync`; absence from `nexus-domain` is a **layering choice**. Risk is **onboarding confusion** (readers think “missing”) — mitigated by this doc + architecture-alignment summary.

### 3.5 Plan 21 — **checklist only in OSS**

- Artifact: [plan-21-openapi-freeze-checklist.md](../reports/2026-04-10-v1-spec-openapi-freeze-checklist/plan-21-openapi-freeze-checklist.md).  
- **Done criterion** lives in **platform** OpenAPI / route governance, not in nexus `git diff`.

### 3.6 Codegen / consumer sharpness (from coverage matrix)

- Explore **feed** entries may remain `serde_json::Value` until a stricter hit shape is frozen.  
- **TS**: nested inline objects may duplicate enum literals vs `common.schema.json`.  
- **Rust**: some schema `enum` strings still map to plain `String` depending on generator path.  
These are **engineering polish** items; they do not by themselves mean “V1.1 OSS incomplete” if product accepts current wire strictness.

### 3.7 Test hygiene — **verified warning (pre-existing)**

- `crates/nexus42/tests/acp_registry.rs` imports `serde_json::Value` and `tempfile::TempDir` with **no other references** in file (only `use` lines) → `cargo test` reports **unused import** warnings.  
- **Fix**: remove unused imports or use them in tests; suitable for a small cleanup commit (e.g. under `v1-tech-debt-cleanup` or opportunistic PR).

## 6. 「V1.1 是否全部完成」— 边界分析（问答口径）

本节回答「开源 nexus 仓库语境下，1.1 相关事情是否都做完了」。**不**断言私有平台仓或整条产品线的完成度。

### 6.1 本仓库 **计划层面**（`status.json`）

| 判断 | 说明 |
| --- | --- |
| **离散 V1.1 交付计划** | 对齐收口、wire sprint、CLI sync/fork/explore/publish、ExploreCreatorCard 字段、codegen ref、**specs alignment remediation** 等均为 **Done**。 |
| **`Todo` 计划** | **0** — 无排队未启动的 plan 行。 |
| **仍为 `InProgress`** | **`v1-tech-debt-cleanup`**：长期计划；**Batch A–D 已完成**；**Batch E（TD-11、TD-13）**在计划文中标为 **V1.2 排期**，但 **plan id 不变**。因此机器状态上 **仍有未关闭计划**，语义上 **不全是「零未完成」**。 |

**结论 A**：若「V1.1」= **本仓已立项的 V1.1 功能/对齐/合约补齐** → **可以认为已完成**。若「全部完成」= **仓库里没有任何 InProgress plan** → **否**（还剩 `v1-tech-debt-cleanup`）。

### 6.2 跨仓与合同消费（nexus-platform）

| 项 | 说明 |
| --- | --- |
| **HTTP / OpenAPI** | 真源在平台仓；nexus 仅提供 **JSON Schema + npm/crate 生成物**。 |
| **Plan 15 / 质量门** | 属 **平台** program 状态，**不在**本 `status.json`。 |
| **Plan 21** | OSS 侧仅有 **checklist**；**勾选完成**需在平台执行。 |

**结论 B**：**不能**仅凭 nexus OSS「计划全 Done」推出「平台 V1.1 也全 Done」。

### 6.3 规格与实现的已知差分（非 plan 行）

| 项 | 是否算「V1.1 没做完」 |
| --- | --- |
| **全局 `state.db` vs 多工作区路径** | 属 **版本/产品承诺**问题；当前实现与 **单工作区** 假设一致；升级到多工作区通常归入 **V1.2+** 或专项 ADR。 |
| **Domain 未覆盖 sync 聚合** | **架构边界**，除非产品要求 domain 层全覆盖。 |
| **Codegen 松散 JSON / String** | **工程质量债**；见 §3.6。 |
| **acp_registry 测试 unused import** | **卫生**；见 §3.7。 |

**结论 C**：这些是 **尾巴与债**，不是「还有一个未登记的 V1.1 plan id」。

### 6.4 汇总话术（对内/对 PM）

- **「nexus OSS 里登记的 V1.1 相关工作」**：**已完成**（无 Todo；remediation Done）。  
- **「整个 Nexus 产品 / 双仓联合的 V1.1」**：需 **另看平台 roadmap + OpenAPI 冻结 + Plan 21 执行** — **本文件不背书**。  
- **「仓库里再也没有 open work」**：**否** — `v1-tech-debt-cleanup` 仍为 InProgress（含 V1.2 排期的 Batch E）。
