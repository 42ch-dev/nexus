## 5. Open design items (resolve during implement)

These are tracked as per-plan design decisions, not iteration-level unknowns:

1. **JSON Schema validation library** — Rust crate choice for validating ComputeInput/ComputeOutput against manifest-declared schemas (resolve in P1 T3). Candidates: `jsonschema`, `json-schema-validator`. Trade-off: compile time vs spec compliance.
2. **Validation strictness** — fail-fast on first validation error vs collect all errors (resolve in P1 T3). Recommendation: fail-fast with detailed JSON path (single error per call; matches ComputeError existing pattern).
3. **Schema fragment reuse** — should `manifest.json` allow `$ref` to external files for complex modules, or stay strictly inline? (V1.62 decision: strictly inline; defer `$ref` to V2 if needed.)
4. **`schemas/local-api/` future expansion pattern** — when next Local API DTO migrates from `nexus-contracts/src/local/`, what's the naming convention? (V1.62 decision: one subfolder per concern, mirroring the current `local/{acp,domain,orchestration,schedule}/` structure; e.g. `schemas/local-api/orchestration/`, `schemas/local-api/schedule/`. Re-evaluate at first migration.)
5. **Boundary doc naming** — should `schemas-wire-platform-sync-boundary.md` rename now that it covers more than platform-wire? (V1.62 decision: rename to `schemas-external-consumer-boundary.md` for clarity; update all in-repo links in the same P0 T15 commit.)
