- **A4 — Daemon validation + atomic persistence (LOCKED — cross-referenced with entity-scope-model §§5.5.1-5.5.2, §5.1.1)** (`crates/nexus-daemon-runtime/src/handlers/world_kb.rs`):
  - Parse + validate against World KB semantics before accepting any patch. Validation is **pre-write** — all rules below checked before any DB mutation:
    1. **ID existence**: `world_id` resolves to one World owned by the creator; `entity_id` (key_block_id) exists in that World; `job_id` / `candidate_id` exist in that World; path ids match body ids when duplicated (body authority rejected with 400 on mismatch).
    2. **Authorization**: creator owns the World (entity-scope-model §1.2 ownership path); cross-creator world blocks rejected (mirrors `nexus.kb_snapshot.write` cross-world guard). When KeyBlock carries `source_work_id`, prefer Work author per §5.5.7.3.
    3. **Promotion transition validity**: candidate `adopt` requires `status = pending`; `reject` requires `pending`; `merge` requires `pending` + valid target `key_block_id` in same World (confirmed or manual status, not deleted/merged/deprecated). Confirmed/rejected/merged are terminal states — repeated promotion attempts return 422. `manual` state entities (direct inserts) bypass promotion; their edits use `patch-entity` only.
    4. **Entity edit validity**: `block_type` re-classification respects entity-scope-model §5.1.1 BlockType enum — allowed values on wire (snake_case): `character`, `ability`, `scene`, `organization`, `item`, `conflict`, `info_point`, `event`, `species`, `faction`, `magic_system`, `technology`, `deity`, `level`, `economy_tier`, `dialogue`, `beat`, `act`. Body re-runs body-layer validation (Novel/GameBible/Script per the World's profile); `ValidationMode` is inferred from the active `work_profile`. Aliases uniqueness within same `(world_id, block_type, canonical_name)` composite per entity-scope-model §3.1. `expected_version` is required for all writes — missing `expected_version` returns 400.
    5. **Version precondition (OCC)**: `expected_version` equals the current row version (`kb_key_blocks.revision` for entity edits; `kb_extract_jobs.version` for promotion writes) before the daemon writes; stale values short-circuit as 409 `WorldKbConflictError`, not 422 validation failures. Both tables use CAS pattern; if `rows_affected() = 0` on UPDATE, the version precondition failed.
  - On success: atomic DB transaction. For promotion: uses V1.50 `insert_key_block_in_tx` + `mark_confirmed_in_tx` pattern with per-row CAS version guard. ## 6. Risk notes

### 6.1 Backend prerequisite risk (Track A)

V1.73 promotes World KB read+write from CLI-direct-DB to first-class Local API routes. The existing CLI commands (`creator world kb adopt/reject/edit/delete`) call `nexus_local_db::kb_extract_job` / `kb_store` directly. Refactoring them to call a Local API client (thin shell) preserves behavior but introduces a new transport layer. **Mitigation**: the CLI thin-shell refactor is behavior-preserving (same DB operations, new transport); hermetic tests verify CLI output unchanged before/after. The `nexus.kb_snapshot.read`/`write` host tools (orchestration-side) are NOT refactored in V1.73 — they remain on the direct-DB path (orchestration is a trusted in-process writer, not a canvas-driven remote writer).

Concrete mitigation / test scenarios:
- CLI `creator world kb adopt/reject/edit/delete/pending/list/show` output is byte-identical before/after the thin-shell refactor (regression snapshot tests).
- The new Local API routes return the same DB operation results as the CLI path (parity tests).
- Orchestration `nexus.kb_snapshot.read`/`write` host tools continue to work unchanged (not in refactor scope).

### 6.2 Per-row OCC storage decision (Track A — LOCKED)

V1.71 used YAML `revision:` (Strategy); V1.72 used markdown frontmatter `outline_revision:` (Outline). World KB lives in SQLite, so the revision mechanism differs. **LOCKED mechanism: per-row OCC on existing `kb_key_blocks.revision` + `kb_extract_jobs.version`.** No new migration required. `kb_key_blocks` has `revision INTEGER` from migration `20260525_kb_key_blocks.sql` line 14; `kb_extract_jobs` has `version INTEGER NOT NULL DEFAULT 0` from migration `202606190001_kb_extract_jobs_and_pool_version.sql`. The existing `update_key_block` in `crates/nexus-local-db/src/kb_store.rs` (line 672-693) must be enhanced with `WHERE revision = ?` and `rows_affected()` check for CAS. Per-row OCC is entity-scoped (editing character A doesn't block editing character B), which matches the canvas conflict UX (per-entity conflict modal). A world-level single revision counter is rejected because World KB edits are entity-scoped, not world-scoped. **Verified**: migration columns exist; update path lacks CAS; no data migration needed — existing rows with `revision = NULL` are treated as version 0.

Concrete mitigation / test scenarios:
- Entity A read at version `N`; daemon/orchestration mutates entity A to `N+1`; canvas submits with `expected_version: N` → 409 `WorldKbConflictError` with `current_version: N+1`.
- Entity A and entity B submit non-overlapping edits; both succeed independently (per-row OCC, no false conflict).
- Promote candidate read at version `N`; daemon auto-promote (CLI) flips to `N+1`; canvas adopt submits with `expected_version: N` → 409.
- UI conflict modal keeps the user's draft patch, refetches canonical state, exercises Use current / Reapply / Review side-by-side paths.

### 6.3 Conflict modal UX reuse risk (Track A)

V1.71/V1.72 shipped the conflict modal UX pattern. V1.73 reuses it with KB-flavored copy (entity name replaces chapter/event label; entity field replaces outline field; entity version replaces outline revision). Default action **Use current** preserves canonical server state.

Concrete mitigation / test scenarios:
- On 409 conflict, KB-flavored copy renders correctly (entity name not chapter title).
- Use current is default and preserves canonical state; draft remains in inspector.
- Reapply my edit resubmits with `expected_version = current_version`.
- Review side-by-side enables for non-overlapping fields; disables for same-field conflicts.

### 6.4 Outline β hardening regression risk (Track B1–B4)

Track B adds validation to the V1.72 outline patch routes. New validation (slug/volume/foreshadow/published) could reject previously-accepted patches. **Mitigation**: validation only rejects genuinely-invalid inputs (the 4 MEDIUM gaps are real correctness issues); existing valid patches continue to pass. Each B1–B4 fix adds a regression test proving valid input still passes + invalid input now rejected with structured 422.

### 6.5 outline-canvas.tsx split risk (Track B5)

Splitting the 825-line `outline-canvas.tsx` into 6 modules risks behavior regression. **Mitigation**: preserve all existing behavior; no API/signature changes; public facade re-export preserves imports; Track A imports through the public facade only until B5 merges. Each split file ≤ 250 lines. Existing vitest tests must pass unchanged after the split.

### 6.6 Concurrent daemon + canvas World KB writes (Track A)

World KB has two writer classes: the canvas (remote, via Local API) and orchestration (in-process, via `nexus.kb_snapshot.write` host tool / review-time extraction hook). Per-row OCC must catch stale writes from BOTH. When orchestration mutates a KeyBlock between canvas read and patch submission, the patch must fail with 409 rather than silently applying over a newer state.

Concrete mitigation / test scenarios:
- Canvas reads entity at version `N`; orchestration `nexus.kb_snapshot.write` mutates to `N+1`; canvas submits with `expected_version: N` → 409.
- Review-time extraction hook inserts a new candidate; canvas promote-candidate on a stale candidate view → 409 if the candidate was already adopted/rejected by another path.
