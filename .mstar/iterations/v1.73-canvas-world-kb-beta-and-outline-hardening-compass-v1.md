## 1. Scope / non-goals / out-of-scope

### 1.1 In scope

**Track A — Canvas World KB β (Full depth, the lead)**

Implement the Canvas World KB β slice per [canvas-strategy-surface.md §3.3 surface 3](../knowledge/specs/canvas-strategy-surface.md) and the V1.71/V1.72 patch-route/per-row-conflict pattern. This promotes the World KB canvas surface from Draft to **Shipped β** (read + write + conflict + non-spatial alternate view).

- **A1 — Schemas + codegen (the additive wire-contract work)**: New Local API DTOs in `schemas/local-api/world-kb/`:
  - `WorldKbPatchEntityRequest` (carries `world_id`, `key_block_id`, `base_version`, and the patch payload: title, body, aliases, block_type where re-classification is allowed).
  - `WorldKbPromoteCandidateRequest` (carries `world_id`, `extract_job_id`, `base_version`, and the promotion action: `adopt` | `reject` | `merge` + optional merge target `key_block_id`).
  - `WorldKbPatchResponse` (echoes `new_version`, `validation_summary`, optional `side_effects`).
  - `WorldKbConflictError` (mirrors V1.71/V1.72 conflict shape: `current_version`, `key_block_id`/`extract_job_id`, `conflicting_path`, recovery hint).
  - `WorldKbValidationError` (mirrors V1.71/V1.72: structured validation summary).
  - Read-side projection DTOs: `WorldKbGraphProjection` (nodes: entities + candidates with lifecycle; edges: source-anchor provenance + derived references) following the shared `CanvasGraphDocument` shape (§3.4).
  - All generated to TS (`apps/web/src/api-types/world-kb/`) and Rust (`crates/nexus-contracts/src/generated/world-kb/`). `pnpm run codegen` regenerates downstream; CI `verify-codegen` verifies committed artifacts.
- **A2 — Backend prerequisite: promote World KB read+write from CLI-direct-DB to Local API routes (LOCKED shape, architect confirms mechanism)**: The existing `creator world kb adopt/reject/edit/delete/pending` CLI commands call the DB layer directly (`nexus_local_db::kb_extract_job`, `kb_store`). V1.73 promotes these to first-class Local API routes so the canvas can drive them. Read endpoints: `GET /v1/local/worlds/{world_id}/kb/graph` returns the `WorldKbGraphProjection`; `GET /v1/local/worlds/{world_id}/kb/candidates` returns pending promotion candidates. Write endpoints: see A3. The CLI commands are refactored to call the new Local API client (thin shell) — preserving CLI behavior; no CLI surface regression.
- **A3 — Local API routes (LOCKED set; architect confirms exact paths)**: New routes following V1.64 cursor + ErrorResponse conventions and V1.71/V1.72 patch-route convention:
  - `POST /v1/local/worlds/{world_id}/kb/patch-entity` with `WorldKbPatchEntityRequest` → `WorldKbPatchResponse` (operations: edit entity title/body/aliases/block_type with Novel validation re-run per entity-scope-model §5.5.5).
  - `POST /v1/local/worlds/{world_id}/kb/promote-candidate` with `WorldKbPromoteCandidateRequest` → `WorldKbPatchResponse` (operations: `adopt` confirms a pending candidate → confirmed KeyBlock; `reject` dismisses to `Logs/kb/rejected/`; `merge` folds a candidate into an existing confirmed KeyBlock).
  - Both carry `world_id`, target identifiers, `base_version`, and a minimal patch payload; path identifiers are authoritative and must match any repeated body identifiers if present.
  - Error envelope: `WorldKbConflictError` (409) + `WorldKbValidationError` (422) + standard envelope (400/500).
- **A4 — Daemon validation + atomic persistence (LOCKED rules; architect confirms storage mechanism)** (`crates/nexus-daemon-runtime/src/handlers/world_kb.rs`):
  - Parse + validate against World KB semantics before accepting any patch:
    1. **ID existence**: `world_id` resolves to one World owned by the creator; `key_block_id` / `extract_job_id` exist in that World; path ids match body ids when duplicated.
    2. **Authorization**: creator owns the World (entity-scope-model §1.2 ownership path); cross-creator world blocks rejected (mirrors `nexus.kb_snapshot.write` cross-world guard).
    3. **Promotion transition validity**: candidate `adopt` requires `status = pending`; `reject` requires `pending`; `merge` requires `pending` + valid target `key_block_id` in same World; confirmed/rejected/merged are terminal (entity-scope-model §5.5.1–§5.5.2).
    4. **Entity edit validity**: block_type re-classification respects entity-scope-model §5.1.1 BlockType enum; body re-runs `ValidationMode::Novel` (entity-scope-model §5.5.5); aliases uniqueness within World.
    5. **Version precondition (OCC)**: `base_version` equals the current row version before the daemon writes; stale values short-circuit as 409 conflicts, not 422 validation failures.
  - On success: atomic DB transaction (promote uses the V1.50 `insert_key_block_in_tx` + `mark_confirmed_in_tx` pattern; entity edit uses the V1.51 per-row CAS UPDATE…WHERE version = ? pattern) with version increment.
  - On conflict: short-circuit before write; return `WorldKbConflictError` with current canonical state.
- **A5 — TS client methods (`NexusClient` in `apps/web/src/api/`)**: `worldKbPatchEntity`, `worldKbPromoteCandidate` methods. Refetch canonical graph after successful patch; expose `lastKnownVersion` per-entity for client-side conflict anticipation.
- **A6 — Canvas UI write affordances** (`apps/web/src/features/canvas/world-kb/`):
  - **World KB graph projection** (per `canvas-strategy-surface.md` §3.3 surface 3 + §3.4 WorldKbNodeData/WorldKbEdgeData): World → KeyBlock entities (character/ability/scene/organization/item/conflict/info_point/event + computable blocks) as graph nodes; promotion-state lifecycle rendered on candidate nodes (pending/confirmed/rejected/merged badges); source-anchor provenance edges; computable-state badges on computable blocks. Entity cards show name/BlockType/lifecycle/source-anchor count.
  - **Entity inspector**: edit title/body/aliases/block_type; submit `world_kb.patch_entity`; show validation errors inline; show conflict modal on 409.
  - **Promotion inspector**: pending candidate → adopt/reject/merge affordances; submit `world_kb.promote_candidate`; merge flow selects target confirmed KeyBlock; same conflict UX.
  - **Relationship edges (read-only this slice)**: derived reference/source-anchor edges render on the canvas (read-side projection from `kb_source_anchors`); no patch affordance (deferred V1.74).
  - **Conflict modal UX** (reuse V1.71/V1.72 pattern, KB-flavored copy):
    - **Headline**: "This world entry changed while you were editing."
    - **Body**: "Nexus updated **{entity_name}** to version **{current_version}** while you were editing **{field}**. Your change is still in the inspector." (`{entity_name}` adapts: entity name for `patch_entity`, candidate name for `promote_candidate`.)
    - **What changed** / **What you were about to do** / **Actions** (Use current / Reapply my edit / Review side-by-side / Cancel) — same semantics as V1.72 Outline conflict modal.
  - **World KB freshness indicator** in command palette: "World KB · {entity_count} entries · updated {relative time} — refresh now" (reuse V1.71/V1.72 revision copy pattern).
  - **Non-spatial alternate view** (Draft §4.4 accessibility): sortable entity list (name / BlockType / lifecycle / source-anchor count / updated) — available as toggle from the canvas toolbar; default for keyboard-only / screen-reader users.
- **A7 — Tauri WKWebView parity smoke**: V1.70/V1.71/V1.72 established the canvas-vs-WKWebView parity pattern; V1.73 extends it to World KB entity inspector, promotion inspector, and conflict modal. Browser + Tauri shell parity check on World KB interactions.
- **A8 — Accessibility (conflict UX + canvas)**: conflict modal announces via ARIA live region; focus moves into modal and is trapped until action selected; closing returns focus to originating inspector; keyboard-accessible paths for all actions; honor `prefers-reduced-motion`; non-spatial alternate view reachable and equivalent (Draft §4.4 invariant).

## 6. Risk notes

### 6.1 Backend prerequisite risk (Track A)

V1.73 promotes World KB read+write from CLI-direct-DB to first-class Local API routes. The existing CLI commands (`creator world kb adopt/reject/edit/delete`) call `nexus_local_db::kb_extract_job` / `kb_store` directly. Refactoring them to call a Local API client (thin shell) preserves behavior but introduces a new transport layer. **Mitigation**: the CLI thin-shell refactor is behavior-preserving (same DB operations, new transport); hermetic tests verify CLI output unchanged before/after. The `nexus.kb_snapshot.read`/`write` host tools (orchestration-side) are NOT refactored in V1.73 — they remain on the direct-DB path (orchestration is a trusted in-process writer, not a canvas-driven remote writer).

Concrete mitigation / test scenarios:
- CLI `creator world kb adopt/reject/edit/delete/pending/list/show` output is byte-identical before/after the thin-shell refactor (regression snapshot tests).
- The new Local API routes return the same DB operation results as the CLI path (parity tests).
- Orchestration `nexus.kb_snapshot.read`/`write` host tools continue to work unchanged (not in refactor scope).

### 6.2 Per-row OCC storage decision (Track A — architect Phase 2b)

V1.71 used YAML `revision:` (Strategy); V1.72 used markdown frontmatter `outline_revision:` (Outline). World KB lives in SQLite, so the revision mechanism differs. Phase 2b architect locks the mechanism; **default candidate is per-row OCC** reusing V1.51 `kb_extract_jobs.version` CAS + adding a `version` column to `kb_key_blocks` if absent. Per-row OCC is entity-scoped (editing character A doesn't block editing character B), which matches the canvas conflict UX (per-entity conflict modal). A world-level single revision counter is rejected because World KB edits are entity-scoped, not world-scoped.

Concrete mitigation / test scenarios:
- Entity A read at version `N`; daemon/orchestration mutates entity A to `N+1`; canvas submits with `base_version: N` → 409 `WorldKbConflictError` with `current_version: N+1`.
- Entity A and entity B submit non-overlapping edits; both succeed independently (per-row OCC, no false conflict).
- Promote candidate read at version `N`; daemon auto-promote (CLI) flips to `N+1`; canvas adopt submits with `base_version: N` → 409.
- UI conflict modal keeps the user's draft patch, refetches canonical state, exercises Use current / Reapply / Review side-by-side paths.

### 6.3 Conflict modal UX reuse risk (Track A)

V1.71/V1.72 shipped the conflict modal UX pattern. V1.73 reuses it with KB-flavored copy (entity name replaces chapter/event label; entity field replaces outline field; entity version replaces outline revision). Default action **Use current** preserves canonical server state.

Concrete mitigation / test scenarios:
- On 409 conflict, KB-flavored copy renders correctly (entity name not chapter title).
- Use current is default and preserves canonical state; draft remains in inspector.
- Reapply my edit resubmits with `base_version = current_version`.
- Review side-by-side enables for non-overlapping fields; disables for same-field conflicts.

### 6.4 Outline β hardening regression risk (Track B1–B4)

Track B adds validation to the V1.72 outline patch routes. New validation (slug/volume/foreshadow/published) could reject previously-accepted patches. **Mitigation**: validation only rejects genuinely-invalid inputs (the 4 MEDIUM gaps are real correctness issues); existing valid patches continue to pass. Each B1–B4 fix adds a regression test proving valid input still passes + invalid input now rejected with structured 422.

### 6.5 outline-canvas.tsx split risk (Track B5)

Splitting the 825-line `outline-canvas.tsx` into 6 modules risks behavior regression. **Mitigation**: preserve all existing behavior; no API/signature changes; public facade re-export preserves imports; Track A imports through the public facade only until B5 merges. Each split file ≤ 250 lines. Existing vitest tests must pass unchanged after the split.

### 6.6 Concurrent daemon + canvas World KB writes (Track A)

World KB has two writer classes: the canvas (remote, via Local API) and orchestration (in-process, via `nexus.kb_snapshot.write` host tool / review-time extraction hook). Per-row OCC must catch stale writes from BOTH. When orchestration mutates a KeyBlock between canvas read and patch submission, the patch must fail with 409 rather than silently applying over a newer state.

Concrete mitigation / test scenarios:
- Canvas reads entity at version `N`; orchestration `nexus.kb_snapshot.write` mutates to `N+1`; canvas submits with `base_version: N` → 409.
- Review-time extraction hook inserts a new candidate; canvas promote-candidate on a stale candidate view → 409 if the candidate was already adopted/rejected by another path.
