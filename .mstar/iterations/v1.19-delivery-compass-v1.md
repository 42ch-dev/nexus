## 2) V1.19 positioning

V1.19 is a **hardening release** ‚Äî no new crates, no new architectural layers. The goal is to make the V1.18 skeleton **functionally complete** for real usage:

1. **Safety first**: shutdown wiring (D-007) and admission enforcement (D-008) are correctness bugs, not enhancements.
2. **Capability honesty**: the descriptor must not lie about what the provider supports (D-003).
3. **Multi-turn is basic**: per-op spawn makes native providers unusable for conversation (D-001).
4. **Permission handling is mandatory**: ACP providers hang without it (D-002).
5. **Timeout enforcement prevents indefinite hangs**: all provider paths must be bounded (D-004).
6. **QC residuals close out**: input validation (R4), path traversal (R5), and cross-platform probe (R3) are hardening staples.

## 4) Item detail

### D-001: Native CLI multi-turn session support [HIGH]

**Problem**: `ClaudeCliProvider::execute()` spawns a fresh subprocess per call (claude.rs L239-325). Each operation gets a new `HostSessionId`. There is no persistent native session ‚Äî stdin is written once and closed. Multi-turn conversation is broken.

**V1.18 audit note**: "Multi-turn is a basic feature, not a reasonable simplification." The `ProviderAdapter` trait and session state machine were designed for persistent sessions. `ManagedSessionHandle` carries a `session_id` expected to survive across operations.

**Design**:

1. `ClaudeCliProvider::launch()` must spawn the child process and store it in a session-scoped `NativeSession { child, stdin, stdout }` (struct already defined at L28-55 but fields unused).
2. `stdin` pipe stays open across `execute()` calls.
3. Each `execute()` writes a new prompt to `stdin` and reads the response from `stdout`.
4. `shutdown()` closes stdin (EOF), waits for process exit with `shutdown_ms` timeout, then force-kills.
5. Claude CLI `--print` mode is single-turn by design ‚Äî multi-turn requires switching to interactive/repl mode or using the Claude CLI's session-resume flags if available. Research required at T1.
6. If Claude CLI does not support persistent interactive stdin in a usable way, consider Claude CLI's `--resume` flag or `--session-id` for session continuity across invocations (multi-process multi-turn instead of single-process multi-turn).

**Open question**: Does Claude CLI `--print` support multi-turn via persistent stdin, or does it require `--resume` / `--session-id` for multi-turn continuity? This must be resolved at T1 before implementation.

**Risk**: If Claude CLI cannot support multi-turn at all in Wave 2, D-001 scope reduces to "best-effort session continuity via resume flags" with a note that full multi-turn requires ACP provider.

**Related**: claude.rs L28-55 (NativeSession), L212-237 (launch), L239-325 (execute).

### D-002: ACP session/request_permission handling [HIGH]

**Problem**: `AcpProvider::execute()` does not handle ACP `session/request_permission` messages. When an ACP provider requests permission to use a tool, the host must evaluate and respond. Currently the provider hangs or timeouts.

**V1.18 compass requirement (R-003)**: "ACP provider permissions delegate to `nexus-acp-host::PermissionPolicy::evaluate_for_agent()`."

**Design**:

1. The ACP provider's event stream loop must monitor for permission request messages (distinct from `session/update` content events).
2. On receipt, extract tool name, description, and risk classification.
3. Call through to `nexus_acp_host::PermissionPolicy::evaluate_for_agent()` with tool context and session identity.
4. If risk classifier (D-005) is available, use `AutoToolRiskClassifier` to determine risk level; otherwise fall back to `StaticToolRiskClassifier`.
5. Permission outcome preference order: `approve_for_session > allow_always > allow_once`; reject otherwise.
6. Send the permission response back through the ACP SDK.
7. Emit a `ToolCallUpdate` host event so the caller knows a permission decision was made.

**Depends on**: D-005 (risk classifier enables informed policy decisions; without it, all unknown tools default to deny/ask).

**Related**: acp.rs L212-289 (execute loop), risk.rs, policy/admission.rs, `nexus-acp-host::PermissionPolicy`.

### D-003: SetModel / SetMode on ACP provider [MEDIUM]

**Problem**: `AcpProvider::execute()` returns `CapabilityUnsupported` for `SetModel` and `SetMode` (acp.rs L219-232). `CapabilityDescriptor::acp_full()` claims both as `true`. This is a capability lie.

**Design** (two options, decide at T1):

- **Option A (implement)**: Use ACP SDK session config APIs to apply model/mode changes. If the provider rejects, apply graceful fallback per OpenDesign pattern ‚Äî emit `Status` warning and continue with default (only when `allow_model_fallback = true`).
- **Option B (remove claim)**: Set `set_model = false` and `set_mode = false` in `acp_full()` static descriptor. Only enable after successful ACP initialize negotiation confirms support.

**Recommendation**: Option A if the ACP SDK exposes session config methods; Option B if not. The key constraint is that the descriptor must never claim what the adapter cannot deliver.

**Related**: acp.rs L219-232, model.rs L353-370.

### D-004: Stage-level timeout enforcement [MEDIUM]

**Problem**: `TimeoutConfig` defines `launch_ms`, `initialize_ms`, `session_ms`, `prompt_ms`, `shutdown_ms` but none are enforced. A hung provider blocks the host indefinitely.

**Design**:

1. Wrap each provider operation with `tokio::time::timeout()` using the corresponding config value.
2. Operations to wrap:
   - `probe()` ‚Üí `launch_ms` (use launch timeout as a reasonable default for probe)
   - `launch()` ‚Üí `launch_ms` + `initialize_ms` (two-stage: spawn timeout + initialize timeout)
   - `execute()` ‚Üí `prompt_ms`
   - `shutdown()` ‚Üí `shutdown_ms` (drain timeout with force-kill fallback)
3. On timeout, cancel the operation and emit `OpFailed` with `error_category = "timeout"`.
4. Shutdown drain in `HostManager::shutdown()` must use `shutdown_ms` per-session, then force-kill survivors. This closes R7 (shutdown drain not bounded).

**Related**: config.rs (TimeoutConfig), all providers, manager.rs.

### D-005: Auto tool-risk classification [MEDIUM]

**Problem**: Only `StaticToolRiskClassifier` (manual registration) exists. OpenDesign's regex-based auto-classification was identified as valuable (R-006) but only the extension point was created.

**Design**:

1. Implement `AutoToolRiskClassifier` using the `ToolRiskClassifier` trait.
2. Classification rules (borrowed from OpenDesign `classifyConnectorToolSafety()`):
   - Destructive: `drop|truncate|purge|erase|wipe|rm |force|kill|destroy` ‚Üí `Destructive`
   - Write: `create|update|delete|send|manage|write|insert|modify|edit|remove` ‚Üí `Write`
   - Read: `get|list|search|fetch|view|read|show|find|query|check|describe` ‚Üí `Read`
   - Default (unknown tools): `Write` ‚Äî not `Read`
3. Compiled regex patterns (not runtime-compiled per call).
4. Integration: `HostManager` uses `AutoToolRiskClassifier` as the default classifier; `StaticToolRiskClassifier` overrides for specific tools when configured.
5. Used by ACP permission handling (D-002) to inform `PermissionPolicy` decisions.

**Related**: risk.rs L24-60 (trait + static impl).

### D-006: Provider-level streaming adaptation [MEDIUM ‚Üí LOW priority]

**Problem**: ACP streaming events are not fully translated to `HostEvent::MessageDelta` / `ToolCall` etc. The scaffold (`into_event_stream`) exists but is not wired for all ACP event types.

**Design**:

1. Complete the ACP event ‚Üí `HostEvent` mapping:
   - `session/update` with `thinking` content ‚Üí `HostEvent::ThoughtDelta`
   - `session/update` with `text` content ‚Üí `HostEvent::MessageDelta`
   - `session/update` with `tool_use` ‚Üí `HostEvent::ToolCall`
   - `session/update` with `tool_result` ‚Üí `HostEvent::ToolCallUpdate`
   - `session/update` with `plan` / `reasoning` ‚Üí `HostEvent::PlanUpdate`
2. Handle partial line buffering for JSON-RPC newline-delimited messages.
3. Ensure each prompt produces exactly one terminal `OpFinished` or `OpFailed`.

**Priority note**: Current buffered event approach works for single-turn. This item can be deferred to V1.20 if Batch 2 runs long ‚Äî it is an enhancement, not a correctness fix.

**Related**: acp.rs, capability/model.rs (HostEvent variants).

### D-007: HostManager::shutdown ‚Üí ProviderAdapter::shutdown() [MEDIUM]

**Problem**: `HostManager::shutdown()` (manager.rs L320-347) transitions sessions to Stopped and clears provider mappings but never calls `ProviderAdapter::shutdown()`. ACP sessions are not cleanly closed; native child processes are not terminated ‚Äî they become orphans.

**Design**:

1. Before clearing session registry state, iterate all active sessions.
2. For each session, call the corresponding `ProviderAdapter::shutdown()`.
3. Wait up to `shutdown_ms` per session (from `TimeoutConfig`).
4. Force-kill any surviving processes.
5. Only then clear session registry and provider mappings.
6. This is the managed-only safety boundary.

**Effort**: Small ‚Äî the adapter `shutdown()` methods already exist (acp.rs L317-330, claude.rs L344-358). The gap is purely in `HostManager::shutdown()` not calling them.

**Related**: manager.rs L320-347, acp.rs L317-330, claude.rs L344-358.

### D-008: AdmissionPolicy enforcement in create_session / exec [MEDIUM]

**Problem**: `AdmissionPolicy` is instantiated in `HostManager` but its methods (`check_before_launch`, `check_before_exec`, `check_provider`, `check_session_limit`, `check_ops_per_session`) are never called from `create_session()` or `exec()`.

**Design**:

1. In `create_session()` (manager.rs L159-213):
   - Call `admission.check_provider(&request.provider_id)` ‚Äî verify provider is allowed.
   - Call `admission.check_session_limit(session_count)` ‚Äî enforce `max_sessions`.
2. In `exec()` (manager.rs L215-268):
   - Call `admission.check_ops_per_session(active_ops)` ‚Äî enforce `max_ops_per_session`.
   - Call `admission.check_before_exec(&session, &op)` ‚Äî verify capability and policy.
3. On denial, return `HostError::PolicyDenied` with descriptive message.
4. Update `AdmissionPolicy::from_config()` to consume discovered providers from the catalog (not just config entries).

**Effort**: Small ‚Äî the methods exist; the gap is purely wiring.

**Related**: manager.rs L159-268, policy/admission.rs.

### D-009: Cross-platform command probe [LOW]

**Problem**: `path_scan.rs` uses Unix-only `which` command to check if a binary exists. This breaks on Windows.

**Design**:

1. Replace `Command::new("which")` with a cross-platform check:
   - Use `which::which()` from the `which` crate (already a reasonable lightweight dependency), or
   - Implement platform-conditional probe: `which` on Unix, `where` on Windows via `Command::new`.
2. Prefer the `which` crate for reliability ‚Äî it handles PATH, PATHEXT, and edge cases.
3. Update probe in `ClaudeCliProvider::probe()` and `PathScanDiscovery::discover()`.

**Related**: path_scan.rs, claude.rs (probe).

### D-010: API handler input validation [MEDIUM]

**Problem**: API handlers in `daemon-runtime/api` accept session ID path parameters without validation. Malformed or non-UUID values could cause panics or unexpected behavior.

**Design**:

1. Parse session ID path params as `uuid::Uuid` at the handler boundary.
2. Return 400 Bad Request with descriptive error for invalid IDs.
3. Return 404 Not Found for valid UUIDs that don't match any session.
4. Apply to all `/v1/local/agent-host/sessions/{id}/*` routes.

**Related**: daemon-runtime/api agent-host route module.

### D-011: Config path traversal protection [MEDIUM]

**Problem**: Config paths from `AgentHostConfig` (e.g., `config_path`, `workspace_root`) are not validated against directory traversal attacks (e.g., `../../etc/passwd`).

**Design**:

1. Validate `workspace_root` in `CreateSessionRequest`:
   - Must be an absolute path.
   - Must not escape above a configured trust boundary (default: current user's home directory or the workspace root passed to `HostManager::start()`).
2. Validate `config_path` in `HostStartConfig`:
   - Must be under `{NEXUS_HOME}/agent-host/` or an explicitly allowed path.
3. Return `HostError::PolicyDenied` for traversal attempts.
4. Use `std::path::canonicalize()` for resolution and prefix checking.

**Related**: config.rs, manager.rs (create_session).
