## 4) V1.5 workstreams (execution decomposition)

### WS-A — Schedule FSM hardening (Track A, first priority)

- **Goal**: Resolve all 7 schedule-related residuals (WS7 R1–R7) to deliver a robust Schedule state machine with correct persistence, transitions, and UX.
- **Authoritative design**: [creator-schedule-and-core-context.md](creator-schedule-and-core-context.md) §3 (data model), §4 (lifecycle FSM), §5 (concurrency), §8 (CLI surface), §10 (SQLite schema).
- **Dependencies**: none — can begin immediately.
- **Deliverables**:
  - R1: Persist `RunningSet` / `CompletedSet` across daemon restart — extend `creator_schedules` table or add admission state to `orchestration_sessions`; `ScheduleSupervisor` repopulates from SQLite on daemon init.
  - R2: Duplicate `schedule_id` detection — add UNIQUE constraint enforcement or early-check in `schedule add` path.
  - R3: Implement `paused→running` FSM transition — extend `ScheduleSupervisor` to handle resume-from-paused-mid-exec correctly (re-associate with existing session).
  - R4: Implement pause signal propagation in FSM — `ScheduleSupervisor.pause()` sends `engine.signal(session, Pause)` and flips schedule status atomically.
  - R5: Delete cascade for schedules — add `ON DELETE CASCADE` on foreign keys or implement cascading delete in `schedule remove` command (cancel active session + delete dependency rows + delete schedule row).
  - R6: Fix `CoreContextManager` version bump race — use `SELECT … FOR UPDATE`-style serialisation or optimistic concurrency with retry on the `(schedule_id, version)` primary key.
  - R7: Simplify paused→running UX — single `schedule resume <id>` command that handles both pre-exec and mid-exec resume paths without requiring a two-step sequence.
- **Evidence**: unit tests for each FSM transition edge; pause/resume roundtrip across daemon restart; concurrent `schedule add` duplicate detection; delete cascade with in-flight session; version bump race test with concurrent edits.

### WS-B — Orchestration + DB residual cleanup (Track A)

- **Goal**: Resolve the remaining 12 residuals across orchestration skeleton, preset loader, local DB, and sqlx compile-time migration to bring the entire V1.4 codebase to low/nit residual level.
- **Authoritative design**: [orchestration-engine.md](../knowledge/specs/orchestration-engine.md) §4 (engine), §6 (worker model), §8 (preset loader); [local-db-refactor.md](archived/knowledge/local-db-refactor.md); [sqlx-compile-time-migration.md](archived/knowledge/sqlx-compile-time-migration.md).
- **Dependencies**: none — can begin immediately (parallel with WS-A).
- **Deliverables**:
  - **WS2 R1**: Repopulate in-memory session tracker from SQLite on daemon restart — engine init queries `orchestration_sessions WHERE status IN ('running', 'waiting_for_input')` and reconstructs the in-memory map.
  - **WS2 R3**: Verify `FlowRunner` clone-per-step correctness — add deterministic test that runs a multi-step graph, clones mid-execution, and confirms both branches produce correct independent results.
  - **WS2 R4**: Implement Worker shutdown SIGTERM→SIGKILL path — replace direct `start_kill()` with graceful `SIGTERM` → timeout → `SIGKILL` sequence per [acp-client-tech-spec.md](../archived/../archived/knowledge/acp-client-tech-spec.md) §2.3.
  - **WS8 R1**: Fix `read_versions` error handling — return the malformed value alongside the error so callers can diagnose.
  - **WS8 R4**: Move Outbox schema DDL from inline to `sqlx::migrate` — add a timestamped `.sql` migration file under `crates/nexus-local-db/migrations/` for the outbox table.
  - **WS3 R1**: Deduplicate `EngineProxy` vs `GraphFlowEngine` — extract shared logic into the engine itself or collapse the proxy into a thin delegation wrapper.
  - **WS3 R2**: Strengthen E2E test assertions — replace `let _ = …` patterns with actual result validation in `e2e_novel_writing.rs`.
  - **sqlx R1**: Add `// SAFETY:` comments on all test-only runtime queries in `pool.rs`, integration tests, and `test_utils.rs`.
  - **sqlx R2**: Move inline `FromRow` structs (`OutboxRow`, `PartialApplyRow`) to module level in `outbox.rs`.
  - **sqlx R3**: Normalize `?1` vs `?` placeholder syntax to the consistent `?` form across compile-time macros.
  - **sqlx R4**: Fix misleading SAFETY comment in `schedules.rs` (copy-paste error; update reason to match actual dynamic-SQL rationale).
  - **sqlx R5**: Convert test-only queries in `schema.rs` to compile-time macros where feasible.
- **Evidence**: `cargo clippy --all -- -D warnings` clean; `SQLX_OFFLINE=true cargo test --workspace` passes; E2E assertions non-trivial; `rg 'rusqlite|deadpool[_-]sqlite' crates/` returns zero hits; sqlx prepare check passes.

### WS-C — `context.summarize` capability (Track B)

- **Goal**: Implement the LLM-driven `core_context` summarization capability that V1.4 reserved in the data model. Users (via preset YAML) can declare `context_update.kind: llm_summarize` on a state's exit, causing the engine to invoke an LLM-backed summarization step and write a new `core_context_versions` row with `derivation_kind = 'llm_summarize'`.
- **Authoritative design**: [creator-schedule-and-core-context.md](creator-schedule-and-core-context.md) §6.3 (reserved derivation kind), §11 (V1.5 forward compatibility — full design preview).
- **Dependencies**: **WS-A must complete first** — `context.summarize` builds on the Schedule FSM and `CoreContextManager` infrastructure hardened in WS-A (especially R6 race fix).
- **Deliverables**:
  - New capability `context.summarize` at `crates/nexus-orchestration/src/capabilities/context_summarize.rs` — implements the `Capability` trait; takes `(current core_context, state execution trace, optional template)` as input; invokes a configured LLM-backed judge capability; writes a new `core_context_versions` row with `derivation_kind = 'llm_summarize'`.
  - Preset YAML extension: `context_update.kind: llm_summarize` (new variant alongside existing `append` / `struct_merge`). Preset loader validates the new kind and routes to `context.summarize` capability.
  - Zero schema migration: `DerivationKind::LlmSummarize` variant already exists in the Rust enum; `core_context_versions.derivation_kind` TEXT column already accepts `'llm_summarize'`; CLI `schedule context-history` already handles the label.
  - Configuration: `context.summarize` requires a user-configured LLM endpoint (via the existing `judge.llm` capability infrastructure); if no LLM is configured, the capability returns a `CapabilityError::TransientExternal` and the preset's context_update is skipped gracefully.
  - Integration with `novel-writing` preset: optional demonstrator that uses `context_update.kind: llm_summarize` on one state (e.g. after `brainstorming`) to summarize the brainstorm output into a condensed `core_context` for downstream states.
- **Evidence**: unit test with mock LLM producing a summary and writing `core_context_versions` row; preset YAML with `llm_summarize` validates and loads; integration test with `novel-writing` preset showing the summarised context is read by the next state; `cargo test --workspace` green; clippy/fmt clean.

### WS-D — Cron / wall-clock triggers (Track B)

- **Goal**: Implement the clock poller that automatically moves `Pending` Schedules to `Running` when `now() >= scheduled_at`, subject to admission rules. This activates the `scheduled_at` column that V1.4 reserved but never consumed.
- **Authoritative design**: [creator-schedule-and-core-context.md](creator-schedule-and-core-context.md) §2 (OQ-6 — trigger model), §10 (SQLite schema — `scheduled_at` column, index `creator_schedules_by_scheduled`); [orchestration-engine.md](../knowledge/specs/orchestration-engine.md) §11 OQ-6; [crate-selection-best-practices.md](crate-selection-best-practices.md) §3.7 (four hard constraints for crate selection).
- **Dependencies**: **WS-A must complete first** — cron triggers interact with the Schedule FSM admission logic hardened in WS-A.
- **Deliverables**:
  - Clock poller module at `crates/nexus-orchestration/src/scheduler/` — a background Tokio task that periodically queries `creator_schedules WHERE status = 'pending' AND scheduled_at IS NOT NULL AND scheduled_at <= now()`, then submits each to the `ScheduleSupervisor` for admission.
  - Crate selection: evaluated at implementation time per [crate-selection-best-practices.md](crate-selection-best-practices.md) §3.7. Candidates: `tokio-cron-scheduler` 0.15.x (quickest integration; DST risk requires wrapper) or hand-rolled `cron` + `chrono-tz` + `tokio::time::sleep_until` (~150 LOC, full control). Both must satisfy the four hard constraints: (1) wall-clock + tz-aware, (2) per-key serialisation (same creator + same Schedule MUST NOT double-run), (3) graceful shutdown (waits for current node or force-abort after timeout), (4) DST / clock-jump safety (recompute next-run on discontinuity).
  - Zero schema migration: `scheduled_at` column and `creator_schedules_by_scheduled` index already exist from V1.4 WS7.
  - CLI addition: `nexus42 schedule add --preset <id> --creator <id> --scheduled-at <ISO-8601 datetime>` (accepts the `scheduled_at` value that V1.4 defined but ignored).
  - `ScheduleSupervisor` integration: admission rules from [creator-schedule-and-core-context.md](creator-schedule-and-core-context.md) §5 still apply — cron-triggered schedules still respect concurrency declarations, dependency chains, and the single-ACP-worker-per-creator constraint.
  - Graceful shutdown: daemon lifecycle HSM `Stopping` state signals the clock poller to stop; in-flight admission checks complete or abort within the shutdown grace period.
- **Evidence**: time-triggered schedule starts correctly at the appointed wall-clock time (mock clock test); duplicate-fire prevention test (same schedule queried twice in one poll cycle → admitted once); DST transition test (clock jumps forward/backward, next-run recomputed correctly); graceful shutdown test (daemon stops mid-poll, no orphan admissions); `cargo test --workspace` green; clippy/fmt clean.

### WS-E — Platform coordination (Track C)

- **Goal**: Deliver nexus-side schema and contract changes required by platform V1.5 plans 36–43, so the platform team can proceed with their dependent work without being gated on nexus.
- **Authoritative design**: `{platform-plans}/knowledge/v1.5-program-compass-v1.md` (themes A–F); individual plan files `{platform-plans}/<plan-id>.md`.
- **Dependencies**: none — can begin immediately (parallel with WS-A and WS-B). However, nexus schema changes should be coordinated with the platform team to avoid wasted work on plans that may still evolve.
- **Deliverables**:
  - **Plan 40 (`as_of`) — nexus JSON Schema sync**: add `as_of` query parameter and response fields to the relevant `schemas/` files (context-assembly wire contracts). Run `pnpm run codegen` and commit generated TypeScript + Rust output. This is the highest-priority platform dependency.
  - **Plans 38–39 (identity plane + OpenAPI)**: evaluate whether nexus schemas need updates for any new auth-related wire types. Currently nexus contracts are auth-agnostic (platform handles auth); but if new token-exchange routes require nexus schema changes, deliver those.
  - **Plan 41 (e2e)**: ensure `nexus42` CLI commands that the e2e test suite exercises remain stable. No schema changes expected, but regression testing against platform's e2e plan is valuable.
  - **Plan 42 (quota enforcement)**: evaluate whether nexus contracts need `quota_exceeded` error shapes or entitlement-related wire types. If so, add schemas and regenerate.
  - **Plan 43 (Skills Export L0)**: evaluate whether static skill descriptors live in nexus schemas or are purely platform-side. If nexus-hosted, add schemas.
  - Cross-repo handoff: after each schema change, commit regenerated output (`packages/nexus-contracts/src/generated/`, `crates/nexus-contracts/src/generated/`) and notify platform team for contracts upgrade.
- **Evidence**: `pnpm run codegen` clean; `pnpm run validate-schemas` clean; `git diff --exit-code packages/nexus-contracts/src/generated/ crates/nexus-contracts/src/generated/` after codegen; platform team confirms receipt and compatibility.

## 7) Cross-cutting policy

### Branch strategy

- Integration branch: `feature/v1.5-stabilization` from `main`.
- Per-WS topic branches off the integration branch when ≥2 concurrent writers.
- Merge to `main` only after M4 gate passes.

### CI requirements (mandatory before merge)

- `pnpm run validate-schemas` — clean.
- `pnpm run codegen` + `verify-codegen` — no diff on generated output.
- `cargo +nightly fmt --all -- --check` — clean.
- `cargo clippy --all -- -D warnings` — clean.
- `SQLX_OFFLINE=true cargo sqlx prepare --workspace --all -- --all-targets --check` — passes.
- `cargo test --workspace` — green.
- CI linkage gate (no ACP SDK in daemon) — passes (inherited from V1.4; still applies).

### sqlx compile-time macro policy

- All new queries MUST use `sqlx::query!()` / `sqlx::query_as!()` / `sqlx::query_scalar!()`.
- Runtime `sqlx::query()` only acceptable for DDL, PRAGMA, or truly dynamic SQL — each usage MUST include a `// SAFETY: dynamic SQL — compile-time macro not applicable.` comment explaining why.
- After any query change: `cargo sqlx prepare --workspace --all -- --all-targets` and commit `.sqlx/` metadata.

### Platform sync cadence

- Nexus schema changes (WS-E) trigger immediate `pnpm run codegen` + commit of generated output.
- Platform team notified after each schema merge to coordinate `@42ch/nexus-contracts` upgrade.
- Platform can wait for nexus schema work to complete before starting dependent plan work (per Q4 decision).

## 8) Risk register

### RISK-V15-01 — `graph-flow` 0.3.x breaking change

- **Source**: inherited from V1.4 RISK-02.
- **Symptom**: upstream releases a 0.3.x with API changes during V1.5 build.
- **Control**: pin `=0.2.3` in `Cargo.toml`; `OrchestrationEngine` adapter trait isolates callers; if upstream breaks, upgrade in a separate plan. WS-B R1 (session tracker) and R3 (FlowRunner clone) work directly with the pinned version — no expectation of upstream changes in V1.5 window.

### RISK-V15-02 — Platform dependency timing

- **Symptom**: platform V1.5 plans 36–43 need nexus schema changes before they can proceed; nexus schema work in WS-E is delayed or produces incompatible shapes.
- **Control**: WS-E begins early (parallel with WS-A/B); schema changes are additive and backward-compatible; platform team has visibility into nexus commits; `pnpm run codegen` CI gate catches drift immediately.

### RISK-V15-03 — Cron crate selection constraints

- **Symptom**: neither `tokio-cron-scheduler` nor hand-rolled approach satisfies all four hard constraints from [crate-selection-best-practices.md](crate-selection-best-practices.md) §3.7 without significant wrapper complexity.
- **Control**: evaluate both candidates at WS-D design time; if neither is acceptable, WS-D scope reduces to a `Scheduler` trait placeholder (same as V1.4 deferral state) and cron triggers slip to V1.6 — the rest of V1.5 (WS-A/B/C/E) is unaffected.

### RISK-V15-04 — Residual discovery during cleanup

- **Symptom**: WS-A or WS-B cleanup work reveals additional defects beyond the 19 known residuals, expanding scope.
- **Control**: each WS has a bounded scope defined by the residual inventory. New findings are triaged: if `critical`/`high`, address in V1.5; if `medium`/`low`/`nit`, log as a new residual and evaluate for V1.5 inclusion or V1.6 deferral. Scope expansion beyond the 19 known items requires PM approval.

### RISK-V15-05 — `context.summarize` LLM dependency

- **Symptom**: the capability requires a user-configured LLM endpoint (via `judge.llm` infrastructure); if no LLM is configured, the capability cannot function, and presets that declare `llm_summarize` context updates will silently skip them.
- **Control**: capability returns `CapabilityError::TransientExternal` when no LLM is available; preset execution continues on the previous `core_context` version; clear CLI feedback when a context_update is skipped due to missing LLM configuration.

### RISK-V15-06 — Schedule FSM change regression

- **Symptom**: fixes for R1–R7 in WS-A introduce regressions in existing V1.4 schedule functionality (CRUD, dependency chains, context versioning).
- **Control**: WS-A must pass the full V1.4 regression gate (R11–R14 from V1.4 compass §6) before merge; new V1.5 regression tests (R16–R19) cover the specific FSM edge cases.
