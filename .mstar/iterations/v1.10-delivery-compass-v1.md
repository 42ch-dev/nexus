## 2) V1.10 定位

**V1.10 = 「Device Flow Login + Device Identity」**

V1.9 完成了 Preset Ecosystem + Hardening。V1.10 聚焦两个由 platform V1.10 compass §12 驱动的 OSS 端任务：

1. **CLI Device Flow Login** — 实现 `nexus42 auth login` 的真实 RFC 8628 Device Authorization Grant，从 mock daemon 代理切换到直接调用 platform 端点。
2. **DeviceID Header** — 生成持久化机器标识，并在所有 platform HTTP 请求中携带 `X-Device-ID` header。

**V1.10 一句话**：让 CLI 用户能通过真实的 OAuth 2.0 Device Flow 登录 platform，并获得 rate-limit 优先的 device identity。

## 3) Confirmed decisions (locked)

- **Architecture: CLI calls platform directly for device flow.** The current mock path (CLI → daemon → mock response) is replaced with CLI → platform (real HTTP). Rationale: (a) platform compass §12 says "CLI 调用 platform POST /api/v1/auth/device/code"; (b) daemon is a local supervisor, not an auth proxy; (c) eliminates the daemon-as-dependency requirement for auth login.
- **Token storage:** Platform JWT stored in `~/.nexus42/auth.json` (extend existing `AuthStore`). Currently `AuthStore` holds creator API keys only; V1.10 adds a `user_token` field. This avoids daemon SQLite dependency for auth state and makes `nexus42 auth status` work without daemon.
- **DeviceID generation:** One-time generation on first CLI platform request. Store a random UUID v4 to `~/.nexus42/device-id`. Persisted as plain text (0600 permissions), not in auth.json or SQLite.
- **DeviceID injection:** Add `X-Device-ID` header to all `PlatformClient` (registration/verification) and any new device flow HTTP client. The `SyncClient` may also need it if it makes platform requests — but the sync client already has its own bearer auth; deviceID is additive, not replacing auth.
- **Daemon mock endpoints:** **Fully remove** all mock auth HTTP endpoints from daemon. The entire `crates/nexus42d/src/api/handlers/auth.rs` file is removed (4 mock handlers: `status`, `device_authorization`, `exchange_token`, `logout`). The `crates/nexus42d/src/auth/device_flow.rs` module (TD-10 stub) is removed. The `crates/nexus42d/src/api/handlers/context.rs` mock `assemble()` handler is removed (returns empty arrays — full mock). The CLI → daemon local HTTP dependency for auth is an incorrect architecture and must be completely eliminated. Any mock/placeholder needs are handled in the CLI internally.
- **CLI context assembly fallback:** `nexus42 context assemble` currently calls daemon's mock `/v1/local/context/assemble`. After removing the daemon handler, the CLI returns a "not yet available" error internally. Code and tests tagged with TODO markers for future platform integration.
- **Branch strategy:** `feature/v1.10` from `main`.

## 5) Platform contract (external, reference only)

### Device Flow Endpoints (deployed)

**`POST /api/v1/auth/device/code`**
- Input: `{ client_id?: string, scope?: string }` (both optional)
- Output: `{ success: true, data: { device_code, user_code, verification_uri, verification_uri_complete?, expires_in, interval } }`
- Rate limit: `withDeviceRateLimit()` — prefers `X-Device-ID` over IP

**`POST /api/v1/auth/device/token`**
- Input: `{ device_code: string, client_id?: string }`
- Output: `{ success: true, data: { access_token, token_type: "Bearer", expires_in } }`
- Error responses (400): `{ success: false, error: { code, message, details: { error: "authorization_pending"|"slow_down"|"expired_token"|"access_denied" } } }`.
- Rate limit: `withDeviceRateLimit()` — prefers `X-Device-ID` over IP

### Platform JWT Claims (snake_case)

```json
{
  "sub": "<userId>",
  "iss": "nexus-platform",
  "aud": "nexus-api",
  "userId": "<userId>",
  "tenantId": "<tenantId | null>",
  "tier": "free",
  "entitlements": {
    "official_creator_eligible": false,
    "max_concurrent_jobs": 1,
    "runtime_policy": "...",
    "memory_structured_write": false,
    "memory_vector_index": false
  },
  "iat": ...,
  "exp": ...
}
```

### Error Codes

| Platform error code | HTTP status | Meaning |
|---|---|---|
| `authorization_pending` | 400 | User hasn't approved yet — keep polling |
| `slow_down` | 400 | Polling too fast — increase interval |
| `expired_token` | 400 | Device code expired — restart flow |
| `access_denied` | 400 | User denied authorization |
| `VALIDATION_ERROR` | 400 | Invalid input |
| `INTERNAL_ERROR` | 500 | Platform internal error |

## 6) V1.10 workstreams

### WS-A — CLI Device Flow Login (TD-10 closure)

- **Goal**: Replace the mock daemon-proxy device flow with real platform-direct RFC 8628 implementation. TD-10 closed.
- **Design inputs**: TD-10 residual; platform device flow endpoints (§5); `crates/nexus42/src/auth/user_auth.rs` (current mock flow).
- **Dependencies**: Platform endpoints deployed (done).
- **In scope**:
  - **New HTTP client**: Create a lightweight platform HTTP client for device flow (or extend `PlatformClient` in `nexus-sync`). Handles `POST /api/v1/auth/device/code` and `POST /api/v1/auth/device/token`. Uses `platform_url` from `CliConfig`. Includes `X-Device-ID` header (see WS-B).
  - **Rewrite `login()`**: Replace daemon-calling `login()` in `user_auth.rs` with direct platform calls. Implement RFC 8628 polling loop:
    1. POST `/device/code` → get `device_code`, `user_code`, `verification_uri`, `expires_in`, `interval`
    2. Print verification URI + user code (same UX as current)
    3. Poll `/device/token` with `device_code`, respecting `interval` and `slow_down`
    4. On success, store platform JWT in `AuthStore`
  - **Extend `AuthStore`**: Add `user_token: Option<UserTokenState>` field to `AuthStore` struct (`crates/nexus42/src/auth/mod.rs`). `UserTokenState` stores `access_token`, `token_type`, `expires_at` (ISO 8601), `user_id`. Persist to `~/.nexus42/auth.json` with 0600 permissions.
  - **Rewrite `status()`**: Read user token from `AuthStore` (not daemon). Show authenticated/not + user_id + expiry without requiring daemon running.
  - **Rewrite `logout()`**: Clear user token from `AuthStore` (local only; no platform endpoint needed for device flow logout).
  - **Remove daemon auth handlers entirely**: Delete `crates/nexus42d/src/api/handlers/auth.rs` (all 4 mock handlers). Update daemon router in `auth_middleware.rs` to remove `auth_routes` — the `GET /v1/local/auth/status`, `POST /v1/local/auth/device`, `POST /v1/local/auth/token`, `POST /v1/local/auth/logout` routes are gone. Update `auth_middleware.rs` tests that reference these routes.
  - **Remove daemon auth internals**: Delete `crates/nexus42d/src/auth/device_flow.rs` (TD-10 stub). Remove `device_flow` from `crates/nexus42d/src/auth/mod.rs` re-exports. Keep `session.rs`, `creator_session.rs`, `token_manager.rs` (they serve daemon-internal middleware, not CLI auth).
  - **Remove daemon DB table `device_code_sessions`**: This SQLite table only existed for the mock device flow. Drop it via a new migration, or remove from initial schema if the daemon's auth module is being cleaned.
  - **Update `login_with_token()`**: Keep the dev-mode helper but have it write to `AuthStore` instead of calling daemon. Or deprecate it if the main `login()` flow covers the use case.
  - **Remove daemon context mock**: Delete `crates/nexus42d/src/api/handlers/context.rs` (mock `assemble()` handler that returns empty arrays). Remove `context_routes` from daemon router. Update daemon `mod.rs` to drop the handler module re-export.
  - **CLI context assemble fallback**: Update `crates/nexus42/src/commands/context.rs` to return a "Context assembly not yet available in V1.10. This feature is planned for a future release." error instead of calling the daemon's now-removed endpoint. Add TODO markers in code and tests. Update `crates/nexus42/src/context/client.rs` tests to match the new error behavior rather than expecting mock data.
- **Out of scope**:
  - Token refresh (platform JWT refresh mechanism — future version).
  - Daemon SQLite token storage for device flow (deprecated, not removed).
  - Creator API key handling changes (existing flow unchanged).
  - `nexus42 auth login --flow device` flag — the existing `nexus42 auth login` command becomes device flow by default (it's the only flow). The `--flow` flag can be added later for other grant types.
- **Evidence**:
  - Unit test: platform `/device/code` call returns valid response.
  - Unit test: platform `/device/token` handles `authorization_pending`, `slow_down`, `expired_token`, `access_denied`.
  - Unit test: `AuthStore` round-trips user token (serialize + deserialize).
  - Unit test: `login()` polling loop exits correctly on success / expiry / denial.
  - Integration test: `nexus42 auth login` starts flow (prints verification URI).
  - `cargo test --workspace` green; clippy/fmt clean.

### WS-B — DeviceID Header

- **Goal**: Generate a persistent machine identifier and inject `X-Device-ID` header into all platform HTTP requests.
- **Design inputs**: Platform rate limit middleware (`getRateLimitId()` prefers `X-Device-ID` over IP); `PlatformClient::build_request()` in `nexus-sync`.
- **Dependencies**: None (independent of WS-A, though WS-A's new HTTP client should also include the header).
- **In scope**:
  - **DeviceID generation**: Generate a random UUID v4 on first use. Store at `~/.nexus42/device-id` (plain text, 0600 permissions on Unix). Add `device_id_path()` to `nexus-home-layout` crate.
  - **DeviceID provider**: New module `nexus-sync/src/device_id.rs` (or inline in platform_client.rs). Function `get_or_create_device_id(home: &Path) -> Result<String>` that reads existing or creates new.
  - **Inject into `PlatformClient::build_request()`**: Add `.header("X-Device-ID", &self.device_id)` to the request builder. `PlatformClient::new()` now takes a `device_id: String` parameter.
  - **Inject into WS-A device flow client**: The new device flow HTTP client also includes `X-Device-ID`.
  - **Inject into `SyncClient`**: If `SyncClient` makes platform requests, add `X-Device-ID` header. (Check during implementation — `SyncClient` may not talk to platform directly.)
  - **Update CLI bootstrap**: In `nexus42` main/init, resolve device ID and pass to all platform client constructors.
- **Out of scope**:
  - DeviceID rotation or revocation.
  - Multi-device sync (platform side already handles per-device rate limiting).
  - DeviceID validation (platform already validates format).
  - `nexus42 daemon` DeviceID — daemon doesn't make platform requests directly.
- **Evidence**:
  - Unit test: first call creates `~/.nexus42/device-id` with valid UUID.
  - Unit test: second call reads existing device ID (stable across restarts).
  - Unit test: platform client request includes `X-Device-ID` header.
  - Unit test: missing `~/.nexus42/device-id` triggers regeneration.
  - `cargo test --workspace` green; clippy/fmt clean.

## 9) Cross-cutting policy

### Branch strategy

- Integration branch: `feature/v1.10` from `main`.
- WS-A and WS-B can proceed in parallel (independent modules: auth CLI vs platform client).
- Both small enough to potentially complete in a single implementation session.

### CI requirements (mandatory before merge)

- `cargo +nightly fmt --all -- --check` — clean.
- `cargo clippy --all -- -D warnings` — clean.
- `cargo test --workspace` — green.

### Platform sync

- V1.10 platform endpoints are already deployed. No platform-side blocker.
- V1.10 has **no expected new platform dependency** — both workstreams are OSS-side implementation of existing platform contracts.

### Daemon mock endpoint removal

- **`crates/nexus42d/src/api/handlers/auth.rs`** → **entire file deleted.** All 4 handlers (`status`, `device_authorization`, `exchange_token`, `logout`) are removed. These were mock implementations that should never have been a CLI dependency.
- **`crates/nexus42d/src/api/handlers/context.rs`** → **entire file deleted.** Mock `assemble()` handler that returned empty arrays is removed.
- **`crates/nexus42d/src/auth/device_flow.rs`** → **entire module deleted.** TD-10 stub `verify_device_code()` removed.
- **`crates/nexus42d/src/api/auth_middleware.rs`** → remove `auth_routes` and `context_routes` from test `build_router()`. Update tests that reference the removed routes.
- **`crates/nexus42d/src/api/handlers/mod.rs`** → remove `context` and `auth` module re-exports (if present — check during implementation).
- **`crates/nexus42d/src/auth/mod.rs`** → remove `device_flow` from re-exports.
- **Daemon SQLite** → drop `device_code_sessions` table. Keep `auth_tokens` table (used by `require_auth` middleware for daemon-internal route protection). No migration needed for `auth_tokens` — it's already in the initial schema.
- **CLI → daemon HTTP dependency for auth and context** → **completely eliminated.** `user_auth.rs` no longer imports or uses `DaemonClient` for auth. `ContextClient` no longer proxies through daemon — returns a local placeholder error instead.
