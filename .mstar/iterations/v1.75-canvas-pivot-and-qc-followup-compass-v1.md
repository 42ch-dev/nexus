### Phase 2b technical decisions (LOCKED — `@product-manager` + `@architect` countersigned)

> `@architect` codebase verification completed 2026-06-29. The architect corrected one persistence detail from the PM draft: V1.65 outline prose lives in per-chapter `outline_path` markdown files, while V1.72 `outline_revision` CAS lives in the work-level `Outlines/outline.md` frontmatter. The V1.75 content patch must bridge both without touching `body_path`. The corrected persistence model is locked below.

- **Outline-prose patch field (LOCKED — architect countersigned, corrected persistence detail)**: add a `content` field (type `string`, description "Chapter outline prose notes — the rich-text outline content for this chapter, replacing the V1.65 whole-document editor's per-chapter outline content.") to `schemas/local-api/canvas/outline/outline-patch-chapter-set.schema.json`. Field name **`content`** is locked because it matches the existing V1.65 read/write shape: `ChapterOutline.content`, `PutChapterOutlineRequest.content`, and `usePutChapterOutline({ content })`. Current code confirms the parity gap: `outline-patch-chapter-set` is metadata-only (`title`/`slug`/`planned_word_count`/`actual_word_count`/`volume`/`status`), while `chapter-page.tsx` edits `outline.data.content` and saves it through the V1.65 PUT route. **Persistence correction**: the outline prose content is not in the work-level `Outlines/outline.md` body today; it lives in the per-chapter markdown file referenced by `work_chapters.outline_path` (e.g. `Works/{work_ref}/Outlines/chapters/ch01-outline.md`). V1.72 CAS lives separately in `Works/{work_ref}/Outlines/outline.md` frontmatter (`outline_revision`). Therefore the V1.75 daemon patch must, under the existing `RuntimeLockGuard` and `base_revision` check, write the per-chapter `outline_path` markdown content with the same temp-file + rename + file fsync + dir fsync durability pattern, then bump/persist the work-level `outline_revision` via the existing `outline.rs` frontmatter writer. If `outline_path` is missing, use the existing V1.65 fallback path and persist it through `work_chapters::update_outline_path` before writing. Regression tests must prove: content patch updates the per-chapter outline file, stale `base_revision` returns 409, `outline_revision` bumps, and `body_path` + body file content remain untouched.
- **Body-ownership invariant (LOCKED — architect countersigned)**: the V1.75 `content` patch is an OUTLINE-prose write only. It MUST NOT mutate `work_chapters.body_path`, call the chapter body writer, or rewrite files under `Works/{work_ref}/Stories/**`. Codebase verification: `work_chapters` stores `outline_path` and `body_path` as distinct columns; V1.65 `GET/PUT /chapters/{n}/outline` reads/writes `outline_path`, while `GET /chapters/{n}/body` reads `body_path`. ## 6. Risk notes

### 6.1 Parity-gap underestimation risk (Track A)

The parity audit found the metadata-only gap (A1-A4 closes it). Risk: the V1.65 editor has rich-text capabilities (specific TipTap extensions, markdown round-trip edge cases) the in-inspector TipTap might not perfectly mirror. **Mitigation**: A4 parity verification checklist + round-trip vitest; if a gap is found, extend the inspector TipTap config before A5 retire. The retire (A5) is GATED on A4 parity passing — do not retire until parity is verified.

### 6.2 Retire regression risk (Track A Phase 2)

Removing the V1.65 editor + putOutline save path could break existing user workflows or tests that depend on chapter-page editing. **Mitigation**: the chapter-page morph keeps the read-only body render (the reading/preview value stays); only the EDITING moves to canvas. Existing chapter-page tests updated for the read-only morphology. The `usePutChapterOutline` removal is gated on confirming no other consumer (A6 dead-code audit).

### 6.3 Body-ownership invariant (Track A)

The outline-prose `content` patch MUST NOT touch the chapter `body_path` (AI-owned). **Mitigation**: A2 daemon validation explicitly scopes the write to the per-chapter `outline_path` markdown file; regression test proving both the `body_path` DB column and body file bytes are untouched after a content patch (mirrors V1.72 §6.4 body-ownership invariant).

### 6.4 Codegen Eq regeneration blast radius (Track B B1)

Fixing the codegen Eq-stripping regenerates Rust derives across ALL generated structs (Eq added back to non-f64 structs). Large mechanical diff but deterministic. **Mitigation**: regenerate + commit; `verify-codegen` + schema-drift-detection gate; the added `Eq` derives are additive (more derives, not fewer) — no consumer breaks.

### 6.5 Cross-track file-disjointness

P0 (outline) and P1 (world-kb/codegen) are file-disjoint. The only shared surface is generated codegen output (P0 A1 regenerates outline DTOs; P1 B1 regenerates ALL Rust derives via the codegen Eq fix). **Mitigation**: PM serializes the codegen+commit on the integration branch after merging both (P0 and P1 do schema/codegen-source edits in worktrees; PM runs the canonical codegen on integration). Same coordination as V1.74.

### 6.6 Two-file outline write + CAS consistency (Track A A2)

The V1.75 content patch spans two files: the per-chapter `outline_path` markdown file holds prose, while work-level `Outlines/outline.md` frontmatter holds `outline_revision`. A crash or I/O failure between writes could leave prose updated without a revision bump (or vice versa) if implemented naively. **Mitigation**: do both writes under `RuntimeLockGuard`; prepare/flush temp files before rename; use the existing temp+rename+file fsync+dir fsync pattern for both targets; add a failure-path regression around stale revision/conflict behavior; if a post-content-write revision persist fails, return 500 and preserve a retry-safe recovery path (prefer best-effort rollback to the pre-read outline content when feasible).

### 6.7 Inspector line cap + conflict-label drift (Track A A3)

`chapter-inspector.tsx` is already 238 lines, so inlining TipTap would violate the V1.73 split cap and make the inspector hard to review. The existing conflict modal also lacks a content-specific changed-field label. **Mitigation**: extract `chapter-outline-content-editor.tsx` (or equivalent) for TipTap + toolbar, keep the inspector ≤250 lines, and extend `OutlineChangedField` / `changedFieldsOf` with `chapter_outline_content` so conflict UX accurately reports prose edits.
