## 1. Scope / non-goals / out-of-scope

### 1.1 In scope

**Track A — Outline & Structure Authoring UI (V1.65 lead slice)**

Extend the V1.64 `apps/web` SPA from control-room (read + setup) to an **authoring entry surface** for chapter outlines and structure. All screens consume the V1.63 codegen TS contracts + the new V1.65 chapter-content Local API (Track B/P0 backend).

- **Chapter structure table** (per-Work): chapter #, title, slug, planned word count, volume, status (`not_started`/`outlined`/`draft`/`finalized`/`published`), actual word count. Sortable by chapter #. Multi-Work switcher (reuse V1.64 Works dashboard entry).
- **Outline rich-text editor**: edit a chapter's `outline_path` markdown in a rich-text editor (framework decision in §5 item #1 — TipTap recommended). Soft-concurrency: warning UI when editing outline of a chapter already in `draft`/`finalized` status ("this chapter is already drafted; changing the outline will not auto-redraft; the next orchestration draft will be based on the new outline"). Atomic file write on save (reuse the reconcile atomic-write pattern), DB metadata (`outline_path`, `updated_at`) updated in the same transaction.
- **Structure CRUD**: title / slug / planned word count / volume assignment / status progression (`not_started → outlined` only — reverse transitions gated). Protection policy for `finalized`/`published` chapters: confirmation dialog (hard-block deletion; warn on structural edits). Detail in §5 item #3.
- **Body read-only rendering**: render a chapter's `body_path` markdown (frontmatter-aware — surface status/metadata, render body prose read-only). Right-click context menu: **"Copy path"** only (browser clipboard write; path sourced from API). **No** "open with" / "reveal in file manager" in V1.65 (deferred to V1.66 Tauri shell per Q5).
- **New Local API consumed** (backend built in P0): `GET /v1/local/works/{work_id}/chapters` (list, cursor + `items`), `GET /v1/local/works/{work_id}/chapters/{n}` (detail incl. `outline_path`/`body_path`), `GET /v1/local/works/{work_id}/chapters/{n}/outline` (read outline markdown), `PUT /v1/local/works/{work_id}/chapters/{n}/outline` (atomic write outline), `PATCH /v1/local/works/{work_id}/chapters/{n}` (structure: title/slug/planned_word_count/volume/status progression), `GET /v1/local/works/{work_id}/chapters/{n}/body` (read body markdown, read-only). All follow V1.64 cursor + `ErrorResponse` conventions (`local-api-surface-conventions.md`).
- **DESIGN.md increment** (Prepare gate, architect-owned Phase 2b): editor component tokens, data-table tokens, context-menu tokens appended to `apps/web/DESIGN.md` (Standard → Standard+; Production still V1.66). Required before P2 implement.
- **`tauri-api` adapter boundary preserved**: new screens route through `NexusClient` → `BrowserClient` (no browser-only APIs in editor core); V1.66 `TauriClient` swaps transparently.

## 5. Open design items (resolve during Phase 2b / implement)

1. **Outline rich-text editor framework.** TipTap (React-friendly, ProseMirror-based, markdown-bridge mature) vs Lexical (Meta, smaller bundle) vs Slate (lower-level). **Lean**: TipTap (V1.64 SPA is React 18; TipTap's `tiptap-markdown` extension gives MD round-trip; matches the "edit a markdown file" semantics with least bespoke serializer code). MD fidelity strategy: restrict to a markdown subset (headings, lists, bold/italic, code, blockquote, links) — preserve unknown frontmatter/HTML as raw blocks. **Owner: P2 implementer, architect-validated Phase 2b.**

2. **Outline PUT concurrency semantics.** Soft model chosen (Q2/Q3): no hard lock; UI save = atomic file write + DB `outline_path`/`updated_at` update in one transaction. Orchestration reads outline at draft-time (reads whatever is on disk → natural snapshot). UI warns when editing outline of a chapter already `draft`/`finalized`. **Sub-decision**: does a successful outline PUT on a `draft` chapter bump status back to `outlined` (forcing re-draft)? **Lean: no** — status is author-controlled via explicit structure-CRUD; outline edit is content-only. **Owner: P0 implementer (PUT semantics) + P2 implementer (warning UX), architect-validated.**

3. **Structure CRUD protection for `finalized`/`published` chapters.** Deletion = hard-block (with reason); structural edits (title/slug/word-count/volume) on `finalized` = confirmation dialog; status reverse-transition (`finalized → outlined`) = explicit action with cascade note (re-draft required). **Owner: P0 implementer (DB constraints) + P2 implementer (UX), product-manager-validated.**

4. **Body read-only render fidelity.** Markdown renderer choice (react-markdown + remark-gfm matches V1.64 stack). Frontmatter parsing: surface `status`/`title` as a read-only header strip above the rendered body. **Owner: P2 implementer.**

5. **`work_profile` field classification (R-V164-P2-G1).** Additive optional (daemon defaults when absent — safer, fully backward-compatible) vs pre-1.0 breaking required (cleaner schema, forces explicit choice). **Lean: additive optional** — preserves V1.64 UI behavior (daemon assigns) while letting V1.65 UI select explicitly; no breaking change needed. **Owner: @architect Phase 2b.**

6. **Chapter list pagination size.** Cursor-based, default page size (lean: 50, matching peers). Whether to expose `status` filter query param in V1.65 (lean: yes — the structure table benefits from status filtering). **Owner: P0 implementer.**

7. **DESIGN.md increment scope.** Editor component (toolbar, surface, save-state indicator), data-table (structure table — rows, status badges, inline edit), context-menu (copy-path). Light + dark tokens. Production-level polish (animations, full modal/menu specs) deferred V1.66. **Owner: @architect Phase 2b.**

8. **`tauri-api` adapter reuse audit.** Confirm P2 editor routes through `NexusClient` interface (no direct `fetch` bypasses); confirm no browser-only API leaks into editor core (so V1.66 `TauriClient` swaps cleanly). **Owner: P2 implementer, architect-reviewed.**

9. **Outline PUT authorization / path guard.** The outline `body_path`/`outline_path` is DB-sourced; the PUT endpoint must apply the same W-002-style path guard as `host_tool_handlers.rs` (resolve within workspace root, reject traversal). **Owner: P0 implementer.**
