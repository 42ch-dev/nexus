## 1. Scope / non-goals / out-of-scope

### 1.1 In scope

**Track A — Local Web UI (Control Room + Setup MVP)**:

A brand-new `apps/web` React + Vite + TypeScript SPA, daemon-served and Tauri-ready.

- **App scaffold** (`apps/web`): Vite + React 18 + TypeScript + TailwindCSS + shadcn/ui + TanStack Query (server state) + React Router. pnpm workspace `apps/*` registration. CI build job. `tauri-api` adapter boundary (browser `fetch localhost:<port>` impl today; Tauri `invoke` impl behind the same interface for V1.65).
- **DESIGN.md** (Prepare gate, `@architect`-led with `@product-manager` input): design tokens — Color (light + dark), Typography, Spacing, Elevation, Motion, Shapes, Component primitives, Voice & Content. Completeness level: **Standard** (MVP-appropriate; Production-level polish is V1.65+). Lives at `apps/web/DESIGN.md`; referenced from the `web-ui.md` spec.
- **Control Room screens (READ-heavy — *visibility* for the author who wants to see without the terminal)**:
  - Works dashboard: list (cursor-paginated post-F-P1) + status + completion %. Work detail view.
  - Orchestration sessions view: list + per-session status (running/completed/failed).
  - Schedule view: cron roles per Work + next-fire (UTC) + local-time display (parity with CLI `creator works cron`).
  - Capability registry browser: list `nexus.*` capabilities + descriptions.
  - Findings view: list findings (post-F-P2 endpoint) with status/severity filtering.
- **Setup screens (writes — *configuration* of the authoring starting point)**:
  - Work CRUD: create / patch (status, stage) / archive. Foundational for any authoring.
  - Preset management: full CRUD (list / show / create / update / delete / **validate** — `validate` dry-run is product-priority #1 per @product-manager: highest-trust affordance for a non-CLI author "is this safe before I commit?"). Schemas already promoted in V1.63.
- **Daemon serving wiring**: `tower-http::ServeDir` (dev/fallback) + `rust-embed` embed of `apps/web/dist` into the `nexus42` binary (release); release build pipeline; dev = Vite dev server proxying `/v1/local/*` to the running daemon. Unauthenticated static SPA shell; data endpoints are keyless on loopback (V1.20 model).
- **CLI entry** (small): a `nexus42 daemon start` note that the UI is served at `http://localhost:<port>/`, and optionally a `nexus42 ui` / `nexus42 web` convenience command (starts daemon if not running + opens browser). Final shape decided in §5 item #4 + @architect review.

## 5. Open design items (resolve during implement)

1. **F-P1 Works cursor migration: breaking shape change handling.** The Works list endpoint shape changes (offset/limit → cursor + `PaginationInfo`). Since the only pre-V1.64 consumer is codegen itself (no shipped external TS app), landing the break alongside the first UI consumer is safe. Decision: land in P0; UI P2 consumes the cursor shape directly (no adapter). **Owner: P0 implementer.**

2. **F-P1 vs F-P3 decoupling.** The V1.63 surface audit bundled "Works cursor + rename `works`→`items`" under F-P1. This iteration closes the **pagination** part (F-P1) but **defers the array rename** (F-P3) — renaming `works`/`schedules`/`sessions`/`capabilities` arrays is a multi-handler breaking change better done as a coordinated sweep. P0 keeps the existing array names; the UI maps them via a thin TanStack Query transformer. **Owner: P0 implementer (pagination only) + P2 implementer (adapter).**

3. **`rust-embed` vs `include_dir` vs serve-from-disk.** Release embedding strategy for the SPA bundle into the `nexus42` binary. `rust-embed` is the recommended default (macro-friendly, handles gzip, rebuilds on dist change). Serve-from-disk (`~/.nexus42/web/`) is the fallback if embedding creates CI/release-pipeline friction. **Architectural sub-decisions for P3**: (a) embed implementation — `rust-embed`-backed custom service in release vs `tower-http::ServeDir` from disk in dev/fallback; (b) route mount — UI at server root `/` vs `/ui` (lean: `/` for V1.64, SPA is the primary surface; revisit if root conflicts with future endpoints); (c) SPA fallback + cache headers (`index.html` fallback for client-side routes; cache-control for hashed assets, no-cache for index.html). Decision in P3. **Owner: P3 implementer.**

4. **Web UI entry command shape.** Options: (a) `nexus42 daemon start` serves + logs the UI URL at `http://localhost:<port>/`; (b) add `nexus42 ui`/`nexus42 web` convenience that starts daemon (if not running) + opens the OS browser. **PM-locked (per @product-manager rec)**: ship **(a)** as the V1.64 target (minimal CLI-surface change, never the sole path); **(b)** optional in P3 only if low-cost, deferred to V1.65 otherwise. **Owner: P3 implementer.**

5. **TanStack Query vs SWR vs vanilla fetch.** Server-state library for the SPA. TanStack Query is the recommended default (matches the cursor-pagination + ErrorResponse retry model; mature). Decision in P1. **Owner: P1 implementer.**

6. **DESIGN.md completeness level.** `Standard` (MVP-appropriate: tokens defined, light + dark, core components specified; Production-level polish/animations deferred). Promote to `Production` in V1.65 alongside Tauri shell. **Owner: @architect (Phase 2b).** — **DONE Phase 2b**: `apps/web/DESIGN.md` landed at Standard with first-class dark tokens; Production gaps (separate `DESIGN.dark.md`, visual QA, full modal/menu/toast specs, Tauri/mobile refinements) deferred to V1.65.

7. **`tauri-api` adapter contract.** The boundary abstraction (browser `fetch localhost` vs Tauri `invoke`). Must be defined in P1 so P2 screens are transport-agnostic. Lean: a `NexusClient` interface with two impls (`BrowserClient`, stubbed `TauriClient`). **Architectural sub-decision**: the `BrowserClient` operates same-origin against the daemon's loopback listener (V1.20 keyless-localhost model — no API key on `localhost`); confirm same-origin semantics when the SPA is served from the same daemon port. **Owner: P1 implementer.**

8. **CI scope.** Add a `web-build` CI job (pnpm install `apps/web` + `pnpm --filter web build` + typecheck). Whether to add Tauri CI prep now (no — Tauri is V1.65; CI matrix then). **Owner: P1 implementer + @ops-engineer review.**

9. **Auth model terminology (clarification, per @product-manager risk #3).** The Local API data endpoints are **keyless on loopback** (V1.20 model) — NOT "API-key-gated" in the V1.64 local-UI context. The static SPA shell carries no data; all data flows through the loopback Local API. The `require_api_key` middleware applies only when keyed-all mode is enabled (non-loopback). **Clarified in `daemon-runtime.md` (architect amend) + `web-ui.md` §4.**
