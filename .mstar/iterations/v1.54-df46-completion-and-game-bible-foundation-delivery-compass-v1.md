## 6. Deferred design (compass §6)

This section captures **interface design sketches** for deferred-feature items that are planned but not implemented in V1.54. Future iteration compasses can reference these designs as starting points.

### 6.1 DF-29 — registry.refresh interface

**Purpose**: Allow agents to refresh capability registry from a remote source (CDN/repository).

**Design sketch**:
- Capability: `nexus.registry.refresh`
- Admission: network-available + policy-enabled gate
- Input: `source` (URL or named channel), optional `force` flag
- Output: diff of added/removed/changed capabilities since last refresh
- Persistence: refreshed registry version + timestamp stored in local DB
- Failure modes: network-unavailable → stale-continue; source-untrusted → policy_blocked

**Blocked by**: No network/CDN integration in local daemon. Requires `nexus-cloud-sync` or equivalent transport layer.

**Re-evaluate at**: V1.56+ when local network layer is available.

### 6.2 DF-31 — workspace.open / workspace.commit interface

**Purpose**: Allow agents to open a workspace context (file-level or directory-level) and commit changes back.

**Design sketch**:
- Capability: `nexus.workspace.open`, `nexus.workspace.commit`
- `workspace.open`: accepts `path` (relative to workspace root); returns `session_id` + file snapshot
- `workspace.commit`: accepts `session_id` + `changes[]` (diffs or full content); validates no conflicting writes
- Admission: workspace-scoped policy; creator-identity gate
- Persistence: session lives in daemon memory; commit writes through `nexus-home-layout`

**Blocked by**: `nexus-home-layout` wiring incomplete. V1.39 shipped Files/ subdir but no structured workspace session API.

**Re-evaluate at**: V1.55+ after nexus-home-layout refactor.

### 6.3 DF-56 — Post-V1.42 conditional routing roadmap

V1.42 shipped minimal GO/NOGO routing (llm_judge → two next edges). The remaining roadmap:

| Phase | Feature | Dependencies |
| --- | --- | --- |
| Phase 2 | Arbitrary stage-level conditional `next` (non-judge nodes) | DF-56 Phase 1 (V1.42 Done) |
| Phase 3 | Expression / rule-based routing beyond boolean GO/NOGO | Preset expression language (new spec) |
| Phase 4 | Multi-branch graphs with >2 outgoing edges + merge points | Phase 3 |
| Phase 5 | `registry.refresh` conditional edges | DF-29 |
| Phase 6 | `workspace.open` / `workspace.commit` branch inputs | DF-31 |

**Re-evaluate at**: V1.56+ for Phase 2. Phase 3-4 require a preset expression language spec (new major design).

### 6.4 DF-44 — Reference body refreshable scan pipeline

**Purpose**: Extend the static reference source registration (V1.26) with a refreshable scan pipeline that detects upstream changes and re-ingests body content.

**Design sketch**:
- Scan trigger: manual (`creator reference refresh`) or scheduled (cron)
- Refresh pipeline: fetch → diff → re-index changed bodies → update KB anchors
- Persistence: `reference_sources` table gets `last_refreshed_at`, `etag`, `body_hash`
- KB integration: source anchors in `kb_key_blocks` re-linked to refreshed body

**Blocked by**: Static registration works; auto-refresh needs network layer (same block as DF-29).

**Re-evaluate at**: V1.56+.
