## 3. Plan topology

V1.56 uses **7 registered plans** across two implement tracks (Track A + Track B) with a 3-wave topology. Wave 1 runs P0 (Track A) and P1 (Track B) in parallel; Waves 2 and 3 are sequential within each track.

**Track-ownership pattern** (PM-locked during P-1): each track has one canonical owner across all waves. Track A = `@fullstack-dev` (P0, P2). Track B = `@fullstack-dev-2` (P1, P3). Wave 1 dispatches P0 on `fullstack-dev` and P1 on `fullstack-dev-2` in parallel on separate worktrees. Wave 2 dispatches P2 on `fullstack-dev` and Wave 3 dispatches P3 on `fullstack-dev-2` — both sequential within their track. Each active writer requires a separate git worktree per `mstar-branch-worktree`.

**3-wave dispatch rhythm**:

- **Wave 1 (parallel)**: P0 (Track A) and P1 (Track B) implement in parallel on separate worktrees from `iteration/v1.56`; each merges to integration independently; each receives its own tri-review + mid-QA + targeted fix-wave. Wave 1 ends when BOTH P0 and P1 are accepted (or PM-recorded override).
- **Wave 2 (single)**: P2 (Track A) implements from `iteration/v1.56` after P0+P1 acceptance; merges to integration; receives tri-review + mid-QA + targeted fix-wave. Wave 2 ends when P2 is accepted.
- **Wave 3 (single)**: P3 (Track B) implements from `iteration/v1.56` after P0+P1+P2 merged; merges to integration; receives tri-review + mid-QA + targeted fix-wave. Wave 3 ends when P3 is accepted.
- **P-last**: runs after Wave 3 completes (all four implement plans integrated and reviewed). Report-only QA, not tri-review.

PM dispatches Wave 1 in lockstep — when both P0 and P1 are merged, PM dispatches mid-QC for both within the same dispatch round. Cross-wave: Wave 2 cannot start until Wave 1 plans have their fix-waves closed (or PM override). Wave 3 cannot start until Wave 2 is accepted.

## 4. Branch policy

V1.56 uses the project-standard **two-tier branch model** from [`.mstar/AGENTS.md`](../AGENTS.md): an iteration integration branch plus per-plan topic branches. Topic branches are created **from `iteration/v1.56`, not from `main`**. PM creates `iteration/v1.56` from `main` after this Prepare task is reviewed.

| Tier | Branch | Purpose |
| --- | --- | --- |
| Iteration integration | `iteration/v1.56` (PM creates from `main` AFTER this Prepare task is reviewed) | Single integration line and QC/QA review branch; final PR target is `main`. |
| Prepare working branch | `feature/v1.56-prepare` (where P-1 artifacts are authored) | Compass + plan stubs + tracker activation + light spec-status updates. PM commits after review. |
| P0 topic | `feature/v1.56-df31-df42-full-redesign` | DF-31 full + DF-42 full Redesign; from `iteration/v1.56` after PM signoff; merges back to integration. |
| P1 topic | `feature/v1.56-df29-registry-refresh` | DF-29 `registry.refresh`; from `iteration/v1.56` after PM signoff; merges back to integration. |
| P2 topic | `feature/v1.56-df56-independent-slice` | DF-56 independent slice; from `iteration/v1.56` after P0+P1 merged; merges back to integration. |
| P3 topic | `feature/v1.56-df56-dependent-slice` | DF-56 dependent slice; from `iteration/v1.56` after P0+P1+P2 merged; merges back to integration. |
| P-last closeout | `feature/v1.56-hygiene-and-closeout` | Spec consolidation, R-V155P2-F002 fix-wave, Profile B, tracker, residual closure; merges to integration. |

**Dispatch / merge policy**: PM dispatches Wave 1 (P0 + P1) in lockstep — when both topics branch, both implement in parallel on separate worktrees. Wave 2 dispatches P2 after Wave 1 fix-waves close. Wave 3 dispatches P3 after Wave 2 fix-waves close. P-mid captures the 3-wave rhythm; P-last is the closeout. Every implement plan must be merged to `iteration/v1.56` before its formal tri-review and mid-QA. QC/QA assignments use `Working branch: iteration/v1.56` at integrated `HEAD` for the relevant checkpoint.
