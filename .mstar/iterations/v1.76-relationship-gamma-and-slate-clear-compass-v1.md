### Phase 2b technical decisions (`@product-manager` + `@architect` LOCKED)

> Codebase-verified and locked by `@architect` in Phase 2b. Key verification points: `nexus.llm.extract` output schema is currently defined in both `crates/nexus-orchestration/src/capability/builtins/llm_extract.rs` and the normative Master `.mstar/knowledge/specs/llm-extract.md`; `LlmExtractTask` is a pure parser/invoker and does not persist; review-time candidate persistence lives in `quality_loop::persist_candidates` / `kb_extract_jobs`, with `upsert_pending_candidate` as the rescan dedup precedent; `extract_finalize` is the `kb.extract_work` direct-KeyBlock insert helper, not the pending-candidate upsert path; `kb_relationships` currently has no `needs_review` or `source` columns; GET graph currently has no query param and projects all stored relationships plus symmetric reverse projections.

- **Extraction output shape (corrected + architect LOCKED)**: extend the `nexus.llm.extract` capability output and `llm-extract.md` Master spec (not a new standalone schema unless implementation chooses a local Rust helper) from `required: ["candidates"]` to `required: ["candidates"]` plus optional `relationships` for backward compatibility. Existing `candidates` item shape remains unchanged: `{ canonical_name, block_type, summary?, confidence, source_quote }`. New `relationships` item shape is: `{ source_canonical_name, source_block_type?, target_canonical_name, target_block_type?, relation_type, custom_label?, symmetric, confidence, source_quote }`. `relation_type` uses `WorldKbRelationshipKind` snake_case values; `relation_type="custom"` requires `custom_label`; `confidence` is clamped/validated to `0.0..=1.0`; `source_quote` mirrors the existing entity candidate provenance (verbatim excerpt), because current extraction has no chapter-span locator contract. **Canonical-name resolution**: resolve endpoints by `(world_id, block_type, canonical_name)` when `*_block_type` is provided; if not provided, resolve by case-insensitive `(world_id, canonical_name)` only when exactly one non-deleted KeyBlock matches, otherwise skip + warn. **Entity-existence ordering**: review-time extraction creates pending entity candidates, not confirmed KeyBlocks, so relationship persistence MUST NOT assume entities are confirmed earlier in the same pass. Persist only relationships whose endpoints are already existing non-deleted KeyBlocks in the same world; candidates whose endpoints are missing/ambiguous are skipped + logged and may be proposed on a later rescan after author promotion. **Dedup key**: idempotent extraction upsert keyed on `(world_id, source_entity_id, target_entity_id, relation_type, COALESCE(custom_label,''), source='extraction')`; symmetric rows use this stored direction and do not insert a reverse row (per entity-scope-model §5.6.2).
- **`needs_review` + `source` schema (architect LOCKED)**: migration `crates/nexus-local-db/migrations/202606300001_kb_relationships_needs_review.sql` adds `needs_review INTEGER NOT NULL DEFAULT 0` + `source TEXT NOT NULL DEFAULT 'manual' CHECK (source IN ('manual','extraction'))` to `kb_relationships`. Existing rows: `needs_review=0`, `source='manual'`. Add index `(world_id, needs_review)` for the graph filter and `(world_id, source, source_entity_id, target_entity_id, relation_type)` (or equivalent lookup index) for extraction upsert. The `WorldKbRelationshipProjection` DTO gains `needs_review: bool` + `source: 'manual' | 'extraction'`. `source` is read-only in normal author patching; `needs_review` must be updateable by the existing patch route so Promote can clear it without inventing a second promotion state machine.
- **GET graph `needs_review` filter (architect LOCKED)**: update `get_graph` from `Path(world_id)` only to `Path(world_id) + Query<GraphQuery>`, with `include_suggested: Option<bool>`. Default is `false`: relationship storage query filters `needs_review = 0`, so suggested rows are hidden from the confirmed graph. `?include_suggested=true` includes both confirmed and suggested relationships and preserves the current symmetric reverse projection rule. The candidates endpoint remains entity-candidate-only and is unaffected.
- **Confidence-weighting UX (product-manager countersigned — LOCKED)**: the weighting curve uses **three stepped visual bands** (not continuous linear — discrete bands communicate "weak/medium/strong" at a glance, matching the existing DESIGN.md confidence tokens): **low** (<0.4): stroke 1px, opacity 30%; **mid** (0.4–<0.7): stroke 2px, opacity 60%; **high** (≥0.7): stroke 3px, opacity 100%. The **threshold filter slider** lives in the canvas toolbar (right of the Suggested toggle), default `0.0` (show all), range 0.0–1.0 step 0.05, label "Confidence ≥ 0.XX". The threshold **only applies to confirmed edges** — the Suggested pane always shows all `needs_review` rows regardless of threshold (to never hide untriaged suggestions). **Confidence badges** on edge labels use the shipped DESIGN.md `canvas-worldkb-relationship-confidence-*` tokens (low=red-700, mid=amber-700, high=green-700) as an 8px colored dot appended to the edge label; badge size is uniform (not proportional — the color band IS the proportion signal). The **Suggested pane** displays confidence prominently: each row shows a confidence badge + numeric value (e.g., "0.82 ■"), sortable by confidence descending (default — triage highest-confidence suggestions first).
- **Wire contracts (architect LOCKED)**: `wire_contracts_changed: TRUE` for Track A. Bump `@42ch/nexus-contracts` **0.11.0 → 0.12.0**. Rationale: fields are additive, but this repo is pre-1.0 and Track A changes Local API behavior (GET graph default filter for new extraction-sourced rows) plus adds a relationship-extraction DTO and relationship update input support for clearing `needs_review`; use the minor bump, not the fallback `0.11.1`.

Current codebase correction: `extract_finalize` inserts a direct `KeyBlock` for `kb.extract_work`; pending entity candidates are persisted by `quality_loop::persist_candidates` / `kb_extract_jobs`, and `upsert_pending_candidate` is the rescan dedup precedent. Relationship persistence should live beside the extraction caller after endpoint resolution, via idempotent upsert into `kb_relationships` (`needs_review=1`, `source='extraction'`, confidence, optional source_anchor_ids/metadata). Entity-existence prerequisite: skip + warn when either endpoint cannot resolve to a non-deleted KeyBlock in the same world; review-time extraction does not auto-confirm same-pass entities. - **A3 — GET graph filter + Suggested pane**: GET graph defaults `needs_review=0`; `?include_suggested=true` surfaces suggestions. The canvas toolbar gains two controls: (1) a **"Suggested (N)" toggle button** with a live count badge of un-triaged `needs_review` relationships; (2) a **confidence threshold slider** (default 0.0, right of the Suggested toggle). Toggling "Suggested" opens the **Suggested pane** — a sortable table (source entity, relation type, target entity, confidence badge + numeric value, source marker "extraction") default-sorted by confidence descending. Suggested relationships render on the graph as **dashed edges** (distinct from confirmed solid edges) with confidence-weighted stroke, plus a small "suggested" badge on the edge label.
- **A4 — Promotion/curation**: the author triages from the Suggested pane (the primary curation surface). **Per-row actions**: "Promote" (clears `needs_review` via `world_kb.patch_relationship` → the edge becomes a confirmed, solid graph edge) and "Delete" (removes the relationship row). **Bulk promote**: a "Promote all" button at the top clears `needs_review` for all visible suggested relationships (confirmation dialog for N > 1). Both actions are optimistic (immediate UI update with daemon confirmation). The Suggested badge counter updates immediately. The graph also supports inline promote/delete via click-to-select an edge and use the relationship inspector (V1.74 reuse — the inspector gains `needs_review`/`source` badges and Promote/Delete buttons when viewing a suggested relationship).
- **A5 — Confidence-weighting UX**: graph edge stroke width + opacity follow three stepped bands (low <0.4: 1px/30%, mid 0.4–<0.7: 2px/60%, high ≥0.7: 3px/100%) matching existing DESIGN.md confidence tokens. A threshold filter slider in the canvas toolbar (default 0.0) hides confirmed edges where `confidence < threshold`; edges without confidence (manual/author-created) are always visible. Suggested edges (dashed) always show regardless of threshold. Edge-label badges use the shipped DESIGN.md confidence-band colors (low=red, mid=amber, high=green) as uniform 8px dots. The Suggested pane shows confidence as a sortable column (value + colored badge), default-sorted high→low so the author triages the most reliable extraction suggestions first.

## 6. Risk notes

### 6.1 Extraction quality / flooding risk (Track A)
Extraction may propose many low-confidence co-occurrence relationships. **Mitigation**: the `needs_review` gate (default-excluded from the graph) + confidence-weighting (low-confidence faint) + the threshold filter. The author triages in the Suggested pane. If extraction is too noisy, the LLM prompt can be tuned (V1.77) — V1.76 ships the mechanism.

### 6.2 Entity-existence prerequisite (Track A)
A relationship candidate referencing an unconfirmed or ambiguous entity can't be persisted (no FK target). **Mitigation**: skip + warn/log by default; review-time extraction does **not** confirm entity candidates in the same pass, so relationships involving newly suggested entities appear only after the author promotes entities and reruns/rescans extraction. Direct `kb.extract_work` paths may persist relationships only after the direct KeyBlock insert has committed and the counterpart endpoint already exists. This avoids a hidden relationship promotion state machine in V1.76.

### 6.3 Extraction output schema change (Track A)
Extending `nexus.llm.extract` output is a change to the extraction capability contract. **Mitigation**: the `relationships` array is optional/additive (existing required `candidates` array unchanged); older extraction results without relationships are tolerated as an empty relationship list. The normative contract lives in `llm-extract.md` and the built-in capability output schema, so both must be updated together.

### 6.4 GET graph default-filter change (Track A)
Changing GET graph to default-exclude `needs_review` relationships is a behavior change (existing callers see fewer relationships). **Mitigation**: all existing relationships have `needs_review=0` (migration default) → no visible change for existing data; only newly-extracted suggestions are filtered. The `include_suggested` param opts in.

### 6.5 Confidence-weighting subjectivity (Track A)
The weighting curve is a UX judgment. **Mitigation**: product-manager locked **stepped confidence bands** in Phase 2b (low `<0.4` = 1px/30%, mid `0.4–<0.7` = 2px/60%, high `≥0.7` = 3px/100%) instead of continuous linear weighting. The threshold slider remains tunable in V1.77, but V1.76 implementation must use the locked bands.

### 6.6 Relationship prompt and taxonomy quality (Track A)
The entity extraction prompt currently asks only for entities and the built-in schema only frames `candidates`. **Mitigation**: update prompt framing, parser normalization, and tests together so `relationships` use the existing `WorldKbRelationshipKind` taxonomy; unknown/unsupported relation types should either map to `custom` with `custom_label` or be skipped with a warning rather than inventing parallel enum values.

### 6.7 Source grounding mismatch (Track A)
Current graph source-anchor projections are `sa_<key_block_id>` and validation expects anchors to point at existing KeyBlocks with provenance; extraction only has a verbatim `source_quote`, not a chapter span/source-anchor row. **Mitigation**: keep `source_quote` in the extraction relationship candidate and/or row metadata for audit; populate `source_anchor_ids` only when existing endpoint anchors can be resolved. Do not claim a new chapter-span anchor model unless a separate migration/spec update adds it.

## 7. Success criteria (DoD)

1. **Extraction proposes relationships**: An author who runs `nexus.llm.extract`-backed extraction on a chapter sees relationship candidates appear in the Suggested pane once both endpoint entities resolve to existing non-deleted KeyBlocks — entity pairs (e.g., "Character A → allied_with → Character B") with an LLM-assigned `confidence` (0.0–1.0) and verbatim `source_quote` provenance. Candidates land in `kb_relationships` with `needs_review=1` + `source='extraction'`, idempotent across rescans (same stored entity pair + relation_type + custom_label + source='extraction' → no duplicate).
2. **`needs_review` gate prevents co-occurrence flooding**: Confirmed relationships (needs_review=0, source='manual') render on the graph as solid edges. Extraction candidates (needs_review=1) are hidden by default — the author sees only a "Suggested (N)" badge count in the canvas toolbar. Toggling the Suggested pane reveals the full triage list; `?include_suggested=true` surfaces them in the graph as dashed edges (visually distinct from confirmed).
3. **Author-curated promotion**: From the Suggested pane (sortable by confidence, default highest first), the author promotes a relationship (clears `needs_review` → the dashed edge becomes solid/confirmed on the graph) or deletes it. A bulk "Promote all" action clears `needs_review` for all visible suggestions (confirmation dialog for N > 1). Promotion and deletion reuse the existing `world_kb.patch_relationship` route extended with `needs_review` update support; `source` remains read-only provenance. The Suggested pane counter updates immediately.
4. **Confidence-weighting gives extraction data visual meaning**: Graph edges render with stroke width + opacity in three stepped bands (low <0.4: 1px/30%, mid 0.4–<0.7: 2px/60%, high ≥0.7: 3px/100%), consuming the shipped DESIGN.md confidence tokens (`canvas-worldkb-relationship-confidence-low/mid/high`). A threshold filter slider in the canvas toolbar (default 0.0 — show all, range 0.0–1.0 step 0.05, label "Confidence ≥ 0.XX") hides confirmed edges below threshold; manual/author-created edges without confidence always show. Confidence-band colored badges (red high-contrast dot for low, amber for mid, green for high) appear on edge labels at uniform 8px size — the color band IS the proportion signal, consistent with the existing `badge-status-pill` visual language. Suggested edges render as dashed with the same confidence weighting, always visible regardless of threshold.
