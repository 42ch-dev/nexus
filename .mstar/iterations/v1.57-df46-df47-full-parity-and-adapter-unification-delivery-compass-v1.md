### 9.2 Adapter topology (Option A — Q3)

```
                  ┌──────────────────────────────┐
                  │   capability::Registry        │
                  │   (SSOT: dispatch, admission,  │
                  │    handler, failure mode,     │
                  │    test vector)               │
                  └──────────┬───────────────────┘
                             │ registry::dispatch(tool_id, input)
         ┌───────────────────┼───────────────────┐
         │                   │                   │
  ┌──────▼──────┐   ┌───────▼───────┐   ┌───────▼──────┐
  │ CLI entry   │   │ Worker entry  │   │ HTTP entry   │
  │ host-call   │   │ agent_tool_   │   │ ToolExecute   │
  │ subcommand  │   │ request IPC   │   │ Request       │
  └─────────────┘   └───────────────┘   └──────────────┘
```

- **3 caller entry points** in `host_tool_executor.rs` (≈800 lines total post-refactor).
- Each entry normalizes its caller-specific wire format into the internal registry shape, then calls `capability::Registry::dispatch(tool_id, input)`.
- Wire codecs stay in the per-caller entry; admission, handler, failure mode, test vector live in `capability/builtins/` only.
- The 7 currently-duplicated `execute_X` functions are removed; their handlers live in `capability/builtins/`.

### 9.3 CLI surface (Q4)

- One new low-level debugging entry:
  ```
  nexus42 host-call <tool_id> --args <json>
  ```
- Walks the daemon host_tool path (CLI → worker `agent_tool_request` → `capability::Registry::dispatch`).
- **No** per-`nexus.*` CLI subcommands added (would collide with existing `creator *`).
