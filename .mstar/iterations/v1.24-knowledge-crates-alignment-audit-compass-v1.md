## 0. Position

This compass captures the **second full audit** after V1.21 (crate split) and V1.23 (entity scope + wiring reference). It is the iteration-scoped SSOT for:

1. What is **wrong or stale** in normative knowledge docs.
2. What is **implemented at compile time** but **not yet product-wired** (handlers, CLI, persistence).
3. What **design risks** remain even when Cargo edges look correct.

**Normative long-term rules** stay in `knowledge/specs/` — especially [entity-scope-model.md](../knowledge/specs/entity-scope-model.md) and [local-cloud-crate-architecture.md](../knowledge/specs/local-cloud-crate-architecture.md). This compass **does not** replace them; it drives a follow-up spec-refresh program.

When this compass conflicts with an active implementation plan, the active plan wins for delivery. When it conflicts with outdated wording inside normative specs, **fix the normative spec** (or archive superseded sections) in the remediation wave below.

## 4. Design issues (normative)

### 4.1 Dual “KB” semantics without route qualification (KCA-003)

**Problem:** Daemon exposes `/v1/local/kb/entries` and CLI exposes `creator kb`, but both implement the **CLI local work KB index** (`~/.nexus42/.../kb/index.json` + markdown files). [entity-scope-model.md §5.3](../knowledge/specs/entity-scope-model.md#53-cli-creator-kb--local-work-scope-file-index) states they are not `nexus-kb`, yet the HTTP path uses `kb` without scope prefix.

**Risk:** Contributors wire World KeyBlocks into daemon `handlers/kb.rs` thinking it is `nexus-kb`.

**Locked direction (C2):** Keep `/v1/local/kb/*` for this iteration; extract and label implementation as **`scope=work`** only (file index under active creator/workspace). Add spec/handler banners; enforce work as the only supported scope until a future KB route redesign. World KB (`nexus-kb`) stays out of these endpoints.

### 4.2 Missing Local API: context assemble (KCA-002)

**Problem:** [local-runtime-boundary.md](../knowledge/specs/local-runtime-boundary.md) and `nexus42` `DaemonClient` expect `POST /v1/local/context/assemble`. Router in `api/mod.rs` has **no** such route. CLI platform context may use in-process `nexus-moment-context-assembly` instead of daemon proxy — behavior is **split-brain**.

**Locked direction (B2):** Retire `POST /v1/local/context/assemble` from the Local API contract; update CLI to use in-process `nexus-moment-context-assembly` only (no daemon proxy). Document in cli-spec + boundary spec.

### 4.3 Compile-time-only domain linkage (KCA-005)

**Problem:** `architecture_assertions.rs` only references types from narrative/kb/knowledge to satisfy Cargo dep policy. No daemon handler constructs `InMemoryNarrativeGateway` or calls `assemble_moment`.

**Recommended direction:** Treat V1.24 **product wiring** separately from V1.23 **dependency wiring** — see §5 Theme C.

### 4.4 Persistence ownership split (KCA-004)

**Problem:** `nexus-local-db` owns `reference_sources` SQLite table; `nexus-knowledge` documents SQLite-backed storage as future; daemon `GET /v1/local/references` lists via local-db, not `nexus-knowledge`.

**Recommended direction:** Spec clarify: until migration, **references list** is local-db + daemon handler; **user knowledge entries** are in-memory prototype in `nexus-knowledge` only used by moment assembly tests.
