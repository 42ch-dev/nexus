## 3. Plan topology

V1.55 uses **7 registered plans** across two implement tracks plus a mid-QC meta-plan. The mid-QC rhythm is intentionally heavy: each implement plan lands on integration before the next dependent plan starts.

| Slot | Plan ID | Track | Effort | Owner | Review | Dependency / checkpoint |
| --- | --- | --- | --- | --- | --- | --- |
| P-1 | `2026-06-22-v1.55-compass-and-plan-stubs` | prepare | M | `@project-manager` (PM whitelist; authored by `@architect`) | QC skipped | — |
| P0 | `2026-06-22-v1.55-df43-sqlite-alignment` | Theme B / Track A | M | `@fullstack-dev` (Track A canonical owner) | Tri-review + mid-QA | P-1 PM signoff |
| P1 | `2026-06-22-v1.55-df31-workspace-interface` | Theme B / Track A | M | `@fullstack-dev` (Track A canonical owner) | Tri-review + mid-QA | P0 merged to `iteration/v1.55` |
| P2 | `2026-06-22-v1.55-game-bible-depth-35` | Theme D / Track B | L | `@fullstack-dev-2` (Track B canonical owner) | Tri-review + mid-QA | P-1 PM signoff; watch P0 spec drift |
| P3 | `2026-06-22-v1.55-script-scaffold` | Theme D / Track B | M | `@fullstack-dev-2` (Track B canonical owner) | Tri-review + mid-QA | P2 merged to `iteration/v1.55`; may consume P1 skeleton if needed |
| P-mid | `2026-06-22-v1.55-mid-qc-and-fix-waves` | meta | M | `@project-manager` | Meta tracking only | Captures P0/P1/P2/P3 mid-review rhythm |
| P-last | `2026-06-22-v1.55-hygiene-and-closeout` | closeout | M | `@project-manager` | Report-only QA | P0–P3 merged + mid reviews complete |

**Track-ownership pattern** (PM-locked during P-1 final review, 2026-06-22): each track has one canonical owner across both waves to ensure Wave-1 parallel dispatch lands on distinct `fullstack-dev` and `fullstack-dev-2` slots. Track A = `@fullstack-dev` (P0, P1). Track B = `@fullstack-dev-2` (P2, P3). Wave 1 dispatch therefore lands P0 on `fullstack-dev` and P2 on `fullstack-dev-2` in parallel; Wave 2 dispatch lands P1 on `fullstack-dev` and P3 on `fullstack-dev-2` in parallel. Each active writer still requires a separate git worktree per `mstar-branch-worktree`.

**Mid-QC checkpoints (parallel waves; each plan still individually merges + gets its own mid-QC)**:

The dual-track topology allows Track A and Track B to run in parallel. Within each track, plans are sequential (Track A: P0 → P1; Track B: P2 → P3). Cross-track parallel means the mid-QC rhythm is **two parallel waves**, not four sequential reviews:

- **Wave 1 (parallel mid-QC)**: P0 (Track A) and P2 (Track B) implement in parallel on separate worktrees; each plan merges to `iteration/v1.55` independently; each receives its own tri-review + mid-QA at its own integrated checkpoint; each may have its own targeted fix-wave. Wave 1 ends when BOTH P0 and P2 are accepted (or PM-recorded override).
- **Wave 2 (parallel mid-QC)**: P1 (Track A, requires P0 acceptance) and P3 (Track B, requires P2 acceptance) implement in parallel on separate worktrees; each plan merges to `iteration/v1.55` independently; each receives its own tri-review + mid-QA at its own integrated checkpoint; each may have its own targeted fix-wave. Wave 2 ends when BOTH P1 and P3 are accepted (or PM-recorded override).
- **P-last**: runs after Wave 2 completes (all four implement plans integrated and reviewed). Report-only QA, not tri-review.

PM dispatches mid-QC waves in lockstep — when both plans in a wave are merged, PM sends mid-QC for both within the same dispatch round. Cross-wave: Wave 2 cannot start until both Wave 1 plans have their fix-waves closed (or PM override).

## 4. Branch policy

V1.55 uses the project-standard **two-tier branch model** from [`.mstar/AGENTS.md`](../AGENTS.md): an iteration integration branch plus per-plan topic branches. Topic branches are created **from `iteration/v1.55`, not from `main`**.

| Tier | Branch | Purpose |
| --- | --- | --- |
| Iteration integration | `iteration/v1.55` | Single integration line and QC/QA review branch; final PR target is `main`. |
| P-1 prepare | `iteration/v1.55` | Compass + plan stubs + status/tracker/spec-status headers only. |
| P0 topic | `feature/v1.55-df43-sqlite-alignment` | DF-43 SQLite/crate alignment; from `iteration/v1.55`; merges back to integration. |
| P1 topic | `feature/v1.55-df31-workspace-interface` | DF-31 workspace interface skeleton; from current integration after P0. |
| P2 topic | `feature/v1.55-game-bible-depth-35` | Game-bible Depth 3.5; from `iteration/v1.55`; merges back to integration. |
| P3 topic | `feature/v1.55-script-scaffold` | Script scaffold; from current integration after P2 and after any required P1 interface seam. |
| P-last closeout | `feature/v1.55-hygiene-and-closeout` | Spec promotions, Profile B, tracker, residual closure; merges to integration. |

**Dispatch / merge policy**: Track A and Track B may progress in parallel only when each active writer has a separate worktree. Every implement plan must be merged to `iteration/v1.55` before its formal tri-review and mid-QA. QC/QA assignments use `Working branch: iteration/v1.55` at integrated `HEAD` for the relevant checkpoint.
