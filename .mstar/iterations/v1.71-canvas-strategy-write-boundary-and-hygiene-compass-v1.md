## 1. Scope / non-goals / out-of-scope

### 1.1 In scope

**Track A ‚Äî Canvas Strategy Œ≤ (Full depth, the lead)**

Implement the Draft ¬ß3.5 structured write boundary from [canvas-strategy-surface.md ¬ß3.5](../knowledge/specs/canvas-strategy-surface.md) in full. This promotes the Strategy canvas surface from Œ±-read to Œ≤-write, with conflict detection and UI retry/merge.

- **A1 ‚Äî Schemas + codegen (the breaking wire-contract work)**: New Local API DTOs in `schemas/local-api/canvas/strategy/`:
  - `StrategyPatchStateRequest` / `StrategyPatchTransitionRequest` / `StrategyPatchPromptTemplateRequest` (each carries `strategy_id`, target ids, `base_revision`, and the patch payload).
  - `StrategyPatchResponse` (echoes `new_revision`, `validation_summary`, optional `side_effects`).
  - `StrategyConflictError` (extends existing error envelope: `current_revision`, `node_id`, `conflicting_path`, recovery hint).
  - `StrategyValidationError` (structured validation summary, mirrors Draft ¬ß3.4 `CanvasValidationSummary`).
  - All generated to TS (`apps/web/src/api-types/canvas/`) and Rust (`crates/nexus-contracts/src/generated/canvas/strategy/`). `pnpm run codegen` regenerates downstream; CI verifies the generated artifacts are committed (`verify-codegen` job).
- **A2 ‚Äî graphRevision tracking + persistence (LOCKED)**: store the Strategy graph revision as a `revision:` key in the preset YAML header. This keeps preset YAML as the canonical SoT, preserves V1.70's `GetPresetResponse.yaml` read path, and avoids a V1.71 DB migration whose only durable value would be high-frequency edit analytics that the local single-author product does not yet need. On every accepted patch, the daemon reads the current YAML revision, verifies `base_revision`, applies the structured change, increments `revision` exactly once, and commits the YAML through the established temp-write + rename + fsync durability path. Read endpoints (`GET /v1/local/presets/{id}` and any future Strategy graph projection) return/expose the current `revision`. Write endpoints reject stale `base_revision` with `StrategyConflictError`. **Migration note**: existing preset YAML without `revision:` is treated as revision `0`; the first successful patch writes `revision: 1`. A future `strategy_revisions` DB table is deferred until multi-device sync, audit history, or high-frequency collaborative edits require it; if introduced later, it must backfill from the YAML header rather than becoming a second source of truth.
- **A3 ‚Äî Local API routes (new)**: New routes following V1.64 cursor + ErrorResponse conventions:
  - `POST /v1/local/strategies/{strategy_id}/states/{state_id}/patch` with `StrategyPatchStateRequest` ‚Üí `StrategyPatchResponse`
  - `POST /v1/local/strategies/{strategy_id}/transitions/patch` with `StrategyPatchTransitionRequest` ‚Üí `StrategyPatchResponse`
  - `POST /v1/local/strategies/{strategy_id}/states/{state_id}/prompt/patch` with `StrategyPatchPromptTemplateRequest` ‚Üí `StrategyPatchResponse`
  - All three requests carry `strategy_id`, `base_revision`, target identifiers, and a minimal patch payload; path identifiers are authoritative and must match any repeated body identifiers if present.
  - Error envelope: `StrategyConflictError` (409) + `StrategyValidationError` (422) + standard envelope (400/500).
- **A4 ‚Äî Daemon validation + atomic persistence** (`crates/nexus-daemon-runtime/src/handlers/strategy.rs`):
  - Parse + validate against preset semantics before accepting any patch:
    1. **ID existence**: `strategy_id` resolves to one preset bundle; `state_id`, source/target state ids, transition handles, and prompt/template ids exist in that bundle; path ids match body ids when duplicated.
    2. **Reachability / graph integrity**: the patch must not orphan the start state, create unreachable required states, remove terminal reachability, break inner-graph `depends_on` references, or create invalid Converge incoming/outgoing structure.
    3. **Condition syntax**: branch/default/edge conditions parse against the preset condition grammar; unknown variables/capability refs produce structured validation paths.
    4. **Prompt template reference resolution**: prompt refs resolve to an existing template/node-scoped prompt; TipTap/Markdown patches round-trip to the allowed prompt subset and cannot escape the node/template scope.
    5. **Revision precondition**: `base_revision` equals the current YAML `revision:` before the daemon writes; stale values short-circuit as 409 conflicts, not 422 validation failures.
  - On success: atomic YAML write (temp + rename + fsync, established durability pattern) with the incremented YAML `revision:` in the same committed file image.
  - On conflict: short-circuit before write; return `StrategyConflictError` with current canonical state.
- **A5 ‚Äî TS client methods (`NexusClient` in `apps/web/src/api/`)**: `strategyPatchState`, `strategyPatchTransition`, `strategyPatchPromptTemplate` methods. Refetch canonical preset after successful patch; expose `lastKnownRevision` for client-side conflict anticipation.
- **A6 ‚Äî Canvas UI write affordances** (`apps/web/src/features/canvas/strategy/`):
  - **State inspector**: edit label/description ‚Üí submit patch; show validation errors inline; show conflict modal on 409.
  - **Edge inspector**: edit condition/target ‚Üí submit patch; same conflict UX.
  - **Prompt template node**: in-node TipTap editor (V1.65 editor survives as in-node capability per Draft ¬ß2 invariant) ‚Üí submit patch.
  - **Conflict modal UX** (Draft ¬ß3.5 conflict policy #2, #3): on 409 `StrategyConflictError`, keep the user's draft patch in memory, refetch the canonical Strategy graph, and present a modal with:
  - **Headline**: "This node changed while you were editing."
  - **Body**: "Nexus updated **{node label}** to revision **{current_revision}** while you were editing **{field}**. Your change is still in the inspector."
  - **What changed**: a field-level summary of the canonical value that now differs from the user's last known revision.
  - **What you were about to do**: a field-level summary of the user's draft value for the same path.
  - **Actions**:
    - **Use current** (primary/default): keep the canonical server state and close the modal. The draft remains in the inspector so the author can reapply it manually.
    - **Reapply my edit**: resubmit the unchanged draft patch with `base_revision = {current_revision}`.
    - **Review side-by-side**: enabled only when the draft and canonical changes touch non-overlapping fields (e.g., label vs. condition); opens a field-level diff view where the author picks which fields to keep before resubmitting. Disabled when both changes touch the same field/path or prompt-template content.
  - **Cancel/close**: returns focus to the originating inspector with the draft intact.
  - **graphRevision tracking**: client tracks `lastKnownRevision` per strategy; surfaced in command palette ("Strategy last refreshed 3 min ago ‚Äî refresh now").
- **A7 ‚Äî Tauri WKWebView parity smoke**: A6 P0 Œ± parity smoke was for read+overlay+steer; V1.71 Œ≤ adds write affordances. Same browser + Tauri parity check: inspector modal, edge drag-rewire keyboard alternative, prompt in-node TipTap save.
- **A8 ‚Äî Accessibility (conflict UX)**: the conflict modal announces the conflict and the current-vs-draft difference via an ARIA live region; focus moves into the modal and is trapped until the author selects an action; closing returns focus to the originating node or inspector control; provide keyboard-accessible paths for **Use current**, **Reapply my edit**, **Review side-by-side**, and **Cancel**; honor `prefers-reduced-motion` for modal entrance and any graph-edge animation.
- **A9 ‚Äî Tests**: unit (validation, revision increment, conflict detection), integration (3 routes via NexusClient mock + canonical daemon), e2e (canvas edit ‚Üí daemon save ‚Üí refetch), regression (`R-V152TA-S006`-style promise gap test: ensure "if conflict happens, client sees current canonical state, never silent overwrite"). All in `crates/nexus-daemon-runtime/tests/strategy_patch.rs` + `apps/web/src/features/canvas/strategy/__tests__/`.

### 1.3 Wire contracts note

**`wire_contracts_changed: TRUE` (Track A only; @architect countersigned in Phase 2b).** Track A introduces new Local API DTOs and 3 new routes, which constitutes a breaking change to `@42ch/nexus-contracts` 0.6.0 ‚Üí **0.7.0** (pre-1.0 breaking bump per repo SemVer policy: "Pre-release Development, breaking changes are expected"). Generated artifacts in both `apps/web/src/api-types/` (TS) and `crates/nexus-contracts/src/generated/canvas/strategy/` (Rust) must be regenerated via `pnpm run codegen` and committed; CI `verify-codegen` job gates merge. Track B has `wire_contracts_changed: FALSE` ‚Äî only adds a CLI flag (`--sign-identity`) and workflow conditional; no schema, codegen, or DTO change.

**Why breaking is justified**: V1.71 promotes paper DTOs (Draft ¬ß3.5 "illustrative until a future wire-contract plan promotes them") to schema-backed contracts. The 3 patch operations are entirely new; consumers (TS web, Rust CLI) must regenerate. No existing field renames or removals ‚Äî the bump is additive at the JSON-field level, but the version bump remains **0.7.0** because (a) the DTO schemas themselves are first-class contracts that downstream `nexus-platform` consumes via npm semver lock; (b) adding 3 routes is a new surface; (c) pre-1.0 policy permits breaking without deprecation period. **Fallback decision**: if the downstream consumer explicitly rejects a 0.7.0 bump during PR coordination, the only acceptable fallback is an additive **0.6.1** release with no field renames/removals, no required-field changes to existing DTOs, and PM + maintainer sign-off recorded in the PR. The default remains 0.7.0. The bump is communicated in the V1.71 PR description and `nexus-platform` is notified via the `metadata.latest_ship.wire_contracts_changed: true` flag.

## 6. Risk notes

### 6.1 Breaking wire-contract risk (Track A)

V1.71 introduces new Local API DTOs and 3 new routes. Pre-1.0 breaking is permitted but the bump from `@42ch/nexus-contracts` 0.6.0 ‚Üí 0.7.0 affects downstream `nexus-platform` (npm semver lock). Phase 2b locks the default to **0.7.0**, with additive **0.6.1** available only if consumer coordination explicitly rejects the bump and no existing DTO field is renamed, removed, or made newly required.

Concrete mitigation / test scenarios:
- Run `pnpm run codegen`, then `git diff --exit-code` after a second run to prove deterministic generated artifacts (no timestamp/whitespace drift).
- Verify the CI `verify-codegen` job fails if generated TS/Rust outputs for the 3 Strategy DTOs are omitted.
- Add a consumer-shape smoke that imports `StrategyPatchStateRequest`, `StrategyPatchTransitionRequest`, `StrategyPatchPromptTemplateRequest`, `StrategyPatchResponse`, `StrategyConflictError`, and `StrategyValidationError` from `@42ch/nexus-contracts` and asserts stable export names.
- PR checklist must document `wire_contracts_changed: TRUE`, the 0.7.0 bump, and downstream notification; if the 0.6.1 fallback is used, record the maintainer approval and the additive-only proof.

### 6.2 graphRevision storage decision (Track A)

Phase 2b locks **option (a): `revision:` key in the preset YAML header**. The YAML bundle remains the single source of truth for Strategy/preset structure, so colocating `revision` with the patched graph avoids DB/YAML split-brain and keeps V1.70 `GetPresetResponse.yaml` projection unchanged. Option (b), a `strategy_revisions` DB table, is deferred until audit history, multi-device sync, or collaborative high-frequency edits justify a migration.

Concrete mitigation / test scenarios:
- Existing YAML without `revision:` loads as revision `0`; first successful patch persists `revision: 1` and returns `new_revision: 1`.
- Two sequential accepted patches monotonically increment `revision` without skipping or double-incrementing.
- A failed validation patch does **not** increment `revision`.
- Crash/failure during temp-write/rename does not leave a response claiming a revision that is absent from canonical YAML.
- A future DB-table migration must be introduced by a separate plan and backfill from YAML `revision:`; it must not write a second authoritative revision in V1.71.

### 6.3 UI retry/merge UX design risk (Track A)

Product guidance for the conflict modal: (a) **Default action** is **Use current** ‚Äî it preserves the canonical server state and cannot silently overwrite newer daemon/orchestration work; the author's draft stays in the inspector for explicit reapplication. (b) **Reapply my edit** resubmits the unchanged draft against `base_revision = current_revision`; use it when the author has reviewed the current state and still wants the same change. (c) **Review side-by-side** (the V1.71 merge path) is enabled only when the draft and canonical changes modify non-overlapping fields of the same node (e.g., label vs. condition) or entirely different nodes; it is disabled when both changes touch the same field/path, prompt-template content, or graph structure, because automatic merging would risk semantic corruption. (d) The modal must show both what changed and what the user was about to do before any action is taken. (e) The ARIA live region announces the conflict once on open, not on every background poll, to avoid screen-reader spam. A8 accessibility review (T3 in QC) must explicitly cover the conflict modal.

### 6.4 Concurrent daemon + canvas writes (Track A)

Draft ¬ß3.5 conflict policy describes the design; V1.71 must ensure graphRevision catches stale writes from BOTH the canvas and daemon/orchestration-side writers. Strategy patch routes write only structured preset/prompt-template state; raw `body_path` remains orchestration-owned and must never be overwritten by the browser. When daemon-side work mutates Strategy-adjacent prompt/template state between the canvas read and the patch submission, the patch must fail with 409 rather than silently applying over a newer graph.

Concrete mitigation / test scenarios:
- Canvas reads Strategy at revision `N`; daemon/orchestration mutates a prompt/template or Strategy YAML field and commits revision `N+1`; canvas submits with `base_revision: N` ‚Üí 409 `StrategyConflictError` with `current_revision: N+1`, target id, conflicting path, and recovery hint.
- Canvas and daemon submit non-overlapping Strategy edits against the same `base_revision`; first accepted write increments to `N+1`, second receives 409 and must refetch/reapply instead of last-write-wins.
- If orchestration is actively writing a body artifact (`body_path`) for a node, the canvas prompt/body-adjacent edit UI is read-only for that node and the patch route rejects any raw body mutation attempt as validation error.
- UI conflict modal keeps the user's draft patch, refetches canonical state, and exercises **Use current**, **Reapply my edit**, and **Review side-by-side** paths without losing keyboard focus or ARIA announcement context.

### 6.5 File overlap risk between Track A and Track B (P0 ‚à• P1)

Both tracks touch `crates/nexus-daemon-runtime/`. Track A creates new files (`handlers/strategy.rs`, `strategy/persistence.rs`, `strategy/validation.rs`). Track B edits existing files (`handlers/schedule.rs`, `handlers/sessions.rs`, `handlers/presets.rs`, `AGENTS.md`). The only real overlap is `AGENTS.md` ‚Äî Track A may reference error-envelope rule for strategy conflict errors (Draft ¬ß3.5 conflict error is structured), Track B V1.67P0-S-AGENTS writes the same rule for general error envelope. `@architect` Phase 2b locks the wording to avoid merge conflict. The `R-V167P0-QC1-S-AGENTS` rule wording is the source of truth; Track A references it rather than re-writing.

### 6.6 Sign infrastructure without cert risk (Track B)

B2 ships `SIGNING.md` + `--sign-identity` flag + CI guard, but no actual signing. Risk: doc/flag could be misused (e.g., user passes `--sign-identity=""` and expects signing). Hard constraint: flag is a no-op unless `APPLE_SIGN_IDENTITY` env var is non-empty AND the identity exists in the keychain. Documented in `SIGNING.md`. CI guard: workflow skips signing step entirely when secrets absent (build-time notice). This is infrastructure prep, not signing rollout.

### 6.7 Token-name preservation invariant continues (Track A8)

V1.69 + V1.70 established that token names in `apps/web/DESIGN.md` are preserved verbatim. V1.71 A8 *adds* canvas-write-mode tokens under the same invariant. If a token name needs changing, that is a defect to fix before merge.
