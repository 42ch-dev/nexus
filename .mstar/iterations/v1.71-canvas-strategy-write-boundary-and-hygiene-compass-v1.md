## 1. Scope / non-goals / out-of-scope

### 1.1 In scope

**Track A — Canvas Strategy β (Full depth, the lead)**

Implement the Draft §3.5 structured write boundary from [canvas-strategy-surface.md §3.5](../knowledge/specs/canvas-strategy-surface.md) in full. This promotes the Strategy canvas surface from α-read to β-write, with conflict detection and UI retry/merge.

- **A1 — Schemas + codegen (the breaking wire-contract work)**: New Local API DTOs in `schemas/local-api/canvas/strategy/`:
  - `StrategyPatchStateRequest` / `StrategyPatchTransitionRequest` / `StrategyPatchPromptTemplateRequest` (each carries `strategy_id`, target ids, `base_revision`, and the patch payload).
  - `StrategyPatchResponse` (echoes `new_revision`, `validation_summary`, optional `side_effects`).
  - `StrategyConflictError` (extends existing error envelope: `current_revision`, `node_id`, `conflicting_path`, recovery hint).
  - `StrategyValidationError` (structured validation summary, mirrors Draft §3.4 `CanvasValidationSummary`).
  - All generated to TS (`apps/web/src/api-types/canvas/`) and Rust (`crates/nexus-contracts/src/generated/canvas/strategy/`). `pnpm run codegen` regenerates downstream; CI verifies the generated artifacts are committed (`verify-codegen` job).
- **A2 — graphRevision tracking + persistence**: introduce a `revision` field on Strategy/preset (decision: separate DB column on a new `preset_revisions` table, or `revision:` key in preset YAML header — `@architect` Phase 2b confirms). On every patch, increment revision atomically. Read endpoints (`GET /v1/local/presets/{id}` and any new read) return current `revision`. Write endpoints reject stale `base_revision` with `StrategyConflictError`.
- **A3 — Local API routes (new)**: New routes following V1.64 cursor + ErrorResponse conventions:
  - `POST /v1/local/strategies/{strategy_id}/states/{state_id}/patch` → `StrategyPatchResponse`
  - `POST /v1/local/strategies/{strategy_id}/transitions/patch` → `StrategyPatchResponse`
  - `POST /v1/local/strategies/{strategy_id}/states/{state_id}/prompt/patch` → `StrategyPatchResponse`
  - Error envelope: `StrategyConflictError` (409) + `StrategyValidationError` (422) + standard envelope (400/500).
- **A4 — Daemon validation + atomic persistence** (`crates/nexus-daemon-runtime/src/handlers/strategy.rs`):
  - Parse + validate against preset semantics: ids exist, reachability intact, condition expression syntactically valid, prompt template references resolve.
  - On success: atomic YAML write (temp + rename + fsync, established durability pattern) + revision row update in same DB transaction.
  - On conflict: short-circuit before write; return `StrategyConflictError` with current canonical state.
- **A5 — TS client methods (`NexusClient` in `apps/web/src/api/`)**: `strategyPatchState`, `strategyPatchTransition`, `strategyPatchPromptTemplate` methods. Refetch canonical preset after successful patch; expose `lastKnownRevision` for client-side conflict anticipation.
- **A6 — Canvas UI write affordances** (`apps/web/src/features/canvas/strategy/`):
  - **State inspector**: edit label/description → submit patch; show validation errors inline; show conflict modal on 409.
  - **Edge inspector**: edit condition/target → submit patch; same conflict UX.
  - **Prompt template node**: in-node TipTap editor (V1.65 editor survives as in-node capability per Draft §2 invariant) → submit patch.
  - **Conflict modal UX** (Draft §3.5 conflict policy #2, #3): structured error → "Nexus updated this node since you started. View current → Reapply your edit / Merge / Discard." UX design in Phase 2b `@product-manager` (or fallback to PM-authored if subagent unavailable).
  - **graphRevision tracking**: client tracks `lastKnownRevision` per strategy; surfaced in command palette ("Strategy last refreshed 3 min ago — refresh now").
- **A7 — Tauri WKWebView parity smoke**: A6 P0 α parity smoke was for read+overlay+steer; V1.71 β adds write affordances. Same browser + Tauri parity check: inspector modal, edge drag-rewire keyboard alternative, prompt in-node TipTap save.
- **A8 — Accessibility (conflict UX)**: conflict modal must announce current vs draft state via ARIA live region; provide keyboard reapply/merge/discard paths; honor reduced motion for any transition animations.
- **A9 — Tests**: unit (validation, revision increment, conflict detection), integration (3 routes via NexusClient mock + canonical daemon), e2e (canvas edit → daemon save → refetch), regression (`R-V152TA-S006`-style promise gap test: ensure "if conflict happens, client sees current canonical state, never silent overwrite"). All in `crates/nexus-daemon-runtime/tests/strategy_patch.rs` + `apps/web/src/features/canvas/strategy/__tests__/`.

### 1.3 Wire contracts note

**`wire_contracts_changed: TRUE` (Track A only).** Track A introduces new Local API DTOs and 3 new routes, which constitutes a breaking change to `@42ch/nexus-contracts` 0.6.0 → **0.7.0** (pre-1.0 breaking bump per repo SemVer policy: "Pre-release Development, breaking changes are expected"). Generated artifacts in both `apps/web/src/api-types/` (TS) and `crates/nexus-contracts/src/generated/canvas/strategy/` (Rust) must be regenerated via `pnpm run codegen` and committed; CI `verify-codegen` job gates merge. Track B has `wire_contracts_changed: FALSE` — only adds a CLI flag (`--sign-identity`) and workflow conditional; no schema, codegen, or DTO change.

**Why breaking is justified**: V1.71 promotes paper DTOs (Draft §3.5 "illustrative until a future wire-contract plan promotes them") to schema-backed contracts. The 3 patch operations are entirely new; consumers (TS web, Rust CLI) must regenerate. No existing field renames or removals — the bump is purely additive at the schema level, but the version bump is still `0.7.0` because (a) the DTO schemas themselves are first-class contracts that downstream `nexus-platform` consumes via npm semver lock; (b) adding 3 routes is a new surface; (c) pre-1.0 policy permits breaking without deprecation period. The bump is communicated in the V1.71 PR description and `nexus-platform` is notified via the `metadata.latest_ship.wire_contracts_changed: true` flag.

## 6. Risk notes

### 6.1 Breaking wire-contract risk (Track A)

V1.71 introduces new Local API DTOs and 3 new routes. Pre-1.0 breaking is permitted but the bump from `@42ch/nexus-contracts` 0.6.0 → 0.7.0 affects downstream `nexus-platform` (npm semver lock). `@architect` Phase 2b must:
- Confirm generated artifacts are deterministic (no whitespace drift)
- Confirm `pnpm run codegen` is reproducible (CI gates via `verify-codegen`)
- Document the bump in PR description + `metadata.latest_ship.wire_contracts_changed: true` + notify `nexus-platform` team
- If pre-1.0 bump is rejected by the consumer, fallback: ship as additive 0.6.1 (semver non-breaking) — this requires DTOs to be strictly additive (no field renames/removals) and consumers to tolerate new optional fields. `@architect` Phase 2b decides.

### 6.2 graphRevision storage decision (Track A)

`@architect` Phase 2b decides between two storage options:
- **(a)** Add `revision` key to preset YAML header + atomic write on every patch. No schema change; fits preset YAML being the SoT.
- **(b)** Add `strategy_revisions` DB table + revision row update in same DB tx as the YAML write. More robust for high-frequency edits; requires DB migration.
V1.70 P0 client-side projection reuses `GetPresetResponse.yaml`, so option (a) keeps read paths unchanged. Option (b) is more reliable but adds migration risk. `@architect` confirms; default to (a) for simplicity.

### 6.3 UI retry/merge UX design risk (Track A)

The conflict modal UX is novel — Draft §3.5 conflict policy #2/#3 describe the rules but the visual design is open. `@product-manager` Phase 2b either authors the design (canvas UX is V1.70 Draft territory) or PM-authored fallback if subagent unavailable. Risks: copy could be confusing, button placement could be ambiguous, ARIA live region timing could cause screen-reader spam. A8 accessibility review (T3 in QC) must explicitly cover the conflict modal.

### 6.4 Concurrent daemon + canvas writes (Track A)

Draft §3.5 conflict policy describes the design; V1.71 must ensure graphRevision catches stale writes from BOTH the canvas AND the orchestration engine (which writes `body_path` via host-tool paths). Test scenario: simulation where orchestration writes `body_path` between canvas edit's `base_revision` read and patch submission — verify canvas patch returns 409 with structured error. `@architect` Phase 2b confirms test plan.

### 6.5 File overlap risk between Track A and Track B (P0 ∥ P1)

Both tracks touch `crates/nexus-daemon-runtime/`. Track A creates new files (`handlers/strategy.rs`, `strategy/persistence.rs`, `strategy/validation.rs`). Track B edits existing files (`handlers/schedule.rs`, `handlers/sessions.rs`, `handlers/presets.rs`, `AGENTS.md`). The only real overlap is `AGENTS.md` — Track A may reference error-envelope rule for strategy conflict errors (Draft §3.5 conflict error is structured), Track B V1.67P0-S-AGENTS writes the same rule for general error envelope. `@architect` Phase 2b locks the wording to avoid merge conflict. The `R-V167P0-QC1-S-AGENTS` rule wording is the source of truth; Track A references it rather than re-writing.

### 6.6 Sign infrastructure without cert risk (Track B)

B2 ships `SIGNING.md` + `--sign-identity` flag + CI guard, but no actual signing. Risk: doc/flag could be misused (e.g., user passes `--sign-identity=""` and expects signing). Hard constraint: flag is a no-op unless `APPLE_SIGN_IDENTITY` env var is non-empty AND the identity exists in the keychain. Documented in `SIGNING.md`. CI guard: workflow skips signing step entirely when secrets absent (build-time notice). This is infrastructure prep, not signing rollout.

### 6.7 Token-name preservation invariant continues (Track A8)

V1.69 + V1.70 established that token names in `apps/web/DESIGN.md` are preserved verbatim. V1.71 A8 *adds* canvas-write-mode tokens under the same invariant. If a token name needs changing, that is a defect to fix before merge.
