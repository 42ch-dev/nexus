## 1. North Star

Nexus OSS should make the split crates real product modules, not compile-only islands. The architecture should express:

1. A clear entity-scope hierarchy.
2. A clear distinction between user knowledge, creator identity/memory, world-scoped KB graph, narrative state, and moment context assembly.
3. A crate graph where product-critical domain crates are reachable from `nexus42` and/or the daemon runtime.
4. Specs that describe the actual implementation state and the target state separately when they differ.

---

## 2. Entity Scope Model

Future architecture should use these ownership scopes consistently:

| Scope | Meaning | Examples |
| --- | --- | --- |
| `Global` | Product-wide scope | schema versions, public registry defaults, product metadata |
| `User` | Human account scope | user knowledge index, user-level preferences, cloud account bridge |
| `Creator` | Agent / creator persona scope | Creator identity, SOUL, creator memory, active workspace |
| `World` | World / narrative universe scope | world KB graph, world state, forks, world-scoped references |
| `Timeline` | Timeline scope inside a world | timeline branches, timeline-local state |
| `Event` | Event scope inside a timeline | narrative events, source anchors tied to events |
| `Moment` | Session-start context point | pre-session assembled context snapshot |

### 2.1 Non-fungible uniqueness

For non-fungible unique entities, the same class/template has at most one instance under the same scope ID. Different classes/templates do not conflict with each other.

Example:

- One `DefaultWorldKBIndex` under `world_id = W1`.
- One `DefaultWorldKBIndex` under `world_id = W2`.
- `DefaultWorldKBIndex` and `NarrativeTimelineIndex` may both exist under `world_id = W1`.

## 3. Crate Responsibility Reference

### 3.1 `nexus-moment-context-assembly`

**Purpose**: session-start **Moment** context assembly.

It should run before a session begins and collect the slices needed at that moment:

- Creator memory and SOUL-derived context.
- Current narrative state from `nexus-narrative`.
- World-scoped KB graph entries from `nexus-kb`.
- User-scoped knowledge entries from `nexus-knowledge`.
- Optional platform-enhanced context only behind explicit cloud-stage behavior.

It should not become the owner of memory, narrative, KB graph, or knowledge persistence. It is an aggregator and formatter.

### 3.2 `nexus-narrative`

**Purpose**: creative-work narrative state gateway.

It should be the core entry point for understanding:

- What work is currently active.
- Which world, timeline, event, and moment the creator is operating in.
- The current world-state background and narrative constraints.
- How narrative context relates to world-scoped KB assets.

It should not own global user knowledge or cloud sync transport.

### 3.3 `nexus-knowledge`

**Purpose**: **User-scoped** global knowledge base.

It should own:

- Tag-driven user knowledge indexing.
- Knowledge storage / lookup mechanics.
- Efficient selection of user knowledge that may be pulled into moment context.

It is **not** Creator-scoped and is **not** the world/narrative KB graph.

### 3.4 `nexus-kb`

**Purpose**: World/narrative-scoped KB graph module.

It should own:

- Creator knowledge assets that belong under a narrative entity, especially `World`.
- KB graph insertion and query primitives.
- KeyBlock / SourceAnchor style domain logic where these are tied to narrative scope.

The KB scope must be under a narrative entity (`World` and below), not directly under Creator.

### 3.5 `nexus-cloud-domain`

**Purpose**: cloud bridge domain logic for User and Pairing.

`nexus-cloud-sync` should use this crate for User/Pairing invariants and conversions instead of duplicating or bypassing domain behavior in HTTP transport code.

### 3.6 `nexus-cloud-sync`

**Purpose**: cloud transport and sync flow.

It owns HTTP clients, sync/outbox transport, and platform communication. It should depend on `nexus-cloud-domain` for cloud bridge domain logic while keeping daemon-runtime isolated from platform HTTP.

## 4. Desired Product Graph

This is the future target graph. It is not a claim about the current implementation.

```text
nexus42
  ├── nexus-daemon-runtime
  │   ├── nexus-orchestration
  │   ├── nexus-agent-host
  │   ├── nexus-creator
  │   ├── nexus-creator-memory
  │   ├── nexus-narrative
  │   ├── nexus-kb
  │   ├── nexus-knowledge
  │   ├── nexus-moment-context-assembly (default features)
  │   └── nexus-local-db
  ├── nexus-cloud-sync
  │   └── nexus-cloud-domain
  └── nexus-moment-context-assembly (cloud-stage only for CLI/platform flows)

nexus-moment-context-assembly
  ├── nexus-creator-memory
  ├── nexus-narrative
  ├── nexus-kb
  ├── nexus-knowledge
  └── nexus-contracts
```

Hard boundary:

- `nexus-daemon-runtime` must not depend on `nexus-cloud-sync`.
- `nexus-daemon-runtime` must not expose platform sync or registration proxies.
- Cloud-stage context enhancement must remain explicit and feature-gated.

## 5. Future Iteration Themes

### Theme A — Spec and terminology alignment

Clarify in knowledge specs:

1. User-scoped `knowledge` vs world-scoped `kb`.
2. Creator memory vs Creator identity vs narrative state.
3. Moment context assembly as a session-start aggregator.
4. Actual implementation graph vs target graph.

Potential outputs:

- Scope/entity model spec or section.
- Updated `local-cloud-crate-architecture.md`.
- Updated `cli-spec.md` terminology for `creator kb`.
- Updated `docs/ARCHITECTURE.md`.

### Theme B — Domain interfaces

Introduce minimal APIs before full graph features:

1. `nexus-knowledge`: store/list/search user knowledge by tags and scope.
2. `nexus-kb`: insert/query world-scoped KB graph entries.
3. `nexus-narrative`: provide current narrative context for world/timeline/event/moment.
4. `nexus-moment-context-assembly`: aggregate read-only context slices from the above modules.

Avoid building a full graph query engine before minimal product paths exist.

### Theme C — Product wiring

Make split crates reachable:

1. Wire `nexus-cloud-sync -> nexus-cloud-domain`.
2. Wire `nexus-moment-context-assembly -> narrative/kb/knowledge`.
3. Wire daemon/CLI paths to domain services where product behavior needs them.
4. Move storage logic out of `nexus42` command handlers when a crate should own it.

### Theme D — Drift guards

Add checks that fail when architecture drifts:

1. Forbidden dependency checks (`daemon-runtime` must not pull cloud transport).
2. Required product dependency checks once the target graph is implemented.
3. CLI command tests for any renamed/split KB/knowledge commands.
4. Moment assembly integration tests with memory + narrative + KB + knowledge fixtures.

## 6. Open Decisions for a Future Iteration

These are intentionally not resolved by this reference compass.

| Decision | Recommended default | Why |
| --- | --- | --- |
| Should `creator kb` remain the command name? | Re-evaluate after scope spec | Current name overlaps with `nexus-kb` but current behavior is work-index-like |
| Should world-scoped KB commands live under `creator kb` or a narrative/world command group? | Prefer scope-explicit UX | World ownership should be visible to users and tests |
| Should `nexus-knowledge` persist in SQLite or file-backed index first? | Start with the existing local DB layer if feasible | Avoid another storage owner; keep CLI/daemon consistent |
| Should `nexus-narrative` be required by daemon-runtime immediately? | Only after a minimal read-model exists | Avoid wiring an empty dependency just to satisfy a graph diagram |
| Should moment context assembly expose a trait boundary? | Yes, if it prevents direct storage coupling | Keeps assembly as aggregation, not storage owner |
