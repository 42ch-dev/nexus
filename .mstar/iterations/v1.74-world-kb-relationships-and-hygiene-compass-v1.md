  - Validation is **pre-write** — all rules checked before any DB mutation:

### 2.2 `entity-scope-model.md` §5.6 amendment wording (lift-ready)

````md
### 5.6 World KB relationship semantics (V1.74 normative)

> **Status**: Normative — V1.74 Shipped β. This section defines first-class inter-entity relationships in the World KB graph. The V1.50–V1.73 model defined KeyBlocks, SourceAnchors, and the promotion state machine; V1.74 adds typed edges between confirmed/non-deleted World KB entities.

World KB relationships are World-scoped graph edges owned by the same World as their source and target KeyBlocks. A relationship row does not transfer ownership, change the KeyBlock lifecycle, or replace SourceAnchor provenance; it records author-visible semantic linkage between two World KB entities.

#### 5.6.1 Storage and identity

- A relationship is stored as one `kb_relationships` row with `relationship_id`, `world_id`, `source_entity_id`, `target_entity_id`, `relation_type`, optional `custom_label`, `symmetric`, optional `confidence`, optional `source_anchor_ids`, optional `metadata`, timestamps, and `revision`.
- `source_entity_id` and `target_entity_id` MUST refer to non-deleted `kb_key_blocks.key_block_id` rows in the same `world_id`.
- Self-loops are prohibited: `source_entity_id == target_entity_id` is invalid and MUST be rejected as a validation error.

#### 5.6.2 Directed + symmetric model

Relationships are stored as directed edges. The `symmetric` flag marks a directed row whose author-visible meaning should be projected in both directions. Implementations MUST NOT store a second reverse row for symmetry. On canonical graph reads, a symmetric row emits both the stored projection and a derived reverse projection that share the same `relationship_id`; editing or deleting either projection targets the single stored row.

#### 5.6.3 Taxonomy

The wire taxonomy is the `WorldKbRelationshipKind` enum. V1.74 core values are: `allied_with`, `opposes`, `parent_of`, `child_of`, `member_of`, `located_in`, `rules_over`, `references`, `serves`, `rival_of`, `mentor_of`, and `custom`. Wire values are snake_case. When `relation_type = custom`, `custom_label` is required and carries the author’s narrative label. When `relation_type` is any core enum value, `custom_label` is ignored or normalized to absent.

#### 5.6.4 Source-anchor grounding

Relationship grounding is optional. `source_anchor_ids` may be empty for author-asserted worldbuilding relationships that are true in the author’s model but not yet grounded in extracted text. Non-empty `source_anchor_ids` indicate grounding and MUST resolve to source-anchor projections in the same World. UI may display “author-asserted” for empty grounding and “N anchors” for grounded relationships.

#### 5.6.5 Confidence

`confidence` is optional and display-only in V1.74. It may be author-set or future-extraction-set, but it MUST NOT drive validation, graph layout weighting, filtering, or conflict resolution in this iteration. Values, when present, are in the inclusive range `0.0..=1.0`.
````

## 6. Risk notes

### 6.1 New `kb_relationships` migration risk (Track A)

V1.74 introduces the first new KB table since V1.50's `kb_key_blocks` / `kb_source_anchors` / `kb_extract_jobs`. **Mitigation**: the migration is additive (no change to existing tables); FK targets existing `kb_key_blocks.key_block_id`; per-row OCC mirrors the proven `kb_extract_jobs.version` CAS pattern. Migration tested via the existing migration runner + a fresh-DB integration test.

### 6.2 Symmetric read-projection correctness (Track A)

Symmetric relationships (`symmetric = 1`) emit two graph edges (forward + derived reverse) from one storage row. The read projection must keep them referring to the same `relationship_id` so editing either updates the single row (no double-write). **Mitigation**: unit test proving (a) symmetric relationship renders 2 edges sharing 1 id, (b) updating via the reverse edge updates the single row, (c) deleting via either edge removes the row and both projections.

### 6.3 Hybrid taxonomy validation drift (Track A)

The core enum is fixed in schema/code; `custom` + `custom_label` is the escape hatch. Risk: authors overuse `custom` and the enum becomes meaningless, OR the enum is too narrow and forces custom for common cases. **Mitigation**: the architect-locked enum covers the common narrative cases (allied/opposes/parent/child/member/located/rules/references/serves/rival/mentor); enum usage vs custom-label usage can be observed post-ship and the enum extended in a later iteration.

### 6.4 Codegen coordination between tracks (Track A A1 ↔ Track B B3/B4)

B3 (naming reconciliation) + B4 (idempotency_key removal) edit V1.73 world-kb schema files that P0 A1 regenerates from. Uncoordinated regeneration causes merge conflicts on generated TS/Rust files. **Mitigation**: PM sequences the codegen+commit so B3/B4 schema edits land and regenerate *before* or *atomically with* P0 A1's new relationship schemas in a single codegen pass on the integration branch (not in parallel worktrees). The worktree split is for source files; the integration-branch codegen+commit is serialized by PM.

### 6.5 world-kb-canvas.tsx split + relationship UI sequencing (Track A A10 ↔ A6)

A10 (split) must precede A6 (relationship UI) within P0 — adding relationship UI to the verified current 561-line `apps/web/src/components/canvas/world-kb/world-kb-canvas.tsx` monolith defeats the split purpose. **Mitigation**: P0 is single-stream (`single_stream_justified: yes`); the split is the first UI task, relationship UI builds on the split modules. This is intra-plan sequencing, not a cross-track dependency.

### 6.6 Slate-clear ambition (Track B)

Closing 11 residuals in one track is more than V1.73's Track B (7). Risk: Track B expands beyond M. **Mitigation**: all 11 are low/nit; 6 are polish (docs/test/copy/cosmetic). The 2 correctness items (B1/B2) + 2 reconciliation items (B3/B4) + 1 UX fix (B5) are the substance. If Track B slips, polish items (B6/B7/B9/B10/B11) can re-defer to V1.75 with durable pointers without losing the slate-clear's correctness value — but the default plan closes all 11.

### 6.7 Concurrent daemon + canvas relationship writes (Track A)

Relationships have two writer classes: canvas (remote, via Local API) and orchestration (in-process). Per-row OCC must catch stale writes from both (mirrors V1.73 §6.6). **Mitigation**: regression test — canvas reads relationship at version N; orchestration mutates to N+1; canvas submits with expected_version N → 409.

### 6.8 Migration rollback / downgrade expectation (Track A)

The `kb_relationships` migration is additive, but SQLite migrations in this repo are forward-only; rollback means restoring the DB backup or creating a forward cleanup migration, not editing historical migrations. **Mitigation**: fresh-DB migration test + existing-DB migration test with pre-existing `kb_key_blocks`; document that deleting V1.74 relationship rows is a data-destructive forward fix and requires maintainer approval.

### 6.9 Symmetric projection performance at large graph scale (Track A)

Symmetric read projection can double the number of relationship edges returned to React Flow. For small V1.74 Worlds this is acceptable, but large Worlds can hit graph-rendering pressure faster than entity counts alone suggest. **Mitigation**: add `(world_id, source_entity_id, target_entity_id)` index, keep graph entity/edge caps consistent with the existing `GRAPH_ENTITY_CAP`, and add a test for deterministic projection ordering so future pagination/filtering can be introduced without changing semantics.
