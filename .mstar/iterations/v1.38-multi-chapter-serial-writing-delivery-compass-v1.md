## 0. Position

V1.37 made the V1.36 novel-writing foundation reliable: `novel-project-init` input reaches `preset.input`, runtime gates fail fast, scaffold writes are atomic, and the spec now contains a roadmap for DF-62 multi-chapter behavior. V1.38 selects the next narrow axis:

> **Make serial writing actually continue from chapter 2 through N using DB-selected chapter state, without introducing automatic continuation or multi-work automation.**

V1.38 is not a broad novel platform iteration. It prepares two implementation plans that are small enough to verify:

- **P0** establishes chapter selection and status truth: choose the next chapter from `work_chapters`, update `current_chapter` only on finalize, render useful status/next-action UX, and stop at completion.
- **P1** makes `novel-writing` consume the selected chapter as prompt/preset input, replacing chapter-1 assumptions while keeping a single reusable preset for chapter N.

When this compass conflicts with a later locked implementation plan, the active plan wins for delivery batching only. Normative behavior changes still require spec updates before implementation.

## 2. Target User Journey

```text
creator run status <work_id>
  -> progress: 1 / 3 chapters finalized
  -> current_chapter: 1
  -> next action: Chapter 2 is not started; run creator run continue <work_id>

creator run continue <work_id>
  -> gate checks pass
  -> next_chapter(work_id) selects chapter 2 from work_chapters
  -> novel-writing receives selected chapter input
  -> outline/draft/finalize operate on ch02 paths
  -> finalize GO updates chapter 2 row + frontmatter, then current_chapter = 2

creator run status <work_id>
  -> progress: 2 / 3 chapters finalized
  -> next action: Chapter 3 is not started; run creator run continue <work_id>
```

The user remains in control of continuation. V1.38 may make the next action obvious; it must not silently reenqueue chapter 3.
