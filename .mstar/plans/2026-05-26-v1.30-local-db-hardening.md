## 1. Problem Statement

Five residuals deferred from V1.26 reference/narrative/KB work target post-V1.29 hardening:

| ID | Issue |
| --- | --- |
| R5 | `body.md` written before SQL INSERT — orphan files on DB failure |
| R6 | `list_references` unbounded |
| R7 | `content_hash` always NULL |
| R8 | `db_err` maps all sqlx errors to `ValidationError` — no Storage variant |
| R9 | `KbQuery` loads all blocks then filters in Rust — no DB LIMIT |

Registered as fourth V1.30 plan; **parallel** with core serial chain is allowed.

---

## 2. Goals

### L1 — Reference write order (R5)

1. Write `body.md` after successful SQL INSERT, or wrap file + SQL in a single transactional discipline (document chosen approach).
2. Test: simulated DB failure does not leave orphan body file (or cleanup on rollback).

### L2 — Reference pagination (R6)

1. Add `LIMIT` / `OFFSET` (or cursor) to `list_references` with sensible default.

### L3 — Content hash (R7)

1. Populate `content_hash` on insert/update (static hash of body bytes).
2. **Not** DF-44 full refresh pipeline.

### L4 — Narrative error variant (R8)

1. Introduce `NarrativeError::Storage` (or `Database`) for sqlx failures instead of overloading `ValidationError`.

### L5 — KB list pagination (R9)

1. Add SQL `LIMIT` to `list_by_world` / hot path used by `KbQuery` where all rows are loaded.

---

## 3. Non-goals

- DF-44 reference refresh pipeline.
- Schema migrations unless required for hash column usage (prefer filling existing column).
- Changes to `kb_extract_job` (sibling kb-extract plan).
