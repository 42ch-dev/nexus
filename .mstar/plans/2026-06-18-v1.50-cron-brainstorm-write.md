## 1. Problem Statement

`works.schedule_json` is now writable (T-A P0), but the daemon does not yet fire cron-triggered schedules for `novel-brainstorm` / `novel-write`. Authors can configure schedule but nothing happens.

---

## 2. Goals

1. Daemon tick evaluates per-Work cron; fires `novel-brainstorm` and `novel-write` schedules per role enabled + cron match.
2. Per-Work gating: skip when intake != complete / runtime_lock != NULL / completion_locked.
3. Idempotency: skip when prior same-role same-work schedule still active.
4. Cron fire enqueues a `Schedule` with `preset_id = <role-preset>` and `work_ref` from source Work.

---

## 3. Non-goals

- Review cron (T-A P2).
- Auto-chronology (T-A P3).

---

## 4. Acceptance Criteria

1. `cargo test -p nexus-orchestration --test cron_supervisor` covers: fire on cron match, skip on intake/lock/completion, idempotent re-fire skip.
2. `creator run status` after cron fire shows the auto-chain step advanced.
3. Test: synthetic tick advances `now` to a cron match minute; expect enqueue; advance to non-match → no enqueue.
4. Hermetic test: two cron fires within same minute for same role+work → second skipped.

### Per-finding summary

#### R-V150P1CRONBW-03 — `cron_fires_at_minute` per-tick re-parse (qc3 W-001)
- **Fix**: in-process `OnceLock<Mutex<HashMap<(work_id, role), (raw_cron_str, cron::Schedule)>>>` memoisation cache (`type CronScheduleCache`). The cached resolver `cron_fires_at_minute_for_work` looks up by `(work_id, role)`; on hit with matching raw string → reuse, on miss or content drift → re-parse + store (content-based invalidation). The resolved `Schedule` is cloned out of the lock before the µs-scale match (`schedule_fires_at_minute`) so the critical section stays tight. Cache hits are O(1) per Work per tick after first observation.
- **Pure matcher**: `cron_fires_at_minute` refactored to delegate to the extracted `schedule_fires_at_minute` helper (behavior-preserving); retained as `#[cfg(test)]` fixture for the existing matcher-semantics tests. The daemon hot path (`try_fire_role`) now calls `cron_fires_at_minute_for_work`.
- **Invalidation**: `pub fn invalidate_cron_schedule_cache()` exposed from `nexus-orchestration`; `nexus42::handle_set` calls it after a successful `set_schedule_json_tx` CAS write (daemon is read-only on `schedule_json`, so the CLI write site is the only invalidation hook). The cache also self-heals on raw-string drift, so explicit invalidation is memory hygiene, not correctness.
- **Regression test**: `cron_fires_at_minute_uses_memoised_schedule` — asserts via an instrumented `AtomicU64` parse counter that 100 calls for the same `(work, role, cron)` parse exactly once; content drift + new keys re-parse; non-match minute does not re-parse; explicit invalidation forces a re-parse.

#### R-V150P1CRONBW-04 — info!-level idempotency-skip too noisy (qc3 W-002)
- **Fix**: changed the `info!(... "prior schedule still active; skipping fire")` to `debug!(...)` with an inline rationale comment. One-line behavioural change; aligns with the existing disabled/gated skip paths (already `debug!`). The `sweep complete` summary line already aggregates the counter for default-level observability.
- **Regression test**: skipped per assignment (verifying log level is purely behavioural; qc3 confirms the sweep summary covers the counter observably).

#### R-V150P1CRONBW-05 — preset-version drift surface (qc3 W-003)
- **Fix**: added an explicit `"novel-brainstorm" | "novel-write" => 1` match arm in `preset_version_for_id` (discoverability for a future version bump; `novel-brainstorm` ships at v1, `novel-write` deferred until R-V150P1CRONBW-01 lands). Justified `#[allow(clippy::match_same_arms)]` documents that the named arms share the catch-all body intentionally — the sync test guards the value, not the arm presence.
- **Test**: renamed `preset_version_mapping_matches_yaml` → `preset_version_mapping_matches_yaml_includes_cron_presets` and extended `known_ids` with `novel-brainstorm` + `novel-write`. `novel-brainstorm` is strictly validated against its embedded YAML; `novel-write`'s absent YAML (R-V150P1CRONBW-01 deferral) is tolerated with `mapping == 1` assertion until authored, after which the strict branch enforces sync automatically.
- **Test**: added focused `preset_version_for_id_novel_brainstorm_resolves` that guards the novel-brainstorm mapping against silent drift even if the `known_ids` array is later pruned.
