## 1. Problem Statement

`SEC-V131-01` records that `judge.llm` and `context.summarize` still read raw `creator_id` / `session_id` values from capability args, while the hardened `acp.prompt` path uses context-injected IDs. This repeats the same IDOR pattern at a lower-risk boundary: orchestration normally calls these capabilities internally, but preset args should not be allowed to spoof execution identity.

P3 closes this single security residual without broad capability refactoring.

---

## 2. Goals

### C1 — Align identity source with `acp.prompt`

1. `judge.llm` reads only context-injected `_creator_id` / `_session_id` for execution identity.
2. `context.summarize` reads only context-injected `_creator_id` / `_session_id` for execution identity.
3. Raw `creator_id` / `session_id` preset args are ignored or rejected according to the existing `acp.prompt` pattern.

### C2 — Regression tests

1. Add tests proving raw args cannot spoof creator/session identity.
2. Add tests proving context-injected IDs still allow legitimate execution.
3. Keep tests deterministic; mock worker/IPC as needed.

### C3 — Residual closure evidence

1. Completion report cites `SEC-V131-01`.
2. QA verifies the fix before PM/QA closes or archives the residual in `status.json`.
