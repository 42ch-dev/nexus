## 1. Problem Statement

V1.36 proved a one-chapter novel-writing flow and V1.37 hardened its foundation. The remaining DF-62 gap is that serial writing does not yet reliably select chapter 2..N from `work_chapters`, update `current_chapter` as chapters finalize, or show a status surface that tells the user what to do next.

This plan implements the state foundation for multi-chapter writing before prompt parameterization. It should make the selected chapter discoverable and testable even if P1 has not yet updated every `novel-writing` prompt.

---

## 2. Goals

1. Implement or expose the equivalent of `next_chapter(work_id)` over `work_chapters`.
2. Select the next chapter from DB state, not filename order:
   - lowest `not_started` row first.
   - resume lowest `draft` row when no `not_started` row should take precedence.
   - do not skip `outlined` rows in favor of later chapters.
3. Update `works.current_chapter` only when a chapter transitions to `finalized`.
4. Apply completion semantics only when all planned rows are finalized and intake is complete.
5. Render `creator run status <work_id>` with chapter progress, row statuses, current chapter, and next action.
6. Add tests for selection, resume, completion, status UX, and relevant DB/frontmatter conflict behavior.

---

## 3. Non-goals

- Prompt/template parameterization for selected chapter content (P1).
- Automatic continuation / auto-reenqueue after a chapter finalizes.
- Full DF-53 auto-chain (`research -> produce -> review -> persist`).
- Multi-volume PK migration to `(work_id, volume, chapter)`.
- World KB implementation (DF-63).
- Findings/rules/logs quality loop (DF-64/65/66/67).
- Platform publish, multi-work switch, or selection pool.

## 4. Design Constraints

### 4.1 Selection Contract

Use the selection contract from `novel-workflow-profile.md` §4.5.2:

```sql
next_chapter(work_id):
  SELECT chapter FROM work_chapters
   WHERE work_id = ? AND status = 'not_started'
   ORDER BY chapter ASC LIMIT 1
  if no row: SELECT chapter FROM work_chapters
              WHERE work_id = ? AND status = 'draft'
              ORDER BY chapter ASC LIMIT 1
  if no row: work is at novel-completion
```

Implementation may include `outlined` explicitly if that better matches local state transitions, but it must not skip an `outlined` chapter in favor of a later row.

### 4.2 `current_chapter`

`works.current_chapter` is the latest finalized chapter number. It is not the chapter currently being outlined or drafted.

### 4.3 Completion and Next Action

Completion stops scheduling and shows a terminal message. Non-terminal status shows a user-controlled next action such as `creator run continue <work_id>`. This is not auto-continuation.

### 4.4 DB / Filesystem Truth

`work_chapters` remains the queryable SSOT. If frontmatter and DB status disagree, DB wins and the next run or reconciliation should re-sync frontmatter without inventing a new chapter transition.

## 6. Acceptance

1. A Work with chapters 1 finalized and 2..N not started selects chapter 2.
2. A Work with an existing draft chapter resumes that draft and does not create/select a later chapter.
3. An `outlined` chapter is not skipped.
4. `current_chapter` updates only on finalize and becomes the just-finalized chapter number.
5. Completion fires only when every planned chapter row is `finalized`, `current_chapter >= total_planned_chapters`, and intake is complete.
6. `creator run status <work_id>` shows chapter progress, row statuses, and a next action without auto-enqueueing.
7. Tests cover chapter 2..N selection and one-chapter compatibility.
