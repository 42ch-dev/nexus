## 1. Problem Statement

After P0 selects chapter 2..N from DB state, `novel-writing` still needs to consume that selected chapter throughout prompt rendering and preset execution. Any chapter-1 assumptions in preset YAML, prompt templates, path builders, or schedule input would keep serial writing from actually producing the selected chapter.

This plan parameterizes `novel-writing` around selected chapter input while preserving a single preset. It must not create `novel-writing-chapter-2`, per-chapter copied presets, or an automatic continuation chain.

---

## 2. Goals

1. Replace chapter-1 hardcoding in `novel-writing` preset/prompt execution with selected chapter inputs.
2. Preserve one `novel-writing` preset for chapter N.
3. Ensure selected chapter context includes at least:
   - `chapter`.
   - zero-padded chapter label / number where templates need it.
   - `outline_path`.
   - `body_path`.
   - `slug` / title fields when available.
   - previous chapter summary or recent finalized chapter context where existing prompt surfaces require continuity.
4. Keep P0 selection and P1 prompt rendering separated enough that each can be tested independently.
5. Add tests proving chapter 2+ renders and writes the selected chapter paths, not chapter 1 paths.

---

## 3. Non-goals

- Implementing `next_chapter(work_id)` or status UX if P0 has not landed.
- Automatic continuation / auto-reenqueue.
- Full DF-53 auto-chain.
- New per-chapter presets.
- Multi-volume PK migration.
- World KB prompt block implementation (DF-63).
- Quality-loop prompt additions for findings/rules/logs (DF-64/65/66/67).
- Platform publish.

---

## 4. Design Constraints

### 4.1 Single Preset

`novel-writing` remains the canonical produce-stage preset for every chapter. Selected chapter data should arrive as `preset.input` or an equivalent typed runtime context, not as a new preset id per chapter.

### 4.2 Selected Chapter Authority

P1 consumes the P0-selected chapter. It must not reimplement a divergent chapter selection algorithm inside prompts or template code.

### 4.3 No Auto-Chain

Finishing chapter N may update status and show the next action, but P1 must not enqueue chapter N+1 automatically.

### 4.4 Chapter 1 Compatibility

The one-chapter and first-chapter paths remain valid. Parameterization should make chapter 1 a normal selected chapter, not a special legacy branch.
