## 0. Contract and file-boundary lock

P1 consumes only the P-1 frozen `@42ch/nexus-contracts` types generated from:

- `MemoryFragmentInfo` — optional `world_id: string | null` plus existing
  `fragment_id`, `summary`, `keywords?`, `created_at?`.
- `ListMemoryFragmentsQuery` — optional `world_id` query param; omitted means
  Creator SOUL whole, present means a specific world projection.
- `SoulNarrativeRequest` — `creator_id` plus optional `force_regenerate`.
- `SoulNarrativeResponse` — `creator_id`, `state`, optional `narrative`, optional
  `generated_at`, `stale`, optional `fragment_count_at_generation`, optional
  `max_fragment_created_at_at_generation`, `current_fragment_count`,
  `current_distinct_keyword_count`, `min_fragment_count`, and
  `min_distinct_keyword_count`.

**File boundary**: P1 may touch only `apps/web/**`, specifically
`apps/web/src/components/soul/**`, `apps/web/src/components/memory/**`,

component/test files adjacent to those surfaces. P1 must not edit `schemas/**`,
`packages/nexus-contracts/**`, `crates/**`, `.sqlx/**`, or migration files.

**Single collision risk**: `apps/web/src/api/queries.ts` and

## 1. Problem statement

The V1.79 SOUL surface (keyword clusters + temporal drift, Creator-scoped,
manual refresh, no narrative) needs four additions to deliver the V1.81
"Creator SOUL Maturation" experience: a narrative card, a world projection
selector, an independent growth-curve, and auto-refresh.

## 2. Scope

### 2.1 Creator-SOUL narrative card (SP-1 / ④)

- New component under `apps/web/src/components/soul/` (e.g.
  `soul-narrative-card.tsx`) consuming the `POST /soul/reflect` endpoint via a
  new `useSoulNarrative` query/mutation in `apps/web/src/api/queries.ts`.
  The route is exactly `POST /v1/local/memory/soul/reflect` and the hook consumes
  the generated `SoulNarrativeRequest` / `SoulNarrativeResponse` types.
- **Five states with explicit UX contract** (each testable, each with copy):

  | State | Trigger | UX | Copy tone |
  |-------|---------|-----|-----------|
  | `ungenerated` | Narrative never generated + fragment count ≥ threshold | CTA card: "Reflect on my SOUL" primary button; below it, a preview hint ("Nexus will reflect on your themes, shifts, and creative growth") | Forward-looking, inviting |
  | `generating` | Synthesis in progress after CTA click | Loading skeleton with a pulse animation on the narrative card body; button disabled with spinner + "Reflecting…" label; no timeout panic (synthesis is synchronous but LLM latency warrants a loading state) | Active, patient |
  | `current` | Cached narrative returned; not stale | Narrative prose block with `generated_at` timestamp ("Reflected on Jul 1, 2026"); a subtle "Re-reflect" secondary action below; the narrative text is read-only and styled with a reflective prose typography token from DESIGN.md | Calm, reflective |
  | `stale` | Cached narrative exists but new fragments have arrived | The cached narrative is still visible but a stale banner sits above it: "You've grown since this reflection — new fragments have arrived." with a "Re-reflect" primary CTA on the banner; `generated_at` is visible | Prompting, not alarming |
  | `insufficient-data` | Fragment count < threshold (by default: < 10 fragments OR total distinct keywords < 20) | Empty-state illustration + copy: "Your SOUL is still forming. Keep writing and reviewing — once you've accumulated enough creative experience, Nexus can reflect on who you are becoming." Below: a count of current fragments + a quiet "X more to go" if close to threshold | Encouraging, educational |

- Placed in the SOUL section (`apps/web/src/components/memory/soul-section.tsx`)
  above the existing keyword/drift viz. The narrative card is the first thing the
  author sees on the SOUL tab — it is the headline feature of V1.81.

### 2.2 World projection selector (SP-2 / ①)

- A selector control (default "All worlds" = Creator SOUL whole) that drives a
  `world_id` query param on the fragments query, re-scoping the keyword
  clusters, temporal drift, and growth-curve to the selected world's subset.
- **Honest subset semantics (UX contract)**:
  - **Default "All worlds"**: the selector shows "All worlds" as the selected
    option with a subtle label: "your whole Creator SOUL."
  - **Worlds with creative activity**: listed when they have at least one
    fragment or at least one existing Work bound to the world. Fragment-backed
    worlds show world name + fragment count badge (e.g., "Eryndor (42
    fragments)").
  - **Worlds with no creative activity (omitted)**: worlds with zero fragments
    and zero Works are **not listed** — no dead-end empty options.
  - **Work-backed but no-fragment world**: if a listed world has Works but zero
    fragments, selecting it shows: "No fragments in this world yet — your
    Creator SOUL is still shaped by your work here when fragments arrive." This
    is distinct from the insufficient-data state (which is about the whole
    Creator SOUL being too thin).
- The selector lives in the SOUL section header; the page-level state
  coordinates with the existing `fragmentKeyword` lift pattern in
  `apps/web/src/pages/memory-page.tsx`.

### 2.3 Growth-curve (SP-3 / ②, BL-10)

- New `growth-curve.tsx` component under `apps/web/src/components/soul/`:
  cumulative fragment count over time, independent of the temporal-drift
  timeline. Respects the world projection.
- **Three density states** (reusing the V1.79 `densityFor` branching pattern,
  which V1.80 raised to a testing bar):
  - `empty` (0 fragments): forward-looking illustration + "Your SOUL begins
    here — every review session adds a fragment to your creative growth."
  - `low-data` (1–9 fragments over ≥2 distinct days): simple line chart + "Your
    SOUL is taking shape. Keep writing to see your growth curve emerge."
  - `rich` (≥10 fragments OR ≥5 distinct days): fully rendered cumulative
    growth-curve with axis labels and a summary stat ("142 fragments over 23
    days").
  The threshold constants mirror the narrative insufficient-data gate for
  consistency, but the growth-curve always renders a chart (degrading to
  simpler forms), unlike the narrative which gates entirely below threshold.

### 2.4 Auto-refresh (SP-4 / ③)

- Poll interval on the SOUL fragments query (react-query `refetchInterval`) +
  invalidation of the SOUL queries after the V1.78 review mutation completes,
  so the viz refreshes without a manual reload.

### 2.5 Tests

- Component tests for:
  - Narrative card: all 5 states rendered with correct copy + CTA per the
    UX contract table in §2.1 (ungenerated, generating, current, stale,
    insufficient-data).
  - World-selector: world list with fragment counts, default "All worlds"
    selection, selected-world re-scoping coverage, Work-backed/no-fragment empty
    state, and the zero-activity omitted-world invariant (test that worlds with
    0 fragments and 0 Works do not render as options).
  - Growth-curve: all 3 density states (`empty`/`low-data`/`rich`) with
    correct thresholds and copy.
  - Auto-refresh: `refetchInterval` configured + query invalidation on
    review-mutation settle.
- Mirror the V1.79 P1 test discipline; the V1.80 frontend-hygiene plan raised
  the bar on reading/SOUL test coverage — match it. This means: each component
  state gets a dedicated test, copy assertions are included, and the
  insufficient-data / empty-state paths are tested (not just the happy
  `current`/`rich` states).
