# V1.9 WS-D — Wire/Local Schema Drift Detection

## Summary

Build a compile-time / test-time validation tool that detects when wire schemas in `schemas/` diverge from local Rust type definitions in `crates/nexus-contracts/src/local/`. Add as a CI gate so drift is caught before merge.

## Background

- **Deferred feature**: DF-08 ("wire/local auto-detect drift") tracked in `.mstar/knowledge/deferred-features-cross-version-tracker.md`.
- **Current state**: The codegen pipeline (`tooling/codegen/`) generates Rust types from JSON Schema into `crates/nexus-contracts/src/generated/` (wire types). Separately, `crates/nexus-contracts/src/local/` contains hand-written local types for types that never cross the wire (`orchestration/preset.rs`, `schedule/core_context.rs`, `domain/`, etc.). There is presently **no automated check** that these local types stay structurally consistent with their wire counterparts, or that all schemas have corresponding type definitions.
- **Existing pattern reference**: The delivery compass mentions `context_summarize_*_schema_valid()` tests as a pattern to generalize. These likely exist in `crates/nexus-contracts/tests/` as per-schema validation tests.
- **`tooling/check-schema-drift.sh`**: Existing script checks DB schema version consistency and DDL duplication (different concern). WS-D adds a NEW check for wire↔local Rust type consistency.

## Design Decisions

1. **Approach: structural field comparison via serde introspection**. For each schema in `schemas/`, the tool:
   - Parses the JSON Schema to extract expected field names and their JSON types.
   - Introspects the corresponding generated Rust type (in `crates/nexus-contracts/src/generated/`) via `serde` field names (using `#[serde(rename = "...")]` annotations).
   - For local types that have wire counterparts: validates that field names and types match.
   - Reports any mismatches: fields present in schema but missing in Rust, fields in Rust but missing in schema, type mismatches.
2. **Implementation as a Rust test**: Add a test file `crates/nexus-contracts/tests/schema_drift_detection.rs` that runs the drift check. This ensures it runs with `cargo test --workspace` and integrates with CI naturally.
3. **Schema inventory**: Maintain a mapping of `schema_file_path → RustStructPath` that lists which schemas correspond to which Rust structs (both generated and local). This mapping is the single source of truth for what gets checked. New schemas are added to this mapping explicitly (opt-in).
4. **Two validation modes**:
   - **Strict match**: Schema field count and names must exactly match Rust struct fields.
   - **Subset match** (local-only): If a local type is intentionally a superset of the wire type (has extra fields for internal use), mark it as `subset_ok`. The tool checks that all wire fields are present in the local type, but not vice versa.
5. **CI integration**: The test runs as part of `cargo test --workspace`. If drift is detected, the test fails with a clear error message listing the specific mismatch (schema name, field name, expected type vs actual type).
6. **Not a CLI command**: Per delivery compass §5 WS-D scope, this is a test/CI gate, not a user-facing CLI tool. However, a convenience script `tooling/check-wire-drift.sh` can be added wrapping `cargo test --test schema_drift_detection`.

## Risks

| ID | Risk | Mitigation |
|----|------|------------|
| RISK-WSD-01 | Field name mismatches due to `#[serde(rename)]` not being introspectable at compile time | Use `serde`'s `Serializer` trait to introspect field names at runtime in the test; this is a well-known pattern |
| RISK-WSD-02 | Schema `$ref` resolution adds complexity | Reuse codegen's `$ref` resolution logic from `tooling/codegen/src/schema-loader.ts`; implement equivalent in Rust or use a simple flat-file approach (pre-resolve refs in the test) |
| RISK-WSD-03 | False positives due to optional fields | `Option<T>` in Rust maps to optional fields in JSON Schema; handle this case explicitly |
| RISK-WSD-04 | Performance: parsing all schemas every test run | Schemas are small (< 50 files, < 5KB each); compile-time overhead is negligible. Cache is unnecessary for initial implementation. |
