## 1. Problem Statement

DF-44 (Reference body externalization — refreshable scan pipeline) is partially shipped (V1.40 static registration). Authors cannot re-sync `kb_extract_jobs` from current chapter text after editing the chapter. KB rows drift from chapter reality.

---

## 2. Goals

1. CLI `creator kb rescan <work_ref>/<chapter>` re-scans the chapter's current text, upserts `kb_extract_jobs` rows (idempotent on `(source_chapter_id, canonical_name_guess)`), and diffs into `nexus-kb` rows.
2. Dry-run mode (`--dry-run`): shows what would change without writing.
3. Output: list of new / updated / removed extract job rows + KB row changes.
4. Author identity check (same as T-B P1).

---

## 3. Non-goals

- Daemon-driven auto-rescan (V1.51+).
- FS watcher (V1.51+).
- LLM-driven extraction (V1.51+).

---

## 4. Acceptance Criteria

1. CLI re-scans chapter; idempotent re-run on unchanged text produces no diff.
2. Edited chapter text triggers upsert of `kb_extract_jobs` rows + delta write to `nexus-kb`.
3. `--dry-run` shows diff without writing.
4. Cross-author attempt returns `403`.
5. Hermetic test: rescan after edit updates KB rows; rescan with no edit produces empty diff.

### Key design decisions (DB-constraint-aware)

1. **Upsert composite identity** is `(source_chapter_id, canonical_name_guess)`
   scoped to the rescanned chapter, but the V1.50 P1 migration reuses
   `work_entry_id = canonical_name_guess`, so the DB enforces one non-failed
   row per `(creator, world, canonical_name)`. `upsert_pending_candidate`
   therefore reuses any existing pending/confirmed row by that key and refreshes
   its `source_chapter_id` + payload — candidates never duplicate across rescans
   of the same or different chapters. Confirmed rows are terminal (§5.5.2) and
   are never mutated by upsert.
2. **nexus-kb delta** (`diff_and_apply`) refreshes the bodies of active
   `KeyBlock`s whose `canonical_name` matches a freshly-extracted candidate,
   so "KB rows reflect current chapter text". It computes a full diff
   (inserted/updated/removed) for reporting but **applies only body refreshes** —
   it never inserts/deletes `kb_key_blocks` (§5.5 reserves those for
   `creator world kb adopt|edit|delete`); `inserted`/`removed` are advisory.
3. **Authz on world ownership**: `resolve_work` resolves `<work_ref>` globally
   so a cross-author attempt resolves the work and surfaces `403` at the
   `narrative_worlds.owner_creator_id` gate (consistent with T-B P0/P1), rather
   than a misleading "work not found".
4. **R-V150KBED-02** (resolve_workspace_id staleness) self-corrects: the rescan
   resolves `workspace_id` fresh from `narrative_gateway` (falling back to
   `creator_id`) for every upserted row.

- **R-V150KBED-06** (pre-existing, **fixed**): duplicate migration version

  `Schema::init` path (all DB tests + CLI runtime). Verified pre-existing:
  `world_kb_promotion_cli` failed on the unmodified branch base. Fix
  (`2cbd5498`): renumbered the partial-index migration to `202606180003`
  (content unchanged; `CREATE INDEX IF NOT EXISTS` is idempotent; the

  Recommended for PM to register in `status.json` root `residual_findings`
  with `lifecycle: resolved`.
- **R-V150KBED-07** (new, low): the refreshable-scan delta deliberately does
  NOT insert/delete `kb_key_blocks` (§5.5 invariant). V1.51+ LLM-driven
  extraction may want a fuller sync surface; revisit the `diff_and_apply`
  apply-subset then. Owner: `@fullstack-dev` / `@product-manager`, target V1.51+.
- **R-V150KBED-08** (new, low): `--dry-run` candidate preview is chapter-scoped
