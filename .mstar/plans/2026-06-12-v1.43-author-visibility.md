## 1. Problem Statement

V1.39 shipped findings, review presets, and 96h banner **in runtime**, but authors must infer state from sparse CLI output. Grill-me **C** requires quality-loop and completion to be **author-visible** without reading API JSON.

---

## 2. Goals

1. `creator run status` / `creator works status` answers: active chapter, completion, open findings count/summary.
2. Surface 96h stale-finding banner on the status path authors use during serial writing (if not already wired).
3. Completion stop messaging aligned with quickstart Part I §6 (coordinate with P1 strings).
4. Update quickstart Part I §4–§6 if implementation reveals doc drift.

---

## 3. Non-goals

- New findings schema or review preset logic
- Platform publish
- Multi-volume runtime fixes → **P-last** (R-V142P1-F-003 etc.)

---

## 4. Acceptance Criteria

1. Author with open findings sees count + highest severity in status output.
2. Completed Work shows unambiguous terminal state per novel-workflow-profile §6.
3. Hermetic CLI or API test covers findings summary + completion display.
4. novel-author-experience §4 table rows marked implemented.
