## 1. Problem Statement

The daemon `POST /v1/local/presets:validate` path is weaker than the `nexus-orchestration` loader validation, and historical CLI validation has drifted from runtime loader semantics. A preset can look valid to a user-facing command/API but still fail or behave incorrectly when loaded as an orchestration graph.

V1.32 P1 creates a shared semantic quality gate so CLI/API validation and runtime loading reject the same structural preset defects.

---

## 2. Goals

### A1 — Shared validation facade

1. Define a reusable validation entry point in or adjacent to `nexus-orchestration` that wraps loader-equivalent checks without duplicating semantics in daemon/CLI layers.
2. Preserve existing size/depth defenses.
3. Return structured diagnostics suitable for CLI text and daemon JSON responses.

### A2 — Logical completeness checks

Reject or report:

1. `initial_state` missing or unable to reach a terminal state.
2. Terminal state marker/header mismatch.
3. Bundle directory id vs manifest id mismatch for bundle validation.
4. Referenced inner graph missing.
5. Orphan inner graphs with no state reference (warning vs error must be an explicit architect decision).

### A3 — Asset and path safety checks

1. Verify `template`, `prompt`, `system_prompt`, `prompt_file`, and `system_prompt_file` references exist when file-backed.
2. Reject traversal, absolute paths, and links outside the preset bundle sandbox.
3. Keep inline prompt/template values legal when supported by the spec.

### A4 — Capability compatibility checks

1. Validate capability names against the active registry.
2. Catch obvious argument/schema drift where capability metadata exposes expected args.
3. Emit diagnostics that explain whether a problem is a missing capability, unknown arg, or unsupported schema check.

### A5 — Validation surface parity

1. `POST /v1/local/presets:validate` uses the same semantic validator as the loader-facing validation path.
2. If P1 adds a user-facing CLI wrapper, it must first choose the command surface explicitly and update `cli-spec.md` plus command-surface contract tests. Do not assume the removed top-level `nexus42 preset ...` group exists.
3. Invalid fixture tests cover every shipped presentation path.

## 3. Non-goals

- Conditional routing / branching engine implementation
- New preset YAML schema version
- DF-29 `registry.refresh`
- DF-31 `workspace.*`
- Platform HTTP or `nexus-platform` changes
- Rewriting all embedded presets (P2 owns repairs after the gate lands)

## 5. Acceptance (plan close)

1. One semantic validation surface is used by runtime loading and user-facing validation paths, or any remaining wrapper has tests proving equivalent failures.
2. Tests cover initial-to-terminal reachability, terminal marker mismatch, id mismatch, orphan/missing inner graphs, missing prompt/template files, unsafe paths, and capability drift.
3. Existing valid embedded presets either pass or fail only on issues assigned to P2.
4. Diagnostics are actionable enough for CLI users and daemon clients.
5. No new platform API work is introduced.
