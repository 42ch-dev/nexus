#### W-P3-01: Sequential LLM judge evaluations add latency to multi-state preset runs
**Scope**: P3 (`crates/nexus-orchestration/src/tasks/mod.rs`)

Each state transition with `exit_when.kind=llm_judge` triggers a synchronous `cap.run(input).await` IPC round-trip. In a typical `novel-writing` preset (gathering → brainstorming → outlining → drafting), four sequential judge evaluations execute back-to-back with no parallelization. At ~300–800ms per LLM call, this adds **1.2–3.2s of wall-clock latency** per full preset run.

**Mitigation**: Acceptable for V1.33 (feature completeness > latency optimization). Future iteration could batch evaluations or cache rendered templates across states.

**Evidence**:
```rust
// crates/nexus-orchestration/src/tasks/mod.rs:767
let (result, reason) = judge_task.evaluate(&context).await?;
```

#### W-P4-01: `save_memory` uses non-atomic file write — corruption risk on crash or disk-full
**Scope**: P4 (`crates/nexus-creator-memory/src/memory_io.rs:149`)

`std::fs::write(&path, &content)` writes directly to the target file. If the process crashes or the disk fills mid-write, the LTM file is left in a partially written (truncated) state. There is no atomic write-then-rename pattern.

**Fix**: Write to a temporary file in the same directory, then `std::fs::rename` to the target. This guarantees atomicity on POSIX and is best-effort on Windows.

**Evidence**:
```rust
// crates/nexus-creator-memory/src/memory_io.rs:149
std::fs::write(&path, &content)
    .map_err(|e| MemoryError::ValidationError(format!("cannot write memory file: {e}")))?;
```

#### W-P4-02: `PassthroughSummarizer` has no size limit — large digests create unbounded LTM files
**Scope**: P4 (`crates/nexus-daemon-runtime/src/api/handlers/memory.rs:635-652`)

The `PassthroughSummarizer` returns `raw_digest` verbatim. A session with a very large digest (e.g., a long brainstorming session with thousands of tokens) will create a correspondingly large LTM file. No truncation, no size guard, and no early rejection.

**Fix**: Add a configurable max size (e.g., 64 KiB) in `promote_to_long_term`; truncate or reject digests exceeding the threshold.

**Evidence**:
```rust
// crates/nexus-daemon-runtime/src/api/handlers/memory.rs:650
Ok(raw_digest.to_string())
```

#### W-P4-03: `memory_fragments` keyword search lacks index — full table scan on `LIKE`
**Scope**: P4 (`crates/nexus-local-db/migrations/20260417_000001_initial.sql`)

The `list_fragments_filtered` query uses `keywords LIKE '%"keyword"%'` (line 202 of `memory_fragment.rs`). The only index is `idx_memory_fragments_creator_id` (line 135). With a large fragment table, the `LIKE` pattern with leading wildcard forces a **full table scan** even when `creator_id` is indexed (SQLite may use the creator index for filtering but still scans all matching rows for the LIKE condition).

**Fix**: Add a `CREATE INDEX idx_memory_fragments_keywords ON memory_fragments(keywords)` or migrate to SQLite FTS5 for keyword search.

**Evidence**:
```sql
-- crates/nexus-local-db/src/memory_fragment.rs:202
WHERE creator_id = ? AND keywords LIKE ?
```

### 🟢 Suggestion

#### S-P3-01: Handlebars template rendering could cache compiled templates
**Scope**: P3 (`crates/nexus-orchestration/src/tasks/mod.rs`)

The static `HANDLEBARS` registry (line 1226) avoids per-call allocation, but `render_template` still parses the template string on every invocation. For presets with large templates and frequent state re-evaluations, this is unnecessary CPU work.

**Fix**: Pre-register templates by name in the registry and use `render()` instead of `render_template()`.

---

#### S-P3-02: `min_interval` throttle stores DateTime as RFC3339 string — parse cost on every check
**Scope**: P3 (`crates/nexus-orchestration/src/tasks/mod.rs:725-775`)

The throttle stores `chrono::Utc::now().to_rfc3339()` as a string in context, then re-parses it with `last.parse::<chrono::DateTime<chrono::Utc>>()` on each evaluation. For high-frequency state machines, this string serialization/deserialization is wasteful.

**Fix**: Store `Duration` or epoch millis as an integer/float in context to avoid parsing.

---

#### S-P4-01: `list_fragments_filtered` uses runtime `sqlx::query` — acceptable but note for future
**Scope**: P4 (`crates/nexus-local-db/src/memory_fragment.rs`)

The dynamic SQL for optional keyword filtering correctly uses runtime `sqlx::query` with a `// SAFETY:` comment (compliant with `nexus-daemon-runtime/AGENTS.md`). The query is parameterized (no injection risk) and bounded by `LIMIT ?`. This is acceptable but should be revisited if the schema stabilizes and a compile-time path becomes feasible.
