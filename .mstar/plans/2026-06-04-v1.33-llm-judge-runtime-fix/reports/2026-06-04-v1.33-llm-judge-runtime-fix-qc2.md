## Findings
### 🔴 Critical
- (cross-cutting, P4 primary) Missing active creator / auth context enforcement for new memory review and fragments endpoints. `POST /v1/local/memory/review` (body `creator_id`), `GET /v1/local/memory/fragments?creator_id=...`, `list_pending_reviews`, `count_pending_reviews`, and `create_pending_review` accept `creator_id` from client-controlled request field (JSON body or query string), validate only format (`is_valid_creator_id` → 400 InvalidInput on bad), then filter DB and invoke `process_review_queue` / `promote_to_long_term` / `list_fragments*` using the supplied value. No call to `read_active_creator_id(state.nexus_home())` (present in works.rs, creators.rs, workspaces.rs) and no `.ok_or(NexusApiError::AuthRequired)?` or equivalent 401/403 for mismatch. In `KeylessLocalhost` mode (tests + possible dev daemon), any localhost caller can target arbitrary `ctr_*` data. This bypasses the creator isolation / SEC-V131-01 pattern (P1 re-QC wave 2 and V1.32 baseline) for the new memory surface. `promote_to_long_term` itself has no internal auth; trusts caller-supplied creator_id for LTM file writes under `~/.nexus42/<creator>/long-term-memory/`.
  → Fix: add active creator extraction + strict match (or 403) in all memory handlers before DB ops (consistent with works pattern); treat missing active as `AuthRequired` (401); add cross-creator tests (seed ctr_a, POST review as ctr_b → 403 or 404).

- (P4) `memory_review_fragments_api.rs` test coverage only exercises 200 OK and 400 (invalid creator format). No coverage of 401 (AuthRequired from middleware), 403 (ownership/cross-creator in delete or review), 404 (not-found pending), or cross-creator isolation matrix (e.g. review as wrong creator returns empty vs error; pending created under ctr_a is invisible / 403 under ctr_b). Tests hardcode `KeylessLocalhost` + single `ctr_testuser`; never exercise real `require_api_key` or `read_active_creator_id` paths.
  → Fix: extend the test server to cover keyed mode + explicit active_creator config; add cases for 401/403/404 and cross-creator 404/403 per lens and T5 acceptance.

- (P4) `PassthroughSummarizer` (used unconditionally for PromoteToLongTerm) returns `raw_digest` verbatim as LTM `body`. `promote_to_long_term` writes it to Markdown under the creator's LTM dir with no `UNTRUSTED`, `SOURCE: session-capture`, length cap beyond review heuristics, or escaping. Downstream consumers (context assembly, moment prompts, SOUL) load LTM bodies as trusted creator memory and interpolate into LLM prompts. A malicious or compromised ACP session can inject prompt-jailbreak content that survives into long-term memory and affects future generations.
  → Fix: either mark LTM bodies from review pipeline with explicit untrusted/provenance frontmatter or tag, or route through a trusted summarizer that adds provenance + caps + sanitization before storage.

### 🟡 Warning
- (cross-cutting) Auth pattern inconsistency between P3/P4 additions and established V1.32/P1 works surface. Memory handlers return `InvalidInput` (400) for bad-format creator_id and `Forbidden` (403) only on explicit ownership mismatch in delete; never surface `AuthRequired` (401) for missing active creator context. Works handlers explicitly do the active read + AuthRequired. Middleware provides the 401 for no API key, but post-auth creator binding is absent for memory. Review/fragments effectively treat the supplied creator_id as self-authenticating within an authed connection.
  → Fix: standardize on active-creator extraction + mismatch → Forbidden (or dedicated error) + missing → AuthRequired (401) for all creator-scoped local endpoints; update memory handlers and add a shared helper if not already in context.

- (P3) `parse_iso8601_duration` only supports `PT?H?M?S` (after leading P and optional T); returns `None` for `P1D`, `P2DT3H`, malformed, or day-based values (no error, no log). In `StateCompositeTask` llm_judge path the throttle block is `if let Some(duration) = parse... { if elapsed < duration { reuse previous } }`; on None it falls through to unconditional `LlmJudgeTask::evaluate` (which may call judge.llm + worker). A preset or schedule with `min_interval: "P1D"` (or future extension) silently disables throttling, amplifying LLM judge calls (cost, rate-limit, worker load). Presets currently use `PT6H`/`PT1H` (supported), but no validation at load time rejects unsupported units.
  → Fix: either reject unsupported min_interval at preset load / schedule admission (with clear error), or log a warning on parse failure + document that only H/M/S are supported; consider full chrono-iso8601 parser if days/weeks needed.

- (P3) LlmJudgeTask template render failure path: `render_core_context_template(...).unwrap_or_else(|e| { warn!; self.template.clone() })` — the raw (possibly containing unsubstituted `{{...}}` handlebars from the preset bundle) is used as the `prompt` value sent to `judge.llm`. The capability then frames it as "You are a judge... {prompt_text}" and ships to worker/acp_prompt. Judge LLM may see literal template syntax in its evaluation input, affecting GO/NOGO reliability or leaking preset internals in the `reason`.
  → Fix: on render failure either hard-fail the exit condition (WaitForInput + error in context) or fall back to a safe static prompt; never send raw template source as evaluation content.

- (P3) `WorkerUnavailable` from `judge.llm` (no provider injected, or worker IPC down) is mapped in LlmJudgeTask to `(false, "...unavailable...")` → `NextAction::WaitForInput` (NOGO). State machines using `exit_when: llm_judge` can be held in perpetual wait if the worker is unavailable (crash, restart race, IPC break). Design documents as "safe default: don't advance without evaluation", but is a liveness/DoS vector for any schedule relying on llm_judge gates (novel-writing gathering, reflection, research). No circuit-breaker, fallback to rule, or operator signal path.
  → Fix: document the liveness impact explicitly in orchestration-engine.md and preset docs; consider a timeout + rule fallback or operator "force continue" signal for llm_judge states; at minimum expose the judge reason in schedule status so operators can see "stuck on unavailable judge".

- (P4) `list_fragments` handler + `list_fragments_filtered` (runtime `sqlx::query` with SAFETY comment for dynamic LIKE) + manual `row.get` mapping after alias with `!` suffix. When `keyword` present, uses filtered path (workaround in handler: non-keyword path uses compile-time `list_fragments` + truncate). The `!` alias is known to have had sqlx compile/runtime issues; using two code paths for the same logical query increases drift risk and means keyword search may hit the "buggy" path in prod. No test coverage exercising the keyword= path end-to-end against the daemon handler (tests only hit the no-keyword 200/empty path).
  → Fix: unify on a single safe query path (or small query builder); add keyword-filter integration test through the handler; ensure the `!` aliases are exercised or the workaround removed.

- (P4) `pending_id` (and `session_id`) values supplied by client in `CreatePendingReviewRequest` (and path param in delete) are used as DB primary keys and passed through to `process_review_queue` / `PendingReviewInput` without additional sanitization or length bounds beyond the validate fn (which only checks creator_id format + non-empty strings). While current code paths only use them in parameterized SQL and for LTM idempotency (via session_id), a future change that derives LTM filenames or fs paths from `pending_id` would be a classic path traversal vector. No max-length or allowlist on pending_id.
  → Fix: add explicit length/format validation for `pending_id` / `session_id` (e.g. `^pending_[0-9a-f]{8,}$`); centralize in a shared validator; document that they are untrusted identifiers only.

- (cross / P4) In `daemon_client.rs`, `list_memory_fragments` builds the query string with `format!(".../fragments?creator_id={}", creator_id)` (no `urlencoding`). Although CLI-side creator_id comes from validated config and `is_valid_creator_id`, any future caller or compromised value could inject `&...` or other query params. `review_pending_memories` correctly uses JSON body.
  → Fix: use `reqwest::Url::parse_with_params` or `urlencoding::encode` for all query values (even validated ones) as belt-and-suspenders.

### 🟢 Suggestion
- (P3) Add a unit test (or extend existing) that supplies a handlebars template containing `{{untrusted_user_content}}` (with special chars) and asserts the rendered prompt sent to the mock judge provider contains the literal substituted value (no double-escaping, since no_escape is intentional for prompts). Already have `acp_prompt_task_nested_handlebars_rendering` QC2-S-001; mirror for judge path.
- (P3) The min_interval throttle key is `"_judge_last_eval_{state_id}"` stored in the graph context (in-memory for the session). On session resume / crash-restart the throttle state is lost; a quick restart can bypass min_interval. Consider persisting last-eval timestamps (per state + schedule) if strict rate limiting is required.
- (P4) The review bulk endpoint has no pagination / cursor (unlike list_pending_reviews); for a creator with thousands of pending it will load all into memory + process sequentially. Add a note or future limit.
- (cross) Extract the duplicated `read_active_creator_id` (and the workspace slug variant) into a single `crate::api::context` helper so new endpoints (memory, future) don't forget to call it.
- (P4) `PassthroughSummarizer` is a zero-cost "I trust the digest" path; consider naming it `RawDigestPassthrough` and adding a comment that it is the review-pipeline's explicit choice to store unsummarized capture content (with the injection risk noted).
- Minor: in `PendingReviewInfo` and response DTOs, `task_kind` and `created_at` are String with no further parsing/validation at API layer (created_at comes from DB or client default); consistent with other local endpoints but worth a comment if strict ISO or enum is ever required.
- The capability registry tests and judge_llm unit tests already cover the _creator_id injection stripping (SEC-V131-01); good, but add one that exercises the full LlmJudgeTask + registry path with a spoof attempt in the rendered prompt (to prove the separate context.get injection wins).
