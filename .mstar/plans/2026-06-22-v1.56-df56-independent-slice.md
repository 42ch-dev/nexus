## Goal

Complete the three DF-56 sub-items that do NOT depend on P0 or P1: (1) arbitrary stage-level conditional `next` for any node type, not just `llm_judge`; (2) expression/rule-based routing beyond the V1.42 P2 GO/NOGO binary; (3) multi-branch graphs with >2 outgoing edges and merge points — extending `preset-conditional-routing.md` from minimal slice to full engine.

## Scope In

- **Arbitrary stage-level conditional `next`**: any stage node (not just `llm_judge`) can declare a `next` with a conditional expression that selects the target at runtime.
- **Expression/rule-based routing**: routing decisions driven by a predicate expression language (e.g. field comparisons on context state, stage output, or work metadata) rather than hardcoded GO/NOGO.
- **Multi-branch graphs**: a single stage node can fan out to >2 `next` targets with labeled edges (e.g. `on_success`, `on_partial`, `on_failure`); the expression result picks the edge.
- **Merge points**: two or more branches can converge at a shared successor node that resumes the workflow; merge semantics (wait-for-all or first-arrival) are configurable.
- Update `orchestration-engine.md` and extend `preset-conditional-routing.md` body for the new routing primitives.
- Tests: reachability across multi-branch graphs, merge semantics, expression evaluation correctness, and regression coverage for existing V1.42 GO/NOGO paths.

## Scope Out

- `registry.refresh` conditional edges — that is P3 (depends on P1).
- `workspace.open`/`workspace.commit` branch inputs — that is P3 (depends on P0).
- New Work profiles or preset authoring for conditional routing.
- Platform/MCP integration.
- Real-time expression evaluation via external service (expressions are locally evaluated).

## Risks

| Risk | Mitigation |
| --- | --- |
| Expression language design grows into a full DSL | Keep expression language minimal: field comparisons, boolean operators, literal values; reject feature creep. |
| Multi-branch graphs create unreachable or cyclic paths | Reachability validator already exists from V1.42 P2; extend it for multi-branch and merge points. |
| Merge semantics (wait-for-all vs first-arrival) introduce non-determinism | Default to wait-for-all; first-arrival requires explicit opt-in and is documented as non-deterministic. |
| Regression on existing V1.42 GO/NOGO paths | V1.42 GO/NOGO tests are preserved and extended; backward compatibility is a hard acceptance criterion. |
