## 1. Problem Statement

`creator works status --json` dumps work API only — no `findings[]`. Remediation footer still suggests `creator run reflection-loop` instead of per-finding hints + `novel-review-master` when empty. Generic (non-novel) branch must stay findings-free per grill #6.

---

## 2. Goals

1. **`--json` enrichment**: append root-level `findings[]` (same shape as list API) + optional `findings_stale` object; preserve daemon GET work fields unchanged.
2. **Remediation (Grill #7)**: remove blanket `reflection-loop` footer; per-row `routing_hint`; empty findings → suggest `creator run novel-review-master <work_id>`.
3. **Novel-only scope**: findings fetch/display only when `work_profile=novel`; document in spec §4.
4. **P1 handoff**: remediation strings should target spec paths (P1 may adjust copy in same file — coordinate serial merge).

---

## 3. Non-goals

- Reintroducing list behavior on `creator run`.
- Findings for generic work profile.
- Quickstart or spec body rewrites (P1).
- Chapter on-disk hints (P2).

---

## 4. Acceptance Criteria

1. `creator works status <novel_work_id> --json` includes non-empty `findings[]` when open findings exist.
2. Human output uses per-finding hints; no blanket `reflection-loop` footer.
3. Generic work `works status` does not fetch findings (human + json).
4. Unit/integration tests cover json shape + novel-only gate.
5. Spec §4.1 documents JSON contract.
