## 1. Problem Statement

V1.36 shipped the novel-writing vertical slice, but the front door is not yet reliable enough for ordinary users. The init preset can collect the right answers, yet those values are not fully threaded into schedule / `preset.input`; `gates:` are specified but not evaluated at enqueue time; scaffold writes are recoverable but do not satisfy the single-transaction atomicity clause in the spec. Users can still fail late inside the graph instead of receiving a clear remediation before scheduling.

---

## 2. Goals

1. Close or directly target `R-V136P1-01`: `novel-project-init` grill-me output reaches schedule input and `preset.input.*`.
2. Close or directly target `R-V136P3-02`: runtime `preset.gates` evaluation runs at enqueue time and returns structured `preset_gates_failed` errors.
3. Close or directly target `R-V136P1-02`: scaffold `work_chapters` row seeding and `works` PATCH are committed in one DB transaction or equivalent all-or-nothing unit.
4. Preserve the V1.36 pre-1.0 no-shim policy for `Works/<work_ref>/`.
5. Improve the first-run UX so missing scaffold or incomplete intake produces explicit next-step guidance.
6. Optionally clean `R-V136P3-01` legacy prompt assets and extract `R-V136P2-02` path helpers only if they reduce the P0 implementation risk.

## 5. Acceptance

1. Running `novel-writing` without required scaffold fails before enqueue with `preset_gates_failed`, including the missing gate and a command-level remediation.
2. `novel-project-init` structured answers are visible as `preset.input.*` to scaffold and prompt rendering.
3. A failed scaffold cannot leave `work_chapters` rows committed without the corresponding `works` novel-profile PATCH.
4. Forced gate override is explicit and audit-logged.
5. Tests cover success and failure paths for gates, input wiring, and scaffold atomicity.
6. Residuals `R-V136P1-01`, `R-V136P1-02`, and `R-V136P3-02` are either closed or left with a precise remaining gap.
