## Goal

Consolidate the dual-outbox architecture into a single, unified outbox schema. Currently two outbox concepts exist: (1) a daemon legacy `outbox` table in `nexus-local-db` (command queue), and (2) a sync client `Outbox` in `nexus-cloud-sync` with its own `outbox_entries` table (DDL in `init_pool_with_schema`, NOT in migrations). This plan migrates the sync outbox DDL into `nexus-local-db/migrations/`, enforces a single-writer rule per event type, narrows/deprecates the daemon legacy table, and wires the currently-stubbed `outbox.flush`/`outbox.compact` orchestration capabilities to real implementations.

---

## Scope

- **T1**: Migrate cloud-sync outbox DDL to `nexus-local-db/migrations/`. Extract the `outbox_entries` table schema from `crates/nexus-cloud-sync/src/outbox.rs` `init_pool_with_schema` into a new migration file under `crates/nexus-local-db/migrations/`. Update `nexus-cloud-sync` to reference the migration-managed schema instead of inline DDL. Ensure forward/rollback migration is testable.
- **T2**: Document & enforce single-writer rule per event type. Create a new Draft spec `outbox-consolidation.md` under `.mstar/knowledge/specs/` that defines: which event types map to which writer (daemon or sync), exclusivity contracts, and the schema ownership boundary. Add runtime enforcement (e.g. assert or debug-mode check) in the outbox write paths. Update `orchestration-engine.md` with an outbox consolidation section referencing the new spec.
- **T3**: Deprecate/narrow the daemon legacy `outbox` table. Audit all references to `nexus-local-db::schema::outbox` (see `crates/nexus-local-db/src/schema.rs`). If the legacy table has active daemon-side consumers beyond the sync client, add a deprecation notice + plan for phased removal. If unused or only used by sync paths already consuming the cloud-sync `Outbox`, narrow the schema (remove unused columns, add deprecation comment). Do NOT drop the table in this plan — scope is narrowing + deprecation marking.
- **T4**: Wire `outbox.flush` and `outbox.compact` orchestration capabilities to real implementations. Current stubs in `crates/nexus-orchestration/src/capability/builtins/outbox.rs`:
  - `flush` returns "cloud line disabled" — replace with actual outbox drain logic (local-only for now: drain pending entries to a local delivery log or mark as flushed). Document the contract in `outbox-consolidation.md`.
  - `compact` is a smoke stub — replace with actual compaction logic (remove successfully-delivered entries older than a configurable retention window). Document the contract.
  - Both capabilities remain local-only (platform paused). Add per-capability test vectors (success + failure paths).
- **T5**: Regression tests for sync CLI. Verify all `nexus42 sync *` commands (which use `nexus_cloud_sync::outbox::Outbox` directly — see `crates/nexus42/src/commands/sync/mod.rs`) still function correctly with the migration-managed schema. Add integration tests that exercise the sync CLI path end-to-end with the consolidated outbox.
- **T6**: Amend `daemon-runtime.md` — add flush/compact invocation path for outbox capabilities; document single-writer enforcement in outbox dispatch.
- **T7**: Verify: `cargo test -p nexus-cloud-sync` + `cargo test -p nexus-local-db` + `cargo test -p nexus-orchestration --test capability_registry` (new flush/compact test vectors) + `cargo clippy --all -- -D warnings` + `cargo +nightly fmt --all -- --check`.

## Risks

- **Legacy outbox table has active daemon-side consumers**: If the daemon legacy `outbox` table in `nexus-local-db` is used by daemon command queue operations beyond the sync client, deprecation may require coordination. Mitigation: T3 audit step identifies all consumers; if active consumers exist, narrow scope to deprecation notice + plan for phased removal (do NOT drop the table in V1.59).
- **Sync CLI regression**: Existing `nexus42 sync *` commands use `nexus_cloud_sync::outbox::Outbox` directly. If migration changes the table structure or initialization path, sync commands may break. Mitigation: T5 regression tests catch this; migration preserves existing table structure (only changes initialization from inline DDL to migration-managed).
- **Flush/compact semantics undefined**: Current stubs return "cloud line disabled" with no documented contract. Defining flush/compact semantics requires product decision (what does "flush" mean in local-only mode?). Mitigation: Draft spec `outbox-consolidation.md` documents the contract; implementers follow the spec; PM reviews before merge.
- **Cross-track merge conflict on `nexus-local-db`**: P0 (Track A) may touch `nexus-local-db` for manuscript DAO. P1 touches `nexus-local-db` for outbox migration + legacy table deprecation. Mitigation: different subsystems (manuscript vs outbox); PM resolves conflicts on `iteration/v1.59` if both plans land schema changes.
