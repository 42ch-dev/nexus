## 1. Problem Statement

### Sub-feature A: Work‚ÜíKeyBlock provenance gap (R-V150KBED-02)

V1.50 T-B P1 introduced World KB with `creator world kb` CLI surface and `kb_key_blocks` persistence. KB ownership currently resolves via `narrative_worlds.owner_creator_id`. This is sufficient when only one Work exists per World, but when multiple Works share a World (multi-Volume, multi-author, or cross-Work collaboration), the KB row owner cannot distinguish which Work extracted which entity. R-V150KBED-02 (deferred from V1.50 kb-editor-cli QC) flags this as an author/provenance gap: promoted KeyBlocks carry no `source_work_id`, making it impossible to trace which Work's extraction produced a given confirmed row.

Additionally, the V1.51 T-A P0 LLM extraction pathway enriches candidates with `llm_confidence`, `llm_source_quote`, and `source_chapter_id` in `kb_extract_jobs`. When those candidates are adopted, the provenance is available in the extract job but not carried into the target `kb_key_blocks` row.

### Sub-feature B: first non-novel work profile ‚Äî essay

Currently only `work_profile: novel` is implemented (V1.36‚ÄìV1.51). The `essay` Feature line spec is fully authored in P-1 (`essay-profile.md`). T-A P2 implements the first non-novel profile with a single-artifact `Drafts/draft.md` layout, lightweight quality check, optional World binding, and chapter-free stage chain (`intake ‚Üí outline ‚Üí draft ‚Üí revise ‚Üí finalize`).

## 7. Design Notes

### 7.1 Work‚ÜíKeyBlock provenance columns

| Column | Type | Description |
|--------|------|-------------|
| `source_work_id` | TEXT (nullable FK ‚Üí `works.work_id`) | The Work that produced this KB row via extraction |
| `source_chapter` | INTEGER (nullable) | Chapter number where the entity was extracted (novel only; NULL for essay) |
| `source_provenance_kind` | TEXT (CHECK enum) | How the row entered the KB graph |

`source_provenance_kind` enum values:
- `manual` ‚Äî Author directly added via `creator world kb add`
- `review_time_extract` ‚Äî review-time extraction hook (V1.50/V1.51)
- `finalize_time_extract` ‚Äî finalize-time extraction (novel finalize hook)
- `cross_chapter_rescan` ‚Äî V1.51 T-A P1 cross-chapter reconciliation
- `author_explicit` ‚Äî Author explicitly confirmed/adopted a pending candidate

### 7.2 Author gate with provenance

When checking authorization for `creator world kb edit` / `delete`, the gate logic considers:
1. If `kb_key_blocks.source_work_id` is present ‚Üí verify `works.creator_id` matches the active Creator
2. Otherwise ‚Üí fall back to `narrative_worlds.owner_creator_id` (existing behavior)

### 7.3 Essay profile preset stages

```
states:
  intake ‚Üí outline ‚Üí draft ‚Üí finalize ‚Üí done
```

Each state gate:
- `intake`: `work_profile IN (null, essay)`, `workspace_slug` required, no intake yet
- `outline`: `intake_status = complete`, outline file does not exist or is incomplete
- `draft`: outline exists, `Drafts/draft.md` status != `finalized`
- `finalize`: draft exists, `Drafts/draft.md` status = `draft` or `revised`

### 7.4 Essay project scaffold

The `essay.project_scaffold` capability:
1. Creates `Works/<work_ref>/` directory
2. Creates `Works/<work_ref>/Outlines/outline.md` (template per spec ¬ß4)
3. Creates `Works/<work_ref>/Drafts/draft.md` (template per spec ¬ß4)
4. Creates `Works/<work_ref>/README.md`
5. PATCHes `works` row: `work_profile = 'essay'`, `work_ref = <ref>`, `intake_status = 'pending'`

### 7.5 Essay completion

Per `essay-profile.md` ¬ß8:
```
Drafts/draft.md frontmatter status == finalized
AND works.intake_status == complete
‚Üí works.status = completed
```

No novel `work_chapters`, `current_chapter`, or `total_planned_chapters` involvement.
