# V1.9 WS-B — Template Path Hardening + Name Validation

## Summary

Add path traversal validation for `template_file` references in presets (both embedded and filesystem-loaded), and add max-length validation for the `--name` flag in `creator register`. This hardens the preset loading surface against malicious paths and closes residual R-WC-001.

## Background

- **Deferred feature**: DF-09 ("Template_file path traversal protection") tracked in `.mstar/knowledge/deferred-features-cross-version-tracker.md`.
- **Residual merge**: R-WC-001 ("No max-length validation on name") from `2026-04-23-v1.8-ws-c-name-positional-to-flag` — see `.mstar/status.json` **root** `residual_findings` entry.
- **Existing defense pattern**: `nexus-home-layout/src/lib.rs` lines 49-83 — `creator_soul_md_path()` calls `assert_creator_id_safe()` which rejects `/`, `\`, `..`, and control characters. WS-B applies the same pattern to `template_file` paths in presets.
- **Why after WS-A**: Third-party presets loaded from `~/.nexus42/presets/<name>/` are the primary attack surface. WS-A must be stable so the validation knows which root directory to use as the sandbox boundary.
- **Existing name validation**: `validate_handle()` in `crates/nexus42/src/commands/creator.rs` (line 156-164) validates handle format via regex but there is NO length check on the `--name` string in `register_creator()`.

## Design Decisions

1. **Path traversal check in `load_preset_from_str()` and `load_preset()`**: After parsing, walk all `template_file` fields in the manifest. For each path:
   - Reject if path contains `..` (directory traversal)
   - Reject if path starts with `/` (absolute path)
   - Reject if path contains null bytes
   - For filesystem-loaded presets (`load_preset()`): resolve relative to the preset's bundle root and verify the canonical path stays within the bundle root (sandbox check)
   - For embedded presets (`load_preset_from_str()`): verify the path is a simple relative path without traversal components
2. **Reuse existing pattern**: Model validation after `assert_creator_id_safe()` in `nexus-home-layout/src/lib.rs` lines 62-83. Create `assert_template_file_safe(path: &str)` in `loader.rs`.
3. **Report as validation error**: Invalid paths produce `ValidationProblem` entries with path `"<state_path>.template_file"` or `"<state_path>.context_update.template_file"`, consistent with existing loader validation.
4. **Name max-length validation**: Add check in `register_creator()` in `creator.rs` BEFORE the platform API call. Max 64 chars for `name`. Error message: `"Creator name exceeds maximum length (64 characters)"`. Use existing `CliError` variant pattern.

## Risks

| ID | Risk | Mitigation |
|----|------|------------|
| RISK-WSB-01 | Path traversal validation too aggressive, breaks legitimate preset layouts | Pattern after existing `assert_creator_id_safe()`; test with real `novel-writing` preset paths like `"prompts/gathering-exit.md"` |
| RISK-WSB-02 | Sandbox check (`canonicalize`) may fail on symlinks or nonexistent paths | Only validate format (no `..`/absolute) for structure check; sandbox check is a defense-in-depth layer for filesystem presets only; gracefully handle `canonicalize` errors |
| RISK-WSB-03 | Scope creep beyond template_file + name validation | WS-B is intentionally S-size; reject additional validation requests |
