## 1. Problem Statement

V1.42 P0 shipped `runtime_lock_holder` DB column + `RuntimeLockGuard` RAII in `nexus_local_db::runtime_lock`. V1.49 §1.2 OUT-B O14 retained the **single-writer daemon model** with atomic temp+rename as the V1.49 choice; R-V149P1-01 note flagged a hard advisory lock as a V1.50+ follow-up if multi-writer arrives.

V1.50 introduced cron staggering which now creates real multi-writer contention: daemon cron-side fires `novel-brainstorm`/`novel-write`/`novel-review-master` while author runs CLI commands like `creator works cron set` or `creator run outline-chapter`. Without a hard cross-process advisory lock, the read-modify-write window for `cron set` (R-V150P0-W5) is racy.

T-B P0 adds a file-based companion lock `Works/<work_ref>/.lock` (flock + heartbeat) for cross-process safety, integrated into both daemon cron-side and CLI-side mutating paths.

---

## 2. Goals

1. File-based companion lock at `Works/<work_ref>/.lock` with `flock(LOCK_EX)` + heartbeat (write `<pid>:<holder>:<expires_at_ms>` to lock file; refresh every 30s while held).
2. New `nexus_local_db::file_lock` module: `try_acquire(work_ref) -> Result<FileLockGuard, Locked>`; RAII release on drop.
3. Daemon cron-side: every mutating path (cron evaluator + cron-fire enqueue) acquires file lock before read-modify-write of `works.schedule_json` or `kb_extract_jobs`.
4. CLI-side: every mutating command (`creator works cron set`, `creator run`, `creator world kb adopt`) acquires file lock; on contention, returns `E_LOCK: work is held by <pid>:<holder>` with stable exit code 75.
5. Lock holder visible via `creator works status --json` + `creator world show <work_ref>` (informational).
6. Closes R-V149P1-01 advisory-lock note portion (spec-reconciliation portion already closed V1.49 P-last).
