# V1.11 — Daemon Stub Cleanup + DeviceID TOCTOU Fix

## Summary

Clean 14 daemon legacy items (dead modules, stub functions, placeholder code, unnecessary `#[allow(dead_code)]` annotations, and test infrastructure debt) explicitly deferred from V1.10 WS-C. Fix the TOCTOU race condition in `get_or_create_device_id()` (R-V110-004, warning) by replacing check-then-create with atomic `create_new` file operation.

## Background

- **V1.10**: Done. Device Flow Login + DeviceID Header + Daemon Auth/Context Mock Removal shipped. 14 daemon non-HTTP cleanup items were explicitly deferred to V1.11 (V1.10 compass §6 WS-C).
- **R-V110-004**: TOCTOU race in `get_or_create_device_id()` — `std::fs::metadata(path)` check followed by `std::fs::write(path, uuid)` creates a window where the file can be tampered with.
- **Design compass**: `.mstar/iterations/v1.11-delivery-compass-v1.md`

## Design Decisions

1. **Cleanup scope**: Only the 14 items recorded in V1.10 compass §6 WS-C. No expansion to unrecorded code smells.
2. **TOCTOU fix**: Use `std::fs::OpenOptions::new().write(true).create_new(true)` for atomic file creation. `AlreadyExists` error falls back to reading existing file. API unchanged.
3. **No new dependencies**: V1.11 introduces no new crate dependencies.
4. **Group-by-blast-radius execution**: 14 cleanup items split into 4 groups ordered by risk — delete entire files first (highest blast radius, easiest to verify), then surgical edits, then annotation cleanup, then test infrastructure.
5. **WS-A and WS-B are independent**: WS-A touches `crates/nexus42d/` only; WS-B touches `crates/nexus-sync/` only. No code conflicts, fully parallelizable.

## Out of Scope

- CLI refresh_token support (separate plan: `2026-04-28-v1.11-cli-refresh-token`)
- Backlog residual governance (13 items — R-WA/R-WC/R-M1 series)
- DF-11 Handlebars template engine
- Daemon runtime behavior changes, API route changes, SQLite schema changes, new features
