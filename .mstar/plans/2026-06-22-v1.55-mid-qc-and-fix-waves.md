## Goal

Make the V1.55 “mid merge + tri-review + mid-QA + fix-wave” rhythm explicit and auditable for P0, P1, P2, and P3.

## Scope In

- Track each implement plan’s merge to `iteration/v1.55`.
- Ensure QC tri-review and mid-QA happen on the integrated branch checkpoint.
- Track targeted fix-waves and re-review decisions.
- Prevent the next dependent plan from starting before the prior checkpoint is accepted or PM-overridden.

## Scope Out

- No product code.
- No replacement for actual QC reports under `.mstar/plans/reports/<plan-id>/`.
- No QA execution by this plan; it records routing and evidence only.

## Plan Steps

The mid-QC rhythm runs as **two parallel waves** (Track A || Track B) — each plan still individually merges + receives its own mid-QC at its own integrated checkpoint, but cross-track plans progress in parallel rather than sequentially.

### Wave 1 (parallel mid-QC; Track A || Track B)

| ID | Step | Deliverable |
| --- | --- | --- |
| T1 | After P0 (Track A) merges, dispatch QC tri-review + mid-QA on `iteration/v1.55` at P0's integrated HEAD | P0 report set |
| T2 | Record P0 findings, fix-wave, targeted re-review, and acceptance | P0 checkpoint note |
| T3 | After P2 (Track B) merges, dispatch QC tri-review + mid-QA on `iteration/v1.55` at P2's integrated HEAD | P2 report set |
| T4 | Record P2 findings, fix-wave, targeted re-review, and acceptance | P2 checkpoint note |
| T5 | Wave 1 acceptance gate: BOTH P0 and P2 accepted (or PM override recorded). Wave 2 may begin. | Wave 1 gate note |

### Wave 2 (parallel mid-QC; Track A || Track B; requires Wave 1 accepted)

| ID | Step | Deliverable |
| --- | --- | --- |
| T6 | After P1 (Track A, requires P0 acceptance) merges, dispatch QC tri-review + mid-QA on `iteration/v1.55` at P1's integrated HEAD | P1 report set |
| T7 | Record P1 findings, fix-wave, targeted re-review, and acceptance | P1 checkpoint note |
| T8 | After P3 (Track B, requires P2 acceptance) merges, dispatch QC tri-review + mid-QA on `iteration/v1.55` at P3's integrated HEAD | P3 report set |
| T9 | Record P3 findings, fix-wave, targeted re-review, and acceptance | P3 checkpoint note |
| T10 | Wave 2 acceptance gate: BOTH P1 and P3 accepted (or PM override recorded). P-last may begin. | Wave 2 gate note |

PM dispatch rule for parallel waves: when both plans in a wave are merged, PM dispatches the corresponding mid-QC invocations within the same dispatch round (per `mstar-dispatch-gates` parallel-dispatch rule).

## Acceptance Criteria

- [ ] P0, P1, P2, and P3 each have a recorded integrated checkpoint before formal review.
- [ ] Each checkpoint has three QC reports and one mid-QA report, or a PM-recorded exception.
- [ ] Mid-QC runs in **two parallel waves** (Wave 1: P0 || P2; Wave 2: P1 || P3), not four sequential reviews.
- [ ] Wave 2 begins only after both Wave 1 plans are accepted (or PM override recorded).
- [ ] Fix-waves are targeted and tied to specific findings.
- [ ] P-last does not start until all four implement checkpoints are accepted or explicitly overridden by PM.
- [ ] Review ranges for each checkpoint are reproducible and recorded in assignments/reports.

## QC Strategy

This meta-plan is not itself tri-reviewed. It exists to ensure the four implement plans receive tri-review + mid-QA at integrated checkpoints. P-last QA verifies that the evidence exists.

## Risks

| Risk | Mitigation |
| --- | --- |
| Review overload from 12 QC reviews + 4 mid-QA | Keep scopes per checkpoint narrow and use targeted re-review for fixes. |
| QC reviews wrong branch | Assignments must use `iteration/v1.55` and identical review range/diff basis per checkpoint. |
| Fix-wave drift | Every fix maps to a finding; unrelated work becomes a new residual or plan. |
