## 5. Technical Design

### 5.1 Write-tool pattern

Every write tool follows the established registry pattern:

```
client request → ACP wire → host_tool_executor::execute()
  → dispatch → CapabilityRegistry::dispatch(capability_id, input)
    → AdmissionGate chain check (allowlist → ownership → policy → audit)
    → handler(input) → response (JSON)
    → audit_tool_execution()
```

**Write-specific admission gates**:
- `RequireWorldOwnership` — for world-scoped tools (kb_snapshot.write, world.configure)
- `WorkspaceBounds` — for work-scoped tools (manuscript.chapter.update, work.schedule.set)
- `PermissionPolicy` — all writes require explicit policy check
- `AuditLog` — all writes generate audit trail

**Handler locations**: all in `crates/nexus-daemon-runtime/src/api/handlers/tools.rs` or `host_tool_executor.rs`, following existing handler patterns.

|---------|--------|-----------------|---------|---------|
| `nexus.kb_snapshot.write` | Write | Allowlist, RequireWorldOwnership, PermissionPolicy, AuditLog | `handle_kb_snapshot_write` | `creator world kb edit/adopt` |

### 5.3 Registry allocation cache

**Current**: `host_tool_registry()` constructs a new `CapabilityRegistry` on every call, allocating `HashMap` + 19 `CapabilityRow` + 19 `Vec<AdmissionGate>`.

**V1.54**: Convert to `static` singleton via `std::sync::LazyLock`:

```rust
use std::sync::LazyLock;

static REGISTRY: LazyLock<CapabilityRegistry> = LazyLock::new(|| {
    let mut reg = CapabilityRegistry::new();
    // ... register all 19 rows (13 existing + 6 new) ...
    reg
});

pub fn host_tool_registry() -> &'static CapabilityRegistry {
    &REGISTRY
}
```

**Secondary optimization**: Convert `CapabilityRow::admission: Vec<AdmissionGate>` → `admission: &'static [AdmissionGate]`. Eliminates per-row heap allocation at init time.

**Benchmark**: Add Criterion benchmark at `crates/nexus-daemon-runtime/benches/dispatch_latency.rs`. Before cache: measure `host_tool_registry()` construction + 19 lookups. After cache: measure 19 lookups on `&REGISTRY`.

## 8. Verification Plan

### Per-tool test matrix (≥18 tests)

| Test type | Assertion |
|-----------|-----------|
| Happy path | Tool receives valid input → returns Ok(JSON) with expected fields |
| Admission reject | Unknown/not-allowlisted tool id → Err(NOT_SUPPORTED) |
| Invalid input | Missing or wrong-typed field → Err(INVALID_INPUT) |
| Missing world/workspace | World-scoped tools without world → Err(FORBIDDEN) |
| Audit trail | Write operations produce audit log entries |

### Cross-cutting tests

| Test | Verifies |
|------|----------|
| Concurrent dispatch (10× parallel) | No deadlock, no data race on `LazyLock` + `&self` dispatch |
| Schedule caller-kind admission | `dispatch_for_schedule` produces same admission as direct dispatch |
| Catalog↔registry bijection | Every catalog id has a registry row; every registry id has catalog entry |
| ALLOWLIST ↔ registry sync | No tool in registry without allowlist entry; no orphan allowlist entries |
| Criterion benchmark | Before/after latency improvement (target: <1µs/lookup after cache) |

### CI Gate

```
cargo clippy --all -- -D warnings
cargo test --all
cargo +nightly fmt --all --check
```

All `sqlx` queries use compile-time macros; SQL `LIMIT` uses `LIMIT ?` with regenerated `.sqlx/` snapshots.
