# P2: Track A — Control Room + Setup Screens

The MVP product surface: author-facing Web UI screens consuming the hardened Local API (P0) via the scaffold (P1). This is the largest plan; may split (P2a Control-Room READ ‖ P2b Setup CRUD) if @product-manager/@architect advise in Phase 2b review.

## Tasks

- [x] T1: **TanStack Query data layer** — per-resource query hooks (`useWorks`, `useWork`, `useSessions`, `useSchedule`, `useCapabilities`, `useFindings`, `usePresets`, `usePreset`) consuming `NexusClient` (P1). Cursor-pagination handling; `ErrorResponse` → toast notifications. **F-P3 adapter**: thin transformer normalizing `works`/`schedules`/`sessions`/`capabilities` arrays → `items` (compass §5 item #2). **F-F1 adapter**: client-side sort for small lists.
- [x] T2: **Works dashboard** (Control Room) — cursor-paginated list (status badges, completion %), filters, work-detail view (chapters/status for novel; sections for game-bible/script; draft for essay). Empty/loading/error states.
- [x] T3: **Orchestration sessions view** (Control Room) — list sessions (running/completed/failed), per-session status detail, stage progression. READ-only.
- [x] T4: **Schedule view** (Control Room) — cron roles per Work (brainstorm/write/review) + next-fire (UTC) + local-time display. Parity with CLI `creator works cron` output. READ-only.
- [x] T5: **Capability registry browser** (Control Room) — list `nexus.*` capabilities + descriptions + admission gates. READ-only.
- [x] T6: **Findings view** (Control Room) — list findings (post-F-P2 endpoint) with status/severity filtering; link to source work. READ-only.
- [x] T7: **Work CRUD** (Setup) — create work (profile enum: novel/game-bible/script/essay), patch (status, stage), archive. Form validation against `@42ch/nexus-contracts` types.
- [x] T8: **Preset management** (Setup) — full CRUD: list / show / create / update / delete / validate. Schemas already promoted (V1.63). Editor form for preset YAML/JSON; validate action surfaces errors.
- [x] T9: Responsive layout + a11y (keyboard nav, ARIA, color-contrast per DESIGN.md tokens).
- [x] T10: Verify — `pnpm --filter web build` + typecheck; manual smoke against running daemon (each screen loads, CRUD operations succeed, errors surface as toasts).
- [x] T11: Capture screenshots for the `web-ui.md` spec (Shipped promotion evidence in P-last).

### Implementation notes (P2 delivery)

- **T1**: F-P3 `normalizeList` + F-F1 `sortByDate` live in `src/lib/nexus/adapters.ts` (covered by `adapters.test.ts`). Cursor pagination via `useInfiniteQuery` for Works + Findings (`fetchNextPage`/"Load more"). Toast system (`src/lib/use-toast.tsx`) wired into `QueryCache.onError` so every read failure surfaces one parsed `ErrorResponse` toast (W-1 fix live). `listFindings` wired across the interface/Browser/Tauri.
- **T2**: completion % is derived on the detail view from `current_chapter`/`total_planned_chapters` (novel profile); `WorkSummary` carries no completion field so the list shows status/intake badges only.
- **T7 scope gap (residual)**: `CreateWorkRequest`/`PatchWorkRequest` have **no `work_profile` field** — the daemon assigns profiles internally. The UI offers foundational fields (title/goal/idea + status/stage patch + archive) and surfaces this gap rather than faking a profile selector.
- **T8 scope gap (residual)**: the daemon exposes only list/scaffold/validate/reload for presets (no get/update/delete routes or contracts). The UI delivers those four; `validate` (dry-run) is the product-priority #1 hero feature with structured error/warning surfacing.
- **T5 scope gap (residual)**: `CapabilityInfo` carries name + I/O schemas only; admission-gate data is not exposed in the list response.
- **T10/T11**: `pnpm --filter web typecheck` + `build` + `test` (30 tests) green. Preview-server smoke captured screenshots of all 7 routes + dark mode + Create Work dialog (`apps/web/screenshots/`). Live daemon serving smoke is deferred to P-mid/QA — P3 owns the rust-embed binary that serves the dist.
