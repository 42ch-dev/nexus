  - manifest.json format = required {module_id, name, version, nexus_abi_version, required_key_block_types, compute_export, init_export}; optional {description, author, host_functions, battle_report_kind, max_fuel, max_memory_mib, max_wall_time_ms} (open item #3).

  - host ABI surface = two whitelisted imports, module namespace `nexus`: `kb_read(id_ptr,id_len,out_ptr,out_cap)→i64` and `narrative_query(q_ptr,q_len,out_ptr,out_cap)→i64`, memory-buffer marshalling, sentinels −1/−2 (open item #4).
- Residual findings: none.
- Risks for Wave 3 (P3 will use WasmEngine):
  - `compute()` is synchronous and spawns+joins a wall-time watchdog thread per call; under high call volume this is fine (each call is μs–ms) but P3 should reuse one `WasmEngine` for all calls (engine construction is the expensive part). Modules are compiled once and reused via `WasmModule`.
  - Memory cap uses wasmtime `StoreLimits` (grow-time limiter). For V1.61 the fuel + wall-time combination bounds compute; `basic-combat` is well under every limit. A future hard memory cap that also bounds initial instantiation can be layered into `InvocationState` if a hostile module is ever embedded.
  - basic-combat uses the inline `key_blocks` snapshot (canonical path per schema) and does NOT call host imports; host-function wiring is validated end-to-end by a WAT probe unit test + a non-whitelisted-import-rejection test. If P3 wants a module that demonstrates `kb_read` from real compute, it can add one.
  - `narrative_query` is V1 pass-through of `narrative_state`; a richer query engine is a later iteration.
