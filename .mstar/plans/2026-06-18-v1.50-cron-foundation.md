## 1. Problem Statement

Per-Work cron config is not implemented. `works.schedule_json` column does not exist. `creator works cron` subcommands do not exist. Authors have no way to set staggered schedules for brainstorm/write/review.

---

## 2. Goals

1. Add `works.schedule_json TEXT` column (nullable; empty = defaults).
2. DAO: `set_schedule_json(work_id, json)` / `get_schedule_json(work_id)`.
3. CLI: `creator works cron set/show/list` per spec ¬ß3.
4. Default schedule table constant (`DEFAULT_SCHEDULE` in `cron.rs`).
5. Hermetic tests for migration (forward + back-compat) + DAO + CLI smoke.

---

## 3. Non-goals

- Cron firing into auto-chain (T-A P1).
- Review cron (T-A P2).
- Auto-chronology (T-A P3).

---

## 4. Acceptance Criteria

1. Migration `202606180001_works_schedule_json.sql` adds the column; existing Works get NULL (= defaults).
2. `cargo test -p nexus-local-db --test works_schedule_migration` passes (forward + rollback).
3. `cargo test -p nexus42 --test cron_cli` covers: set with flags, show with TZ display, list across workspace.
4. `creator works cron set my-work --brainstorm "0 3,9,15,21 * * *" --tz Asia/Shanghai` round-trips through DAO and re-renders correctly on `show`.
5. Invalid cron expression returns `ValidationError` with stable error code; invalid TZ returns `ValidationError`.

### Design decisions (Think Before Coding)

1. **CLI opens local `state.db` directly** (not via daemon HTTP) ‚Äî the plan's code-touch list excludes daemon handler files; the CLI already depends on `nexus-local-db` with precedent (`db::Schema::init`, `soul::open_global_db`). The daemon's cron *firing* (T-A P1) reads the same `works.schedule_json` column. This keeps the foundation surgical while making `set/show/list` functional at runtime.
2. **Dedicated DAO functions** (`set/get_schedule_json` on the column only) rather than extending `WorkRecord`/`WorkPatch` ‚Äî the latter would touch the daemon PATCH handler and rebalance the 22-column INSERT value lists = out of scope for foundation.
3. **5-field cron normalization** ‚Äî the `cron` crate (zslayton) requires ‚â•6 fields (seconds first); spec uses 5-field crontab expressions, so `validate_cron_expr` prepends `0 ` (seconds=0) for 5-field input, preserving semantics.
4. **"rollback" test** = `ALTER TABLE works DROP COLUMN` simulation ‚Äî sqlx::migrate in this repo is forward-only (no `.down.sql` convention); DROP COLUMN (SQLite 3.35+) is the canonical down-migration for an added column.
