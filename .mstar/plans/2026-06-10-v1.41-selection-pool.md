## 1. Problem Statement

Authors with multiple novel ideas need **选题库** (active / queued / completed) and **灵感池** tracking. OSS Work is single-Work scoped; pool state must live **outside** Work rows as creator-scoped DB SSOT with optional markdown mirrors for inspiration.

---

## 2. Goals

1. **`novel_pool_entries`** landed in P0 — P1 extends DAO for promote/archive/list flows beyond `works use`.
2. Add `inspiration_items` table: `item_id`, `creator_id`, `rel_path`, `title`, `status`, optional `promoted_work_id`.
3. Scaffold `Works/_pool/灵感池/<slug>.md` on `inspiration add`; DB holds canonical path reference.
4. CLI: `creator works pool list|promote|archive|inspiration add|list|promote`.
5. **`promote`**: prior `active` → `queued`; promoted → `active`; bind/scaffold `work_id` if missing.
6. **`inspiration promote`**: read MD → `run start --idea`; pool `queued` row; item → `promoted`.
6. Wire completion hook from P0 to update pool row + clear `active`.
7. Optional export of pool to markdown (DB wins on conflict).
8. Hermetic tests for CRUD + promote + default pointer.

---

## 3. Non-goals

- `work_profile: novel-pool` as a Work type (pool is meta-layer, not Work).
- `creator work switch` / global switch mutex.
- Full Obsidian bidirectional sync.
- Multi-volume PK (V1.42).

---

## 4. Acceptance Criteria

1. Exactly one `active` pool entry per creator (enforced at DB or service layer).
2. `inspiration add` creates MD file + DB row atomically.
3. `promote --set-default` sets pool active without pausing other Works.
4. `list` output matches DB regardless of stale markdown.
5. Distinct from per-Work `inspiration_log` (documented in help).
6. Hermetic API + CLI tests.
