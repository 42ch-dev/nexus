### 🟡 Warning

- **W-001 — Work create idempotency is not atomic and ignores the idempotency write failure.** `create_work` checks `works_idempotency`, inserts into `works`, then calls `record_idempotency` and discards the result. A concurrent duplicate `client_request_id` can create two `works` rows, and any idempotency-table failure still returns `201 Created` for a Work that is no longer repeatable by the original key. This breaks the P1 acceptance goal of idempotent create and creates a fragile table boundary for P2/P3 follow-up flows. Recommended fix: make create + idempotency mapping one transaction, use `INSERT ... ON CONFLICT`/unique handling around `(creator_id, client_request_id)`, and return the canonical Work from the same transaction. Suggested residual severity if deferred: `medium` (or `high` if PM treats idempotent create as iteration-blocking).

- **W-002 — `nexus-local-db::works` uses runtime SQL for static DML, bypassing the repo's compile-time SQLx contract.** The new repository uses `sqlx::query` / `query_as` for static `INSERT`, `SELECT`, `UPDATE`, and `DELETE`-like DML with comments saying sqlx prepare has not run yet. This conflicts with `crates/nexus-local-db/AGENTS.md`, `crates/nexus-daemon-runtime/AGENTS.md`, and the T1 contract line that DML should use compile-time-checked SQLx. It makes the Work entity more prone to silent drift from migrations and blocks the normal `.sqlx/` review trail. Recommended fix: convert static DML to `query!` / `query_as!`, keep only truly dynamic `list_works` / `patch_work` as runtime SQL with bounded generated clauses, and run/commit `cargo sqlx prepare` metadata. Suggested residual severity if deferred: `medium`.

- **W-003 — `run_intents` validation is advisory and does not enforce the locked contract.** The plan requires creator presets to have at least one allowed creator-run intent (`work_init` or `work_continue`) as an `ERROR`, and calls for a distinct invalid-intent diagnostic category/test coverage. The implementation only warns when `run_intents` is empty and accepts creator presets whose intents are non-empty but not usable by `creator run` (e.g. `knowledge_ingest` / `work_maintenance`). This weakens the loader as the SSOT for P2's intake preset and future command routing. Recommended fix: encode the intended policy explicitly (or update the contract if knowledge/maintenance creator presets are intentional exceptions), add unit tests for empty creator intents and non-run creator intents, and ensure strict validation behavior is documented. Suggested residual severity if deferred: `medium`.

- **W-004 — Work API response shape drifts from the T1 contract and exposes DB JSON columns as strings.** The contract's `WorkRecord` response shape models `creative_brief` as JSON, `inspiration_log` as `Vec<InspirationLogEntry>`, and `schedule_ids` as `Vec<String>`, but the API returns `nexus_local_db::WorkRecord` directly, where all three JSON-valued columns are serialized as raw strings. This creates a Work entity SSOT split between DDL, repo struct, API contract, and CLI `serde_json::Value` consumers. P2/P4 clients will either parse nested JSON manually or accidentally rely on stringified JSON. Recommended fix: introduce an API DTO conversion layer that parses JSON columns into the contract shape, or amend the T1 contract and CLI output contract to state that these fields are stringified JSON. Suggested residual severity if deferred: `medium`.

### 🟢 Suggestion

- **S-001 — Keep `system preset` in its own module to preserve CLI module boundaries.** The T1 contract names `crates/nexus42/src/commands/system/preset.rs` and asks to mirror `system db` / `creator memory` patterns, but the implementation places command types, handler logic, HTTP calls, and tests directly in `commands/system/mod.rs`. It passes compile checks, but increases `mod.rs` responsibility and makes future preset subcommands harder to maintain. Suggested residual severity: `low`.

- **S-002 — Expand the test suite to lock the new architecture boundaries, not only command parsing.** `command_surface_contract` passes, and local DB unit tests cover basic CRUD, but the plan's explicit T6 API-contract and run_intents validation tests are not present as standalone daemon-runtime/API and orchestration validation cases. Adding them would catch W-001/W-003/W-004 regressions directly and reduce reliance on manual QC. Suggested residual severity: `low`.
