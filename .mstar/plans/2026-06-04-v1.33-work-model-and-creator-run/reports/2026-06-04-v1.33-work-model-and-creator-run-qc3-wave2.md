### 🟢 Suggestion

#### S-001: Document scale threshold for `append_inspiration` full-column rewrite
- **Observation**: The `append_inspiration` fix (R-V133P1-04) replaced fragile `substr/CASE` SQL with a transaction-wrapped Rust append + full `UPDATE`. This rewrites the entire `inspiration_log` column on every append. For the pre-release volume expectation (<100 entries per Work), this is negligible — a 100-entry JSON array is perhaps 5–10 KB and SQLite handles it comfortably.
- **Future concern**: If a single Work accumulates >1,000 inspiration entries, the column rewrite becomes measurable (each append copies the full JSON text). The current code has no documented scale threshold or migration path.
- **Recommendation**: Add a one-line comment in `append_inspiration` noting the rewrite behavior and a scale threshold (e.g., "acceptable for <1,000 entries; migrate to normalized `work_inspirations` table if scale exceeds this"). Not blocking for V1.33.

## Performance & Reliability Assessment

### Transaction overhead (R-V133P1-01)
For V1.33 single-process SQLite, `BEGIN ... COMMIT` adds ~1–2 ms per create call. The idempotency SELECT hits an indexed column. Acceptable for pre-1.0; no scalability concern at current volumes.

### Full-column rewrite (R-V133P1-04)
Each `append_inspiration` rewrites the entire `inspiration_log` JSON text. At <100 entries (pre-release expectation), the column is <10 KB and SQLite rewrites are sub-millisecond. Suggestion S-001 flags a future threshold (>1,000 entries) for normalized table migration. Not a V1.33 blocker.

### DTO conversion (R-V133P1-10)
`WorkApiDto::from(WorkRecord)` is a compile-time struct expansion — no heap allocation, no JSON parsing. `list_works` uses the even-smaller `WorkSummary` shape. Negligible in hot path.

### Error-path branching (R-V133P1-06)
The `is_connect() || is_timeout()` check only runs when `reqwest::Error` is converted to `CliError` — i.e., on failure. Common (success) path is untouched. Zero overhead in happy path.

### Test determinism
