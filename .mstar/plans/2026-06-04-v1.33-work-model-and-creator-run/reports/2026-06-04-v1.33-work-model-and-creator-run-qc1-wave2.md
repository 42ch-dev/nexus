### 🟡 Warning

- **W-001 — The required `run_intents_validation` integration test target is missing, and the fix remains semantically partial vs the locked T1 contract.** The assigned command `cargo test -p nexus-orchestration --test run_intents_validation` fails with exit `101` because no such integration-test target exists. The fix placed three unit tests in `validation.rs` instead, despite T1 §7.5 explicitly listing `crates/nexus-orchestration/tests/run_intents_validation.rs`. More importantly, `check_run_intents` now only errors on an empty creator `run_intents` list; it still allows a `kind: creator` preset with only `knowledge_ingest` or `work_maintenance`, even though T1 §7.4 says every creator preset must have at least one of `work_init` or `work_continue`. System presets claiming non-system work intents also remain a Warning rather than a hard contract failure. Recommended fix: add the required integration test target and tighten `check_run_intents` to enforce the exact contract predicates.

- **W-002 — `create_work_atomic` closes the major race, but the transaction boundary is awkward and should be simplified before this becomes the repository pattern.** The new `create_work_atomic` fits the local-db layer better than the old handler-level SELECT/INSERT/record flow, and it no longer leaks transaction details to daemon handlers. However, its `Result<Result<WorkRecord, WorkRecord>, LocalDbError>` return shape encodes `Ok(existing)` vs `Err(new)` in a way that is hard to read and easy to misuse. The replay path also performs `get_work(pool, ...)` on the pool while the transaction is still open, then rolls the transaction back, and silently falls back to `record.clone()` if the idempotency row points at a missing work. Recommended fix: introduce a small domain enum such as `CreateWorkOutcome::{Created, Replayed}` and fetch the replayed row inside the same transaction (or fail loudly on a dangling mapping). This is not a new Critical because the original non-atomic handler leak is fixed.

### 🟢 Suggestion

- **S-001 — `works_api.rs` is useful coverage, but its helper setup should be tightened for maintainability.** The file is a reasonable first dedicated API test target and the creator-isolation tests are present. The test organization would be easier to maintain if repeated no-creator/other-creator setup used one explicit fixture owner rather than `std::mem::forget(tmp)` plus an unrelated `_tmp` holder. Also, several route tests are direct handler invocations because of an `axum-test` path limitation; the module comments document this, but future route-level coverage should revisit it once the helper can represent `wrk_*` path values cleanly.

- **S-002 — `WorkApiDto` is in the right layer for P1, but P2/P4 should decide whether JSON columns remain strings at the API boundary.** Keeping the DTO in the handler is acceptable for this local-only P1 surface, because it prevents leaking DB-only owner fields and avoids a premature shared contract module. The conversion is currently a field copy, so P2 creative brief and P4 inspiration patches may duplicate parsing policy unless the next plan either parses `creative_brief`/`inspiration_log`/`schedule_ids` into structured values here or explicitly locks the API contract to stringified JSON.
