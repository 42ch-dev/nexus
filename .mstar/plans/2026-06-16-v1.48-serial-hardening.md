## 7. Baseline Review Notes (T1)

**Gaps relative to spec §4.5.7 #4–#5 and §4.5.3:**

1. `reconcile_from_filesystem` (`crates/nexus-local-db/src/work_chapters.rs`) currently treats `Stories/` frontmatter `status` as authoritative for existing rows: when file `status` differs from the DB row, it **overwrites** the DB status. This violates novel-workflow-profile §4.5.3 (“`work_chapters` is the queryable source of truth … the DB row wins”).
2. The reconcile routine does not re-sync the file frontmatter back to the DB status, so prompt-visible state can remain stale after reconcile.
3. Resume selection (`next_chapter` / `next_chapter_volume_aware`) is already correct: it returns the lowest active chapter including `draft`, so a single `draft` row is resumed. The missing coverage is a hermetic acceptance test that asserts no duplicate row is created and the same row is selected.
4. CLI `creator works reconcile-chapters <work_id>` and the daemon `POST /v1/local/works/{work_id}/reconcile-chapters` handler delegate fully to `reconcile_from_filesystem`; the gap is therefore in the DB-layer routine, not the command wiring.
