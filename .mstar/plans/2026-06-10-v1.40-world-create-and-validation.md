## 1. Problem Statement

`novel-project-init` grill-me offers "create new World" but init path is not atomically bound to world create. `world_refs` validation is spec-only. CLI flags diverge from spec (`--title` vs `--name`). `creator world show` is missing. World-bound users lack structured `world_refs` gate behavior.

---

## 2. Goals

1. **Align** `creator world` CLI with spec (add `--name` alias or update spec to `--title`; document narrative kind default).
2. Add `creator world show <world_id>` (reuse `SqliteNarrativeGateway` / existing list row shape).
3. Wire init grill-me "create new" path to existing `create_world` (atomic with scaffold transaction).
4. Reuse / extend `novel_scaffold` `world_id` FK validation on Work PATCH/create; structured remediation.
5. Implement `world_refs` validator: required for World-bound Works, warnings at outline, errors at finalize.
6. Enforce mandatory World binding for new V1.40 Works; preserve legacy V1.39-and-earlier worldless Work read behavior unchanged.

---

## 3. Non-goals

- Rewriting `create_world` / narrative persistence (already in `nexus-local-db`).
- Full taxonomy (P1), prompt context block (P2), kb-extract binding (P3).
- Multi-volume PK, platform publish, multi-work switch.

---

## 4. Acceptance Criteria

1. Existing `creator world create` + `list` continue to work; `show` added.
2. Init "create new World" binds `world_id` or fails closed (no partial scaffold).
3. PATCH with invalid `world_id` returns remediation pointing to `creator world create` or `list`.
4. World-bound outline with invalid `world_refs` emits warnings; finalize blocks with error unless `--force`.
5. V1.40 Works: `world_id` is mandatory at creation/init. Legacy V1.39-and-earlier worldless Works continue to function read-only; new Works without `world_id` fail closed.
6. Hermetic tests cover valid/invalid `world_id` and `world_refs` paths.

## 6.1 Implementation Notes â€” mandatory World binding amendment

1. New `world_id` is required at `Works/<work_ref>/` creation time (P0 T3 path). `novel-project-init` must fail closed before scaffold side effects if no valid `world_id` can be produced.
2. The `create_world` scaffold path now runs on **all** init paths as World-binding resolution: create-new invokes world create; bind-existing validates the selected `world_id` from `creator world list` before scaffold.
3. Worldless Works from V1.39 and earlier remain legacy-compatible: existing code may read `world_id == NULL` and continue to operate; new V1.40 creation/init code rejects `world_id == NULL` only at creation time.
4. `world_refs` becomes required (not warn-only) for World-bound Works. Outline-time invalid ids may warn with remediation; finalize-stage gate blocks invalid/missing required refs unless `--force` is used with audit reason.
