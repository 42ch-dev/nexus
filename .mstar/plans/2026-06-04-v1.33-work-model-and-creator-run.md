## 1. Problem Statement

Users cannot run a complete narrative cycle without knowing `daemon schedule`, preset IDs, and seed strings. There is no **Work** container for long-term goals, structured briefs, or inspiration supplements. `system preset list` only shows `_system.*` presets, hiding `novel-writing` and other embedded assets.

---

## 2. Goals

### A1 — Work persistence

1. Add `works` table (or equivalent) to per-workspace `state.db` with fields per work-experience-model §3.
2. Local API: `POST/GET /v1/local/works`, `GET /v1/local/works/{work_id}`, `PATCH` for brief/intake/log.
3. Idempotent create; stable `work_id` generation.

### A2 — `creator run` CLI

1. Subcommand group under `creator`: `start`, `continue`, `list`, `status`.
2. `start` creates Work + schedules intake (P2) + primary preset per policy.
3. `continue` appends inspiration and merges into schedule `core_context`.
4. Register commands in `command_surface_contract` tests.

### A3 — Schedule ↔ Work linkage

1. Schedule metadata field `work_id` when created via `creator run`.
2. `daemon schedule list/inspect` surfaces `work_id` when present.

### A4 — `system preset` CLI

1. `system preset list` — all embedded + user presets via `GET /v1/local/presets`.
2. `system preset validate <path>` — wraps `POST /v1/local/presets:validate` or direct facade call.
3. Display `run_intents` in list output.

### A5 — `run_intents` in loader

1. Extend `preset.yaml` schema with `run_intents: []`.
2. Loader validation: closed enum, at least one intent per creator preset.
3. Tag all embedded presets per work-experience-model §5.2.
