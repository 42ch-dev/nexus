## §1 Overview

Implement LLM fallback path for the challenge solver when the pure logic pipeline fails. Currently, `solve_challenge()` returns `Err(ChallengeError::ParseError)` on failure with no fallback. The frozen spec §3.5 defines an LLM fallback: invoke agent runtime LLM with the challenge-solver skill file as system prompt.

**Compass**: [v1.8-delivery-compass-v1.md](.mstar/iterations/v1.8-delivery-compass-v1.md)
**Frozen spec**: `creator-challenge-solver-v1.md` §3.5 (platform v1-spec, via local-paths)

**Effort (agent-oriented)**: M (~1 agent session)
**Dependencies**: None — can begin immediately.

---

## §2 Scope

### In scope

- Modify `solve_challenge()` in `crates/nexus42/src/challenge/mod.rs` to catch `ChallengeError::ParseError` and invoke LLM fallback.
- LLM invocation: use existing agent runtime infrastructure to send challenge text + skill file content (`challenge-solver-skill.md`) as system prompt.
- Return LLM response as numeric answer.
- Add unit tests for fallback path (mocked LLM response).
- LLM timeout / error: fall through gracefully, returning original parse error.

### Out of scope

- Challenge difficulty changes.
- Multi-language challenges.
- LLM provider selection logic.
