## 1. Problem Statement

`nexus-kb` and ingest paths do not validate novel-profile `body` minimums. Spec §5.1.1 previously described a seven-value `block_type` enum that conflicts with shipped wire `BlockType`. Prompt context, `world_refs`, and `kb-extract` need one agreed model: wire enum + novel semantics in `body`.

---

## 2. Goals

1. Document and enforce wire `BlockType` at `nexus-kb` create/update (reuse `nexus_contracts::BlockType`).
2. Add optional `body.attributes.novel_category` validation for novel-profile items per §5.1.1 mapping table.
3. Stable `canonical_name` convention aligned with `world_refs` (e.g. `char_lin_xia`).
4. Align `kb-extract/prompts/extract.md` output schema with wire types + `novel_category` (not a parallel enum).
5. Hermetic tests per wire `block_type` + representative novel categories.

---

## 3. Non-goals

- Changing `schemas/common BlockType` enum (no codegen bump in V1.40).
- Prompt injection (P2), extract job wiring (P3).
- Per-Work `Worldbuilding/` subtree.

---

## 4. Acceptance Criteria

1. Invalid wire `block_type` fails with structured error.
2. Novel-profile ingest accepts minimum `body` per mapping table (e.g. `character` + `novel_category: character`).
3. `(world_id, block_type, canonical_name)` active uniqueness preserved.
4. `world_refs` from P0 resolve against stored `canonical_name` items.
5. `kb-extract` prompt JSON schema matches P1 contract (wire types).
