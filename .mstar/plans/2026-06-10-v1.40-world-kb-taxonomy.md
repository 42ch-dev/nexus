## 1. Problem Statement

V1.37 P2 defined seven narrative `block_type` categories in spec but `nexus-kb` and `kb_key_blocks` do not enforce the vocabulary or minimum `body_json` fields. Prompt context, `world_refs`, and `kb-extract` need a shared category SSOT. Legacy `kb-extract` prompts/capability still use PascalCase types (`Character`, `Ability`, …) — **delete** in P1 (no backward compat per grill-me).

---

## 2. Goals

1. Implement typed category enum (or validated `block_type` vocabulary) for: `foundation`, `background`, `character`, `location`, `society`, `rules`, `economy`.
2. Enforce minimum `body_json` fields per category at create/update.
3. Stable item `id` generation convention aligned with `world_refs` (e.g. `char_*`, `loc_*`).
4. Migration for any new constraints on `kb_key_blocks` (document in local-db-schema).
5. Hermetic tests for each category minimum viable payload.

---

## 3. Non-goals

- Prompt injection (P2), extract pipeline (P3).
- Platform World fork (PD-01).
- Per-Work `Worldbuilding/` subtree.

---

## 4. Acceptance Criteria

1. Creating a KeyBlock with invalid `block_type` fails with structured error.
2. Each of seven categories accepts minimum viable `body_json` per entity-scope-model §5.1.1.
3. `(world_id, block_type, canonical_name)` active uniqueness preserved.
4. `world_refs` ids from P0 resolve against taxonomy-stored items.
5. Migration is reversible and documented in plan notes.
