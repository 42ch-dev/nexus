### 🟢 Suggestion

- **S-001 (performance / future work)**: `try_platform_assemble` now unconditionally returns `None`
  for `LocalFirst` / `CloudEnhanced` modes. The caller path still records
  `guard.record_platform_result(false, "assemble unavailable")` and then re-runs `build_stage0_from_local`,
  which means the **two-stage fallback branch always rebuilds `Stage0Assembly` exactly once**, same as
  `LocalOnly` — net behavior is identical, with one fewer HTTP round trip (good).
  However, while the daemon proxy is retired, **all `LocalFirst`/`CloudEnhanced` invocations now
  silently degrade to Stage-0**. Consider, in a follow-up iteration:
  (a) emit a tracing `warn!` (not just `debug!`) on each fall-through so operators notice the
  cloud-stage gap; or (b) gate the warn behind a single per-process `Once` to avoid log spam in
  hot loops. Not in this plan's scope (KCA-002 explicitly defers the direct platform call).

- **S-002 (reliability surface area / future work)**: Daemon `kb` handlers and request DTOs
  (`ListKbEntriesQuery`, `AddKbEntryRequest`) have **no `scope` field**. Plan C2 acceptance reads
  "API accepts only work scope today; other scopes return explicit 'not implemented'". The current
  implementation enforces work-scope only by **omission** (no scope param exists; any client-supplied
  `?scope=world` is silently dropped by Axum's `Query<T>` deserializer). The new doc comments make
  the contract clear and the CLI explicitly maps `KbScope::World → world_scope_deferred()`, so the
  user-visible behavior is correct. As a follow-up, the daemon could either:
  (a) add an optional `scope: Option<String>` to the query/body DTOs and return HTTP 501 / 400
  for non-`work` values, or (b) keep the implicit contract but add a `#[serde(deny_unknown_fields)]`
  guard so an unexpected `scope` field at least surfaces a 400. Either tightens the boundary at
  near-zero perf cost. Not blocking for V1.24.

- **S-003 (test coverage / future work)**: The new
  `resolve_kb_paths_uses_work_scope_layout` test is a welcome reliability invariant for the
  work-scope path layout. A symmetric CLI-side test (e.g. asserting `kb_list(scope=World)` prints
  the deferred message and does not touch the filesystem or the daemon) would lock the
  CLI-side world-scope short-circuit behavior. The current change only edits prose around the
  function and adds five copies of the same `if scope == &KbScope::World { world_scope_deferred(); return Ok(()); }`
  guard — a unit test would cheaply prevent a future regression where one of the five guards
  is removed.

## Performance & Reliability Assessment (deep dive)

### 1. Daemon proxy removal (KCA-002 B2)
- **Hot-path cost change**: Two-stage `assemble_context` previously issued one `POST /v1/local/context/assemble`
  per call (TCP + TLS via `DaemonClient::send_authenticated`, JSON serialize ~hundreds of bytes,
  parse ~kilobytes response). Now zero IO before falling back to Stage-0. **Improves** p95 latency
  and removes a daemon dependency from the read-only context path. **No regression** identified.
- **Resource lifecycle**: `DaemonClient::call_assemble` and `ContextClient::assemble` are removed
  cleanly; no dangling `reqwest::Client` pools, no orphaned Tokio tasks, no leaked auth tokens.
  The `nexus42::context::client` module is removed and unlinked from `mod.rs`.
- **Error handling**: Previous code branched on HTTP 503 → `None` (fallback) and HTTP 403 →
  `PlatformOperationProhibited`. The 403 path (local-only mode prohibition) is no longer reachable
  because the request itself is never sent. **This is correct** under KCA-002 B2 — local-only mode
  by definition does not want to touch the platform, so silently producing Stage-0 is the spec'd
  behavior. The `RuntimeGuard::record_platform_result` accounting still records `false` on the
  cloud-enhanced/local-first branches, which preserves observability counters.
- **Fallback semantics**: The fallback path is **deterministic and unbounded-safe** (Stage-0 reads
  bounded local files — SOUL.md, listed memories, fragments). No retry loops, no exponential backoff
  needed because no remote call occurs.

### 2. Work-scope KB handler (KCA-003 C2)
- **Performance**: Code change is **doc-only** in handlers plus one new unit test. Hot paths
  (`list_entries`, `add_entry`, `get_entry`, `delete_entry`) are byte-identical — confirmed by
  `git diff` showing only `///` and `//` lines changed in the handler bodies. The
  `KB_ENTRY_INDEX` `LazyLock<Mutex<HashMap>>` cache from prior W-005 work is preserved.
  **No perf delta.**
- **Reliability**: New unit test `resolve_kb_paths_uses_work_scope_layout` adds a static invariant
  on the work-scope path prefix. This is a **net positive reliability guard** against future
  refactors that might point KB writes into a non-work-scope path. Test runs in microseconds,
  no IO.
- **Concurrency**: `KB_ENTRY_INDEX` mutex semantics unchanged. No new locks added.

### 3. CLI `creator kb` scope guards
- Five guard sites (`kb_list`, `kb_search`, `kb_show`, `kb_add`, `kb_remove`) early-return on
  `KbScope::World`. Pattern is **trivially correct** and short-circuits before any daemon HTTP
  health-check or filesystem read — meaning a `--scope world` invocation costs O(1) work and
  one `println!`. No degradation.
