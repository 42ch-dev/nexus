### 🟢 Suggestion

#### S1 — Dead legacy code accumulation in `system.rs` (test-only)
- **Location**: `crates/nexus42/src/commands/system.rs` lines 87–147
- **Issue**: `SystemPresetCommand` enum and `run_legacy` function are no longer reachable from production code (`main.rs` now routes through `SystemCommand`). They exist solely for two unit tests (`system_preset_list_parses`, `system_preset_subcommand_required`). The `run_legacy` function carries `#[allow(dead_code)]` to suppress the compiler warning.
- **Impact**: Low today, but during the next wave of CLI restructuring (Plan 4) this dead code will confuse reviewers about which enum is authoritative. It also bloats the compile unit slightly.
- **Fix**: Either (a) move `SystemPresetCommand`, `SystemPresetCli`, and `run_legacy` into a `#[cfg(test)]` block, or (b) remove them entirely and update the two unit tests to parse `SystemCommand::Preset` directly.

#### S2 — `current_state_system_subcommands` no longer verifies absence of unexpected subcommands
- **Location**: `crates/nexus42/tests/command_surface_contract.rs` lines 219–228
- **Issue**: The old test explicitly asserted that `version` was **not** a `system` subcommand (to prevent premature leaks). After the rewrite, the assertion was flipped to check presence, but the corresponding negative checks for other commands (e.g., `clone`, `world`, `init`) were not added. The comment says "Verify no unexpected subcommands" yet only one positive assertion remains.
- **Impact**: If a future refactor accidentally adds a top-level command (e.g., `schedule`) as a `system` subcommand, the test will not catch it.
- **Fix**: Add assertions that the `commands_section` does **not** contain known non-system commands such as `clone`, `world`, `init`, `acp`, etc.

#### S3 — `SystemCommand::Completion` accepts unvalidated shell string
- **Location**: `crates/nexus42/src/commands/system.rs` lines 45–48
- **Issue**: The `Completion` variant takes a bare `shell: String` positional argument without validation or shell enumeration. The stub `print_completion_help` prints a generic message for any input. When `clap_complete` is eventually integrated, an invalid shell name will likely need to be rejected at parse time.
- **Impact**: Low — stub behavior; no runtime failure. But it defers validation design to the future implementer.
- **Fix**: Add a TODO comment referencing the future `clap_complete` integration, or introduce a small `Shell` enum with `ValueEnum` so clap rejects invalid values early.
