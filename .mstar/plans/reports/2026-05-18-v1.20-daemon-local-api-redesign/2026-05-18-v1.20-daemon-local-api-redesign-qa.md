## QC Warning Fix Verification

### W-001: SSE broadcast capacity (QC3)

**Issue**: `EVENT_BROADCAST_CAPACITY = 256` shared across all sessions, risk of silent event loss.

**Fix**: Increased to 1024.

**Verification**: ✅ Verified
- `manager.rs:33-37` — `const EVENT_BROADCAST_CAPACITY: usize = 1024;`
- Comment explains rationale: "higher capacity gives each subscriber more headroom before `RecvError::Lagged`"

---

### W-002: SSE zombie connections (QC3)

**Issue**: SSE stream never terminates, consumes memory after session ends.

**Fix**: Terminate stream after yielding `SessionStopped` event.

**Verification**: ✅ Verified
- `agent_host.rs:461-481` — unfold state includes `done` flag
- `is_terminal = matches!(HostEvent::SessionStopped(e) if e.session_id == sid)`
- Returns `(rx, is_terminal)` as next state; `done = true` triggers `return None`

---

### W-003: Blocking execute_operation (QC3)

**Issue**: Handler drains full stream before returning, causes HTTP timeout for long operations.

**Fix**: Return immediately (fire-and-forget), spawn background task to drain stream.

**Verification**: ✅ Verified
- `agent_host.rs:397-418` — `tokio::spawn(async move { ... })` drains stream in background
- Response returns `status: "started"` immediately
- Comment: "This prevents blocking the HTTP handler for the duration of long-running operations"

---

### W-004: Workspace scan O(n) per request (QC3)

**Issue**: `scan_workspaces()` traverses filesystem on every paginated request.

**Fix**: 60-second TTL cache, invalidated on create.

**Verification**: ✅ Verified
- `workspaces.rs` diff shows `WORKSPACE_CACHE_TTL_SECS = 60`
- `WORKSPACE_CACHE: LazyLock<Mutex<Option<WorkspaceCache>>>` stores `(Instant, Vec<WorkspaceSummary>)`
- `get_cached_workspaces()` checks TTL before returning
- `invalidate_workspace_cache()` called after `create_workspace()`

---

### W-005: KB entry O(n) lookup (QC3)

**Issue**: `get_entry`/`delete_entry` scan all creators/workspaces for entry ID.

**Fix**: Daemon-wide KB entry index (`entry_id → (creator_id, workspace_slug)`).

**Verification**: ✅ Verified
- `kb.rs` diff shows `KB_ENTRY_INDEX: LazyLock<Mutex<Option<EntryLocationMap>>>`
- `rebuild_kb_entry_index()` scans all `index.json` files on first access
- `lookup_entry_location()` provides O(1) hash lookup
- `invalidate_kb_entry_index()` called after add/delete
- Fast path reads entry from known location; slow path (stale index) falls through to filesystem scan

---

### W-006: KB add_entry crash-consistency gap (QC3)

**Issue**: Entry content written directly to `{id}.md`, not atomically. Crash between steps 2-3 leaves orphan file.

**Fix**: Write content to temp file first, atomic rename after index commit.

**Verification**: ✅ Verified
- `kb.rs:439-452` — `tmp_content_path = entries_dir.join(format!("{entry_id}.md.tmp"))`
- Write to temp → atomic index rename → atomic content rename
- Sequence: temp content → temp index → atomic index rename → atomic content rename
- Comment: "QC3 W-006: crash-consistent write sequence"
