### 🟡 Warning

**W-001: New V2 command paths print legacy deprecation notices**

- Location: `crates/nexus42/src/commands/acp.rs:219-221`, `crates/nexus42/src/commands/platform.rs:50-52`, `crates/nexus42/src/commands/system.rs:121`; notices in `session.rs:58`, `policy.rs:50`, `permission.rs:77`, `explore.rs:100`, `doctor.rs:48`.
- Issue: The deprecation notices were added inside shared legacy command modules. The new V2 entry points delegate to those same modules, so non-deprecated commands emit legacy deprecation text. Confirmed at runtime: `nexus42 acp session list` prints `Note: \`nexus42 session\` is deprecated...`, and `nexus42 system doctor check` prints `Note: \`nexus42 doctor\` is deprecated...`.
- Impact: This is a backward-compatibility and correctness regression for the promoted command surface. It also makes `system doctor` look deprecated immediately after being introduced.
- Fix: Move deprecation notices to legacy top-level dispatch only, or split legacy shim entry points from V2 implementation functions (for example `run_with_notice` vs `run_internal`). Ensure `acp session/policy/permission`, `platform explore`, and `system doctor` do not emit legacy notices.
- Confidence: High.

**W-002: `system doctor` delegates to legacy `doctor` instead of implementing the planned combined diagnostics**

- Location: `crates/nexus42/src/commands/system.rs:39-43`, `crates/nexus42/src/commands/system.rs:121`, `crates/nexus42/src/commands/doctor.rs:47-51`.
- Issue: The plan requires `system doctor` to run a unified diagnostic that includes daemon connectivity, ACP registry reachability, and home directory health. The implementation only wraps the existing `DoctorCommand` and calls `super::doctor::run(...)`, which runs the legacy `nexus42 doctor` checks and prints the legacy title.
- Impact: Users invoking the new system-level diagnostic do not get the ACP registry check promised by the plan, and the command surface is not actually consolidated.
- Fix: Implement a `system::doctor` path that calls the intended checks directly (or exposes reusable no-notice helpers), including ACP registry reachability and home directory health. Keep the hidden top-level `doctor` command as the only place that emits the deprecation notice.
- Confidence: High.

**W-003: `acp run` does not explicitly validate `agent_ref` or sanitize `cwd` before spawning**

- Location: `crates/nexus42/src/commands/acp.rs:842-872`; lookup behavior in `crates/nexus-acp-host/src/registry.rs:424-439`.
- Issue: `cmd_run` passes raw `agent_ref` to `RegistryClient::find_agent`. `find_agent` treats an empty query as a prefix of every string, so an explicitly empty `agent_ref` can resolve to the first registry agent instead of failing validation. The `cwd` option is accepted as a raw `PathBuf` and passed to `AgentSpawner::new`; there is no canonicalization, existence check, directory check, or explicit policy boundary before `Command::current_dir` is used.
- Impact: The new primary execution path can select an unintended agent for empty input and can attempt subprocess launch from arbitrary or invalid paths. The latter may fail at spawn time, but the plan explicitly asked for validation/sanitization before subprocess spawn.
- Fix: Reject empty/whitespace-only `agent_ref` before registry lookup, consider ambiguity handling for partial matches, and validate `cwd` with `canonicalize`, `is_dir`, and any intended workspace/security boundary before constructing `AgentSpawner`.
- Confidence: High.

### 🟢 Suggestion

- Issue: The new contract tests assert only help-surface presence. They do not catch the V2-path notice regression above or verify that legacy paths still produce the same stdout plus the expected one-line stderr notice.
