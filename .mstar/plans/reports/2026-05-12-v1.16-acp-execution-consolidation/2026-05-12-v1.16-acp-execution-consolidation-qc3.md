### 🟡 Warning

**W-PR-001: Double `child.wait()` in single-shot mode creates brittle lifecycle semantics**

- **Location**: `crates/nexus42/src/commands/acp.rs:887`, `acp.rs:905`, `acp.rs:949`
- **Issue**: In single-shot mode (`message` is `Some`), `cmd_run` calls `wait_for_agent_exit(&mut child, ...)`, which internally awaits `child.wait()` (line 949). When `wait_for_agent_exit` returns `Ok(())`, the caller immediately awaits `child.wait()` *again* at line 905.
- **Impact**: Tokio caches the `ExitStatus` after the first `wait()`, so the second call returns the cached value and does not crash today. However, this is a brittle pattern:
  1. It assumes tokio's internal caching semantics, which are not part of the stable API contract.
  2. If error-path refactoring ever bypasses `wait_for_agent_exit`, the second `child.wait()` may be the *only* wait, creating an asymmetry.
  3. Confusing lifecycle management increases the risk of regressions in future subprocess-related refactors.
- **Fix**: Refactor `wait_for_agent_exit` to return `Result<std::process::ExitStatus, String>` (or `io::Result<ExitStatus>`) and use that status directly in the caller. Remove the redundant `child.wait()` in the `Ok` branch of `cmd_run`.
- **Confidence**: High

**W-PR-002: Timeout path does not terminate the orphaned child process**

- **Location**: `crates/nexus42/src/commands/acp.rs:942-969` (`wait_for_agent_exit`); `acp.rs:916-918` (caller error branch)
- **Issue**: If the agent subprocess exceeds the 5-minute timeout, `wait_for_agent_exit` returns `Err("Agent ... timed out after ...")`. The caller at line 916 prints the error string and returns `Ok(())`. **No attempt is made to kill the still-running child**.
- **Impact**: The child process becomes orphaned (or a zombie if it exits after the timeout future is dropped). In interactive or CI environments, this leaks processes and file descriptors. Over repeated invocations, this can exhaust PID limits or fill log directories.
- **Origin**: Pre-existing behavior inherited from `agent.rs` (moved, not introduced). However, because `acp run` is now the **primary** execution path, the impact surface has expanded from a hidden command to the canonical entry point.
- **Fix**: In the `Err(_)` timeout branch of `cmd_run`, attempt `let _ = child.kill().await;` before printing the error and returning.
- **Confidence**: High

**W-PR-003: `system doctor` lacks planned combined diagnostics and emits incorrect deprecation notice**

- **Location**: `crates/nexus42/src/commands/system.rs:121`; `crates/nexus42/src/commands/doctor.rs:48`
- **Issue**: The plan (T6) explicitly requires `system doctor` to run a unified diagnostic covering (1) daemon connectivity, (2) ACP registry reachability, and (3) home directory health. The implementation delegates directly to `doctor::run`, which:
  1. Prints `Note: \`nexus42 doctor\` is deprecated. Use \`nexus42 system doctor\` instead.` — even though the user **is** calling `system doctor`.
  2. Runs only the legacy 5 checks (daemon, config, database, workspace, version). No ACP registry reachability check is included.
- **Impact (Reliability)**: Users invoking the unified diagnostic miss ACP registry health. They must run `acp doctor` separately, defeating the consolidation goal. The incorrect stderr notice also breaks scripts that capture stderr.
- **Overlap**: This is the same root cause as QC2 W-002. From the performance/reliability angle, the extra command invocation (two diagnostics instead of one) adds unnecessary latency and user friction.
- **Fix**: Implement a `system::run_combined_doctor` that calls reusable check helpers directly (without the deprecation notice), including an ACP registry reachability probe. Keep the hidden `doctor` command as the sole deprecation-notice emitter.
- **Confidence**: High

### 🟢 Suggestion

**S-PR-001: Subprocess lifecycle coverage is absent for the primary `acp run` path**

- **Location**: `crates/nexus42/src/commands/acp.rs:842-1034`
- **Issue**: There are **zero** unit tests for `cmd_run`, `wait_for_agent_exit`, `setup_cancel_handler`, or `interactive_prompt_loop`. The only test coverage is 6 contract tests that verify help-text presence. The `agent.rs` shim tests (`agent_list_delegates_to_acp`, `agent_status_delegates_to_acp`, `agent_skills_delegates_to_acp`) do not exercise the `agent run` → `acp run` delegation path.
- **Impact**: Resource-lifecycle bugs (W-PR-001, W-PR-002, zombie processes, signal handling) have no automated regression protection.
- **Fix**:
  1. Add a test that mocks/spawns a fast-exiting dummy binary and asserts `cmd_run` returns `Ok(())` without double-wait errors.
  2. Add a test for the timeout branch that asserts the child is killed (or at least that the timeout error is surfaced).
  3. Add a test for `setup_cancel_handler` that verifies the spawned task does not panic when the sender is dropped.
- **Confidence**: High

**S-PR-002: `work_dir` fallback does not validate directory existence before spawn**

- **Location**: `crates/nexus42/src/commands/acp.rs:853`
- **Issue**: `let work_dir = cwd.unwrap_or_else(|| std::env::current_dir().unwrap_or_default());`
  - If `std::env::current_dir()` fails (directory deleted, permission denied), `work_dir` becomes an empty `PathBuf`.
  - `AgentSpawner::new(work_dir)` receives this without any `exists()` / `is_dir()` check.
  - The spawn failure is deferred to the OS, producing a generic error rather than a helpful validation message.
- **Impact**: Minor UX degradation on the error path. Not a security issue because `AgentSpawner` / `Command::current_dir` will fail safely.
- **Fix**: Add explicit validation: reject empty/invalid `work_dir` with a clear error before constructing `AgentSpawner`.
- **Confidence**: Medium

**S-PR-003: Binary size impact is neutral — forward-looking maintainability note**

- **Location**: `crates/nexus42/src/commands/acp.rs` (1246 lines post-change)
- **Issue**: None — positive observation. `cargo build --release` produces a binary of **~14 MB** on both `28a4903` and `HEAD`. The `cmd_run` code was relocated, not duplicated; no new crate dependencies were added.
- **Note**: `acp.rs` is now 1246 lines. If future plans add more subcommand groups, consider extracting submodules (e.g., `acp/run.rs`, `acp/session.rs`) to keep compile-unit sizes manageable. This aligns with QC1 S-001.
- **Confidence**: High

**Verdict**: **Request Changes**

The implementation correctly relocates code with no binary-size regression and passes lint/contract tests. However, three reliability issues block approval:

1. **W-PR-001** (double wait): Brittle subprocess lifecycle pattern that should be refactored to return `ExitStatus` from the helper.
2. **W-PR-002** (orphaned child on timeout): The primary `acp run` path can leak processes on timeout. This is inherited from the old `agent.rs` but now affects the canonical execution entry.
3. **W-PR-003** (incomplete `system doctor`): Missing the planned ACP registry check, and the incorrect deprecation notice on the V2 path degrades script reliability.

The suggestion **S-PR-001** (test coverage) is strongly recommended because the moved subprocess code has zero automated regression protection.
