### 🟢 Suggestion

1. **Mock provider duplication between test modules**
   - `MockGoProvider` (judge_llm.rs) and `MockSummaryProvider` (context_summarize.rs) are structurally identical: both capture `creator_id` via `std::sync::Mutex<String>`.
   - If the `WorkerHandleProvider` interface changes, both mocks require coordinated updates.
   - **Recommendation**: Consider extracting a generic `CapturingMockProvider` to a shared test utility module (e.g., `crates/nexus-orchestration/src/capability/test_helpers.rs`) to reduce maintenance burden. Low priority — test-only code.

2. **Implicit "default" fallback not explicitly asserted**
   - Existing tests `judge_llm_with_mock_worker_go` and `context_summarize_with_mock_worker` run without any identity fields, implicitly exercising the `unwrap_or("default")` fallback.
   - Neither test asserts that the captured `creator_id` equals `"default"`, so a regression that changed the fallback value would not be caught.
   - **Recommendation**: Add a one-line assertion in each test (or a dedicated test) verifying `"default"` is passed when no `_creator_id` is present. Low priority — the SEC-V131-01 tests already cover the security boundary comprehensively.

3. **Synchronous blake3 hash in async context (pre-existing)**
   - `context_summarize.rs:133` computes `blake3::hash(prompt.as_bytes())` synchronously before the async worker call.
   - For very large prompts, this blocks the async task. This is pre-existing code unchanged by this PR.
   - **Recommendation**: If prompt sizes grow significantly in production, consider offloading to a blocking task (`tokio::task::spawn_blocking`) or computing the hash asynchronously. Not actionable for this PR.

### Rationale

The PR closes SEC-V131-01 by removing `creator_id`/`session_id` from the public JSON Schema and only accepting context-injected `_creator_id`/`_session_id` fields. From a **performance and reliability** perspective:

- **No hot-path regressions**: The production `run()` methods change only the JSON key being looked up (`creator_id` → `_creator_id`, `session_id` → `_session_id`). No new allocations, no added blocking operations, no increased algorithmic complexity.
- **Graceful degradation**: Missing context-injected IDs fall back to `"default"` via `unwrap_or`, preventing panics. This preserves the pre-existing behavior.
- **Deterministic, fast tests**: All 6 new SEC-V131-01 regression tests and all 325 existing tests pass in under 0.5 seconds. Tests use in-memory mock providers (no real LLM calls, no real worker IPC, no network I/O).
- **No resource leaks**: The `std::sync::Mutex` in test mocks is never held across `.await` points; locks are acquired and released immediately within the synchronous mock body.
- **CI clean**: `cargo +nightly fmt`, `cargo clippy -D warnings`, and `cargo test --lib` all pass.

The three Suggestions are minor test-quality or pre-existing observations with no blocking impact.
