## Detailed Security & Correctness Analysis (qc-specialist-2 focus)

### SEC-V131-01 Closure Assessment
- **Gap closed**: Both `judge.llm` and `context.summarize` no longer declare or read raw `creator_id` / `session_id` from `input` for the values passed to `call_acp_prompt(...)`.
- **New behavior**:
  - `input_schema` for both now omits the raw fields (matching acp_prompt.rs exactly; only "prompt" or "content" + optional trace/template are user/preset-visible).
  - Extraction uses `.get("_creator_id")` / `.get("_session_id")` with clear security comments citing SEC-V131-01 and "prevents cross-creator IPC routing (IDOR)".
  - Raw values supplied in input are silently ignored (fall back to "default"); this is the intended "defense-in-depth" behavior per plan (orchestration engine is the only legitimate injector).
- **Three regression tests added per capability** (exactly as required by plan C2 and compass risk note):
  1. `*_raw_creator_id_ignored_on_spoof_attempt` — raw "spoofed_*" present → captured by mock = "default" (asserted).
  2. `*_context_injected_identity_trusted` — only _* present → captured = "legit_*".
  3. `*_context_identity_overrides_raw_spoof` — both present → _* wins (the critical "context wins" case).
- All 6 new SEC tests + pre-existing tests pass cleanly.
- No change to the actual worker dispatch or prompt content.

### Attack Surface & Trust Boundary
- **Before**: Preset YAML or schedule --var could supply `creator_id` / `session_id` → directly used as the identity for `acp.prompt` IPC call → potential for a malicious/compromised preset or schedule to cause cross-creator action under wrong identity (even if low-probability because most calls are internal).
- **After**: Only values injected by the orchestration engine (via `session.context.set("_session_id", ...)` and equivalent creator wiring, visible in engine.rs) are ever forwarded. The three capabilities now enforce the same boundary as the already-hardened `acp.prompt`.
- **Context injection path itself**: Trusted (engine code path, not user-controlled). The new tests prove the *extraction* prefers the injected form even under adversarial input. No new surface opened.
- **Other uses of "creator_id" / "session_id" in the crate** (reviewed via rg):
  - `soul_experience_aggregate.rs`: Legitimately requires `creator_id` as a *business parameter* (the target creator whose memories to aggregate). This is **not** the execution identity for IPC routing. Schema and code unchanged and correct.
  - `acp_session_load.rs`: Uses `session_id` as its primary domain argument (which session to resume). Distinct semantic; not related to the IDOR vector.
  - Engine / schedule / graph_flow: Internal `SessionId`, `creator_id` on structs, and explicit `_session_id` injection into context. This is the *source of truth* for the secure path.
  - Test fixtures / multi_agent_worker.rs: Use `session_id` for ACP session management tests (expected).
  - Embedded presets (soul-experience-refresh, kb-extract): Pass `creator_id: "{{preset.input.creator_id}}"` to soul/kb aggregate capabilities — correct business usage, not execution identity.
  - No other builtins were found using raw creator/session for the `call_acp_prompt` identity path.
