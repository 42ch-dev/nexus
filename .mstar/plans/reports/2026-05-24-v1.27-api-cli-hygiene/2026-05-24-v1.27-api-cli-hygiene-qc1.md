**S-001**: `get_entry` and `delete_entry` handlers lack explicit scope validation
- **Detail**: The `list_entries` and `add_entry` handlers now validate `scope=work` with `validate_scope()`, returning 400 for non-work values. ### 2. KB scope validation (H3) — `crates/nexus-daemon-runtime/src/api/handlers/kb.rs`

**Architecture assessment**: ✅ Clean

- `validate_scope()` is a small, pure, well-documented private function with exhaustive match coverage (`None | Some("work")` → Ok, `Some(other)` → 400).
- Error uses `NexusApiError::InvalidInput` (400) with a descriptive message citing `entity-scope-model §5.3` and explaining that World KB and User knowledge have separate routes.
- `#[serde(default)]` on the new `scope` fields means existing clients (that don't send `scope`) continue to work — no breaking API change.
- Validation is placed at the top of `list_entries` and `add_entry`, failing fast before any filesystem or DB operations.
- Tests cover all branches: `None`, `"work"`, `"world"`, `"user"` (unknown). The `"world"` test asserts the error message contains both `"entity-scope-model"` and `"work-scope file index only"`.

**Maintainability note**: The validation function could be reused if future handlers (e.g., bulk operations) also accept `scope`. Its placement in a private module-level function is correct.
