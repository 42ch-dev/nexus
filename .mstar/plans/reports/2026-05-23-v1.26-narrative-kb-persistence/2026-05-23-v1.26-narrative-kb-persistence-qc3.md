### 🟡 Warning
- **W-001: `get_narrative_context` makes redundant `get_event` queries**
  Phase 3 of `SqliteNarrativeGateway::get_narrative_context` calls `get_event()` (up to 2x: once for the explicit `query.event_id`, once for the `timeline_position.current_event_id`). However, Phase 2's `get_timeline()` already returned all events for the branch — including the head event. The helper already resolves `current_event_id` from the in-memory timeline data but discards the full `TimelineEvent` struct; Phase 3 then re-queries the DB for data already present in memory. This adds up to 2 unnecessary DB round-trips per context load. While per-call overhead is low (indexed PK lookup), it adds up under frequent reads. → **Fix**: Return the resolved `TimelineEvent` from Phase 2 alongside `TimelinePosition`, reuse in Phase 3 instead of re-querying.

- **W-002: `KbStore::query()` loads all active blocks into memory then filters in Rust**
  `SqliteKbStore::query()` calls `list_by_world()` (fetch all non-deleted blocks for a world), then applies `block_type`, `canonical_name`, and `text_search` filters in-memory. The doc comments acknowledge this trade-off ("efficient for per-world datasets which are typically small"), but there is **no upper bound or pagination at the DB level**. If a world accumulates thousands of active key blocks (over years of use), every `query()` call will linearly scan and allocate all of them. → **Fix**: Document a soft cap / consider adding `LIMIT` in SQL and moving simple equality filters (block_type, canonical_name) to the SQL layer.

- **W-003: `get_narrative_context` silently swallows head event resolution errors**
  In the `else if let Some(ref head_id) = world_state.current_timeline_head_id` branch of Phase 2, `self.get_event(head_id).await.ok()` silently converts any DB error or missing head event into `None` timeline position. If `current_timeline_head_id` points to a non-existent or corrupted event, the caller receives an incomplete `NarrativeContext` without any indication that data integrity was compromised. The explicit `query.event_id` path in Phase 3 uses the same `.ok()` pattern. → **Fix**: Log a warning on resolution failure (`tracing::warn!`) so operators can detect dangling references.

- **W-004: Restart test uses seed helpers instead of public API for data insertion**
  `data_survives_simulated_restart` in `db/restart_tests.rs` seeds data via `narrative_gateway::seed::world` and `kb_store::seed::key_block` — direct `INSERT` statements — then verifies via the public `NarrativeGateway`/`KbStore` APIs. This tests that **raw SQL data** survives a pool close/reopen cycle, but does **not** verify that `WorkspaceState::init()` (the production daemon startup path) can successfully reconstruct the gateway and store from an existing database file. The test also uses `nexus_local_db::open_pool` (raw sqlx pool) rather than `DbPool::new()`, further diverging from production. → **Fix**: Add a higher-level integration test that calls `WorkspaceState::init()` → reads back via `narrative_gateway()` and verifies persisted data.

### 🟢 Suggestion
- **S-001: Test pool uses `max_connections(2)`, production defaults to 8**
  `WorkspaceState::new_for_testing()` sets `max_connections(2)`. Sequential tests won't contend, but the mismatch means the test suite never exercises production pool sizing behavior. Minor.

- **S-002: CLI `open_world_kb_store` opens a separate pool instance on the same DB file**
  Each `nexus42 creator kb ...` command calls `open_world_kb_store()` which opens a fresh pool via `Schema::init()`. When the daemon is concurrently running with its own pool, two pool instances access the same SQLite file. WAL mode handles this correctly (concurrent readers, serialized writes), but explicit documentation of this multi-pool coexistence would aid operational understanding.

- **S-003: No explicit `pool.close().await` in production daemon shutdown**
  `WorkspaceState` relies on `Drop` to close the `DbPool`, which triggers sqlx's pool shutdown. The restart test explicitly calls `pool.close().await`. For SQLite, `Drop` is sufficient (WAL checkpoint happens on close), but explicit close would be more defensive and consistent with the test pattern.
