## Warning

### W-1: Anonymous identity `active_creator_id` persists across sessions without identity existing (Medium)

**Location**: `crates/nexus42/src/commands/identity.rs` â€” `use_identity()` (line 183) and `create_identity()` (line 173)

**Observation**: When a user creates an anonymous identity, it is auto-set as `active_creator_id` in `CliConfig` without being stored in the database. On subsequent CLI invocations, `resolve_active_identity()` (line 245) detects that the `ctr_anon*` ID doesn't exist in the DB and reconstructs a minimal `ResolvedIdentity`. However, `use_identity()` allows users to set ANY `creator_id` (including a stale anonymous one) as active without checking if it actually exists in the database (line 186 only checks persistent identities via `get_local_identity`, which returns `None` for `ctr_anon*` IDs that were never persisted).

**Impact**: A user who creates an anonymous identity, exits, then runs `nexus42 identity use <stale_anon_id>` will set a non-existent identity as active. The resolution path handles this gracefully (line 248-257), but the UX is inconsistent â€” `use` succeeds even though the identity is gone.

**Recommendation**: `use_identity()` should either:
1. Reject anonymous IDs (`ctr_anon*`) since they cannot be persisted and will not survive a session, OR
2. Emit a warning when setting a non-database identity as active.

**Severity**: Medium â€” runtime behavior is graceful but UX is misleading.

### W-2: `link_to_platform` silently succeeds when identity does not exist (Medium)

**Location**: `crates/nexus-local-db/src/identity.rs` â€” `link_to_platform()` (line 111)

**Observation**: The SQL UPDATE uses `WHERE creator_id = ?2 AND platform_linked = 0`. When no rows match (identity doesn't exist), `affected == 0`. The error-handling block (line 122-136) then re-queries to distinguish "already linked" from "doesn't exist", but returns the same `QueryReturnedNoRows` error for both cases. Callers cannot distinguish the two failure modes.

**Impact**: `link_identity()` in `commands/identity.rs` (line 204) calls `link_to_platform` and only prints a success message. If the identity doesn't exist, the error bubbles up and is printed as a generic "local database error" â€” not helpful for debugging.

**Recommendation**: Return a distinct error variant (`LocalDbError::IdentityNotFound`) or include the reason in the error message so callers can give actionable feedback.

**Severity**: Medium â€” error messages are misleading but the operation fails safely.

### W-3: ID generation is UUID v4-based, not counter-based (Low-to-Medium)

**Location**: `crates/nexus-domain/src/local_identity.rs` â€” `generate_anonymous_id()`, `generate_local_id()` (lines 175, 188)

**Observation**: Both functions use `uuid::Uuid::new_v4()` for ID generation. The plan mentions `ctr_anon*` / `ctr_local*` as if they might be counter-derived, but the implementation uses UUID v4. This is collision-safe (UUID v4 has ~122 bits of entropy) but does not guarantee:
1. Deterministic ordering (IDs are random, not sequential)
2. A stable "next ID" for a given session

**Spec reference**: The plan does not explicitly require counter-based ID generation. The schema comment says "ctr_anon* for anonymous, ctr_local* for persistent" without specifying the suffix format. The `^ctr_[a-zA-Z0-9]+$` pattern in `is_valid_creator_id()` accepts both.

**Recommendation**: If sequential/counter IDs are needed for auditability or ordering, the implementation should switch to a counter stored in the database. If random UUIDs are acceptable, document this choice in the domain model.

**Severity**: Low-to-Medium â€” depends on whether sequential IDs are required by the spec.

## Suggestions

### S-1: Add integration test for ephemeral anonymous identity lifecycle

**Location**: `crates/nexus42/src/commands/identity.rs`

**Observation**: `create_identity()` auto-sets anonymous identities as active but does not persist them. There's no test that verifies an anonymous identity is NOT queryable after creation. A test should:
1. Create anonymous identity (auto-set as active)
2. Run `resolve_active_identity()` â€” should return the anonymous ID
3. Restart (simulate by clearing `active_creator_id` or reloading config) â€” identity should NOT be recoverable

**Severity**: Low â€” existing unit tests cover domain model behavior, but CLI-level ephemeral behavior is untested.

### S-2: Add CLI-level test for `resolve_active_identity` with stale platform ID

**Location**: `crates/nexus42/src/commands/identity.rs` â€” `resolve_active_identity()` (line 221)

**Observation**: The fallback at line 258-268 handles "might be a platform creator ID" but this path is untested. When `active_creator_id` contains a platform creator (e.g., from before the local identity feature existed), the resolution returns `identity_type: "platform"`. This path should have a test to ensure it doesn't panic or error.

**Severity**: Low â€” defensive code with no test coverage.

### S-3: Document `open_global_db` path in CLI vs daemon context

**Location**: `crates/nexus42/src/commands/identity.rs` â€” `open_global_db()` (line 70)

**Observation**: The function opens `~/.nexus42/state.db` directly, bypassing the ADR-014 workspace layout (`~/.nexus42/creators/<creator_id>/workspaces/<slug>/state.db`). This is intentional for global identity storage, but:
1. It's not documented WHY global identity is outside the per-creator workspace layout
2. The daemon likely needs access to the same identity store; if it uses a different path, identity state could diverge

**Recommendation**: Add a doc comment explaining this is intentional (global singletons vs per-creator workspaces) and verify the daemon uses the same path.

**Severity**: Low â€” design intent is clear from context but undocumented.

### S-5: `link_identity` has no domain-level validation before DB call

**Location**: `crates/nexus42/src/commands/identity.rs` â€” `link_identity()` (line 204)

**Observation**: The CLI command calls `link_to_platform()` directly without validating the `platform_creator_id` format against the CreatorId pattern (`^ctr_[a-zA-Z0-9]+$`). The DB layer accepts any string. The domain model has `is_valid_creator_id()` but it's not used here.

**Recommendation**: Validate `platform_id` format before calling `link_to_platform`:
```rust
if !nexus_domain::local_identity::is_valid_creator_id(&platform_id) {
    return Err(CliError::Other(format!("Invalid platform creator ID format: {}", platform_id)));
}
```

**Severity**: Low â€” malformed IDs will fail at the DB constraint or platform sync layer, not silently corrupt data.

---

### S-6: Consider adding a `delete` subcommand for identity management

**Location**: `crates/nexus42/src/commands/identity.rs` â€” `IdentityCommand` (line 19)

**Observation**: The CRUD set includes `list`, `create`, `use`, `link`, but not `delete`. The `delete_local_identity` function exists in `nexus-local-db` (line 146) but is not exposed via CLI. Users cannot delete identities they no longer need.

**Recommendation**: Add a `Delete { creator_id }` variant to `IdentityCommand` and implement `delete_identity()` that calls `delete_local_identity`.

**Severity**: Low â€” feature gap, not a bug.

- **Domain model is well-tested**: 14 unit tests covering ID generation, identity type conversion, platform linking, serialization roundtrips.

- **Generated code is consistent**: Rust and TypeScript contracts match the schema and each other.
- **No platform calls**: The identity module makes zero outbound network calls. All operations are local.
- **`link_to_platform` correctly prevents re-linking**: The SQL `WHERE platform_linked = 0` and the domain `link_to_platform()` both check `already_linked` state.
- **ADR-014 layout is respected for workspace state**: `~/.nexus42/state.db` is used for global identity storage (intentional singleton), while workspace state uses the ADR-014 per-creator path. The distinction is conceptually sound even if undocumented.
- **Migration system is idempotent and safe**: `migrate_v2` uses `CREATE TABLE IF NOT EXISTS` and the migration runner is version-guarded.
- **Error messages are user-friendly**: `use_identity()` gives actionable feedback when identity not found.
- **`resolve_active_identity` handles stale IDs gracefully**: Falls back to `platform` type for unknown IDs rather than erroring.
