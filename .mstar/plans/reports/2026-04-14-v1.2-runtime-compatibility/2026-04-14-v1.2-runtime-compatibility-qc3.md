- **[C-001] Snapshot restoration causes incorrect re-degradation**
  - **File**: `crates/nexus42/src/commands/context.rs:212-227`
  - **Source**: manual-reasoning
  - **Evidence**: `DegradationSnapshot::from_guard` captures `failure_count` BEFORE `degrade()` resets it to 0. When restoring a degraded snapshot, the for-loop replays `failure_count` failures, which pushes the count above threshold and triggers an UNINTENDED second degradation.
  - **Scenario**: Guard is Normal (failure_count=2), records 3rd failure â†’ degrades to DegradedLevel1, resets failure_count=0. If `from_guard` captures snapshot here (state=DegradedLevel1, failure_count=0) â†’ restore: loop doesn't run (0 iterations) â†’ CORRECT. BUT if from_guard captures while failure_count=3 before reset â†’ restore: loop runs 3 times, failure_count goes 1â†’2â†’3, exceeds threshold â†’ degrade to DegradedLevel2 â†’ WRONG (already at DegradedLevel1).
  - **Impact**: On restart, degraded guard may end up at a WORSE degradation level than saved. Compounding effect: subsequent failures accumulate on inflated failure_count.
  - **Fix**: In `create_degradation_guard`, skip replaying failures if `snap.state != DegradationState::Normal` OR capture `failure_count` AFTER the reset in `from_guard`.
