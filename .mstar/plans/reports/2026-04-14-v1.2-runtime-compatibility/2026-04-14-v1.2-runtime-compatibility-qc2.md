### 🟡 Warning

#### W-1: Daemon/CLI request shape mismatch breaks `context assemble` path
- **Location:** `crates/nexus42/src/context/client.rs:25-34`, `crates/nexus42d/src/api/mod.rs:103-111`, `crates/nexus42/src/api/daemon_client.rs:164-218`
- **Issue:** `ContextClient::assemble` sends `ContextAssembleRequestV1` (platform wire contract) to the daemon endpoint `POST /v1/local/context/assemble`. The daemon handler `assemble` (`crates/nexus42d/src/api/handlers/context.rs:25-67`) expects the internal `AssembleRequest` shape (`creator_id`, `workspace_slug`, `runtime_mode`, `prompt_hint`). Axum cannot deserialize `ContextAssembleRequestV1` into `AssembleRequest`, so the call returns HTTP 422.
- **Impact:** The user-facing `nexus42 context assemble --world-id <id>` command will always fail with a deserialization error from the daemon. The `call_assemble` method in `DaemonClient` was explicitly designed to send the correct `AssembleRequest` shape, but it is marked `#[allow(dead_code)]` and is never invoked.
- **Fix:** Wire `DaemonClient::call_assemble` into `try_platform_assemble` (or replace `ContextClient::assemble` with it) so the CLI and daemon speak the same request type. Alternatively, update the daemon to accept `ContextAssembleRequestV1` and map it internally.

#### W-2: `create_degradation_guard` fails to restore degraded state from snapshot
- **Location:** `crates/nexus42/src/commands/context.rs:212-227`
- **Issue:** When restoring from a persisted `DegradationSnapshot` with a non-Normal state, the code replays failures by calling `record_platform_result(false, ...)` in a loop. After a real degradation event, `failure_count` is reset to 0, so a snapshot capturing `state = DegradedLevel1` and `failure_count = 0` will loop zero times and leave the guard in `Normal` state. Additionally, each replayed call overwrites `last_failure_time` and `last_health_check` to `Utc::now()`, destroying the original timestamps.
- **Impact:** A persisted degradation snapshot is silently ignored on restart, causing the CLI to operate at the original (non-degraded) mode until fresh failures re-trigger degradation.
- **Fix:** Add a direct state-restoration API to `DegradationGuard` (e.g., `restore_from_snapshot`) that sets `degradation_state`, `current_mode`, and `failure_count` directly without replaying synthetic failures.

#### W-3: `should_degrade` uses incorrect failure-window semantics
- **Location:** `crates/nexus-domain/src/degradation.rs:334-346`
- **Issue:** The method checks `elapsed_since_last_failure <= window`, not "did `failure_threshold` failures occur within the last `failure_window_secs`". This means widely spaced failures can trigger degradation as long as the final failure is recent.
- **Example:** With `failure_threshold=3` and `failure_window_secs=60`, failures at T=0, T=10, and T=70 will degrade the system even though only one failure fell inside the 60-second window.
- **Fix:** Maintain a bounded queue or sliding-window timestamp list of recent failures, and require that `failure_threshold` of them fall within `failure_window_secs` of the current time (or of each other).

#### W-4: `AssembleLocal` two-stage path calls platform with a placeholder `world_id`
- **Location:** `crates/nexus42/src/commands/context.rs:376-420`
- **Issue:** `try_platform_assemble` hardcodes `world_id: "wld_default".to_string()` and documents that it will "result in a 404/422 if no world is configured, which is treated as 'platform unavailable' for fallback purposes." Because of W-1, the call already fails at the deserialization layer; even if W-1 were fixed, the placeholder world ID guarantees the platform path cannot return meaningful data.
- **Impact:** Two-stage assembly in `local_first`/`cloud_enhanced` can never successfully retrieve platform context, making the fallback path the only viable path. This undermines the value of the new `TwoStageAssembly` model.
- **Fix:** Require a real `world_id` for two-stage assembly (either as a CLI argument or resolved from workspace state) before attempting the platform call.

### 🟢 Suggestion

#### S-1: `max_file_size` CLI argument is accepted but ignored
- **Location:** `crates/nexus42/src/commands/context.rs:84-86`, `121-122`, `161-164`
- **Issue:** The `Assemble` command accepts `--max-file-size` but explicitly suppresses it with `let _ = max_file_size;`. This creates a CLI surface that does not behave as advertised.
- **Fix:** Either remove the argument until the backend supports it, or forward it to the daemon request and wire it into `SummaryGenerator`.

#### S-2: Missing truncation support in `TwoStageAssembly`
- **Location:** `crates/nexus-domain/src/context_assembly.rs:389-496`
- **Issue:** `TwoStageAssembly` exposes `max_tokens: Option<usize>` and has an `assemble()` method, but unlike `Stage0Assembly` it does not implement `assemble_with_truncation()`. In `assemble_local_with_routing`, truncation is only applied in `local_only` mode or in the fallback path; the successful two-stage path ignores `max_tokens`.
- **Fix:** Implement token-budget truncation for `TwoStageAssembly` that respects the same non-truncatable rules (Personality never truncated).

#### S-3: `DegradationSnapshot` restoration should preserve `last_health_check` timestamps
- **Location:** `crates/nexus-domain/src/degradation.rs:205-226`
- **Issue:** `DegradationSnapshot::from_guard` converts `checked_at` to RFC 3339 string, but there is no symmetric method to restore it back into a `HealthCheckResult` with the original `DateTime<Utc>`.
- **Fix:** Add a `to_guard` or `restore` helper that round-trips the timestamp correctly.

## Cross-Reviewer Ready Notes

- **Exploitability of W-3:** Low-Medium — premature degradation causes unnecessary downgrade of service quality but does not expose sensitive data.

Fixing W-1 and W-2 is required before merge. W-3 should be addressed or at least documented with a residual tracking item. S-1 through S-3 can be deferred to a follow-up if documented.
