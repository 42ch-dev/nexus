## Critical

### C1: `personality_core` Missing from JSON Schema Wire Contract (BLOCKING)

**Severity**: Critical
**Files**: `schemas/common/common.schema.json` line 181; `crates/nexus-domain/src/memory_item.rs` line 44-46

**Finding**: The `MemoryKind` enum in `schemas/common/common.schema.json` defines:
```json
"enum": ["story_summary", "research_material", "review_note", "character_note", "world_building", "plot_outline", "theme_analysis", "custom"]
```

But `crates/nexus-domain/src/memory_item.rs` adds a 9th variant `PersonalityCore` (line 44-46) with wire value `"personality_core"`, which is used throughout the pipeline:
- `personality_sync.rs` line 72: creates memories with `memory_kind = "personality_core"`
- `context_assembly.rs` skips `personality_core` (line 224) because it's sourced from SOUL.md
- `experience_aggregation.rs` line 377-383 explicitly excludes `personality_core` from experience aggregation

**Spec Reference**: `creator-memory-soul-lifecycle-v1.md` §4.2 defines the personality track → memory projection that uses `personality_core` kind. The spec §3.3 frontmatter schema says `memory_kind` "must align with MemoryKind enum from data-model / ADR-001".

**Impact**: Any `personality_core` memory that reaches the wire contract (e.g., for `cloud_enhanced` sync to platform) will fail schema validation. The platform's `MemoryKind` enum in `common.schema.json` is the authoritative wire contract — the Rust domain model cannot unilaterally add values.

**Required Fix**: Add `"personality_core"` to the `MemoryKind` enum array in `schemas/common/common.schema.json`, or document why `personality_core` is intentionally excluded from wire contracts (e.g., it stays local-only). Per the spec, this enum lives in data-model/ADR-001; any addition requires an ADR patch.

## Warning

### W1: Session Capture Has No Durability Guarantee on Abnormal Termination

**Severity**: Warning
**File**: `crates/nexus42/src/acp/session_capture.rs` lines 187-245

**Finding**: `submit_to_daemon()` is fire-and-forget with a 5-second timeout. If the daemon is unreachable (or the process is killed with `kill -9` before the spawn completes), the session capture data is lost. The spec §6.2 says session-end "must" write to `memory_pending_review` table.

**Evidence**: `session_capture.rs` line 267-269 spawns a background task, but `spawn()` has no guarantee the task will complete before process exit.

**Runtime Impact**: On `kill -9` or power loss, any in-flight session captures are lost. The pending review queue will have gaps.

**Suggestion**: Consider writing session capture data to a local temp file or the daemon's WAL journal before acknowledging session end, so abnormal terminations can be recovered on restart.

---

### W2: `memory_fragments` Table Has No Cleanup Mechanism

**Severity**: Warning
**File**: `crates/nexus-local-db/src/schema.rs` lines 195-205

**Finding**: The `memory_fragments` table stores fragments with an optional `ttl` column, but:
1. No TTL-based expiration logic is implemented (no daemon timer, no cleanup cron)
2. No DELETE endpoint exists in the daemon API for fragments (only for pending reviews)
3. The CLI `memory fragments` command (lines 229-233) is a stub that just prints "requires daemon"

**Spec Reference**: spec §7.2 defines fragment TTL behavior but doesn't specify who enforces expiration.

**Suggestion**: Either implement TTL cleanup in the daemon, add a DELETE endpoint for fragments, or document that TTL is advisory only and fragments persist until manually deleted.

---

### W3: Slug Generation from `memory_id` Can Produce Collisions

**Severity**: Warning
**File**: `crates/nexus-domain/src/review.rs` line 628

**Finding**: Promotion creates slug via:
```rust
let slug = memory.frontmatter.memory_id.replace("mem_", "memory-");
```
If two sessions produce memory IDs like `mem_abc123` and `mem_abc12345`, both would produce slug `memory-abc123` — the longer ID's suffix would be truncated silently.

**Evidence**: UUID-based IDs make this unlikely in practice, but the collision is not prevented.

**Suggestion**: Use the full `memory_id` (with `mem_` prefix or hashed) as the slug, or use a hash of the memory_id to ensure uniqueness.

---

### W4: `EXPERIENCE_MEMORY_KINDS` Hardcoded List Not Aligned with Schema

**Severity**: Warning
**Files**: `crates/nexus-domain/src/experience_aggregation.rs` lines 17-24

**Finding**: `EXPERIENCE_MEMORY_KINDS` is a hardcoded Rust constant:
```rust
const EXPERIENCE_MEMORY_KINDS: &[&str] = &[
    "story_summary", "review_note", "character_note",
    "world_building", "plot_outline", "theme_analysis",
];
```

This list must manually stay in sync with the `personality_core` exclusion and the JSON Schema enum. If a new experience-type memory kind is added to the schema but not to this list, it would be silently excluded from experience aggregation.

**Suggestion**: Derive `EXPERIENCE_MEMORY_KINDS` at runtime from the validated `MemoryKind` enum (excluding `personality_core` and `research_material`), or add a compile-time assertion that verifies the hardcoded list matches the schema.

### W5: Review Command is a Stub

**Severity**: Warning
**File**: `crates/nexus42/src/commands/memory.rs` lines 221-227

**Finding**: The `review` subcommand does nothing except print instructions to start the daemon:
```rust
async fn review(_config: &CliConfig, _creator_id: &str) -> Result<()> {
    println!("Review queue requires a running daemon.");
    println!("Start the daemon with `nexus42 daemon start` and retry.");
    Ok(())
}
```

The spec §7 says the daemon runs scheduled review. But there is no CLI command to manually trigger a review cycle, no way to see what actions the review algorithm would take, and no dry-run mode.

**Suggestion**: Either implement a real review command that calls the daemon API, or remove the stub and document that review is fully daemon-driven.

## Suggestion

### S1: `deduped_keywords()` Deduplication Is Case-Sensitive After Normalization

**Severity**: Suggestion
**File**: `crates/nexus-domain/src/context_assembly.rs` lines 190-199

**Finding**: Keywords are normalized to lowercase during deduplication:
```rust
let set: BTreeSet<String> = self
    .fragment_keywords
    .iter()
    .map(|k| k.trim().to_lowercase())  // normalized to lowercase
    .filter(|k| !k.is_empty())
    .collect();
```

But if the same keyword appears as "Character" and "character" in the input, only one would appear in the output (correct behavior). However, the deduplication happens after lowercase transformation, so the result preserves original case from whichever entry was processed first. This means `"Character"` and `"character"` inputs produce one output but the case is arbitrary.

**Suggestion**: Document that keyword dedup is case-insensitive but output case reflects the first-seen variant, or normalize to lowercase consistently.

---

### S2: Token Estimation Uses Simple `chars/4` Heuristic

**Severity**: Suggestion
**File**: `crates/nexus-domain/src/context_assembly.rs` line 206-208

**Finding**: `estimate_tokens()` uses `text.len().div_ceil(4)`. This is a rough approximation. Actual token counts vary by tokenizer (cl100k_base used by GPT-4 is ~0.4 chars/token for English, but can be 2-3x for code or CJK characters).

**Suggestion**: Add a comment that this is an approximation and document when calibration may be needed. Consider adding actual token count tracking for observability.

### S3: `PendingReviewInput` world_id Is Optional but Not Indexed

**Severity**: Suggestion
**File**: `crates/nexus-local-db/src/schema.rs` lines 179-189

**Finding**: `memory_pending_review` table:
```sql
CREATE TABLE IF NOT EXISTS memory_pending_review (
    pending_id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    creator_id TEXT NOT NULL,
    world_id TEXT,   -- optional, no index
    task_kind TEXT NOT NULL DEFAULT 'unknown',
    raw_digest TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

`world_id` is nullable and has no index. Queries filtering by `world_id` would be slow on large tables.

**Suggestion**: Add an index on `world_id` if queries will filter by it, or clarify that `world_id` filtering is not a supported query pattern.

## Cross-Reviewer Ready Notes

**Performance concerns**:
- `check_session_already_promoted` loads ALL memories for a creator on every promotion check (line 532-541 in `review.rs`). For creators with many memories, this is O(n) I/O per review.
- `list_memories` reads the entire directory on every call; no caching.

**Observability gaps**:
- No metrics emitted for `memory_pending_queue_depth`, `memory_review_runs_total`, etc. (spec §10).
- `tracing::info!` in `promote_to_long_term` provides audit trail but no structured metrics.

**Edge case: empty memory directory**: `list_memories` correctly returns empty Vec when dir doesn't exist (line 63-64 in `memory_io.rs`).

**Edge case: all memories are personality_core**: Experience aggregation will produce empty result (correct per spec), but there's no explicit test for this scenario.
