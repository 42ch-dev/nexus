## Critical

### C1: `create_pending_review` handler — no idempotent upsert on duplicate `pending_id`

**File**: `crates/nexus42d/src/api/handlers/memory.rs:105-110`
**Risk**: Session dedup reliability (§6.2) — CLI may retry on network failure

The `POST /v1/local/memory/pending-review` handler does a bare `INSERT`:
```rust
conn.execute(
    "INSERT INTO memory_pending_review (pending_id, ...) VALUES (?1, ...)",
    rusqlite::params![pending_id, ...],
)
```

If the CLI's `submit_to_daemon` fires a retry (e.g., 5s timeout expires but daemon actually processed the first request), the second INSERT will hit `PRIMARY KEY` constraint violation and return a 500 error to the caller. The spec (§6.2) requires session dedup to work correctly.

**Fix**: Use `INSERT OR IGNORE` or `INSERT OR REPLACE` with ON CONFLICT, or check-exists-before-insert. The `submit_to_daemon` in `session_capture.rs` generates a fresh UUID for `pending_id` each call, so there is no idempotency key at the CLI level — but the daemon handler should still be resilient to duplicates.

**Priority**: Must fix. Either (a) make the CLI generate a deterministic `pending_id = "pending_{session_id}"` and use `INSERT OR IGNORE`, or (b) add a UNIQUE constraint on `session_id` and handle conflicts in the handler.

### C2: `create_pending_review` handler — no validation of `pending_id` / `session_id` / `raw_digest`

**File**: `crates/nexus42d/src/api/handlers/memory.rs:64-128`
**Risk**: Input validation gap — only `creator_id` is validated

The handler validates `creator_id` starts with `ctr_`, but does not validate:
- `pending_id` — no length/format check; an extremely long string wastes storage
- `session_id` — no format check; could be empty string
- `raw_digest` — no length limit; arbitrarily large payloads accepted

While rusqlite parameterized queries prevent SQL injection, these unvalidated values flow directly into the DB and could cause storage exhaustion or downstream issues when consumed by the review algorithm.

**Fix**: Add minimum/maximum length checks:
```rust
if req.pending_id.is_empty() || req.pending_id.len() > 128 {
    return Err(NexusApiError::InvalidInput { field: "pending_id".into(), reason: "...".into() });
}
if req.raw_digest.len() > 64 * 1024 {
    return Err(NexusApiError::InvalidInput { field: "raw_digest".into(), reason: "raw_digest exceeds 64KB".into() });
}
```

**Priority**: Must fix before merge. At minimum, reject empty IDs and cap `raw_digest` size.

## Warning

### W1: Duplicate code between `aggregate_experience` and `aggregate_experience_preview`

**File**: `crates/nexus-domain/src/experience_aggregation.rs:48-80` vs `crates/nexus-domain/src/experience_aggregation.rs:85-125`

These two functions share ~90% identical code (list → filter → sort → generate). The only difference is step 5 (one writes to SOUL.md, the other doesn't).

**Fix**: Extract a shared `collect_experience_entries(home, creator_id) -> Vec<ExperienceEntry>` helper. Both functions then call it and diverge only at the SOUL.md write step.

### W2: `experience_aggregation.rs` swallows `load_memory` errors silently

**File**: `crates/nexus-domain/src/experience_aggregation.rs:58`
```rust
let memory = match memory_io::load_memory(home, creator_id, slug) {
    Ok(m) => m,
    Err(_) => continue, // Skip unreadable memories
};
```

Same pattern in `aggregate_experience_preview`. Unreadable memories are silently skipped with no logging. If a memory file is corrupted, the user has no way to know why it's excluded from experience aggregation.

**Fix**: Add `tracing::warn!(slug, error = %e, "Skipping unreadable memory during experience aggregation")`.

### W3: `context.rs` spawns a tokio runtime synchronously for fragment keyword fetch

**File**: `crates/nexus42/src/commands/context.rs:262-267`
```rust
let rt = tokio::runtime::Builder::new_current_thread()
    .enable_all()
    .build()
    .map_err(|_| ())?;
rt.block_on(async { ... })
```

This creates a nested tokio runtime inside an already-async context (the `run` function is `async`). If `run` is already inside a tokio runtime, `Builder::new_current_thread().build()` may work but is wasteful. If called from a sync context, it's correct but the `run()` dispatch should ideally be consistent.

**Fix**: Since `assemble_local` is called from an async `run()`, pass through the existing runtime context rather than spawning a new one. Or make `collect_fragment_keywords` async and call it from the existing async context.

### W4: `review.rs` classification has no `MergeIntoExisting` or `TriggerSoulExperienceOnly` rule path

**File**: `crates/nexus-domain/src/review.rs:150-224`

The `ReviewAction` enum has 5 variants (Drop, FragmentOnly, PromoteToLongTerm, MergeIntoExisting, TriggerSoulExperienceOnly), but `classify_pending_review` only returns Drop, FragmentOnly, and PromoteToLongTerm. The docstring says "MergeIntoExisting: Content overlaps with existing memory (Phase 2 feature)" — acceptable as future work, but the gap should be documented in the plan notes so downstream consumers don't assume all enum variants are reachable.

**Risk**: Low, but downstream code that exhaustively matches on `ReviewAction` may have dead branches. No current compile-time hazard since matches don't require exhaustiveness for non-`#[non_exhaustive]` enums.

### W5: `session_capture.rs` — `#![allow(dead_code)]` module-wide

**File**: `crates/nexus42/src/acp/session_capture.rs:4`

The entire module is suppressed for dead_code because functions are "not yet wired into session flow." This is a temporary measure, but it means:
- No compile-time enforcement that public API is actually used
- Function signatures can drift without detection
- The module will grow stale if the wiring is delayed

**Fix**: Add a tracking issue / TODO comment with the expected integration point (which T-task / which module will call these). Remove `#![allow(dead_code)]` once wired.

### W6: `memory.rs` CLI commands — `Review` and `Fragments` are stubs

**File**: `crates/nexus42/src/commands/memory.rs:185-194`

```rust
async fn review(_config: &CliConfig, _creator_id: &str) -> Result<()> {
    println!("Review queue requires a running daemon.");
    Ok(())
}
```

These commands print a message and return Ok without actually performing any operation. This is acceptable as a V1.2 limitation, but:
- No error exit code — scripts parsing stdout won't know review failed
- `_creator_id` is unused (should not be in signature, or should be used in the error message)

**Fix**: Return an actual `CliError` with a non-zero exit, or at minimum use `eprintln!` instead of `println!` and include the `creator_id` in the message.

## Suggestion

### S1: `experience_aggregation.rs` — `EXPERIENCE_MEMORY_KINDS` is duplicated with `MemoryKind` enum

The constant list `["story_summary", "review_note", "character_note", "world_building", "plot_outline", "theme_analysis"]` should derive from `MemoryKind` enum variants rather than being a separate string array. This creates a maintenance coupling: adding a new experience-worthy kind requires updating two places.

### S2: `review.rs` — `extract_keywords` could use a `HashSet` for O(1) dedup

Current implementation uses `Vec::contains()` for dedup (O(n) per word). For a typical digest this is negligible, but using `HashSet<String>` would be cleaner:
```rust
let mut seen = HashSet::new();
for word in words {
    if seen.insert(word.clone()) {
        keywords.push(word);
    }
}
```

### S3: `context_assembly.rs` — `truncate_with_budget` partial truncation may produce malformed markdown

When a truncatable section is partially truncated (line 280-290), the content is cut at a character boundary with `truncate_to_char_count`. This may split in the middle of a markdown element (e.g., mid-URL, mid-code-fence). For V1 this is acceptable since the output is for LLM consumption, but worth documenting as a known limitation.

### S4: `long_term_memory.rs` — `render()` silently swallows serde_yaml errors

```rust
let yaml = serde_yaml::to_string(&self.frontmatter).unwrap_or_default();
```

If serialization fails (unlikely but possible), the memory file would be written with empty frontmatter, which would fail to parse on the next load. Consider returning `Result<String, DomainError>` from `render()`.

### S5: DB migration v4 — no rollback path

`migrate_v4` creates tables with `CREATE TABLE IF NOT EXISTS` which is safe for forward migration but provides no DROP TABLE rollback. For a local SQLite DB this is acceptable (users can delete the DB file), but should be noted in the migration comments.

### S6: `memory_io.rs` — `list_memories` sorts slugs alphabetically, not by recency

```rust
slugs.sort();
```

Callers that want "most recent first" must load each memory to read `updated_at`. Consider adding `list_memories_with_meta()` that returns `(slug, memory_kind, updated_at)` tuples. This is a performance consideration for creators with many memories.
