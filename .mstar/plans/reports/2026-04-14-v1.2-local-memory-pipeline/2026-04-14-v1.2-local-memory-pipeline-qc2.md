## Summary

The local memory pipeline is a substantial, well-structured feature addition. Module decomposition is clean, path-traversal protections are in place, and domain-logic test coverage is solid. However, there are **two correctness issues** that can cause runtime panics, plus several architectural and performance concerns that should be addressed or tracked before merge.

## Critical

- **C1 — Nested Tokio runtime panic in `context assemble-local`**  
  `crates/nexus42/src/commands/context.rs::try_fetch_fragment_keywords()` creates a new `tokio::runtime::Builder::new_current_thread().build()` and calls `rt.block_on(...)` from inside `assemble_local`, which is invoked synchronously from the async `run()` function (the CLI uses `#[tokio::main]`). Starting a runtime from within a Tokio runtime **will panic at runtime** with *“Cannot start a runtime from within a runtime”*. Because `include_fragments` defaults to `true`, `nexus42 context assemble-local` is effectively guaranteed to crash whenever the daemon is reachable.  
  **Fix**: make `assemble_local` `async`, await the HTTP call directly, and remove the nested runtime creation.

- **C2 — `expect()` in production path (`daemon_client.rs`)**  
  `crates/nexus42/src/api/daemon_client.rs` (~line 60) calls `.expect("Failed to build reqwest Client")` on `reqwest::Client::builder().build()`. This is production (non-test) code; a bad proxy config or exhausted file-descriptor limit would cause an unrecoverable panic.  
  **Fix**: return `crate::errors::CliError` instead of panicking.

## Warning

- **W1 — Weak `creator_id` validation in daemon API handlers**  
  `crates/nexus42d/src/api/handlers/memory.rs` validates `creator_id` only with `starts_with("ctr_")`. This is weaker than `nexus_domain::is_valid_creator_id` (which rejects path separators, `..`, backslashes, and control characters). Malformed IDs such as `ctr_../../../etc` can reach downstream logic. While SQL injection is impossible here (parameterized queries), future file-system consumers are at risk.  
  **Fix**: reuse `nexus_domain::is_valid_creator_id(&req.creator_id)` in the handlers.

- **W2 — Duplicated logic in `experience_aggregation.rs`**  
  `aggregate_experience` and `aggregate_experience_preview` share ~40 identical lines of filtering, sorting, and synthesis. Any future fix to the filtering logic (e.g., adding a new experience kind) must be edited in two places.  
  **Fix**: extract a private `build_experience_markdown(home, creator_id, synthesizer)` helper.

- **W3 — O(N) file I/O on every promotion idempotency check**  
  `crates/nexus-domain/src/review.rs::check_session_already_promoted` lists **all** memory files on disk and parses each frontmatter to look for a single `session_id`. As memory volume grows this becomes a significant latency bottleneck.  
  **Mitigation**: track promoted sessions in the SQLite DB (a simple `promoted_sessions` table or index) so the check is O(1).

- **W4 — Silent data loss risk in `LongTermMemory::render()`**  
  `crates/nexus-domain/src/long_term_memory.rs` uses `serde_yaml::to_string(&self.frontmatter).unwrap_or_default()`. If serialization ever fails, the frontmatter block becomes empty YAML (`---\n---\n`) and the metadata is silently discarded.  
  **Fix**: make `render()` return `Result<String, DomainError>` and propagate the serialization failure.

## Suggestion

- **S1 — Inefficient deduplication in `get_all_keywords`**  
  `crates/nexus-local-db/src/memory_fragment.rs::get_all_keywords` uses `Vec::contains` inside a nested loop for deduplication. Replace the `Vec` with a `BTreeSet<String>` (or `HashSet`) for clarity and better complexity.

- **S2 — Inconsistent async trait styles**  
  `ExperienceSynthesizer` (in `experience_aggregation.rs`) uses `Pin<Box<dyn Future>>` while `SessionDigestSummarizer` (in `review.rs`) uses `impl Future` in trait (RPITIT). Standardize on the modern RPITIT style now that the workspace supports it.

- **S3 — Windows reserved filenames not rejected**  
  `slug_is_safe` does not block Windows reserved names (`CON`, `PRN`, `AUX`, `NUL`, `COM1`–`COM9`, `LPT1`–`LPT9`). Creating a memory with one of these slugs would fail cryptically on Windows. Add a small static deny-list.

- **S4 — Sparse CLI command tests**  
  `crates/nexus42/src/commands/memory.rs`, `soul.rs`, and `context.rs` test modules only perform compile-time enum checks. Add at least one roundtrip integration test per command (e.g., create → show → delete) using a temporary home directory, mirroring the domain-layer test patterns.

- **S5 — Digest thresholds use byte length, not char count**  
  `review.rs::classify_pending_review` uses `record.raw_digest.len()` (byte length) for all thresholds. For multi-byte UTF-8 input (e.g., CJK characters) a 50-byte string may be only ~16 characters, causing unexpectedly aggressive dropping. Consider `chars().count()` or document that thresholds are byte-based.

## Architecture Notes

- **Module decomposition**: Excellent. The separation between `nexus-domain` (business rules), `nexus-local-db` ( persistence), `nexus42d` (REST API), and `nexus42` (CLI commands) is clean and follows the repo conventions.
- **Trait design**: `SessionDigestSummarizer` and `ExperienceSynthesizer` provide clean seams for LLM integration without pulling ACP client dependencies into the domain layer.
- **DB migrations**: V4 migration (`memory_pending_review` + `memory_fragments`) is idempotent and correctly registered in `migration.rs`.
- **Path-traversal defense**: `memory_io.rs` validates both `creator_id` and `slug` before any filesystem operation. This is the correct defense-in-depth posture.
- **Generated code / fmt**: `cargo fmt --check` reports diffs only in `crates/nexus-contracts/src/generated/`. This is expected because stable `rustfmt` cannot apply the `ignore` directive from `.rustfmt.toml` (documented in `AGENTS.md`). No action needed.
