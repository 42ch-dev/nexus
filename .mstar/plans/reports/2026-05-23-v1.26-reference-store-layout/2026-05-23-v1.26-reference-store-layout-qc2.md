### 🟡 Warning
- **W-01: Inconsistent defense-in-depth for `creator_id` validation in new `creator reference` commands**  
  The new `crates/nexus42/src/commands/creator/reference.rs` (run_register / run_list / run_show / resolve_creator_context) does **not** call `paths::validate_creator_id_safe` (or equivalent) at entry, unlike the sibling `run_kb` in `creator/mod.rs:763-767` which has explicit `// F002: Validate active_creator_id before constructing any paths.` guard + early `CliError`.  
  While `nexus_home_layout::reference_body_path` (and `assert_creator_id_safe`) will panic on bad ID during `register` (defense-in-depth), list/show proceed silently, and early structured error is preferable for UX/security parity.  
  Source: `reference.rs:115-140` (no validate), vs `mod.rs:763` (KB has it); `paths.rs` re-exports `validate_creator_id_safe` from `nexus-home-layout`.  
  Risk: config tampering or corrupted `active_creator_id` (persisted in TOML) could reach FS path logic without early rejection (though assert mitigates).  
  **Fix**: Add the same `if let Some(cid) = &config.active_creator_id { paths::validate... }` guard in reference command entry points (or a shared `resolve_creator_context` helper). Update tests.  
  Confidence: High.

### 🟢 Suggestion
- **S-01: Non-atomic FS write in `register`**  
  In `nexus-local-db/src/reference_source.rs:130-148`: `create_dir_all(parent)` then `tokio::fs::write(body_abs, ...)` with no temp-file + atomic rename, no rollback on partial failure. Error path returns `LocalDbError::Io` (good propagation), but leaves orphan `<reference_id>/` dir without `body.md` on disk-full / permission / crash.  
  Subsequent register uses fresh UUID so no collision, but state is inconsistent until manual cleanup.  
  Per spec §2, body is canonical; recommend temp write + rename for durability (or `fs::write` + explicit dir cleanup in Err arm).  
  Low risk (local dev only, pre-1.0).

- **S-02: No body size / content validation on register**  
  `--body <text>` or `--file <path>` (stdin or disk) accepted without length limit, UTF-8 enforcement beyond `read_to_string`, or sanitization. Written verbatim to `body.md`.  
  Local CLI (user-controlled input), so acceptable for V1.26 minimal surface; add note for future `refreshable` pipelines or large PDF ingest.

- **S-03: `workspace_id` column stored but queries unfiltered (by design via DB isolation)**  
  `reference_sources` INSERT sets `workspace_id = "wrk_<slug>"`; `list_references` / `get_by_id` / daemon handlers perform cross-row SELECTs with **no `WHERE workspace_id = ?`**.  
  Correct in practice because each workspace opens its own `state.db` file (`resolve_state_db_path` + `WorkspaceState`); rows are partitioned by SQLite file, not logical column.  
  Column remains useful for future cross-db queries, metadata, or daemon multi-tenant. Document explicitly in schema / spec to avoid future confusion. No security impact.

- **S-04: `reference_id` from untrusted sources (CLI arg / HTTP Path) used only in parameterized queries**  
  `nexus42 creator reference show <id>`, `GET /v1/local/references/{id}` pass raw string to `get_by_id(..., &reference_id)` which uses `WHERE reference_source_id = ?`. No FS join or string interpolation. Safe (no traversal/SQLi).  
  If future code materializes `body.md` from the returned `content_path` or reconstructs via `reference_body_path(..., id_from_db_or_input)`, the helper's `assert_reference_id_safe` / `validate_reference_id_safe` will catch. Already covered by tests. Good pattern.

- **S-05: Migration is destructive-but-documented (pre-1.0)**  
  `20260523_reference_store_split.sql` does `UPDATE reference_sources SET content = NULL`. Matches plan/spec §7 "Pre-release wipe: local persistence may be wiped rather than migrated." No data loss surprise for users on fresh installs. Reversible only by re-registering bodies.
