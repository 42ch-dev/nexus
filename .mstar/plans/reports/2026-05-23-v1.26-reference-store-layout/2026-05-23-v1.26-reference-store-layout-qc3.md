### 🟡 Warning

- **F-001 — Orphaned body.md on SQL insert failure (partial-write atomicity)**:
  In `crates/nexus-local-db/src/reference_source.rs:register()`, `body.md` is written to disk via `tokio::fs::write()` **before** the SQLite `INSERT`. If the database insert subsequently fails (e.g., constraint violation, pool exhaustion, I/O error on the DB file), the `body.md` file remains on disk as an orphan with no corresponding registry row. This violates write atomicity across the two storage backends (filesystem + SQLite). No cleanup or rollback is attempted.
  → **Fix**: Swap the order — `INSERT` first, then write `body.md`. If the file write fails, the orphaned DB row can be cleaned up or is harmless (the `content_path` column correctly points to a non-existent file, and `scan_status = 'pending'` signals incomplete state). Alternatively, wrap both operations in a compensating cleanup block (`delete` from DB + `remove_file` on disk) on failure.

- **F-002 — `list()` queries have no pagination — unbounded result set**:
  In `crates/nexus-local-db/src/reference_source.rs:list()`, all rows are fetched with `fetch_all()` and returned in a single `Vec`. There is no limit, offset, or cursor-based pagination. For a creator with thousands of reference sources, this will consume proportional memory and cause linear latency growth. The daemon handler (`nexus-daemon-runtime`) and CLI (`nexus42`) both route through this function.
  → **Fix**: Add pagination parameters (`limit`, `offset` or cursor) to `list()`. For pre-1.0, document the unbounded behavior as a known pre-release tradeoff.

- **F-003 — Missing index on `created_at` for sort-heavy `list()`**:
  The `list()` query uses `ORDER BY created_at DESC`. No index exists on the `created_at` column in the `reference_sources` table. SQLite will perform a full table scan + sort for every invocation. With hundreds of rows the impact is negligible; with thousands, latency will grow linearly.
  → **Fix**: Add `CREATE INDEX idx_reference_sources_created_at ON reference_sources(created_at)` in a follow-up migration. This is a low-risk, high-impact improvement.

- **F-004 — No body size limit in `register()` — unbounded memory allocation**:
  The CLI reads body text from stdin (`std::io::stdin().read_to_string(&mut buf)`) or file (`std::fs::read_to_string(path)`) without any size cap, then passes the entire content as `&str` to `register()`. A user could pipe a multi-gigabyte file and exhaust process memory or crash.
  → **Fix**: Add a configurable size limit (e.g., 10 MiB) with pre-allocation size check, or stream the body in chunks with a progressive hash. For pre-1.0, document the limit as a known gap.

- **F-005 — `content_hash` column always set to NULL — integrity field unused**:
  The `reference_sources.content_hash` column exists in the schema but `register()` always sets it to `NULL`. No hash (SHA-256 or similar) of `body.md` is computed at registration time. This means there is no way to detect body file corruption or tampering without re-reading the file. The `ReferenceSourceRow.content_hash` field is `Option<String>` and is surfaced in the CLI `show` output (it only appears when Some).
  → **Fix**: Compute a SHA-256 hash of the body content during registration and store it in `content_hash`. This enables integrity verification in future refresh/scan pipelines.

### 🟢 Suggestion

- **S-001 — `SourceMutability` stored as `String` in `ReferenceSourceRow`**:
  The `reference_source.rs` module defines a proper `SourceMutability` enum with `as_str()`, but the `ReferenceSourceRow` struct stores it as `String`. The `From<ReferenceSourceRow>` conversion in the daemon handler passes it through as-is. Storing the enum directly would prevent invalid string values from propagating through the system.
  → **Consider**: Store `SourceMutability` as the enum in `ReferenceSourceRow` and parse on construction. The current string-based approach is safe because the INSERT always writes a valid value via `SourceMutability::as_str()`, but the type system doesn't enforce it downstream.

- **S-002 — `register()` creates directories without cleanup on failure paths**:
  `tokio::fs::create_dir_all()` is called before the SQL INSERT. If `create_dir_all` succeeds but the subsequent `tokio::fs::write(&body_abs, ...)` fails, an empty directory is left behind. Conversely, if both filesystem operations succeed but the SQL INSERT fails, both the directory and `body.md` are orphaned (see F-001). There is no cleanup attempt on any failure path.
  → **Consider**: Wrap the whole operation in a cleanup block that removes the unit directory on any error after `create_dir_all` succeeds. This is more comprehensive than just swapping the write order.

- **S-003 — `tokio::fs` for file I/O in `register()` not truly async on local filesystems**:
  On most platforms (Linux, macOS), `tokio::fs` delegates to `spawn_blocking` for local filesystem operations. While functionally correct, this introduces a thread pool hop per operation. For the CLI use case this is irrelevant; for the daemon (if `register` is ever exposed there), the overhead is negligible but worth noting.
  → **Note**: Current usage is CLI-only. No action needed for pre-1.0. Track if daemon-side registration is planned.

- **S-004 — `reference_source_id` generated client-side via UUID v4**:
  The ID is generated in `register()` via `format!("ref_{}", uuid::Uuid::new_v4().simple())`. UUID v4 collision probability is negligible, but this pattern means the ID is not idempotent — retrying a failed registration after a partial write will produce a new row + new body file (rather than overwriting the orphaned one from F-001).
  → **Consider**: Accept an optional client-provided `reference_source_id` parameter for idempotent retry scenarios (e.g., CLI resubmit after transient DB error).
