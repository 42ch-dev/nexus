## Cross-Reviewer Ready Notes (QC3 — Performance & Reliability)

### Hot-path complexity
The refactored `insert_nested` function replaces `key.split('.').collect::<Vec<&str>>()` with `key.rsplit_once('.')` followed by `prefix.split('.')`. This eliminates an intermediate `Vec` allocation for every dot-separated key. For typical context keys (1–3 segments) this is a minor but measurable reduction in heap pressure. The loop body is otherwise unchanged — O(n) over the key string, with one `HashMap` entry insertion per segment.

### Resource lifecycle
No resources are held across await points or function boundaries. `insert_nested` takes ownership of `value: serde_json::Value` and either inserts it at the root or drills through intermediate `serde_json::Map` entries (created on demand via `or_insert_with(|| serde_json::json!({}))`). No leaks, no double-inserts, no partial mutations on failure paths.

### Concurrency risk
None. Both `build_nested_payload` and `insert_nested` are synchronous, pure functions with no shared mutable state. They operate on a freshly allocated `serde_json::Map` owned by the caller.

### Regression risk — Low
1. **Empty key behavior change (theoretical)**: The old `set_nested` would panic on an empty key (`""`) because `parts.pop()` would fail. The new `insert_nested` inserts the empty key directly. Since keys originate from `serde_json::to_value(context)["data"]` — i.e., serialized struct field names — empty keys are not expected in practice. If they do occur, the new behavior (insert instead of panic) is arguably more correct and does not break downstream handlebars rendering.

2. **Single-segment keys**: Verified equivalent. `rsplit_once('.')` returns `None` → direct insertion, matching the old logic where `parts` had length 1, the loop was skipped, and the sole segment was inserted.

3. **Multi-segment keys**: Verified equivalent. `rsplit_once('.')` yields the same leaf and prefix decomposition as `split('.').collect().pop()`.

4. **Test coverage**: All 240 library tests in `nexus-orchestration` pass, including `tasks::tests::core_context_template_repeated_renders_are_deterministic` which exercises the template rendering path that consumes `build_nested_payload`.

### Observable degradation
No degradation expected. The function is not instrumented for latency/throughput metrics, but the change is strictly a simplification with fewer allocations. No new failure modes are introduced.
