### 🟡 Warning

**W-001: Unbounded allocation in `validate_preset` — file read before size check**
- **Issue**: `crates/nexus42/src/commands/preset.rs:299` reads the entire file via `std::fs::read_to_string(file_path)` before checking `yaml.len() > VALIDATE_MAX_YAML_SIZE`. A malicious multi-gigabyte file will be fully allocated in memory before rejection, risking OOM or excessive memory pressure.
- **Fix**: Check `std::fs::metadata(file_path).len()` (or open and read in chunks up to the limit) before calling `read_to_string`.
- **Source**: `crates/nexus42/src/commands/preset.rs:299`

**W-002: `serde_yaml` double-parse expands stack-overflow attack surface before depth guard**
- **Issue**: `load_preset_from_str_with_limits` in `loader.rs` parses YAML to `serde_yaml::Value` first (for depth measurement), then parses again to `PresetManifest`. `serde_yaml::from_str` has no internal recursion limit; a YAML with extreme nesting (>~10,000 levels) can stack-overflow during either parse *before* our depth check (`actual_depth > max_depth`) is reached. The pre-change code had one unprotected parse; the post-change code has two.
- **Fix**: Use `serde_yaml::from_value(yaml_value)` for the second deserialization to avoid re-parsing from string. For defense-in-depth against the first parse, consider wrapping `serde_yaml::from_str` in a thread with a limited stack size, or document the upstream limitation and monitor for `serde_yaml` updates that add recursion limits.
- **Source**: `crates/nexus-orchestration/src/preset/loader.rs:187,197`

### 🟢 Suggestion

**S-001: Duplicate `yaml_value_depth` implementation across crates**
- **Issue**: Identical `yaml_value_depth` functions exist in `crates/nexus-orchestration/src/preset/loader.rs` and `crates/nexus42/src/commands/preset.rs`. This creates a maintenance hazard if depth semantics need to change.
- **Fix**: Export `yaml_value_depth` from `nexus-orchestration` (e.g., `pub use loader::yaml_value_depth`) and reuse it in `nexus42`.
- **Source**: `crates/nexus-orchestration/src/preset/loader.rs:914`, `crates/nexus42/src/commands/preset.rs:378`

**S-002: `build.rs` `rerun-if-changed` uses `..` relative path**
- **Issue**: `cargo:rerun-if-changed=../nexus-orchestration/embedded-presets` relies on Cargo resolving `..` relative to the package root. While this works in current Cargo versions, it is less robust than an absolute path derived from `CARGO_MANIFEST_DIR`.
- **Fix**: Compute the absolute path and emit `cargo:rerun-if-changed={absolute_path}`.
- **Source**: `crates/nexus42/build.rs:48`

**S-003: `is_scan_cache_fresh` could use fs precision documentation**
- **Issue**: The mtime comparison (`current_mtime != cached_mtime`) is correct and timezone-safe, but filesystems with 1-second mtime resolution can miss sub-second modifications. This is acceptable for presets but worth documenting.
- **Fix**: Add a one-line comment in `is_scan_cache_fresh` noting the 1-second-resolution limitation.
- **Source**: `crates/nexus-orchestration/src/user_preset_dir.rs:260`
