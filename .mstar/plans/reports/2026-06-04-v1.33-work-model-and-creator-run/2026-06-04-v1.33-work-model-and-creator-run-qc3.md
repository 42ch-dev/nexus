### 🟡 Warning

#### W-001: Idempotency error silently swallowed — retry creates duplicate Works
- **Issue**: In `crates/nexus-daemon-runtime/src/api/handlers/works.rs:153-155`, the idempotency recording uses `let _ = works::record_idempotency(...).await;`, which silently drops all errors via the `let _ =` pattern.
- **Impact**: If `record_idempotency` fails (e.g., unique constraint violation from a concurrent request, database timeout, or disk-full), the request returns `201 CREATED` with a new `work_id`, but the idempotency mapping is not persisted. On retry with the same `client_request_id`, the client receives a *different* `work_id` because the initial check finds no mapping. This violates the idempotency guarantee and can orphan Works in the database.
- **Race scenario**:
  1. Request A: idempotency check → not found; INSERT works (wrk_A); record_idempotency → fails silently
  2. Request B (retry): idempotency check → not found; INSERT works (wrk_B); record_idempotency → succeeds
  3. Client now has two different `work_id`s for the same `client_request_id`
- **Fix**: Do not silently discard the error. At minimum, surface it so the request fails and the client knows to retry. Ideally, wrap the idempotency check + work creation + idempotency recording in a single SQLite transaction so the operation is atomic.

#### W-002: `total` field in list response is misleading for pagination
- **Issue**: In `crates/nexus-daemon-runtime/src/api/handlers/works.rs:189`, `let total = records.len();` assigns the count of the *current page* (after `LIMIT`/`OFFSET`) to a field named `total`. Consumers expect `total` to mean the total number of matching rows across all pages.
- **Impact**: Pagination UIs or API consumers will compute incorrect total pages (e.g., showing "Page 1 of 1" when there are 150 works and limit=100).
- **Fix**: Either (a) rename the field to `returned` or `count` to accurately reflect its semantics, or (b) execute a separate `SELECT COUNT(*)` query (with the same `WHERE` filters but without `LIMIT`/`OFFSET`) to populate a true total.

### 🟢 Suggestion

#### S-001: Missing index for potential `world_id` query pattern
- **Observation**: The `works` table has three indices: `idx_works_creator_status`, `idx_works_creator_intake`, `idx_works_creator_updated`. These cover the current `list_works` filter patterns. However, `world_id` is nullable and not indexed. If future features query "all works bound to world X" (e.g., for a world dashboard), this will table-scan.
- **Recommendation**: Add `CREATE INDEX IF NOT EXISTS idx_works_world_id ON works (world_id);` when that query pattern is introduced. Not required for V1.33.

#### S-002: JSON column scalability — flag for backlog
- **Observation**: `creative_brief`, `inspiration_log`, and `schedule_ids` are stored as `TEXT` containing JSON. The current code does not use `JSON_EXTRACT` or `LIKE '%...%'` queries against these columns, so there is no immediate query degradation.
- **Recommendation**: Acceptable for pre-release volumes (expected: dozens to low hundreds of works per creator). If works scale beyond ~10K per creator, consider:
  - Migrating `inspiration_log` to a separate `work_inspirations` table (normalized, indexed by `work_id`)
  - Using SQLite's native `JSON` type (if json1 extension is guaranteed) for `creative_brief`
- **Action**: Add a tech-debt item to `metadata.tech_debt_summary` for post-V1.0 migration planning.

#### S-003: `system preset list` makes two sequential API calls
- **Observation**: In `crates/nexus42/src/commands/system/mod.rs:246-259`, `list_system_presets` calls `GET /v1/local/orchestration/presets` followed by `GET /v1/local/presets` to enrich with `run_intents`. This doubles the round-trip latency for every `nexus42 system preset list` invocation.
- **Impact**: Minor for current embedded preset counts (~6–10), but scales linearly with preset count.
- **Recommendation**: Acceptable for V1.33. Consider adding `run_intents` to the orchestration `ListPresetsResponse` in a future iteration to eliminate the second call.
