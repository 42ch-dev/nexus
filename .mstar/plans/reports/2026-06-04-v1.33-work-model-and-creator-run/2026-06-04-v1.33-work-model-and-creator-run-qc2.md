## Findings
### 🔴 Critical
- Idempotency race in Work creation: `create_work` handler performs a non-transactional SELECT-then-INSERT pattern (find_by_client_request_id → create_work → record_idempotency with `let _ =` swallow on failure). Concurrent POST /v1/local/works with the same client_request_id can produce duplicate `works` rows and return different `work_id`s on retries. The UNIQUE on `works_idempotency` is defeated by the TOCTOU window. SQLite write serialization does not protect the app-level check.
  → Fix: wrap the find/check + create + record in a single sqlx transaction (or use INSERT ... ON CONFLICT / upsert at app level with proper error mapping to 409).

- Wrong status code for unauthenticated / uninitialized creator context on all 5 new `/v1/local/works*` endpoints: code does `read_active_creator_id(...).ok_or(NexusApiError::Uninitialized)?` which maps to 409 Conflict. T1 contract §7.2 explicitly requires `401 Unauthorized` when no active creator in context. `Uninitialized` is also overloaded with real conflict cases.
  → Fix: use (or introduce) `NexusApiError::AuthRequired` (already exists and maps to 401) or a dedicated `NoActiveCreator` variant for the works (and other) handlers; update contract/docs if semantics differ.

- Missing API contract tests for the new works endpoints: T1 contract §7.2 and T6 explicitly require `crates/nexus-daemon-runtime/tests/works_api.rs` (axum Router + ServiceExt::oneshot tests covering 200/201/400/401/404/409, auth boundary, idempotency 200-vs-201, error shapes). No such file or equivalent coverage exists in the P1 delta (only DB roundtrips in works.rs and CLI surface tests).
  → Fix: add the test file (or equivalent) before merge; cover the exact status matrix and creator isolation (cross-creator work_id must 404 even with valid API key).

- `append_inspiration` response count and atomicity issues: handler fetches `record` (for verify + count base) *before* calling `append_inspiration`, then always returns `len(pre) + 1`. Concurrent appends produce wrong (under) counts for clients. The DB-side JSON append (substr/CASE) is a single atomic UPDATE (good for no partial note loss) but the Rust-side construction + count is not; substr logic is also fragile to any whitespace or non-compact JSON in the column.
  → Fix: re-fetch the WorkRecord *after* successful append and derive count from the post state (or have append_inspiration return the new count); consider json_array_insert or a small Rust append + full UPDATE for robustness.

### 🟡 Warning
- `run_intents` validation + trust boundary incomplete vs T1 contract §7.4 and work-experience-model:
  - `check_run_intents` only emits `Warning` (not Error) for creator with empty run_intents (contract: "ERROR — the preset is unreachable via `creator run`").
  - Creator check is only `!intents.is_empty()`; does not require at least one of `work_init` or `work_continue` (allows e.g. `[knowledge_ingest]` only on a creator-kind preset).
  - System check only warns "has run_intents but none is system_maintenance"; does not reject `_system.*` claiming `work_init`/`work_continue` etc., nor creator presets claiming `system_maintenance`.
  - No runtime gating in `creator run start --preset <id>` (or in create_work handler) that the chosen preset actually declares a usable intent.
  - `GET /v1/local/presets` (and thus `system preset list --intent`) returns `run_intents: []` for all system + user presets (never parses their YAML); only embedded presets are loaded via the registry for intents.
  - Contract-specified unit tests (`validate_preset_rejects_unknown_intent`, `validate_preset_requires_creator_intent`, `validate_embedded_presets_all_have_run_intents`) were never added (no `run_intents_validation.rs` test target; `preset_validation.rs` has zero mentions of RunIntents or the new category).
  → Fix: elevate the relevant cases to Error, tighten the predicate to the exact rules in the model, load/parse run_intents for user/system presets too, add the 3 tests, and (for P2) wire a capability-style gate in the create path.

- `creator run *` (start/continue/list/status) on daemon-down yields a raw "Network error: ..." (reqwest connect refused/timeout) instead of the friendly `CliError::DaemonNotRunning` message with the "Suggestion: Start it with `nexus42 daemon start`" text. (No panic — rg confirmed 0 unwrap/expect in run.rs — but not the "clear message" required by the assignment lens.)
  → Fix: either special-case connection errors inside DaemonClient::post/get (or the From<reqwest::Error>) to map to DaemonNotRunning, or have the high-level handle_run catch Network and re-map when it looks like "connection refused".

- Naive query-string construction for `creator run list --status`: `path = format!("{path}?status={s}")` with no urlencoding before passing to client.get (which just concatenates). A status value containing `&`, `?`, space, etc. produces a malformed URL or extra query params (server-side filter may still work but is incorrect and a latent injection vector for query params).
  → Fix: use `reqwest::Url` or `urlencoding::encode` for filter values.

- T1 contract file-layout drift + incomplete implementation: contract §7.3 explicitly said "New file: crates/nexus42/src/commands/system/preset.rs" (plus registration in mod.rs); actual change is a 151-line edit entirely inside `system/mod.rs` (SystemPresetSubcommand + list/validate logic inlined). The `list_system_presets` function (despite name and help text "all discoverable presets (embedded + user + system)") only ever surfaces embedded presets (from the legacy orchestration endpoint) and labels everything "embedded"; user/system groups from the /v1/local/presets call are queried only for the intent map and then discarded.
  → Fix: either update the contract post-facto with justification, or refactor to the specified file layout; make list actually merge embedded + system + user from the local presets endpoint.

- Silent swallow of idempotency record failure + no tx: after successful create_work, `let _ = works::record_idempotency(...)` discards any UNIQUE violation (or other error). A racy or duplicate-crid call leaves the mapping absent, so future identical client_request_id calls will create *new* works instead of returning the prior one.
  → Fix: surface the error (map unique violation to 409 Conflict) and perform the whole create + record sequence inside a transaction.

### 🟢 Suggestion
- PATCH /v1/local/works/{id} accepts any string for status/intake_status and only discovers invalid values via DB CHECK constraint violation (which becomes 500 Internal + "DATABASE_ERROR"). Contract §7.2 wants 400 for schema violation. Add app-level validation against the exact CHECK sets before the repo call (return InvalidInput / 400).
- Duplicate `read_active_creator_id` / `read_active_workspace_slug` helpers (identical toml reads) are copy-pasted into works.rs, creators.rs, and workspaces.rs. Extract to a shared `crate::api::context` or `workspace` helper.
- creative_brief is defined as "JSON text" in DDL and stored/retrieved as Option<String> in WorkRecord/WorkPatch. Contract example used Option<serde_json::Value>. P1 never sets it (intake is P2), but when P2 arrives the input path should parse + validate as JSON (no unwrap on user text) and the response type should probably surface it as Value for clients. Add a note / TODO in the handler.
- The works_idempotency table has a UNIQUE but no FK or ON DELETE CASCADE to works; fine for V1.33 but worth an index or comment if creator_id+crid lookups become hot.
- `system preset list --intent` and the JSON path in CLI swallow errors from the /v1/local/presets call (`.unwrap_or_else(|_| json!({}))`) and may return incomplete data without surfacing why. At minimum log or return a degraded response.
- Add at least smoke-level axum integration tests for the works routes (even if not the full dedicated binary yet) before claiming "API contract tested." The existing `preset_validation` style tests show the pattern.
- Minor: migration filenames in tree omit the `_v133_` infix used in the T1 contract example (allowed by the contract's own "judgment calls" note, but document the final names).
- Minor: in CLI `run start`, the default `long_term_goal` is the literal "Complete creative work" and title derivation is a simple 60-char truncate; consider making goal/ title derivation policy-driven or at least documented.
- Consider promoting the dynamic-SQL SAFETY comments in works.rs (list_works, patch_work, append_inspiration) to a small query-builder helper or at least a module comment so future maintainers don't copy the pattern without the bind discipline.
