### 🟡 Warning

**F-W1-01: `assert_template_file_safe` missing backslash check — inconsistent with `assert_creator_id_safe` and vulnerable on Windows**

- **Source Type**: manual-reasoning + git-diff
- **Source Reference**: `loader.rs:226` vs `nexus-home-layout/src/lib.rs:113`
- **Confidence**: High

The existing `assert_creator_id_safe()` in `nexus-home-layout` rejects **both** `/` and `\` as path separators:

```rust
// nexus-home-layout/src/lib.rs:113
if ch == '/' || ch == '\\' {
    panic!("creator_id contains path separator: ...");
}
```

But the new `assert_template_file_safe()` only checks for `/` prefix:

```rust
// loader.rs:226
if path.starts_with('/') { ... }
// Missing: check for '\\' or contains '\\'
```

This is an inconsistency with the "modeled after assert_creator_id_safe" design intent (per plan §Design Decisions #2). On Windows systems, paths like `..\..\etc\passwd` or `\Windows\System32` would pass `assert_template_file_safe()` but would be caught by `assert_creator_id_safe()`.

While the `validate_template_files_in_sandbox()` canonicalization check would catch real filesystem traversal on the current platform, `load_preset_from_str()` (embedded presets) relies **solely** on `assert_template_file_safe()` for structural validation. A malicious embedded preset with `template_file: "..\\..\\secret"` would pass on non-Windows systems and could become exploitable if the codebase ever runs on Windows.

**Fix**: Add a `path.contains('\\')` or `path.starts_with('\\')` check to `assert_template_file_safe()`, consistent with `assert_creator_id_safe()`. Update doc comment and add a unit test.

**F-W1-02: Template path collection logic duplicated between `validate_manifest()` and `validate_template_files_in_sandbox()`**

- **Source Type**: git-diff + manual-reasoning
- **Source Reference**: `loader.rs:254-314` (sandbox) vs `loader.rs:598-664` (manifest validation)
- **Confidence**: High

Both functions independently walk the manifest to collect all `template_file` paths from four locations:
1. `states[].exit_when.template_file` (LlmJudge variant)
2. `states[].context_update.template_file`
3. `inner_graphs.<name>.nodes[].template_file`
4. `preset.initial_action.template_file` (SeedExpansion variant)

The extraction logic (pattern matching on enum variants, format! for dot-paths) is structurally duplicated with ~25 lines of near-identical code in each function. If a future task adds a new `template_file` field to the manifest (e.g., a new `ExitWhen` variant or a new `InitialAction` kind), the developer must remember to update **both** functions. Missing one creates a silent validation gap.

**Fix**: Extract a shared helper that returns an iterator over `(dot_path, template_file)` pairs from the manifest. Both `validate_manifest()` (for structural `assert_template_file_safe` checks) and `validate_template_files_in_sandbox()` (for canonicalization checks) can consume this single iterator. Example signature:

```rust
fn collect_template_files(manifest: &PresetManifest) -> Vec<(String, &str)> { ... }
```

**F-W1-03: Name length validation not fail-first — runs after `validate_handle()`**

- **Source Type**: git-diff
- **Source Reference**: `creator.rs:288-294` (validation order in `register_creator`)
- **Confidence**: Medium

The name length check is placed **after** `validate_handle()` and **before** `AuthStore::load()`:

```rust
// Line 281-287: validate_handle runs first
let validated_handle = match &handle { ... };
// Line 288-294: name length check second
if name.len() > MAX_CREATOR_NAME_LENGTH { ... }
```

While functionally correct (both validations run before the expensive platform API call), the ordering is inconsistent with "fail fast" principles. A long name with an invalid handle would waste cycles on regex matching before the cheaper `.len()` check rejects it. This is a minor maintainability concern: future developers reading the function may not notice that validation order matters for performance.

**Fix**: Move the name length check to the very beginning of `register_creator()`, before any other validation. Add a brief comment explaining the ordering: `// Validate cheap constraints first (fail-fast).`

**F-S1-02: Test `filesystem_preset_rejects_template_file_escaping_bundle` does not exercise the sandbox canonicalization check**

- **Source Type**: git-diff + manual-reasoning
- **Source Reference**: `loader.rs` test at ~line 1948
- **Confidence**: High

The test uses `template_file: "../outside-secret.md"` which is caught by the structural `assert_template_file_safe()` check (the `..` pattern) in `load_preset_from_str()`, **before** `validate_template_files_in_sandbox()` ever runs. The test assertion even confirms this: `problems.iter().any(|p| p.error.contains(".."))`.

To truly test the sandbox canonicalization defense-in-depth layer, a test should use a path that passes `assert_template_file_safe()` but escapes the bundle — e.g., a symlink inside the bundle pointing outside. Create a symlink `bundle_root/prompts/escape.md -> /tmp/outside-secret.md` and verify the canonicalization check catches it.

**F-S1-03: Duplicate test cases for null byte and control character rejection**

- **Source Type**: git-diff
- **Source Reference**: `loader.rs` tests `reject_template_file_null_byte` and `reject_template_file_control_char` vs `template_file_safe_rejects_null_byte` and `template_file_safe_rejects_control_characters`
- **Confidence**: High

Four tests cover the same ground:
- `template_file_safe_rejects_null_byte` and `reject_template_file_null_byte` both test `assert_template_file_safe("...\0...")`
- `template_file_safe_rejects_control_characters` and `reject_template_file_control_char` both test `assert_template_file_safe("...\x01...")`

The `reject_template_file_*` prefixed tests are in the integration test section (WS-B T5) but only call `assert_template_file_safe()` directly, making them duplicates of the unit tests. Consider removing the redundant pair or repurposing them to test the end-to-end `load_preset_from_str()` rejection path.

### Rationale

F-W1-01 (missing backslash check) is a security consistency gap with the existing `assert_creator_id_safe()` pattern that the plan explicitly references as the design model. On Windows, this creates a path traversal vulnerability for embedded presets. This must be fixed before merge.

F-W1-02 (duplicated template collection logic) is a maintainability risk that will compound as the manifest schema evolves. A shared iterator helper would eliminate the risk of future template_file fields being missed in one of the two validation passes.

F-W1-03 (validation ordering) is a minor structural issue that's easy to fix and improves code clarity.
