## Critical (Blocking)

### C1: `stage_bundle_if_not_exists` — unnecessary write transaction blocks concurrent readers

**File**: `crates/nexus-sync/src/outbox.rs`, `stage_bundle_if_not_exists()`

```rust
let mut tx = self.pool.inner().begin().await?;
let exists: bool = sqlx::query_as::<_, (bool,)>(
    "SELECT EXISTS(SELECT 1 FROM outbox_entries WHERE bundle_id = ?1)",
)
.bind(&bundle_id)
.fetch_one(&mut *tx)
.await?
.0;

if exists {
    tx.rollback().await?;
    ...
```

**Issue**: A full write transaction (`begin()`) is opened for what is fundamentally a read-check, then rolled back if the bundle exists. In SQLite WAL mode, a write transaction acquires a write lock, which blocks ALL concurrent readers until the transaction ends. When the bundle already exists (the common idempotent retry case), the code needlessly holds a write lock during the rollback.

**Recommended fix**:
```rust
// Read-only check first (no lock contention)
let exists: bool = sqlx::query_as::<_, (bool,)>(
    "SELECT EXISTS(SELECT 1 FROM outbox_entries WHERE bundle_id = ?1)",
)
.bind(&bundle_id)
.fetch_one(self.pool.inner())
.await?
.0;
if exists {
    return Ok(None);
}

// Only begin write tx if we know we need to insert
let mut tx = self.pool.inner().begin().await?;
sqlx::query("INSERT INTO outbox_entries ...")
    ...
    .execute(&mut *tx)
    .await?;
tx.commit().await?;
```

**Risk**: Under concurrent sync/pull scenarios (e.g., CLI pull + daemon push), this pattern causes unnecessary serialization and potential timeout under load.

### C2: `mark_failed` — `tx.commit()` before `return Err` has no rollback safety

**File**: `crates/nexus-sync/src/outbox.rs`, `mark_failed()`

```rust
if retry_count >= MAX_RETRIES {
    let now = chrono::Utc::now().to_rfc3339();
    sqlx::query("UPDATE outbox_entries ... SET delivery_state = 'failed' ...")
        .bind(error)
        .bind(&now)
        .bind(outbox_entry_id)
        .execute(&mut *tx)
        .await?;
    tx.commit().await?;
    return Err(SyncError::OutboxMaxRetriesExceeded { ... });
}
```

**Issue**: After `tx.commit().await?`, if the subsequent `return Err(...)` path is taken, the transaction has already been committed. This is **not a bug per se** (the commit is intentional before returning the error), but the pattern is misleading. The original rusqlite code had the same semantics (commit before returning error), so this is a faithful port. However, there is an **implicit risk**: if `commit()` succeeds but the return-path code panics before returning, the transaction state is ambiguous.

**Severity**: This is a **pattern readability** issue rather than a data correctness bug. The commit-before-return semantics are correct. Marking as Critical because this pattern, if copied elsewhere (e.g., by future developers who assume the `?` macro handles it), could cause silent double-commits.

**Recommendation**: Add a clarifying comment or restructure:
```rust
// commit first, then return error — this is intentional so the 'failed' state
// is persisted even though the caller receives an error.
tx.commit().await?;
return Err(...);
```

### C3: CLI `open_workspace_db` creates a new pool per command invocation — PRAGMA overhead

**File**: `crates/nexus42/src/commands/manuscript.rs`, `open_workspace_db()`

```rust
async fn open_workspace_db(workspace_root: &std::path::Path) -> Result<nexus_local_db::SqlitePool> {
    let pool = crate::db::Schema::init(&db_path).await?;  // opens pool + migrations + seed_versions
    Ok(pool)
}
```

This is called from `manuscript run()`, which may invoke `set_phase`, `promote`, `verify` in sequence — **each call opens a new pool**. The same pattern exists in `research.rs` (`list_references`, `extract_references`, `cache_scan_results_at`).

**Impact**: Each pool creation runs `PRAGMA journal_mode = WAL` and `PRAGMA foreign_keys = ON`. While these are idempotent, opening and closing a fresh pool for each database operation in a single CLI invocation is wasteful and could cause file-lock contention.

**Recommended fix**: Pass a single pool through the call chain for commands that make multiple DB calls:
```rust
// In manuscript::run()
let pool = open_workspace_db(&workspace_root).await?;
manager.set_phase(&title, &phase, &pool).await?;
```

**Severity**: Not a data correctness issue, but a **scalability and correctness** concern — multiple pools on the same DB file could cause `SQLITE_BUSY` under edge cases, especially if migrations run concurrently.

## Warnings

### W1: `nexus42d::db::pool.rs` and `nexus_local_db::open_pool` both set PRAGMAs — redundant execution

Both `DbPool::new()` (lines 149-154) and `nexus_local_db::open_pool()` (lines 102-107) execute:
```rust
sqlx::query("PRAGMA journal_mode = WAL").execute(&pool).await?;
sqlx::query("PRAGMA foreign_keys = ON").execute(&pool).await?;
```

When `WorkspaceState::init()` calls `Schema::init()` (which calls `nexus_local_db::init_pool()` → `open_pool()` + PRAGMAs) and then `DbPool::new()` (which also sets PRAGMAs), the pragmas are set **twice** on the same database file.

**Impact**: Harmless (PRAGMAs are idempotent), but confusing and violates DRY. The daemon creates two separate pools on the same file, each setting pragmas independently.

**Recommendation**: Set PRAGMAs in exactly one place — either in `nexus_local_db::open_pool()` (recommended as the shared entry point) or remove them from `DbPool::new()` and rely on `Schema::init()` having already done so.

### W2: `PoolStatus` lost `available` and `waiting` metrics without migration path

**File**: `crates/nexus42d/src/api/handlers/monitoring.rs`

The old `deadpool-sqlite::Status` provided `available` (idle connections) and `waiting` (checkout queue depth). These fields were silently dropped from `PoolStatus` and the JSON response.

**Impact**: If any external monitoring tool or dashboard consumes `/v1/local/monitoring/pool`, it will receive a changed response shape (two fields removed). No versioning or deprecation notice was added.

**Recommendation**: Either (a) document this as a breaking API change, (b) provide approximate equivalents (sqlx has `size()` but not `available`/`waiting`), or (c) add a note to the API changelog.

### W3: `read_versions()` error reporting loses the actual bad value

**File**: `crates/nexus-local-db/src/lib.rs`, `read_versions()`

```rust
.parse::<u32>()
.map_err(|e| LocalDbError::InvalidVersionValue {
    key: "db_schema_version".to_string(),
    value: "".to_string(),  // <-- loses the actual string that failed to parse
    reason: e.to_string(),
})?;
```

The `value` field is hardcoded to `""`. The previous version captured the actual string from the database.

**Recommendation**: Capture the parsed string before calling `.parse()`:
```rust
let raw = db_schema_version_row.0;
raw.parse::<u32>().map_err(|e| LocalDbError::InvalidVersionValue {
    key: "db_schema_version".to_string(),
    value: raw.clone(),
    reason: e.to_string(),
})?
```

### W4: `validate()` no longer checks `schema_version` key existence

**File**: `crates/nexus-local-db/src/lib.rs`, `validate()`

The old `validate()` function verified:
1. `workspace_meta` table exists
2. `db_schema_version` key exists
3. `schema_version` key exists
4. Both values are valid u32

The new `validate()` only calls `read_versions()` and checks `db_schema_version` against `DB_SCHEMA_VERSION`. While `read_versions()` does verify `schema_version` key exists (via `ok_or_else`), it does **not** check if `schema_version` matches `SCHEMA_VERSION` — the old code didn't either, but it at least validated parseability.

**Impact**: A corrupted `schema_version` value (e.g., `"not_a_number"`) would pass `validate()` in the new code because `read_versions()` fails on `db_schema_version` first and never reaches `schema_version`.

**Severity**: Low risk in practice (both keys are seeded together), but the validation surface has shrunk.

### W5: `nexus42d/src/db/schema.rs` — `Schema::init()` returns a `SqlitePool` but callers may ignore it

**File**: `crates/nexus42d/src/db/schema.rs`

```rust
pub async fn init(db_path: &std::path::Path) -> Result<sqlx::SqlitePool, nexus_local_db::LocalDbError> {
    nexus_local_db::init_pool(db_path).await
}
```

This returns a pool, but in `WorkspaceState::init()` (`workspace/mod.rs`), the returned pool is immediately dropped and a new `DbPool::new()` is created:
```rust
crate::db::schema::Schema::init(&db_path).await?;  // pool returned here is dropped
let db = DbPool::new(&db_path, PoolConfig::from_env()).await?;
```

**Impact**: This opens **two** pools on the same database file in sequence. The first pool (from `Schema::init`) is immediately dropped.

**Recommendation**: Either reuse the pool from `Schema::init()` or change `Schema::init()` to not return a pool (e.g., `-> Result<(), ...>`). Given that `Schema::init()` calls `nexus_local_db::init_pool()` (open + migrate + seed), the caller should reuse this pool.

## Suggestions

### S1: Consider deriving `FromRow` for more structs instead of positional tuple access

Many handlers use `sqlx::query_as::<_, (String, String, ...)>()` with positional `.0`, `.1` access:

```rust
let row: (String, String, String, String, String) = sqlx::query_as(...)
    .fetch_optional(pool)
    .await?;
let expires_at = DateTime::parse_from_rfc3339(&row.3)...
```

Files: `token_manager.rs`, `memory.rs`, `sessions.rs`, `sync.rs`, `identity.rs`, `soul_meta.rs`, `memory_fragment.rs`, `pending_review.rs`

**Recommendation**: Add `#[derive(sqlx::FromRow)]` to existing or new struct definitions (like `CreatorInfo`, `PendingReviewInfo`, `ReferenceInfo`, `SessionInfo` already do). This makes the code self-documenting and resilient to column reordering.

### S2: Consider sqlx offline mode for CI

The migration removed `crates/nexus-local-db/src/migration.rs` (287 lines) in favor of `sqlx::migrate!()`. For CI environments without a running SQLite instance, sqlx requires offline mode (`sqlx prepare`).

**Check**: The CI workflow (`.github/workflows/ci.yml`) was updated to include `sqlx offline` in the WS8 T8 commit. Ensure `crates/nexus-local-db/.sqlx/` directory with query metadata files is committed.

### S3: `nexus42d::db::pool.rs` test `create_test_pool()` runs migrations + creates a second pool

```rust
async fn create_test_pool() -> (tempfile::TempDir, std::path::PathBuf, DbPool) {
    let pool = nexus_local_db::open_pool(&db_path).await.unwrap();
    nexus_local_db::run_migrations(&pool).await.unwrap();
    nexus_local_db::seed_versions(&pool).await.unwrap();

    let db_pool = DbPool::new(&db_path, ...).await.expect("...");
    (tmp, db_path, db_pool)
}
```

This opens a pool, runs migrations, seeds, then opens a **second** pool (`DbPool::new`). Consider using `nexus_local_db::init_pool()` for a single-step setup, or wrapping `DbPool::new` to accept an already-migrated path.

### S4: Hardcoded placeholder style `?1` vs `$1`

The codebase consistently uses SQLite positional parameters (`?1`, `?2`). This is correct for sqlx SQLite (which supports both `?` and `$N`), but `$N` is more idiomatic in the sqlx ecosystem. No functional issue, just a consistency preference.

Due to bash permission constraints, `cargo clippy --all -- -D warnings` could not be executed in this review. The last commit message ("lint fixes for final CI gate") suggests the developer already ran clippy locally. **This should be verified in CI before merge.**

## Positive Observations

1. **DDL centralization**: Replacing 802 lines of inline SQL DDL in `schema.rs` with a 150-line migration file is excellent. The `sqlx::migrate!()` approach provides versioned, idempotent migrations with compile-time embedding.

2. **FromRow derive pattern**: The adoption of `#[derive(sqlx::FromRow)]` for `CreatorInfo`, `PendingReviewInfo`, `ReferenceInfo`, and `SessionInfo` eliminates boilerplate row-mapping closures and is idiomatic sqlx.

3. **Error handling simplification**: Removed the double `.map_err()` pattern that was required with `deadpool-sqlite`'s `InteractError`. The new `sqlx::Error` → `NexusApiError` / `CliError` / `SyncError` conversion is clean.

4. **Test asyncification**: All test fixtures (`create_test_workspace`, `create_test_app`, `setup_pool`) are consistently `async fn` with `#[tokio::test]`. No mixed sync/async test patterns found.

5. **`PooledConn` elimination**: Removing the `PooledConn` wrapper type (~230 lines of async delegation) is a net win. Direct `sqlx::SqlitePool` access is simpler and more maintainable.

6. **Transaction correctness in outbox**: The `append()`, `stage_bundle()`, `mark_sent()`, `mark_acked()`, `mark_failed()` methods all use proper `begin()`/`commit()` patterns. The `mark_failed` read-then-write in a single transaction is correct for the retry-count check.

7. **CLI `open_global_db()` graceful degradation**: The `soul.rs` command returns `Option<Pool>` and degrades gracefully when DB doesn't exist — this is good defensive design.

## Cross-Reviewer Ready Notes

**Integration risk**: Medium. The migration is comprehensive and mechanically consistent — every `rusqlite::Connection` → `sqlx::SqlitePool`, every `conn.interact(...)` → `sqlx::query(...).execute(pool).await`. The risk is in edge cases:
- Multiple pools on same file (W5, W1)
- Transaction lock contention (C1)
- CLI pool-per-operation overhead (C3)

**Migration cost**: High effort to implement, but low risk to revert since the old code is cleanly removed (not commented out). Rollback would require restoring `deadpool-sqlite` + `rusqlite` dependencies and reverting all handler signatures.

**Future evolution**: The sqlx migration positions the codebase for:
- Adding PostgreSQL/MySQL support in the future (sqlx supports all three)
- Compile-time query verification with `sqlx::query!()` macro
- Connection pool metrics via sqlx's `PoolOptions` callbacks
