### 🟡 Warning

#### W-001: AdmissionPolicy not integrated into session creation flow

**Issue**: `HostManager::create_session` does not call `AdmissionPolicy::check` before launching the provider. The `admission` field in `HostManager` is marked `#[allow(dead_code)]` and is never used in the actual session admission gate.

**Location**: `crates/nexus-agent-host/src/core/manager.rs:45-46`, `manager.rs:159-213`

**Fix suggestion**: 
1. In `create_session`, before calling `adapter.launch`, check:
   - `admission.check_provider_allowed(&request.provider_id)`
   - `admission.check_session_capacity()` against configured `max_sessions`
2. Remove `#[allow(dead_code)]` once admission checks are active.
3. Compass §Done #9 states "Policy/admission gates run before provider launch" — currently unmet.

**Impact**: Security/policy bypass — unknown providers may launch sessions without policy evaluation.

---

#### W-002: HostManager::shutdown does not call provider shutdown for active sessions

**Issue**: `HostManager::shutdown` only transitions session states and clears mappings. It never calls `ProviderAdapter::shutdown` for each active session, leaving provider processes potentially orphaned.

**Location**: `crates/nexus-agent-host/src/core/manager.rs:320-346`

**Fix suggestion**:
1. Iterate over `session_providers` and call `adapter.shutdown(handle)` for each session.
2. Wait for configured `shutdown_grace_ms` timeout per session before force-kill.
3. Compass §ACP provider implementation #10 states "Cancel active op if any, wait for configured graceful timeout, terminate child, then force kill if needed" — should apply to host shutdown too.

**Impact**: Resource leak — provider processes may remain running after daemon shutdown.

---

#### W-003: Native CLI probe uses `which` command (Unix-only)

**Issue**: `ClaudeCliProvider::probe` uses `tokio::process::Command::new("which")` to check if the command exists. This is Unix-specific and will fail on Windows where the equivalent is `where`.

**Location**: `crates/nexus-agent-host/src/providers/native_cli/claude.rs:188-191`

**Fix suggestion**:
1. Use a cross-platform approach: check `std::env::var("PATH")` and scan manually, or use the `which` crate.
2. Alternatively, attempt to spawn the command directly with a timeout and check for `ENOENT` / process not found error.

**Impact**: Portability — native provider discovery fails on Windows.

---

#### S-002: OperationRegistry missing from core/

**Issue**: Compass §Crate structure lists `operation.rs` under `core/` for `OpRegistry + op admission/cancel state`, but only `session.rs` exists. Session-level op tracking is in `SessionRegistry::active_op_id`.

**Location**: `crates/nexus-agent-host/src/core/mod.rs` — missing `pub mod operation;`

**Recommendation**: 
1. Wave 1 one-op-per-session constraint makes SessionRegistry-level tracking sufficient.
2. If OpRegistry is needed for future multi-op or cross-session op tracking, add `core/operation.rs` with a stub and document in compass as deferred.

---

#### S-003: No integration tests for ACP provider lifecycle

**Issue**: `providers/acp.rs` tests cover only content block conversion and stream update translation. No tests for `launch`, `execute`, or `shutdown` with a mock/fixture ACP SDK.

**Location**: `crates/nexus-agent-host/src/providers/acp.rs:353-439`

**Recommendation**: 
1. Add a fixture/mock `NexusAcpClient` implementation for integration tests.
2. Test the full `probe → launch → execute → shutdown` lifecycle.
3. Compass §M3 requires "ACP provider implemented with fake/fixture provider tests and deterministic event streams" — currently partial.

---

#### S-004: HostEventStream wrapper in exec relies on correct terminal event emission

**Issue**: `HostManager::exec` wraps the provider stream to transition `Busy → Ready` on `OpFinished` or `OpFailed`. This assumes providers always emit exactly one terminal event. If a provider stream ends without terminal event, session remains in `Busy` state.

**Location**: `crates/nexus-agent-host/src/core/manager.rs:253-266`

**Recommendation**: 
1. Add a safety timeout wrapper that emits `OpFailed` if stream ends without terminal event.
2. Or add a post-stream cleanup check in HostManager to force transition if session is still `Busy`.

## Architecture Assessment

### Positive Observations

1. **Crate structure matches compass**: `lib.rs` with facade traits, `core/` for manager/session, `capability/` for DTOs, `discovery/` for catalog, `policy/` for gates, `providers/` for adapters, `telemetry/` for events — exactly as specified.

2. **Trait boundaries are narrow and correct**: 
   - `HostFacade` is consumed by daemon-runtime only
   - `ProviderAdapter` is implemented by ACP and native providers
   - `ProviderDiscovery` is separate from adapters
   - No trait bloat or inappropriate exposure.

3. **Layering is clean**:
   - `nexus-daemon-runtime` → `HostFacade` (Arc<dyn HostFacade>)
   - `nexus-agent-host` → `nexus-acp-host` (ACP adapter uses existing SDK)
   - No circular dependencies or inappropriate downward imports.

4. **R-002 resolution correctly implemented**: `HostOperation` contains only `Prompt`, `SetModel`, `SetMode`. Cancel and Health are control-plane on `HostFacade` — matches compass PM review note R-002.

5. **Session state machine matches compass**: All states (`Created → Starting → Ready → Busy(op_id) → Ready → Stopping → Stopped`, plus `ErrorRecoverable/ErrorTerminal`) and transitions are implemented with correct source state validation.

6. **Error taxonomy is well-structured**: `HostError` variants map compass §Error handling with context fields (`provider_id`, `session_id`, `op_id`, `category`, `message`, `source`) and `.with_provider/.with_session/.with_op` context attachment methods.

7. **CapabilityDescriptor correctly models negotiated features**: `acp_full()` and `native_cli_limited()` constructors match compass §Capability negotiation.

### Maintainability Concerns

1. **W-001**: Admission policy integration gap — policy gates exist but are not invoked, reducing future safety enforcement complexity.

2. **W-002**: Shutdown incomplete — missing provider cleanup may require post-shutdown orphan process recovery.

3. **S-002**: OpRegistry deferred to future — acceptable for Wave 1 but adds future refactoring scope.
