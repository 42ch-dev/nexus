**Scope**: 57 files, +4,204 / −196 lines across `crates/nexus-orchestration/`, `crates/nexus42d/`, `crates/nexus-contracts/`, `crates/nexus-local-db/migrations/`.

## Executive Summary

This review finds **3 Critical**, **5 High**, **6 Medium**, and **4 Low/Nit** issues. The most severe problems are:

1. **Daemon panic via malicious worker** (`transport.rs:64`) — an oversized NDJSON line causes an unconditional `expect()` panic, crashing the daemon.
2. **Blind JSON-RPC response acceptance** (`ipc.rs:93`) — hardcoded `id: 1` with no request/response correlation means a malicious or buggy worker can inject arbitrary responses.
3. **Lifecycle shutdown is a no-op for real subsystems** (`main.rs:333-348`) — the `StatigLifecycle` `Stopping.entry` action only shuts down mock stubs, leaving the real orchestration engine and worker manager running.

## Findings

### Critical

#### C1 — Transport panic on oversized NDJSON line (daemon DoS)
- **File**: `crates/nexus-orchestration/src/worker/transport.rs:64`
- **Code**:
  ```rust
  async fn recv(&mut self) -> Option<String> {
      self.reader
          .next()
          .await
          .map(|r| r.expect("LinesCodec error"))
  }
  ```
- **Issue**: `LinesCodec` is configured with `max_length: 1_024 * 1_024` (1 MiB). If a malicious or buggy worker emits a line longer than 1 MiB, `LinesCodec` returns `Err(LengthDelimitedCodecError { .. })`, and the `.expect()` **panics the entire daemon**.
- **Impact**: Remote DoS via worker subprocess (any compromised ACP agent or worker binary can crash the supervisor).
- **Fix**: Replace `expect` with proper error handling — return `None` or a `Result` that propagates to the caller.
- **Cross-check**: Same pattern at `transport.rs:117` in `DuplexTransport`.

#### C2 — Hardcoded JSON-RPC request ID with no response correlation
- **File**: `crates/nexus-orchestration/src/worker/ipc.rs:93`
- **Code**:
  ```rust
  let request = serde_json::json!({
      "jsonrpc": "2.0",
      "id": 1,   // <-- hardcoded
      "method": method,
      "params": params,
  });
  ```
- **Issue**: Every JSON-RPC request uses `id: 1`. The response parser checks for `"error"` and `"result"` but **never validates that the response `id` matches the request `id`**. A malicious worker (or a second concurrent request, if added later) can inject arbitrary responses that will be blindly accepted.
- **Impact**: Response injection / spoofing. A worker could reply with a fabricated result for `worker/shutdown` while ignoring an actual `worker/acp_prompt` request.
- **Fix**: Generate monotonic or random request IDs; maintain a pending-response map; validate the `id` field before accepting the result. The module comment says "A full IpcClient … will be layered on top in WS3" — but WS2 already exposes this to real worker processes, so the one-shot helper must at minimum not be exploitable.

#### C3 — Lifecycle shutdown does not shut down the real engine or workers
- **File**: `crates/nexus42d/src/main.rs:333-348`
- **Code**:
  ```rust
  fn create_subsystems(...) -> Vec<Arc<dyn SubsystemBootstrap>> {
      vec![
          Arc::new(HttpSubsystem::new(port)),
          Arc::new(DbSubsystem::new(...)),
          Arc::new(SyncSubsystem::new()),
          Arc::new(EngineSubsystem::new()),   // Mock - WS2 will provide real impl
          Arc::new(WorkerMgrSubsystem::new()), // Mock - WS2 will provide real impl
      ]
  }
  ```
- **Issue**: The real `GraphFlowEngine` and `WorkerManager` are instantiated in `main()` and wired into `WorkspaceState`, but the `StatigLifecycle` `Stopping.entry` action (`actions.rs:190-260`) iterates `create_subsystems()` and calls `shutdown()` on each. The Engine and WorkerMgr entries are **mock stubs** that do nothing. Therefore, when the daemon receives SIGTERM:
  1. The orchestration engine keeps running (sessions not cancelled).
  2. Worker processes are not shut down via IPC.
  3. The HTTP server never receives a real graceful shutdown signal (`main.rs:241` uses `std::future::pending()`).
- **Impact**: Data loss or corruption — active engine sessions are killed mid-step; workers become orphaned zombies.
- **Fix**: Either (a) register the real engine and worker manager as lifecycle subsystems, or (b) have `Stopping.entry` reach into `WorkspaceState` to shut them down. The spec (`orchestration-engine.md` §6.1) explicitly says "lifecycle HSM `Stopping` state sends `Cancel` to every active session before shutting the engine down."

### High

#### H1 — HTTP server graceful shutdown is never triggered
- **File**: `crates/nexus42d/src/main.rs:238-243`
- **Code**:
  ```rust
  axum::serve(listener, app)
      .with_graceful_shutdown(async {
          std::future::pending::<()>().await;
      })
      .await?;
  ```
- **Issue**: `std::future::pending()` never resolves. The HTTP server will run until the tokio runtime is dropped (i.e., process exit). In-flight requests may be aborted mid-stream.
- **Impact**: Unclean HTTP termination; API clients receive connection resets instead of drained responses.
- **Fix**: Drive this from the lifecycle HSM — await a `tokio::sync::Notify` or `CancellationToken` that `Stopping.entry` fires.

#### H2 — `get_session` endpoint cannot find terminal sessions
- **File**: `crates/nexus42d/src/api/handlers/orchestration/sessions.rs:79-87`
- **Code**:
  ```rust
  let sessions = engine
      .list_active(nexus_orchestration::engine::SessionFilter::default())
      .await
      ...
  ```
- **Issue**: `get_session` calls `list_active`, which filters for non-terminal sessions (`Running | Paused | WaitingForInput`). A session that has `Completed` or `Failed` will return `404 NOT_FOUND` even though it exists in storage. The spec (`acp-client-tech-spec.md` §4.3) says this endpoint returns "Full session state" — that implies all sessions.
- **Impact**: Users cannot inspect finished sessions; debugging and audit workflows broken.
- **Fix**: Add `Engine::get_session(&self, id) -> Option<SessionSummary>` that queries storage directly, or modify `list_active` to accept a flag for including terminal sessions.

#### H3 — System preset uses isolated CapabilityRegistry instead of daemon's registry
- **File**: `crates/nexus-orchestration/src/system_preset.rs:120`
- **Code**:
  ```rust
  pub fn build() -> Arc<Graph> {
      let registry = Arc::new(CapabilityRegistry::with_builtins());
      ...
  }
  ```
- **Issue**: `build()` constructs a **new** `CapabilityRegistry` rather than accepting the one wired into the daemon. If the daemon registers additional capabilities after startup (e.g., platform-provided caps), the system preset graph will not see them. Also, any capability state (caches, counters) is not shared.
- **Impact**: State divergence; system preset capabilities run in a different registry instance than HTTP-triggered capabilities.
- **Fix**: Change `build()` to accept `Arc<CapabilityRegistry>`.

#### H4 — Worker shutdown does not wait for clean exit and skips SIGTERM
- **File**: `crates/nexus-orchestration/src/worker/manager.rs:170-204`, `282-305`
- **Issue**:
  1. `WorkerHandle::shutdown()` sends the `worker/shutdown` RPC but returns `Ok(())` immediately — the caller has no way to await the process exit.
  2. The supervisor task (`manager.rs:282-305`) on cancellation timeout calls `child.start_kill()` directly. `start_kill()` sends **SIGKILL** (Unix) or terminates immediately (Windows). The spec (`orchestration-engine.md` §6.1) says "`SIGTERM` → `SIGKILL` path" — but the code jumps straight to SIGKILL.
- **Impact**: Workers are killed abruptly; ACP sessions may be left in an inconsistent state; no graceful termination window.
- **Fix**: Add a `SIGTERM` step (send via `nix::sys::signal` on Unix, or a configurable mechanism) before `start_kill()`, and expose an async `shutdown_and_wait(&mut self, timeout) -> Result<ExitStatus, …>` on `WorkerHandle`.

#### H5 — `call_json_rpc` consumes stdio pipes, making handle single-use
- **File**: `crates/nexus-orchestration/src/worker/manager.rs:142-164`
- **Code**:
  ```rust
  pub async fn call_json_rpc(&mut self, ...) -> Result<Value, WorkerError> {
      let stdin = self.stdin.take().ok_or_else(...)?;
      let stdout = self.stdout.take().ok_or_else(...)?;
      ...
  }
  ```
- **Issue**: Every call to `call_json_rpc` consumes the stdin/stdout `Option`s. After one call, subsequent calls (or shutdown) fail with `"stdin already consumed"`. The WS2 test (`worker_ipc_roundtrip.rs`) only makes one call, so it passes, but any real multi-RPC worker session will break.
- **Impact**: Worker handle is effectively single-use; violates the IPC catalogue spec which expects `worker/initialize`, then multiple `worker/acp_prompt`, then `worker/shutdown`.
- **Fix**: Hold an `Arc<tokio::sync::Mutex<StdioTransport>>` or wrap the transport in a reusable client (as the module comment foreshadows for WS3).

### Medium

#### M1 — Session ID collision risk under rapid creation
- **File**: `crates/nexus-orchestration/src/engine.rs:213`
- **Code**:
  ```rust
  let session_id = format!("{}:{}", preset_id, chrono::Utc::now().timestamp_millis());
  ```
- **Issue**: Millisecond timestamps can collide if two sessions for the same preset are started within the same millisecond (e.g., in a tight loop or concurrent requests). No retry or collision detection exists.
- **Fix**: Use ULID or UUIDv7, or append a random suffix.

#### M2 — `SqliteSessionStorage` reads back incomplete session metadata
- **File**: `crates/nexus-orchestration/src/storage/sqlite.rs:96-124`
- **Code**: The `get()` query only selects `session_id`, `current_task_id`, `context_json`. It does **not** read back `creator_id`, `preset_id`, `status`, `created_at`, `updated_at`, `parent_session_id`.
- **Issue**: When a session is loaded after daemon restart, the reconstructed `Session` has `graph_id: "default"` (hardcoded) and loses all metadata. The engine's in-memory `SessionSummary` list is also lost on restart (it's not rebuilt from DB).
- **Impact**: Crash recovery and daemon restart resume are broken — sessions lose identity and status.
- **Fix**: Select all columns in `get()`; reconstruct `SessionSummary` from the DB row; rebuild the in-memory sessions list on engine startup.

#### M3 — `new_session` creates session with empty current_task_id
- **File**: `crates/nexus-orchestration/src/engine.rs:284-302`
- **Code**:
  ```rust
  let session = graph_flow::Session::new_from_task(session_id.clone(), "");
  ```
- **Issue**: The empty string `""` as the current task ID is not a valid graph task. `FlowRunner::run()` may fail or behave unexpectedly when it tries to look up task `""`.
- **Fix**: Require a valid start task ID (e.g., from the graph's `start_task_id()`), or make `new_session` accept a graph parameter.

#### M4 — Blocking operations in async context (Unix socket & failed state)
- **File**: `crates/nexus42d/src/main.rs:247-255`, `crates/nexus42d/src/lifecycle/actions.rs:278`
- **Code**:
  ```rust
  // main.rs
  if path.exists() { std::fs::remove_file(&path)?; }
  if let Some(parent) = path.parent() { std::fs::create_dir_all(parent)?; }
  
  // actions.rs
  std::thread::sleep(std::time::Duration::from_millis(100));
  ```
- **Issue**: Both use blocking std APIs inside async Tokio tasks. Under high load or slow filesystems, this blocks the async runtime thread.
- **Fix**: Use `tokio::fs` equivalents.

#### M5 — `nexus42d/Cargo.toml` dependency hygiene issues
- **File**: `crates/nexus42d/Cargo.toml:45`, `46`
- **Issues**:
  1. `tempfile = "3"` is in `[dependencies]`, not `[dev-dependencies]`. It is only used in tests/test-support code. Bloats release binary.
  2. `async-trait = "0.1"` is a direct dependency; the workspace already declares `async-trait`. Should use `{ workspace = true }`.
  3. `toml = "0.8"` is a direct dependency; check if workspace has it.
- **Fix**: Move `tempfile` to `[dev-dependencies]`; use workspace deps where available.

#### M6 — Panic hook always reports `SubsystemKind::Engine`
- **File**: `crates/nexus42d/src/main.rs:206-215`
- **Code**:
  ```rust
  lifecycle_for_panic.dispatch(Event::FatalError {
      kind: SubsystemKind::Engine, // Generic subsystem
      err: info.to_string(),
  });
  ```
- **Issue**: Regardless of which subsystem panicked, the lifecycle is told `SubsystemKind::Engine`. This makes degraded-state reporting and post-mortem analysis inaccurate.
- **Fix**: At minimum, use `SubsystemKind::Unknown` or try to detect the thread name. Better: thread-local panic context.

### Low / Nit

#### L1 — Engine started before lifecycle transition (spec divergence)
- **File**: `crates/nexus42d/src/main.rs:142-156`
- **Issue**: The `_system.maintenance` session is started in `main.rs` **before** the lifecycle HSM is created or transitioned to `Running`. The spec (`orchestration-engine.md` §9.1) says "Enters engine when statig lifecycle transitions `Running → entry`." The comment in `actions.rs:135-139` acknowledges this divergence: "Engine sessions … are started in main.rs before the lifecycle transitions to Running."
- **Impact**: Low — the engine is functional, but lifecycle observers won't see the causal ordering. May confuse future WS4 integration tests.
- **Fix**: Move engine start into `Running.entry` action (requires refactoring `WorkspaceState` to be accessible from the action context).

#### L2 — Orchestration routes lack auth middleware (documented, but verify)
- **File**: `crates/nexus42d/src/api/mod.rs:174-194`
- **Issue**: The orchestration routes are merged without `.route_layer(auth_middleware::require_auth)`. The spec says these are "local-only HTTP" and the code comment agrees. This is acceptable for V1.4, but should be explicitly revisited when platform UI consumes these endpoints.
- **Impact**: Low — local loopback only.
- **Fix**: Add a code comment or TODO referencing the future auth requirement.

#### L3 — `jsonrpsee` dependency in `nexus-orchestration` but unused in WS2
- **File**: `crates/nexus-orchestration/Cargo.toml:20`
- **Issue**: The crate depends on `jsonrpsee` (per crate-selection spec §3.1), but the actual IPC implementation uses hand-rolled JSON-RPC via `serde_json` + `LinesCodec`. `jsonrpsee` types/macros are not imported anywhere in the WS2 code.
- **Impact**: Nit — unnecessary dependency bloat until WS3 wires it in.
- **Fix**: Either (a) remove `jsonrpsee` from WS2 and add it in WS3, or (b) add a module comment explaining it's pre-wired for WS3.

#### L4 — `SignalSessionRequest.signal` lacks case normalization
- **File**: `crates/nexus42d/src/api/handlers/orchestration/sessions.rs:113-124`
- **Issue**: Signal matching is exact lowercase: `"pause"`, `"resume"`, etc. `"Pause"` or `"PAUSE"` returns `400 BAD_REQUEST`. Most HTTP APIs are case-insensitive for enum-like values, or at least accept common variants.
- **Impact**: Nit — minor UX friction.
- **Fix**: Normalize with `body.signal.to_ascii_lowercase()` before matching.
