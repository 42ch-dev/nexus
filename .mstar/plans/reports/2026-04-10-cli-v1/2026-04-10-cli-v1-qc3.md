#### QC3-W1: Rust codegen emits `serde_json::Value` for `$ref` string-pattern fields (TS codegen is correct)

**Severity**: Warning (medium ΓÇö runtime type safety degraded; works but not type-safe)

**Files Affected**:
- `crates/nexus-contracts/src/generated/publish_story_request.rs` (lines 16, 18)
- `crates/nexus-contracts/src/generated/publish_history_request.rs`
- `crates/nexus-contracts/src/generated/publish_history_response.rs`

**Evidence**:
```rust
// Generated Rust (incorrect - uses Value)
pub struct PublishStoryRequest {
    pub manuscript_id: serde_json::Value,        // line 16
    pub story_manifest_id: Option<serde_json::Value>, // line 18
}

// Generated TypeScript (correct - uses string)
export interface PublishStoryRequest {
  manuscript_id: string;        // line 14
  story_manifest_id?: string;   // line 15
}
```

**Root Cause**: The JSON Schema `publish-story-request.schema.json` references `common.schema.json#/definitions/ManuscriptId` via `$ref`, but the Rust codegen does not follow the `$ref` chain when the target is a `string` with a `pattern`. It emits `serde_json::Value` instead of `String`.

**Impact**: 
- Rust consumers must use `.as_str().unwrap_or("")` or similar workarounds (see `crates/nexus42d/src/api/handlers/publish.rs` line 30-32: `manuscript_id_str()` helper)
- Type safety is lost ΓÇö any JSON that deserializes can pass validation even if it doesn't match the `pattern` regex
- CLI code uses `serde_json::Value::String(...)` casting (line 106-107 in `publish.rs`)

**Suggested Fix**: Update `tooling/codegen/src/rust-generator.ts` to:
1. Follow `$ref` chains to common schema definitions
2. When the resolved type is `string` with `pattern`, emit `String` instead of `serde_json::Value`

**Note**: This was partially acknowledged in QC-self report (`PUBLISH-CODEGEN-01`) but flagged as "Suggestion" rather than Warning since it was tracked as a known residual. I elevate this to Warning because it represents a **cross-language inconsistency** between Rust and TypeScript codegen that could cause platform interoperability issues.

#### QC3-W2: `SyncClient` auth token validation is overly restrictive

**Severity**: Warning (low ΓÇö may cause legitimate tokens to be rejected)

**File**: `crates/nexus-sync/src/sync_client.rs` (lines 57-76)

**Evidence**:
```rust
fn validate_auth_token(token: &str) -> SyncResult<()> {
    // ... enforces 64+ chars and only alphanumeric, hyphens, underscores, dots
    for c in token.chars() {
        if !c.is_ascii_alphanumeric() && c != '-' && c != '_' && c != '.' {
            return Err(SyncError::AuthTokenInvalid(...));
        }
    }
}
```

**Issue**: Tokens formatted with `+`, `=`, or `/` (common in base64 encoding) would be rejected. If the platform issues JWT tokens or other base64-variant tokens, they would fail validation.

**Current Usage**: `MIN_AUTH_TOKEN_LENGTH = 64` ΓÇö this appears to be checking for a specific token format, but:
1. No evidence the platform issues tokens in this exact format
2. If the platform uses OAuth2/JWT, the token format may include characters beyond the allowed set

**Suggested Fix**: Either:
1. Document the exact token format expected and verify platform issues tokens matching this pattern
2. Loosen validation to allow base64 characters (`+`, `/`, `=`) if JWT/OAuth tokens are used
3. Add configuration to disable validation for specific deployment scenarios
