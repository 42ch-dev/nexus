## Architecture & Maintainability Review

### Positive Patterns

1. **Layered Architecture**: Clear separation between `SyncClient` (HTTP), daemon handlers (local API proxy), and CLI commands (user interface). Each layer has a single responsibility.

2. **Schema-Driven Development**: All wire types flow from JSON Schema → codegen → generated Rust/TypeScript. No handwritten DTOs in the critical path.

3. **Consistent Error Handling**: `SyncClient` returns `SyncResult<T>` with well-defined error variants (`SyncError::PlatformError`, `SyncError::AuthTokenInvalid`, etc.). Daemon handlers map these to local RPC responses consistently.

4. **Test Pyramid**: Good coverage at the `SyncClient` level with wiremock, testing both success and error paths (401, 422).

5. **Validation Functions**: Input validators (e.g., `validate_scope`, `validate_manuscript_id`) are pure functions with clear error messages, making them easy to test and reason about.

### Areas of Concern

1. **Daemon Proxy Pattern**: The daemon handlers (`POST /v1/local/explore/browse`, etc.) are thin wrappers that call `SyncClient` methods. This is intentional (daemon mediates platform auth), but creates a duplication pattern: every new platform API requires (a) schema, (b) `SyncClient` method, (c) daemon handler, (d) CLI command. Consider whether a codegen approach could reduce (b) and (c).

2. **Env Var Dependency**: CLI commands depend on `NEXUS_SYNC_PLATFORM_URL` and `NEXUS_SYNC_PLATFORM_TOKEN` being set in the daemon process. This is documented but not validated at CLI startup. Users may get opaque errors if the daemon lacks these.

3. **Generated Code Drift**: The codegen pipeline is critical infrastructure. If `tooling/codegen` diverges from schema semantics (e.g., array-of-$ref handling), both Rust and TypeScript generated output will be affected. The recent fix to `ts-generator.ts` (lines 47+) is a good catch, but this area needs ongoing vigilance.

4. **Residual Tracking**: Two nits are tracked (`FORK-SNAP-01`, `PUBLISH-CODEGEN-01`) regarding `serde_json::Value` usage where typed structs would be preferable. These are codegen limitations, not bugs, but should be addressed before V1.2 to improve type safety.
