### 🟡 Warning
- **F-QC3-001: `desktop-release.yml` can double-run a tag release and then clobber already-uploaded assets.** The workflow triggers on both tag pushes and `release.published`. In the tag-push path, the job builds for up to 75 minutes and then runs `gh release create "$GITHUB_REF_NAME" ...` with release assets. Publishing that release can emit the `release: published` event for the same tag, causing a second macOS build for the same release. The second path then runs `gh release upload ... --clobber`, and `gh release upload --help` states that `--clobber` deletes existing assets before re-uploading them; if upload fails, original assets are lost. This undermines both stated goals: release reliability and reduced redundant macOS usage. It also interacts poorly with `concurrency.cancel-in-progress: true` on `desktop-release-${{ github.ref }}`, because duplicate runs for the same tag can cancel one another or replace assets mid-release.
  - **Fix:** Make the release workflow single-source per release. Prefer one trigger path: either (a) tag push builds and creates/uploads the release assets, with no `release: published` trigger, or (b) release publication builds/uploads to an existing release using `github.event.release.tag_name`, with no tag-push `gh release create` path. If both triggers are intentionally retained, add explicit guards so a release generated by the tag-push workflow does not start the `release.published` upload path, and avoid `--clobber` unless the workflow first verifies that it is a deliberate replacement run.

### 🟢 Suggestion
- **S-QC3-001: Add `--verify-tag` to the tag-push `gh release create` command as a defensive release invariant.** The workflow is triggered by `push.tags`, so the tag should already exist; `--verify-tag` would make that assumption explicit and prevent accidental release creation against a default-branch tag if the workflow is later refactored or manually reused.
