### 🟢 Suggestion

#### S1: `refresh_access_token` returns raw API error body verbatim in non-invalid_grant 400 responses

**Source**: `crates/nexus42/src/auth/user_auth.rs`, `refresh_access_token()`

**Details**: When the platform API returns a 400 status that is NOT an `invalid_grant` error, the function returns:
```rust
return Err(CliError::Api {
    status,
    message: text,  // raw API response body
});
```

The `text` variable contains the full raw HTTP response body. If the API returns verbose error details (e.g., stack traces, internal error messages, rate limit details), these would be exposed to the CLI user.

**Scenario**: A 400 response like `{"error": {"code": "rate_limit_exceeded", "message": "Too many requests", "details": {"retry_after": 30}}}` would show the full JSON to the user, potentially revealing internal rate-limiting behavior.

**Impact**: Low — requires a non-invalid_grant 400 from the platform; refresh_token value itself is never leaked.

**Suggested fix**: Return a sanitized error message for non-invalid_grant 400s:
```rust
if status == 400 {
    if let Ok(envelope) = serde_json::from_str::<serde_json::Value>(&text) {
        // ... existing invalid_grant check ...
    }
    return Err(CliError::Api {
        status,
        message: "API error (see server logs for details)".into(),
    });
}
```

#### S2: `serde_json::from_str` error in token response parsing includes parsed JSON in message

**Source**: `crates/nexus42/src/auth/user_auth.rs`, `refresh_access_token()`

**Details**: When parsing the token response succeeds but `serde_json::from_value::<DeviceTokenResponse>(data)` fails (e.g., `access_token` field missing from response), the error message includes the full JSON structure:

```rust
.map_err(|e| CliError::Other(format!("Failed to parse token response: {e}")))?;
```

The `serde_json::Error` display impl includes the JSON that failed to parse. If the API returns a structured error with internal details in the JSON body, this could be exposed.

**Impact**: Low — only occurs on malformed success responses from the platform; refresh_token itself is never in this path (it's in the success parsing which succeeds).

**Suggested fix**: Use `serde_json::from_value::<DeviceTokenResponse>(data).map_err(|e| CliError::Other("Failed to parse token response".into()))` to avoid including the JSON in the error message.

## Security & Correctness Assessment

### ✅ TOCTOU Fix — `device_id.rs`

The `create_new` + retry loop implementation is **correct and atomic**:
- `OpenOptions::new().write(true).create_new(true)` atomically fails if the file exists
- `AlreadyExists` error triggers read of existing content
- Empty file retry (5ms sleep) handles concurrent writer race
- No check-then-act pattern remains

No security issues in the TOCTOU fix.

### ✅ refresh_token Not Leaked in Logs/Errors

- `status()` only prints "Refresh token: present/absent" — never the token value
- `invalid_grant` error path returns `"Refresh token expired. Please run \`nexus42 auth login\` again."` — no token in message
- Token values never appear in `println!` or `log::` calls

### ✅ `invalid_grant` Handling

When the platform returns `invalid_grant`:
1. All tokens are cleared via `store.clear_user_token()?`
2. A user-friendly error is returned (no token value)

This prevents token-fixation attacks where a revoked token could be used.

### ✅ Backward Compatibility

- `UserTokenState` uses `#[serde(default, skip_serializing_if = "Option::is_none")]` for `refresh_token` and `refresh_expires_at` — pre-V1.11 `auth.json` files without these fields deserialize correctly
- `DeviceTokenResponse` uses the same pattern — platform responses without refresh fields parse as `None`
- Tests verify: `user_token_backward_compat_without_refresh_fields`, `device_token_response_backward_compat_without_refresh`

### ✅ `ensure_valid_token` Buffer Logic

The 60-second proactive refresh buffer (`REFRESH_BUFFER_SECS = 60`) is reasonable:
- If token expires within 60s and `refresh_token` exists → proactively refresh
- If token expired and no `refresh_token` → error with clear message
- If token valid → return as-is (no unnecessary refresh)

### ✅ Auth Store Double-Load (Not a Security Issue)

`refresh_access_token` calls `AuthStore::load()?` three times. While this is suboptimal for performance, it is not a security issue — all loads read the same file, and the file write is atomic.

### ✅ Health Check — `runtime.rs`

The `SELECT 1` query properly checks DB connectivity:
- Returns `HealthStatus::Up` on success
- Returns `HealthStatus::Down` on error
- No sensitive data exposure

### ✅ `test_utils.rs` Extraction

The `seed_valid_token` / `seed_expired_token` functions now accept a `refresh_token` parameter and properly insert all four token fields. The extraction from `auth_middleware.rs` is clean and correct.
