### 🟡 Warning

#### W-T2-1 — `submit_with_retry` resubmits the same wrong answer, wasting attempts
**File**: `crates/nexus42/src/commands/creator.rs`  
**Lines**: 365–408

The doc comment on `submit_with_retry` states:

> "After the first wrong answer, re-solves the challenge and re-submits."

However, the implementation retries with the **identical** `answer` parameter:

```rust
let response = client.verify_creator(verification_code, answer).await?;
```

Because `solve_challenge` is a pure function, retrying the same answer can never succeed. This wastes one of the user’s limited verification attempts and accelerates progression toward the `Locked` state.

**Impact**: Correctness defect; user-facing harm (unnecessary attempt consumption).  
**Fix**: Either (a) remove the retry loop since it provides no value for a deterministic solver, or (b) if the intent was to allow platform jitter/new challenge generation, fetch a new challenge before re-submitting. Update the doc comment to match the actual behavior.

---

#### W-T2-2 — Verification HTTP errors are misreported as registration failures
**File**: `crates/nexus42/src/errors.rs`  
**Lines**: 329–343

The `From<nexus_sync::errors::SyncError> for CliError` impl maps **all** `PlatformError` variants to `CreatorRegistrationFailed`:

```rust
nexus_sync::errors::SyncError::PlatformError { status, body } => {
    CliError::CreatorRegistrationFailed { status, message: body }
}
```

This conversion is used for both `register_creator()` and `verify_creator()`. When the verification endpoint returns a 4xx/5xx, the user sees:

> "Creator registration failed (HTTP 500): ..."

instead of a verification-specific error.

**Impact**: Misleading UX; diagnostic confusion.  
**Fix**: Make the conversion context-aware. The simplest fix is to stop using `?` + `From` inside `submit_with_retry` and instead map the error explicitly to `CreatorVerificationFailed` when the call is `/api/v1/creators/verify`.

### 🟢 Suggestion

#### S-T2-1 — Misleading test name and coverage gap for HTTP-error mapping
**File**: `crates/nexus42/src/commands/creator.rs`  
**Lines**: 722–741

The test `sync_http_error_maps_to_cli_network_error` does **not** test HTTP error mapping. The comment explains that constructing a `reqwest::Error` is difficult, so the test falls back to verifying `PlatformError` mapping again (already covered by the preceding test).

**Fix**: Rename the test to reflect what it actually verifies (e.g., `sync_platform_error_falls_back_to_creator_registration_failed`), or delete it to avoid duplication.

---

#### S-T2-2 — `obtain_auth_token` token selection is non-deterministic with multiple creators
**File**: `crates/nexus42/src/commands/creator.rs`  
**Lines**: 414–427

`obtain_auth_token` iterates `auth_store.creators.values()` and returns the first non-empty `access_token`. Because `HashMap` iteration order is non-deterministic, if multiple creators have tokens the CLI may pick an arbitrary one.

The inline comment acknowledges this is a temporary proxy for V1.3.  
**Fix**: Track a dedicated top-level `user_access_token` field in `AuthStore` so the CLI does not need to guess which creator token to use.

---

#### S-T2-3 — No integration test for the full `register_creator` orchestration
**File**: `crates/nexus42/src/commands/creator.rs`  
**Lines**: 245–363

Tests cover `obtain_auth_token`, error display, `SyncError` conversion, and `submit_with_retry` in isolation, but there is **no test** that exercises the full `register_creator` flow end-to-end (token lookup → platform register → expiry check → challenge solve → verify → credential store → active creator set).

**Fix**: Add a single wiremock-based integration test (or a faked `PlatformClient`) that drives the entire orchestration function and asserts on stdout/stderr and on-disk config state.

---

#### S-T2-4 — `summary` CLI argument is accepted but silently ignored
**File**: `crates/nexus42/src/commands/creator.rs`  
**Lines**: 245–249

The `Register` variant accepts `--summary`, and `register_creator` receives it as `_summary`, but it is never sent to the platform. The design doc §4 shows the request body as `{ display_name, registration_source }`, so this is consistent with the current spec, yet the CLI still exposes the flag to users.

**Fix**: Either wire `summary` into the `PlatformClient::register_creator` call (if the platform API supports it) or remove the CLI flag to avoid user confusion.
