## Warnings

### W1 — `submit_with_retry` doc comment claims re-solve, but code re-submits the same answer

**File**: `crates/nexus42/src/commands/creator.rs`, lines 365–369

The doc comment states:
```rust
/// After the first wrong answer,
/// re-solves the challenge and re-submits.
```

But the actual loop body re-submits the **same** `answer` string — there is no call to `solve_challenge` inside the retry loop:

```rust
for attempt in 1..=max_attempts {
    let response = client.verify_creator(verification_code, answer).await?;
    // ... no re-solve
}
```

**Impact**: The doc misleads future maintainers about the retry semantics. If the platform issues a *new* challenge after a wrong answer, re-submitting the old answer would be incorrect. If the platform keeps the *same* challenge, the doc is inaccurate about the mechanism.

**Fix**: Either (a) call `solve_challenge` again in the retry path (if the challenge text changes per attempt), or (b) update the doc comment to say "re-submits the same answer" (if the challenge is static per verification session).

### W2 — `SyncError::PlatformError` maps all HTTP errors to `CreatorRegistrationFailed`, losing semantic distinction between registration-time and verify-time failures

**File**: `crates/nexus42/src/errors.rs`, lines 329–342

The `From<nexus_sync::errors::SyncError> for CliError` impl maps *all* `PlatformError` variants (4xx, 5xx from *any* platform call — register, verify, or future endpoints) to `CliError::CreatorRegistrationFailed`:

```rust
nexus_sync::errors::SyncError::PlatformError { status, body } => {
    CliError::CreatorRegistrationFailed { status, message: body }
}
```

In the register flow, both `client.register_creator()` (step 2) and `client.verify_creator()` (step 5) can produce `PlatformError`. If the verify endpoint returns a 500, the user sees "Creator registration failed (HTTP 500)" — which is misleading since registration already succeeded and verification failed.

**Impact**: User-facing error messages are inaccurate in the verify phase. This is especially problematic for the D4 retry flow — if the verify endpoint is temporarily down, the user gets a "registration failed" suggestion to check `auth status`, which is not the right guidance.

**Fix**: Either (a) introduce a `CreatorVerificationHttpError` variant for verify-phase HTTP errors, or (b) handle `SyncError` at the call site in `register_creator` with context-aware conversion (e.g., `.map_err(|e| match e { ... })`).

## Suggestions

### S1 — `obtain_auth_token` conflates user-level auth with creator-level auth (design concern)

**File**: `crates/nexus42/src/commands/creator.rs`, lines 414–426

The function iterates over `creators` in the `AuthStore` and returns *any* non-empty `access_token` as a proxy for the user token. The inline comment acknowledges this:
```
// In a more complete implementation, we'd have a dedicated user token field.
```

**Risk**: If multiple creators are cached with different tokens, the iteration order of `HashMap` is non-deterministic. The token returned may belong to a creator that was authenticated via a different OAuth flow, not the current user session.

**Recommendation**: Acceptable as a V1.3 workaround, but should be tracked as a residual (`R`-item) with a clear TODO for V1.4 to introduce a proper user-level token field.

### S2 — Missing test: `SyncError::HttpError` → `CliError::Network` mapping

**File**: `crates/nexus42/src/errors.rs`, test `sync_http_error_maps_to_cli_network_error` (line 722)

The test name claims to test `HttpError` → `Network`, but the test body actually creates `SyncError::PlatformError { status: 502, ... }` and asserts `CreatorRegistrationFailed`. The comment explains the difficulty of constructing a `reqwest::Error`, but the test name and assertion do not match.

**Recommendation**: Rename the test to reflect what it actually tests, or construct a `SyncError::HttpError` variant (via `SyncError::from(reqwest::Error)`) to test the intended mapping.

### S3 — Verification code truncation lacks explanatory comment

**File**: `crates/nexus42/src/commands/creator.rs`, lines 271–277

```rust
println!(
    "  Verification code: {}",
    &verification.verification_code[..verification
        .verification_code
        .len()
        .min(16)]
);
```

This safely truncates the code for display, but the reason (avoiding log leakage? UX readability?) is not documented.

**Recommendation**: Add a one-line comment explaining the rationale.

### S4 — `remaining_secs` calculation uses `timestamp()` (loses sub-second precision)

**File**: `crates/nexus42/src/commands/creator.rs`, line 291

```rust
let remaining_secs = (expires_at.timestamp() - now.timestamp()).max(0);
```

`DateTime::timestamp()` returns `i64` (whole seconds). For very short remaining times (< 1s after the 10s buffer), this would show `0` even though the challenge is still valid. Not a correctness issue (the expiry check already passed), but a UX minor nit.

**Recommendation**: Use `timestamp_millis()` or `(expires_at - now).num_seconds()` for slightly more accurate display, or accept this as a cosmetic trade-off.
