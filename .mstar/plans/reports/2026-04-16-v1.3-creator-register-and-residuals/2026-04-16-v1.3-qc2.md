## Warnings

### W1 — Unconditional Unix import breaks cross-platform compilation
**File**: `crates/nexus42/src/auth/mod.rs`  
**Line**: 13  
`use std::os::unix::fs::PermissionsExt;` is imported unconditionally, yet the project already gates Unix-specific deps (`nix`) behind `#[cfg(unix)]`. This will fail to compile on Windows. The `PermissionsExt` usage in `save()` should be `cfg`-gated or replaced with a cross-platform permission helper.

**Impact**: Compilation failure on non-Unix targets.  
**Fix**: Wrap the `set_permissions` call in `#[cfg(unix)]` and leave a TODO/compile-error for Windows.

---

### W2 — Number-word converter does not handle "hundred" compounds
**File**: `crates/nexus42/src/challenge/numbers.rs`  
**Lines**: 61–89  
`convert_number_words` handles `tens + units` (e.g., "thirty five" → 35) but has no logic for "hundred" combinations such as "one hundred twenty three". The map contains `"hundred" → 100`, yet a challenge like *"one hundred apples and someone adds twenty three more"* would emit `"1 100 apples and someone adds 23 more"`, and the parser would extract `1` and `100` instead of `100` and `23`.

**Impact**: Wrong answer for any platform challenge using numbers ≥ 100.  
**Fix**: Extend the compound logic to handle `[units] hundred [tens] [units]` patterns, or document the supported range explicitly and reject/alert on unsupported input.

---

### W3 — Math parser blindly extracts the first two numbers
**File**: `crates/nexus42/src/challenge/parser.rs`  
**Lines**: 42–52  
`extract_math_problem` collects *all* numbers via `filter_map(|w| w.parse::<u32>().ok())` and always takes `numbers[0]` and `numbers[1]`. If the platform ever inserts an extra number in the scene description (e.g., *"A basket has 10 apples and 5 oranges and someone adds 3 more"*), the solver will use the wrong operands.

**Impact**: Fragile coupling to a very specific sentence structure; brittle against future platform obfuscation.  
**Fix**: Anchor number extraction to the tokens immediately before and after the operation keyword, or validate that exactly two numbers appear.

---

### W4 — `pending_id` mismatch between daemon submit and local fallback
**File**: `crates/nexus42/src/acp/session_capture.rs`  
**Lines**: 340–375 (`spawn_submit_capture`) vs 193 (`submit_to_daemon`)  
`spawn_submit_capture` generates a local `pending_id`, but `submit_to_daemon` generates a *different* `pending_id` inside the method. The local fallback file is named after the outer ID, while the daemon request uses the inner ID. This makes it impossible to correlate a locally-stored capture with the daemon-side log.

**Impact**: Operational/debugging friction; data correlation breaks on daemon-unavailable paths.  
**Fix**: Pass the same `pending_id` into `submit_to_daemon` (add it as a parameter) so both paths use one ID.

---

### W5 — Byte-index truncation risks panic on multi-byte UTF-8
**Files**: 
- `crates/nexus-domain/src/context_assembly.rs` (line 306: `text[..max_chars]`)
- `crates/nexus-domain/src/experience_aggregation.rs` (line 232: `body[..max_chars]`)

Both `truncate_to_char_count` and `truncate_body` use byte slicing (`text[..max_chars]`) rather than character boundaries. If the text contains multi-byte UTF-8 characters (e.g., em-dashes, CJK, smart quotes), slicing at an arbitrary byte position will panic.

**Impact**: Runtime panic on non-ASCII content.  
**Fix**: Use `chars().take(n).collect()` or `unicode-segmentation` for safe truncation.

---

### W6 — Silent error swallowing in `get_creator_api_key`
**File**: `crates/nexus42/src/auth/mod.rs`  
**Lines**: 106–125  
`get_creator_api_key` converts *any* disk error (permissions denied, JSON corruption, I/O failure) into `Ok(None)`. This masks real operational problems.

**Impact**: Silent data loss / auth failures that are hard to diagnose.  
**Fix**: Distinguish "file not found" (legitimate `None`) from parse/I/O errors (should propagate).

## Suggestions

### S1 — Explicit overflow handling in challenge evaluation
**File**: `crates/nexus42/src/challenge/eval.rs`  
`Operation::Add` and `Operation::Multiply` use `saturating_add`/`saturating_mul`. Given the current domain (words 0–100) overflow is practically impossible, but for correctness the function should return an explicit `ChallengeError::Overflow` rather than silently saturating.

### S2 — Add retry policy to `PlatformClient`
**File**: `crates/nexus-sync/src/platform_client.rs`  
Transient network errors (timeouts, 503s) are not retried. Consider adding a small retry loop (with exponential backoff) around `execute_request`, or document the intentional absence of retries.

### S3 — Cache DB connection in `resolve_active_identity`
**File**: `crates/nexus42/src/commands/identity.rs`  
`resolve_active_identity` opens the SQLite database on every call. For a CLI this is acceptable, but if this path becomes hot in future daemon integrations, connection caching will be needed.

### S4 — `TaskKind::from_str` should return `Err` for invalid inputs
**File**: `crates/nexus-domain/src/review.rs`  
`FromStr` for `TaskKind` logs a warning but returns `Ok(TaskKind::Unknown)` for every invalid string. This violates the principle of least surprise for a `FromStr` impl. Consider returning `Err(ParseTaskKindError)` and providing a separate `parse_fallible` convenience method.

### S5 — Add boundary/edge-case tests for number conversion
**File**: `crates/nexus42/src/challenge/numbers.rs`  
Tests cover 0–99 well, but "hundred" is only tested in isolation. Add tests for:
- Teens after tens (e.g., "twenty twelve" — should this be 20 12 or 32? Currently 32, which may be surprising)
- Zero-prefixed compounds (not applicable to natural language, but good for defensive testing)

## Gate Decision

**Verdict: Request Changes**

While the overall design is sound and `clippy` is clean, the **Warning** items above represent real correctness, portability, or observability defects that should be fixed before merge:

1. **W1** (Unix-only import) — blocks cross-platform builds.
2. **W4** (pending_id mismatch) — breaks operational correlation.
3. **W5** (UTF-8 panic risk) — can crash on real user content.
4. **W2/W3** (challenge number handling) — may produce wrong answers if the platform varies challenge format.
5. **W6** (silent error swallowing) — masks real auth failures.

Once these are addressed (or explicitly deferred with PM sign-off and tracked as new residuals), this reviewer is happy to re-review and approve.
