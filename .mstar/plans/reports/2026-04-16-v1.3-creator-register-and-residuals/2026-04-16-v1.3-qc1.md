## Warnings

### W1: `restore_from_snapshot` synthetic failure timestamps cause immediate re-degradation risk

**File**: `crates/nexus-domain/src/degradation.rs`, lines 316-319

```rust
// Replicate failure_count as synthetic timestamps so window logic works.
// Place them at "now" so they'll expire naturally via the window.
failure_timestamps: (0..snap.failure_count)
    .map(|_| chrono::Utc::now())
    .collect(),
```

**Issue**: When restoring from a persisted snapshot with `failure_count > 0`, all synthetic timestamps are placed at `chrono::Utc::now()`. If `failure_count >= failure_threshold` (default: 3), the guard will immediately trigger re-degradation on the very next `should_degrade()` check, even though the system was intentionally restored at a specific degradation level.

**Impact**: A user who was at `DegradedLevel1` with 2 failures, restores from snapshot, then encounters 1 more real failure → the 2 synthetic + 1 real = 3 triggers a second degradation to `DegradedLevel2`. This contradicts the stated design goal of C-001 (avoid unintended re-degradation).

**Fix**: The `restore_from_snapshot` should either (a) place synthetic timestamps spread across the window so they don't all hit the threshold at once, or (b) cap synthetic timestamps to `failure_threshold - 1` to ensure they alone can never trigger degradation, or (c) use a different field to track "restored failure count" separate from "active window timestamps." The comment on line 317 acknowledges the approach but does not mitigate the risk.

**Severity**: Warning — measurable impact on degradation behavior under restore-from-snapshot scenario.

## Suggestions

### S1: `detect_operation` redundant `to_lowercase()` call

**File**: `crates/nexus42/src/challenge/parser.rs`, line 55

The `extract_math_problem` function receives text that has already been lowercased by the calling pipeline (noise removal → `to_lowercase()` → number conversion). Yet `detect_operation` calls `text.to_lowercase()` again. This is a minor CPU waste and could be removed.

**Fix**: Accept `&str` that is already lowercase; add a doc comment clarifying the precondition.

### S2: `extract_keywords` stop words list rebuilt on every call

**File**: `crates/nexus-domain/src/review.rs`, lines 341-453

The `STOP_WORDS` array is defined inside the function body, meaning it's reconstructed on every invocation. With 80+ stop words, this is a noticeable allocation for a function that may be called in the review pipeline for each pending entry.

**Fix**: Move to a `static` const or use `LazyLock` for the stop words `HashSet` to achieve O(1) lookup without repeated allocation.

### S3: Emoji character in CLI output may not render in all terminals

**File**: `crates/nexus42/src/commands/context.rs`, line 261

```rust
eprintln!(
    "⚠ Degradation: {} (failures: {})",
    ...
);
```

The `⚠` (U+26A0) emoji may not render correctly in all terminal environments (Windows CMD, some SSH sessions, CI logs). Consider using ASCII `WARNING:` or `[!]` for consistency with the rest of the CLI, which uses plain text.

### S4: `PlatformClient` creates new `reqwest::Client` per instance

**File**: `crates/nexus-sync/src/platform_client.rs`, line 145

Each `PlatformClient::new()` creates its own `reqwest::Client`. If multiple instances are created (e.g., during testing), this creates separate connection pools. For the current single-instance usage this is fine, but if the client is ever reused across the codebase, consider accepting a shared `Client` or using a lazy-static pool.

### S5: `challenge/mod.rs` has `#![allow(dead_code)]` at module level

**File**: `crates/nexus42/src/challenge/mod.rs`, line 19

The `#![allow(dead_code)]` attribute silences all dead code warnings for the entire module and its children. This is overly broad — better to apply `#[allow(dead_code)]` selectively to specific items that are intentionally unused (e.g., public API for future use) and let clippy catch genuinely dead code.
