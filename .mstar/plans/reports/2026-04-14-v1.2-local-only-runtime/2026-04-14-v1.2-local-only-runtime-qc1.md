## Warnings

### W1: `sync status` subcommand blocked by top-level guard (incorrect scope)

**File**: `crates/nexus42/src/commands/sync.rs`, line ~222
**Severity**: Medium

The `runtime_guard::require_platform(...)` call is placed at the top of `run()`, before the `match cmd`. This means **all** sync subcommands — including `sync status` — are blocked in `local_only` mode.

**Problem**: `sync status` is a **read-only local operation** (checks local outbox/DB state). It does not make HTTP calls to the platform. Blocking it in `local_only` mode contradicts ADR-017 which says "Platform unreachable/login failure is not a Creator failure" — the user should still be able to inspect local sync state.

**Evidence** (from diff):
```rust
pub async fn run(cmd: SyncCommand, config: &CliConfig) -> Result<()> {
    runtime_guard::require_platform(&config.runtime_mode(), "sync")?;  // <-- guards ALL subcommands
    let client = DaemonClient::from_config(config);
    match cmd {
        ...
    }
}
```

The plan's file map says "Add local_only guard" for sync — but the integration test (`crates/nexus42/tests/integration.rs`) was changed to expect `sync status` to fail with "not available in local_only mode". This test change codifies the **wrong** behavior.

**Recommended fix**: Move the guard inside the match arm for subcommands that actually need platform (e.g. `sync pull`, `sync push`). Leave `sync status` and other read-local subcommands unguarded. Update the integration test to assert the original behavior (success) for `sync status`.

**Cross-Reviewer Ready Notes**: This is the same pattern risk as `explore` and `publish` — if those commands have read-only subcommands, they'd be incorrectly blocked too. Other reviewers should verify.

### W2: `DomainError::PlatformOperationProhibited` → `CliError::Other` loses error classification

**File**: `crates/nexus42/src/errors.rs`, line 199-202
**Severity**: Medium

The existing `From<DomainError> for CliError` maps **all** domain errors (including the new `PlatformOperationProhibited`) to `CliError::Other`:

```rust
impl From<nexus_domain::errors::DomainError> for CliError {
    fn from(err: nexus_domain::errors::DomainError) -> Self {
        CliError::Other(format!("Domain error: {}", err))
    }
}
```

**Problem**: This loses the semantic distinction between a platform-prohibited error and other domain errors. Downstream code (tests, error handlers, user-facing messages) cannot pattern-match on this specific case. The error message is prefixed with "Domain error: " which adds noise.

**Recommended fix**: Add a dedicated `CliError::PlatformOperationProhibited { mode: String, operation: String }` variant and match on `DomainError::PlatformOperationProhibited` in the `From` impl:

```rust
impl From<nexus_domain::errors::DomainError> for CliError {
    fn from(err: nexus_domain::errors::DomainError) -> Self {
        match err {
            DomainError::PlatformOperationProhibited { mode, operation } => {
                CliError::PlatformOperationProhibited { mode, operation }
            }
            other => CliError::Other(format!("Domain error: {}", other)),
        }
    }
}
```

With a `Display` impl that outputs `"Operation 'X' is not available in Y mode"` (matching the integration test's expectation) — without the "Domain error: " prefix.

**Note**: The current `CliError::Other` impl still produces a message containing "not available in local_only mode" (from the `DomainError` Display), so the integration test passes. But this is fragile — any change to `DomainError`'s Display would silently break the test.

## Suggestions

### S1: `auth_register` classified but command variant doesn't exist (yet)

**File**: `crates/nexus-domain/src/runtime_guard.rs`, line 33
`classify_operation` maps `"auth_register"` to `PlatformRequired`, but `AuthCommand` has no `Register` variant. This is dead classification code. It's harmless now, but if someone adds `AuthCommand::Register` later and forgets the guard, the classifier won't help because the guard is placed manually in `auth.rs` (`require_platform`), not via `check_operation`.

**Suggestion**: Either remove `"auth_register"` from `classify_operation` until the command exists, or add a `#[allow(dead_code)]` comment. Alternatively, consider using `check_operation` in command dispatch instead of manual `require_platform` calls for consistency.

### S2: `classify_operation` defaults unknown operations to `LocalOnly` (fail-open)

**File**: `crates/nexus-domain/src/runtime_guard.rs`, line 41
```rust
_ => OperationCategory::LocalOnly,
```

If a new platform-dependent command is added and the developer forgets to add it to `classify_operation`, it will silently be classified as `LocalOnly` and **not** be blocked in `local_only` mode. This is a fail-open design.

**Suggestion**: Consider `LocalOnly` for unknown ops (current) but add a `tracing::warn!` log, or change the default to `PlatformRequired` (fail-closed) for safety. At minimum, document the rationale in a comment.

### S3: Daemon reads runtime mode only at startup — stale after CLI config change

**File**: `crates/nexus42d/src/workspace/mod.rs`, lines 96-99

The daemon reads `runtime_mode` from `CliConfigSnapshot::load()` once during `WorkspaceState::open()` and stores it in the struct. If the user runs `nexus42 runtime-mode set cloud_enhanced` while the daemon is running, the daemon's status API will still report `local_only` (or whatever the mode was at startup).

**Acceptable for T3**: The plan (Decision D4) says T3 only enforces `local_only` guards; T6 handles mode-switch degradation. But this should be documented as a known limitation or tracked as a residual.

### S4: `runtime_mode set` does not notify running daemon

**File**: `crates/nexus42/src/commands/runtime_mode.rs`, line 73-76

After `updated.save()`, the daemon is unaware of the mode change (see S3). For T3 this is acceptable, but future work should either: (a) have the daemon re-read config on each status call, or (b) send a config-change notification to the daemon.

### S5: `runtime_mode set` has no validation that the daemon is running

**File**: `crates/nexus42/src/commands/runtime_mode.rs`

The `set` command writes to `~/.nexus42/config.json` without checking if the daemon is running. If the daemon is running with a different mode, the config and daemon state will diverge. Not a T3 blocker, but worth noting.

### S6: Integration test scope change may hide daemon connectivity regression

**File**: `crates/nexus42/tests/integration.rs`, lines 276-279

The test `sync_status_without_daemon` was renamed and its assertion changed from:
- **Before**: `.success().stdout(predicate::str::contains("not running"))`
- **After**: `.failure().stderr(predicate::str::contains("not available in local_only mode"))`

This means the test no longer verifies the "daemon not running" code path for `sync status`. If `local_only` mode is ever changed to a non-blocking default, this test won't catch the regression. Consider adding a separate test that sets mode to `cloud_enhanced` and then verifies the "daemon not running" behavior.
