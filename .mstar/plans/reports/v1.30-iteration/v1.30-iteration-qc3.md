### 🟡 Warning

#### W-001: KeyBlock inserted before mark_done — partial failure inconsistency
- **File**: `crates/nexus-orchestration/src/capability/builtins/kb_extract_work.rs`, lines 308–326
- **Issue**: The `KeyBlock` is inserted via `SqliteKbStore` (Phase 4, line 308) *before* the job is marked done (Phase 5, line 324). If `mark_extract_job_done()` fails after the KeyBlock has been committed, the following inconsistent state results:
  - A `KeyBlock` exists in the database (insert succeeded).
  - The extract job remains in "running" status (mark_done failed).
  - On retry, the orchestrator will re-extract the same work entry, producing a *duplicate KeyBlock* (same canonical name, same world — triggering a `Duplicate` error from `SqliteKbStore`).
  - The duplicate error would then mark the (retry) job as failed, but the original job remains "running" forever.

  The same pattern applies to `mark_failed` paths (lines 277, 290, 311) — those are called *before* `KeyBlock` insertion and are safe. The issue is specific to the `mark_done` after successful insert.

- **Fix**: Either (a) wrap KeyBlock insert + mark_done in a single DB transaction (idempotent on retry), or (b) insert the KeyBlock with an idempotency key derived from the job_id so re-insertion is safe, or (c) insert the KeyBlock only after mark_done succeeds (but this risks a "done" job with no corresponding KeyBlock).

- **Severity rationale**: This is a **Warning** (not Critical) because the failure scenario requires a DB error *after* a successful KeyBlock insert, which is rare. However, it produces unrecoverable state (stuck "running" job + no retry path) without operator intervention.

#### W-002: SessionCapture RwLock — all accesses use write() lock exclusively
- **File**: `crates/nexus42/src/commands/acp_worker/mod.rs`, lines 391, 518, 652, 737, 2000, 2012, 2018, 2027, 2063, 2104, 2121
- **Issue**: All 11 access points to `state.session_captures` use `.write()` exclusively. Several are read-only operations that could use `.read()`:
  - Line 518: `.get_mut()` requires write for mutation, but `.record_message()` + `.update_context()` are called. The write lock is correct here.
  - Line 737: `.remove()` requires write lock. Correct.
  - **Line 2115**: `.read().expect("lock ok").len()` — this uses `.read()` but could've used `.read()` consistently elsewhere.
  - Lines 2000, 2012, 2018: test code — not a runtime concern.

  Since the IPC loop is single-threaded (sequential JSON-RPC dispatch), this does **not** cause contention in practice. However, using `.write()` when `.read()` suffices is misleading and would become a problem if the worker ever processes requests concurrently.

- **Fix**: Use `.read()` for read-only operations (e.g., test assertions on line 2115 already does this correctly). The production code paths (391, 518, 652, 737) actually need write (insert, get_mut for mutation, remove), so the issue is limited to code clarity rather than a live bug.

- **Severity rationale**: Warning because the code pattern is technically incorrect and fragile against future concurrency changes, even though it's benign under the current single-threaded IPC model.

#### W-003: cleanup_row spawns detached task — failure silently lost
- **File**: `crates/nexus-local-db/src/reference_source.rs`, lines 194–205
- **Issue**: When `register()` fails after a successful SQL INSERT (e.g., file write fails), `cleanup_row()` spawns a detached `tokio::spawn` to DELETE the row. The DELETE result is discarded with `let _ =`. If the DELETE fails (pool closed, connection broken, table missing), the dangling registry row persists indefinitely with **no log, no retry, no operator visibility**.

  The `tokio::spawn` also introduces a race: the spawned task may not execute before the function returns an error, and the caller may observe the dangling row before cleanup completes.

- **Fix**: At minimum, add `tracing::error!` on DELETE failure. Better: await the cleanup inline with a short timeout, or use a transactional approach (DELETE the row before returning the error, synchronously).

- **Severity rationale**: Warning because dangling rows are a data integrity concern that compounds over time, even though the primary path (successful registration) is unaffected.

### 🟢 Suggestion

#### S-001: 150ms Drop wait may be insufficient for heavy children
- **File**: `crates/nexus-agent-host/src/providers/native_cli/claude.rs`, lines 762–766
- **Issue**: The `Drop` implementation sends SIGTERM, waits **150ms**, then sends SIGKILL. For LLM processes that need to flush large stdout buffers, tear down MCP server connections, or write session state to disk, 150ms is a tight budget. A Claude process mid-response could easily need 500ms–2s to flush output.
- **Suggestion**: Consider making this timeout configurable via `TimeoutConfig` with a default of at least 500ms (for production) or 2s (for safety). The comment acknowledges this is "best-effort" — but best-effort should give children a reasonable chance to exit cleanly. Also consider logging which PIDs received SIGKILL to aid post-mortem.

#### S-002: LIST_BY_WORLD_LIMIT=500 silently truncates world queries
- **File**: `crates/nexus-local-db/src/kb_store.rs`, line 86, lines 260–288
- **Issue**: `list_by_world()` hard-caps at 500 rows with no indication of truncation. The `query()` method (line 290) builds on top of this, so a world with 600 entries only searches within the first 500. The caller receives a `KbQueryResult` with `has_more` based on in-memory filtering, not the full dataset.
- **Suggestion**: Return a warning or include metadata indicating truncation. Add a `total_count_in_world` field to `KbQueryResult` that reflects the actual database count (pre-filter), or add a separate COUNT query before applying the LIMIT. At minimum, document the LIMIT in the `KbStore` trait contract so callers are aware.

#### S-003: extraction prompt allocates full content via format!()
- **File**: `crates/nexus-orchestration/src/capability/builtins/kb_extract_work.rs`, lines 89–106
- **Issue**: `extraction_prompt()` constructs the full prompt via `format!(...)`, which clones the entire `work_content` into the prompt string. For large work entries (e.g., 1MB+ Markdown files imported as KB entries), this doubles memory usage at peak.
- **Suggestion**: For the current use case (typical work entries are small structured summaries), this is acceptable. If large-file support is planned, consider a streaming approach or `Cow<str>` based template with minimal copying.

#### S-004: mark_running lacks status guard
- **File**: `crates/nexus-local-db/src/kb_extract_job.rs`, lines 254–264
- **Issue**: `mark_running()` unconditionally updates status to "running" regardless of the current status. If called on a "done" or "failed" job, it silently reverts it.
- **Suggestion**: Add a `WHERE status = 'queued'` clause to the UPDATE, mirroring the guard in `claim_job()`. This prevents accidental state reversion by misbehaving callers.

#### S-005: claim_job re-fetches row after commit
- **File**: `crates/nexus-local-db/src/kb_extract_job.rs`, lines 323–343
- **Issue**: After the transaction commits, `claim_job` re-fetches the row (lines 324–341) solely to get the updated `started_at` timestamp. SQLite's `datetime('now')` returns UTC, and the `started_at` field is `None` in the SELECT result. The re-fetch adds one extra database round-trip.
- **Suggestion**: Compute `started_at` in Rust using `chrono::Utc::now().to_rfc3339()` after the commit, set it on the returned `KbExtractJob` struct, and avoid the re-fetch. This saves a database query and is equivalent (clock skew between SQLite's `datetime('now')` and Rust's timestamp is negligible).
