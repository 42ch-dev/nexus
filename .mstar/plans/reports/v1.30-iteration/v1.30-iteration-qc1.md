### 🟢 Suggestion

1. **S-001: `reference_source.rs` dynamic SQL for pagination loses compile-time guarantees**
   - `list()` now uses `sqlx::query()` with `format!()` for `LIMIT`/`OFFSET` instead of `sqlx::query!()`. This is necessary because sqlx offline mode does not support bind parameters for `LIMIT`/`OFFSET` in SQLite, but it removes compile-time query checking for the entire `list()` query.
   - **Mitigation**: The query is read-only and the `limit`/`offset` values are bounded `i64` from `Option<i64>` callers. The SAFETY comment documents the trade-off. Consider a future migration to `sqlx::query_as!()` with a fixed set of pagination constants if the offline-mode limitation is resolved.

2. **S-002: `kb_store.rs` uses `format!()` for `LIMIT` constant**
   - Similar to S-001, `list_by_world()` uses `format!()` to inject `LIST_BY_WORLD_LIMIT` (500) into the SQL. The comment correctly identifies this as a workaround for sqlx offline mode.
   - **Mitigation**: Acceptable for now since `LIST_BY_WORLD_LIMIT` is a module-level constant and the query is read-only. No action required.

3. **S-003: `cleanup_row()` in `reference_source.rs` is fire-and-forget with no observability**
   - The best-effort `cleanup_row()` spawns a detached tokio task to delete a row when file write fails after DB insert. If this cleanup fails, the dangling row is invisible (no log, no metric).
   - **Mitigation**: Consider adding a `tracing::warn!()` inside the spawned task on cleanup failure. This is low priority since the scenario (DB insert succeeds, file write fails) is already an exceptional path.

4. **S-004: `KbExtractWork::run()` is ~170 lines — consider phase extraction in a future refactor**
   - The `#[allow(clippy::too_many_lines)]` annotation is justified by the comment "Single-pass claim→extract→insert→finalize pipeline; splitting would obscure the state machine." The five phases (load/claim → load content → check LLM response → parse/insert → mark done) are clearly demarcated with comment banners.
   - **Mitigation**: The state machine is coherent as-is. If the pipeline grows further (e.g., adding retry logic, multi-turn extraction), consider extracting each phase into a private method.

## Architecture Assessment

### Module Boundaries (creator/kb.rs extraction — R19)

The extraction of `~973` lines of KB command handlers from `creator/mod.rs` into `creator/kb.rs` is clean and well-executed:

- **Public API preserved**: `KbCommand`, `KbScope`, and `run()` are re-exported from `kb.rs` via `pub use kb::{KbCommand, KbScope};` in `mod.rs`. Existing consumers and pattern matches on `CreatorCommand::Kb` remain unchanged.
- **No leakage**: The extracted module keeps `KbIndex` and `KbIndexEntry` as `pub(crate)`, preventing external visibility of implementation details.
- **Slimmed parent**: `creator/mod.rs` dropped from ~1900 lines to ~950 lines, improving readability and compile times.
- **Atomic file write utility**: `write_kb_index()` (atomic tmp+rename) is preserved as a public utility in `kb.rs`, even though `kb_add` currently inlines the pattern.

**Verdict**: Clean boundary. No regressions.

### kb_extract_work e2e Lifecycle (R16)

The `KbExtractWork` capability has been transformed from a placeholder ("returns structured response indicating extraction was prepared") into a full stateful pipeline:

- **Pool injection**: The capability now holds `Option<Arc<SqlitePool>>`, with `KbExtractWork::new()` (placeholder, no pool) and `KbExtractWork::with_pool()` (full e2e). The registry provides both `with_builtins()` and `with_builtins_and_pool()` for backward compatibility.
- **Two-phase design**: 
  - **Prompt phase** (no `llm_response`): builds and returns the extraction prompt for `acp.prompt` execution.
  - **Finalize phase** (with `llm_response`): parses JSON, maps block types, inserts `KeyBlock` via `SqliteKbStore`, and marks job done.
- **Error handling**: Parse errors and insert failures explicitly mark the job as failed via `nexus_local_db::mark_extract_job_failed()` before returning the error upstream. This prevents stuck "running" jobs.
- **Job status validation**: Rejects jobs with status other than `queued` or `running`, preventing accidental re-processing of done/failed jobs.
- **Claim semantics**: When `job_id` is omitted, uses `claim_extract_job()` (atomic SELECT+UPDATE in transaction) to prevent double-claiming.

**Verdict**: Architecturally sound. The state machine is explicit, error paths are handled, and the capability integrates cleanly with both the preset orchestrator and direct invocation.

### SessionCapture in acp_worker (R14)

The SessionCapture lifecycle management is coherent with multiplexed sessions:

- **Per-session captures**: `MultiplexedWorkerState` now holds `session_captures: RwLock<HashMap<String, SessionCapture>>`, keyed by session ID.
- **Lifecycle hooks**:
  - **Creation**: Captures are created during `initialize` (for each session in the `agents` array) and `agent_start` (for dynamically started sessions).
  - **Mutation**: `handle_acp_prompt` records message count and updates context on the session's capture.
  - **Termination**: `handle_agent_stop` removes the capture from the map and submits it via `spawn_submit_capture()`.
- **Fallback**: If no capture is found (e.g., session started before this code), a fallback capture is created with a warning log.
- **No memory leak**: Captures are removed from the HashMap at session stop, preventing unbounded growth for long-running workers with many transient sessions.

**Verdict**: Coherent lifecycle. The `RwLock<HashMap>` pattern is appropriate for the multiplexed worker where sessions are dynamically created/destroyed.

### NarrativeError Variant Introduction (R8)

The introduction of `NarrativeError::Storage(String)` and the change of `db_err()` to return `Storage` instead of `ValidationError` is a correctness improvement:

- **Semantic accuracy**: Database errors (connection failures, lock timeouts, corruption) are not validation errors. The old mapping misled callers.
- **Blast radius**: `db_err` is a private helper in `narrative_gateway.rs`. The only downstream consumer within the crate is the SQLite gateway implementation. The `InMemoryNarrativeGateway` in `nexus-narrative/src/gateway.rs` uses its own error paths and is unaffected.
- **Test coverage**: A new unit test `test_db_err_maps_to_storage` verifies the mapping.

**Verdict**: Correct hierarchy. No breaking changes to public APIs.

### Cross-Cutting Concerns

1. **Bounded listing consistency (R9, R20)**: Both `kb_store::list_by_world()` (limit 500) and `kb_extract_job::list_by_creator()` (caller-provided limit, default 100 in CLI) now enforce bounded result sets. This prevents unbounded memory usage on large datasets.
2. **Write ordering consistency (R5)**: `reference_source::register()` now performs the SQLite INSERT before writing `body.md` to disk, with a best-effort cleanup on file-write failure. This prevents orphan body files on DB failure.
3. **Content hashing (R7)**: `register()` now computes a `blake3` hash of the body bytes at registration time and stores it in `content_hash`. This enables future integrity verification.
4. **Child process cleanup (R17)**: The `ClaudeCliProvider` Drop implementation now sends `SIGTERM`, waits 150ms, then sends `SIGKILL`. This is a significant improvement over the previous immediate `SIGKILL`, giving well-behaved children time to flush state.
5. **UUID job IDs (R18)**: `kb_extract_job` now uses `uuid::Uuid::new_v4()` instead of a timestamp-based ID, eliminating collision risk under rapid successive calls. A single retry on `UNIQUE` constraint violation (SQLite code 1555) provides defense in depth.
