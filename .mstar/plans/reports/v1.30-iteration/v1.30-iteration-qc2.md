## Findings

### 🔴 Critical
- **Atomic claim correctness race (R15) — kb_extract_job::claim_job**: `SELECT ... WHERE status='queued' LIMIT 1` followed by `UPDATE ... SET status='running' WHERE job_id=? AND status='queued'` inside `BEGIN ... COMMIT` tx. No `rows_affected()` check on UPDATE; no `FOR UPDATE` / locking. Two concurrent `claim_job` calls (tokio::spawn) can both SELECT the same row (SQLite snapshot or before commit), both issue UPDATE (one affects 1 row, other 0), both return `Some(claimed_job)` from their pre-tx SELECT + post-commit re-fetch. The `test_claim_job_concurrent_double_claim_prevented` passes only due to default SQLite locking/BUSY in the test harness timing; not deterministic under WAL, high load, or different scheduler. 

  Impact: duplicate KB extract work dispatched, race on KeyBlock insertion / job state, wasted LLM quota, potential duplicate or corrupted narrative/KB data. Violates "atomic claim" guarantee in plan R15 and e2e lifecycle (R16).

  Source Trace: git diff .../kb_extract_job.rs:263-340 (claim_job impl + tx), 540-583 (concurrent test + full lifecycle test); cross-ref to kb_extract_work.rs:170 (claim call site).

  Confidence: High (code + test inspection + known SQLite concurrency pattern).

- **Dynamic SQL interpolation for user-controlled LIMIT/OFFSET (R20 + injection-adjacent) — reference_source::list**: `sqlx::query(&format!( "SELECT ... LIMIT {limit} OFFSET {offset}" ))` with `Option<i64>` params, no validation/clamping inside fn. Caller-controlled (pagination in higher layers). "SAFETY" comment claims callers always pass positive; not enforced. Numeric interpolation cannot do classic string injection, but enables DoS (huge LIMIT → OOM / slow scan), negative values (SQLite LIMIT -N semantics vary, may error or return all), and violates the parameterized-query discipline used everywhere else in the crate. Similar (lower-risk, const-bounded) pattern repeated in kb_store::list_by_world.

  Impact: correctness (unbounded results possible despite R20 intent), security (resource exhaustion via crafted pagination calls if any untrusted path reaches it).

  Source Trace: .../reference_source.rs:210-240 (format! query + comment), 425-460 (pagination test); also kb_store.rs:260-280.

  Confidence: High.

- **Incomplete `mark_failed` coverage on ALL error paths in full e2e lifecycle (R16) — kb_extract_work::run**: After `claim_extract_job` (or load) which transitions job → 'running', early returns (empty `work_content` → `needs_content` json; missing `llm_response` → prompt json) leave job in 'running' with no terminal state. On parse / block_type / insert errors: `mark_failed` called (good, via `_ = ...`). But if `mark_done` fails *after* successful `store.insert_key_block`, job remains 'running' while KeyBlock exists (inconsistent). No scope guard / RAII / defer pattern; two-phase prompt/finalize design amplifies risk if second call never arrives or panics.

  Impact: leaked 'running' jobs (observability debt, queue stalls), KeyBlock orphans or untracked provenance, violates "full e2e lifecycle" and "mark_failed on ALL error paths" intent of R16 / plan K4.

  Source Trace: .../kb_extract_work.rs:164-320 (run fn phases 1-5, mark calls only in 3 Err arms), 340 (mark_done after insert).

  Confidence: High.

- **PID reuse / orphan / zombie risk in Drop hardening (R17) — claude.rs**: SIGTERM → 150ms sleep → SIGKILL on raw PIDs collected from `persistent_children`. No `kill(pid, 0)` / procfs existence + owner check before kill; classic PID reuse window after child exit but before Drop. `std::process::Command` fire-and-forget (no status check on kill itself). Sleep blocks the Drop thread. Non-Unix: only warning, children orphaned. Explicit `shutdown()` path is documented as preferred (uses real tokio::Child), but Drop is the safety net.

  Impact: correctness (may kill innocent reused PID; zombies if KILL fails silently); security (local privilege if PID reuse exploited in multi-tenant, low but present).

  Source Trace: .../claude.rs:736-785 (Drop impl + comments).

  Confidence: High.

### 🟡 Warning
- **RwLock poison + silent loss in SessionCapture (R14) — acp_worker/mod.rs**: `state.session_captures: std::sync::RwLock<HashMap<...>>`; every access uses `if let Ok(mut g) = write()` (poison ignored). On prior panic while holding, subsequent records/removes are no-ops; `handle_agent_stop` falls back to zero-metric capture. Short critical sections (no await inside), but still lossy under panic.

  Source Trace: .../acp_worker/mod.rs:87 (state), 390-410 (init inserts), 510 (prompt record), 740-770 (stop remove + fallback).

- **Fire-and-forget orphan cleanup (R5) — reference_source::register**: On file write fail after successful INSERT, `tokio::spawn` async DELETE; no await/join/monitoring. Cleanup can silently fail → dangling registry row with no body.md (inconsistent state).

  Source Trace: .../reference_source.rs:160-175 (cleanup_row fn + spawn), 200-220 (error paths).

- **Test for atomicity is timing-dependent**: `test_claim_job_concurrent_double_claim_prevented` asserts exactly 1 succeeds, but relies on SQLite runtime behavior rather than deterministic protocol (e.g., rows_affected check). Can flake under load or different SQLite compile options.

### 🟢 Suggestion
- Enforce `limit: 1..=1000`, `offset >= 0` with clamping/sanitization at entry to `reference_source::list` (and callers); prefer `sqlx::query_as` + bind where possible or document the dynamic-query exception narrowly.
- In `claim_job`: capture `rows_affected` from UPDATE; if 0 { tx.rollback().await?; return Ok(None); } before commit. Add `SELECT ... FOR UPDATE` if SQLite version allows.
- Introduce `JobLifecycleGuard` or explicit state machine (enum + transition table) in `KbExtractWork` so every terminal path (success or any error) calls mark_done or mark_failed exactly once.
- Child cleanup: add `libc::kill(pid, 0) == 0 && verify owner` before SIGTERM/KILL; make sleep cancellable or move to background thread; consider `setsid` / process groups at spawn time for reliable reaping.
- `insert_with_retry`: the final `Configuration` error after 2 attempts is odd for a "should never happen" case; return a dedicated `JobIdCollision` error type instead.
- Session capture: use `parking_lot::RwLock` (poison-free) or `std::sync::Mutex` + `unwrap_or_else` recovery; or record metrics outside the lock where possible.
- Creator extract (R19): clean ~950-line slimming; verify no stray direct uses of old handlers in other crates (grep for `creator::` paths).
- Overall: the .sqlx/ metadata churn (~50 files) in diff is expected after query changes; ensure CI step "sqlx prepare" is enforced so offline mode stays green.
- Consider adding a `job_state` enum column + DB trigger or app-level FSM assertions in tests for lifecycle invariants.
