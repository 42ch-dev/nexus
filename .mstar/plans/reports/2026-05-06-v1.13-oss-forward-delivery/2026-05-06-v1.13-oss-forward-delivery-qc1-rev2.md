### 🟢 Suggestion

**S-REV2-001: `build_nested_payload` could document intermediate path collision behavior**
- **Location**: `crates/nexus-orchestration/src/tasks/mod.rs:1061-1080`
- **Issue**: `build_nested_payload` constructs nested JSON from flat dot-separated keys. If two keys like `foo.bar` and `foo` both exist in the same context, the implementation creates `{"foo": {"bar": ...}}` but what happens if `foo` itself is also a leaf value is not documented. The current `set_nested` implementation would overwrite intermediate nodes if a later key claims a path segment already used as an intermediate object.
- **Impact**: Low — context keys in the orchestration layer are conventionally structured (e.g., `world.title`, `state.<id>.output`), not arbitrarily mixed. Current test coverage proves the intended nested path access works.
- **Recommendation**: Add a doc comment clarifying expected key structure conventions (e.g., "keys should be hierarchical with consistent intermediate path segments"). Alternatively, add a debug assertion or log warning if path collision occurs during payload construction.
- **Confidence**: Medium

**S-REV2-002: Error body sanitization patterns could extend to more secret formats**
- **Location**: `crates/nexus-sync/src/platform_client.rs:68-72`
- **Issue**: The current `sanitize_error_body` redacts known patterns (Bearer tokens, `sk-`, `nexus_live_`, long hex/base64). Other common secret patterns like JWT structure, AWS-style keys (`AKIA...`), or UUIDs in sensitive contexts are not covered. The 200-char cap provides a safety net, but structured secrets shorter than 200 chars could still leak.
- **Impact**: Low — the current patterns cover the primary secret formats used in the nexus platform integration (nexus_live_/nexus_test_ keys, Bearer tokens). The 200-char cap limits exposure for unrecognized patterns. The fix is correct for the current scope.
- **Recommendation**: Future enhancement (V1.14+): consider extending pattern library for common cloud-provider key formats if platform integration expands. Current implementation is acceptable for V1.13.
- **Confidence**: High

**S-REV2-003: Template rendering fallback behavior could be documented more clearly**
- **Location**: `crates/nexus-orchestration/src/tasks/mod.rs:940-949`
- **Issue**: `AcpPromptTask::render_template` falls back to raw template on render failure with a `tracing::warn` log. The fallback is non-fatal and correct for stub mode, but production orchestrations may benefit from clearer documentation about whether render errors should be treated as hard failures in certain execution contexts.
- **Impact**: Minimal — the fallback behavior is explicit and tested. The tracing log provides observability for debugging.
- **Recommendation**: Add a doc comment on `render_template` clarifying the fallback policy (e.g., "for stub/test mode, render errors are non-fatal; production presets may override this behavior by validating templates at preset load time").
- **Confidence**: Medium
