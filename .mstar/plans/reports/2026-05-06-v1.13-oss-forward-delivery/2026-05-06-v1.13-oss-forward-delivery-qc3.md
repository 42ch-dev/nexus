### 🟡 Warning

#### W-001: `render_core_context_template` recreates Handlebars registry on every call
- **Source**: `crates/nexus-orchestration/src/tasks/mod.rs:1051`
- **Issue**: The function instantiates a fresh `handlebars::Handlebars` registry on every invocation:
  ```rust
  let reg = handlebars::Handlebars::new();
  let rendered = reg.render_template(template, payload)?;
  ```
  Handlebars template compilation (parsing, AST construction, and partial registration) is non-trivial overhead. If this function is later used in a hot path — e.g., per-prompt rendering inside the orchestration engine's task loop — it introduces avoidable O(n) latency on every render where n = template complexity.
- **Fix** (pick one):
  1. Accept `&Handlebars` as a parameter so callers can cache and reuse a registry:
     ```rust
     pub fn render_core_context_template(
         reg: &Handlebars,
         template: &str,
         payload: &serde_json::Value,
     ) -> anyhow::Result<String>;
     ```
  2. Or provide a cached singleton variant (e.g., via `std::sync::LazyLock` or `once_cell`) for the common case.
  3. Or explicitly document in the doc comment that this is for **one-off / test use only** and that production callers should cache `Handlebars`.
- **Confidence**: High

### 🟢 Suggestion

#### S-001: Add template syntax error test for `render_core_context_template`
- **Source**: `crates/nexus-orchestration/tests/e2e_novel_writing.rs:288`
- **Issue**: The only test for the new template rendering path is a happy-path test (`core_context_template_is_rendered_into_prompt`). There is no coverage for malformed templates, missing keys, or nested-path failures.
- **Fix**: Add at least one test asserting that `render_core_context_template` returns `Err` for an unclosed tag or invalid syntax, and one test for a missing key (to document whether handlebars renders empty string or errors).
- **Confidence**: High

#### S-002: Evaluate compile-time/binary-size tradeoff of `handlebars` dependency
- **Source**: `Cargo.lock`, `crates/nexus-orchestration/Cargo.toml`
- **Issue**: `handlebars = "6"` pulls in a substantial transitive tree: `pest` + `pest_derive` + `pest_generator` + `pest_meta`, `derive_builder`, `num-order`, `num-modular`, `ucd-trie` — roughly 12 new crates. The current usage is limited to simple JSON-path interpolation (`{{world.title}}`). A lighter alternative (`minijinja`, `liquid`, or a minimal custom interpolator) might achieve the same functional goal with less compile-time and binary-size overhead.
- **Fix**: If the project anticipates complex templating (partials, helpers, block logic) in future presets, `handlebars` is justified. If the need remains limited to flat/nested variable substitution, consider evaluating `minijinja` (lighter, no proc-macro parser generator) as a follow-up.
- **Confidence**: Medium

#### S-003: Document expected latency bound of UpstreamTimeout e2e test
- **Source**: `crates/nexus42/tests/creator_register_e2e.rs:679`
- **Issue**: The `UpstreamTimeout` test uses `http://192.0.2.1:1` to simulate an unreachable platform. TCP connection timeout is OS-dependent and can range from a few seconds to 30s+ (the `DEFAULT_TIMEOUT_SECS` in `PlatformClient`). The test already acknowledges proxy behavior (accepting 502), but does not document expected wall-clock duration for CI timeout budgeting.
- **Fix**: Add a code comment or test attribute documenting the expected upper-bound duration (e.g., `#[timeout(60s)]` equivalent, or a note that the test may take up to `DEFAULT_TIMEOUT_SECS`).
- **Confidence**: Medium
