### 🟡 Warning
- **[F-QC2-WSB-01] `..` detection via `contains("..")` is imprecise — may reject legitimate filenames like `foo..bar`**
  - **Location**: `crates/nexus-orchestration/src/preset/loader.rs:231` (`assert_template_file_safe`)
  - **Problem**: `if path.contains("..")` uses a simple substring check. Any path containing the two-character sequence `".."` is rejected, even when those characters appear mid-filename (e.g., `foo..bar`, `a..b/c.md`). This is overly conservative and may produce confusing errors for valid use cases.
  - **Fix**: Split on `Path::separator` (`/`) first, then check whether any **component** equals `".."` exactly:
    ```rust
    if path.split('/').any(|c| c == "..") { return Err(...); }
    ```
  - **Severity**: Warning (secure in practice — the check never allows traversal — but may cause false positives)
  - **Confidence**: Medium (the overly strict behavior is conservative from a security standpoint)

- **[F-QC2-WSB-02] Misleading test comment — no actual symlink is created in `filesystem_preset_rejects_template_file_escaping_bundle`**
  - **Location**: `crates/nexus-orchestration/src/preset/loader.rs:1973`
  - **Problem**: Comment says `// Create a symlink outside the bundle` but the code uses `std::fs::write(&outside, "secret")` — a regular file, not a symlink. The test still validates the correct behavior (structural `..` rejection via `assert_template_file_safe`), but it does not actually exercise the symlink-based sandbox escape path.
  - **Impact**: The defense-in-depth sandbox check (`canonicalize()` in `validate_template_files_in_sandbox`) that catches symlink-based bundle escapes is not covered by any test. A future regression could silently remove that protection.
  - **Fix**: Add a separate test that creates a **symlink inside the bundle** pointing outside (e.g., `prompts/ -> /tmp/outside/`) and uses a safe-looking path (`prompts/file.md`) that actually resolves through the symlink:
    ```rust
    std::os::unix::fs::symlink("/tmp/outside", bundle_root.join("prompts")).unwrap();
    // use template_file: "file.md" — resolves via symlink to /tmp/outside/file.md
    ```
  - **Severity**: Warning (implementation is correct; test coverage gap only)
  - **Confidence**: High

- **[F-QC2-WSB-03] Name length validation occurs after auth token fetch in `register_creator`**
  - **Location**: `crates/nexus42/src/commands/creator.rs:285-291`
  - **Problem**: `MAX_CREATOR_NAME_LENGTH` validation is performed **after** `AuthStore::load()?` which obtains an auth token. For an invalid name, the user authenticates successfully before receiving the length error, which is wasted work and a suboptimal UX.
  - **Fix**: Move the length check to the very beginning of `register_creator`, before the auth token fetch.
  - **Severity**: Warning (functional correctness is not affected; UX only)
  - **Confidence**: High

## Detailed Analysis

### Path Traversal Defense (T1 + T3)
The implementation uses a two-layer defense:
1. **Structural check** (`assert_template_file_safe`): Rejects `..` anywhere in path, `/`-prefixed paths, null bytes, and control characters. This is applied to all `template_file` fields in `validate_manifest()` (called by `load_preset_from_str()`).
2. **Sandbox check** (`validate_template_files_in_sandbox`): After YAML parsing in `load_preset()`, resolves each `template_file` by joining to `bundle_root` and calling `canonicalize()`. Checks `canonical.starts_with(canonical_root)`.

The defense-in-depth is sound. However, the `contains("..")` check is overly conservative (F-QC2-WSB-01), and the symlink sandbox path lacks test coverage (F-QC2-WSB-02).

### Symlink Handling
`std::path::Path::canonicalize()` on Unix resolves symlinks, so a symlink-based escape (e.g., bundle contains a symlink pointing outside) would be correctly caught by the `starts_with()` check. The implementation is correct; the test gap is the only concern.

### Name Validation (T4)
`MAX_CREATOR_NAME_LENGTH = 64` is enforced in `register_creator()`. The boundary tests (`name_64_chars` passes, `name_65_chars` fails) are correct. The check placement after auth token fetch is a minor UX issue (F-QC2-WSB-03).

### Coverage of `template_file` Fields
All four locations in `PresetManifest` are validated:
- `states[].exit_when.llm_judge.template_file` (optional)
- `states[].context_update.template_file` (required `String`)
- `inner_graphs[].nodes[].template_file` (optional)
- `preset.initial_action.seed_expansion.template_file` (optional)

The optional/required distinction is correctly handled — `Some(ref tf)` for optional fields, direct access for required fields.
