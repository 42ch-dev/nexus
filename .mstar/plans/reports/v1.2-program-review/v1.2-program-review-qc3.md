## Security Assessment

### Runtime Guard ✅

**Design**: `runtime_guard.rs` classifies operations into three categories:
- `LocalOnly`: always available
- `PlatformRequired`: blocked in `local_only`
- `LocalWithPlatformEnhancement`: available but enhanced in other modes

**Security property**: The `check_operation` function is the single enforcement point for platform-required operations. Every call path was checked against `degradation.rs` — `DegradationGuard::check_operation` delegates to `runtime_guard::check_operation`, consistent.

**Classification correctness**: `sync`, `publish`, `auth_login`, `auth_register`, `context_assemble`, `explore` are correctly classified as `PlatformRequired`. `identity`, `daemon`, `agent`, `session`, `runtime_mode` are correctly `LocalOnly`.

**No bypass vectors found**: All platform-required operations go through `check_operation`. No unchecked `require_platform` calls outside this module.

### Degradation Guard ✅

**C-001 mitigation verified**: `restore_from_snapshot` intentionally clears `last_failure_time`, `last_health_check`, and `last_upgrade_attempt` to prevent "re-degradation on restart." This is documented and correct.

**Degradation chain**: `cloud_enhanced → local_first → local_only` enforced. No way to skip levels.

**ForcedLocalOnly**: Can only be set via `force_local_only()` — not triggered by failure count. Correct.

**Upgrade cooldown**: Correctly implemented. `try_upgrade()` checks both cooldown elapsed time AND platform health.

**Potential concern (low)**: `failure_window_secs` is checked but `failure_count` is never reset based on elapsed time within the window — if failures span more than `failure_window_secs`, the guard still counts them cumulatively toward the threshold. However, `should_degrade` checks both `failure_count >= threshold` AND `elapsed <= window_secs`. If `elapsed > window_secs`, `should_degrade` returns `false` even with high `failure_count`. This means **after the window expires, failure_count remains elevated but degradation won't trigger until a new failure occurs within a new window**. The guard is slightly more conservative than a strict sliding window — not a security issue but a behavioral quirk worth noting.

### Identity Security ✅

**ID generation**: `ctr_anon` + 12 hex chars and `ctr_local` + 12 hex chars both match `^ctr_[a-zA-Z0-9]+$` pattern. `is_valid_creator_id` validates `ctr_` prefix, minimum length 4, and alphanumeric-only suffix.

**Platform link validation**: `link_to_platform` validates `platform_creator_id` against `is_valid_creator_id` before setting the link. No unchecked linking.

**SQL injection**: All identity functions use `rusqlite::params!` (parameterized queries). No string interpolation.

**No identity impersonation via DB**: `link_to_platform` uses `UPDATE ... WHERE platform_linked = 0` and checks the affected row count. No race condition window since the DB is local (single-process access typical in V1.2).

### Config File Security ✅

**Path traversal**: Config path is hardcoded to `~/.nexus42/config.json` — no user-controlled path. ✅

**File write**: Uses `serde_json::to_string_pretty` then `std::fs::write` — no arbitrary file write. ✅

**Deserialization**: `serde_json::from_str` with error propagation — no `unsafe`. ✅

**Empty file handling**: `load()` returns `Default::default()` for empty/missing files. ✅

**Config key allowlist**: `VALID_CONFIG_KEYS` is a hardcoded slice — no arbitrary key injection. ✅

**Sensitive data**: No API keys or secrets stored in `CliConfig`. `auth.json` is separate. Platform URL is not sensitive.

### DegradationSnapshot Persistence ⚠️ MEDIUM

**Issue**: `DegradationSnapshot` is stored inline in `config.json` (V1.2 MVP per docs). If the snapshot is corrupted or malformed, `serde_json::from_str` on the entire config file will fail, potentially causing CLI failure on next startup.

**Current behavior**: `CliConfig::load()` propagates the serde error as `anyhow::Error`. If config.json is corrupted, `nexus42 config get runtime_mode` would error out.

**Recommendation**: Add a recovery path: if config parsing fails, rename to `.bak` and start with defaults. This is tracked as R-QC3-001 below.

## Findings

### R-QC3-001 — Config file corruption recovery missing (Medium)

**File**: `crates/nexus42/src/config.rs`  
**Severity**: medium  
**Category**: reliability / degrade behavior

**Description**: `CliConfig::load()` calls `serde_json::from_str` directly on the config file. If `config.json` is corrupted (e.g., partial write, disk error, or malformed `DegradationSnapshot`), the entire CLI fails to start.

```rust
// Current: any parse error propagates up
Ok(serde_json::from_str(&content)?)
```

**Impact**: A failed `config save` mid-write (e.g., crash, OOM, signal) could leave the user unable to run any `nexus42` command, including fixing the config.

**Recommendation**: Add a recovery path that backs up and re-initializes on parse failure:

```rust
let config_path = Self::config_path()?;
if config_path.exists() {
    let content = std::fs::read_to_string(&config_path)?;
    if !content.trim().is_empty() {
        match serde_json::from_str::<CliConfig>(&content) {
            Ok(cfg) => return Ok(cfg),
            Err(e) => {
                // Backup corrupted file and re-initialize
                let bak = config_path.with_extension("json.bak");
                std::fs::rename(&config_path, bak)?;
            }
        }
    }
}
Ok(Self::default())
```

**Status**: Not yet fixed. Recommend opening a follow-up issue.
