## Findings

### 🔴 Critical

- **C-001 (P4) — R-V133P4-01 remains partial because existing `pending-review` CRUD still trusts request-supplied `creator_id` without active-creator enforcement.** The fix correctly added `AuthRequired` + active creator comparison to the new `POST /v1/local/memory/review` and `GET /v1/local/memory/fragments` handlers (`memory.rs:477-533`, `memory.rs:701-775`), but the same file's pre-existing `create_pending_review`, `list_pending_reviews`, `count_pending_reviews`, and `delete_pending_review` paths still only validate/request-match supplied creator IDs (`memory.rs:79-128`, `210-262`, `285-315`, `332-406`). This leaves cross-creator list/count/delete and arbitrary pending-row seeding available to any caller that can reach the local API, and it misses the assignment's explicit check to apply the fix to pending-review CRUD too. Recommended fix: extract one shared `require_active_creator_matches(nexus_home, requested_creator_id, resource)` helper and call it before DB operations in all creator-scoped memory handlers; add 401/403/404 tests for pending-review list/count/delete/create as appropriate.

### 🟡 Warning

- **W-001 (P3) — R-V133P3-02 is only fixed for embedded presets; disk-loaded user/system preset bundles still validate `template_file` paths but do not load their contents at runtime.** `with_resolved_template` calls `crate::preset::read_embedded_template(preset_id, path)` (`tasks.rs:579-605`), and `build_wired_outer_graph` invokes it with `loaded.id` (`loader.rs:862-868`). `LoadedPreset` does not retain the filesystem `bundle_root` (`loader.rs:25-49`), and `load_preset(bundle_root, ...)` performs sandbox validation then returns `load_preset_from_str(&yaml, caps)` (`loader.rs:288-331`), discarding the canonicalized bundle location. That means a supported user-installed preset under `~/.nexus42/presets/<id>/` with `exit_when.kind=llm_judge` and `template_file: prompts/foo.md` can pass A3 validation but still send the literal path string as the judge prompt. Recommended fix: resolve template contents during `load_preset`/`load_preset_from_str` using a source enum (`Embedded(id)` vs `Filesystem(bundle_root)`), or store a validated template resolver/bundle root in `LoadedPreset`; use canonicalize + `starts_with(bundle_root)` for filesystem reads.

- **W-002 (cross-cutting) — the assigned required orchestration test command is invalid as written.** `cargo test -p nexus-orchestration judge parse_judge_response all_embedded_presets` exits `1` with `unexpected argument 'parse_judge_response'`; Cargo accepts at most one test-name filter before `--`. Supplemental targeted checks passed (`parse_go`, `parse_nogo`, `all_embedded_presets`, and `llm_judge`), but the required command itself cannot pass. Recommended fix: update the PM/plan verification command to separate invocations or use `cargo test -p nexus-orchestration --lib judge` plus exact filters.

- **W-003 (P4) — R-V133P4-06's 256 KiB cap can panic on non-ASCII digests and is hard-coded.** `PassthroughSummarizer` truncates with `&raw_digest[..MAX_DIGEST_BYTES]` (`memory.rs:672-690`). Because Rust string slicing requires a UTF-8 character boundary, a digest whose byte 262144 lands inside a multibyte character will panic instead of safely truncating. The cap also lives as a local constant rather than configuration, which is acceptable for V1.33 but should be explicit. Recommended fix: truncate with `char_indices()` / `floor_char_boundary` equivalent before slicing, and consider plumbing the limit through daemon configuration later.

### 🟢 Suggestion

- **S-001 (P4) — R-V133P4-05 fixed the direct partial-write risk, but the temp path is deterministic for a slug.** `save_memory` now writes `<slug>.md.tmp` then renames (`memory_io.rs:153-162`), which is a clean minimal atomic-write pattern and cleans up on rename failure. Concurrent writes to the same creator/slug can still collide on the same temp path. This is not the original corruption mode, but a unique temp name in the same directory would make the pattern safer as the memory pipeline grows.
