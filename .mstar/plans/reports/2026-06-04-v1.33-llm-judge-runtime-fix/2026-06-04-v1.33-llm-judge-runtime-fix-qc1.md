## Findings

### 🔴 Critical

- **F-001 (P3) — `judge.llm` can treat an exact `NOGO` response as GO, so the new `llm_judge` path may advance when the judge explicitly says not to.** `JudgeLlm::run` asks the worker to respond with `GO` or `NOGO`, but `parse_judge_response` checks `GO_WORDS` first with substring matching. Because `"nogo"` contains `"go"`, an exact `NOGO` response matches the GO branch before the NOGO branch. The new P3 tests use `"NO — stop"`, not `"NOGO"`, so they do not exercise the capability contract. Recommended fix: parse normalized leading verdict tokens / word boundaries and check NOGO before GO, then add tests for exact `"NOGO"`, `"NOGO — reason"`, and `"GO — reason"` through both `judge_llm::parse_judge_response` and `StateCompositeTask`.

- **F-002 (P3) — `template_file` is still treated as inline prompt content, not loaded from the preset bundle, so real embedded presets send path strings such as `prompts/gathering-exit.md` to the judge.** The loader validates `template_file` as a safe, existing relative asset path, and P3’s locked goals require “Load and render `template_file` from bundle root.” However `StateCompositeTask` passes `template_file.as_deref()` directly into `LlmJudgeTask::new`, and `LlmJudgeTask::evaluate` renders that string as the template body. The tests pass inline strings (`"Evaluate: is gathering complete?"`) rather than a manifest path, so they miss the production path. This is not an acceptable runtime convention without a broader contract change, because existing manifest validation and preset authoring semantics say `template_file` is a file path. Recommended fix: resolve/load template content during preset loading or task construction (preferred), keeping path safety/existence validation in one loader-owned boundary; alternatively explicitly rename the manifest field and update docs/tests, but that is a breaking contract change.

### 🟡 Warning

- **F-003 (P4) — Keyword-filtered `GET /v1/local/memory/fragments` still uses runtime sqlx aliases with `!`, so the filtered path is likely a 500 and is untested.** The no-keyword branch works around `list_fragments_filtered` by calling compile-time `list_fragments` and truncating. But the keyword branch still calls `list_fragments_filtered`, whose runtime SQL aliases columns as `"fragment_id!"`, `"session_id!"`, etc. SQLite exposes those literal names (`fragment_id!|session_id!` confirmed), while the mapper calls `row.get("fragment_id")`, `row.get("session_id")`, etc. The result is a runtime column lookup failure for `keyword=...`. Recommended fix in this QC wave: remove `!` suffixes from runtime aliases or switch to compile-time checked static query variants; add an API test for `/v1/local/memory/fragments?creator_id=...&keyword=...`.

- **F-004 (P4) — Creator scoping remains split between body/query-supplied creator IDs and P1’s active-creator pattern, leaving the Local API ownership convention ambiguous.** The new review/fragments handlers match existing `pending-review` routes by validating `creator_id` from body/query, and the CLI supplies the active creator from `config.toml`. That is pragmatic for V1.33 memory-route parity, but it contradicts the P4 plan wording (“Auth: active creator from request context”) and the newer P1 `works` handlers’ `AuthRequired` / active creator pattern. Recommended fix: either document memory routes as explicit-creator legacy endpoints for P5 or introduce a shared Local API creator-context extractor and migrate memory + works toward one convention. This is an architecture/maintainability warning rather than a blocker in this lens because P4 follows the pre-existing pending-review pattern.

### 🟢 Suggestion

- **F-005 (P3) — `LlmJudgeTask` factoring is acceptable, but the old `JudgeTask` references and template comments should be cleaned up with the loader fix.** Replacing the rule-only `JudgeTask` with a purpose-built `LlmJudgeTask` is clearer than extending a stub, provided the loader boundary supplies actual template content. After the fix, update comments that currently mix “template_file path” with “template content” to prevent future readers from reintroducing the confusion.

- **F-006 (P3) — `WorkerUnavailable → WaitForInput` is the right safe default, but non-availability vs capability failure should be observable as separate operational signals.** Returning NOGO/WaitForInput prevents silent advancement and is safer than `Continue`. For non-`WorkerUnavailable` capability errors, the code propagates `TaskExecutionFailed` without local `tracing::warn!`/`error!`; add structured logging with state/capability/session context if the engine does not already log GraphErrors at a comparable level.

- **F-007 (P3) — `min_interval` parsing supports only `PT#H/#M/#S`; document backlog for full ISO-8601 duration support or adopt a parser.** The current parser is adequate for the immediate `PT6H` style examples, but ignores days/weeks/months and date components. Keep it as a tracked backlog item so manifest authors do not assume full ISO-8601 semantics.

- **F-008 (P4) — `PassthroughSummarizer` is a sound V1.33 abstraction seam, but review processing should return or log per-entry failures if it becomes user-visible.** The summarizer trait keeps the future LLM summarizer boundary clean. Today failed promote/fragment operations are logged and skipped while the endpoint returns only counts, which is acceptable for closing the loop but may make support harder; consider a future response envelope with skipped IDs/reasons.

- **F-009 (cross-cutting) — The assigned P3 verification commands are misleading without `--lib`.** `cargo test -p nexus-orchestration judge` and `cargo test -p nexus-orchestration all_embedded_presets` exited 0 but ran 0 tests in this workspace. The actual relevant tests run with `--lib` (`judge`: 41 passed; `all_embedded_presets`: 1 passed). Update future PM/plan verification commands to include `--lib` or exact test targets.

- **F-010 (cross-cutting) — Plan checkbox updates landed on the integration worktree before merge; keep this as a workflow observation.** The diff includes report-scope plan checkbox edits (`318df8c`) on the integration line. It is not a code issue, but future parallel tracks should keep plan bookkeeping tied to the responsible topic branch or PM consolidation step to reduce audit noise.
