## Findings

### 🔴 Critical

#### C-001: R-V133P3-02 — `template_file` still treated as inline template string, not loaded from bundle

**Scope**: P3 (`crates/nexus-orchestration/src/tasks/mod.rs:738`)

The fix wave did **not** address R-V133P3-02. `StateCompositeTask` still passes `template_file.as_deref()` directly into `LlmJudgeTask::new`, and `LlmJudgeTask::evaluate` renders that string as the template body. The tests use inline strings (`"Evaluate: is gathering complete?"`) rather than manifest paths, so they miss the production path.

**Evidence**:
```rust
// crates/nexus-orchestration/src/tasks/mod.rs:738
let template = template_file.as_deref().unwrap_or("");
// ...
let judge_task = LlmJudgeTask::new(template.to_string(), cap_name.to_string(), registry);
```

The `template_file` field in presets contains paths like `prompts/gathering-exit.md`, not inline template content. At runtime, the judge capability receives the raw path string as the prompt, which is incorrect.

**Status**: **NOT FIXED** — remains open from wave 1.

### 🟡 Warning

#### W-001: R-V133P3-04 — WorkerUnavailable liveness vector documented but not mitigated

**Scope**: P3 (`crates/nexus-orchestration/src/tasks/mod.rs:465-484`)

The fix wave added a detailed `R-V133P3-04 (deferred, documented)` comment explaining the liveness/DoS vector, but no code change was made to add a circuit-breaker, timeout, or rule fallback. The behavior remains: no worker → perpetual `WaitForInput`.

**Evidence**:
```rust
// R-V133P3-04 (deferred, documented): This creates a liveness/
// DoS vector — an attacker who controls worker connectivity can
// lock states in NOGO. Acceptable for V1.33 local-only single-user
```

**Status**: **DEFERRED WITH DOCUMENTATION** — acceptable for V1.33 local-only pre-release, but should be tracked for V1.34+.

#### W-002: W-P3-01 — Sequential LLM judge evaluations still add latency

**Scope**: P3 (`crates/nexus-orchestration/src/tasks/mod.rs:803`)

The fix wave made no changes to the synchronous per-state LLM judge pattern. Each state with `exit_when: llm_judge` still triggers a blocking IPC round-trip. No batching, parallelization, or caching was introduced.

**Status**: **STILL OPEN** — appropriate to defer for V1.33 (feature completeness priority).

#### W-003: W-P4-03 — `memory_fragments` keyword search still lacks database index

**Scope**: P4 (`crates/nexus-local-db/src/memory_fragment.rs:197-209`)

The `list_fragments_filtered` query uses `keywords LIKE '%"keyword"%'` with no index on the `keywords` column. The fix wave addressed the `!` alias bug (R-V133P4-04) but did not add an index. With a large fragment table, this forces a full table scan within the creator_id-filtered rows.

**Status**: **STILL OPEN** — appropriate to defer for pre-release (low volumes expected).

#### W-004: Test naming inconsistency in cross-creator isolation test

**Scope**: P4 (`crates/nexus-daemon-runtime/tests/memory_review_fragments_api.rs:313-346`)

The test `cross_creator_isolation_review_other_creator_returns_403` is misnamed. The test asserts `StatusCode::OK` (200), not 403, because the auth check passes (active_creator = ctr_alice, request creator_id = ctr_alice) and the DB query simply returns zero rows for Bob's entries. The behavior is correct (proper isolation), but the test name and comment claim 403 which is confusing.

**Evidence**:
```rust
// Line 314: "cross_creator_isolation_review_other_creator_returns_403"
// Line 340: resp.assert_status(axum::http::StatusCode::OK);  // NOT 403
```

**Fix**: Rename test to `cross_creator_isolation_review_filters_other_creator` to match actual behavior.

### 🟢 Suggestion

#### S-001: R-V133P3-03 — `parse_iso8601_duration` days parsing uses seconds arithmetic

**Scope**: P3 (`crates/nexus-orchestration/src/tasks/mod.rs:1349-1438`)

The fix wave extended `parse_iso8601_duration` to support `P1D`, `P7D` etc. by converting days to `days * 86400` seconds. This is correct for relative durations (throttle intervals) and DST-safe because it represents elapsed time, not wall-clock time. The `chrono::Duration::seconds(total_seconds)` approach is simpler than `Duration::days(1)` and avoids DST boundary issues.

**Status**: **FIXED — no action needed.**

#### S-002: `memory` auth checks add one config file read per request

**Scope**: P4 (`crates/nexus-daemon-runtime/src/api/handlers/memory.rs:783-791`)

The new `read_active_creator_id` helper reads `config.toml` on every request to `review` and `fragments` endpoints. This is ~1-2ms on SSD and acceptable for local daemon use. For high-throughput scenarios, consider caching the active creator in `WorkspaceState` or using a `OnceLock`.

**Status**: **Acceptable for V1.33.**

#### S-003: Test suite timing improved

**Scope**: Cross-cutting

Full test suite wall-clock improved from ~73s (wave 1) to ~49s (wave 2), a 33% improvement. This is likely due to incremental compilation artifacts from the fix wave build. Monitor CI duration going forward.
