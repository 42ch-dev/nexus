### 🟢 Suggestion

**S-001: Device ID caching across CLI invocations**
- Source: `crates/nexus42/src/main.rs` lines 234-243, `crates/nexus-sync/src/device_id.rs`
- Issue: `get_or_create_device_id` reads from disk on every CLI invocation. While acceptable for a CLI that typically runs once per session, it could benefit from a config-level cache.
- Suggestion: Store the resolved `device_id` in the CLI config and persist it, only regenerating if missing or corrupted. This would avoid unnecessary file I/O on every command.

**S-002: Exponential backoff for transient network errors**
- Source: `crates/nexus-sync/src/device_flow_client.rs` lines 176-247
- Issue: The polling loop handles `AuthorizationPending` and `SlowDown` but doesn't apply backoff to transient network errors like connection timeouts or DNS failures. `DeviceFlowError::Other` errors continue polling without any backoff.
- Suggestion: Add exponential backoff with jitter for transient network errors, capped at a reasonable maximum (e.g., 30 seconds). This would reduce unnecessary load on both client and server during outages.

**S-003: Add tests for network failure scenarios**
- Source: `crates/nexus-sync/src/device_flow_client.rs` tests
- Issue: Current tests don't simulate network failures (connection timeouts, DNS failures). The behavior of `DeviceFlowError::Other` is important for reliability but currently lacks test coverage.
- Suggestion: Add tests using wiremock's `ResponseTemplate` with `Delay` to simulate timeouts, or use mockito's connection failure simulation.

**S-004: Logging consistency for polling failures**
- Source: `crates/nexus42/src/auth/user_auth.rs` line 107
- Issue: The `poll_device_token` error path for `DeviceFlowError::Other` uses `eprintln!` with minimal context and only prints every 6 attempts.
- Suggestion: Use `tracing::warn!` with consistent formatting including the error message and attempt number for better observability during long polling sessions.

## Performance Review Details

### ✅ Device ID Resolution
- Device ID is resolved once at startup in `main.rs` before command dispatch
- `get_or_create_device_id` only called if `~/.nexus42/` directory already exists (post-init)
- UUID v4 generation is fast; file I/O is minimal (single read/create + chmod)
- File permissions set to 0600 (Unix) for security ✓

### ✅ Platform Client & Device Flow Client
- `PlatformClient::new` creates a fresh reqwest client per instance (30s timeout)
- `DeviceFlowClient::new` also creates a fresh reqwest client per instance
- Header injection for `X-Device-ID` happens at build time
- device_id is stored as an owned String on each client - reasonable for infrequent auth operations

### ✅ RFC 8628 Polling Loop (user_auth.rs)
- Interval enforcement: `tokio::time::sleep(interval).await` executed before each poll ✓
- SlowDown handling: increases interval by 5 seconds (RFC 8628 compliant) ✓
- Max attempts: capped at `(expires_in / interval).min(60)` - reasonable bound ✓
- Terminal conditions: exits on success, `ExpiredToken`, or `AccessDenied` ✓
- JWT user_id extraction uses base64 decode without verification - acceptable since token comes from platform over HTTPS ✓

### ✅ Resource Lifecycle
- HTTP connections managed by reqwest's internal pool (default behavior)
- No explicit resource leaks identified
- Auth store reads/writes use standard Rust fs operations with proper error handling
- Tokio timers properly async (no blocking executor)

### ✅ Concurrency
- CLI is single-process, single-threaded by default (tokio multi-threaded but auth operations are sequential)
- Auth store operations serialize via file I/O - no in-memory race conditions
- Device ID file creation: uses standard atomic write pattern (write file then chmod)
- No `unsafe` code or raw thread::spawn in reviewed modules

### ✅ Degradation Behavior
- Platform unreachable → `DeviceFlowError::Other`, polling continues with current interval
- Device ID file missing/corrupted → regenerates new UUID
- `~/.nexus42/` doesn't exist yet → device_id resolution skipped gracefully with tracing::warn
- Auth store read failures → propagated as CliError, not silently swallowed
- JWT parse failures → falls back to "unknown" user_id, doesn't break login flow

### ✅ Test Quality
- Device flow tests cover: client creation, normalization of trailing slashes
- Device code request: success + server error responses
- Token polling: success, authorization_pending, slow_down, expired_token, access_denied, server error
- JWT extraction: valid token, fallback to sub, invalid formats
- Auth store: roundtrip serialization, backward compatibility with existing files
- Device ID: create/read cycle, permissions on Unix, missing file regeneration
- Platform client: header injection verification
