### 🟡 Warning

**F-001: Job claim (next_queued + mark_running) lacks explicit transaction or row lock — race condition under concurrent orchestration**
- **File:** `crates/nexus-local-db/src/kb_extract_job.rs:191-217` (next_queued), `224-234` (mark_running); `crates/nexus-orchestration/embedded-presets/kb-extract/preset.yaml`; no consumer wiring visible in V1.29 diff for `mark_extract_job_*` outside CLI enqueue.
- **Severity:** Warning (Correctness)
- **Issue:** `next_queued` is a bare `SELECT ... WHERE status='queued' ORDER BY created_at ASC LIMIT 1`; `mark_running` is a separate `UPDATE`. No `BEGIN IMMEDIATE`, `FOR UPDATE`, or application-level claim token. SQLite default locking allows window for two `orchestrate run --preset kb-extract` (or daemon + CLI) to both claim the same job.
- **Impact:** Duplicate extraction work, wasted LLM calls, potential duplicate KeyBlocks or job state corruption (`running` -> `running` lost update). Failed jobs may never reach `mark_failed` if the preset graph does not yet wire the job_id through to error paths (current `KbExtractWork::run` is a placeholder returning "ready_for_acp_prompt").
- **Evidence:** DB functions use only `sqlx::query!` single statements; no `pool.begin()` or `with_transaction` wrapper in the module. Tests are single-threaded. Preset YAML shows capability call but no job state machine. Re-exports in `nexus-local-db/src/lib.rs` expose the raw fns.
- **Fix:** Wrap claim in `BEGIN IMMEDIATE; SELECT ...; UPDATE ...; COMMIT` (or use a `claimed_by` + `claimed_at` column + UPDATE ... WHERE status='queued' returning the row). Add integration test with two concurrent tasks. Wire `job_id` through the preset execution and ensure `mark_failed` on any error path in the extraction_graph / acp.prompt failure.

**F-002: Persistent child process cleanup (R-011) is best-effort, Unix-only, and has PID tracking / lifecycle races**
- **File:** `crates/nexus-agent-host/src/providers/native_cli/claude.rs:713-746` (Drop), `137-138` (persistent_pids), `843-919` (execute_spawn_persistent), `732-744` (kill -9 via std::process::Command).
- **Severity:** Warning (Security / Reliability)
- **Issue:** 
  - `Drop` iterates `persistent_pids` (Arc<Mutex<Vec<u32>>>) and shells out `kill -9 <pid>` synchronously; ignores all errors.
  - No `Child` handle retained for persistent mode (handles behind tokio::sync::Mutex in sessions map; Drop cannot await/kill cleanly).
  - Windows: only a warn log, child left orphan.
  - PID added on spawn, removed only on explicit `shutdown`; crash or panic can leave stale PIDs; no `waitpid` / reaping, no SIGTERM + grace + SIGKILL sequence.
  - `pids` list is append-only in the reviewed diff; removal paths not fully shown in all error branches.
- **Impact:** Orphan `claude` processes (resource leak, potential secret leakage if prompts contain PII), zombie accumulation on Unix, silent failure to terminate on host shutdown / panic. Attacker-controlled prompt or session_id could influence child if args not strictly validated (though current arg assembly looks controlled).
- **Evidence:** Drop uses `std::process::Command::new("kill")` not `tokio::process::Child::kill`; comment explicitly says "We don't have Child handles". `#[cfg(unix)]` only. Tests use `kill` checks post-facto.
- **Fix:** Retain `Arc<tokio::process::Child>` (or wrapped) for persistent sessions so Drop / shutdown can call `.kill().await` with timeout. Implement cross-platform (use `taskkill /F /PID` on Windows or `sysinfo` / `nix` crate). Add explicit `remove_pid` on normal shutdown paths + on child exit detection. Consider `std::process::Child` for the kill path or a dedicated reaper task. Add test that Drop actually terminates the observed PID.

**F-003: `generate_job_id` is time-based (millis + subsec nanos | shl20) rather than UUIDv4 / crypto random; collision risk (low but non-zero)**
- **File:** `crates/nexus-local-db/src/kb_extract_job.rs:38-44`
- **Severity:** Warning (Correctness)
- **Issue:** `job_id = "xj_" + 12 hex from (millis | (nanos as u64 << 20)) & 0xFFFF_FFFF_FFFF`. Relies on clock resolution and single machine; under clock skew, fast calls, or VM restore, duplicates possible before DB unique constraint saves it. The idempotency index is partial (excludes failed), so collision on active job would be masked as "already queued".
- **Impact:** Silent job reuse or enqueue failure under load/clock issues; harder to debug than opaque UUID. DB CHECK (job_id LIKE 'xj_%') + PRIMARY KEY prevents bad data, but generation is weak.
- **Evidence:** Function comment and impl; no `uuid` crate usage here (other parts of monorepo use proper IDs).
- **Fix:** Switch to `uuid::Uuid::new_v4()` (or feature-flagged deterministic for tests) formatted as `xj_<hyphenless-uuid12-or-full>`. Keep the 'xj_' prefix for human grep. Update migration CHECK if format changes.

**F-004: Placeholder implementation in `kb.extract_work` + missing job state transition wiring leaves extract jobs unmanaged on error / completion**
- **File:** `crates/nexus-orchestration/src/capability/builtins/kb_extract_work.rs:121-140` (placeholder returns "ready_for_acp_prompt"); `preset.yaml:28-46` (loading → extracting_graph → done, no job_id flow); no call sites for `mark_extract_job_*` outside CLI in the diff.
- **Severity:** Warning (Correctness / Data Integrity)
- **Issue:** The capability that is supposed to drive extraction currently does no LLM call, no JSON parse of real response, no KeyBlock insert, and no `mark_running` / `mark_done` / `mark_failed` on the `job_id` supplied in input schema. The preset graph calls it but has no error edge that calls `mark_failed`. If the real acp.prompt inside the inner_graph fails or panics, the queued job may stay `queued` or `running` forever (or until manual intervention).
- **Impact:** Stuck jobs, silent data loss (work KB never becomes world KB), incorrect `extract-status` output, user-visible "queued" that never progresses. Violates the documented lifecycle in the module docs.
- **Evidence:** Capability `run()` is almost entirely a no-op returning static JSON; tests only cover the placeholder path and `parse_extraction_response` (which is not even called from the capability yet). DB layer has the state machine and tests, but the consumer is incomplete in V1.29.
- **Fix:** Either (a) make `KbExtractWork` own the full flow (load via job_id, call acp.prompt internally or via context, parse, insert KeyBlock, then mark_*) or (b) have the preset / orchestration engine own the job_id and perform the marks in `on_enter` / `on_error` hooks. Add end-to-end test that enqueues, runs preset, and observes `done` + new KeyBlock.

**S-001: Add explicit `BEGIN IMMEDIATE` helper or `with_tx` wrapper around job queue mutations for future-proofing**
- Centralize in `nexus-local-db` so every `next_queued` + `mark_*` pair is obviously atomic. Even for single-writer daemon today, this eliminates a correctness class of bug when concurrency is later introduced (e.g., multiple workers or gRPC API).

- Currently the prompt template does not surface the job_id; makes debugging failed extractions harder.
