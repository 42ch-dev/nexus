## Findings

### 🔴 Critical

#### C-001: Corrupted identity cache can bypass creator-id path-safety validation

**Location**: `crates/nexus42/src/creator_identity.rs:118-135`

**Issue**: `resolve_creator_ref()` trusts cached `creator_id` values before validating them. If `creator-identities.json` is corrupted or maliciously edited, an unsafe creator ID can be returned through either an exact cache-key hit or a handle match. `use_creator()` then persists that returned value to `CliConfig.active_creator_id` (`crates/nexus42/src/commands/creator.rs:963-974`). Later code has path-safety checks in some paths, but this feature's success criterion is to persist a normalized canonical creator ID, not an unvalidated cache value.

**Security impact**: The local identity cache is best-effort display metadata and should not become a trusted authority for filesystem-safe IDs. A corrupted cache entry such as a handle mapping to `../etc` can cause `creator use <handle>` to persist an unsafe active creator ID, reintroducing the path-traversal class this plan was intended to close.

**Recommended fix**: Validate every resolved `creator_id` with `validate_creator_id_safe()` before returning or persisting it, including cache-key hits and handle matches. Treat invalid cached entries as corrupt and skip or error with a clear message. Add tests for corrupted cache entries where the key and/or `entry.creator_id` contains `/`, `\`, `..`, control characters, and empty values.

#### W-003: `CreatorAuthHeaders` derives `Debug` while containing bearer secrets

**Location**: `crates/nexus42/src/auth/creator_auth.rs:15-21`

**Issue**: `CreatorAuthHeaders` derives `Debug`, and the `authorization` field contains `Bearer <creator_api_key>` or `Bearer <user_access_token>`. Any accidental debug logging, panic output, or test failure formatting of this struct can expose secrets.

**Impact**: The plan explicitly requires auth helpers and tests not to leak secret values. Deriving `Debug` on a bearer-token container creates an avoidable secret-handling footgun.

**Recommended fix**: Remove the derived `Debug`, or implement a custom redacted `Debug` that prints only header presence and redacted token metadata.

#### W-004: `creator_auth_headers()` does not validate `creator_id` before header construction

**Location**: `crates/nexus42/src/auth/creator_auth.rs:38-56`

**Issue**: The user-token fallback copies `creator_id` directly into `x_creator_id` without validating it. Current callers may pass a normalized ID, but the helper is public and is intended to centralize Creator-context auth behavior.

**Impact**: Invalid or control-character creator IDs can propagate into header assembly. Downstream HTTP libraries may reject them, but this helper should enforce the auth boundary before returning header values.

**Recommended fix**: Validate `creator_id` at the start of `creator_auth_headers()` and test rejection of path separators, `..`, control characters, and empty values.

#### W-005: `creator use` input validation is path-safe but not ID-shape/non-empty validation

**Location**: `crates/nexus42/src/creator_identity.rs:133-135`, `crates/nexus-home-layout/src/lib.rs:176-195`

**Issue**: The fallback accepts any string that is merely path-safe. `validate_creator_id_safe()` rejects separators, `..`, and control characters, but it does not reject empty strings or enforce a creator-ID shape. The plan wording says unknown input should look like a valid/path-safe Creator ID.

**Impact**: Programmatic or shell-quoted empty input can become an active creator ID, and arbitrary safe aliases can be persisted as IDs instead of failing clearly.

**Recommended fix**: Add explicit non-empty and creator-ID-shape validation for the fallback path, or document that pre-1.0 compatibility intentionally accepts any non-empty path-safe value and add tests for that boundary.

### 🟢 Suggestion

#### S-001: Production `creator_auth_headers()` is not directly covered

**Location**: `crates/nexus42/src/auth/creator_auth.rs:38-65`, `crates/nexus42/src/auth/creator_auth.rs:199-232`

**Issue**: Tests use `build_headers_from_store()` instead of exercising `creator_auth_headers()` itself. This avoids disk I/O, but it duplicates the implementation and can drift from production behavior.

**Improvement**: Refactor header selection into a pure helper over `&AuthStore`, have `creator_auth_headers()` load the store and call it, and test the pure helper directly.
