### 🟡 Warning

#### W-1: Lost-update race on `creator-identities.json` under concurrent CLI invocations
**Source**: `crates/nexus42/src/creator_identity.rs:100-104`

`set_creator_identity()` implements a classic read-modify-write cycle without any inter-process locking:

```rust
pub fn set_creator_identity(entry: CreatorIdentityEntry) -> Result<()> {
    let mut cache = load_creator_identity_cache();  // Read
    cache.creators.insert(entry.creator_id.clone(), entry);  // Modify
    save_creator_identity_cache(&cache)  // Write
}
```

While `save_creator_identity_cache()` uses an atomic `rename()` to avoid producing a half-written file, **two concurrent `nexus42` processes can still interleave as follows**:

1. Process A reads cache {X}
2. Process B reads cache {X}
3. Process A inserts Y → cache {X, Y}, saves
4. Process B inserts Z → cache {X, Z}, saves
5. Final state is {X, Z}; **Y is lost**

This is not theoretical — a user running `nexus42 creator register` in two parallel terminals (or a script firing rapid registrations) will hit it. The identity cache is labeled "best-effort display data" in `commands/creator.rs:801`, but silent data loss in a local cache is still a reliability defect that complicates debugging and degrades user trust.

**Fix direction**: Acquire an advisory file lock (e.g., `fs2::FileExt` or `nix::fcntl::flock`) on `creator-identities.json` before the read-modify-write sequence. Alternatively, switch to an append-only log or SQLite (already a project dependency) where concurrent writes are handled by the engine.

#### W-2: `AuthStore::save()` writes directly to `auth.json` without atomic rename
**Source**: `crates/nexus42/src/auth/mod.rs:95-109`

`AuthStore::save()` performs:

```rust
std::fs::write(&path, &content)?;
```

This is **not atomic**. A crash or SIGKILL mid-write leaves `auth.json` truncated or partially written. On the next `AuthStore::load()`, `serde_json::from_str` will fail and propagate an error (not gracefully degraded), potentially locking the user out until they manually delete the corrupt file.

By contrast, `save_creator_identity_cache()` in the **same PR** correctly uses `tmp + rename`. The inconsistency is notable: the new code knows the right pattern, but the pre-existing auth store path does not apply it.

**Fix direction**: Change `AuthStore::save()` to write to `auth.json.tmp` and `rename()` into place, matching `creator_identity.rs`.

#### W-3: TOCTOU window in `ensure_valid_token()` between expiry check and rotation
**Source**: `crates/nexus42/src/auth/creator_auth.rs:107-126`

```rust
pub async fn ensure_valid_token(config: &CliConfig, creator_id: &str) -> Result<String> {
    let store = AuthStore::load()?;
    if let Some(creators) = &store.creators {
        if let Some(state) = creators.get(creator_id) {
            let expires = chrono::DateTime::parse_from_rfc3339(&state.expires_at)?;
            if chrono::Utc::now() < expires {
                return Ok(state.access_token.clone());
            }
        }
    }
    // Token expired or missing — rotate
    rotate_credentials(config, creator_id).await?;
    let store = AuthStore::load()?;  // Re-load after rotation
    ...
}
```

Two issues here:
1. **Time-of-check to time-of-use (TOCTOU)**: The expiry check happens against an in-memory snapshot. By the time the token is actually used in an HTTP request, it may have expired.
2. **Double `AuthStore::load()`**: After `rotate_credentials()`, the store is reloaded from disk. If another concurrent process also rotated credentials, the second load may see a different token state. This is not handled explicitly.

**Fix direction**: Add a short "freshness buffer" (e.g., reject tokens expiring within 30s) so the TOCTOU window is reduced. Document the concurrency expectation for token rotation (is it caller-serialized or not?).

#### W-4: Test `HOME` manipulation is not panic-safe within the test process
**Source**: `crates/nexus42/src/creator_identity.rs` (multiple tests)

All disk-touching tests follow this pattern:

```rust
let original_home = std::env::var("HOME").ok();
std::env::set_var("HOME", tmp.path());
// ... test body ...
if let Some(home) = original_home {
    std::env::set_var("HOME", home);
} else {
    std::env::remove_var("HOME");
}
```

If the test panics between `set_var` and restoration, `HOME` remains pointed at the temp directory for the remainder of the test process. While Rust's default test harness runs tests in separate processes, **this is not guaranteed** when:
- Running with `cargo test -- --test-threads=N` where N > 1 (tests share the process environment)
- Using custom test harnesses
- Future refactoring introduces `#[tokio::test]` with shared runtime threads

The tests happen to pass with `--test-threads=1` because each test runs in its own process and the env leak is harmless, but the pattern is brittle.

**Fix direction**: Use a RAII guard (e.g., a `TempHome` struct that restores `HOME` in `Drop`) so restoration is panic-safe. Or use a crate like `serial_test` to force serialization of env-mutating tests.

### 🟢 Suggestion

#### S-1: Add a concurrent-stress test for `set_creator_identity`
The identity cache module has no test exercising concurrent writes. A simple `#[test]` spawning two threads that each call `set_creator_identity()` with different entries, then asserting both entries exist in the final cache, would document the expected behavior and catch regressions if locking is added later.

#### S-2: Unify local-persistence atomic-write pattern across the CLI
There are at least three local JSON files now (`auth.json`, `config.toml`, `creator-identities.json`) with two different write patterns. A shared utility like `atomic_write_json(path, value)` would eliminate the inconsistency and reduce future copy-paste errors.

#### S-3: `load_creator_identity_cache()` silently swallows I/O errors
If `std::fs::read_to_string()` fails with `PermissionDenied` (e.g., file created by another user or with wrong umask), the function returns an empty cache with no log output. In a debugging scenario this makes the symptom ("my creators disappeared") hard to trace. Consider `tracing::warn!`-logging the error before returning the default.

#### S-4: `resolve_creator_ref()` does O(n) handle scan
Handle resolution iterates over all entries (`for entry in cache.creators.values()`). For the expected scale (tens of creators) this is fine, but if the cache grows to hundreds of entries, a secondary `HashMap<String, String>` (handle → creator_id) would provide O(1) lookup. Document the expected scale or add a `TODO` if platform integration may increase it.
