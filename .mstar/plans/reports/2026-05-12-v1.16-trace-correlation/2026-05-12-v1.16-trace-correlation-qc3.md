### 🟡 Warning

#### W-001: Trace files grow unbounded — no rotation, size limits, or TTL cleanup
- **Location**: `crates/nexus42/src/commands/acp_trace.rs:36-81`, `crates/nexus-home-layout/src/lib.rs:197-213`
- **Trigger**: Long-running daemon orchestration schedules or frequent `acp run` invocations accumulate `~/.nexus42/acp/runs/<run_id>/trace.jsonl` files indefinitely.
- **Impact**: Disk space leak. A busy user or daemon with many runs could exhaust home-directory quota. No metrics or alerts exist to observe growth.
- **Fix**: Add one or more guardrails in a follow-up plan (V1.17 recommended):
  1. Per-run size limit (e.g., 100 MiB) with rotation to `trace.jsonl.1`, or
  2. Run-directory TTL (e.g., auto-delete runs older than 30 days), or
  3. A `nexus42 system cleanup` subcommand that prunes old traces.
- **Residual**: Track as `R-001` — operational guardrail needed before GA.

#### W-002: Per-event file open/close/serialize overhead on hot path
- **Location**: `crates/nexus42/src/commands/acp_trace.rs:36-81`
- **Trigger**: Every `append_trace_event` call performs `create_dir_all` → `OpenOptions::new().create(true).append(true).open()` → `serde_json::to_string` → `write_all` → implicit drop/close.
- **Impact**: O(n) syscalls with no batching or handle reuse. For a run with hundreds of capability calls, this is measurable overhead (especially on spinning disks or network-backed home directories). Each event is independently serialized and flushed to the kernel page cache; no `BufWriter` or file-handle pool is used.
- **Fix**: For V1.17, consider buffering:
  1. Open the trace file once per run and hold a `BufWriter` in a run-scoped context, or
  2. Batch events and flush on a timer or explicit checkpoint.
  Current best-effort semantics must be preserved — buffered mode should still not fail the run on I/O error.
- **Residual**: Track as `R-002` — performance optimization deferrable to orchestration integration.

#### W-003: Concurrent writes to the same trace file risk interleaved bytes
- **Location**: `crates/nexus42/src/commands/acp_trace.rs:55-78`
- **Trigger**: If the same `run_id` is reused concurrently (e.g., daemon orchestration with multiple workers sharing a run ID, or a user running two `acp run --run-id <same>` instances), both processes open the same file in append mode and issue `write_all` without any POSIX advisory lock (`flock`) or atomic write guarantees.
- **Impact**: On some filesystems, `O_APPEND` guarantees atomicity only for writes <= `PIPE_BUF` (4 KiB on Linux). A large trace event (e.g., `error` field with a very long stack trace) could exceed this and result in interleaved JSONL lines, corrupting the file.
- **Fix**: Either (a) acquire an advisory lock (`flock`) around the write, or (b) document that `run_id` must be unique per concurrent execution and add a runtime check that fails fast if a trace file is already locked.
- **Residual**: Track as `R-003` — correctness under concurrency.

#### W-004: Best-effort I/O errors lack structured observability
- **Location**: `crates/nexus42/src/commands/acp_trace.rs:51-52, 62-63, 70-71, 76-77`
- **Trigger**: Disk full, permission denied, or filesystem remounted read-only during trace write.
- **Impact**: Errors are silently swallowed with `eprintln!` only. In production (daemon mode or CI), stderr may be discarded or rotated away. There is no `tracing::warn!` event, no metric counter, and no way for operators to detect that traces are being lost.
- **Fix**: Replace `eprintln!` with `tracing::warn!` (the project already depends on `tracing`). Additionally, expose a simple counter metric (e.g., `trace_write_failures_total`) that can be scraped by the daemon health endpoint in V1.17.
- **Cross-ref**: QC1 S-001 made the same observation. This elevates it from a style suggestion to a reliability concern because silent data loss is an operational risk.

### 🟢 Suggestion

#### S-001: Missing edge-case tests for concurrent writes and disk-full scenarios
- **Location**: `crates/nexus42/src/commands/acp_trace.rs:83-169`
- **Issue**: Tests cover happy path (append two events), ID format validation, and path-traversal rejection. Missing: (a) concurrent append from two threads/processes, (b) behavior when disk is full or directory is read-only, (c) very large event payload (>1 MiB), (d) corrupted JSONL recovery/rejection on read.
- **Fix**: Add at least a concurrent-append test using `std::thread::spawn` and verify file integrity. Disk-full simulation can use a small tmpfs mount or mock filesystem (optional for pre-1.0).

#### S-002: No explicit fsync / durability guarantee for trace events
- **Location**: `crates/nexus42/src/commands/acp_trace.rs:76-78`
- **Issue**: `write_all` returns `Ok` when data is in the kernel page cache. A crash immediately after write may lose the last event(s). This is acceptable for best-effort telemetry, but the lack of guarantee should be documented.
- **Fix**: Document in module docstring that trace writes are best-effort and may lose events on crash. If V1.17 needs stronger durability, add an optional `sync: bool` parameter or flush on run completion.

#### S-003: `append_trace_event` could validate event-run_id consistency
- **Location**: `crates/nexus42/src/commands/acp_trace.rs:36-81`
- **Issue**: The function receives both a `run_id: &NexusRunId` for the path and an `event: &NexusTraceEvent` that embeds its own `run_id`. A bug in the caller could write an event with a mismatched `run_id` into the wrong file.
- **Fix**: Add a debug-only assertion (or a cheap check in test builds) that `event.run_id() == run_id`. Zero runtime cost in release builds.
- **Cross-ref**: QC2 S-001 raised the same point. This is a low-cost defense-in-depth addition.

## Cross-Reviewer Ready Notes

### Performance & Reliability Assessment (Reviewer #3 focus)

**JSONL Append Efficiency — Acceptable for Seed, Needs Optimization for Scale:**
- Per-event `OpenOptions::open` + `serde_json::to_string` + `write_all` is correct and simple for V1.16 seed scope.
- Hot-path overhead will become noticeable once orchestration propagates trace events per capability call (V1.17). A `BufWriter` or run-scoped file handle should be added before GA.
- Estimated overhead: ~3-5 syscalls per event (open, write, close) plus JSON allocation. At 100 events/run this is negligible; at 10,000 events/run it is measurable.

**Trace File Growth — Operational Risk:**
- `~/.nexus42/acp/runs/<run_id>/trace.jsonl` has no size cap, rotation, or TTL cleanup.
- For CLI `acp run` (human-initiated, short-lived), this is fine — humans don't generate enough traces.
- For daemon orchestration (long-running, automated, potentially infinite loops), this is a disk-space leak.
- **Recommended residual**: Add a follow-up task in V1.17 for trace retention policy.

**Subprocess Env Impact — Negligible:**
- `spawn_with_env` adds only `.env(key, value)` calls to `tokio::process::Command`.
- No extra heap allocation beyond the `&[(&str, &str)]` slice on the stack.
- `NEXUS_RUN_ID` propagation latency is <1 µs; not a concern.

**Concurrency & Correctness — Gap Identified:**
- Concurrent append to the same `trace.jsonl` lacks file locking. `O_APPEND` is atomic only up to `PIPE_BUF` (4 KiB). Large error messages could corrupt the file.
- This is a latent bug that will manifest only when orchestration runs multiple capability tasks in parallel under the same `run_id` (V1.17).
- **Recommended fix**: Add `flock` around write, or enforce one-writer-per-run-id invariant.

**Observability — Silent Data Loss Risk:**
- `eprintln!` for I/O failures means trace loss is invisible in daemon/CI contexts.
- `tracing::warn!` + a counter metric would make this observable.
- QC1 S-001 noted the same pattern; this reviewer elevates it to a reliability concern.

**Test Coverage — Good for Contract, Thin for Edge Cases:**
- 8 DTO round-trip tests + 4 writer tests + 3 path-helper tests + 2 CLI contract tests = 17 trace-related tests pass.
- Missing: concurrency, disk-full, large payload, corrupted read.
- Pre-1.0 seed scope justifies deferring these; they should be added before GA.
