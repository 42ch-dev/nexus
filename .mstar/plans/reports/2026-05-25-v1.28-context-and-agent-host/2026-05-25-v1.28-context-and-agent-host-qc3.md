### 🟡 Warning

#### W-001: KbQuery filtering is in-memory in current backend (C2.2)

**Issue**: `InMemoryKbStore::query()` (crates/nexus-kb/src/store.rs:229-289) reads all blocks, then filters and paginates in Rust memory. A `KbQuery` with `limit=1` still iterates every active block in the world. The previous `list_by_world()` path had identical performance characteristics, so this is not a regression, but the query infrastructure isn't yet leveraging the structured query interface for efficiency gains.

**Evidence**: The `query()` method calls `self.read_blocks()` to get a `HashMap` snapshot, then runs `.values().filter(...)` on all entries before applying `skip(offset).take(limit)`.

**Risk**: O(n) scan per query where n = total key blocks in the store. Acceptable for the current in-memory store (likely < 10k blocks) but will become a performance bottleneck at scale.

**Fix**: The `KbQuery` struct and trait already forward limit, offset, and filter parameters to the backend. A future `SqliteKbStore` (referenced in comments) would push filtering to SQL `WHERE world_id = ? AND block_type = ? LIMIT ?`, resolving this at the DB level. Mark this as architectural debt, not a bug.

#### W-002: Persistent child processes survive daemon crash (DF-24/DF-18)

**Issue**: When `ClaudeCliProvider` is dropped without explicit `shutdown()` calls (e.g., daemon process crash, SIGKILL, panic unwind), persistent child processes survive as orphans. This is because `tokio::process::Child`'s `Drop` implementation does not kill the child process — it only detaches it.

**Evidence**: `crates/nexus-agent-host/src/providers/native_cli/claude.rs:873-881` — `PersistentHandles` contains `child: tokio::process::Child` with no custom `Drop`. The session entry is only removed from the HashMap in `shutdown()` (line 651-667), which calls `handles.kill().await` and `handles.wait_with_timeout(...)`. When the provider is dropped without `shutdown()`, the HashMap is dropped, `NativeSession` is dropped, `PersistentHandles` is dropped, but tokio's `Child` is NOT killed.

**Risk**: In daemon runtime scenarios:
- **Graceful shutdown**: The `HostManager` calls `shutdown()` on all sessions (verified by `shutdown_respects_timeout_and_does_not_hang` test). This path is safe.
- **Crash/unclean exit**: If the daemon process is killed (SIGKILL, panic, OOM kill), zombie child processes survive. On repeated restarts without cleanup, this could accumulate orphaned processes.
- **Drop path**: If a `ClaudeCliProvider` is dropped without calling `shutdown()` for any reason, the same orphan scenario applies.

**Fix**: Implement `Drop` for `PersistentHandles` that calls `child.kill()` (best-effort, non-blocking). For async cleanup, consider a `Drop` that calls `child.start_kill()` (tokio signal) without awaiting. Additionally, consider storing the PID in a well-known location (or a PID file) for daemon startup cleanup.

**Mitigation**: The existing `nt1_2_shutdown_kills_persistent_child` test verifies the happy path. The test uses `kill -0` to confirm the child is dead after `shutdown()`. This is correct for the designed shutdown flow.

#### W-003: Persistent mode cancel() is a no-op (DF-18)

**Issue**: `cancel()` for persistent-mode children does nothing — it only emits a tracing log (line 559-569). If a long-running operation hangs or blocks, there is no mechanism to interrupt the child process or the delimited event stream. The caller must wait for the prompt timeout (`timeouts.prompt_ms`, default 5 minutes per `default_prompt_ms()`) to fire.

**Evidence**: The `cancel()` method explicitly documents this limitation:
```
// For persistent-mode children, there's no cancel since the child
// stays alive between operations. The event stream will end on the
// next delimiter or timeout.
```

**Risk**: For per-invocation mode, cancellation is handled naturally by stdin close (child exits). For persistent mode, a stuck child (e.g., waiting for tool approval that never comes) cannot be interrupted. The only recourse is the session-level timeout or a full shutdown.

**Fix**: Implement a per-operation timeout watch in the delimited stream reader, or store a `CancellationToken` alongside persistent handles that `cancel()` can signal to interrupt the stream reader.

### 🟢 Suggestion

#### S-001: Token budget approximation could be more accurate (C2.3)

**Issue**: Cross-domain truncation uses a `chars/4` heuristic for token estimation. This approximates "approximately 4 characters per token" for English text. For CJK languages (Chinese, Japanese, Korean), the ratio is closer to `chars/1.5`. For code or mixed content, the ratio varies unpredictably.

**Evidence**: `crates/nexus-moment-context-assembly/src/moment.rs:227` — `let max_chars = max_tokens.saturating_mul(4);`

**Risk**: Budget overruns or underruns depending on content language. For a 2000-token budget:
- English: 8000 chars (about right for typical English)
- Chinese: ~3000 chars would suffice, but 8000 are allocated (underfill)
- Code: variable, typically overestimates

**Fix**: Consider a lightweight tokenizer library (e.g., `tiktoken-rs` with cl100k_base encoding) for accurate token counting. For the current pre-release phase, the chars/4 heuristic is acceptable and documented.

**Note**: The implementation correctly preserves personality (never truncated), breaks at line boundaries, and truncates domains in priority order (`world_state` > `timeline` > `world_kb` > `user_knowledge`). The ordering and line-boundary break are well-implemented.

#### S-002: Stderr drain task has no explicit lifecycle management (DF-18)

**Issue**: The stderr drain background task (`execute_spawn_persistent`, lines 852-866) is spawned via `tokio::spawn` without storing a `JoinHandle`. The task terminates naturally when the child process exits (stderr pipe EOF), but there is no explicit abort mechanism for cases where the child hangs with stderr still producing output.

**Evidence**: `tokio::spawn(async move { ... })` at line 855 — no `JoinHandle` captured, no `AbortHandle` created.

**Risk**: Low — the drain task performs only `tracing::warn!` calls and exits on EOF. If the child is killed during shutdown, the pipe EOF terminates the task. The only scenario where this could be problematic is if the child exits abnormally without closing stderr, which is extremely rare.

**Fix**: Store a `JoinHandle` or `AbortHandle` in `PersistentHandles` and abort the drain task in the `Drop` implementation (if added per W-002). This would also provide a hook for health checks (`is_stderr_drain_alive()`).

### Rationale

No critical performance or reliability issues found. All 182 agent-host tests pass, all 21 context-assembly lib tests pass. The three warnings are:

1. **W-001 (KbQuery in-memory)**: This is an architectural observation about the current in-memory backend — functionally correct, and the trait interface properly supports future DB-level optimization. Not a regression from the previous `list_by_world()` approach.

2. **W-002 (Orphaned children on crash)**: The designed shutdown path (via `HostManager`) correctly cleans up persistent children. The risk is limited to crash/unclean-exit scenarios, which is a known reliability concern for any process-manager architecture. The fix (adding `Drop` that calls `start_kill()`) is straightforward.

3. **W-003 (No cancel for persistent)**: This is a documented design tradeoff of the persistent mode. For the current testing/development use case, the prompt timeout provides an upper bound. Production use of persistent mode would need cancellation support, but this is a feature gap, not a bug.
