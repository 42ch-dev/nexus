### 🟡 Warning
- **C2.3 Token truncation (personality section boundary)**: The `split_stage0_personality` / `apply_cross_domain_truncation` implementation in `MomentContext` relies on heuristic `find("## Personality")` + byte slicing + `find("\n## ")` to extract the non-truncatable section before applying budget to other domains. While the current `Stage0Assembly::assemble()` output format (using `PERSONALITY_HEADING` const + `\n\n` + content) is compatible and the dedicated test (`cross_domain_truncation_respects_budget`) confirms personality content survives, the parser is fragile to markdown variations (extra blank lines after header, no trailing newline before next section, embedded `## ` strings in content, or future stage0 formatting changes). This could silently drop or duplicate non-personality stage0 content, or (in pathological cases) mis-classify sections.

  **-> Fix / residual**: Replace the string heuristic with either (a) a small dedicated section extractor that operates on the structured `Stage0Assembly` fields before string assembly (preferred — keep `personality` as a separate non-truncatable payload through the pipeline), or (b) a proper markdown parser (e.g., `pulldown-cmark` event stream or a strict `^## \S` anchored regex with header stack). Add negative tests for malformed/ambiguous stage0 output. Track as low-severity tech debt unless a concrete failure case is demonstrated.

- **DF-18 / DF-24 Native multi-turn process hygiene (minor observability gap)**: Persistent child stderr is drained by a detached `tokio::spawn` task with no retained `JoinHandle` or abort signal on session shutdown. In normal operation this is harmless (short-lived warning logs), but under high session churn or panic paths it could leave orphan tasks. Kill/wait paths in `PersistentHandles` and `shutdown()` are otherwise correct (try_wait, kill + timed wait, PID checks in tests).

  **-> Suggestion only (no blocking)**: Retain abort handles for the drain tasks (or use a bounded channel + explicit shutdown) if future requirements demand strict task cleanup. Current behavior is acceptable for V1.28.

### 🟢 Suggestion
- **DF-26 Cross-platform probe (documentation only)**: Primary path uses the `which` crate (safe, no shell, handles PATHEXT/symlinks/exec bits). Manual fallback (only when `which` internally fails) does pure `is_file()` checks inside PATH dirs — no shell, no `Command`, no format-into-exec. The code comment correctly calls out Windows edge cases. No command-injection surface exists.

  **-> Polish**: Consider making the fallback also attempt `which::which_in` for test dirs (already done in `scan_custom_path`) and add a one-line security note in the module doc that "all resolution paths are free of shell interpolation."

- **DF-25 Admission policy (race window documentation)**: `check_session_limit` and `check_before_launch` are called under manager locks in the observed paths. However, the policy struct itself is stateless and the check is a simple integer comparison. Document (or add an integration test) that the caller (`HostManager`) serializes launch attempts so the `current_sessions` snapshot cannot be stale at the point of the check.

- **CLI input validation (context command)**: `validate_world_id` (clap value_parser) correctly enforces `wld_[alnum]+`. Other flags (`--creator-id`, limits, booleans) flow into `MomentRequest` as typed `Option<String>/usize` with no further shell/exec paths. No injection risk observed (no `format!` into commands, no `eval`, no filesystem writes based on raw strings in the hot path). Good hygiene.
