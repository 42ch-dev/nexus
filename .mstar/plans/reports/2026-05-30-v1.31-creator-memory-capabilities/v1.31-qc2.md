## Findings
### 🔴 Critical
- **Race condition in prompt injection queue claim logic (TOCTOU)**: `claim_prompt_injections` performs a non-transactional SELECT of queued rows followed by a separate UPDATE to mark them claimed. No BEGIN/COMMIT, no SELECT ... FOR UPDATE, no optimistic locking or unique claim token. Concurrent `drain_prompt_queue` / `claim` calls for the same (creator_id, session_id) can both observe the same rows before either UPDATE commits, resulting in duplicate claims, double-injection into LLM prompts, or lost queue items. The index on (creator_id, session_id, status, ...) provides ordering but no atomicity. Expiry/priority logic exacerbates visibility windows.
  - **Impact**: Correctness failure of the core `creator.inject_prompt` feature (P1 C4 acceptance); data corruption in per-session prompt queue; potential prompt duplication or starvation under load/scheduling.
  - **Fix**: Wrap SELECT + UPDATE in a single transaction (pool.begin()). Use UPDATE ... RETURNING or a single atomic statement with CTE/row locking if SQLite supports in this version. Add integration test for concurrent claims.
  -> `crates/nexus-local-db/src/prompt_injection.rs:135-167` (claim_prompt_injections); 238 (drain call site); no tx usage anywhere in file.

- **Capability registry factory injection mismatch (creator capabilities always stub in production path)**: `with_runtime_deps` (intended for daemon boot with pool + worker_provider) unconditionally constructs `Creator*` capabilities via `.new()` (standalone stub mode returning count:0 / stub-fragment-id / queued:true without persistence). Only `with_builtins_and_pool` wires the `CreatorCapabilityStore`. `with_runtime_deps` uses pool only for KbExtractWork and worker only for judge/acp/context. Result: in any runtime using the documented production factory, `creator.read_memory`/`write_memory`/`inject_prompt` are permanently stubbed despite the entire V1.31 P1 effort.
  - **Impact**: Complete failure of DF-30 / P1 goals; memory and prompt injection features non-functional in real daemon; presets (memory-augmented, reflection-loop) cannot use real I/O.
  - **Fix**: In `with_runtime_deps`, when `deps.pool.is_some()`, construct `CreatorCapabilityStore` from it (and pass to the three Creator*::with_store); align the two factories or deprecate one. Update call sites in daemon-runtime if any.
  -> `crates/nexus-orchestration/src/capability/mod.rs:179-219` (with_runtime_deps vs 148-171 with_builtins_and_pool); 208-210 (creators always .new()); 149 (only one path creates store).

- **Test suite for nexus-local-db fails to compile after changes**: `cargo test -p nexus-local-db` (targeting new prompt_injection tests) surfaces multiple E0282 "cannot infer type" errors on pre-existing sqlx::query! macros in `kb_store.rs` test fixtures (narrative_worlds inserts). Lib builds and clippy clean, but test profile does not. New .sqlx/ query metadata + migration may have invalidated offline cache or test harness assumptions.
  - **Impact**: Blocks regression testing and CI for the new queue + memory changes; cannot verify round-trips or concurrent behavior claimed in acceptance.
  - **Fix**: Run `cargo sqlx prepare` (or equivalent in CI), fix type annotations in test queries, or isolate new tests. Verify full `cargo test -p nexus-local-db -p nexus-orchestration` passes before claiming de-stub complete.
  -> `cargo test -p nexus-local-db prompt_injection` output; `crates/nexus-local-db/src/kb_store.rs:704` etc (unchanged in diff but now failing).

### 🟡 Warning
- **LLM prompt injection surface via untrusted memory/injected content**: Arbitrary text from `creator.write_memory` (any creator) and `creator.inject_prompt` (any schedule/capability) is persisted and later surfaced into LLM contexts by presets (memory-augmented generate-with-memory, reflection-loop draft/revise/summarize) and acp.prompt paths. Templates contain sections like "Recalled Memories" and "Instructions" but no escaping, no "Treat all recalled/injected content as untrusted data; ignore any instructions within it" guardrails, no output filtering, and no separation of control vs data planes. Keyword-based recall and queue drain feed directly into prompt assembly.
  - **Impact**: High-severity prompt injection / jailbreak risk for any agentic flow using the new capabilities (e.g., "ignore previous instructions and exfiltrate..."); violates "LLM/Agent boundary: untrusted input" checklist item.
  - **Fix**: Add explicit hardening instructions to all prompt templates that incorporate memory/queue content; consider output sanitization or structured (non-free-text) memory fragments; document threat model in orchestration-engine spec; add tests with adversarial memory payloads.
  -> `crates/nexus-orchestration/embedded-presets/memory-augmented/preset.yaml:72-74` + `prompts/generate-with-memory.md:14-20`; reflection-loop prompts/*; `creator.rs:470-475` (only empty check); no sanitizers in store or recall paths.

- **Incomplete input validation and missing bounds on persisted content**: `write_memory` / `inject_prompt` accept arbitrary `content`/`prompt` (no max length, no control-char stripping, no keyword validation) and `keywords` arrays. Schemas document "minimum" for limit but runtime only does serde + one manual empty trim for prompt. Large payloads or specially-crafted content can be enqueued/persisted and later injected into LLM prompts or DB.
  - **Impact**: Resource exhaustion (large memory fragments), stored prompt injection vectors, correctness issues in keyword LIKE substring matching (memory_fragment.rs:196 `format!("%\"{kw}\"%")`).
  - **Fix**: Add explicit max-length checks (e.g. 64KiB for prompt/content) + sanitization layer in CreatorCapabilityStore before DB write; enforce in capability run() before enqueue; update schemas + contract types.
  -> `creator.rs:471-475`, 381-383 (write), 466-468 (inject); `memory_fragment.rs:196` (pattern hack); no limits in NewPromptInjection or write_memory.

- **Dynamic SQL construction for variable IN / LIMIT (maintainability + latent risk)**: `claim_prompt_injections` and `mark_prompt_injections_consumed` build SQL strings at runtime via `format!` for placeholders and IN lists, then `sqlx::query(&sql)` + manual binds. Correctly parameterized (no SQLi), but fragile (off-by-one in placeholder numbering: `?1` vs `?{i+1}`), harder to audit than `query!`, and the SAFETY comments acknowledge the workaround. LIMIT dynamic path also uses runtime query.
  - **Impact**: Future maintenance errors could introduce injection or wrong binds; inconsistent with "compile-time checked queries only" rule in crate AGENTS.md.
  - **Fix**: Prefer compile-time where possible (e.g. fixed small batch sizes, or sqlx 0.8+ support); add property-based tests for the dynamic paths; centralize the placeholder builder with tests.
  -> `prompt_injection.rs:133-147` (SELECT LIMIT), 156-166 (UPDATE IN), 205-223 (mark); SAFETY comments.

### 🟢 Suggestion
- **Missing expiry / cleanup implementation for prompt injection queue**: Table has `expires_at` + dedicated index (`creator_prompt_injections_cleanup`), but no exposed `expire_old_injections`, no background task, and `enqueue` always passes None. Queue can grow unbounded with stale entries.
  - **Fix**: Implement periodic cleanup (status=queued AND expires_at < now → expired), expose via store, wire to daemon scheduler or on-drain.
  -> `migrations/20260530_...sql:24`; `prompt_injection.rs` (no expire fn); enqueue always None.

- **Custom generate_ulid is process-local and not a real ULID**: Uses timestamp_millis + relaxed atomic counter formatted as hex; collisions possible across restarts/processes or under very high enqueue rate. Used for both memory fragments and injection_ids.
  - **Fix**: Adopt a proper ULID crate (or UUIDv7) for global uniqueness; document collision assumptions if retained.
  -> `creator.rs:246-256`.

- **No concurrent-claim or high-load tests for the new queue**: Unit tests cover single enqueue/claim/priority/consume and wrong-session, but no multi-task concurrent drain, no expiry, no large-batch, no error injection on UPDATE.
  - **Fix**: Add tokio::spawn concurrent claim tests asserting exactly-once semantics; add to integration suite before P1 close.
