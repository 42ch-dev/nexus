### 🟡 Warning

**W-001: `with_runtime_deps` omits pool injection for creator capabilities**
- The `CapabilityRegistry::with_runtime_deps()` factory constructs creator capabilities with `::new()` (no store), even when `deps.pool` is `Some`. This means the production boot path, if it uses `with_runtime_deps`, will silently have creator memory capabilities in standalone mode — returning stub data while the pool sits unused. Only `with_builtins_and_pool()` correctly injects the `CreatorCapabilityStore`.
- **Impact**: Silent data loss in production if daemon boot switches to `with_runtime_deps`. The creator capabilities (`read_memory`, `write_memory`, `inject_prompt`) silently return stubs.
- **Fix**: In `with_runtime_deps()`, add the same `CreatorCapabilityStore` construction from `deps.pool` as `with_builtins_and_pool()` does, or document clearly that `with_runtime_deps` is intended only for worker-backed capabilities and `with_builtins_and_pool` must be used for the full set.

**W-002: `claim_prompt_injections` SELECT-then-UPDATE race condition**
- `claim_prompt_injections()` performs a SELECT followed by an UPDATE without a transaction or `UPDATE ... WHERE status = 'queued' RETURNING` pattern. Under concurrent callers (multiple daemon threads or processes claiming from the same queue), two callers can SELECT the same rows, then both UPDATE them — resulting in duplicate delivery.
- **Impact**: In single-process, single-threaded SQLite (WAL mode), this is low risk since SQLite serializes writes. However, the code is architected as a general-purpose queue and does not document the single-writer assumption. Future multi-process or multi-pool use would hit this.
- **Fix**: Either (a) wrap in `sqlx::Transaction`, or (b) use `UPDATE creator_prompt_injections SET status = 'claimed', claimed_at = ? WHERE injection_id IN (SELECT injection_id FROM creator_prompt_injections WHERE creator_id = ? AND session_id = ? AND status = 'queued' ORDER BY priority DESC, created_at ASC LIMIT ?) RETURNING *`, or (c) document the single-writer assumption with a `// SAFETY:` comment.

**W-003: Keyword LIKE pattern embeds untrusted input in format string**
- `list_fragments_filtered()` and `count_fragments()` build the LIKE pattern via `format!("%\"{kw}\"%")` where `kw` comes from `keyword` parameter. While the result is then parameterized via `.bind()`, a keyword containing `"` or `%` or `_` characters could produce incorrect matches (not injection, but semantic corruption). For example, keyword `"test` would produce `%"test"%` which never matches anything.
- **Impact**: Incorrect search results for keywords containing special LIKE characters or quotes, not a security vulnerability (parameterized bind prevents injection).
- **Fix**: Escape `%`, `_`, and `"` in the keyword before formatting the pattern, or use a JSON-aware search approach (e.g., `json_each()` or full-text search).

**S-002: `generate_ulid()` is not a real ULID**
- `generate_ulid()` in `creator.rs` uses `SystemTime + AtomicU64` counter, not the ULID spec (Crockford Base32 encoding, 48-bit timestamp, 80-bit randomness). The hex format `{ts:016x}{counter:08x}` is unique enough for local use but doesn't interoperate with external ULID generators. Consider renaming to `generate_local_id()` or adopting the `ulid` crate.

## Rationale

The three warnings are architecturally significant:

1. **W-001** (factory inconsistency) is the highest priority — it creates a latent production bug path where a seemingly correct constructor silently degrades creator capabilities to stub mode. This needs a PM/architect decision on whether to fix the factory or deprecate it.

2. **W-002** (race condition) is low-risk under current single-process SQLite but represents a design debt that should be acknowledged (SAFETY comment) or fixed before multi-process deployment.

3. **W-003** (LIKE pattern) will cause incorrect behavior at the boundary (keywords with quotes or LIKE metacharacters), which is a functional correctness issue for users.

None are blocking release, but W-001 should be resolved before the merge target is promoted. The remaining suggestions are low-impact maintainability improvements.
