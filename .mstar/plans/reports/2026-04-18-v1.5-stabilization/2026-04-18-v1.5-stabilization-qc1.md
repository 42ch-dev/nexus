## Warnings

### W-1: `cancel` signal does not use `pause_schedule()` for running-cache consistency

**File**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs` (line ~542–545)
**Severity**: Warning
**Review range**: `git diff 9a3b901..HEAD` — signal_schedule handler, cancel arm

When `cancel` is received for a `running` schedule, the handler calls `supervisor.pause_schedule()` to clear the running cache, but the subsequent status UPDATE and tick use direct SQL:

```rust
// Line 543-544
if current_status_str == "running" {
    let _ = supervisor.pause_schedule(&schedule_id).await;
}
// Then direct SQL below to set status = 'cancelled'...
```

The `pause_schedule()` call is `let _ =`, meaning **errors are silently ignored**. If `pause_schedule()` fails (e.g. DB connection issue), the running cache retains a stale entry, but the cancel proceeds to set the DB status to `cancelled`. A subsequent `tick()` would see the schedule as both `Running` (in cache) and `cancelled` (in DB), potentially corrupting admission logic.

**Recommendation**: Change to proper error propagation (`?`) or at least `tracing::error!` the failure. Since `pause_schedule()` is already imported and working in the `pause` arm, reusing it with error handling is straightforward.

### W-2: `resume_schedule` has a TOCTOU race between status check and update

**File**: `crates/nexus-orchestration/src/schedule/supervisor.rs` (lines ~596–623)
**Severity**: Warning
**Review range**: `git diff 9a3b901..HEAD` — `resume_schedule()` method

The method reads the current status, verifies it is `paused`, then later performs an unconditional `UPDATE ... WHERE status = 'paused'`. Between these two steps, a concurrent `pause_schedule()` or `cancel` could change the status. The UPDATE's WHERE clause provides some protection (it won't match if status changed), but the **admission check** (`admit()`) is performed against a snapshot that may be stale by the time the UPDATE executes.

If two clients concurrently call `resume_schedule()` on the same paused schedule:
1. Both read `paused`
2. Both load all schedules and compute `admit()`
3. Both execute `UPDATE ... WHERE status = 'paused'` — only one succeeds (SQLite serializes writes)
4. The loser gets `0 rows affected` but proceeds to return `"running"` because it already computed `should_run = true`

The UPDATE returns success even with 0 rows affected in sqlx. The method should check `rows_affected()` after the UPDATE and return `NotFound` or `InvalidTransition` if no rows were updated.

**Recommendation**: After each `UPDATE` in `resume_schedule()`, check `result.rows_affected() == 0` and return an appropriate error. Alternatively, use a single atomic operation: `UPDATE ... SET status = CASE WHEN ... THEN 'running' ELSE 'pending' END WHERE schedule_id = ? AND status = 'paused'`.

### W-3: `Scheduler::tick()` duplicates DB query with `tick_clocked()`

**File**: `crates/nexus-orchestration/src/scheduler/mod.rs` (lines ~93–133)
**Severity**: Warning
**Review range**: `git diff 9a3b901..HEAD` — `Scheduler::tick()`

The `Scheduler::tick()` method:
1. Queries `creator_schedules` for `status = 'pending' AND scheduled_at IS NOT NULL AND scheduled_at <= ?` to get `due_rows`
2. Builds a `HashSet<ScheduleId>` of admitted IDs (but only iterates for logging)
3. Calls `supervisor.tick_clocked(now)` which **re-queries all schedules** and does its own filtering

The `due_rows` query is effectively dead code — the `admitted_ids` set is built but never used to prevent anything, and the supervisor's `tick_clocked()` already handles all the admission logic. This wastes one DB query per poll cycle.

**Recommendation**: Either (a) remove the `due_rows` query entirely and just call `supervisor.tick_clocked(now)` directly, or (b) pass the `due_rows` IDs to the supervisor so it doesn't need to re-query. Option (a) is simpler and more maintainable.

### W-4: DST safety constraint not satisfied by `SystemClock`

**File**: `crates/nexus-orchestration/src/scheduler/mod.rs` (module doc, lines 7–11)
**Severity**: Warning
**Review range**: `git diff 9a3b901..HEAD` — module doc and `SystemClock` impl

The module documentation claims **DST/clock-jump safety** as constraint #4: "DST / clock-jump safety: `ClockSource` abstraction allows mock for testing." However, `SystemClock` simply returns `chrono::Utc::now().timestamp()` with no comparison to elapsed time. True DST safety requires detecting when `SystemTime::now() - boot_instant != wall_clock_elapsed` and adjusting accordingly.

The test `dst_clock_jump_recomputes_next_run` (`cron_trigger.rs`) only verifies that the scheduler fires when `clock.set()` jumps past `scheduled_at` — which is correct behavior for a mock clock but doesn't exercise any DST detection logic.

This is acceptable for pre-1.0 but the docstring overstates the guarantee. The `ClockSource` trait **enables** future DST detection but doesn't implement it.

**Recommendation**: Clarify the docstring to state that DST detection is a pre-1.0 gap tracked as a residual, or implement basic `SystemTime` vs `Instant` comparison in `SystemClock::now_unix()`.

## Suggestions

### S-1: `CoreContextManager::schedule_guards` unbounded growth (R6)

**File**: `crates/nexus-orchestration/src/schedule/derivation.rs` (lines 48–52)
**Severity**: Nit

The `HashMap<String, Arc<Mutex<()>>>` grows indefinitely as schedules are created. For a pre-1.0 local daemon this is acceptable (SQLite itself caps practical scale), but a `Weak<Mutex<()>>` + periodic cleanup or `entry().or_insert_with()` pattern with an LRU would be cleaner for long-running daemons.

### S-2: `signal_schedule` start path bypasses supervisor for running-cache update

**File**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs` (lines ~469–478, 609–631)
**Severity**: Nit

The `start` signal uses `match current_status_str => "running"` then falls through to direct `UPDATE ... SET status = ?` at line 611. This bypasses the supervisor's `inner.running_by_creator` cache, unlike `pause` and `resume` which properly use `pause_schedule()` and `resume_schedule()`. The `signal_schedule` method does call `supervisor.tick().await` at line 630, which will refresh the cache, but the window between UPDATE and tick is a consistency gap.

**Recommendation**: Use a dedicated supervisor method for the `start` transition, consistent with `pause`/`resume`.

### S-3: `signal_schedule` cancel arm — pause errors silently swallowed

**File**: `crates/nexus42d/src/api/handlers/orchestration/schedules.rs` (line ~544)
**Severity**: Nit (also covered by W-1)

`let _ = supervisor.pause_schedule(&schedule_id).await;` ignores errors. Even if this is acceptable for cancel (we want to cancel regardless), logging the failure would aid debugging.

### S-4: Missing test for `tick_clocked()` vs `tick()` re-entrancy interaction

**File**: `crates/nexus-orchestration/src/schedule/supervisor.rs` (tests_t9 module)
**Severity**: Nit

The shared `tick_in_progress` flag between `tick()` and `tick_clocked()` is documented but not tested. A concurrent test spawning `tick()` while `tick_clocked()` is in progress would verify the H4 guard works correctly across both entry points.

### S-5: `AddScheduleRequest.scheduled_at` uses string for Unix timestamp

**File**: `crates/nexus-contracts/src/local/schedule/http.rs` (lines 34–36)
**Severity**: Nit

The `scheduled_at` field is `Option<String>` holding a Unix timestamp as a string. While this avoids JSON number precision issues for large timestamps, a dedicated `Option<i64>` with serde `#[serde(with = "as_i64_string")]` or similar would be more type-safe. The CLI parses ISO-8601 and converts to a string timestamp, which is an extra parse step.

## Positive Observations

1. **R6 per-schedule locking** — The `HashMap<String, Arc<Mutex<()>>>` pattern is correctly implemented. The `schedule_guard()` method acquires the HashMap lock only for entry lookup/creation, then releases it before acquiring the per-schedule Mutex. This avoids holding two locks simultaneously and prevents deadlocks. The concurrency test (`r6_concurrent_apply_same_schedule_produces_sequential_versions`) is an excellent race-condition test.

2. **R2 duplicate detection** — Using `(creator_id, preset_id, label)` as a uniqueness key is reasonable for pre-1.0. The `COALESCE(label, '')` normalization handles NULL vs empty-string equivalence. Tests cover both rejection and allowance of different labels.

3. **R3+R7 smart resume** — The `resume_schedule()` method correctly re-evaluates admission rules before transitioning. The `should_run` computation rebuilds the full running/completed sets from DB and calls `admit()`. The fallback to `pending` + `tick()` is sound. Tests cover both the direct-to-running and fallback paths.

4. **R5 delete cascade** — The delete handler properly handles non-terminal schedules by canceling the session first, NULLing `current_session_id`, canceling the schedule, then deleting. The test `r5_cannot_delete_schedule_that_is_a_dependency_target` correctly verifies FK constraint enforcement.

5. **ClockSource abstraction** — The `ClockSource` trait with `SystemClock` and `MockClock` is a clean, dependency-free design. `MockClock` uses `AtomicI64` for thread-safe time manipulation, enabling deterministic testing without `tokio::time::pause()`.

6. **Test quality** — 5 cron tests cover the core scenarios (scheduled-at trigger, duplicate prevention, graceful shutdown, DST jump, existing tick behavior). WS-A residual tests are comprehensive and well-named.

7. **Code quality** — Good use of SAFETY comments for runtime SQL, proper error mapping to HTTP status codes, consistent naming, and thorough inline documentation. The `status.json` pre-merge checklist is well-followed.

8. **WS2 R4: SIGTERM→SIGKILL escalation** — The `worker/manager.rs` changes correctly implement graceful shutdown with configurable grace period, SIGTERM first, then SIGKILL escalation. Uses `nix` crate for POSIX signals.

## Cross-Reviewer Ready Notes

- **Integration risk**: The `cancel` path (W-1) and `start` path (S-2) have different consistency guarantees than `pause`/`resume`. Other reviewers should verify that the HTTP handler's error handling doesn't mask state corruption.
