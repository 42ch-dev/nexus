## Warnings

### W1: Platform Portability â€” `nix` crate unconditionally included (WS2 R4)
- **File**: `crates/nexus-orchestration/Cargo.toml`, `crates/nexus-orchestration/src/worker/manager.rs`
- **Finding**: The `nix` crate (v0.29, features=["signal"]) is added unconditionally to `Cargo.toml` and used for `kill(pid, SIGTERM)` in the worker manager. This will fail to compile on Windows.
- **Impact**: Prevents Windows builds. The project appears Unix-focused (daemon, SIGTERM semantics), but this is an undeclared platform constraint.
- **Recommendation**: Either (a) gate `nix` behind `#[cfg(unix)]` with a Windows fallback that skips SIGTERM and uses `child.kill()` directly, or (b) document in `AGENTS.md` or `README` that Windows is not a supported target. Given pre-1.0 status, option (b) is acceptable if explicitly noted.
- **Severity**: low (pre-1.0, Unix-focused project)

### W2: Recovered sessions lack runners on first `run_step` (WS2 R1)
- **File**: `crates/nexus-orchestration/src/engine.rs`
- **Finding**: `recover_sessions()` adds recovered sessions to the `sessions` map but **not** to the `runners` map. The doc comment correctly states this is intentional ("runners are created lazily when `run_step` is called"). However, if `run_step` is called on a recovered session before a runner is created, `run_step_internal` will return `EngineError::NoGraphLoaded`.
- **Impact**: A recovered session cannot be resumed until its runner is recreated. This is acceptable if the engine's caller (e.g., supervisor) handles `NoGraphLoaded` by re-initializing or cancelling the session.
- **Recommendation**: Verify that the supervisor or engine caller has logic to handle `NoGraphLoaded` for recovered sessions. If not, consider storing the graph/preset_id alongside the session summary so a runner can be recreated on demand.
- **Severity**: low (documented behavior, but verify caller handling)

### W3: `Outbox::new` now triggers all nexus-local-db migrations (WS8 R4)
- **File**: `crates/nexus-sync/src/outbox.rs`
- **Finding**: The inline DDL was replaced with `nexus_local_db::run_migrations(pool.inner())`. This runs **all** migrations in `crates/nexus-local-db/migrations/`, not just outbox tables.
- **Impact**: If `nexus-local-db` gains future migrations that conflict with `nexus-sync`'s usage pattern, this coupling could cause issues. Currently the migrations are independent and idempotent, so this is safe.
- **Recommendation**: Acceptable for now, but consider adding a comment noting this cross-crate migration dependency. If migration scope expands, this may need to be decoupled.
- **Severity**: low

## Positive Observations

1. **EngineSharedState extraction is clean** (WS3 R1). The `EngineSharedState` struct properly consolidates `storage`, `runners`, and `sessions`, and both `GraphFlowEngine` and `EngineProxy` delegate to it. The `run_step_internal` method eliminates duplicate status-translation logic.

2. **Arc<FlowRunner> is idiomatic** (WS2 R3). `FlowRunner::run(&self, ...)` takes `&self`, so wrapping in `Arc` is safe and avoids clone overhead. The `cloned()` call in `run_step_internal` correctly clones the `Arc` (cheap) rather than the `FlowRunner` (expensive).

3. **Session recovery is idempotent** (WS2 R1). `recover_sessions` checks `!sessions.iter().any(|s| s.session_id == summary.session_id)` before inserting, preventing duplicates on repeated recovery calls.

4. **SIGTERMâ†’SIGKILL sequence is correct** (WS2 R4). The supervisor sends SIGTERM, waits for the configurable grace period, then escalates to SIGKILL. Error paths are handled (SIGTERM failure falls back to `start_kill`). The grace period is configurable via `spawn_with_grace`.

5. **SAFETY comments follow canonical format** (sqlx R1). All SAFETY comments in `pool.rs` use the standard pattern: `SAFETY: <reason> â€” compile-time macro not applicable. <explanation>`.

6. **FromRow structs extracted to module level** (sqlx R2). `OutboxRow` and `PartialApplyRow` are defined once at module scope instead of being duplicated in each function. This reduces code duplication and improves maintainability.

7. **E2E assertions are meaningful** (WS3 R2). The new assertions verify: (a) at least one step executed, (b) inner graph nodes ran if brainstorming was reached, (c) the session ends in a valid terminal or waiting state.

8. **context.summarize capability is well-tested** (WS-C). The stub includes 6 unit tests covering: name, input/output schema validity, happy path, stub marker, missing content error, optional trace, and long content truncation.

9. **Preset loader validates capability existence** (WS-C). `ContextUpdateOp::LlmSummarize` is validated against the `CapabilityRegistry`, with both positive and negative test cases.

10. **apply_llm_summarize uses per-schedule locking** (R6). The method acquires the same per-schedule mutex as `apply_seed`, `apply`, and `apply_preset_hook`, preventing version chain corruption under concurrent writes.

11. **DerivationStep::LlmSummarize uses #[non_exhaustive]** (WS-C). This is the correct forward-compatibility pattern â€” the variant can be added without breaking downstream consumers, and the `llm_summarize()` constructor provides a controlled creation path.
