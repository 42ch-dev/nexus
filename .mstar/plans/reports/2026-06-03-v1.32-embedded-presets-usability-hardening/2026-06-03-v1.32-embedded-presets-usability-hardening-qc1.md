### 🟡 Warning
- **W-001 — Runbook examples still document a non-existent `--var` schedule interface.** P2 acceptance B6 requires manual smoke commands to be real and to avoid documenting commands that do not exist. The refreshed README and preset comments correctly remove stale `daemon orchestrate run`, but they now use `nexus42 daemon schedule add --preset ... --var ...` in multiple places (`embedded-presets/README.md`, `kb-extract/preset.yaml`, `memory-augmented/preset.yaml`, `novel-writing/preset.yaml`, `reflection-loop/preset.yaml`, `soul-experience-refresh/preset.yaml`). The actual CLI help for `nexus42 daemon schedule add --help` is `Usage: nexus42 daemon schedule add [OPTIONS] --preset <PRESET> --creator <CREATOR>` and exposes `--seed`, `--label`, etc.; it does **not** expose `--var`. Source confirms `ScheduleCommand::Add` requires `--creator` and has no `var` argument (`crates/nexus42/src/commands/daemon/schedule.rs:24-71`). This makes the new manual smoke snippets fail for users. -> Fix the runbook snippets to match the shipped CLI surface, likely using `--creator <id>` plus the supported seed/core-context path, or explicitly document that arbitrary preset input vars are not currently expressible through `schedule add` if that is the current product limitation.

- **W-002 — The embedded preset “strict” smoke test downgrades all `CapabilityArgDrift` errors category-wide, which can hide future real drift.** In `all_embedded_presets_pass_strict_validation_gate`, every semantic diagnostic with `severity == Error` and `category == CapabilityArgDrift` is moved to `all_warnings` (`crates/nexus-orchestration/src/preset/mod.rs:950-957`). The observed warnings include not only the implementer-flagged orchestration-layer args (`prompt_file`, `vars`) but also missing required capability args (`creator.inject_prompt` requires `prompt`) for `novel-writing` and `research`. Because this is a category-wide exception over all embedded presets, a later embedded preset could introduce an unrelated capability argument mismatch and the smoke test would still pass. That weakens the P1 gate that P2 is meant to enforce. -> Keep R-P2-01/R-P2-02 as non-blocking if capability semantics are out of P2 scope, but make the exception narrow and explicit: allowlist the known preset/path/message combinations or update the validator/capability schema to understand orchestration-layer `prompt_file`/`vars`, while preserving hard failure for unexpected `CapabilityArgDrift` diagnostics.

### 🟢 Suggestion
- **S-001 — Consider locking the `soul-experience-refresh` always-true rule comment with a targeted test as well.** The TD-V131-08 memory-augmented behavior is now covered by `memory_augmented_rule_exit_is_explicit_always_true`, and `soul-experience-refresh` documents the same bare `rule` semantics. If soul refresh is intended to share the same explicit contract, add a small parallel test so future validator changes cannot silently reinterpret that preset.
