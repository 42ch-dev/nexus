### Warning (4)

#### W1: Test duplication between `loader.rs` inline and `tests/preset_validation.rs` — maintenance burden

**Severity**: Warning
**Files**: `crates/nexus-orchestration/src/preset/loader.rs` vs `crates/nexus-orchestration/tests/preset_validation.rs`

The following test scenarios appear verbatim (same YAML, same assertions) in both locations:
- `reject_unknown_next_state` — identical YAML, same path/error checks
- `reject_missing_capability` — identical YAML, same error substring
- `reject_inner_graph_cycle` — identical cycle YAML
- `reject_conditional_next` / `reject_conditional_next_not_yet_supported`
- `reject_unknown_judge_capability`
- `reject_unknown_inner_graph_reference`
- `reject_terminal_with_next` / `reject_terminal_state_with_next`
- `reject_invalid_initial_state`
- `reject_invalid_terminal_state` / `reject_invalid_terminal_state_ref`
- `reject_unknown_depends_on_in_inner_graph`

This duplication means any future change to validation logic requires updating tests in two places. The integration tests should cover *different* scenarios (e.g., filesystem-level loading, end-to-end flows) rather than repeating the unit tests.

**Recommendation**: Keep the comprehensive YAML-level validation tests in one place (preferably `tests/preset_validation.rs` for integration) and remove the duplicates from `loader.rs` inline `#[cfg(test)] mod tests`. Or conversely, keep unit tests inline and make the integration tests exercise distinct scenarios (file I/O, embedded preset loading, etc.).

#### W2: E2E novel-writing tests have silent/weak assertions — do not verify end-to-end behavior

**Severity**: Warning
**Files**: `crates/nexus-orchestration/tests/e2e_novel_writing.rs`

Three of six E2E tests discard their results or make trivial assertions:

- `e2e_inner_graphs_execute` (line 126): `let _ = (outcomes, has_inner_output);` — computes `has_inner_output` but never asserts on it. The test comment says "that's OK for E2E smoke" but a smoke test should at minimum assert the session reached *some* expected state.
- `e2e_schedule_advance_past_outlining` (line 166): `let _ = status;` — advances but never checks the resulting status.
- `e2e_novel_writing_traverses_all_outer_states` — only asserts the session is not `Running`, which is trivially true after any `WaitingForInput`.

These tests exercise the code paths but do not verify *correctness*. They are essentially compile+run smoke tests, not functional verification.

**Recommendation**: At minimum, assert that the session reached `WaitingForInput` or `Completed` after the expected number of steps. For `e2e_schedule_advance_past_outlining`, verify the status changed from the pre-advance state.

#### W3: `StateCompositeTask` creates fresh `CapabilityRegistry` per capability execution — ignores injected registry

**Severity**: Warning
**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, lines 511–513

```rust
let cap_task = CapabilityTask {
    registry: std::sync::Arc::new(CapabilityRegistry::with_builtins()),
};
```

The `StateCompositeTask` struct has `engine: Option<Arc<dyn OrchestrationEngine>>` and is built via `with_engine()`, but the capability execution path creates a *new* `CapabilityRegistry::with_builtins()` on every enter action. This means:
1. Any future ability to inject a mock/custom registry into `StateCompositeTask` is impossible for capability enter actions.
2. Performance cost of re-creating the registry per capability call.
3. Inconsistency: inner graph execution uses `self.engine` (injected), but capability execution uses a hardcoded registry.

**Recommendation**: Add an optional `registry` field to `StateCompositeTask` and use it (falling back to `with_builtins()` only if `None`). Add a `with_registry()` builder method.

#### W4: `load_preset()` (directory-based) is a stub — contradicts "T6 embedded presets" being complete

**Severity**: Warning
**File**: `crates/nexus-orchestration/src/preset/loader.rs`, lines 155–167

```rust
pub fn load_preset(
    _bundle_root: &Path,
    _caps: &CapabilityRegistry,
) -> Result<LoadedPreset, PresetLoadError> {
    Err(PresetLoadError::Validation { ... "not yet implemented (WS3 T6)" ... })
}
```

The commit `3f2ea0f` claims "embed novel-writing preset with 5 states and 2 inner graphs (WS3 T6)" — but the *directory-based* loader (`load_preset`) remains unimplemented. This is acceptable if T6's scope is specifically *embedded* presets (via `load_embedded_preset`), but the function signature and doc comment suggest it should load from a bundle directory.

**Recommendation**: Either implement the filesystem loader or rename/re-document to clarify that T6 only covers `include_dir!`-based embedding, not directory loading. If directory loading is deferred, update the doc comment on `load_preset` to say "deferred to WS7" or similar.

### Suggestion (3)

#### S1: `AcpPromptTask::render_template` uses naive string replacement, not handlebars

**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, lines 761–779

The doc comment (line 706) says "Renders the template with `handlebars`" but the actual implementation uses a simple `while let Some(start) = rendered.find("{{")` loop. The `handlebars` crate is not in `nexus-orchestration/Cargo.toml`.

This works for `{{core_context.version}}` but will break with handlebars features like `{{#if}}`, `{{#each}}`, `{{../parent}}`, whitespace control, or nested expressions.

**Recommendation**: Either (a) add `handlebars` dependency and use the real library, or (b) update the doc comment to say "simple `{{key}}` placeholder substitution (not full handlebars)".

#### S2: `InnerGraphTask` context serialization on every key copy is expensive

**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, lines 236–245

```rust
if let Ok(parent_data) = serde_json::to_value(&context).await { ... }
```

This serializes the entire context to JSON just to extract keys with a given prefix. For large contexts this is O(N) serialization + O(N) deserialization per key prefix.

**Recommendation**: If `graph_flow::Context` supports direct key iteration or prefix filtering, use that. Otherwise, add a TODO comment acknowledging the performance concern.

#### S3: `InnerGraphTask` polls child with hardcoded `256` max iterations

**File**: `crates/nexus-orchestration/src/tasks/mod.rs`, line 262

```rust
for _ in 0..256 {
```

No timeout or step counter is exposed to callers. If a child session enters an infinite loop or long wait, this silently stops after 256 steps with no clear error (the `last_error` is `None` if the loop simply exhausts iterations).

**Recommendation**: Return a timeout error or log a warning when the max iterations are exhausted without the child reaching `Completed`.
