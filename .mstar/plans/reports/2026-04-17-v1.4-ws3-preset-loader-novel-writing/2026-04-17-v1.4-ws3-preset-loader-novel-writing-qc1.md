## Critical Findings (must fix before Approve)

### C-1: CapabilityTask creates a fresh CapabilityRegistry on every execution

**File**: `crates/nexus-orchestration/src/tasks/mod.rs` L510–512

```rust
let cap_task = CapabilityTask {
    registry: std::sync::Arc::new(CapabilityRegistry::with_builtins()),
};
```

**Problem**: `StateCompositeTask::run()` instantiates a brand-new `CapabilityRegistry::with_builtins()` for every capability enter-action. This means:
1. Any custom capabilities registered by the daemon (e.g. plugin capabilities in future V1.5+) are **never** visible to the composite task.
2. The `CapabilityTask.registry` field is **dead code** — the field exists, but the task ignores its own field and creates a fresh registry every time.
3. This breaks the adapter-layer design principle: the daemon should wire a single registry at engine startup, and all tasks should reference that singleton.

**Spec alignment**: §4.4 says tasks are "pure over Context + well-typed capability handles — no global state." A hard-coded `with_builtins()` call is effectively global-state bypass.

**Fix**: `StateCompositeTask` should hold an `Arc<CapabilityRegistry>` (passed via `with_engine` / builder pattern) and reuse it for all capability dispatches. `CapabilityTask` should be constructed once per state, not per execution.

**Evidence**: `git diff 1e75c54..0d0539b -- crates/nexus-orchestration/src/tasks/mod.rs` lines 510–512.

### C-2: requires_capabilities field is NOT validated against registry

**File**: `crates/nexus-orchestration/src/preset/loader.rs` — `validate_manifest()` function (L176–320)

**Problem**: The spec §7.2 defines `requires_capabilities` as `[string[]] — loader rejects preset if any missing`. However, `validate_manifest()` only validates capabilities referenced in `enter` actions and `exit_when.judge_capability`. The `manifest.preset.requires_capabilities` list is **completely ignored** during validation.

**Impact**: A preset could declare `requires_capabilities: ["acp.prompt", "judge.llm"]` but if those capabilities happen to not be registered, the loader would still accept the preset — as long as the YAML doesn't reference them in enter/exit_when. This defeats the purpose of the declares-what-you-need contract.

**Fix**: Add a validation pass that iterates `manifest.preset.requires_capabilities` and checks each against `caps.get(name)`.

**Evidence**: `git diff 1e75c54..0d0539b -- crates/nexus-orchestration/src/preset/loader.rs` — `validate_manifest()` contains no reference to `requires_capabilities`. The grep for `requires_capabilities` in that file shows it appears only in test YAML fixtures, never in validation logic.

## Warning Findings (should fix; increases risk of future regressions)

### W-1: Missing preset.id format validation

**Spec**: §7.2 says `id: [string, /^[a-z][a-z0-9._-]*$/] — must match dir name`

**File**: `crates/nexus-orchestration/src/preset/loader.rs` `validate_manifest()`

**Problem**: No regex/pattern validation on `manifest.preset.id`. A preset with `id: "INVALID"` or `id: "123-bad"` would pass validation. The system preset `_system.maintenance` (underscore prefix) is actually valid per the pattern since `_` is not in `[a-z0-9._-]` — wait, actually `_` IS in the character class, so `_system.maintenance` matches. But there's still no enforcement.

**Fix**: Add regex validation for `preset.id` in `validate_manifest()`. Consider `^[a-z_][a-z0-9._-]*$` to accommodate the `_system.*` prefix convention.

### W-2: Missing preset.description length validation

**Spec**: §7.2 says `description: [string, <=240 chars]`

**File**: `crates/nexus-orchestration/src/preset/loader.rs` `validate_manifest()`

**Problem**: No length check on `manifest.preset.description`. A 10,000-character description would pass.

### W-3: InnerGraphTask polling loop has hard-coded iteration cap with silent truncation

**File**: `crates/nexus-orchestration/src/tasks/mod.rs` L261

```rust
for _ in 0..256 {
```

**Problem**: If an inner graph has more than 256 tasks (or the graph-flow runner produces >256 steps before completion), the loop exits silently without an error. The subsequent `output_binding` read would operate on a partially-executed child session, producing incorrect or empty output.

**Fix**: After the loop, check if the child actually completed. If not, return an error: `"InnerGraphTask: exceeded max steps (256) before completion"`. Consider making the limit configurable or deriving it from the graph's task count.

### W-4: E2E test assertions silently swallow results

**File**: `crates/nexus-orchestration/tests/e2e_novel_writing.rs`

- L126: `let _ = (outcomes, has_inner_output);` — test body is a no-op
- L166: `let _ = status;` — test body is a no-op

**Problem**: These are labeled as tests but assert nothing. If the inner graph execution path is broken, these tests would still pass. The comment on L125-126 says "that's OK for E2E smoke" but a smoke test should at minimum assert that the session reached a known state.

**Fix**: Add minimal assertions: verify `has_inner_output` is true, verify status is one of {Completed, WaitingForInput}.

---

### W-5: EngineProxy duplicates GraphFlowEngine logic (DRY violation)

**File**: `crates/nexus-orchestration/src/engine.rs` L217–394

**Problem**: `EngineProxy` implements `OrchestrationEngine` by duplicating ~90% of the logic from `GraphFlowEngine` (run_step, new_session, start_session_with_graph, get_status, signal, list_active, spawn_child_session, get_context). The only differences are in `start_session_with_preset` wiring and `new_session` returning `NoGraphLoaded`.

**Risk**: Any bug fix or feature addition to `GraphFlowEngine` must be manually mirrored in `EngineProxy`. This is a maintenance hazard.

**Fix**: Consider making `GraphFlowEngine` implement `Clone` or use interior mutability patterns so the proxy can delegate to the real engine instead of reimplementing it. Alternatively, extract shared logic into a helper module.

## Suggestions (nice-to-have; no blocking impact)

### S-1: template_file path validation deferred to "Future"

The spec §7.6 lists "Any template_file path escapes the bundle root or does not exist" as a rejection criterion. The code at `loader.rs` L154–155 has a `TODO` comment: "Future: also reads prompt templates and validates template_file paths." This is acceptable for WS3 scope (embedded presets only) but should be tracked as a residual for when filesystem loading lands.

### S-2: load_preset() directory loader is a stub

`loader.rs` L155–167: `load_preset()` always returns a Validation error saying "not yet implemented (WS3 T6)." This is by design — T6 implements embedded loading via `load_embedded_preset()` which delegates to `load_preset_from_str()`. However, the public API `load_preset` is misleading; consider renaming the directory loader or adding a feature-gate.

### S-3: JudgeTask is a stub heuristic parser, not an LLM delegate

`tasks/mod.rs` L360–401: `JudgeTask` uses a hardcoded `always_true`/`always_false` stub. The `judge.llm` capability (`builtins/judge_llm.rs`) does a simple keyword heuristic parse rather than delegating to an actual ACP agent. This is acceptable for WS3 scope but should be tracked for the full LLM-backed judge implementation.

### S-4: ExitWhen::Timer variant handled as ManualWait

`tasks/mod.rs` L589–593: Timer exit_when falls through to `WaitForInput` with a resume flag. This is a reasonable stub for V1.4 but should be documented.

### S-5: Context serialization roundtrip for child session inheritance

`tasks/mod.rs` L236–245: InnerGraphTask serializes the parent context to JSON, then extracts matching keys. This is functional but fragile — if `graph_flow::Context` serialization changes, key inheritance could silently break. Consider a typed inheritance API.

### S-6: cargo fmt --check shows diffs in generated code

Running `cargo fmt --check` produces diffs in `crates/nexus-contracts/src/generated/` files. These are **not** from the WS3 feature commits (0 changed lines in generated/ in the diff range). The diffs are due to the stable rustfmt channel not honoring the `ignore` directive in `.rustfmt.toml`. Per the AGENTS.md convention, formatting generated Rust is skipped on nightly. This is a pre-existing environmental issue, not a WS3 finding.

## Architecture & Design Observations

### Strengths
1. **Clean trait boundary**: `OrchestrationEngine` trait properly insulates callers from `graph_flow`. `EngineError` is a good error abstraction.
2. **Manifest types in nexus-contracts**: Hand-written types in `nexus-contracts::local::orchestration::preset` correctly follow `schemas-boundary.md` §3 (local types, not wire contracts).
3. **Preset YAML structure matches spec**: The embedded `novel-writing/preset.yaml` closely mirrors the §7.2 example — 5 states, 2 inner graphs, correct output_binding references.
4. **Good test coverage for validation**: 14+ unit tests in `loader.rs` covering unknown states, missing capabilities, cycles, conditional next, etc.
5. **Builder pattern for StateCompositeTask**: `with_engine()`, `with_inner_graphs()`, `with_output_bindings()` is clean and allows partial construction for parsing-only scenarios.

### Integration Risks
1. **No filesystem loading yet**: The `load_preset(bundle_root, caps)` function is a stub. Only `load_embedded_preset()` works. User-installed presets are not supported in WS3. This is acceptable scope-wise but means the daemon API `reload_preset` handler only works on embedded presets.
2. **acp_worker.rs diff (372 lines added)**: The full worker command was reviewed via diff — it implements JSON-RPC 2.0 over stdin/stdout with `jsonrpsee-core`. This is a large, critical path that deserves a separate focused review (out of scope for this QC, but flagged for the team).
3. **Schedule CLI is a thin HTTP client**: `schedule.rs` delegates entirely to the daemon via `DaemonClient`. No local execution — correct architecture, but means the CLI is only a proxy.
