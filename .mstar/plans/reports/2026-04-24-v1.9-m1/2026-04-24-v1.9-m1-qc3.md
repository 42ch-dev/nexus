### 🟡 Warning

#### [W-1] Preset lookup scans all user presets unnecessarily (O(n) vs O(1))
**Severity**: Medium (performance degradation with many user presets)
**Files**: `crates/nexus-orchestration/src/preset/mod.rs`, `crates/nexus-orchestration/src/user_preset_dir.rs`

**Issue**:
In `resolve_preset()`, when looking for a specific preset by ID, the code calls `scan_user_presets()` which:
1. Reads the entire user presets directory
2. Opens and parses **every** preset's YAML file
3. Builds a complete `UserPresetScanResult` with all presets

This is O(n) disk I/O and YAML parsing when only O(1) is needed - we know the exact preset ID being looked for. With 100 user presets installed, this would parse 100 YAML files when only 1 is needed.

**Location**: `resolve_preset()` in `crates/nexus-orchestration/src/preset/mod.rs:108-150`

**Impact**:
- CLI commands like `preset list` correctly use the fast path (`scan_user_presets_local` which only checks directory existence)
- But `resolve_preset()` used during actual preset loading pays the full O(n) cost
- Preset load time scales linearly with number of installed presets, not just the preset being used

**Fix**:
Add a `load_single_user_preset(id: &str, home: &Path, caps: &CapabilityRegistry)` function that:
1. Only checks for `~/.nexus42/presets/{id}/preset.yaml`
2. Reads and parses only that single file
3. Returns `Option<LoadedPreset>` instead of scanning everything
4. Use this in `resolve_preset()` before falling back to the embedded preset

#### [W-2] JoinHandle abort() risk documented but not mitigated
**Severity**: Medium (resource leak potential during rapid shutdown)
**File**: `crates/nexus42d/src/lifecycle/actions.rs`

**Issue**:
The `exit_starting()` function cancels in-flight subsystem start tasks using `JoinHandle::abort()`, but as documented in the code comment (RISK-WSC-02):
> "abort() cancels the tokio task but does not guarantee full resource cleanup"

The risk is acknowledged but not mitigated. Subsystem tasks holding resources (sockets, file handles, DB connections) that are aborted may not run their drop handlers or cleanup logic.

**Location**: `exit_starting()` in `crates/nexus42d/src/lifecycle/actions.rs:169-185`

**Impact**:
- During shutdown while still in Starting state, resources may leak
- File descriptors, sockets, or database connections may not be properly closed
- Could lead to "address already in use" errors on rapid daemon restart

**Fix/Recommendation**:
1. The `SubsystemBootstrap` trait should have a `cancel()` or `shutdown()` method that performs graceful cleanup
2. `exit_starting()` should call this method BEFORE calling `abort()` on the JoinHandle
3. Tasks should be designed to check a cancellation flag periodically rather than relying solely on abort

#### [W-3] Mutex try_lock() in hot path could lose handles under contention
**Severity**: Medium (race condition during subsystem startup)
**File**: `crates/nexus42d/src/lifecycle/actions.rs`

**Issue**:
In `enter_starting()`, after spawning each subsystem task, the code uses `try_lock()` to acquire the mutex:
```rust
let handles = ctx.start_handles.try_lock();
if let Ok(mut handles) = handles {
    handles.push(handle);
} else {
    tracing::warn!("start_handles mutex contested during enter_starting — handle not tracked");
}
```

If the mutex is contended, handles are **silently dropped** with only a warning. Untracked handles won't be aborted in `exit_starting()`, leaving tasks running after state transition.

**Location**: `enter_starting()` in `crates/nexus42d/src/lifecycle/actions.rs:153-161`

**Impact**:
- High concurrency during daemon boot could cause handle tracking failures
- Untracked tasks continue running after exit from Starting state
- Potential resource conflicts, double initialization, or race conditions

**Fix**:
Use `.lock().await` instead of `try_lock()` to guarantee all handles are tracked:
```rust
let mut handles = ctx.start_handles.lock().await;
handles.push(handle);
```
This is in a synchronous context called from a state transition handler - if async lock is problematic, use a `std::sync::Mutex` with blocking lock acquisition.

#### [W-4] Drift detection test may have long execution time with many schemas
**Severity**: Low (developer experience, CI runtime)
**File**: `crates/nexus-contracts/tests/schema_drift_detection.rs`

**Issue**:
The schema drift detection test processes 52 schemas with one or more structs each. For each schema:
1. The JSON schema is parsed and properties are extracted
2. Dummy values are recursively generated for all properties (including $ref resolution)
3. Each checker deserializes and re-serializes the test JSON
4. Field comparison is performed

This is O(n) where n is the total number of schema properties across all schemas. With 52+ schemas, this could become slow as the schema count grows.

**Location**: `schema_drift_detection()` test in `crates/nexus-contracts/tests/schema_drift_detection.rs:761-841`

**Impact**:
- Developer iteration speed affected during schema-related work
- CI test times increase linearly with schema additions
- `$ref` resolution recursion depth could become problematic with deeply nested schemas

**Optimization Opportunities**:
1. Cache parsed schemas and extracted properties across test runs (already partially done via `schema_cache`)
2. Consider `#[ignore]` by default and run only in CI or via explicit flag
3. Add a parallel iteration feature for schema processing (behind `#[cfg(feature = "parallel")]`)
