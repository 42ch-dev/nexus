### 🔴 Critical

#### F-QC2-01: `drive_to_running()` helper in test file dispatches `Event::ProcessStarted` that has no handler in `Starting` state
- **File**: `crates/nexus42d/tests/lifecycle_starting_edges.rs`
- **Line**: 17
- **Issue**: The helper function `drive_to_running()` dispatches `Event::ProcessStarted` before subsystem ups. Reviewing `state.rs` lines 80–137 (`starting()` state handler) — `ProcessStarted` is not handled; it falls through to `Super`, which propagates up to `Top` (no handler there either) and is silently ignored. This means the test helper dispatches an event that does nothing, making the test's assumption about state progression subtly wrong.
- **Fix**: Remove the `m.dispatch(Event::ProcessStarted)` call from `drive_to_running()` — it is inert and misleading. The state machine starts in `Starting` automatically; no `ProcessStarted` event is needed or handled.
- **Source Type**: manual-reasoning
- **Confidence**: High

### 🟡 Warning

#### F-QC2-02: `serde_yaml::from_str` called without depth/decompression limits on untrusted user YAML
- **File**: `crates/nexus-orchestration/src/preset/loader.rs` line 123
- **Issue**: `serde_yaml::from_str` is used to parse user-supplied YAML from `~/.nexus42/presets/<name>/preset.yaml`. `serde_yaml` does not enforce depth limits or bytecode size limits by default. A maliciously crafted YAML with deep nesting or huge scalar values could cause stack overflow or memory exhaustion during parsing. While this is a low-probability attack vector (attacker already has filesystem write access to the preset directory), it violates defense-in-depth principles.
- **Fix**: Set a depth limit via `serde_yaml::Deserializer::new(&yaml).set_limit(128)` or similar. Alternatively, pre-check YAML size before parsing (> 1 MB reject).
- **Source Type**: manual-reasoning
- **Confidence**: Medium

#### F-QC2-03: `init_preset_at` rejects names starting with `_` but `scan_user_presets_local` silently skips them — inconsistent UX
- **Files**: `crates/nexus42/src/commands/preset.rs` lines 77–83, line 239
- **Issue**: `init_preset_at` explicitly rejects preset names starting with `_` (returning `CliError`). But `scan_user_presets_local` silently skips directories starting with `_` (no error, no warning). If a user tries `nexus42 preset init --name _foo` they get an error. If they manually create `~/.nexus42/presets/_foo/` with a `preset.yaml`, it is silently invisible to `preset list`. This inconsistency could confuse users who create system-like presets.
- **Fix**: In `scan_user_presets_local`, emit a warning (or collect into a separate "skipped" list) for `_`-prefixed directories rather than silently ignoring them. At minimum, log at `debug!` level with a message indicating why it was skipped.
- **Source Type**: manual-reasoning
- **Confidence**: High

#### F-QC2-04: `exit_starting` uses `blocking_lock` on tokio sync mutex — may cause deadlocks in single-threaded tokio runtime
- **File**: `crates/nexus42d/src/lifecycle/actions.rs` line 176
- **Issue**: `exit_starting` calls `ctx.start_handles.blocking_lock()` where `start_handles` is `tokio::sync::Mutex<Vec<tokio::task::JoinHandle<()>>>`. In a single-threaded tokio runtime, `blocking_lock()` will deadlock if the current thread holds the async context that needs to release the lock. The `enter_starting` at line 154 uses `try_lock()` (non-blocking), so in-flight handles may not be stored when `exit_starting` tries to abort them.
- **Fix**: Change `start_handles` to `std::sync::Mutex` (a regular std mutex is appropriate here since `enter_starting` stores handles synchronously and `exit_starting` drains them synchronously — no async context needed). Alternatively, use `futures::lock::Mutex` with `await` in both enter/exit.
- **Source Type**: manual-reasoning
- **Confidence**: Medium

### 🟢 Suggestion

#### F-QC2-05: `SchemaEntry::mode` field annotated `#[allow(dead_code)]` on `Subset` variant is misleading
- **File**: `crates/nexus-contracts/tests/schema_drift_detection.rs` line 43
- **Issue**: The `#[allow(dead_code)]` on `CheckMode::Subset` suppresses the "unused variant" warning, but the `Subset` variant is intentionally kept for future use (per comment "Rust may have extra internal fields"). This is fine as documentation intent, but dead code warnings on enum variants that are only used in test code are a code smell.
- **Fix**: Instead of `#[allow(dead_code)]` on the enum variant, use `#[cfg(test)]` on a dedicated test-only wrapper type that uses `Subset`, keeping the production code clean.
- **Source Type**: static-analysis
- **Confidence**: Low

#### F-QC2-06: `resolve_ref_file` silently falls back to raw string for unresolvable `$ref` paths
- **File**: `crates/nexus-contracts/tests/schema_drift_detection.rs` lines 666–687
- **Issue**: When `resolve_ref_file` cannot parse a URL or resolve a path (line 676: `file_part.to_string()` as fallback), the calling code (`make_dummy_from_ref`) falls through to returning `Value::String("dummy".to_string())` (line 661). This means `$ref` references to unmapped schema paths silently produce dummy strings rather than failing the test. A broken `$ref` in a schema could therefore produce false-positive drift detection results (missing fields reported as "dummy").
- **Fix**: Make `resolve_ref_file` return a `Result` and propagate errors up, so unmappable refs cause test failures rather than silent fallback.
- **Source Type**: manual-reasoning
- **Confidence**: Medium
