### 🟡 Warning

**W-1: CapabilityRegistry::get() uses O(n) linear scan instead of O(1) lookup**

The `CapabilityRegistry` stores capabilities in a `Vec<Box<dyn Capability>>` and its `get()` method performs a linear scan (`self.capabilities.iter().find(|c| c.name() == name)`). While the current built-in registry has only 16 capabilities, this is a latent performance anti-pattern. As the registry grows or if preset validation becomes a frequent operation (e.g., called repeatedly during preset authoring workflows), the O(n) per-lookup cost will compound across states and enter actions.

**Impact:** Latent hotspot. Each capability invocation in a preset triggers one linear scan. A preset with 10 states × 2 enter actions × 16 capabilities = 320 string comparisons currently; this scales multiplicatively.

**Fix:** Replace `Vec<Box<dyn Capability>>` with `HashMap<String, Box<dyn Capability>>` (or `HashMap<String, usize>` indexing into a Vec) since the registry is built once at daemon startup and never modified afterward. This gives O(1) lookups with minimal code change.

**Source:** `crates/nexus-orchestration/src/capability/mod.rs:265-270`
**Confidence:** High

### 🟢 Suggestion

**S-1: Capability input schemas are re-parsed on every validation invocation**

`check_capability_arg_compatibility` calls `serde_json::from_str::<serde_json::Value>(schema_str)` for every capability enter action. Pre-parsing schemas into `serde_json::Value` at registry construction time would eliminate repeated parsing overhead, especially for presets with many capability invocations.

**Source:** `crates/nexus-orchestration/src/preset/validation.rs:594-596`
**Confidence:** Medium

**S-2: Asset file existence checks are uncached across validation calls**

`validate_assets_in_bundle` re-stats all referenced files on every call. For presets with many template/prompt references, this issues multiple syscalls per validate request. Since validation may be called repeatedly during authoring workflows, caching the bundle directory listing (e.g., with a `HashSet<PathBuf>` of existing files) would reduce syscall overhead.

**Source:** `crates/nexus-orchestration/src/preset/validation.rs:488-512`
**Confidence:** Medium

**S-3: Consider adding an explicit state count limit**

While BFS reachability is bounded by state count and YAML size/depth limits provide defense, there is no explicit cap on `manifest.states.len()`. A pathological preset with thousands of states could cause validation to take longer than expected before other limits kick in.

**Source:** `crates/nexus-orchestration/src/preset/validation.rs:185-258`
**Confidence:** Low
