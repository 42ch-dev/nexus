## Goal

Harden the workspace OCC (optimistic concurrency control) surface by closing all 6 V1.56 medium workspace OCC residuals — sha2 dependency scoping, path boundary canonicalization, concurrent integration test, async I/O migration, TOCTOU window mitigation, and metrics/tracing at conflict points. Simultaneously converge capability surface quality by absorbing 3 medium + 7 low registry-refresh residuals (tracing gaps, reqwest reuse, help text polish, retry jitter, latency benchmarking, and structured metrics). Close 2 process residuals (fmt drift, .sqlx/ cache hygiene). Absorb V1.57-new low residuals (AC5 wording reconciliation, per-ID failure-path test vectors). This plan delivers the hardened foundation on which P2 (capability quality convergence) builds.

## Scope

- **T1**: Resolve `R-V156P0-M001` — scope sha2 dependency to workspace-managed. Ensure `sha2` is declared only in the crate that actually uses it for OCC version hashing; remove from transitive public dependencies.
- **T2**: Resolve `R-V156P0-M002` — path boundary canonicalization. Add path canonicalization on workspace open; reject symlink traversal attempts; enforce workspace root prefix boundary so sessions cannot escape their workspace directory.
- **T3**: Resolve `R-V156P0-M003` — concurrent integration test for OCC. Write an integration test exercising two concurrent writers on the same workspace session; assert that OCC conflict is detected and surfaced with a stable error code.
- **T4**: Resolve `R-V156P0-M004` — blocking sync I/O → async. Audit workspace session handler for blocking `std::fs` calls; replace with `tokio::fs` equivalents. Verify no blocking calls remain in async context.
- **T5**: Resolve `R-V156P0-M005` — TOCTOU window mitigation. Close the read-check-write window in the workspace session commit path. Use atomic compare-and-swap (CAS) or a transaction guard so that concurrent mutators cannot interleave between version read and version write.
- **T6**: Resolve `R-V156P0-M006` — metrics/tracing at OCC conflict. Add `tracing::warn!` with structured fields (session_id, expected_version, actual_version) at OCC conflict detection sites. Add a metrics counter (`occ_conflict_total`) for operational visibility.
- **T7**: Resolve `R-V156P1-M003` — `force` param wired in registry.refresh handler. Ensure the `force` parameter passed through the host_tool/capability handler path is honored; add a test for the force-refresh code path.
- **T8**: Resolve `R-V156P1-M004` — tracing in capability handler. Add `tracing::info!` / `tracing::debug!` spans to the registry.refresh capability handler covering admission, fetch, and response phases.
- **T9**: Resolve `R-V156P1-M005` — `reqwest::Client` reuse. Extract a shared `LazyLock<reqwest::Client>` with connection pooling (keep-alive, timeout config) instead of constructing a new `Client` per registry.refresh invocation.
- **T10**: Resolve `R-V156P1-L001` — help text HTTPS+public documentation. Update registry.refresh capability help text to explicitly state HTTPS-only and public-internet requirement for CDN URL.
- **T11**: Resolve `R-V156P1-L002` — body-size hammer. Replace the arbitrary 8 MiB body cap with a configurable constant documented in capability-registry.md; add a test for oversized response handling.
- **T12**: Resolve `R-V156P1-L003` — naming prefix. Rename capability-internal symbols to follow the `registry_refresh_*` naming prefix convention (if not already).
- **T13**: Resolve `R-V156P1-L004` — retry jitter. Add randomized jitter (100-500ms) to the registry.refresh retry backoff to avoid thundering herd on CDN.
- **T14**: Resolve `R-V156P1-L005` — latency benchmark. Add a criterion benchmark or integration-level latency measurement for registry.refresh (cold cache + warm cache).
- **T15**: Resolve `R-V156P1-L006` — `generated_at` determinism. Ensure `generated_at` timestamp is set once per refresh invocation (not per retry) so the field is deterministic.
- **T16**: Resolve `R-V156P1-L007` — structured metrics. Add structured metrics counters for registry.refresh: `refresh_total`, `refresh_success`, `refresh_failure`, `refresh_cache_hit`.
- **T17**: Resolve `R-V156-MIDQA-01` — cargo +nightly fmt drift. Run `cargo +nightly fmt --all` and verify clean; add a pre-commit check or CI gate enforcement note.
- **T18**: Resolve `R-V156-PROCESS-01` + `R-V156P1-CACHE-01` — .sqlx/ cache hygiene protocol (both workspace AND test query macros). Document the protocol: run `cargo sqlx prepare --workspace -- --tests` after any migration change (note: `--tests` flag is critical for test query! macros, not just lib queries); commit `.sqlx/` artifacts; add CI check that `.sqlx/` is up-to-date. The two CACHE-01 residuals share the same lesson: future plans touching test queries must run sqlx prepare with the `--tests` flag.
- **T19**: Resolve `R-V157P0-L001` — AC5 plan wording reconciliation. Update V1.57 P0 plan text (or V1.58 P-last spec consolidation) to reconcile the "35 shipped" vs actual "18 shipped / 18 catalog-only" count.
- **T20**: Resolve `R-V157P0-L002` — per-ID failure-path test vectors. Add missing per-ID failure-path test vectors for capability handlers dispatched through `CapabilityRegistry::dispatch()` (scope limited to handlers touched in this plan).
- **T21**: Verify. `cargo test -p nexus-orchestration --test workspace_session` + `cargo test -p nexus-orchestration --test capability_registry` + `cargo clippy --all -- -D warnings` + `cargo +nightly fmt --all -- --check` + `cargo sqlx prepare --workspace -- --tests --check`.

## Risks

- **TOCTOU mitigation may break existing workspace session semantics**: The current read-check-write path may have callers that depend on the non-atomic behavior. Mitigation: add CAS guard as a new code path behind a feature flag initially; run existing test suite to verify no regression; then remove flag.
- **reqwest Client LazyLock extraction changes daemon startup order**: The shared Client must be initialized before any registry.refresh capability is dispatched. Mitigation: use `LazyLock` (initialized on first use); no startup order dependency.
