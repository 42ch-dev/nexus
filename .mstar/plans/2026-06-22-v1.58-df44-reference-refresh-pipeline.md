## Goal

Ship DF-44 (Reference body externalization — refreshable scan pipeline) by introducing the `nexus.reference.refresh` capability, extending the `reference_sources` DB schema with refresh-tracking columns, registering the capability in the acp-capability-set.md §4 roster and capability-registry.md, and adding a daemon-side refresh-scheduler hook. This plan delivers the core refresh pipeline: capability admission, handler, DB migration, roster registration, and a new Draft spec `reference-knowledge.md`. The CLI subcommand (`nexus42 reference refresh`) and cross-cut E2E tests are deferred to P3.

---

## Scope

- **T1**: Design and implement DB migration for reference refresh tracking. Extend `reference_sources` table with `last_refreshed_at TEXT` (ISO-8601 timestamp, nullable), `refresh_policy TEXT NOT NULL DEFAULT 'offline'` (enum: `on_change`, `scheduled`, `offline`), and `refresh_status TEXT` (enum: `fresh`, `stale`, `refreshing`, `error`). Or create a new `reference_refresh_log` table (`id`, `reference_source_id`, `refreshed_at`, `status`, `error_message`, `bytes_fetched`). Add migration file under `crates/nexus-local-db/migrations/`.
- **T2**: Implement `nexus.reference.refresh` capability handler. Register in `capability/builtins/` with a `CapabilityRegistryRow` (all 7 fields). Handler reads the reference source URL, fetches updated content, compares with stored body hash, updates body + metadata if changed. Honor `refresh_policy`: `offline` → skip; `on_change` → fetch + compare; `scheduled` → check `last_refreshed_at` against schedule interval.
- **T3**: If needed, add sibling capability IDs `nexus.reference.refresh_policy.get` (read current refresh policy for a reference source) and `nexus.reference.refresh_status` (read last refresh status). Scope: 1-2 additional IDs if the capability surface requires them.
- **T4**: Extend `acp-capability-set.md` §4 roster from 41 to ~43 rows. Add `nexus.reference.refresh` (+ siblings if T3 is populated). Each new row: capability ID, description, status `shipped` (V1.58 P1), shipped plan, registry row ref. Update the roster governance note if needed.
- **T5**: Extend `capability-registry.md` body: admission contract for `nexus.reference.refresh` (reference source must exist, must have non-offline refresh policy, must have valid URL); per-ID test vectors (success: fetch + update body; failure: offline source, invalid URL, network timeout); handler binding documentation.
- **T6**: Add daemon-side refresh-scheduler hook in `crates/nexus-daemon-runtime/`. A periodic task (configurable interval, default 3600s) that queries `reference_sources WHERE refresh_policy != 'offline' AND (refresh_policy = 'on_change' OR (refresh_policy = 'scheduled' AND last_refreshed_at < now() - schedule_interval))`, then dispatches `nexus.reference.refresh` through `capability::Registry::dispatch()` for each stale source.
- **T7**: Author new Draft spec `reference-knowledge.md` under `.mstar/knowledge/specs/`. Document: refresh policy model (on_change / scheduled / offline), refresh scheduler contract, DB schema for refresh tracking, capability IDs and their admission contracts, integration points with capability::Registry and daemon runtime. Status: Draft (V1.58 P1). Promotion to Master at P-last.
- **T8**: Add per-ID test vectors for new capabilities. ≥ 1 success test + ≥ 1 failure test per new capability ID, dispatched through `CapabilityRegistry::dispatch()`.
- **T9**: Update `catalog_registry_invariant_all_ids_present` cross-validation test to cover the new ~43-row roster.
- **T10**: Codegen if schema changes require it. Run `pnpm run codegen` and commit regenerated output.
- **T11**: Verify. `cargo test -p nexus-local-db --test reference_source` (migration + DAO tests) + `cargo test -p nexus-orchestration --test capability_registry` (new capability handler tests) + `cargo test -p nexus-daemon-runtime --test refresh_scheduler` (daemon hook tests) + `cargo clippy --all -- -D warnings` + `cargo +nightly fmt --all -- --check`.

## Risks

- **DB migration conflicts with existing reference_sources schema**: `reference_sources` may have constraints or indexes that need adjustment for new columns. Mitigation: run migration in a test DB first; verify existing rows survive with DEFAULT values; rollback test.
- **Refresh-scheduler hook may impact daemon startup time**: If the periodic task queries all non-offline sources at boot, this could block daemon readiness. Mitigation: first refresh cycle fires after initial delay (e.g., 60s); boot path is non-blocking.
