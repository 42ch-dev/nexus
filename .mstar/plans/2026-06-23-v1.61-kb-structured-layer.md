## Completion Report

### Design decisions

- **Computable BlockType set**: `Character`, `Item`, `Faction`, `Ability`, `Species` — aligned with wire schemas `entity-attributes.schema.json` and `entity-state.schema.json`. `environment` is not a valid `BlockType` enum variant. Documented in `KeyBlockBody` doc comment (key_block.rs:18) and `block_type_state_key()` doc comment (validation.rs:590).
- **`computable` flag placement**: Inside `body_json` (no DB column) — additive, migration-free. Documented in `KeyBlockBody.computable` doc comment (key_block.rs:40).
- **body_json growth note**: Located at `crates/nexus-local-db/src/kb_store.rs:475` (doc comment in `SqliteKbStore::query()`). Deferred index recommendation: `CREATE INDEX … ON kb_key_blocks(json_extract(body_json, '$.computable'))`.

### Risks for Wave 3 (P3 consumes KB layer)

- P3 (`compute-capability-and-preset`) needs `state` + `computable` fields available on `KeyBlockBody`. These are now present as `Option<serde_json::Value>` and `Option<bool>`. The `block_type_state_key()` helper is public for P3 to consume.
- `SqliteKbStore::query()` applies the `computable` filter in-memory (consistent with all other query filters). If per-world KeyBlock counts grow, the doc comment recommends a deferred SQLite expression index.
