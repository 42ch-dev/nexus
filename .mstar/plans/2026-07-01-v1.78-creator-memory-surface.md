#### 5. `POST /v1/local/memory/review` — trigger review/summarization

- **Request schema file**: `schemas/local-api/memory/review-request.schema.json`
- **Runtime request struct**: `ReviewRequest` (`handlers/memory.rs:471-480`)
- **Fields**:
  - `creator_id: string` — **required** (`String`; `handlers/memory.rs:477-480`)
- **Response schema file**: `schemas/local-api/memory/review-response.schema.json`
- **Runtime response struct**: `ReviewResponse` (`handlers/memory.rs:482-494`)
- **Fields**:
  - `promoted: integer` — **required** (`usize`; `handlers/memory.rs:487-490`)
  - `fragmented: integer` — **required** (`usize`; `handlers/memory.rs:490-492`)
  - `dropped: integer` — **required** (`usize`; `handlers/memory.rs:492-494`)
- **Summarizer trigger semantics (runtime mirror, not redesign)**: POSTing this request processes the active creator's entire pending queue (`handlers/memory.rs:531-587`). Each pending row is classified by `classify_pending_review`; `PromoteToLongTerm` uses `PassthroughSummarizer` and deletes the pending row on success, `FragmentOnly` creates a `memory_fragment` record and deletes the row on success, and `Drop` deletes the row (`handlers/memory.rs:589-688`). The passthrough summarizer prepends an `UNTRUSTED` provenance header, includes `creator_id`/`session_id`/`task_kind`/`world_id`/`captured_at`, and truncates raw digests above 256 KiB (`handlers/memory.rs:704-755`). V1.78 must document this as shipped behavior; it must not introduce a new LLM-backed summarizer or change review rules.
