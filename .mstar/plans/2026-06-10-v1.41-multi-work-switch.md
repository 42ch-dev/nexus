## 1. Problem Statement

V1.39–V1.40 auto-chain completes a novel Work (`mark_work_completed`) but there is no **completion-lock**, **runtime concurrency guard**, or **CLI split** between single-Work actions (`creator run`) and Work management (`creator works`). novels-system Redis + 8-step switch is OUT; OSS needs SQLite + `Works/` mapping per grill-me.

---

## 2. Goals

1. **Extend** `auto_chain::mark_work_completed` (reuse `is_work_completed`): completion-lock + pool row `completed` + clear pool `active`.
2. Write `Works/<work_ref>/.completion-lock.json` on completion (`--no-completion-lock` audited escape).
3. Add `works.runtime_lock_holder` + `runtime_lock_acquired_at` + `completion_locked_at` + `lineage_from_work_id`; holders `cli:<pid>:uuid` / `daemon:schedule:<id>`.
4. Add **`creator works`**: `list`, `status`, `use`, `completion-lock release` — **hard-remove** `creator run list|status`.
5. Add `creator run start --from-work`; `creator run resume --reopen --reason` after lock release.
6. **P0** `novel_pool_entries` + `works use` (upsert, demote prior active → queued); default `work_id` from pool `active`.
7. Daemon: completion-lock + runtime lock on mutate/auto-chain paths.
8. Hermetic tests: completion → lock → release → reopen → resume; runtime lock conflict.

---

## 3. Non-goals

- `creator work switch` / global switch mutex (grill-me OUT).
- novels-system 8-step / 2h cron.
- `inspiration_items` table and `creator works pool *` subcommands (P1) — P0 only `novel_pool_entries` + `works use` + completion row update hook.
- Multi-volume PK (V1.42).
- Platform publish (DF-59).

---

## 4. Acceptance Criteria

1. Completing a novel Work (§6) writes completion-lock and stops auto-chain on that Work only.
2. Other Works may continue auto-chain concurrently.
3. `creator works list|status` replace `creator run list|status` in help and specs.
4. `creator works use` sets pool `active` on `novel_pool_entries` (P0 table).
5. `creator run start --from-work` documents lineage without reviving completed Work chain.
6. Runtime lock rejects concurrent mutating CLI/API on same Work.
7. Hermetic tests cover happy path + lock conflict.
