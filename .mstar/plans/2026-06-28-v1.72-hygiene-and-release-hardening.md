> Architect lock note: P1 is surgical cleanup; the locks below preserve behavior while closing the residuals without creating a new API/schema surface.

- **B1 — Close `R-V171P0-QC1-004` (HIGH handleSave non-atomic)**:
  - **Current behavior**: Strategy `handleSave` chains up to three sequential `mutateAsync` calls (state, transition, prompt template) with no compensating rollback if a later call fails after earlier ones succeeded. Each patch is atomic on the daemon, but the UI-level multi-patch save is not a unified transaction.
  - **Fix (LOCKED)**: **split `handleSave` into per-inspector-section save** — each inspector owns its own save button + per-call error UI (smaller blast radius; matches V1.65 outline editor's per-section save pattern). Do **not** add compensating rollback logic in V1.72.
    - Each inspector (state, edge, prompt template) has its own save button (visible + keyboard-accessible).
    - Saving inspector A while inspector B has unsaved changes does not lose B's draft.
    - A failure in inspector A's save surfaces inline ("Saved label; failed to save transition: …") without rolling back other inspectors' state.
    - The canvas-level `Cmd/Ctrl+S` saves only the active inspector (the one with focus).
    - Existing single-field save still works; existing conflict-modal flow still triggers.
  - Document the chosen approach in `apps/web/src/components/canvas/strategy-canvas.tsx` module doc-comment.
  - **Acceptance**: each inspector has its own save button; failures do not lose other inspectors' drafts; canvas-level keyboard shortcut saves only the active inspector.
- **B2 — Close `R-V171P0-QC1-006` (MEDIUM strategy-canvas.tsx 571-line file split)**:

- **B3 — Close `R-V171-CI-RELEASE-WORKFLOW-INCOMPLETE` (MEDIUM desktop-release.yml signing completion)**:
  - **Current state**: V1.71 B2 shipped `SIGNING.md` + `--sign-identity` flag + CI guard, but the actual release flow is incomplete (no keychain import, no notarize, no staple).
  - **Complete the release signing workflow**:
    - `apps/desktop/SIGNING.md` — update with the actual release flow: how to obtain `APPLE_SIGN_IDENTITY` + `APPLE_SIGN_CERT_P12_BASE64` + Apple ID/password/team-id notarization secrets; how `nexus42 desktop bundle --sign-identity=<id>` activates signing; how CI `desktop-release.yml` runs keychain import → codesign → notarize → staple when secrets present.
    - `.github/workflows/desktop-release.yml` — add:
      - **Keychain import**: create a uniquely named temporary keychain, import `APPLE_SIGN_CERT_P12_BASE64` (base64-decoded into a `.p12` file in the temp keychain), set keychain search list, unlock; restore the original search list and delete the temp keychain in an `always()` cleanup step.
      - **Sign with hardened runtime**: `codesign --force --sign $APPLE_SIGN_IDENTITY --options runtime --timestamp --deep <bundle>`.
      - **Notarize**: `xcrun notarytool submit <dmg> --apple-id $APPLE_ID --password $APPLE_PWD --team-id $APPLE_TEAM_ID --wait`.
      - **Staple**: `xcrun stapler staple <dmg>`.
    - **Hard constraint preserved**: no signing step activates unless **all 5 secrets** are present (`APPLE_SIGN_IDENTITY`, `APPLE_SIGN_CERT_P12_BASE64`, `APPLE_ID`, `APPLE_PWD`, `APPLE_TEAM_ID`). On partial secrets, workflow fails fast before codesign/notarytool and uploads unsigned `.dmg`. CI emits "unsigned build (no signing secrets)" when no secrets are present.
  - **Acceptance**: workflow with no secrets produces unsigned `.dmg` + clear notice; workflow with all secrets produces signed + notarized + stapled `.dmg`; partial secrets fail fast + upload unsigned.
