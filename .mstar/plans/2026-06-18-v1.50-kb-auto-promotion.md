## 1. Problem Statement

V1.49 ¬ß1.2 OUT-B listed O5 (World KB auto-promotion) as deferred. Authors have no way to surface review-time KB candidates for confirmation. V1.40 shipped `kb_extract_jobs` for static reference body extraction (DF-44 partial); review-time extraction is separate.

---

## 2. Goals

1. Extend `kb_extract_jobs` table: `status` (`pending`/`confirmed`/`rejected`), `proposed_payload` JSON, `source_work_id`, `source_chapter_id`, `block_type_guess`, `canonical_name_guess`.
2. Review-time extraction hook in `quality_loop.rs`: when `novel-review-master` runs over a chapter, scan chapter text for candidate entities (simple heuristic: capitalized noun phrases; future iterations may use LLM extraction), insert into `kb_extract_jobs` with `status='pending'`.
3. CLI: `creator world kb pending` lists pending rows; `creator world kb adopt <extract_job_id>` confirms; `creator world kb reject <extract_job_id>` dismisses.
4. Promotion gate: only `works.creator_id` can adopt/reject.
5. Rejected retention: log to `Logs/kb/rejected/`.

---

## 3. Non-goals

- LLM-driven extraction (future iteration; V1.50 uses heuristic).
- Refreshable scan of existing references (T-B P2).
- Write-time extraction (O13).

---

## 4. Acceptance Criteria

1. Migration extends `kb_extract_jobs` with new columns.
2. Review-time extraction hook fires on `novel-review-master` schedule completion; hermetic test verifies insert.
3. CLI commands round-trip; cross-author attempt returns `403`.
4. Adopt inserts a new KeyBlock into `nexus-kb` storage with `status='confirmed'`.
5. Reject writes log entry; row marked `status='rejected'`; archived.
6. Idempotency: re-running review on same chapter does not duplicate pending rows.

### Deviations from plan ¬ß5 (documented)

1. **Column `status` ‚Üí `promotion_status`**: the existing `kb_extract_jobs.status` column carries the V1.29 extraction-queue CHECK (`queued|running|done|failed`); SQLite cannot ALTER a CHECK in place and conflating the two lifecycles in one column is a schema smell. `promotion_status` is a new column with CHECK (`pending|confirmed|rejected`). The entity-scope-model ¬ß5.5.1 state values are preserved.
2. **`source_work_id` not added**: the V1.40 P3 `work_id` column (TEXT) already carries source-work semantics and is reused. Only `source_chapter_id` (INTEGER) is new.
3. **T-A P2 stub not pre-existing**: T-A P2 (`2026-06-18-v1.50-cron-review-staggering`) is still `Todo` at dispatch time. The `quality_loop.rs` module and its `on_schedule_terminal` wiring are created here with a `// COORDINATE-WITH-T-A-P2` code comment; T-A P2 will later wire the per-Work `review` cron role that enqueues `novel-review-master` schedules. The hook is independently testable now (fires for any review-master schedule reaching the supervisor).
