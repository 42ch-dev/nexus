## Evaluation Results

### Plan 40 ‚Äî ADR-009 `as_of` Implementation (Theme C)

- [x] **Schema change delivered**
- **Change**: Added optional `as_of` field (type `["string", "null"]`, format `date-time`) to `ContextAssembleRequestV1` in `schemas/platform/context-assembly-v1.schema.json`.
- **ADR reference**: ADR-009 D1 ‚Äî `ContextAssembleRequestV1` MAY include an optional `as_of` field for limited historical read-only context assembly.
- **Semantics**: When present, timeline events and key blocks are filtered to entities that existed at or before the given timestamp. Vector retrieval on the `as_of` path is optional per ADR-007.
- **Response shape**: Unchanged ‚Äî `ContextAssembleResponseV1` needs no modification (ADR-009 D2 confirms transport isomorphism).
- **Generated output**: `packages/nexus-contracts/src/generated/ContextAssemblyV1.ts` (TS: `as_of?: string | null;`) and `crates/nexus-contracts/src/generated/context_assembly_v1.rs` (Rust: `pub as_of: Option<String>`).
- **Rust fix**: Updated CLI construction site in `crates/nexus42/src/commands/context.rs` to include `as_of: None`.
- **Commit**: `1780f28 docs(schema): add as_of optional field to ContextAssembleRequestV1 (ADR-009)`
- **Verification**: `pnpm run validate-schemas` ‚úì (54/54 valid), `pnpm run codegen` ‚úì, `pnpm run typecheck` ‚úì, `cargo check --workspace` ‚úì.

### Plan 38 ‚Äî Auth & Identity Plane (Theme A)

- [x] **No nexus schema change needed**
- **Rationale**: Plan 38 introduces self-hosted Logto OIDC, platform JWTs, token exchange routes, and Bearer resolution logic ‚Äî all entirely platform-side TypeScript. Nexus contracts are auth-agnostic (per compass and `schemas-boundary.md` ¬ß2.2: "Local API endpoints on `nexus42d`" and "Platform HTTP endpoints" ‚Äî auth routes are platform BFF, not nexus wire contracts). No new auth-related wire types need nexus JSON Schema.
- **Token exchange shape**: Defined as platform-internal type `{ grant_type, subject_token }` ‚Üí `{ access_token, token_type, expires_in }`. This is not a nexus contract type; platform defines and consumes it internally.
- **Cross-repo impact**: None. Platform team may import existing `@42ch/nexus-contracts` types for `CreatorId`, `UserId`, etc., but no new types are required.

### Plan 39 ‚Äî OpenAPI Export (Theme B)

- [x] **No nexus schema change needed**
- **Rationale**: Plan 39 generates an OpenAPI 3.1 specification document (`docs/openapi/v1.yaml`) by introspecting existing platform routes and referencing existing `@42ch/nexus-contracts` wire types. It documents existing shapes ‚Äî it does not create new ones. All referenced types (`SyncConflictResponseV1`, `PublishStoryRequest`, etc.) already exist in nexus schemas.
- **New route from Plan 38**: `auth/token/exchange` will be documented in the OpenAPI spec, but its request/response shapes are platform-internal (see Plan 38 evaluation above).
- **Cross-repo impact**: None. OpenAPI spec is a platform-side artifact.

### Plan 41 ‚Äî CLI + Platform e2e Integration (Theme D)

- [x] **No nexus schema change needed**
- **Rationale**: Plan 41 creates a test harness (`tests/e2e/`) in the platform repo that exercises existing CLI-to-platform flows. It tests existing wire contracts ‚Äî no new schemas are introduced. The harness tests `as_of` in context assembly (Task 4), which depends on the Plan 40 schema change already delivered above.
- **Regression stability**: No CLI commands are changed; existing command surface remains stable. The `as_of` field is optional with `default: null`, so existing CLI behavior is preserved.
- **Cross-repo impact**: None for nexus schemas. Plan 41 is entirely platform-side test code.

### Plan 42 ‚Äî Quota Enforcement (Theme E)

- [x] **No nexus schema change needed**
- **Rationale**: Plan 42 extends existing `QuotaService` and `EntitlementService` with atomic check-and-consume, rollback, and error codes (`QUOTA_EXCEEDED`, `QUOTA_PERIOD_INVALID`, `QUOTA_CONSUME_FAILED`). These error codes are platform-internal error types, not nexus wire contracts. The existing nexus schemas already cover the wire shapes:
  - `schemas/platform/me-entitlements-response.schema.json` ‚Äî `MeEntitlementsResponseV1` (wire)
  - `schemas/platform/official-creator-quota-response.schema.json` ‚Äî `OfficialCreatorQuotaResponseV1` (wire)
- **Error shape**: `QuotaExceededError` with `quota_period_end`, `official_runs_limit`, `official_runs_consumed` is a platform-internal error class, not a wire contract. If the platform team later decides to standardize quota error responses as a formal wire type, that would be a separate nexus schema change request.
- **Cross-repo impact**: None. Platform reads existing quota wire types; adds enforcement logic internally.

### Plan 43 ‚Äî Skills Export L0 (Theme F)

- [x] **No nexus schema change needed**
- **Rationale**: Plan 43 defines a static capability descriptor (`SkillsExportDescriptor`) that maps `nexus.*` capability IDs to V1.3 core capabilities. The descriptor types are defined as platform-internal TypeScript types in `packages/domain/src/types/skills-export.ts`. Per `schemas-boundary.md` ¬ß2.1, a schema belongs in `schemas/` only if nexus-platform observes it AND platform TS code needs it. While the descriptor is served via a platform API endpoint, the current consumer is external (ACP-incapable agent hosts), not the nexus CLI/daemon. No CLI code currently consumes this descriptor.
- **Deferral note**: If the descriptor shape stabilizes and the nexus CLI or daemon needs to consume it (e.g., for capability discovery), it can be added to nexus schemas in a future cycle. The plan itself notes `@42ch/nexus-contracts` usage as optional.
- **Cross-repo impact**: None for nexus schemas.
