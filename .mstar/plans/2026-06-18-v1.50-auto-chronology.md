## 1. Problem Statement

DF-62 (multi-volume PK + seed) shipped V1.42 but volume advancement is manual. Authors want auto-advance when current volume completes.

---

## 2. Goals

1. `works.auto_chronology BOOLEAN` column (default `false`).
2. CLI `creator works chronology set/show/advance`.
3. Daemon `auto_chronology_tick` task (5-min interval; env override).
4. Finish detection (all chapters finalized + intake complete).
5. Atomic outline creation + chapter seed + log entry.
6. Manual override CLI `creator works chronology advance --volume N+1`.

---

## 3. Non-goals

- Multi-volume auto-publication (V1.51+).
- World KB auto-promotion from new volume (T-B P1 separate concern).

---

## 4. Acceptance Criteria

1. Migration adds column with default `false`.
2. Daemon tick hermetic test: positive (all finalized + intake complete → advance), 4 negative edge cases (incomplete chapter / intake not complete / runtime lock / already-advanced).
3. Atomic outline creation + chapter seed; crash mid-advance rolls back cleanly.
4. Manual `creator works chronology advance --volume 3` works regardless of `auto_chronology` flag.
5. Log entry written to `Logs/chronology/<YYYY-MM-DD>-advance-vol<N+1>.md`.

### Issues/Risks

**Scope decisions (documented, spec-faithful within Draft V1.50 latitude):**

1. **"Last planned volume" edge (spec §3.1)** is subsumed by the
   completion-lock gate: there is no `total_planned_volumes` column in this
   migration, so a Work with no further volumes is expected to be
   completion-locked by the author. The `CompletionLocked` skip reason serves
   both the spec §3 step-4 gate and the §3.1 terminal edge. → Residual
   R-V150P3AUTOCHRONO-01.

2. **Auto path seeds zero chapters** (spec §4.2 last paragraph: placeholder
   outline → seed zero). The daemon advance creates the outline + log; the
   author fills the outline and runs the seed step. The manual `advance` CLI
   accepts `--chapters N` to seed immediately. A future enhancement could
   auto-seed from the prior volume's cadence. → Residual
   R-V150P3AUTOCHRONO-02.

3. **File-location reconciliation**: the plan §5 listed
   `nexus-orchestration/src/schedule/supervisor.rs` for the tick; the
   assignment §4 named `nexus-daemon-runtime/src/auto_chronology.rs`. Both
   files were created, following the established `cron_supervisor` precedent:
   the reusable advance logic lives in `nexus-orchestration::auto_chronology`
   (matching the AC §8 test path `nexus-orchestration --test auto_chronology_tick`),
   and the daemon-runtime file is a thin spawn wrapper.
