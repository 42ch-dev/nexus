## 1. Problem Statement

DF-62 (multi-volume PK + seed) shipped V1.42 but volume advancement is manual. Authors want auto-advance when current volume completes.

---

## 2. Goals

1. `works.auto_chronology BOOLEAN` column (default `false`).
2. CLI `creator works chronology set/show/advance`.
3. Daemon `auto_chronology_tick` task (5-min interval; env override).
4. Finish detection (all chapters finalized + intake complete).
5. Atomic outline creation + chapter seed + log entry.
6. Manual override CLI `creator works chronology advance --volume N+1`.

---

## 3. Non-goals

- Multi-volume auto-publication (V1.51+).
- World KB auto-promotion from new volume (T-B P1 separate concern).

---

## 4. Acceptance Criteria

1. Migration adds column with default `false`.
2. Daemon tick hermetic test: positive (all finalized + intake complete → advance), 4 negative edge cases (incomplete chapter / intake not complete / runtime lock / already-advanced).
3. Atomic outline creation + chapter seed; crash mid-advance rolls back cleanly.
4. Manual `creator works chronology advance --volume 3` works regardless of `auto_chronology` flag.
5. Log entry written to `Logs/chronology/<YYYY-MM-DD>-advance-vol<N+1>.md`.

### Design note — R-V150P3AUTOCHRONO-04 test-description interpretation

The fix-wave assignment's test description for R-04 ("mock the outline
write to fail; assert tx rolled back AND no partial outline file exists")
is internally inconsistent with its own Fix paragraph ("open the DB tx
FIRST; on successful commit, write the outline file … This matches the
V1.36 atomicity pattern (DB first, FS second)"). Under "DB first, FS
second", the tx commits BEFORE the outline write, so an outline failure
cannot roll back the already-committed tx.

The implemented regression test `perform_advance_writes_outline_after_tx_commit`
asserts the correct Design-A invariant: when the outline write fails
post-commit, (a) the failure surfaces as `Err` (observable), (b) the DB
tx is **committed** (chapters persist — proving the ordering inversion),
and (c) no partial outline file exists (atomic-write cleanup). This is
the behaviour that actually fixes qc3 W-1: the Work is **not stuck**
because the DB is the source of truth and the outline is recoverable
(next tick for the auto path; `--force` for the manual path).
