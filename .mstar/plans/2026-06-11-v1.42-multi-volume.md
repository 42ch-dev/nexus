## 1. Problem Statement

V1.38–V1.41 shipped single-volume multi-chapter. `work_chapters` PK is `(work_id, chapter)` — insufficient for multi-volume serial works. DF-62 multi-volume PK was deferred to V1.42 (grill-me: **implicit vol=1** backfill).

---

## 2. Goals

1. **DDL migration**: PK `(work_id, volume, chapter)`; backfill existing rows `volume = 1`.
2. **Init**: `novel-project-init` grill-me captures volume count + per-volume chapter totals; seed `work_chapters` rows.
3. **Volume outline**: scaffold `Outlines/volume-{n}-outline.md` per spec §4.5.5.
4. **Auto-chain**: `next_chapter(work_id)` volume-aware; continue across volume boundary when vol N finalized and vol N+1 exists.
5. **Status data plane**: API/DB fields for per-volume progress (`current_chapter` semantics per volume or aggregate per spec §8.1 overlay).
6. Hermetic tests: single-volume regression (V1.41 behavior), two-volume happy path.

---

## 3. Non-goals

- CLI table layout / copy polish → **P-last**
- DF-56 conditional routing → **P2**
- Platform publish (DF-59)

---

## 4. Acceptance Criteria

1. Existing Works behave as single-volume (`volume=1`) without user action.
2. New multi-volume Work: init → chapter 1 vol 1 → finalize → auto-chain picks vol 1 ch 2 or vol 2 ch 1 per rules.
3. `creator works status` returns volume-aware chapter rows (data); CLI formatting may be refined in P-last.
4. Spec §4.5.4–§4.5.7 implement markers updated.
