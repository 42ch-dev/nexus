## Goal

Convert 5 DF-46 catalog-only `orchestration`-scope capability IDs to fully shipped runtime handlers in the `CapabilityRegistry`. Capabilities transition from `catalog-only` (logical contract only, no runtime binding) to `shipped` with handler implementations, admission gates, and per-ID test vectors. Roster goes 27 ‚Üí 32 shipped / 4 sync catalog-only (platform-blocked) / 2 publish OUT.

**Hard fallback (compass Q4)**: If T1 world-delta spec work reveals scope > L effort, PM trims Track A to 3 capabilities (`world.state.query` + `timeline.event.append` + `fork.create`); defers `world.delta.propose` + `world.delta.apply` to V1.61. Roster goes 27 ‚Üí 30 (not 32). PM decision recorded in plan completion report.

The 5 capabilities:
- `nexus.world.state.query` ‚Äî query KB/timeline slices filtered by world_id + creator context
- `nexus.world.delta.propose` ‚Äî produce structured proposed delta package from input changeset
- `nexus.world.delta.apply` ‚Äî apply staged deltas locally under policy (atomic write)
- `nexus.timeline.event.append` ‚Äî append event to narrative timeline (immutable; no canon rewrite)
- `nexus.fork.create` ‚Äî explicit branch creation when rewrite-past is intended (local fork; distinct from PD-01 platform community fork)

## Risks

- **World delta spec depth (compass Q4 primary risk)**: `world.delta.propose`/`apply` have only one-line roster descriptions; acp ¬ß8 lists `world.delta.apply` agent-vs-runtime as an Open Item. T1 Draft overlay answers this, but may reveal scope > L. **Mitigation**: T1 is the hard fallback gate; if scope exceeds L, PM triggers Q4 fallback (ship 3 of 5; defer world deltas to V1.61).
- **Orchestration handler complexity vs host_tool**: V1.59 shipped 9 host_tool-scope capabilities (simple daemon binding). These 5 are orchestration-scope ‚Äî they need engine-level handlers with DB/FS state mutation, admission gates, and atomicity. Effort calibrated at L (not M-L) based on `nexus.reference.refresh` V1.58 P1 precedent.
- **DB schema assumptions**: Plan assumes world/timeline/fork operations use existing `narrative_worlds`, `timeline_events`, `kb_key_blocks` tables (no new migrations). If T1 spec reveals schema gaps, a migration becomes in-scope (effort increases).
- **PD-01 boundary confusion for `fork.create`**: PD-01 rules "World fork is platform-only" ‚Äî but that's about community/social fork. `nexus.fork.create` is local timeline branching. Mitigation: handler doc-comment explicitly cites PD-01 boundary; T1 spec overlay clarifies.
- **Auto-derive regression risk**: Replacing manual 28-element match list in `catalog_registry_invariant_all_ids_present` with iterator-based derivation could miss an edge case. Mitigation: keep the test comprehensive; verify all 32 (or 30) shipped IDs are covered by the iterator walk.
