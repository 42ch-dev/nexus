## 1. Problem Statement

V1.36 shipped a useful `llm_judge` 五问 finalize gate, but it did not ship a durable quality-loop backplane. Reference-system patterns include findings lifecycle, role separation, per-work rules, append-only rules history, process logs, and master-decision escalation. Those are too large for the V1.37 first slice, but they should be planned coherently so later work does not bolt ad hoc review state onto the novel pipeline.

---

## 2. Goals

1. Convert DF-64/65/66/67 into a staged Nexus-native roadmap.
2. Define a local-first replacement for reference-system Redis findings: likely a `findings` table in `state.db`.
3. Define how review output would route to brainstorm/write/none without requiring a full multi-agent cron system immediately.
4. Define three-layer rules architecture in Nexus terms without duplicating SOUL or World KB responsibilities.
5. Define `Works/<work_ref>/Logs/` structure and sync exclusion rules if logs become first-class.
6. Keep implementation deferred unless P0/P1 completion leaves room and a narrower slice is explicitly approved.

---

## 3. Non-goals

- Implementing the full quality-loop in this docs prepare wave.
- Replacing the V1.36 `llm_judge` finalize gate.
- Redis, cron, or platform worker dependencies.
- Platform publish.
- Automatic new-novel switch.

## 4.1 Delivery Notes (2026-06-08)

**Scope decision**: Roadmap-only for V1.37 P3. No code, schema migration, embedded preset, prompt template, daemon scheduled task, CLI subcommand, rules file writer, or platform publish behavior is implemented in this plan.

**Spec amendment**: [novel-writing/workflow-profile.md §5.5](../knowledge/specs/novel-writing/workflow-profile.md#55-quality-loop-roadmap-v137-p3-extension) now defines:

1. `findings` lifecycle `open → resolved | wont_fix`, severity enum `info | minor | major | blocker`, and a future local DB schema sketch.
2. Reference executor mapping: `write` → existing `novel-writing`; `brainstorm` → future `novel-brainstorm`; `none` → no preset/manual decision; `master` → future `novel-review-master`.
3. 96h master-decision timeout as local DB query + `nexus-daemon-runtime` scheduled lifecycle task + `creator run status` banner; escalation remains user opt-in.
4. Three-layer rules architecture: shared craft rules, per-Work `Rules/novel-rules.md`, append-only `Rules/novel-rules-history.md`.
5. `Logs/brainstorm`, `Logs/write`, `Logs/review`, and future `Logs/publish`; `Logs/**` remains excluded from chapter sync.
6. Future `reflection-loop` integration to generate findings and rule suggestions.

**Deferred tracker**: DF-64, DF-65, DF-66, and DF-67 targets and deferral histories now point to V1.37 P3 roadmap-only, with implementation deferred to V1.37+ when a future plan reopens.

**Open design question for PM / future plan**: `novel-review-master` should remain opt-in by default per §5.5.3. A future implementation plan should decide whether a Work-level setting can enable automatic scheduling after the 96h banner, and what audit entry is required for that opt-in.
