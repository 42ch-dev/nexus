# P0 — Memory Review Reliability (REL-01 bounded / serialized / timeout + client drain)

**Iteration**: V1.80 · **Plan ID**: `2026-07-01-v1.80-memory-review-reliability`
**Track**: A · **Owner**: `@fullstack-dev` · **Status**: Todo
**Working branch**: `feature/v1.80-memory-review-reliability` (from `iteration/v1.80`)
**Closes residual**: `R-V178P0-QC3-003` (REL-01)

## Problem statement

`POST /v1/local/memory/review` (`crates/nexus-daemon-runtime/src/api/handlers/memory.rs`) currently fetches the **entire** pending-review queue for a creator and processes it **synchronously** in a single request handler loop (`process_review_queue`), with:

- **No bound** — `fetch_pending_reviews_by_creator` returns every pending row; a large queue blocks the request indefinitely.
- **No timeout** — the HTTP request hangs until the whole loop completes.
- **No per-creator in-flight serialization** — two concurrent `POST /memory/review` calls for the same creator can fetch the same pending rows and double-promote / double-fragment them (the `create_fragment` path mints a fresh `fragment_id`, so duplicates are not deduped at the DB).
- **No client drain signal** — the web `reviewMemory` mutation assumes a single call drains the queue; there is no `has_more` signal to indicate the client should re-request.

V1.78 QC3 (W-QC3-004) accepted this as low residual `R-V178P0-QC3-003` under the local-only / single-active-creator / small-queue threat model, with a named roadmap: chunked processing, per-creator in-flight serialization, and client uncertain-completion handling. V1.80 pays it down.

## specify

- **User value**: The memory-review pipeline returns within bounded time (~5s server budget per call) even on a large queue, never double-processes rows under concurrent calls, and lets the client drain a large queue via repeated `has_more` calls — so the author's desktop shell does not freeze, the SOUL internalization loop (V1.78) is reliable rather than fragile, and the maintainer has a clear reliability contract for the review endpoint.
- **Scope**: `crates/nexus-daemon-runtime/src/api/handlers/memory.rs` (`review` + `process_review_queue` + the existing handler-local bounded projection helper), `crates/nexus-daemon-runtime/src/workspace/mod.rs` (per-creator in-flight guard state), `schemas/local-api/memory/review-response.schema.json`, generated contract output, and the `apps/web` memory client mutation (`useReviewMemory` in `apps/web/src/api/queries.ts`). No `crates/nexus-local-db` change is required for the selected bounded-fetch path unless implementation discovers the handler-local helper cannot be reused.
- **Non-goals**: async background-job infrastructure (job table / status polling / cancellation tokens); changing the promote/fragment/drop classification logic; touching the reading/soul/memory-page components (P1 owns those).
- **DoD**: `R-V178P0-QC3-003` closed in `status.json` (`lifecycle: resolved`); bounded fetch (N rows per call) + per-creator serialization guard + request timeout + client drain loop with max-iterations cap shipped; the author no longer experiences indefinite hangs on large-queue review; QC 3/3 Approve; QA Pass; wire additive `@42ch/nexus-contracts` 0.14.0 → 0.15.0; concurrency regression test asserts no double-promotion; bounded `has_more` drain walk test green; existing classify-correctness regression tests green.

## clarify

Locked by architecture Review & Edit (2026-07-01):

- **Q1 — Bound size**: `REVIEW_BATCH_LIMIT = 50` rows per daemon call. Evidence: the existing memory list pagination defaults to `DEFAULT_QUERY_LIMIT = 50` in `crates/nexus-daemon-runtime/src/api/handlers/memory.rs`, and the V1.78 bounded helper already uses `limit + 1` over-fetch for `has_more`. A 50-row synchronous batch is the smallest policy that aligns review-drain behavior with the shipped list-page default while preserving the local-only / small-queue threat model. Keep it as a named const near the review handler with a short rationale; do not make it user-configurable in this slice.
- **Q2 — In-flight guard mechanism**: use an in-process per-creator mutex map on `WorkspaceState`, not a DB/advisory lock. Add `memory_review_locks: Arc<std::sync::Mutex<HashMap<String, Arc<tokio::sync::Mutex<()>>>>>` (or equivalent aliases) to `crates/nexus-daemon-runtime/src/workspace/mod.rs`, initialize it in both `new_for_testing` and `initialize`, and expose a small accessor that clones/inserts the per-creator `Arc<tokio::sync::Mutex<()>>` while holding the map mutex only briefly. Rationale: the daemon is local-only and single-process for this endpoint; DB locks would add complexity and persistence semantics that REL-01 explicitly does not need.
- **Q3 — Concurrency-overlap response**: serialize by waiting on the per-creator mutex. Do not add a new 409/429 wire/error branch. Evidence: `apps/web/src/pages/memory-page.tsx` already disables the CTA while `reviewMemory.isPending`, so overlapping calls are primarily stale-client or double-click edge cases; waiting gives a simple local UX and avoids adding a second error path to the Local API.
- **Q4 — Timeout threshold + overrun behavior**: use `REVIEW_CALL_TIMEOUT = 5s` as a named daemon const and implement timeout as a deadline-aware bounded loop, not as a coarse cancellation of the entire request future. Before each row/action, check the deadline (and use `tokio::time::timeout_at` around row-level async work where needed); if the deadline is reached, return the partial `ReviewResponse` with counters accumulated so far, `processed` set to rows inspected so far, and `has_more = true`. Do not return 503 for ordinary budget exhaustion. Use `NexusApiError` only for real validation/database failures outside the partial-progress path.
- **Q5 — Client drain cap**: `useReviewMemory` should drain while `has_more === true` for at most `REVIEW_DRAIN_MAX_CALLS = 20` client calls (50 × 20 = 1,000 rows per user action), then stop with a non-error informational toast/state that says review is still draining and the user can run it again. Also break if a call returns `processed === 0` while `has_more === true` to avoid an infinite tight loop on an unprocessable head row; surface the same still-draining message.

- **Q6 — Spec amendment needed?**: The REL-01 behavioral changes (bounded fetch, per-creator serialization, request timeout) refine an endpoint that was never normatively specified — the `web-ui.md` §24 V1.78 product contract describes the existing synchronous review behavior, and `daemon-runtime.md` does not document per-endpoint contracts. **No normative spec amendment is required.** The DTO change (`has_more` + `processed`) is additive and tracked in the wire contracts note. If the architect determines the serialization guard or timeout contract should become normative (future consumers depending on it), record it as a Draft overlay on `web-ui.md` §24 in a follow-up hygiene plan — not in this plan.

## plan

1. **Bounded fetch / DAO approach** — reuse the existing handler-local V1.78 bounded helper `fetch_pending_reviews_page` in `crates/nexus-daemon-runtime/src/api/handlers/memory.rs` with `cursor = None` and `fetch_limit = REVIEW_BATCH_LIMIT + 1`. This helper already uses compile-time checked `sqlx::query!` projections, `ORDER BY created_at DESC, pending_id DESC`, and `LIMIT ?`; it returns `PendingReviewInfo`, which is exactly what `process_review_queue` consumes. Do **not** add a new `crates/nexus-local-db` query for this slice unless reuse proves impossible; `crates/nexus-local-db/src/pending_review.rs` currently only has unbounded `list_pending_reviews`, so moving the helper there would add churn without improving the REL-01 fix. The over-fetched row drives `has_more`; truncate the processing slice back to 50.
2. **Processing order and `has_more` semantics** — keep the existing deterministic newest-first order (`created_at DESC, pending_id DESC`) from the V1.78 pagination helper. Compute `has_more = overfetch_row_exists || stopped_due_to_timeout || processed_less_than_fetched_batch_due_to_deadline`. Per-row classify/promote/fragment/drop semantics stay unchanged; successful promote/fragment/drop still delete the pending row after the side effect succeeds.
3. **In-flight guard** — add per-creator serialized entry to `WorkspaceState` as described in clarify Q2. Handler flow: validate active creator and creator_id first; clone the per-creator lock from state; acquire it; then fetch/process the bounded batch. The guard must be held across fetch, classification, side effects, and `delete_pending_by_id` calls so two overlapping requests cannot fetch/delete the same pending rows. Do not hold the map mutex across `.await`.
4. **Guard release ordering** — although the selected `tokio::sync::Mutex` guard releases synchronously on `Drop` (unlike `RuntimeLockGuard`, which requires async `release().await`), implement the handler with an explicit inner scope so the per-creator guard is dropped only after all row side effects and best-effort deletes for the batch have completed. Avoid spawning detached work while holding the guard. Audit every early return after acquisition; real errors should be converted inside the guarded block and returned only after the guard scope exits. This mirrors the release-order discipline documented in `crates/nexus-daemon-runtime/AGENTS.md` without introducing a custom async-release guard.
5. **Timeout / partial progress** — implement the 5s timeout as a deadline inside `process_review_queue` (or a new `process_review_batch`) rather than wrapping the entire handler in `tokio::time::timeout`, because a coarse timeout would discard local counters and make partial-progress response impossible. The loop checks the deadline before each row and uses row-level `timeout_at` where an async side effect may overrun. On budget exhaustion, return the accumulated partial `ReviewResponse` with `has_more = true`; do not roll back completed side effects/deletes.
6. **Response extension / wire contract** — current schema `schemas/local-api/memory/review-response.schema.json` has required `promoted`, `fragmented`, `dropped` only and `additionalProperties: false`. Add optional properties `has_more` (`boolean`, documented/defaulted by consumers to `false` when absent) and `processed` (`integer`, documented/defaulted by consumers to `0` when absent); do **not** add them to `required` so this is backward-compatible additive for older daemons/clients. The V1.80 handler should still always populate concrete values. Then run `pnpm run codegen`, commit generated Rust/TypeScript output, bump `packages/nexus-contracts/package.json` from `0.14.0` to `0.15.0`, wire the handler to populate both fields, and extend `crates/nexus-daemon-runtime/tests/memory_dto_roundtrip.rs` plus memory-review API tests.
7. **sqlx compliance** — any new or changed static SQL must use compile-time checked macros (`sqlx::query!`, `sqlx::query_as!`, or `sqlx::query_scalar!`) and must update `.sqlx/` via `cargo sqlx prepare --workspace --all -- --all-targets` if query metadata changes. The preferred reuse path in T1 should avoid new SQL entirely; if a distinct helper is added, it must follow the same macro pattern as `fetch_pending_reviews_page`.
8. **Client drain** — update `useReviewMemory` in `apps/web/src/api/queries.ts` (not `apps/web/src/pages/memory-page.tsx`) so its mutation function repeatedly calls `client.reviewMemory({ creator_id })` until `has_more !== true`, `REVIEW_DRAIN_MAX_CALLS` is reached, or `processed === 0` with `has_more === true`. Aggregate counters across calls for the success toast; if the cap/zero-progress guard trips, show a non-error "still draining" message. `apps/web/src/lib/nexus/browser-client.ts` remains the thin transport call unless typegen requires import updates.
9. **Tests** — add/extend tests for: bounded `has_more` drain walk (>50 seeded rows); two overlapping HTTP calls for the same creator produce no duplicate promoted/fragments and no double-delete symptoms; deadline/partial-progress path returns accumulated counters with `has_more=true`; DTO round-trip includes the optional fields while old minimal JSON still deserializes; web `useReviewMemory` aggregates multi-call counters and stops at the cap/zero-progress guard.
