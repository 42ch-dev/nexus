# V1.9 WS-A — Third-Party Preset Loading + CLI

## Summary

Extend Nexus preset loading from "embedded + `_system/` only" to include user-installed third-party presets from `~/.nexus42/presets/<name>/`. Add `nexus42 preset init` and `nexus42 preset list` CLI subcommands. Users can now scaffold, install, and enumerate custom orchestration strategies alongside the built-in `novel-writing` preset.

## Background

- **Deferred feature**: DF-17 ("Preset ecosystem: third-party presets") tracked in `.mstar/knowledge/deferred-features-cross-version-tracker.md`.
- **Current state**: `load_preset()` in `crates/nexus-orchestration/src/preset/loader.rs` (line 177-189) is a stub returning `"not yet implemented"`. Only embedded presets (`include_dir!`) and `_system/` directory presets (via `system_preset_dir.rs`) work.
- **Path layout**: ADR-014 uses `~/.nexus42/`. Existing `system_preset_dir.rs` shows the pattern: scan `<nexus_home>/presets/_system/<name>/` for `preset.yaml`. WS-A extends this to `~/.nexus42/presets/<name>/` (no `_` prefix = user preset).
- **Scope boundary**: This WS does NOT include preset signing, registry, dependency resolution, or hot-reload. Those are deferred to future milestones.

## Design Decisions

1. **New module `user_preset_dir.rs`** in `crates/nexus-orchestration/src/` — analogous to `system_preset_dir.rs`. Scans `~/.nexus42/presets/` for directories NOT starting with `_` (system) or `.` (hidden). Each valid directory with `preset.yaml` becomes a user preset.
2. **Implement `load_preset()` from directory** — replace the stub in `loader.rs` (lines 177-189). Reads `preset.yaml` from the bundle root, parses via the existing `load_preset_from_str()`, validates per §7.6 rules.
3. **Composable search order**: user presets override system presets override embedded presets. If a user preset has the same `id` as an embedded preset, the user version wins (last in chain).
4. **CLI `preset init`** scaffolds `~/.nexus42/presets/<name>/` with `preset.yaml` template + empty `prompts/` directory. Template is a minimal valid preset YAML.
5. **CLI `preset list`** shows all available presets grouped by source: `[embedded]`, `[system]`, `[user]`. Indicates which source wins for duplicate IDs.
6. **`nexus-home-layout`** may need a helper for the user preset base path (analogous to `system_preset_base_dir`). Add `user_preset_base_dir()`.

## Risks

| ID | Risk | Mitigation |
|----|------|------------|
| RISK-WSA-01 | `_` prefix naming collision between user and system presets | Convention: `_` prefix reserved for system; enforced at scan time (`user_preset_dir.rs` skips `_`-prefixed dirs) |
| RISK-WSA-02 | `preset.yaml` in user dirs may reference capabilities not in registry | Existing `validate_manifest()` already rejects unknown capabilities; this is handled at load time |
| RISK-WSA-03 | Duplicate preset IDs across sources cause confusion | `preset list` shows which source wins; override semantics documented |
