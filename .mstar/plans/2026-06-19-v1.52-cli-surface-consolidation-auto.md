## 1. Problem Statement

V1.50 T-B P1 introduced the canonical `creator world kb` CLI surface (list/show/edit/delete + pending/adopt/reject). The legacy `creator kb --scope world` surface predates this and still lives in `crates/nexus42/src/commands/creator/kb.rs`, implementing World KB operations inline via direct `SqliteKbStore` access rather than forwarding to the canonical `world::kb` module.

**R-V150KBED-01** (low, accept — raised at V1.50 kb-editor-cli QC) records the CLI coexistence:
> legacy `creator kb --scope world` coexists with the new `creator world kb` surface.

Having two surfaces for the same World KB operations creates:
1. **Author confusion**: which surface is the "right" one?
2. **Code duplication**: World KB operations implemented in two places (inline in `kb.rs` and hermetic functions in `world/kb.rs`)
3. **Discoverability regression**: `creator kb --help` shows `--scope world` but doesn't point to the canonical surface
4. **Divergence risk**: the inline implementations may diverge from the canonical ones over time

V1.52 T-A P0 added `creator world kb adopt --auto <world_ref>` to the canonical surface. The legacy surface has no auto-adopt path.

---

## 2. Goals

1. Make `creator kb --scope world <subcmd>` a transparent alias for `creator world kb <subcmd>` where the operations map 1:1.
2. Emit a one-time deprecation warning on each invocation of the legacy path, pointing to the canonical surface.
3. Ensure `creator world kb` remains the canonical surface for all World KB operations.
4. Verify the `--auto` flag on `creator world kb adopt` is documented in `--help` output (T-A P0 deliverable).
5. Author the Draft V1.52 overlay body in `cli-spec.md` §6.x documenting the consolidation.

---

## 3. Non-goals

- **Do NOT** remove the legacy `creator kb --scope world` path (planned removal V1.53).
- **Do NOT** rename commands.
- **Do NOT** add new subcommands to either surface.
- **Do NOT** modify the `--auto` flag implementation (T-A P0 scope).
- **Do NOT** change the `creator kb --scope work` operations (local work-scope file index — unaffected).

---

## 4. Acceptance Criteria

1. `creator kb --scope world list --world-id <id>` emits a deprecation warning on stderr and produces the same output as `creator world kb list <id>`.
2. `creator kb --scope world show <entry_id> --scope world --world-id <id>` emits a deprecation warning and produces the same output as `creator world kb show <id> <entry_id>`.
3. `creator kb --scope world search <query> --scope world --world-id <id>` emits a deprecation warning and produces the same output as before (search has no canonical equivalent; keep inline path with warning).
4. `creator kb --scope world add --file <path> --scope world --world-id <id>` emits a deprecation warning and produces the same output as before (add has no canonical equivalent; keep inline path with warning).
5. `creator kb --scope world remove <entry_id> --scope world --world-id <id>` emits a deprecation warning and produces a result equivalent to `creator world kb delete <id> <entry_id> --yes`.
6. `creator world kb adopt --help` documents the `--auto` flag (T-A P0 deliverable; verify present).
7. The deprecation warning is emitted via `tracing::warn!` + `eprintln!` on each invocation, with the message: "`creator kb --scope world` is deprecated; use `creator world kb <subcmd>` (planned removal V1.53)."
8. Spec overlay body in `cli-spec.md` §6.x (Draft V1.52) matches the implementation behavior.
