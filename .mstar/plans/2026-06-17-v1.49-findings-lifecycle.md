## 1. Problem Statement

Findings remain three-state (`open` / `resolved` / `wont_fix`) per V1.37 intentional narrow scope. V1.48 deferred F6 extended lifecycle. Authors cannot triage or mark in-review before resolution.

---

## 2. Goals

1. Add statuses: `triaged`, `in_review`, `duplicate` with validated transitions.
2. DB migration + backward-compatible read for existing rows.
3. Define **actionable** statuses for `open_findings_block` consumer (default: `open`, `triaged`).
4. Hermetic API + DAO tests for transitions.

---

## 3. Non-goals

- Narrative index runtime (P1).
- Author desk UX (P2).
- New `schemas/` wire types.

---

## 4. Acceptance Criteria

1. Overlay §1 lifecycle diagram implemented in DAO validation.
2. PATCH/CLI can move finding through triage → in_review → terminal states.
3. `list_open_findings_for_chapter` (or successor) matches overlay actionable set.
4. P1 consumer tests still pass with updated filter semantics documented.
