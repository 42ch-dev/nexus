## 1. Scope

Refactor `host_tool_executor.rs` (4298 lines) into a 3-caller-entry-point adapter (~800 lines) that dispatches all `nexus.*` tool calls through the single `capability::Registry::dispatch` path. Add the `host-call` CLI debugging subcommand. Inject `CdnConfig` via constructor (closes `R-V156P1-M002`).

### In scope

- **God-file split**: `host_tool_executor.rs` refactored to keep exactly 3 caller entry points:
  1. CLI subcommand entry — `nexus42 host-call <tool_id> --args <json>` → normalizes to internal registry request shape → `registry::dispatch`.
  2. Worker `agent_tool_request` entry — receives IPC messages from orchestration worker → normalizes → `registry::dispatch`.
  3. HTTP `ToolExecuteRequest` entry — receives daemon HTTP POST `/v1/local/acp/tool/execute` → normalizes → `registry::dispatch`.
- **Wire codec per entry**: each entry normalizes its caller-specific wire format into the internal `HostToolRequest` shape; admission, handler, failure mode, and test vector live exclusively in `capability::Registry`.
- **Remove 7 duplicated `execute_X` functions**: their handlers now live in `capability/builtins/` (P0 migrated them); P1 removes the dead dispatch code.
- **File size check**: `host_tool_executor.rs` ≤ 800 lines post-refactor.
- **`host-call` subcommand**: `nexus42 host-call <tool_id> --args <json>` — walks the daemon host_tool path for low-level debugging; documented in `cli-spec.md` §6.2M.
- **CdnConfig constructor injection**: Replace global `RwLock<Option<CdnConfig>>` with constructor-injected config (closes `R-V156P1-M002`).
- **Cross-cut**: `R-V156P3-S003` (field drops) — remove unused capability fields exposed in caller adapters.

### Out of scope

- Registry consolidation (P0 owns handler migration).
- Worker IPC 35-ID extension (P3 owns full IPC matrix).
- `R-V156P1-M001` schema rename (P2 owns it).
- New capability implementations.
- `host-call` becoming a user-facing feature (it is debug-only; CLI help makes this explicit).
