## 1. Problem Statement

T-B P0 shipped N-way labeled routing: a single `llm_judge` state can fan out to 3+ target states via `LabeledNext` edges. However, when multiple `llm_judge` states in a preset graph each produce labeled edges that **converge** on the same target state, the orchestration engine has no way to know when to advance to that state. A "merge node" (or convergence state) needs explicit merge semantics so the engine knows:

1. **`wait-all` (default)**: advance only when ALL incoming labeled edges have produced their target label.
2. **`wait-any`**: advance on the FIRST incoming labeled edge result.
3. **`quorum N/M`**: advance when at least N of M incoming edges have produced their target.

Without explicit merge semantics, a state with 3 incoming labeled edges might advance on the first arrival (implicit any) or deadlock (implicit all) — the author has no control and no contract.

---

## 2. Goals

1. Add `merge:` field to state YAML: `all | any | quorum` with `quorum: { n: <int>, m: <int> }`.
2. Validator: `check_merge_node_integrity` — merge nodes must have ≥2 incoming labeled edges; quorum N/M must satisfy N ≤ M and N ≥ 1.
3. Runtime: merge tracking in `StateCompositeTask` — track incoming label arrivals per merge node; advance when merge condition met.
4. Backward compat: states without `merge:` field default to `wait-all`; existing binary GoNogo + N-way labeled presets continue to work.
5. Spec overlay body: `preset-conditional-routing.md` §3.2 documenting merge semantics.

## 4. Design

### 4.1 Data model

```rust
/// Merge semantics for states with multiple incoming labeled edges.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum MergeKind {
    /// Advance when ALL incoming labeled edges have produced their target.
    All,
    /// Advance on FIRST incoming labeled edge result.
    Any,
    /// Advance when at least N of M incoming edges have produced their target.
    Quorum { n: usize, m: usize },
}
```

Added to `StateDefinition`:
```rust
/// When multiple labeled edges converge on this state, when to advance.
#[serde(default, skip_serializing_if = "Option::is_none")]
pub merge: Option<MergeKind>,
```

### 4.2 YAML format

```yaml
states:
  - id: merge_review
    description: "Wait for all 3 reviewers"
    merge: all
    exit_when: { kind: manual }
    next: finalize

  - id: merge_quick
    merge: any
    exit_when: { kind: manual }
    next: finalize

  - id: merge_quorum
    merge:
      quorum:
        n: 2
        m: 3
    exit_when: { kind: manual }
    next: finalize
```

### 4.3 Runtime tracking

Each `StateCompositeTask` for a merge node tracks incoming label arrivals in context:

- **Context key**: `_merge_<state_id>` — a serialized JSON array of label strings that have arrived.
- **On labeled edge arrival**: the runner writes the label to this array (append).
- **On state entry**: check the merge condition against the incoming edge count (known from the graph topology):
  - `All`: count of unique arriving labels == expected incoming count → advance
  - `Any`: ≥1 arrival → advance
  - `Quorum { n, m }`: count ≥ n → advance
  - Otherwise: return `WaitForInput`
- **On advance**: clear `_merge_<state_id>` from context.

**Incoming count discovery**: the loader re-wires the outer graph so that for each `Labeled` edge targeting a merge state, the runtime context records which states have `Labeled` edges targeting which merge states. The merge node's composite task stores the expected incoming count (pre-computed at load time).

### 4.4 Validation rules

`check_merge_node_integrity`:
- Each state with `merge:` must have ≥2 incoming `Labeled` edges from other states.
- For `quorum N/M`: N ≥ 1 and N ≤ M.
- M must equal the count of incoming `Labeled` edges (author must declare the exact fan-in count).
- No orphan merge nodes: every merge state must have at least one outgoing edge.
