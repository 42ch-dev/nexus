## 1. Problem statement

V1.81 ships an additive wire change (`memory-fragment-info` + `world_id`; new
soul-narrative schemas) and a new on-demand synthesis path. P0‖P1 parallelize
only if the contract shapes are frozen first. This plan does the specify +
clarify + contract authoring + DESIGN.md token work up front, in the Prepare
phase, so neither implement track blocks on a contract decision.

## 2. Scope (Deliverables)

### 2.1 Wire contract shapes (authoritative for P0‖P1)

- `schemas/local-api/memory/memory-fragment-info.schema.json` — add optional
  property `world_id` with JSON Schema type `["string", "null"]`.
  Semantics: absent or `null` = Creator-core-only fragment (no originating
  world); string value = fragment emerged from that `world_id`. The existing
  `required` set stays unchanged: `fragment_id`, `summary` only.
- `schemas/local-api/memory/list-memory-fragments-query.schema.json` — add
  optional `world_id` query param with JSON Schema type `string`. Omitted =
  Creator SOUL whole (all fragments). Present = filter to that world's subset.
  There is no query value for core-only (`world_id IS NULL`) in V1.81; the UI
  only needs whole vs specific-world projection. The existing `required` set
  stays unchanged: `creator_id` only.
- `schemas/local-api/memory/soul-narrative-request.schema.json` — **new**:
  - `required`: `creator_id`
  - `creator_id`: `string` — must match active creator; endpoint rejects
    mismatches through the existing Local API auth/creator checks.
  - `force_regenerate`: optional `boolean`, default `false` — when `true`,
    synthesize and overwrite the cached narrative even if a current/stale cache
    row exists; when omitted/false, the endpoint is read/status-only (returns
    `ungenerated`, `current`, `stale`, or `insufficient_data` without invoking
    the LLM).
  - `additionalProperties`: `false`.
- `schemas/local-api/memory/soul-narrative-response.schema.json` — **new**:
  - `required`: `creator_id`, `state`, `stale`, `current_fragment_count`,
    `current_distinct_keyword_count`, `min_fragment_count`,
    `min_distinct_keyword_count`
  - `creator_id`: `string`
  - `state`: `string` enum `ungenerated | current | stale | insufficient_data`
  - `narrative`: optional `string` — present only for `current`/`stale` rows;
    omitted for `ungenerated` and `insufficient_data`.
  - `generated_at`: optional `string` — RFC 3339 timestamp, present only when a
    cached narrative exists.
  - `stale`: `boolean` — `true` only when a cached narrative exists and any new
    fragment arrived since generation (count or max-created-at snapshot differs).
  - `fragment_count_at_generation`: optional `integer` (minimum `0`) — snapshot
    persisted with the cached narrative; present only when a cached narrative
    exists.
  - `max_fragment_created_at_at_generation`: optional `string` — RFC 3339
    snapshot of the newest fragment included in generation; present only when a
    cached narrative exists and at least one fragment existed at generation time.
  - `current_fragment_count`: `integer` (minimum `0`) — current whole-Creator
    fragment count used for stale and insufficient-data decisions.
  - `current_distinct_keyword_count`: `integer` (minimum `0`) — current distinct
    keyword count across all Creator fragments.
  - `min_fragment_count`: `integer` (constant target `10`) — exposes the gate
    threshold to P1 copy/tests.
  - `min_distinct_keyword_count`: `integer` (constant target `20`) — exposes the
    gate threshold to P1 copy/tests.
  - `additionalProperties`: `false`.
- `packages/nexus-contracts/package.json` — 0.15.0 → 0.16.0 (**P-1 owns** the bump;
  committed alongside the codegen output).
- **Codegen + drift registration — P-1 owns**: run `pnpm run codegen`; commit
  generated Rust (`crates/nexus-contracts/src/generated/`) + TypeScript
  (`packages/nexus-contracts/src/generated/`); register the two new schemas in
  `crates/nexus-contracts/tests/schema_drift_detection.rs` `build_schema_map`;
  run `./tooling/check-wire-drift.sh`. Landing the generated types in P-1 is
  what lets P0 (Rust) and P1 (TS) parallelize against frozen output.

### 2.4 Architect open questions (resolve in this plan)

1. **Narrative persistence store — LOCKED: SQLite cache table, not SOUL file.**
   Rationale: the narrative is an on-demand Local API cache with stale metadata,
   not hand-authored SOUL identity text. Persist it in
   `memory_soul_narratives` via `nexus-local-db`:

   ```sql
   CREATE TABLE IF NOT EXISTS memory_soul_narratives (
       creator_id TEXT NOT NULL PRIMARY KEY,
       narrative TEXT NOT NULL,
       generated_at TEXT NOT NULL,
       fragment_count_at_generation INTEGER NOT NULL,
       max_fragment_created_at_at_generation TEXT,
       created_at TEXT NOT NULL,
       updated_at TEXT NOT NULL
   );
   ```

   `generated_at`, `fragment_count_at_generation`, and
   `max_fragment_created_at_at_generation` are the stale-detection snapshot.
   This table deliberately does **not** write to
   `$HOME/.nexus42/creators/<creator_id>/SOUL.md`; SOUL file I/O remains the
   durable human/agent identity document owned by `nexus-creator-memory`.
2. **Synthesis invocation seam — LOCKED: new trait plus daemon ACP adapter.**
   Grep result: the only production `SessionDigestSummarizer` impl is
   `PassthroughSummarizer` in
   `crates/nexus-daemon-runtime/src/api/handlers/memory.rs:952`; the remaining
   impls are tests/doc examples in `crates/nexus-creator-memory/src/review.rs`.
   The real production ACP prompt seam is not `SessionDigestSummarizer`; it is
   `ProductionWorkerProvider` (`crates/nexus-daemon-runtime/src/worker_provider.rs`)
   injected into `CapabilityRegistry::with_runtime_deps`, consumed by
   LLM-backed capabilities such as `context.summarize` and `acp.prompt`.

   P0 adds `SoulNarrativeSynthesizer` in `nexus-creator-memory` (domain seam,
   no daemon dependency):

   ```rust
   #[allow(async_fn_in_trait)]
   pub trait SoulNarrativeSynthesizer: Send + Sync {
       async fn synthesize(
           &self,
           creator_id: &str,
           input: SoulNarrativeSynthesisInput,
       ) -> Result<SoulNarrativeDraft, MemoryError>;
   }
   ```

   `SoulNarrativeSynthesisInput` contains the capped signal shape below;
   `SoulNarrativeDraft` contains `narrative: String` plus optional diagnostic
   fields kept internal. The real adapter lives in
   `crates/nexus-daemon-runtime/src/api/handlers/memory.rs` (or a sibling
   `memory_soul.rs` module if P0 splits the handler for size) as
   `AcpSoulNarrativeSynthesizer`. It obtains the production ACP path from
   `WorkspaceState::capability_registry()`, calls the registered `acp.prompt`
   capability with context-injected `_creator_id` and a daemon-owned session id
   such as `soul_narrative_reflect`, and extracts `full_text`. Missing registry,
   missing `acp.prompt`, or `WorkerUnavailable` maps to `NexusApiError` through
   the handler (no ad-hoc JSON). The endpoint depends on the trait, not on ACP
   transport details.
3. **Synthesis input cap — LOCKED: signal aggregate, never raw fragment dump.**
   P0 builds `SoulNarrativeSynthesisInput` from the whole Creator SOUL only:
   - Top keywords: at most 30 `(keyword, count)` pairs, descending count.
   - Recent summaries: at most 24 fragment summaries, ordered newest first,
     each truncated to 280 Unicode scalar chars.
   - Temporal signal: at most 8 monthly/weekly buckets (implementation may use
     month buckets when the span is > 8 weeks; otherwise week buckets), each
     with `bucket_start`, `fragment_count`, and top 5 keywords for that bucket.
   - Totals: current fragment count, distinct keyword count, oldest/newest
     fragment timestamps.
   The prompt contract must enforce specificity + temporality + actionable-tone
   using these signals; it must not include `memory_fragments.summary` beyond the
   capped recent-summary list and must never include raw session digests.

### 2.5 Product-side open questions (resolve in this plan)

These are product-owner decisions the architect needs locked before P0 implement:

1. **Insufficient-data threshold — LOCKED**: the narrative endpoint does not
   synthesize when `current_fragment_count < 10` **or**
   `current_distinct_keyword_count < 20`. It returns
   `state: "insufficient_data"` plus the current counts and thresholds. This is
   efficient enough in P0: fragment count is a `COUNT(*)` over `creator_id`, and
   distinct keyword count may initially decode the bounded creator keyword rows
   in Rust (same JSON-array storage model as `get_all_keywords`) unless P0 finds
   an existing normalized keyword table.
2. **Narrative quality gate — LOCKED**: all three axes remain required in the
   prompt contract: specificity (≥2 theme keywords), temporality (≥1 shift from
   the temporal buckets), and actionable tone (closing reflection/question).
   The capped signal shape supports all three. If the LLM output fails local
   structural validation, P0 returns a generation error (`NexusApiError`, not a
   partial narrative) rather than silently dropping an axis.
3. **Omitted worlds — LOCKED**: the selector lists worlds with creative activity:
   at least one fragment in `memory_fragments.world_id` **or** at least one Work
   bound to that `world_id` in the existing Works data. Worlds with neither
   fragments nor Works are omitted. Architecture split: P0's new fragment DAO can
   efficiently distinguish fragment-backed worlds via the
   `(creator_id, world_id, created_at)` index; P1 may combine that with the
   existing Works query/client data to include Work-backed worlds that do not yet
   have fragments. Selecting a Work-backed/no-fragment world shows the subset
   empty state; it does not create a per-world narrative.
4. **Stale granularity — LOCKED**: any new fragment makes the cached narrative
   stale. Algorithm: stale is true when current fragment count differs from
   `fragment_count_at_generation` **or** current max fragment `created_at` differs
   from `max_fragment_created_at_at_generation`. The banner is a prompt, not a
   forced regeneration.
